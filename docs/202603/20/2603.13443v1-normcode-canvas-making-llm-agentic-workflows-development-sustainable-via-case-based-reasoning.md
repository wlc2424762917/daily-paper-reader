## 摘要
我们介绍了 NormCode Canvas (v1.1.3)，这是一个已部署的系统，为多步大语言模型（LLM）工作流实现了两个层级的基于案例的推理（Case-Based Reasoning）。其基础是 NormCode，一种半形式化的规划语言，其经编译器验证的作用域规则（scope rule）确保每个执行检查点（checkpoint）都是一个真正自包含的案例，从而消除了标准编排框架中导致检索不可靠和故障无法定位的隐式共享状态。第一层级将每个检查点视为一个具体案例（挂起的运行时）；Fork 实现了“检索与重用”，Value Override 实现了带有自动陈旧边界传播（stale-boundary propagation）的修订。第二层级将每个编译后的计划视为一个抽象案例；编译流水线本身就是一个 NormCode 计划，从而实现了递归案例学习。由此产生了三个结构特性：(C1) 直接检查点视察；(C2) 通过编译器生成的叙述进行执行前审查；(C3) 作用域限定的选择性重新执行。四个已部署的计划作为结构化证据：PPT 生成（PPT Generation）在商业 API 上以每张幻灯片约 40 秒的速度生成演示文稿；代码助手（Code Assistant）执行跨越至多十个推理周期的多步软件工程任务；NC 编译（NC Compilations）将自然语言规范转换为可执行的 NormCode 计划；而 Canvas 助手（Canvas Assistant）在连接到外部 AI 代码编辑器时，可自动进行计划调试。这些计划共同构成了一个自维持的生态系统，其中计划相互生成、调试和优化，从而在系统规模上实现了累积的基于案例的学习。

## Abstract
We present NormCode Canvas (v1.1.3), a deployed system realizing Case-Based Reasoning at two levels for multi-step LLM workflows. The foundation is NormCode, a semi-formal planning language whose compiler-verified scope rule ensures every execution checkpoint is a genuinely self-contained case -- eliminating the implicit shared state that makes retrieval unreliable and failure non-localizable in standard orchestration frameworks. Level 1 treats each checkpoint as a concrete case (suspended runtime); Fork implements retrieve-and-reuse, Value Override implements revision with automatic stale-boundary propagation. Level 2 treats each compiled plan as an abstract case; the compilation pipeline is itself a NormCode plan, enabling recursive case learning. Three structural properties follow: (C1) direct checkpoint inspection; (C2) pre-execution review via compiler-generated narrative; (C3) scope-bounded selective re-execution. Four deployed plans serve as structured evidence: PPT Generation produces presentation decks at ~40s per slide on commercial APIs; Code Assistant carries out multi-step software-engineering tasks spanning up to ten reasoning cycles; NC Compilations converts natural-language specifications into executable NormCode plans; and Canvas Assistant, when connected to an external AI code editor, automates plan debugging. Together these plans form a self-sustaining ecosystem in which plans produce, debug, and refine one another -- realizing cumulative case-based learning at system scale.

#### 1. 核心问题与整体含义（研究动机和背景）
当前多步大语言模型（LLM）工作流（如 LangChain, AutoGen 等）在开发和维护上面临**不可持续性**的问题。主要挑战在于：
*   **隐式状态依赖**：中间步骤往往依赖于隐藏的对话历史或可变对象，导致执行检查点（checkpoint）并非自包含。
*   **故障定位难**：当第 $k$ 步失败时，原因可能潜伏在之前的任何步骤中，难以回溯。
*   **复用与调试成本高**：由于缺乏结构化隔离，重新运行或修改工作流往往需要从头开始，浪费算力且难以信任重放结果。
**研究动机**：提出一种半正式的规划语言和系统，使推理过程显式化、可检查、可验证，并引入“基于案例的推理”（CBR）框架来管理工作流的生命周期。

#### 2. 方法论：核心思想与关键技术
论文提出了 **NormCode Canvas** 系统，其核心思想是利用 **NormCode** 语言的“作用域规则”实现两级 CBR：
*   **核心技术：NormCode 语言**
    *   **作用域规则（Scope Rule）**：强制要求每个推理步骤只能访问其显式声明的输入块，消除隐式上下文。
    *   **语法标记**：使用 `<-`（赋值）、`<=`（功能函数）、`<*`（循环上下文）三个简单标记构建复杂逻辑。
    *   **三种形态**：`.ncds`（人类可读源码）、`.ncd`（编译器形式化计划）、`.ncn`（自然语言叙述，用于执行前审查）。
*   **两级 CBR 架构**：
    *   **Level 1（具体案例管理）**：将每个执行检查点视为一个“挂起的运行时”。支持 **Fork**（从任意点分支重用）和 **Value Override**（修改中间值并触发选择性重执行）。
    *   **Level 2（抽象案例学习）**：将编译后的计划视为可复用的模式。值得注意的是，**编译流水线本身也是一个 NormCode 计划**，实现了递归的自我演进。
*   **三大结构特性**：
    *   **(C1) 直接检查点视察**：无需重建日志即可查看每一步的精确输入输出。
    *   **(C2) 执行前审查**：通过编译器生成的自然语言叙述（.ncn）在运行前验证逻辑。
    *   **(C3) 范围受限的选择性重执行**：修改某一点后，仅重新运行受影响的下游节点。

#### 3. 实验设计
论文并未采用传统的单一数据集 Benchmark（如 MMLU），而是通过**四个部署在生产环境中的复杂工作流**作为结构化证据：
1.  **PPT 生成（PPT Generation）**：处理复杂的嵌套循环，每张幻灯片生成 HTML 和 PPTX 两种格式。
2.  **代码助手（Code Assistant）**：实现 ReAct 循环（探索、计划、执行、验证），展示多步软件工程任务的处理。
3.  **NC 编译（NC Compilations）**：自我托管的编译器，将自然语言需求转化为可执行计划。
4.  **Canvas 助手（Canvas Assistant）**：连接外部编辑器，自动调试和修复计划。
**对比对象**：在讨论中对比了 LangChain、LangGraph 等主流框架在“结构隔离”和“案例完整性”上的缺失。

#### 4. 资源与算力
*   **模型支持**：实验使用了多种商业 API，包括 `qwen-plus`、`gpt-4o`、`claude-3-opus`、`qwen-turbo` 和 `GLM-4`。

#### 5. 实验数量与充分性
*   **实验规模**：展示了四个具有代表性的生产级计划，涵盖了顺序流、嵌套循环、ReAct 循环和递归编译等多种控制流模式。
*   **充分性评价**：实验在**功能验证**和**结构特性证明**上是充分的。它证明了 C1-C3 特性在不同领域（办公、编程、元编程）的通用性。然而，由于缺乏大规模的定量准确率对比（如在标准智能体基准测试上的得分），其在“任务最终质量”上的优越性主要通过定性案例展示。

#### 6. 主要结论与发现
*   **结构化隔离是关键**：通过编译器强制执行的作用域规则，可以彻底解决 LLM 工作流中状态混乱的问题。
*   **确定性与随机性的平衡**：生产计划中 60-70% 的步骤可以是确定性的语法操作，这大大提升了系统的可靠性和经济性。
*   **自我维持的生态系统**：通过 Level 2 CBR，计划可以相互生成、调试和优化，实现了系统级的累积学习。

#### 7. 优点
*   **极高的可解释性**：`.ncn` 叙述让非技术专家也能在运行前审查 AI 的计划逻辑。
*   **调试效率革命**：选择性重执行（C3）和直接视察（C1）极大地缩短了开发者定位错误和迭代修复的周期。
*   **语言无关性**：支持中文等多种语言编写概念名，对全球开发者友好。

#### 8. 不足与局限
*   **检索依赖人工**：目前的案例检索仍主要依靠操作员判断，尚未实现全自动的基于语义相似度的案例推荐。
*   **存储压力**：随着运行次数增加，案例库会无限制增长，需要引入“基于能力的剪枝”机制。
*   **外部环境依赖**：虽然检查点是自包含的，但如果引用的外部资源（如文件路径）发生变化，重用可能会失败。
*   **自动化边界**：对于逻辑极其复杂的任务，自动编译器（NC Compilations）仍需人工干预或代码助手辅助。
