## 摘要
随着大语言模型（LLM）智能体越来越多地部署在软件工程等开放领域，它们经常会遇到缺乏关键上下文的、描述不明确的指令。虽然人类开发人员通常通过提出澄清性问题来解决描述不明确的问题，但目前的智能体在很大程度上是针对自主执行进行优化的。在这项工作中，我们在 SWE-bench Verified 的一个描述不明确的变体上，系统地评估了 LLM 智能体的澄清寻求能力。我们提出了一种具有不确定性感知能力的多智能体脚手架，它明确地将描述不明确的检测与代码执行解耦。我们的结果表明，使用 OpenHands + Claude Sonnet 4.5 的多智能体系统实现了 69.40% 的任务解决率，显著优于标准的单智能体设置（61.20%），并缩小了与在完全明确指令下运行的智能体之间的性能差距。此外，我们发现该多智能体系统表现出良好的不确定性校准，在简单任务上节省查询，而在更复杂的问题上主动寻求信息。这些发现表明，当前的模型可以转变为主动的协作伙伴，智能体能够独立识别何时提出问题，以在现实世界的描述不明确的任务中获取缺失的信息。

## Abstract
As Large Language Model (LLM) agents are increasingly deployed in open-ended domains like software engineering, they frequently encounter underspecified instructions that lack crucial context. While human developers naturally resolve underspecification by asking clarifying questions, current agents are largely optimized for autonomous execution. In this work, we systematically evaluate the clarification-seeking abilities of LLM agents on an underspecified variant of SWE-bench Verified. We propose an uncertainty-aware multi-agent scaffold that explicitly decouples underspecification detection from code execution. Our results demonstrate that this multi-agent system using OpenHands + Claude Sonnet 4.5 achieves a 69.40% task resolve rate, significantly outperforming a standard single-agent setup (61.20%) and closing the performance gap with agents operating on fully specified instructions. Furthermore, we find that the multi-agent system exhibits well-calibrated uncertainty, conserving queries on simple tasks while proactively seeking information on more complex issues. These findings indicate that current models can be turned into proactive collaborators, where agents independently recognize when to ask questions to elicit missing information in real-world, underspecified tasks.

    *   **意图智能体（Intent Agent）**：充当“监视者”，在每一轮迭代中分析对话历史和主智能体的动作。其唯一职责是判断当前上下文是否存在歧义或信息缺失。
    *   **强制干预机制**：一旦意图智能体检测到不确定性，它会强制主智能体的下一个动作必须是“询问用户（clarify）”，从而阻止盲目执行。
*   **对比架构（UA-SINGLE）**：在单智能体循环中加入提示词，要求模型在每一步执行前先进行“歧义评估”。
*   **用户模拟器**：使用 GPT-5.1 扮演用户，持有原始完整的 Issue 信息，仅在智能体提问时提供被隐藏的细节。

### 3. 实验设计
*   **数据集**：**SWE-bench Verified** 的欠规范变体。研究者使用 GPT-4o 将原始 500 个高质量 GitHub Issue 进行了简化，删除了关键的复现细节，仅保留高层描述。
*   **Benchmark**：以任务解决率（Resolve Rate）为核心指标。
*   **对比方法**：

    5.  **UA-MULTI**：本文提出的多智能体解耦架构。

### 4. 资源与算力
*   **模型使用**：主要实验基于 **Claude Sonnet 4.5** 作为智能体骨干模型。

*   **实验规模**：在全部 500 个 SWE-bench Verified 任务上进行了完整测试。

### 6. 主要结论与发现
*   **性能提升**：UA-MULTI 达到了 **69.40%** 的解决率，显著优于 UA-SINGLE（61.20%），并几乎追平了拥有完整信息的 FULL 基线（70.80%）。
*   **解耦的必要性**：实验证明，即使是顶尖模型（如 Sonnet 4.5），在单智能体模式下也很难同时兼顾“执行代码”和“监控自身不确定性”。
*   **良好的校准性**：UA-MULTI 表现出“知之为知之”的特性。在简单任务上，它能识别出信息已足够并减少提问；在复杂任务上，其提问率显著升高。
*   **提问策略**：UA-MULTI 倾向于在任务的中早期发起更具体、更具技术性的询问，而单智能体往往在任务末尾（尝试失败后）才被动提问。

### 7. 优点
*   **架构简洁有效**：通过简单的角色分工（Intent vs Main）解决了复杂的认知冲突问题。
*   **动态性强**：不同于以往“先问一轮再干活”的静态模式，该系统支持在执行过程中（如发现测试报错后）动态发起后续澄清。
*   **实用性高**：直接针对软件工程中的真实痛点（Issue 描述不清），具有很强的落地潜力。

### 8. 不足与局限
*   **用户模拟器偏差**：实验依赖 LLM 模拟用户，可能无法完全反映真实人类用户的多变性或不耐烦情绪。
*   **模型泛化限制**：目前仅在最强模型（Sonnet 4.5）上验证，较弱的开源模型可能缺乏足够的不确定性校准能力。
*   **推理开销**：多智能体架构带来了更高的 Token 消耗和延迟，在追求极致响应速度的场景下可能受限。
*   **评估范围**：虽然在 SWE-bench 上表现良好，但在安全关键型（Security-critical）任务中的可靠性尚未验证。
