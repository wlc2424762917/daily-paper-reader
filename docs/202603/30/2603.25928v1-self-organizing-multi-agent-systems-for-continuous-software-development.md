## 摘要
基于大语言模型的多智能体系统在自动化软件开发任务方面展现出巨大潜力。然而，大多数现有系统主要关注完成小任务和增量代码更改，而持久、持续的软件开发在很大程度上仍未得到充分探索。我们提出了 TheBotCompany，这是一个用于持续多智能体软件开发的开源编排框架。TheBotCompany 引入了三项关键创新：(1) 用于里程碑驱动开发的阶段性状态机（从策略到执行再到验证）；(2) 自组织智能体团队，其中管理智能体根据项目需求动态雇佣、分配和解雇工作智能体；(3) 异步人类监督。我们在真实软件项目上对 TheBotCompany 进行了为期数天的持续开发评估，衡量了团队适应模式、里程碑完成率、成本效率和代码质量。我们的结果表明，自组织方法能够实现具有可衡量进展的有效长期软件开发，而验证阶段则能捕获原本会持续存在的缺陷。

## Abstract
Large Language Model-based multi-agent systems have shown promise in automating software development tasks. However, most vibe code systems focus on completing small tasks and incremental code changes, leaving persistent, continuous software development largely unexplored. We present TheBotCompany, an open-source orchestration framework for continuous multi-agent software development. TheBotCompany introduces three key innovations: (1) a three-phase state machine (Strategy to Execution to Verification) for milestone-driven development, (2) self-organizing agent teams where manager agents dynamically hire, assign, and fire worker agents based on project needs, and (3) asynchronous human oversight. We evaluate TheBotCompany on real-world software projects over multiple days of continuous development, measuring team adaptation patterns, milestone completion rates, cost efficiency, and code quality. Our results demonstrate that the self-organizing approach enables effective long-term software development with measurable progress, while the verification phase catches defects that would otherwise persist.

### 1. 核心问题与整体含义（研究动机和背景）
传统的基于大语言模型（LLM）的软件开发系统（如 ChatDev、MetaGPT 或 SWE-agent）主要存在三个局限：
*   **缺乏持久编排能力**：现有系统多为“单次会话”模式，处理完一个任务即终止，难以维持长达数周的开发上下文或在中断后自动恢复。
*   **团队结构静态**：智能体角色通常在设计时固定，无法根据项目演进（如从架构设计转向安全加固）动态调整团队构成。
*   **人类干预同步化**：现有框架通常要求人类实时监控或预先设定需求，缺乏像真实开发中“提 Issue-异步处理”的协作模式。

本研究提出了 **TheBotCompany (TBC)**，旨在构建一个能够实现**持续、自组织、里程碑驱动**的多智能体软件开发框架，模拟真实软件团队的长期运作流程。

### 2. 方法论：核心思想与关键技术
TheBotCompany 的核心是一个**三阶段状态机循环**，配合自组织的管理架构：

*   **三阶段里程碑生命周期**：
    1.  **策略阶段 (Strategy)**：由管理智能体 **Athena** 负责。评估项目状态，定义下一个里程碑（Milestone）的目标、预算，并维护长期路线图。
    2.  **执行阶段 (Execution)**：由管理智能体 **Ares** 负责。根据里程碑需求，动态雇佣、分配任务给工作智能体（Workers），并监控进度。
    3.  **验证阶段 (Verification)**：由管理智能体 **Apollo** 负责。独立评估交付物，通过运行测试和代码审查给出“通过/失败”判定。若失败，则触发带有反馈的修复循环，且修复预算呈指数级衰减以防死循环。
*   **自组织团队机制**：仅保留三个管理角色（Athena, Ares, Apollo），工作智能体由管理者根据“技能文件”（Skill Files）动态创建和注销。
*   **持久化与通信**：使用 SQLite 数据库作为内部 Issue Tracker，记录所有决策、评论和代码变更，确保系统在重启后能无缝恢复。
*   **异步人类监督**：通过 Web 仪表盘，人类用户可以随时提交 Issue，管理智能体会在下一个策略周期自动拾取，实现非阻塞式干预。

### 5. 实验数量与充分性
*   **实验规模**：在长期项目中，系统共完成了 **164 个里程碑**，经历了 **616 个智能体周期**，累计运行时间达 **137 小时**。

### 6. 主要结论与发现
*   **持续开发可行性**：TBC 证明了多智能体系统可以在无人类干预的情况下，持续数天推进复杂软件项目，且不会陷入退化状态。
*   **多智能体优势**：在极高难度的任务上，TBC 的得分显著超过单智能体基线（如在某些任务上 Claude Code 得分接近 0，而 TBC 能获得有效分数），证明了“策略-执行-验证”分离的结构优势。
*   **验证阶段的价值**：在具有模糊质量目标的任务（如像素级相似度匹配）中，验证阶段能有效拦截回归缺陷，并通过积累“反模式”经验指导后续规划。
*   **成本特征**：工作智能体占据了约 70% 的成本，管理开销随项目规模增长保持稳定，未出现成本爆炸。

### 7. 优点与亮点
*   **架构解耦**：将“生产”与“评判”职能分离，模拟了工业界的 QA 流程，显著提升了长程任务的可靠性。
*   **动态适应性**：自组织团队允许系统根据任务难度自动调整人力，比固定角色的 ChatDev 等更具灵活性。
*   **持久化设计**：引入 SQLite 和 Issue Tracker 机制，解决了 LLM 智能体常见的“上下文遗忘”和“状态丢失”问题。

### 8. 不足与局限
*   **效率开销**：在简单或紧凑的任务中，多智能体编排的 Token 消耗是单智能体的 **1.65 倍**，且可能因过度设计导致性能反而下降。
*   **领域偏差**：实验主要集中在确定性较强的系统级项目（模拟器、编译器），对于需求高度模糊或涉及复杂 UI/UX 的项目评估不足。
*   **上下文窗口限制**：尽管有持久化存储，但管理智能体在处理超过 90 个里程碑的长周期项目时，仍会因上下文窗口限制而遗忘早期的细微决策（如误判某些已证实的“反模式”）。
