## 摘要
仓库级代码智能体在现实世界的功能添加任务中展现出巨大的潜力，这使得对其能力的可靠评估变得日益重要。然而，现有的基准测试主要基于最终测试的正确性将这些智能体视为黑盒进行评估，对于它们的推理方式以及失败原因提供的见解有限。为了解决这一局限性，我们推出了 RACE-bench，这是一个用于评估仓库级功能添加任务中代码智能体的推理增强型基准测试。RACE-bench 包含来自 12 个开源仓库的 528 个真实功能添加实例。每个实例都配有可执行的补丁验证和结构化的中间推理真值，涵盖了问题理解、文件定位、实现任务和步骤分解。基于这一设计，我们引入了一个双轨评估框架，共同衡量补丁正确性和中间推理质量。我们在 RACE-bench 上评估了三个具有代表性的仓库级代码智能体。在完整基准测试中，不同智能体的解决率在 29% 到 70% 之间。我们的推理级分析进一步表明，虽然目前的智能体在理解高层意图方面表现良好，但在将意图转化为具体的实现步骤时，其性能会大幅下降。我们还发现，与成功案例相比，“应用成功但测试失败”的案例表现出更低的推理召回率（下降 35.7%）和更高的过度预测（增加 94.1%）。这些发现强调了通过检查推理过程的质量，在最终补丁正确性之外评估仓库级代码智能体的重要性。

## Abstract
Repository-level code agents have shown strong promise in real-world feature addition tasks, making reliable evaluation of their capabilities increasingly important. However, existing benchmarks primarily evaluate these agents as black boxes based on final test correctness, providing limited insight into how they reason and where failures arise. To address this limitation, we introduce RACE-bench, a reasoning-augmented benchmark for evaluating code agents on repository-level feature addition tasks. RACE-bench contains 528 real-world feature addition instances from 12 open-source repositories. Each instance is paired with executable patch verification and structured intermediate reasoning ground truth covering issue understanding, file localization, implementation tasks, and step decomposition. Based on this design, we introduce a dual-track evaluation framework that jointly measures patch correctness and intermediate reasoning quality. We evaluate three representative repository-level code agents on RACE-bench. On the full benchmark, Resolved Rates range from 29% to 70% across different agents. Our reasoning-level analysis further shows that while current agents perform well at understanding high-level intent, their performance degrades substantially when translating intent into concrete implementation steps. We also find that apply-success but test-fail cases exhibit lower reasoning recall (35.7% decrease) and higher over-prediction (94.1% increase) compared to successful cases. These findings highlight the importance of evaluating repository-level code agents beyond final patch correctness by examining the quality of their reasoning processes.

随着大语言模型（LLM）驱动的代码智能体（Code Agents）在处理仓库级任务（如功能添加、Bug 修复）中展现出潜力，如何准确评估其能力成为关键。现有的基准测试（如 SWE-bench）主要采用“黑盒评估”模式，即仅根据最终生成的补丁是否通过测试用例来判断成败。这种方式无法揭示智能体在复杂开发流程中的推理逻辑，也难以诊断其在哪个环节（如需求理解、文件定位、任务规划）出现了偏差。
**研究动机**：填补仓库级功能添加任务中“推理过程评估”的空白，通过结构化的中间推理真值，深入分析智能体的失败模式。

#### 2. 论文提出的方法论
论文提出了 **RACE-bench** 基准测试及其配套的**双轨评估框架**：
*   **核心思想**：将软件开发流程抽象为四个推理阶段（五个模块），并为每个实例构建“推理真值”（Reasoning Ground Truth, RGT）。

    1.  **需求理解**：包括领域概念解释和目标预期总结。
    2.  **文件定位**：识别必要的代码文件、辅助文件和测试文件。
    3.  **任务实现**：识别具体的代码实体（函数/方法）及其修改类型（新增/修改/删除）。
    4.  **步骤分解**：将实现过程抽象为最小化的语义步骤序列（如引入新功能、重用现有语义等）。
*   **关键技术流程**：
    *   **实例构建**：从 12 个开源 Python 仓库中提取 PR，通过执行驱动的过滤确保补丁有效且包含测试。
    *   **真值生成**：结合 LLM 辅助生成与基于补丁的程序分析（如测试驱动的消融实验来确定“必要文件”）。

*   **实验规模**：在 528 个全量实例上测试了 3 种智能体，并在 100 个 Lite 实例上进行了跨模型（3 种 LLM）的交叉验证。
*   **充分性**：实验不仅关注成功率，还对“应用成功但测试失败”的案例进行了深入的推理分析，并对比了不同智能体的“探索-利用”风格。

#### 6. 论文的主要结论与发现
*   **性能差异**：不同智能体的解决率差异巨大（29% - 70%），mini-SWE-Agent 表现最优。
*   **“瀑布式”性能下降**：智能体在“高层意图理解”上表现极佳（得分 > 9.2/10），但在将其转化为“具体执行步骤”时能力大幅衰减。
*   **失败原因诊断**：测试失败的案例通常伴随着推理召回率的显著下降（-35.7%）和过度预测的激增（+94.1%），即智能体定位不准且修改了过多无关代码。
*   **架构 vs 模型**：优秀的智能体架构（如 mini-SWE-Agent）能显著降低对基础模型的依赖，表现出更强的鲁棒性。
*   **推理风格**：AutoCodeRover 偏向保守（低过度预测，低召回）；TraeAgent 偏向探索（高召回，高过度预测）。

#### 7. 优点
*   **细粒度评估**：突破了传统的黑盒评估，提供了从文件到语义步骤的多层级推理真值。
*   **真实性强**：基于真实世界的 PR 和功能请求，而非合成数据。
*   **诊断价值**：通过召回率和过度预测指标，为开发者改进智能体提供了明确的方向（如加强步骤分解能力）。
*   **环境完备**：提供了统一的 Docker 镜像，解决了仓库级评估中环境配置难的痛点。

#### 8. 不足与局限
*   **语言局限性**：目前仅覆盖 Python 语言，其他编程语言的泛化性有待验证。
*   **LLM 评估偏差**：虽然使用了 LLM-as-Judge，但语义匹配环节仍可能存在模型固有的偏见。
*   **真值构建成本**：尽管有 LLM 辅助，但构建高质量推理真值的流程较为复杂，难以快速扩展到数万个实例。
*   **静态真值 vs 动态路径**：软件开发具有“等效性”（Equifinality），智能体可能通过不同于真值的路径实现正确功能，这会导致召回率指标在某些成功案例中偏低。
