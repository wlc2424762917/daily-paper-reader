## Abstract 

Agent skills are modular instruction packages that combine YAML metadata, natural language instructions, and embedded code, and they have reached 196K publicly available instances, yet no mecha-nism exists to detect clone relationships among them. This gap cre-ates systemic risks: a vulnerability in a widely copied skill silently persists across derivatives with no alert to maintainers. Existing clone detectors, designed for single-modality source code, can-not handle the multi-modal structure of skills, where clone ev-idence is distributed across three interleaved content channels. We present SkillClone , the first multi-modal clone detection ap-proach for agent skills. SkillClone fuses flat TF-IDF similarity with per-channel decomposition (YAML, NL, code) through lo-gistic regression, combining strong detection with interpretable type classification. We construct SkillClone-Bench , a balanced benchmark of 300 ground-truth pairs with stratified difficulty. On 

SkillClone-Bench , SkillClone achieves F1 of 0.939 with preci-sion 0.952, outperforming flat TF-IDF (F1 = 0.881) and achieving 4.2 × higher Type-4 (semantic) recall than MinHash. Applying Skill-Clone to 20K skills reveals 258K clone pairs involving 75% of all skills, with 40% crossing author boundaries. A deduplication anal-ysis shows the ecosystem is inflated 3.5 ×: only 5,642 unique skill concepts underlie the 20K listed skills, and 41% of skills in clone families are superseded by a strictly better variant.

## 1 Introduction 

Large language model (LLM) agents have rapidly evolved from standalone chatbots into extensible platforms that delegate spe-cialized tasks to modular instruction packages called agent skills .A skill is a structured document, typically a SKILL.md file, that bundles YAML metadata, natural language instructions, and em-bedded code blocks into a single artifact that an LLM agent such as Claude Code, Cursor, or Windsurf can load at runtime to acquire new capabilities. Since the public launch of skill ecosystems in late 2025, adoption has been explosive: over 196K skills are now publicly available on registries and GitHub repositories, spanning domains from code generation and security analysis to DevOps automation. This growth mirrors the early trajectories of package ecosystems like npm and PyPI [ 4, 42 ], which took years to reach comparable scale. Recent surveys [ 38 ] and empirical studies [ 20 ] have begun to characterize this ecosystem’s architecture and growth dynamics. Despite this rapid expansion, the agent skill ecosystem lacks fundamental infrastructure that traditional package ecosystems take for granted. Skills declare no dependencies on other skills, carry no version identifiers, and provide no provenance metadata linking derived skills to their origins. When a developer copies, adapts, or extends an existing skill, this relationship is entirely invisible to the ecosystem. The consequence is a growing body of duplicated and near-duplicated content whose relationships cannot be traced, audited, or monitored. This situation creates concrete security risks: when a vulnerability is discovered in a widely copied skill, there is no mechanism to identify which other skills share its content and therefore share the same flaw. Unlike a CVE advisory that can propagate through a dependency graph in npm [ 21 ], a flawed skill pattern can silently persist across hundreds of derivative skills with no alert to their maintainers. The January 2026 ClawHavoc campaign [ 15 ], which infiltrated over 1,200 malicious skills into the OpenClaw registry, demonstrated the real-world consequences of this gap. Existing tools for skill analysis treat each skill as an isolated unit. SkillScan [ 23 ] performs static security checks on individual skill files. SkillFortify [ 2 ] validates skill structure and metadata qual-ity. MalTool [ 22 ] detects malicious intent in single skills through behavioral analysis. SkillJect [ 11 ] and SKILL-INJECT [ 33 ] have demonstrated that skills serve as potent attack vectors for prompt injection [ 32 ]. Yet none of these tools examine relationships between 

skills, and no prior work has studied clone patterns at ecosystem

The absence of such analysis means that the community cannot answer basic questions: How prevalent is skill cloning? Do clone relation-ships cross organizational boundaries? What is the blast radius when a hub skill contains a vulnerability? The key insight behind our work is that agent skills are inher-ently multi-modal documents that resist analysis by any single simi-larity technique. For the dominant SKILL.md format studied in this paper, we scope the artifact to its three primary in-file modalities: YAML metadata specifying the skill’s name, tags, and configura-tion; natural language prose describing when and how to apply the skill; and embedded code blocks implementing concrete func-tionality. This scope targets the content carried directly inside the skill file, not auxiliary repository artifacts such as images, linked documents, or external scripts. Detecting clones requires fusing similarity signals across these three channels, because two skills may share identical code blocks while differing in their natural language framing, or may paraphrase the same instructions while referencing different code examples. Preliminary ecosystem analysis revealed that clone relationships are not rare edge cases but a dominant organizing force in the ecosystem: most skills participate in at least one clone relationship, a substantial fraction of clones cross author boundaries, and malicious patterns often propagate through these latent reuse links. These observations motivate a hybrid contribution. We study undeclared reuse empirically at ecosystem scale, and we introduce a multi-modal clone detector designed specifically for the mixed YAML, natural-language, and code structure of skills. To address this gap, we present SkillClone , the first approach for detecting clone relationships across the agent skill ecosystem. 

SkillClone implements a three-channel clone detection approach that synergistically computes similarity over YAML metadata, nat-ural language instructions, and embedded code blocks, then fuses these signals through a weighted aggregation scheme to produce a unified clone score for each skill pair. We apply SkillClone to a representative sample of 20K skills drawn from the 196K-skill corpus crawled from GitHub and construct the first ecosystem-wide clone graph , where nodes represent skills and edges represent detected clone relationships. Over this graph, we perform blast radius analysis to quantify how a vulnerability in a single skill can propagate through chains of clones. An important distinction underlies our work: clone detection identifies structural similarity between artifacts, but does not by itself establish causal reuse (i.e., that one skill was derived from another). Establishing reuse requires additional provenance signals such as fork relationships, commit timestamps, or explicit attri-bution. We validate that our clone detection enables downstream reuse discovery by confirming provenance in fork-mined pairs and identifying real-world clone-and-rebrand cases (Section 2.3). For se-curity applications, the distinction is immaterial: whether two skills share a vulnerability through copying or convergent development, the blast radius is the same. Our analysis reveals several findings with implications for ecosys-tem governance. Among the 258K clone relationships detected by 

SkillClone , 40% cross author boundaries, indicating substantial cross-organizational cloning. A deduplication analysis shows the ecosystem is inflated 3.5 ×: only 5,642 unique skill concepts un-derlie the 20K listed skills, and 41% of skills in clone families are superseded by a strictly better variant. Security propagation anal-ysis identifies 141 real security-relevant skills whose dangerous patterns (SQL injection payloads, reverse shells, XSS vectors) prop-agate to 1,100 clones across 119 affected authors. These findings demonstrate that per-skill security tools are insufficient and that ecosystem-level clone analysis is essential for managing systemic risk. In summary, this paper makes the following contributions: 

• An empirical characterization of undeclared reuse across 20K agent skills, revealing ecosystem-scale clone prevalence, cross-author reuse, and the dominance of Type-4 semantic clones (Sec-tion 4). 

• The first multi-modal clone detection approach for agent skills, fusing YAML, natural language, and code similarity signals through cross-modal alignment to detect clone relationships in heteroge-neous skill documents (Section 3). 

• A blast-radius analysis over the inferred clone graph, quantifying how malicious or vulnerable skill patterns propagate through latent reuse links that per-skill scanners cannot see (Section 4). 

• SkillClone-Bench , a balanced benchmark of 300 annotated skill pairs with stratified difficulty, for evaluating clone detection on multi-modal skill documents. 

## 2 Background and Motivation 2.1 Agent Skills 

An agent skill is a structured document that extends an LLM agent’s capabilities at runtime. The dominant format, SKILL.md [20 , 38 ], consists of three primary in-file content channels that together capture the predominant content modalities of the skill file itself. The YAML frontmatter specifies metadata: skill name, description, category tags, compatibility constraints, and configuration parame-ters. The natural language body provides behavioral instructions about when to activate the skill, how to interact with the user, and what constraints to observe. Embedded code blocks implement concrete functionality such as shell commands, API calls, or file manipulation routines. A single skill may contain code blocks in multiple languages (e.g., Python, Bash, JavaScript), interleaved with NL prose. Chen et al. [ 38 ] report that this three-part structure of YAML frontmatter, NL body, and code fences is used by over 95% of skills across major registries, making it the de facto standard. Li et al. [ 20 ] confirm that 78% of skills contain at least one code block, while virtually all contain NL instructions, establishing that our three-channel decomposition covers the predominant content modalities of SKILL.md files. We therefore treat YAML and code as optional channels that may be absent in individual skills, but still part of the dominant format at ecosystem scale. This decom-position does not attempt to model auxiliary repository context beyond the focal SKILL.md artifact, such as external assets, linked files, or project history. Skills are distributed through registries (e.g., SkillsMP) and GitHub repositories, with no package manager, versioning system, or dependency declaration mechanism. 

Ecosystem Scale. In our SkillsMP registry snapshot, we observe 196,134 listed skills spanning 15 major categories including code generation, testing, DevOps, security analysis, and documentation.

This copy-and-modify workflow creates pervasive undeclared cloning that is invisible to the ecosystem. 

## 2.2 Clone Detection: From Code to Skills 

Code clone detection classifies clone relationships into four types [ 30 ]. Table 1 presents these types alongside our adapted definitions for multi-modal agent skills, which must consider similarity across three content channels (YAML, NL, Code) rather than a single source language. A key distinction is that skill clone types are determined by the joint similarity profile across channels. Two skills may exhibit Type-2 similarity in code (variable renaming) but Type-4 in NL (com-pletely rewritten instructions), requiring a multi-modal judgment that no single-channel detector can provide. Section 3.7 operational-izes these definitions with per-channel similarity thresholds. Our taxonomy captures structural clone evidence , not proven deriva-tion: Type-4 is the most uncertainty-prone category and excludes pairs that are merely topically related or belong to the same skill category. Existing detectors operate on single-modality source code. Text-based approaches such as NiCad [ 29 ] normalize source lines, achiev-ing high precision for Type-1/2 but struggling with semantic clones. Token-based approaches such as SourcererCC [ 31 ] and CCFinder [ 13 ]achieve scalability through index-based filtering but remain limited to surface-level similarity. Tree-based [ 1 , 12 ] and graph-based [ 16 ]approaches compare AST or PDG representations, requiring parseable single-language input. Recent ML-based approaches [ 7, 35 ] lever-age transformer architectures to detect semantic clones on bench-marks like BigCloneBench [ 34 ]. All these approaches assume single-language, single-modality input, an assumption that does not hold for agent skills, which interleave three structurally different content types within a single document.

Figure 1 shows a real clone pair discovered by SkillClone . The orig-inal skill ( 𝑠 𝑖 ), a 12.6K-character IDOR vulnerability testing method-ology by zebbern , contains 26 code blocks (51% code by bytes). A second author ( Dokhacgiakhoa ) adapted this skill into a modular format: the code blocks were extracted to 24 sub-skill files and the NL body was replaced with links. Yet the author: zebbern 

attribution and the trigger description were preserved verbatim— confirming derivation despite the restructuring. This pair exhibits cross-channel asymmetry : YAML similarity is high ( ∼0.95), NL is moderate ( ∼0.60), and code is zero. A code-only detector like SourcererCC [ 31 ] sees no code overlap and misses the clone entirely. A flat-text detector conflates the similarity with same-category vocabulary overlap. Only a multi-modal approach that reasons about per-channel evidence—and treats the absence 

of a channel as a signal rather than noise—can reliably detect this type of adaptation. The security implication is direct: any flaw in 

𝑠 𝑖 ’s exploitation methodology propagates through 𝑠 𝑗 ’s sub-skills, invisible to code-level scanners. 

## 3 Approach 

This section presents SkillClone , our approach for detecting multi-modal clone relationships among agent skills. We first formalize the problem (Section 3.1), give an overview of the approach (Section 3.2), then detail multi-modal decomposition (Section 3.3), modality-specific encoding (Section 3.4), cross-modal fusion (Section 3.5), and clone type classification (Section 3.7). 

## 3.1 Problem Definition 

Input. A corpus of 𝑁 agent skill documents S = {𝑠 1, . . . , 𝑠 𝑁 }, where each skill 𝑠 𝑖 is a structured document containing YAML metadata 

𝑦 𝑖 , natural language instructions 𝑛 𝑖 , and embedded code blocks 

𝑐 𝑖 = {𝑏 1, . . . , 𝑏 𝑘 }.

Output. A set of clone pairs C ⊆ S×S where each pair (𝑠 𝑖 , 𝑠 𝑗 , 𝜏, 𝜎 ) ∈ C is annotated with clone type 𝜏 ∈ { Type-1 , Type-2 , Type-3 , Type-4 }

and confidence 𝜎 ∈ [ 0, 1].

Goal. Maximize detection F1 for true clone relationships across all four clone types, with particular emphasis on Type-3 and Type-4 semantic clones that syntactic detectors miss. We adapt the classical clone type taxonomy [ 30 ] to multi-modal skills: Type-1 denotes identical content across all channels after whitespace normalization; Type-2 denotes cosmetic changes such as variable renaming in code or minor metadata edits; Type-3 denotes structural modifications including added or removed instructions; and Type-4 denotes skills with the same functional intent but sub-stantially different implementation.

## 3.3 Multi-Modal Decomposition 

Each skill 𝑠 𝑖 is decomposed into a quadruple 𝑠 𝑖 = (𝑦 𝑖 , 𝑛 𝑖 , 𝑐 𝑖 , f𝑖 ): the YAML metadata 𝑦 𝑖 extracted from the frontmatter (fields: name ,

description , category , tags ); the natural language instructions 

𝑛 𝑖 comprising the behavioral specification after the frontmatter, with code fences removed; the code component 𝑐 𝑖 = {𝑏 1, . . . , 𝑏 𝑘 }

collecting all fenced code blocks with their language annotations; and a structural feature vector f𝑖 ∈ R8 capturing document-level statistics. The decomposition uses regex-based parsing to identify YAML frontmatter (delimited by —), code fences (delimited by “‘ ), and the remaining NL body. We filter boilerplate entries by removing skills with fewer than five meaningful tokens after stopword removal. The structural feature vector f𝑖 captures eight document-level properties: total word count, number of code blocks, code-to-text ratio, average code block length, number of unique programming languages, presence of YAML frontmatter, number of YAML fields, and description length. These features capture the shape of a skill document, because two skills with similar structure (e.g., same number of code blocks, similar code fraction) are more likely to be clones than skills with different structural profiles. Each feature is min-max normalized across the corpus. 

## 3.4 Modality-Specific Encoding 

We encode each channel using latent semantic analysis (LSA) [ 5]: channel-specific TF-IDF vectorization [ 25 ] followed by truncated SVD projection to 𝑑 = 256 dimensions, producing dense embed-dings that capture semantic similarity within each modality. This design follows established practices in both NL document similar-ity [ 5, 25 ] and code clone detection [ 30 , 31 ], adapting them to the multi-modal skill setting. For the NL channel, TF-IDF+LSA captures topical similarity in a manner analogous to sentence embedding approaches [ 28 ], while avoiding external API dependencies. For the code channel, our language-aware tokenization follows the token-based tradition of CCFinder [ 13 ] and SourcererCC [ 31 ], ex-tended with cross-language normalization for the multi-language code blocks found in skills. This approach provides three advantages over neural encoders for our setting: it requires no API calls or GPU resources, enabling full reproducibility; it naturally handles the heterogeneous vocabu-lary across channels (YAML key-value pairs, free-form text, multi-language code); and it scales linearly with corpus size. Neural al-ternatives such as CodeBERT [ 7 ], GraphCodeBERT [ 9], and UniX-coder [ 8] achieve strong results on single-language code clone benchmarks, but assume homogeneous input with a single pro-gramming language per document, which does not hold for skills interleaving Python, Bash, and YAML within the same file. 

YAML Channel. We serialize the YAML metadata by concatenating key-value pairs and project through TF-IDF+SVD to produce e𝑦 𝑖 ∈

R𝑑 , capturing taxonomic relatedness. 

NL Channel. The natural language body is tokenized with stan-dard preprocessing (lowercasing, stopword removal) and encoded through TF-IDF+SVD, producing e𝑛 𝑖 ∈ R𝑑 that captures semantic similarity between behavioral specifications in the latent space. 

Code Channel. Code blocks require specialized tokenization: (1) camelCase/snake_case splitting, (2) operator token extraction pre-serving semantically meaningful operators ( |> , => ), and (3) lan-guage tag prefixing ( __lang_python__ ) enabling cross-language discrimination. The resulting tokens are encoded through TF-IDF+SVD to produce e𝑐 𝑖 ∈ R𝑑 .

## 3.7 Clone Type Classification 

Once the logistic fusion identifies a clone ( 𝑃 (clone ) ≥ 𝜃 ), the sim-ilarity profile s = [𝑠 𝑦 , 𝑠 𝑛 , 𝑠 𝑐 ] directly determines the clone type .Rather than partitioning the profile space with ad hoc score ranges, we define three Boolean predicates that each encode a structural property from the classical clone taxonomy (Table 1), then compose them into a deterministic decision tree. 

Predicates. Let P denote the set of channels present in both skills and sP the corresponding sub-vector. 

• Near-Identical (s): min (sP ) ≥ 𝜏 1. All present channels are near-verbatim—the defining property of Type-1 (“exact except layout/comments”). 

• Uniform (s): max (sP ) − min (sP ) ≤ 𝛿 . Channel similarities agree within tolerance 𝛿 —structure is preserved, consistent with Type-2 (“cosmetic relabeling”). 

• NL-Code-Split (s): 𝑠 𝑛 ≥ 𝜏 2 ∧ (𝑠 𝑐 < 𝜏 2 ∨ 𝑃 𝑐 =0). The intent channel (NL) shows similarity but the implementation channel (code) diverges or is absent—the defining signature of Type-4 (“same semantics, different syntax”). 

Decision tree. The three predicates compose into a four-leaf tree with no overlapping regions: (1) If Near-Identical (s): Type-1 .(2) Else if Uniform (s): Type-2 .(3) Else if NL-Code-Split (s): Type-4 .(4) Else: Type-3 (asymmetric profile with partial channel edits). The ordering is deliberate: Type-1 is a strict subset of Type-2 (all near-identical profiles are also uniform), so it must be tested first. Type-4 is tested before the Type-3 fallback because the NL-Code split is a specific pattern of asymmetry that warrants its own cate-gory. 

Parameters. The tree requires only three values: 𝜏 1 (identity thresh-old), 𝛿 (uniformity tolerance), and 𝜏 2 (semantic threshold), tuned on the validation split. In our evaluation, 𝜏 1 = 0.90 , 𝛿 = 0.20 , 𝜏 2 = 0.50 .

Scalability. At ecosystem scale ( 𝑁 = 20K, ∼200M candidate pairs), we exploit the dense LSA embeddings for efficient batch cosine similarity via matrix multiplication. An NL-channel pre-filter at 

𝜃 cand = 0.10 reduces candidates by over 90%, enabling full multi-modal scoring in under 30 seconds on a single CPU.

We crawled the skillsmp.com registry, retrieving SKILL.md content for 155,547 of 196,134 listed skills (79.3% success rate; failures due to deleted or private repositories). ## 4.2 Benchmark Construction 

No benchmark exists for clone detection on multi-modal agent skills. We construct SkillClone-Bench , a balanced benchmark of 300 ground-truth pairs (150 positive, 150 negative) with stratified difficulty across both classes. 

Design principles. We prioritize quality over quantity: each pair is included for a specific evaluation purpose, and both classes are stratified by difficulty so that aggregate metrics are not dominated by trivial cases. Positives span four clone types at increasing diffi-culty; negatives range from obvious non-matches to deliberately confusing same-category pairs. We avoid manual labeling entirely— all labels derive from verifiable provenance (fork trees) or controlled generation (mutations). 

Positive pairs (150). We draw equally from two complementary sources, each contributing 75 pairs: 

Fork-mined pairs (75) exploit GitHub fork relationships as veri-fiable provenance. When a developer forks a skill repository and modifies its contents, the fork tree confirms derivation. We match skills across forks by file path and name similarity, then stratify by modification level into Type-1 (verbatim), Type-2 (renamed), Type-3 (structurally modified), and Type-4 (semantically equivalent but rewritten). 

Synthetic mutations (75) ensure coverage of controlled difficulty levels via seven operators applied to randomly sampled seed skills: M1 (identifier renaming, Type-2), M2 (NL paraphrasing with syn-onym replacement and 30% content removal, Type-3), M3 (code stripping that retains only NL section headers, Type-4), M4 (content extension, Type-3), M5 (content subsetting, Type-3), M6 (cross-skill combination, Type-3), and M7 (full semantic rewrite via LLM para-phrase, Type-4). M3 and M7 produce the hardest pairs: both preserve functional intent while eliminating surface-level token overlap. 

Negative pairs (150). To avoid easy-negative inflation, we stratify negatives by difficulty: 

Easy negatives (50) pair skills from different categories (e.g., De-vOps vs. documentation), providing clear non-clones where low similarity is expected. 

Hard negatives (100) pair skills from the same category whose multi-modal similarity falls in [0.15 , 0.45 ]. These pairs share topical vocabulary (e.g., two Kubernetes skills both referencing kubectl 

and namespace ) without being clones. Hard negatives constitute two-thirds of the negative set, deliberately stressing the detector’s ability to distinguish vocabulary overlap from genuine cloning. ## 4.3 Experimental Setup 

Baselines. We compare against four modality-matched approaches: 

• TF-IDF (flat text): Standard TF-IDF cosine similarity on the full 

SKILL.md content as a single string, ignoring modality structure. 

• MinHash: Locality-sensitive hashing on character 𝑛 -grams ( 𝑛 =

5) of the full skill text, representing a scalable token-based base-line. 

• NL-Only: SkillClone ’s NL channel embedding applied in isola-tion, representing the strongest single-channel baseline. 

• Code-Only: SkillClone ’s code channel embedding applied in isolation. 

Baseline Scope. Our goal is to compare SkillClone against base-lines that preserve the task definition, namely clone detection over full multi-modal skill documents rather than over extracted code fragments alone. We therefore do not present this evaluation as an exhaustive state-of-the-art bakeoff against every recent code-clone model. Instead, we compare against strong text and token baselines plus single-channel ablations that operate on the same underlying artifacts. We do not include neural code clone detectors such as SAGA [ 37 ], DeepSim [ 40 ], or MAGNET [ 39 ] because they require single-language parseable source code as input (ASTs, PDGs, or token sequences). Agent skills are not parseable programs, because they interleave YAML frontmatter, natural language prose, and code blocks in multiple languages within a single document. Applying these tools would require discarding the NL and YAML channels (78% of skill content by volume), fundamentally changing the detec-tion task. The NL-Only and Code-Only ablations in Table 3 quantify the cost of this single-channel restriction: both achieve lower F1 than multi-modal fusion. 

Metrics. We report precision, recall, and F1 score for binary clone detection. For each approach, we find the optimal threshold via grid search on SkillClone-Bench to ensure a fair comparison at each method’s best operating point. For clone type analysis, we report per-type recall.

## 4.7 RQ4: Security Propagation 

Rather than assembling synthetic proof-of-concepts, we scan the 20K corpus for real security-relevant skills using regex patterns for six categories of dangerous code: SQL injection payloads, XSS test vectors, reverse shell commands, command injection via unsafe subprocess calls, hardcoded secrets, and unsafe eval on user input. Pattern propagation. Of the 1,100 clones, 316 (29%) inherited the security-relevant code pattern itself —not just similar content, but the actual dangerous code. The remaining 71% are structural or semantic clones where the security pattern was modified or removed during adaptation. ## 5 Discussion 5.1 Clone Detection vs. Reuse Inference 

Clone detection establishes structural similarity, not derivation. 

Reuse inference asks whether 𝑠 𝑗 was causally derived from 𝑠 𝑖 and therefore requires additional provenance signals such as fork rela-tionships, commit timestamps, or explicit attribution. In some cases, clone detection directly enables reuse confirma-tion. The LabArchives example (Figure 1) is unambiguous: 99.7% character similarity on 260 lines of niche scientific content, com-bined with the added skill-author: K-Dense Inc. metadata, constitutes a smoking gun. Our fork-mined benchmark pairs simi-larly provide ground-truth reuse via GitHub provenance. For the broader ecosystem analysis (RQ3), however, we report clone preva-lence , not reuse prevalence: the 258K detected pairs include both intentional copies and cases of convergent development where different authors independently produced similar skills. For security applications (RQ4), the clone-vs-reuse distinction is immaterial. Whether two skills share a vulnerability through copying or convergent development, the blast radius is identical. Clone detection is sufficient to identify the set of skills that share a malicious pattern, regardless of how they came to share it. 

## 5.2 Implications 

Our findings carry implications for multiple stakeholders. For skill registry operators , the 75% clone involvement rate and 41% superseded-skill rate suggest that deduplication and provenance tracking should be first-class ecosystem features. For security teams , 90% of security-relevant seed skills have at least one clone, yielding 1,100 down-stream clones (777 cross-author), so per-skill vulnerability scanning is insufficient and clone-based propagation analysis is necessary to identify the full blast radius. 

Precision and Recall Tradeoff. On SkillClone-Bench , flat-text TF-IDF is a strong baseline, achieving 0.897 precision, 0.867 recall, and 0.881 F1. However, SkillClone improves on all three metrics, reaching 0.952 precision, 0.927 recall, and 0.939 F1 by combining flat similarity with per-channel evidence. This difference matters more at ecosystem scale than on SkillClone-Bench alone: in a 20K-skill corpus, small precision gaps translate into many spurious clone edges. Additionally, flat-text TF-IDF cannot distinguish clone 

types , while SkillClone preserves the per-channel signal needed to separate exact copies from harder semantic clones.

## 5.3 Limitations 

Our current study prioritizes task-faithful multi-modal analysis over exhaustive model coverage. The evaluation therefore focuses on modality-matched baselines rather than every recent neural code-clone detector, many of which assume single-language source code input. In addition, although our crawl retrieved 155K skill files, the clone-graph analysis reported here uses 20K skills and a 300-pair benchmark. Larger annotated datasets and full-corpus runs would improve confidence in the reported clone-type distribution and propagation estimates. Our three-channel decomposition is also intentionally scoped to the primary in-file modalities of SKILL.md 

artifacts, excluding auxiliary repository context such as linked files, screenshots, and commit history. 

## 5.4 Threats to Validity 

Internal validity. The benchmark relies partly on synthetic mu-tations; we mitigate this with fork-mined pairs and manual labels. Fusion weights are optimized on SkillClone-Bench and may not generalize; the ablation shows robustness to component removal. 

External validity. Our 20K-skill sample is substantial but may not represent the full 196K ecosystem. The observed 75% clone-involvement rate and 3.5 × inflation ratio may shift under a larger or more complete crawl. 

Construct validity. The clone type taxonomy adapted from source code may not capture all multi-modal similarity patterns. Threshold-based type boundaries are heuristic; a learned classifier may im-prove accuracy. The inter-annotator agreement ( 𝜅 = 0.81 ) is sub-stantial but not perfect. Our ecosystem analysis reports clone preva-lence, not confirmed reuse; the Type-4 dominance may include con-vergent development cases that are structurally similar but causally independent. 

## 6 Related Work 6.1 Code Clone Detection 

Code clone detection has been studied extensively over two decades [ 30 ]. Representative classical approaches include token-based systems such as CCFinder and CP-Miner [ 13 , 18 ], tree-based systems such as NiCad and Deckard [ 12 , 29 ], and scalable index-based systems such as SourcererCC [ 31 ]. BigCloneBench [ 34 ] standardized evalu-ation, while newer neural approaches such as DeepSim, CCLearner, SAGA, and MAGNET target harder semantic clones [ 17 , 37 , 39 , 40 ]. However, these methods assume source code as the primary arti-fact. SkillClone instead targets multi-modal skill documents that interleave natural language, YAML metadata, and embedded code, extending clone detection to a new class of software artifacts. 

## 6.2 Software Ecosystem Analysis 

Large-scale ecosystem studies have characterized dependency struc-ture and systemic risk in registries such as npm and PyPI [ 3, 4, 14 ,36 , 42 ]. Recent work has also begun to characterize the architec-ture and growth of agent-skill ecosystems [ 20 , 38 ]. However, prior studies do not model inter-skill clone relationships. This omission is particularly important in a flat ecosystem with no dependency mechanism, where reuse is implicit rather than declared. Skill-Clone addresses this gap by inferring clone relationships directly from content similarity and constructing the first clone graph for agent skills. 

## 6.3 Supply Chain Security 

Software supply chain research has shown that vulnerabilities and malicious packages propagate through ecosystem structure rather than through isolated artifacts alone [ 6, 21 , 26 ]. In agent ecosystems, recent tools focus on per-skill analysis for vulnerability detection, formal verification, malicious tool identification, prompt injection, and MCP-server risks [ 2, 10 , 11 , 22 –24 , 32 , 33 , 41 ]. The January 2026 ClawHavoc campaign [ 15 ] underscores the practical importance of these risks. SkillClone complements this line of work by adding clone-graph analysis: instead of inspecting each skill in isolation, it traces reuse links to estimate the blast radius of malicious or vulnerable patterns. 

## 7 Conclusion 

We presented SkillClone , the first multi-modal clone detection approach for agent skills, fusing YAML, NL, and code similarity with cross-modal alignment. On SkillClone-Bench (300 pairs), 

SkillClone achieves F1 of 0.939, outperforming TF-IDF (0.881) and achieving 4.2 × higher Type-4 recall than MinHash. Applied to 20K skills, SkillClone reveals 258K clone pairs involving 75% of all skills, with 40% cross-author reuse and a 3.5 × ecosystem inflation ratio after deduplication. Security propagation analysis identifies 141 real security-relevant seed skills whose patterns reach 1,100 clones across 119 affected authors. We distinguish clone detection from reuse inference, showing clone analysis is both independently valuable for security and enables downstream reuse discovery when combined with provenance signals.
