## Abstract 

As LLM agent ecosystems grow, the number of available skills (tools, plug-ins) has reached tens of thousands, making it infeasible to inject all skills into an agent’s context. This creates a need for skill routing —retrieving the most relevant skills from a large pool given a user task. The problem is compounded by pervasive functional overlap in community skill repos-itories, where many skills share similar names and purposes yet differ in implementation details. Despite its practical importance, skill routing remains under-explored. Current agent architectures adopt a progressive disclosure design—exposing only skill names and descriptions to the agent while keeping the full implementation body hidden—implicitly treating metadata as sufficient for selection. We challenge this assumption through a systematic empirical study on a benchmark of ∼80K skills and 75 expert-verified queries. Our key finding is that the skill body (full implementation text) is the decisive signal: removing it causes 29–44 percentage point degra-dation across all retrieval methods, and cross-encoder attention analysis reveals 91.7% of attention concentrating on the body field. Motivated by this finding, we propose S KILL ROUTER , a two-stage retrieve-and-rerank pipeline totaling only 1.2B parameters (0.6B encoder + 0.6B reranker). S KILL -ROUTER achieves 74.0% top-1 routing accuracy and delivers the strongest average result among the compact and zero-shot baselines we evaluate, while remaining deployable on consumer hardware. 

## 1 Introduction 

LLM-based coding and personal assistants—such as Claude Code (Anthropic, 2025), Codex CLI (OpenAI, 2025), and open-source alternatives—have become an integral part of modern software development workflows. A key factor in their effectiveness is the skill (also referred to as tool or plugin) ecosystem: community-contributed skill packages that extend an agent’s capabilities with specialized functionality, from database migration to code linting to API integration. As these ecosystems mature, the number of available skills has grown rapidly, reaching tens of thousands in popular registries. However, this growth brings a critical challenge: community-contributed skill repositories exhibit pervasive functional overlap— many skills share similar names and purposes while differing only in implementation details, creating a highly homogeneous pool that is difficult to navigate. Injecting all available skills into an agent’s context window is infeasible. Context length constraints, inference cost, and the sheer diversity of skill quality and relevance make it impractical to present every skill to the agent for every task. This creates a fundamental need for skill routing : given a user’s task description, automatically selecting the most relevant skill(s) from a large pool. ## 2 Problem Definition and Benchmark 

2.1 Problem Formulation 

We consider the skill routing problem: given a user task query q and a large skill pool 

S = {s1, . . . , sN }, select the skill(s) most relevant to accomplishing the task. Each skill si is a structured text unit comprising three fields: • Name : a short identifier (e.g., pdf-merger ). • Description : a one-sentence summary of the skill’s purpose. • Body : the full implementation or detailed specification, typically hundreds to thousands of tokens. The pool size N can range from tens of thousands to hundreds of thousands in real-world agent ecosystems, making brute-force evaluation infeasible. The goal is to retrieve a ranked list of candidate skills, with the ground-truth skill(s) Gq ⊆ S ranked as highly as possible. 

Hidden-body asymmetry. A distinctive feature of the skill routing setting is what we term 

hidden-body asymmetry : the agent that ultimately uses a selected skill typically operates under context-window constraints and can only see the skill’s name and description. However, the routing component (retriever and reranker) can access the full skill text, including the body. This creates an information asymmetry between the selector (which has full access) and the 

consumer (which sees only metadata). As we demonstrate in Section 3, this asymmetry is not merely a design convenience but a necessity—body access is critical for accurate selection. 

Evaluation metrics. We evaluate with standard retrieval metrics: Hit@1 (top-1 routing accuracy, our primary metric), MRR@10 (mean reciprocal rank in top-10), nDCG@10 

(normalized discounted cumulative gain), Recall@ K (K ∈ { 10, 20, 50 }, proportion of ground-truth skills retrieved), and FC@10 (fraction of queries where all ground-truth skills appear in top-10). For multi-skill queries, Hit@1 counts success when any required skill is ranked first. We therefore report Recall@ K and FC@10 as complementary coverage metrics. Throughout the paper, our headline results should be read as routing-quality results rather than direct end-to-end task-completion measurements. 

2.2 Benchmark Construction 

Evaluating skill routing requires a benchmark with two properties: (1) high task–skill affinity —each query must have clearly defined ground-truth skills that are necessary and sufficient for the task, and (2) realistic pool scale —the skill pool must be large enough that retrieval is non-trivial. 

Task–skill pairs from SkillBench. We build our evaluation queries on top of SkillBench (Huang et al., 2025), a benchmark designed for evaluating LLM agents on complex tasks that require specific skills. SkillBench provides expert-curated task–skill mappings where each task is paired with the specific skills needed to complete it. This strong task–skill affinity—verified by domain experts—makes SkillBench an ideal source for high-quality ground-truth annotations. We extract 87 task queries and their ground-truth skill mappings, then apply a consistency filter to remove queries with ambiguous or annotator-disagreed skill assignments, yielding 75 core queries . These comprise 24 single-skill queries and 

51 multi-skill queries. Further details on single- vs. multi-skill evaluation and metric computation are provided in Appendix A. 

Skill pool construction. To create a realistic retrieval challenge, we construct a skill pool of ∼80K skills by combining the SkillBench skills with a large collection of community-contributed skills from open-source repositories. ## 3 What Signals Drive Skill Selection? 

Before presenting our pipeline, we investigate a fundamental question: which components of a skill’s text actually matter for accurate selection? Each skill consists of three fields—name, description, and body—and the prevailing assumption in current agent frameworks is that name and description provide sufficient information for skill matching. We challenge this assumption through controlled experiments and attention analysis, revealing that the skill body is the decisive signal for accurate selection. 

3.1 Name+Description vs. Full Skill Text 

We compare two input configurations across retrieval methods: nd (name + description only) and full (name + description + body). Body removal causes catastrophic failure. BM25 with nd input achieves exactly zero across all metrics: ground-truth skills’ names and descriptions share virtually no lexical overlap with user queries, making keyword matching completely ineffective. Adding the skill body restores BM25 to 34.7% Hit@1—a result that is itself modest but demonstrates that the dominant discriminative signal resides in the body. 

The effect is universal across model families. Dense encoders show the same pattern: Qwen3-Emb-0.6B drops from 58.7% (full) to 22.7% (nd), a 36.0pp collapse . Even the 8B model cannot compensate—Qwen3-Emb-8B nd (30.7%) is far below the 0.6B full model (58.7%), meaning a 13 × increase in parameters recovers less than one-quarter of the body’s contribution. 

Rerankers without body are actively harmful. Rerankers operating on nd input degrade performance below the encoder-only baseline. 3.2 Attention Analysis: Where Does the Model Look? 

To understand why the body matters, we analyze the attention patterns of the cross-encoder reranker (SR-Rank-0.6B). Since Qwen3 is a causal (decoder-only) model, we examine the attention from the final prediction token to all input tokens, aggregated across 28 layers ×

16 heads and 75 core queries. We partition the attention weights by field (name, description, body) and normalize to sum to 1. Figure 1 (right panel) summarizes the results: body receives 91.7% of total attention , name receives 7.3%, and description receives only 1.0%. 

Layer-wise pattern: body →name →body. The attention distribution evolves across layers. Early layers (0–6) focus almost exclusively on body content (97.3%), likely performing token-level content understanding. Middle layers (7–20) allocate increasing attention to name, peaking at 26.3% in layer 19—suggesting that these layers perform name-based semantic matching against the query. Late layers (21–27) return to body dominance (91.7%), using the body’s detailed information for the final relevance decision. Description attention remains negligible ( ≤1.4%) at all layers, indicating that the information it carries is entirely subsumed by the body. 

3.3 Implications 

These findings overturn the common assumption that skill name and description suffice for selection, and have direct consequences for system design: 1. Retrieval and reranking must access the full body. Any skill-routing system that operates only on name and description is fundamentally limited—not just less accurate, but catastrophically so (30–40pp degradation). 2. Hidden-body asymmetry is a real design constraint. In typical agent architectures, the agent itself cannot see skill bodies (they are too long for the context window). Our findings show that the retrieval/reranking layer must have body access, creating an inherent information asymmetry between the agent and its routing component. 3. These findings directly motivate our pipeline design. In Section 4, we design S KILL -ROUTER to use full skill text in both the encoder and reranker stages—a choice that the above evidence shows is not merely beneficial but necessary. 5Preprint. ## 4 SkillRouter: A First Practical Pipeline 

Motivated by the findings in Section 3—that skill body is the decisive signal and must be accessible to both retrieval and reranking—we design S KILL ROUTER , a compact two-stage retrieve-and-rerank pipeline. The pipeline totals only 1.2B parameters (0.6B encoder + 0.6B reranker), making it suitable for on-device deployment in personal agent products. 

4.1 Overview 

Given a user query q and a skill pool S = {s1, . . . , sN } where N ≈ 80K, S KILL ROUTER 

operates in two stages: 1. Bi-encoder retrieval. SR-Emb-0.6B encodes the query and all skills into a shared em-bedding space using full skill text (name + description + body). The top-K candidates (K=20) are retrieved via cosine similarity, reducing the candidate set from ∼80K to 20. 2. Cross-encoder reranking. SR-Rank-0.6B jointly processes each (query, candidate) pair through cross-attention over the full skill text, producing a refined relevance score. The 20 candidates are re-sorted to produce the final ranking. Both stages use full skill text as input—a design choice that Section 3 shows is not merely ben-eficial but necessary. Skills are pre-encoded offline; at inference time, only query encoding, ANN search, and reranking of 20 candidates are required. 

4.2 Bi-Encoder Retrieval Base model. We fine-tune Qwen3-Emb-0.6B (Qwen Team, 2025), a decoder-based embed-ding model that supports variable-length inputs and instruction-prefixed encoding. The 0.6B scale is chosen to enable on-device deployment while providing sufficient capacity for skill-routing semantics. 

Training data. We construct a training set of 37,979 (query, skill) pairs. Skills are sampled from a community skill repository comprising ∼80K open-source skills across 51 categories, with stratified sampling to ensure category diversity. Hard negative mining. Effective contrastive learning requires informative negatives that are challenging but not false positives. We employ a multi-source mining strategy that produces 10 negatives per query from four complementary sources: • Semantic negatives (4 per query): We pre-compute embeddings for all skills using the base Qwen3-Emb-0.6B model, retrieve the top-50 most similar skills by cosine similar-ity, and sample 4 non-positive skills from this set. These are the hardest negatives— semantically close but functionally distinct. • Lexical negatives (3 per query): BM25 scoring over skill text (name + description + body) on the same top-50 candidate set, capturing term-overlap confounders that semantic search may miss. • Taxonomy negatives (2 per query): randomly sampled from the same category as the positive skill but with a different name, providing same-domain distractors. • Random negatives (1 per query): uniformly sampled from a different category, serving as easy negatives for calibration. 

False negative filtering. Community skill repositories exhibit extensive functional overlap—many skills with different names provide near-identical capabilities. In con-trastive learning, these duplicates become false negatives that corrupt the training signal. We apply a three-layer filtering pipeline: 1. Name deduplication : removing negatives that share the same name as any ground-truth skill for the query. 2. Body overlap : removing negatives whose body text has trigram Jaccard similarity > 0.6 with a ground-truth skill’s body. 3. Embedding similarity : removing negatives with cosine similarity > 0.92 to a ground-truth skill’s embedding, catching semantic duplicates missed by lexical matching. In total, approximately 10% of mined negative pairs are removed. The impact of this filtering is validated in Section 5.4.1, where removing it degrades Hit@1 by 4.0pp.

## 5 Evaluation 

5.1 Experimental Setup Evaluation protocol. All experiments are conducted on our benchmark (Section 2.1), which comprises 75 expert-verified queries (24 single-skill, 51 multi-skill) across two difficulty tiers: 

Easy (78,361 candidate skills) and Hard (79,141 candidate skills). We report seven metrics: Hit@1 (our primary top-1 routing metric), MRR@10 (mean reciprocal rank), nDCG@10 (normalized discounted cumulative gain), Recall@10, Recall@20, Recall@50, and FC@10. For multi-skill queries, Hit@1 counts success when any required skill is ranked first, while Recall@10 and FC@10 provide complementary coverage views. Unless otherwise noted, results are averaged across Easy and Hard tiers. 

Input formats. Each skill s consists of three text fields: name , description , and body . We denote two input configurations: full (name + description + body) and nd (name + descrip-tion only, without body). As established in Section 3, including the full skill body is critical for accurate skill selection. All our fine-tuned models are therefore trained and evaluated with full inputs. We additionally evaluate baselines under both configurations to quantify the impact of skill body access. 

Encoder baselines. We compare against a diverse set of retrieval methods spanning four categories: • Sparse retrieval: BM25 with full skill text as the document. • Traditional encoders: E5-Large-v2 (Wang et al., 2024) (335M), GTE-Large-v1.5 (Li et al., 2024) (434M), and BGE-Large-v1.5 (Xiao et al., 2024) (335M). • Decoder-based encoders: Qwen3-Emb in two sizes—0.6B and 8B (Qwen Team, 2025)— and NV-Embed-v2 (Lee et al., 2024) (7B). • Proprietary models: OpenAI text-embedding-3-large (OpenAI, 2024) and Google gemini-embedding-001 (Google, 2025). All encoder baselines use full skill text as input and retrieve candidates via cosine similarity over the entire skill pool. 

Reranker baselines. For the reranking stage, we evaluate: • Cross-encoder rerankers: Qwen3-Reranker-0.6B and Qwen3-Reranker-8B (Qwen Team, 2025), both zero-shot, tested with both full and nd inputs. • LLM-as-judge rerankers: GPT-4o-mini and GPT-5.4-mini, prompted in listwise mode to select the most relevant skill from the full candidate list, tested with both full and nd 

inputs (see Appendix C for exact prompts). All reranking experiments use the encoder’s top-20 retrieved candidates as input unless otherwise specified. 

Our models. SKILL ROUTER consists of two fine-tuned components: • SR-Emb-0.6B : Qwen3-Emb-0.6B fine-tuned on 37,979 skill-routing queries with hard-negative mining and three-layer false-negative filtering (Section 4.2). • SR-Emb-8B : The same fine-tuning recipe applied to Qwen3-Emb-8B, demonstrating scalability. • SR-Rank-0.6B : Qwen3-Reranker-0.6B fine-tuned with listwise cross-entropy loss on top-20 candidates from SR-Emb-0.6B (Section 4.3). • SR-Rank-8B : The same reranker recipe applied to Qwen3-Reranker-8B, demonstrating scalability. The primary S KILL ROUTER pipeline (SR-Emb-0.6B → SR-Rank-0.6B) totals 1.2B parame-ters, designed for on-device deployment. When the encoder retrieves top-20 candidates for a query, the resulting list is often dominated by semantically similar skills that differ only in implementation details within their body text. Pointwise BCE processes each (query, candidate) pair independently via sigmoid, assigning absolute relevance scores without any awareness of other candidates in the list. Faced with a set of near-identical candidates, the sigmoid outputs converge to a narrow band, making the resulting ranking effectively random. Listwise CE, by contrast, applies softmax over the entire candidate list, forcing the model to directly compare candidates against each other and learn their relative ordering. This comparative mechanism is precisely what is needed when the reranker must pick the one correct skill from a pool of close alternatives—a scenario that arises naturally from the homogeneity of large-scale skill repositories.

5.5.2 Efficiency 

A key motivation for S KILL ROUTER is on-device deployment in personal agent products such as OpenClaw, where cloud-based or large-model inference is impractical. The full pipeline totals only 1.2B parameters (0.6B encoder + 0.6B reranker), compared to 8B for Qwen3-Emb-8B alone or 7B for NV-Embed-v2—much larger models against which S KILL -ROUTER compares favorably on average Hit@1. At inference time, skill embeddings are pre-computed offline and stored in a vector index. For each incoming query, the pipeline requires only: (1) encoding the query with the 0.6B encoder, (2) approximate nearest neighbor (ANN) search over pre-indexed skill embeddings, and (3) reranking the top-20 candidates with the 0.6B cross-encoder. Steps (1) and (3) involve a single forward pass each through a 0.6B model; step (2) is sub-linear in pool size via standard ANN libraries. This architecture is well-suited for CPU or edge-GPU deployment without requiring cloud API access. 

5.5.3 Qualitative Analysis 

We highlight two representative cases that illustrate the strengths of each pipeline stage. The full set of six case studies (four successes and two failures) is provided in Appendix G. 

Case 1: Encoder advantage (video-tutorial-indexer). The query asks an agent to extract chapter timestamps from a local tutorial video. The ground-truth skill is speech-to-text ,which provides Whisper-based audio transcription—the necessary first step for timestamp extraction. SR-Emb-0.6B, through task-specific fine-tuning, learns the indirect semantic mapping “video + timestamps → speech-to-text” and retrieves the correct skill at rank 1. This case demonstrates that fine-tuning on skill-routing data captures reasoning shortcuts that scale alone cannot provide .

## 6 Related Work 

Tool and skill selection for LLM agents. As LLM-based agents gain access to increasingly large tool and skill collections, selecting the right tool for a given task has emerged as a central challenge. Early systems such as Toolformer (Schick et al., 2023) and HuggingGPT (Shen et al., 2023) demonstrate the potential of LLMs to invoke external tools, but assume a small, fixed set of available tools. ToolLLM (Qin et al., 2023) scales this to 16K+ real-world APIs with a retrieval-augmented approach, while Gorilla (Patil et al., 2023) connects LLMs to API documentation through retrieval. AnyTool (Du et al., 2024) and CRAFT (Yuan et al., 2024a) further explore hierarchical tool discovery and context-aware tool retrieval. EASYTOOL (Yuan et al., 2024b) improves tool selection by generating concise, structured tool instructions from raw documentation, highlighting the importance of tool text quality. ToolkenGPT (Hao et al., 2023) takes a different approach by learning special token embeddings (“toolkens”) for each tool, enabling tool selection through next-token prediction. ToolRerank (Zheng et al., 2024) applies adaptive truncation and hierarchy-aware diversification as post-processing on retrieval results, improving tool retrieval quality on ToolBench. 

Neural information retrieval. Our two-stage pipeline builds on the retrieve-and-rerank paradigm widely adopted in neural IR. Dense bi-encoder retrieval, pioneered by DPR (Karpukhin et al., 2020) and extended by Contriever (Izacard et al., 2022), ANCE (Xiong et al., 2021), E5 (Wang et al., 2024), GTE (Li et al., 2024), and BGE (Xiao et al., 2024), encodes queries and documents into a shared embedding space for efficient similarity search. Cross-encoder reranking applies deeper attention-based scoring to a shortlisted candidate set, as demonstrated by BERT passage reranking (Nogueira & Cho, 2020) and RankGPT (Sun et al., 2023). While we adopt this established architecture, skill routing presents unique challenges not found in standard document retrieval: skills are structured multi-field objects (name, description, body) with high inter-skill homogeneity, and the decisive signal resides in the body rather than in metadata—a finding that has no direct parallel in traditional IR settings. 

Benchmarks for agent skills. Several benchmarks evaluate different aspects of tool-augmented agents. ToolBench (Qin et al., 2023) and API-Bank (Li et al., 2023) provide large-scale API collections for evaluating tool selection and execution. MetaTool (Huang et al., 2024) studies tool selection preferences, and T-Eval (Chen et al., 2024) evaluates tool utilization effectiveness. Our evaluation benchmark builds on SkillBench (Huang et al., 2025), which provides expert-curated task–skill mappings with strong affinity guarantees. Unlike prior benchmarks that focus on whether agents can use tools correctly, our bench-mark specifically targets the retrieval problem: can we find the right skill from a pool of 

∼80K candidates? 

## 7 Conclusion 

We presented a systematic study of skill routing for LLM agents, addressing three research questions. For RQ1 , we established that skill routing in large-scale pools ( ∼80K skills) is a genuine and challenging retrieval problem—even the strongest base encoders achieve only 64.0% Hit@1, leaving substantial room for improvement. For RQ2 , we revealed that the skill body is the decisive signal for accurate selection: removing body text causes 29–44pp degradation across all methods, and cross-encoder attention concentrates 91.7% on the body field. This overturns the common assumption that skill name and description suffice. For 

RQ3 , we demonstrated that S KILL ROUTER , a compact retrieve-and-rerank pipeline totaling only 1.2B parameters, achieves 74.0% Hit@1 and compares favorably with larger zero-shot 16 Preprint. Under review. alternatives while remaining suitable for on-device deployment. The same fine-tuning recipe scales to 8B models, reaching 76.0% Hit@1, confirming the generality of the approach. Beyond the pipeline itself, our ablation studies uncovered two practical insights for training skill-routing models: (1) false negative filtering is essential due to the high functional overlap among community-contributed skills, and (2) listwise ranking loss dramatically outperforms pointwise loss (+30.7pp), reflecting the need for comparative scoring in homogeneous candidate pools. 

Limitations. Our evaluation benchmark, while built on high-quality expert-curated task– skill mappings, comprises 75 queries—a scale that limits statistical power. Additionally, our findings are validated on a single benchmark derived from SkillBench; generalization to other skill ecosystems and domains remains to be verified. Finally, because our benchmark is majority multi-skill, coverage-oriented metrics provide useful complementary context to Hit@1; 22% of queries are missed by both encoder and reranker, suggesting that some skill-routing scenarios require reasoning capabilities beyond current retrieval methods. 

Future work. Several directions emerge from our findings. First, constructing larger-scale skill routing benchmarks with broader domain coverage would strengthen the generaliz-ability of our conclusions. Second, integrating LLM-based reasoning with retrieval could address the multi-hop inference cases that our pipeline currently cannot solve. Third, end-to-end evaluation within real agent frameworks would validate whether improved skill retrieval translates to better downstream task performance—our appendix analysis combining S KILL ROUTER with SkillsBench (SkillsBench Team, 2026) results (Appendix H) should be read as a calibrated scenario analysis rather than direct end-to-end evidence. We plan to release our models and evaluation toolkit to facilitate further research in this direction.

## A Evaluation Details 

Single- vs. multi-skill queries. Of the 75 core queries, 24 are single-skill queries (exactly one ground-truth skill) and 51 are multi-skill queries (two or more ground-truth skills required to complete the task). 

Metric computation for multi-skill queries. For queries with multiple ground-truth skills 

Gq = {g1, . . . , gm}, we compute Hit@1 as the indicator of whether any ground-truth skill appears at rank 1. MRR@10 uses the highest-ranked ground-truth skill’s reciprocal rank. Recall@ K measures the fraction of ground-truth skills that appear in the top-K results: R@ K = |G q ∩ top-K|/|G q|. FC@10 (Full Coverage at 10) is the strictest metric: it equals 1 only when all ground-truth skills for a query appear in the top 10.

## H Downstream Scenario Analysis for Agent Task Completion 

Our evaluation so far measures retrieval quality (Hit@1, Recall@ K) in isolation. A natural question is: how does retrieval accuracy translate to downstream agent task completion when skills must be retrieved from a large pool rather than provided as ground truth? 

To answer this, we conduct a calibrated scenario analysis by combining our retrieval results with the independently published SkillsBench benchmark (SkillsBench Team, 2026), which evaluates frontier coding agents on 84 real-world tasks under two conditions: vanilla (no skills) and with skills (gold-standard skills provided directly to the agent). By bridging these two data sources, we can estimate the agent pass rate under a realistic retrieve-then-use deployment scenario, where skills must first be selected from an 80K-scale pool. This appendix should therefore be read as an estimate of downstream implications rather than as direct end-to-end evidence. 

Model. We adopt a linear mixture model, but use coverage proxies rather than a single retrieval metric. Let C denote an effective coverage coefficient capturing how much of the needed skill support is surfaced by retrieval, Pwith the agent pass rate with gold skills, and 

Pwithout the pass rate without skills. The estimated pass rate under retrieval is: 

Pretrieve (C) = C · Pwith + ( 1 − C) · Pwithout (4) The intuition is straightforward: larger values of C move the agent closer to the gold-skill regime, while smaller values of C leave it closer to vanilla performance. 

Why one metric is not enough. SkillBench includes many multi-skill tasks, so treating Hit@1 as the sole proxy is too optimistic. We therefore report three deployment semantics: 1. Hit@1 as a top-1 / upper-bound proxy : appropriate when the agent only consumes one retrieved skill, or when retrieving any one key skill is enough to unlock the task. 2. R@10 as a top-10 partial-coverage proxy : appropriate when the agent can inspect a shortlist and benefit in proportion to how much of the required skill set appears in the top 10. 3. FC@10 as a top-10 full-coverage proxy : appropriate for stricter multi-skill settings where 

all required skills must appear in the top 10. This distinction matters empirically. On our benchmark, the primary 1.2B pipeline (SR-Emb-0.6B × SR-Rank-0.6B) achieves Hit@1 =72.9%, R@10 =FC@10 =87.5% on single-skill queries, but on multi-skill queries the gap widens to Hit@1 =74.5%, R@10 =62.4%, and FC@10 =35.3%. Thus, Hit@1 alone substantially overstates task readiness for multi-skill settings.

The realistic downstream estimate is now a range, not a point. Under the linear model, the fraction of total skill benefit captured equals the chosen coverage proxy C itself. For our primary 1.2B pipeline, this means: • Upper bound (Hit@1): captures 74.0% of the gold-skill uplift, raising average pass rate from 24.3% to 36.3%. • Shortlist estimate (R@10): captures 70.4% of the uplift, yielding 35.8% average pass rate. • Strict lower bound (FC@10): captures 52.0% of the uplift, yielding 32.8% average pass rate. Hence the practically relevant regime for multi-skill tasks is better summarized as a 32.8%– 35.8% expected pass range, with 36.3% serving as an optimistic top-1 upper bound rather than the main point estimate. 

Skills-as-scale-substitute holds under retrieval. SkillsBench’s striking finding is that a smaller model with skills can outperform a larger model without skills—e.g., Claude Haiku 4.5 with gold skills (27.7%) beats Opus 4.5 without skills (22.0%). Under retrieval, this effect becomes conditional on coverage assumptions . For the primary 1.2B pipeline, Haiku 4.5 rises from 11.0% to 19.7%–23.4%; it still exceeds Opus 4.5 without skills (22.0%) under the R@10 / Hit@1 assumptions, but not under the strict FC@10 bound. Sonnet 4.5 rises from 17.3% to 24.8%–28.0%, comfortably exceeding Opus 4.5 without skills across all three assumptions. Thus, the “skills as scale substitute” phenomenon remains visible after retrieval, but its strength depends on whether downstream tasks need one key skill or full multi-skill coverage .

Assumptions and limitations. This simulation rests on several simplifying assumptions that should be noted: 25 Preprint. Under review. 1. Coverage semantics are approximate. Hit@1, R@10, and FC@10 correspond to different assumptions about how agents consume retrieved skills. Real agents may obtain partial benefit from incomplete skill sets, so the true pass rate may lie between the R@10 and FC@10 estimates. 2. Binary benefit model. We still model retrieval as two-state: sufficient coverage yields gold-skill performance, insufficient coverage yields vanilla performance. In reality, partially relevant skills may help somewhat, while misleading skills may hurt. SkillsBench reports that 16 of 84 tasks show negative skill effects, so both deviations are possible. 3. Transferability of C . Our coverage values are measured on our benchmark (150 queries, 

∼80K skills). The actual retrieval difficulty and skill cardinality distribution on Skills-Bench’s 84 tasks may differ. These numbers should therefore be interpreted as calibrated scenario estimates rather than exact predictions. 26
