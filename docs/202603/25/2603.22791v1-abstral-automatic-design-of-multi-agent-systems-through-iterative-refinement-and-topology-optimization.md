## 摘要
多智能体系统应如何设计？这种设计知识能否以一种可检查、可修订且可迁移的形式被捕获？我们介绍了 ABSTRAL，这是一个将多智能体系统（MAS）架构视为演进式自然语言文档的框架，该文档是通过对比轨迹分析（contrastive trace analysis）进行细化的产物。研究得出了三项发现：首先，我们对多智能体协作成本（coordination tax）进行了精确测量：在固定轮次预算下，集成系统的轮次效率仅为 26%，66% 的任务耗尽了限制，但通过发现可并行的任务分解，其表现仍优于单智能体基准。其次，编码在文档中的设计知识具有可迁移性：在一个领域学习到的拓扑推理和角色模板为新领域提供了先发优势，迁移后的种子在单次迭代中即可达到冷启动第 3 次迭代的性能。第三，对比轨迹分析能够发现初始设计中不存在的专家角色，这是以往系统所不具备的能力。在 SOPBench（包含 134 个银行任务，具有确定性真值）上，以 GPT-4o 为骨干网络的 ABSTRAL 达到了 70% 的验证集通过率和 65.96% 的测试集通过率。我们发布了收敛后的文档，作为可检查的设计原理。

## Abstract
How should multi-agent systems be designed, and can that design knowledge be captured in a form that is inspectable, revisable, and transferable? We introduce ABSTRAL, a framework that treats MAS architecture as an evolving natural-language document, an artifact refined through contrastive trace analysis. Three findings emerge. First, we provide a precise measurement of the multi-agent coordination tax: under fixed turn budgets, ensembles achieve only 26% turn efficiency, with 66% of tasks exhausting the limit, yet still improve over single-agent baselines by discovering parallelizable task decompositions. Second, design knowledge encoded in documents transfers: topology reasoning and role templates learned on one domain provide a head start on new domains, with transferred seeds matching coldstart iteration 3 performance in a single iteration. Third, contrastive trace analysis discovers specialist roles absent from any initial design, a capability no prior system demonstrates. On SOPBench (134 bank tasks, deterministic oracle), ABSTRAL reaches 70% validation / 65.96% test pass rate with a GPT-4o backbone. We release the converged documents as inspectable design rationale.

### ABSTRAL：通过迭代细化与拓扑优化的多智能体系统自动设计

#### 1. 核心问题与研究动机
*   **研究背景**：当前大语言模型（LLM）驱动的多智能体系统（MAS）设计主要依赖人工经验。现有的自动化方法（如 ADAS、AFlow）通常将中间知识存储为代码、提示词或神经参数，这些形式难以被人类检查、修订或在不同领域间迁移。
*   **核心问题**：能否将 MAS 的架构设计知识捕获为一种可读、可演化且可迁移的自然语言文档？
*   **研究目标**：提出 ABSTRAL 框架，将 MAS 架构视为一个不断演进的 Markdown 文档（`SKILL.md`），通过分析执行轨迹（Trace）自动优化智能体的拓扑结构和角色分工。

#### 2. 方法论
ABSTRAL 采用三层架构循环优化 MAS 设计：
*   **核心载体（SKILL.md）**：包含四个部分：
    *   **K (领域知识)**：任务相关的特定事实。
    *   **R (拓扑推理)**：条件规则（如“若任务具有属性 P，则倾向于拓扑 T”）。
    *   **T (角色模板)**：发现的专家角色及其系统提示词、工具权限。
    *   **P (构建协议)**：构建可运行智能体图的步骤。
*   **内环：轨迹驱动细化**：
    1.  **构建（Build）**：元智能体根据 `SKILL.md` 生成智能体规范。
    2.  **运行（Run）**：在沙盒中执行任务并捕获全量执行轨迹。
    3.  **分析（Analyze）**：对比成功与失败的轨迹，将失败归类为五类证据（EC1-EC5）。
    4.  **更新（Update）**：根据证据类别对 `SKILL.md` 进行针对性编辑。
*   **中层：收敛与整合**：监控技能差异、成功率平台等信号，定期合并冗余规则，防止语义漂移。
*   **外环：拓扑多样性搜索**：利用**图编辑距离（GED）**和语义距离作为排斥约束，强制系统探索性质不同的拓扑家族（如层级式、集成式、辩论式等），避免陷入局部最优。
*   **关键技术：专家角色发现（EC3）**：当分析发现单个智能体处理了结构上不兼容的子任务时，元智能体会自动推断并创建一个新的专家角色。

#### 3. 实验设计
*   **数据集/场景**：使用 **SOPBench**（2025）基准测试中的银行领域任务，包含 134 个任务。
*   **评估标准**：采用确定性 Oracle 评估（5 项布尔准则：无工具调用错误、无约束违反、数据库匹配、先决条件满足、目标动作正确）。
*   **对比方法**：
    *   **单智能体基准**：GPT-4o, Claude-3.7-Sonnet, o4-mini-high 等。
    *   **消融实验**：仅内环优化（固定拓扑）vs. 全量 ABSTRAL（含外环拓扑搜索）。
    *   **知识迁移实验**：对比“冷启动”与“携带已学习 K 知识启动”的性能。

#### 4. 资源与算力
*   **模型使用**：元智能体（负责构建/分析/更新）使用 **Claude 3.5 Sonnet**；智能体骨干网络使用 **GPT-4o**。
*   **算力消耗**：
    *   **Token 消耗**：整个搜索过程（24 次迭代）消耗约 **15.4 万个 Token**。
    *   **经济成本**：骨干网络推理费用约 **5 美元**，元智能体费用约 0.5 美元。
    *   **时间成本**：总耗时 **28.4 小时**，其中 79% 的时间用于任务执行（RUN 阶段）。
*   **硬件环境**：沙盒环境配置为 2GB RAM。

#### 5. 实验数量与充分性
*   **实验规模**：进行了 3 个外环循环，每个外环包含 8 次内环迭代，共 24 轮次。
*   **验证与测试**：将 SOPBench 拆分为 40 个验证任务和 94 个测试任务。
*   **充分性评价**：实验涵盖了消融研究、知识迁移分析、证据类别分布统计以及拓扑多样性验证。虽然主要在银行领域进行，但通过 24 轮迭代展示了清晰的收敛轨迹。不过，论文提到部分结果来自单次运行，存在一定的随机性偏差（第二次运行峰值从 70% 降至 65%）。

#### 6. 主要结论与发现
*   **协作税（Coordination Tax）的量化**：首次精确测量了 MAS 的开销。在固定轮次预算下，集成系统的轮次效率仅为 **26%**（4 轮中仅 1 轮在调用工具），66% 的任务因达到轮次上限而失败。
*   **MAS 的价值在于并行化**：尽管有协作税，MAS 仍优于单智能体，前提是任务结构允许并行分解。
*   **设计知识的可迁移性**：在一个领域学习到的 `SKILL.md` 能显著加速新领域的优化。迁移后的种子在第 1 次迭代的表现即可匹配冷启动第 3 次迭代的水平。
*   **自动角色发现**：通过对比轨迹分析，系统成功发现了初始设计中未定义的专家角色（如“外汇专家”、“信用申请验证员”）。

#### 7. 优点
*   **可解释性强**：优化目标是自然语言文档，设计逻辑透明，人类可读、可干预。
*   **拓扑感知搜索**：引入 GED 约束，解决了自动化设计中容易陷入相似拓扑结构的难题。
*   **角色演化能力**：能够根据执行失败的因果分析，动态产生新的专家分工，而非简单的提示词微调。

#### 8. 不足与局限
*   **轮次预算敏感**：由于协作开销大，在严格限制对话轮次的场景下，MAS 可能因路由环节过多而导致任务超时失败。
*   **实验覆盖面**：主要集中在 SOPBench 的银行领域，跨 7 个领域的全面迁移能力尚待更大规模验证。
*   **框架开销干扰**：实验中发现 LangGraph 等框架本身的节点转换会消耗轮次，这与直接调用 API 的基准测试之间存在评估偏差。
*   **运行波动**：由于采用小批量（20 个任务）采样进行验证，实验结果存在一定的方差。
