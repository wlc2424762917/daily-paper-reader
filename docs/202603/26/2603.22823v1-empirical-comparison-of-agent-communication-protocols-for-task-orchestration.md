## 摘要
背景：如今，人工智能智能体系统正从单一工具交互向复杂的多智能体编排转型。因此，出现了两种竞争性的通信协议：一种是规范智能体如何调用外部工具的工具集成协议，另一种是使自主智能体能够相互发现并委派任务的智能体间委派协议。尽管这些协议已被数十家企业合作伙伴在工业界广泛采用，但现有文献中尚缺乏对这些协议的实证比较。目标：本研究旨在开发首个系统性基准，在三个复杂度级别的标准化查询下，对纯工具集成、多智能体委派及混合架构进行比较，并量化其在响应时间、上下文窗口消耗、货币成本、错误恢复和实现复杂度方面的权衡。

## Abstract
Context. Nowadays, artificial intelligence agent systems are transforming from single-tool interactions to complex multi-agent orchestrations. As a result, two competing communication protocols have emerged: a tool integration protocol that standardizes how agents invoke external tools, and an inter-agent delegation protocol that enables autonomous agents to discover and delegate tasks to one another. Despite widespread industry adoption by dozens of enterprise partners, no empirical comparison of these protocols exists in the literature. Objective. The goal of this work is to develop the first systematic benchmark comparing tool-integration-only, multi-agent delegation, and hybrid architectures across standardized queries at three complexity levels, and to quantify the trade-offs in response time, context window consumption, monetary cost, error recovery, and implementation complexity.

### 1. 核心问题与研究背景
*   **研究动机**：随着 AI 智能体从简单的工具调用转向复杂的多智能体协作，工业界出现了两种主要的通信范式：一种是 Anthropic 推出的用于标准化工具调用的 **MCP**，另一种是 Google 推出的用于智能体间任务委派的 **A2A**。
*   **核心问题**：尽管这两种协议已被广泛采用，但目前缺乏关于它们在不同任务复杂度下的性能（延迟、成本、上下文消耗等）对比的实证数据。开发者在选择架构时往往依赖直觉而非数据。

### 2. 方法论
论文设计并实现了三种架构来处理相同的业务逻辑（开源项目健康度分析）：
*   **架构 A (纯 MCP)**：单智能体架构。智能体通过 stdio 传输协议连接四个 MCP 工具服务器，所有工具返回的数据都累积在同一个上下文窗口中。
*   **架构 B (多智能体 A2A)**：多智能体架构。一个协调者智能体通过 HTTP/JSON-RPC 协议将子任务委派给四个专门的智能体。每个智能体拥有独立的上下文窗口。
*   **架构 C (混合架构)**：引入一个轻量级的启发式路由层，根据查询关键词判断复杂度。简单查询走 MCP，复杂查询走 A2A。
*   **技术细节**：A2A 智能体通过 `/.well-known/agent.json` 发布能力卡片进行发现；MCP 则侧重于工具的直接集成。

### 3. 实验设计
*   **应用场景**：开发了一个“开源健康分析器”，对接 GitHub、npm、OSV.dev 和 StackOverflow 四个公共 API。
*   **Benchmark 构成**：定义了 30 个标准查询，分为三个难度等级：
    *   **简单**：单数据源、单项目。
    *   **中等**：多数据源、单项目。
    *   **复杂**：多数据源、多项目对比。
*   **对比方法**：对上述三种架构在相同模型（Claude Sonnet 20250514）下进行对比。
*   **评估指标**：端到端延迟、总 Token 消耗、货币成本、API 调用次数及代码复杂度（LOC 和圈复杂度）。

### 4. 资源与算力
*   **算力说明**：论文未提及具体的本地 GPU 训练算力，因为实验是基于 LLM API（Claude Sonnet）进行的。
*   **实验环境**：所有 A2A 智能体部署在 localhost（本地回环）以消除网络波动对延迟的影响。
*   **模型成本**：按每百万输入 Token $3、每百万输出 Token $15 的费率计算成本。

### 5. 实验数量与充分性
*   **实验规模**：每个架构对 30 个查询各执行 5 次，总计 **450 次受控实验执行**。
*   **统计严谨性**：采用了 95% 布特斯特拉普（Bootstrap）置信区间、曼-惠特尼 U 检验（Mann–Whitney U test）以及 Cliff’s delta 效应量分析。
*   **充分性评价**：实验设计较为充分，涵盖了从简单到复杂的梯度，且通过多次重复实验和非参数统计检验确保了结果的客观性和公平性。

### 6. 主要结论与发现
*   **性能交叉效应（Crossover Effect）**：
    *   在**简单任务**中，MCP 显著优于 A2A（延迟 8.9s vs 19.8s），因为 MCP 没有 HTTP 通信和委派决策的开销。
    *   在**复杂任务**中，A2A 反超 MCP（延迟 45.1s vs 51.8s），且 **Token 消耗仅为 MCP 的 1/3 左右**。
*   **上下文膨胀（Context Bloat）**：量化了单智能体架构在处理复杂任务时的缺陷——由于所有工具响应都堆积在单一窗口，MCP 的 Token 消耗呈指数级增长，而 A2A 通过分布式上下文保持了每个智能体的“精简”。
*   **成本优势**：在复杂任务下，A2A 比 MCP 节省了约 39% 的成本。
*   **混合架构表现**：混合架构在各复杂度下均能追踪到最优表现，证明了动态路由的有效性。

### 7. 优点与亮点
*   **首创性**：填补了 MCP 与 A2A 协议实证对比的空白。
*   **量化贡献**：首次定量分析了“上下文窗口膨胀”对成本和性能的具体影响。
*   **实用性强**：提出了一个清晰的决策框架（简单任务用 MCP，复杂编排用 A2A，未知情况用混合路由）。
*   **开源贡献**：提供了完整的 Benchmark 代码和数据集。

### 8. 不足与局限
*   **模型单一性**：实验仅使用了 Claude Sonnet，结果在其他模型（如 GPT-4 或开源模型）上的泛化性有待验证。
*   **网络环境理想化**：A2A 在 localhost 运行，忽略了真实生产环境中的网络延迟，这可能会推迟 A2A 性能反超 MCP 的临界点。
*   **领域局限**：仅测试了开源项目分析一个领域，未覆盖长文本处理或多模态任务。
*   **样本量限制**：每组查询仅 5 次运行，虽然统计显著，但对于捕捉极低概率的异常失败可能不足。
