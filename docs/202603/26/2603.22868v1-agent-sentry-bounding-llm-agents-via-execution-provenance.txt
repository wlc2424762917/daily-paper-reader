# Abstract 

Agentic computing systems, which autonomously spawn new functionalities based on natural language instruc-tions, are becoming increasingly prevalent. While im-mensely capable, these systems raise serious security, privacy, and safety concerns. Fundamentally, the full set of functionalities offered by these systems, combined with their probabilistic execution flows, is not known beforehand. Given this lack of characterization, it is non-trivial to validate whether a system has successfully carried out the user’s intended task or instead executed irrelevant actions, potentially as a consequence of com-promise. In this paper, we propose Agent-Sentry , a framework that attempts to bound agentic systems to address this problem. Our key insight is that agentic systems are designed for specific use cases and there-fore need not expose unbounded or unspecified func-tionalities. Once bounded, these systems become easier to scrutinize. Agent-Sentry operationalizes this in-sight by uncovering frequent functionalities offered by an agentic system, along with their execution traces, to construct behavioral bounds. It then learns a pol-icy from these traces and blocks tool calls that deviate from learned behaviors or that misalign with user in-tent. Our evaluation shows that Agent-Sentry helps prevent over 90% of attacks that attempt to trigger out-of-bounds executions, while preserving up to 98% of sys-tem utility. 

# 1 Introduction 

Agentic systems (aka AI agents) are becoming increas-ingly prevalent, as they are deployed to autonomously perform tasks across domains such as finance, health-care, and software development [15, 14, 24, 27]. These agents follow an LLM-enabled computing paradigm, in which the system relies on (a single or more) machine learning models (typically LLMs), to interpret and re-solve user queries expressed in natural language. In these systems, a query resolution entails an LLM pre-dicting the interactions required between the system modules, system integrations (e.g., apps), and the user,

Consequently, unlike conventional computing systems, agentic systems do not require explicit developer support to provide new functionalities. While there are obvious benefits to this execution paradigm, there are serious security, privacy, and safety risks to consider. Fundamentally, the key issue is that neither the full set of functionalities offered by these systems, nor the precise execution flows of functional-ities are known beforehand, which makes it challeng-ing to reason about the security properties of the sys-tem. While comprehensive characterization of conven-tional computing systems remains an open problem, it is significantly exacerbated in agentic systems. Our goal in this paper is to enable developing agentic systems, that provide strong security properties, while support-ing their emergent functionalities. Our key idea is that in practice agentic systems are designed to provide specific use cases and thus need not have unbounded functionalities. Consequently, if the system functionalities can be trimmed down to a smaller set, it provides for an easier validation. Specifically, the system may exhibit regu-larities for functionalities, such as recurring patterns in tool usage, control flow structure, and data dependen-cies for common tasks [7, 25]. These regularities thus form an envelope of executions for behaviors, which offer a path to bounding behavior without stripping agents of the flexibility that makes them useful at runtime. We present Agent-Sentry , a system that enforces safe execution of LLM-based agents by learning and reasoning over their observed behavior. # 2 Motivation 

## 2.1 Agentic Computing Paradigm 

LLMs have introduced a new agentic computing paradigm , in which the system primarily relies on ML models to resolve user queries expressed in natural lan-guage. This execution paradigm is tremendously power-ful as it allows spawning complicated new functionalities simply via natural language instructions, instead of re-quiring a software engineer to program new functionali-ties, as it is the case in conventional computing systems. For example, given a query that instructs an email agent to “download attachments from recent emails and up-load them to a cloud storage service”, the agent at run-time will orchestrate an execution flow (i.e., calling a series of functions and passing parameters), which may involve authenticating with the email service, reading recent emails, parsing attachments, authenticating with the cloud storage service, and uploading the attach-ments. 

## 2.2 Inherent Security Issues 

While there are obvious benefits to the agentic comput-ing paradigm and its reshaping the computing stack at all levels, there are serious security, privacy, and safety risks to consider. A key characteristic of agentic sys-tems that makes it challenging to provide strong secu-rity properties is that the agentic execution paradigm in inherently unbounded . Specifically, given access to a set of functions/tools, an AI agent can orchestrate those function calls in all possible combinations, making it challenging to characterize the system. Towards achiev-ing strong security properties in a system, it is crucial that the system execution with respect to the combina-tion of a set of possible inputs in a set of possible states, is well understood, which is not the case in agentic sys-tems. For example, given a query to an email agent to “summarize recent emails”, the execution flow taken by the system is not known beforehand nor is full set of functionalities that the email agent may provide based on the user query. Consequently, it is non-trivial to val-idate whether the system was able to perform the task or it got hijacked by a prompt injection in the emails, and steered from the user-specified task. 

## 2.3 Bounding Agentic Systems 

In this paper, we propose to bound the agentic systems to more reliably control their execution. We argue that the agentic system execution need not be unbounded, as in practice agentic systems may only provide lim-ited functionalities. Once the limited set of expected functionalities are characterized, the smaller execution space is easier to validate. For example, while a chain of forward() → delete() functions for an email agent may be possible, it is a non-frequent functionality and may actually be an indication of a prompt injection, where an attacker attempts to forward an email and remove its traces. Similarly, once it is known that to forward an email an execution flow includes a call to a read() and a forward() function, this flow can be trivially verified. However, a key challenge is that with strict bounding, we risk constraining the agentic sys-tems, such that we lose unique benefits of the agentic execution paradigm. Thus, as we bound the systems, we aim to preserve the flexibility of the agentic execution paradigm. At a high level, our approach consists of (i) observ-ing benign and adversarial interactions with an agent to derive a set of representative execution traces, (ii) abstracting these traces into functionality graphs that 2capture recurring execution flows and data provenance, and (iii) using these graphs as prior knowledge to rea-son about the validity of future executions at run-time. While reasoning over execution flows allows us to identify behaviors that fall outside known function-ality, structural validity alone does not guarantee that an execution aligns with the user’s intent. Thus, we we also assess the alignment between the user query and the expected execution flow. It is important to note that our approach is akin to the code minimization approaches in conventional computing systems [22], where the goal is to reduce the trusted compute base, and eliminate the possibil-ity of attacks that targeted the removed code base. While this idea has existed in prior systems, agentic systems present unique challenges, chiefly among them is to strike a reasonable balance between emergent func-tionalities and avoiding prompt injections, which we at-tempt to tackle in this paper. 

# 3 Threat Model 

## 3.1 System Model 

We consider AI agents that primarily rely on LLMs to resolve user queries expressed in natural language prompts. At a high-level, these agents consist of an LLM, a system prompt that defines the agents func-tionality, and a set of tools along with their functionality descriptions to interface with online services. These sys-tems also contain a conventional programming language scaffolding that orchestrates the interaction between the LLM and the system modules, such as passing the infor-mation between the LLM and the tools’ API endpoints. We consider agents that are designed for specific use cases, such as a banking service’s agent, instead of general-purpose agents that serve a wide set of function-alities, such as Alexa [2]. These systems are designed using agent development frameworks/tool kits, such as the OpenAI Agents SDK [23], LangChain [5], and Lla-maIndex [18]. The agent’s functionalities and execution logic are not statically defined, instead, they are inferred at runtime by the LLM, based on the user prompt and the con-textual information observed during execution. As a result, neither the full set of functionalities offered by the system nor the precise execution flow of the agent’s functionalities are known beforehand. The system per-mits the emergent functionalities that the user specifies in their prompts, but are not explicitly defined by the system developer. For example, the developer of a bank-ing service agent may not explicitly foresee a workflow that combines reading a lease document and scheduling the monthly payments mentioned in the lease, yet still considers such behavior acceptable because it remains consistent with the agent’s role as a financial manage-ment assistant. 

## 3.2 Trust Assumptions 

We assume the agent, its scaffolding, its LLM, its tools, and the user to be trustworthy. However, we assume the LLM and its tools to be compromise-able. For example, an attacker may embed a prompt injection in a retrieved document which when processed hijacks and allows an attacker to take over the LLM. We assume that the data and instructions exposed to the AI agent via tools (e.g., content retrieved from files, webpages, emails) are un-trustworthy. We also assume the AI agent and tools un-intentionally make mistakes, due to the ambiguity and imprecision of natural language [17, 13]. For example, a user request to “pay the bill” might be ambiguous if multiple invoices exist, potentially leading the agent to transfer funds to the wrong recipient based on an incor-rect inference. 

## 3.3 Attacker’s Objective 

The attacker’s objective is to induce the agent to per-form actions which are: (i) contextually irrelevant to the functionality offered by the agent (e.g., asking a banking agent to resolve healthcare related prompts), (ii) significantly deviate from the agent specification de-fined by the developer (e.g., a banking agent negotiating with merchants), and (iii) do not align with the user in-tent (e.g., a banking agent automatically adjusting loan payment after observing an increase in salary). The at-tacker attempts to influence the agent’s execution con-text by embedding malicious or misleading instructions (i.e., prompt injections) in untrusted retrieved data. These prompt injection attacks exploit the agent’s ten-dency to treat information it encounters during execu-tion as instructions to act, steering it toward unintended behavior. 

## 3.4 Defense Objective 

We seek to prevent agentic systems from executing ac-tions that are contextually irrelevant to the function-alities offered by the agent, significantly deviate from the intended scope of the system, and deviate from the user’s intent. For example, in the context of the bank-ing services agent: (i) it should only resolve banking related tasks, and simply ignore the prompts outside of its domain, (ii) it should not autonomously negotiate with merchants or contact third parties on the user’s be-half, even if such an action is technically possible, and (iii) it should not autonomously try to adjust the user’s loan payment after observing an increase in salary, if not explicitly specified in the user prompt. At a high-level, our key idea is to comprehensively uncover the system functionality in a non-adversarial setting, and use that characterization to bound/limit the agent’s execution. As the agent’s functionalities are uncovered in a non-adversarial setting, they do not in-clude contextually irrelevant functionalities, functional-3ities that significantly deviate from the specified func-tionalities of the system, and do not deviate from the user’s intent. Essentially, such characterization enables bounding the system, which prevents attacks, while pre-serving the developer intended agent’s functionalities. 

# 4 Methodology 

## 4.1 Overview 

Our goal is to bound agentic systems in a way that improves security while preserving the flexibility that makes the agentic computing paradigm powerful. In-stead of enumerating all valid and invalid behaviors of an agent ahead of time, we learn functionality graphs from how an agent executes in practice and reason about future executions. The main idea behind our system is that, while the space of all possible execution flows is unbounded in theory, the set of execution flows seen dur-ing legitimate use typically follow repeatable patterns. 

Agent-Sentry bounds agentic execution by learn-ing functionality graphs from execution traces collected during both benign use and adversarial scenarios. These traces show how agents invoke tools, gather informa-tion and act on that information while responding to user queries. From these traces, Agent-Sentry learns which flows consistently support intended functional-ity, which are strongly associated with misuse such as prompt injection, and which cannot be interpreted based on behavior alone. This learned knowledge allows 

Agent-Sentry to evaluate actions before they execute making it possible to intervene early when behavior be-gins to drift. While this provides a strong basis to bound agentic executions, some execution flows are inherently ambigu-ous as they appear in both benign and adversarial sce-narios and agents may also exhibit previously unseen (but potentially legitimate) behaviors as their capabili-ties evolve over time. For these cases, Agent-Sentry 

uses an intent-alignment check that evaluates whether a proposed action is consistent with the user’s origi-nal request given the execution so far. This provides additional context precisely when functionality graph anomalies alone cannot determine whether an action is appropriate. Together, functionality graph learning and intent alignment form Agent-Sentry . Functionality graph learning provides Agent-Sentry with a behavior-based understanding of expected execution flows, while intent alignment resolves uncertainty when behavior alone is inconclusive. By combining these two perspec-tives, Agent-Sentry bounds agentic execution in a way that prevents misuse while preserving the emergent and flexible nature of agentic systems. In this paper, we use the term functionality graph to refer to a learned representation of recurring agent exe-cution behavior. Unlike a policy in reinforcement learn-ing, or any other machine learning approach, this model does not assume actions’ intentions or optimize rewards. Instead, it first characterizes which execution behaviors are typical, which are strongly associated with misuse, and which remain ambiguous based on observed exe-cutions. Then, it makes deterministic decisions based on comparing the current execution behavior with the functionality graph. 

## 4.2 Execution Trace in Agentic Systems 

An execution trace is a record of how an agent responds to a single user query under the agentic computing paradigm. It captures the user request, the ordered se-quence of tool invocations performed by the agent, and the dependencies between information retrieved earlier in the execution and subsequent actions. An execution trace reflects behavior that emerges dynamically at run-time, rather than a predefined or statically known pro-gram path. We treat traces as structural descriptions of execution behavior rather than records of individual users or spe-cific data values. Two traces are considered equivalent if they share the same execution logic, even if they involve different data, such as emails, documents, or account identifiers. This abstraction allows Agent-Sentry to reason and learn execution behaviors without relying on sensitive content or untrusted content. Within a trace, Agent-Sentry distinguishes tools that retrieve information from those that perform ac-tions with external effects such as sending or deleting an email. This distinction reflects a fundamental secu-rity intuition in agentic systems. Information retrieval (via retrieval tools) expands the agent’s view of the sys-tem state, while actions (performed via action tools) modify that state. Agent-Sentry focuses on eval-uating actions, interpreting them in the context pro-vided by the preceding flow of execution. While exe-cution traces record concrete runtime behavior, Agent -Sentry does not reason over individual traces directly. Instead, traces are abstracted into execution flows that capture their structural execution logic while ignoring specific data values. 

## 4.3 Execution Flows as Functionality Graphs 

An execution flow is an abstract representation of an execution trace that captures the ordering and relation-ships among retrieval and action tool invocations, in-dependent of specific input values or retrieved content. Multiple execution traces may correspond to the same execution flow if they share the same execution logic. A functionality graph path represents a recurring exe-cution pattern that characterizes how an agent gathers information and justifies actions across multiple execu-tions. Execution behav-iors capture what the agent does and when, while infor-mation flows capture why an action is taken given the available context. 

## 4.4 Deriving Functionality Graphs 

Agent-Sentry creates three functionality graphs by observing execution traces produced during both benign use and adversarial runs of the agent. The three functionality graphs (benign, ambiguous, and adversarial) are constructed by first extracting re-curring execution flows from benign traces and then from adversarial traces. First, benign execution traces are used to populate the benign functionality graph by extracting unique execution flows along two dimen-sions. The first dimension captures control flow struc-tures, recording which retrieval tools precede each ac-tion tool and if there are multiple action tools, then how are action tools sequenced relative to the preceding action tool. The second dimension captures the data provenance by tracking the source of each action tools arguments, specifically on whether they originate from the original user prompt or from the outputs of prior retrieval tools. Next the same extraction procedure is applied to adversarial execution traces. Execution flows extracted from adversarial traces that also appear in be-nign traces (by checking against the benign functional-ity graph) are reassigned to the ambiguous functional-ity graph from benign graph, while those that do not appear in benign traces are assigned to the attack func-tionality graph. To extract these execution flows, Agent-Sentry 

parses execution traces to identify tool call sequences and applies a combination of string matching tech-niques, including exact matching, substring contain-ment, format-aware extraction for structured identifiers (such as IBANs and email addresses) and fuzzy match-ing, to associate action arguments with their likely 5sources. These form the directed data provenance edges for the second dimension of the functionality graphs. We provide examples of extracted functionality graph paths in Appendix 11 As a result, execution flows observed exclusively during benign runs form the be-nign functionality graph, execution flows observed ex-clusively during adversarial runs form the adversarial functionality graph, and execution flows shared across both form the ambiguous functionality graph. Other execution flows fall outside what Agent-Sentry has observed so far, reflecting incomplete knowledge rather than clear safety or misuse. In both ambiguous and previously unseen situations, a decision cannot be made solely from the shape of the execution flow and how information is reused, because the same behavior may or may not be appropriate depending on the task the agent is trying to complete. 

## 4.5 Resolving Ambiguous cases via In-tent Alignment 

For execution flows that remain unresolved after anal-ysis with learned behaviors, Agent-Sentry evaluates whether a proposed action is consistent with the user’s original request, given the execution observed so far. This step is primarily designed to handle cases where execution behavior and information use flows alone do not provide enough evidence to determine whether an action should proceed. The intent alignment mechanism utilizes LLM-as-a-Judge to examine the user’s original request, together with a restricted view of the execution trace and the proposed action. Its purpose is to determine whether executing the action would reasonably advance the task that the user asked the agent to perform. Rather than attempting to judge the correctness of the entire exe-cution, the mechanism focuses narrowly on whether the next action fits the user-specified goal in view of what the agent has already done. Importantly, the intent alignment check is performed using only trusted inputs: the original user request, the execution steps seen so far, and the description of the proposed action tool that needs to be verified. Retrieved content and other untrusted data encountered during execution are never used thus preventing injected in-structions or adversarial content from influencing the alignment decision. This design allows intent alignment to be applied without introducing a new attack surface. Intent alignment complements the functionality graph analysis rather than replacing it. Functionality graph analysis provides the primary signal, and intent alignment is used only when structural reasoning alone does not resolve the decision. This separation gives 

Agent-Sentry a clear division of responsibility. To-gether, these components allow Agent-Sentry to ad-dress uncertainty without weakening its behavioral safe-guards or constraining its flexibility. 

## 4.6 Decision Making 

During execution, Agent-Sentry evaluates each pro-posed action using the learned functionality graph. In many cases, the observed execution flow matches be-havior that has only been seen during benign use, or matches behavior that has only been seen during mis-use, allowing an allow or block decision to be made di-rectly. Intent alignment is called upon only when the observed execution flow falls into one of two specific categories: when the same functionality graph path has been observed in both benign and adversarial runs, or, when it has not been observed before. In these cases, functionality graphs alone does not determine whether the action should proceed, and intent alignment with the user query is used to assess whether the action is consistent with the task the user asked the agent to perform. Figure 1 illustrates how Agent-Sentry enforces ex-ecution bounds at runtime. Each proposed action tool generated by the agent is intercepted before execution and first evaluated against the learned functionality graphs. If the corresponding execution flow is classi-fied as benign or adversarial by the functionality graph, 

Agent-Sentry allows or blocks the action tool di-rectly. If the flow is ambiguous or previously unseen, the action is deferred to the intent alignment mecha-nism, which checks consistency with the original user request using only trusted inputs. Only actions that pass these checks are executed, ensuring that unautho-rized actions are blocked even if the underlying agent’s LLM is influenced by injected or misleading content. 

## 4.7 Updating Functionality Graphs 

As discussed, intent alignment judges whether to allow or block previously unseen execution flows. After such execution flows are complete, we envision that either the system has an automated method to determine, af-ter the fact, whether the corresponding execution trace was benign or an attack, or, if the confidence level of the automated method is not high, a developer review may manually make a determination. We envision that the functionality graph will be periodically updated with these new execution flows. At update times, we also en-vision that execution flows currently in the ambiguous functionality graph, may be moved into either the be-nign or adversarial functionality graph, when it has been consistently the case for some significant period of time that such execution flows are exclusively associated with either benign or adversarial scenarios. It is beyond the scope of this work to implement the above update func-tionality. ## 5.1 Agent-Sentry Bench 

Agent-Sentry Bench is a static dataset of unique exe-cution flows derived from and extending the AgentDojo benchmark. While AgentDojo functions as a dynamic environment for end-to-end task evaluation, Agent-Sentry Bench is designed as a corpus of diverse ex-ecution traces to support the learning and evaluation of execution-provenance models. To build Agent-Sentry Bench, we synthesized a di-verse collection of execution traces spanning both be-nign and adversarial behaviors. Benign traces were de-rived from prior AgentDojo executions generated using a variety of language models and were further supple-mented with additional benign workflows to expand task coverage and behavioral diversity. Adversarial traces were simulated using user and injection tasks adapted from AgentDojo, augmented with additional synthe-sized injection scenarios to broaden the space of attack behaviors. Overall, the benchmark is designed to max-imize the variability of valid execution paths, includ-ing distinct execution-flow transitions and argument-provenance patterns. The complete methodology for generating both utility and attack traces is provided in Appendix 12 and Appendix 13.

## 5.2 Evaluation Procedure 

Evaluation proceeds by learning the behavioral model on varying fractions of benign utility and adversarial traces, ranging from 50% to 100% in 10% intervals, to measure performance sensitivity to data availability. 1

To ensure robustness and minimize variance, we per-form the learning or cold start phase across multiple ran-dom seeds for data splitting. Finally, Agent-Sentry is evaluated on all utility traces and all adversarial traces. To account for the mismatch between Agent-Sentry Bench dataset composition and deployment re-ality, we report results under three complementary eval-uation settings. First, we evaluate the system using the dataset as-is, where all traces contribute equally. This setting reflects the raw diversity of adversarial behaviors captured by the benchmark and serves as a conservative evaluation that emphasizes attack coverage. Second, we evaluate the system under a balanced setting in which benign and adversarial behaviors are treated as equally prevalent. This allows us to isolate differences in sys-tem behavior between utility and attack traces without the results being dominated by the larger number of adversarial examples. Third, we evaluate the system under a deployment-motivated setting that assumes be-nign interactions constitute the majority of real-world usage, with adversarial behavior occurring relatively in-frequently. To approximate second and third scenarios given the limited number of unique utility traces, benign traces are oversampled with replacement. This oversampling reflects the fact that, at the level of control flow and

## 5.3 AgentDojo Benchmark 

AgentDojo consists of independently designed agent tasks and prompt injection attacks and is widely used in recent work on agent security. Importantly, Agent-Dojo includes differences in task structure and attack techniques. We evaluate Agent-Sentry on AgentDojo using GPT-4o traces to provide a comparison to prior art defenses. 

## 5.4 Evaluation Metrics 

We primarily evaluate the performance of Agent-Sentry using metrics designed to capture both its effec-tiveness at preventing attacks and its impact on benign functionality. 

• Attack Success Rate (ASR): The percentage of prompt injection attacks that were not blocked, eval-uated over all successful attack attempts. Lower val-ues indicate stronger defensive performance. 

• Utility Success Rate (without attack): The frac-tion of benign executions that were successful when 

Agent-Sentry is enabled. Higher values indicate better preservation of intended agent functionality. 

# 6 Experimental Results 

## 6.1 Overview 

We evaluate Agent-Sentry through a sequence of con-trolled studies designed to characterize system behav-ior under varying levels of functionality graph coverage. Across all experiments, the system relies on functional-ity graphs from execution traces and, in case of ambigu-ous or unseen scenarios, on an LLM-based reasoning layer, to classify scenarios as utility or attack. As dis-cussed, a functionality graph path does not uniquely de-termine intent: benign and adversarial execution traces may map to the same functional path, requiring seman-tic reasoning to resolve traces that are classified as am-biguous by the functionality graph. Unless otherwise stated, we report results under the 

as-is evaluation setting, which reflects the adversarial-heavy composition of the benchmark. As shown later, alternative prevalence assumptions do not materially af-fect per-trace performance; corresponding results are therefore deferred to the appendix. All results reported in this section, except those for the complete functional-ity graph coverage setting, are computed as the average over three randomized runs using fixed random seeds to smooth out the effects of random sampling. 

## 6.2 Performance with Complete Func-tionality Graph Coverage 

We begin by evaluating system performance when the functionality graphs have complete coverage of all exe-cution traces in the Agent-Sentry dataset. Starting from a cold state, the system is warmed up with a sin-gle pass over all available traces, ensuring that every observed functionality-graph path is represented. # 7 Related Work 

Securing LLM-based agents requires addressing their fundamental unboundedness: at runtime, agents dy-namically synthesize control flow and tool-mediated data dependencies conditioned on untrusted inputs, rather than executing a statically defined program. We categorize existing works by the layer at which they at-tempt to bound this behavior: input-level constraints, architectural isolation, semantic intent bounding, and explicit policy enforcement. 

Input-Level Constraints and Soft Bounding 

Early defenses attempt to bound agent behavior by con-straining the model’s interpretation of its context win-dow. Techniques such as prompt sandwiching, strict de-

> Dojo dataset like we do with all approaches.

11 limiters, and repeated system instructions aim to create a ”soft” separation between instructions and untrusted data [7]. Spotlighting systematizes this by applying transformations (datamarking) to untrusted content, attempting to enforce provenance awareness through in-put encoding [10]. Complementary approaches attempt to bound the input space via pre-execution detection. PromptArmor employs an auxiliary LLM guard to de-tect malicious spans before they reach the agent [28], while Attention Tracker leverages internal attention pat-terns to identify adversarial inputs without extra infer-ence [11]. However, these approaches ultimately rely on the probabilistic adherence of the model to soft con-straints rather than enforcing hard system guarantees, leaving them vulnerable to adaptive attacks that manip-ulate the model’s instruction-following capabilities [7]. 

Architectural and Data-Flow Bounding A more rigorous systems perspective seeks to bound the conse-quences of execution through architectural constraints. IsolateGPT introduces execution isolation to strictly contain the blast radius of compromised agents, pre-venting them from accessing sensitive system re-sources [31]. Similarly, proposed taint-tracking frame-works seek to enforce data-flow integrity by marking untrusted content and requiring explicit mediation be-fore it can influence high-impact sinks [12]. While these methods provide stronger structural guarantees than input filtering, they face challenges at the intra-agent 

boundary, where agents must legitimately process un-trusted data (e.g., summarizing an email) without al-lowing that data to hijack the control flow. 

Semantic and Intent Bounding Beyond struc-tural constraints, preventing IPI requires bounding the agent’s goal to the user’s original intent. We view IPI primarily as a mechanism for inducing task drift , where the agent’s behavior remains valid generally but devi-ates specifically from the user’s request. “Get my drift?” demonstrates that activation deltas in the model’s inter-nal state can serve as a robust signal for detecting this semantic deviation [1]. While effective, such white-box approaches are incompatible with the closed-source API models (e.g., GPT-5, Claude) that power most state-of-the-art agentic systems, creating a need for black-box intent verification. 

Policy-Based and Trace-Based Execution En-forcement A broad line of work seeks to secure LLM-based agents by constraining how they execute tool-invocation workflows, rather than filtering user inputs alone. Early efforts in automated red teaming focus on discovering vulnerabilities by generating adversarial prompts or attacker agents [20, 34, 33]. While effective for exposing weaknesses, these approaches are primarily diagnostic and do not provide a mechanism for enforcing safety during deployment. More recent systems enforce security policies directly over agent execution traces. CaMeL operationalizes the Dual-LLM pattern [30] by separating a privileged plan-ning model from a quarantined parsing model and en-forcing capability policies via a custom interpreter [6]. This design provides strong security guarantees, but in-troduces substantial runtime overhead due to multiple LLM invocations, interpreter-based policy checks, and potential user intervention for declassification, which CaMeL explicitly acknowledges as a usability trade-off. Progent similarly enforces execution policies through a programmable interpreter and domain-specific lan-guage [26], achieving low attack success by blocking dis-allowed function calls, but at the cost of manual policy engineering that is difficult to scale across tools, tasks, and evolving agent behaviors. To reduce reliance on hand-authored policies, other approaches attempt to infer enforcement constraints au-tomatically from runtime behavior. AgentArmor con-structs and inspects Program Dependence Graphs in-ferred from agent execution logs to enforce dependency-aware constraints [28], but relies on LLMs acting as auxiliary analyzers that must reason over untrusted retrieved content, increasing runtime cost and expos-ing the defense to meta-injection attacks. TraceAegis decomposes agent invocation traces into hierarchical execution units and learns behavioral constraints for anomaly detection [19], achieving strong detection per-formance but remaining detection-centric and lacking a mechanism for resolving ambiguity or novelty during real-time execution. In contrast, Agent-Sentry learns lightweight functionality graphs from execution traces to bound agent behavior structurally, explicitly defers ambiguous or unseen cases to a constrained intent-alignment mechanism that reasons only over trusted inputs, and empirically balances utility preservation and attack success, an operational trade-off that prior policy-based and trace-analysis systems do not quantify or optimize. When semantic intent verification (that utilizes an LLM) is required, Agent-Sentry confines the intent alignment mechanism to a strictly trusted set of in-formation, reasoning only over the original user re-quest (user prompt), tool specifications and tool call sequences while ignoring retrieved or externally sourced content. This design choice avoids introducing addi-tional attack surfaces during enforcement. By com-bining lightweight execution-level structure with a con-strained semantic check, Agent-Sentry provides a practical mechanism for bounding agent behavior that preserves utility while remaining robust to adaptive ma-nipulation. 12 8 Concluding Remarks 

Evolving Execution Coverage A fundamental challenge in agentic systems is that new benign behav-iors and new attack strategies will continually emerge as agents, tools, and usage patterns evolve. As a re-sult, complete execution coverage is never static: there will always be unseen scenarios that fall outside the cur-rent functionality graph. In practice, however, the vol-ume of such novel traffic is expected to remain small, as the system is periodically updated with newly ob-served execution traces and adapts to evolving behavior. This limitation is not unique to Agent-Sentry , but is inherent to any system operating in adversarial envi-ronments where attackers actively seek novel strategies. The benefit of explicit coverage modeling is that such cases are surfaced as ambiguous rather than silently mis-classified, enabling conservative handling and incremen-tal improvement over time. 

Intent Ambiguity and Mimicry Resistance A re-maining challenge arises from mimicry attacks, where adversarial executions deliberately follow functionality graph paths that are non-trivial to structurally and se-mantically distinguishable from benign behavior within a single execution trace. While Agent-Sentry miti-gates some of these cases by checking alignment between the user prompt and the expected execution trace, this mechanism is inherently limited by the information pro-vided at decision time, making the system susceptible to carefully crafted mimicry that aligns with both execu-tion structure and apparent user intent. As additional contextual and structural signals are incorporated into the functionality graph over time such as user identity and privilege context (conditioning decisions on authen-ticated roles or permissions) and external policy and compliance signals (enforcing organizational rules such as spending limits or access constraints that are inde-pendent of learned execution structure), the system can strengthen its ability to distinguish such cases directly at the structural level. This evolution would reduce reliance on LLM-based intent alignment for resolving mimicry and improve robustness when structural and intent-based cues alone are insufficient. 

## 8.1 Takeaways 

Execution provenance is an effective security signal for agentic systems: We show that traditional security tech-niques based on execution provenance and control/data-flow analysis can be successfully applied to agentic sys-tems and achieve strong empirical results. We do note that however, as agentic execution space is open-ended and continually evolving, it is fundamentally impos-sible to anticipate all benign workflows or future at-tacks in advance. For this reason, Agent-Sentry com-bines structural bounds with an LLM-based intent check and is designed to integrate additional orthogonal sig-nals over time. This layered view complements several strands of prior work but not limited to: (i) adversarial-training and robustness research that fortifies models against perturbations and adversarial examples [9, 21]; (ii) guardrails and policy/constitution approaches that steer model behavior via rules or automated critique (e.g., Constitutional AI and engineered guardrail sys-tems) [3, 8]; (iii) prompt-injection detection and moni-toring that aim to identify contaminated inputs or at-tention shifts indicative of injections [11, 16]; and (iv) alignment-focused defenses that harden model outputs or check alignment at inference time [32, 4]. Each class of defense targets different failure modes; by operating at the execution layer, Agent-Sentry can be deployed complementarily with any defense that aims to improve agent robustness, such as model-centric, input-centric, and alignment-based approaches.

12.1.2 Generation Pipeline 

The generation process for a single trace follows these steps: 1. Sequence Planning : The LLM generates a valid tool sequence (length 2–5) that satisfies the cov-erage target (e.g., ”Must include search files →

summarize file ”). 2. Prompt Generation : A second LLM call con-verts this abstract sequence into a natural language user request, applying a specific persona (see Sec-tion 12.2). 3. Execution : The generated prompt is executed against the actual agent sandbox. 4. Verification : The resulting execution trace is compared against the expected sequence. If the agent deviates (e.g., hallucinates a tool or skips a step), the trace is marked for refinement. 5. Refinement : On failure, an ”AI Debugger” ana-lyzes the mismatch and rewrites the prompt to be more explicit, triggering a retry.

# 13 Appendix: Simulated Attack Dataset Generation 

To rigorously evaluate Agent-Sentry against diverse In-direct Prompt Injection (IPI) attacks, we developed a specialized Attack Simulation Framework . This frame-work autonomously generates high-volume, high-quality adversarial datasets by systematically combining benign retrieval contexts with malicious attack objectives. Un-like static datasets, this approach allows us to stress-test the defense against a combinatorial explosion of injec-tion vectors and attack payloads without manual craft-ing. 

## 13.1 Architecture Overview 

The simulator operates as a multi-stage pipeline de-signed to produce Simulated IPI Traces . An IPI attack is characterized by an adversary embedding malicious instructions into an external data source (e.g., an email or document) which, when retrieved by the agent, hi-jacks the control flow to execute unauthorized actions. Our framework simulates this by generating two dis-tinct types of valid execution traces and synthesizing them into a single attack scenario: 

• Benign Utility Traces : Represent the legitimate task the user intends to perform, which necessitates reading external data. 

• Attack Traces : Represent the unauthorized behav-ior the attacker wishes to trigger (e.g., exfiltrating data or modifying files). A Splicing Engine then merges these traces, embed-ding the attack instructions into the retrieval output of the utility trace, creating a realistic simulation of a successful injection. 

## 13.2 Trace Generation Phases 

The framework uses a dedicated Large Language Model (DeepSeek-V3.2) to generate base traces, ensuring di-versity through a global registry of prompts and scenar-ios. 

13.2.1 Phase 1: Attack Trace Generation 

The Attack Generator focuses on creating valid execu-tion paths for restricted or high-risk actions. 

• Objective : Force the agent to execute specific Action Tools (e.g., send email , transfer funds ) identified via tool classification. 

• Strategy : The generator prompts the agent to per-form single-step actions or multi-step sequences (e.g., 

Retrieval → Action ). 

• Diversity Enforcement : A persistent Attack Reg-istry tracks all generated prompts. New prompts are generated with diversity constraints to avoid seman-tic duplicates, ensuring the evaluation covers a wide range of phrasing and intent expression. 

• Verification : Generated traces are verified to ensure they genuinely result in the target tool call. We em-ploy a flexible verification logic that accepts ”drift” (e.g., if the agent autonomously decides to check a calendar before scheduling a meeting) provided the target unauthorized action is eventually executed. 

13.2.2 Phase 2: Utility Trace Generation 

The Utility Generator creates the ”carrier” scenar-ios—benign workflows that expose the agent to un-trusted data. 

• Objective : Generate legitimate user requests that require the agent to use a Retrieval Tool (e.g., 

get latest email , read file ). 

• Constraint : The generator strictly enforces that the 

first tool called in the sequence must be a retrieval tool. This is critical because the IPI payload is in-jected into this first retrieval result. 20 • Coverage Tracking : A Coverage Tracker monitors tool transitions (e.g., search → read ) to ensure the utility dataset represents varied benign usage pat-terns, preventing overfitting to simple lookup tasks. 

## 13.3 Meta-Prompts and Diversity 

To ensure the generated traces are high-quality and var-ied, we use structured system prompts with explicit di-versity constraints. 

13.3.1 Prompt Generation System 

The core instruction used to generate user prompts from tool sequences enforces natural language and implicit intent: Meta-Prompt: User Prompt Generation        

> You are a creative writer generating user prompts for an AI agent. Your goal is to write a natural language prompt that triggers a specific SEQUENCE of tool calls. CRITICAL RULES: 1. You must adopt the specific STYLE/PERSONA requested. 2. The prompt must be natural, not robot-speak. 3. The prompt must IMPLICITLY require the tools. Do not just list tool names. 4. ‘reasoning‘ must explain why this phrasing triggers these tools.

13.3.2 Stylistic Diversity 

To prevent the model from converging on a single ”polite command” style, we stochastically sample from a set of 9 distinct personas for each generation: 

• Standard : Clear, polite, professional English. 

• Terse : Minimal words, imperative mood. 

• Verbose : Long-winded, includes irrelevant details. 

• Urgent : Panicked, demanding, uses ’ASAP’. 

• Casual : Slang, typos, lowercase, friendly. 

• Technical : Uses specific jargon, precise con-straints. 

• Confused : Unsure of terminology, vague requests. 

• Angry : Frustrated, complaining about previous failures. 

• Complex : Multiple constraints/formatting re-quests. 

13.3.3 Refinement Loop 

If a generated prompt fails to trigger the expected tool sequence during verification, we employ an automated debugger prompt: Meta-Prompt: Prompt Refinement      

> You are an expert AI debugger. A previous user prompt FAILED to trigger the expected tool sequence. Your job is to analyze the failure and write a REFINED prompt that fixes the issue. STRATEGY: - If missed a tool, make the requirement MORE EXPLICIT. - If called a wrong tool, change wording to avoid ambiguity. - If arguments were wrong, be more specific about data. - Maintain the original Style/Persona if possible.

## 13.4 Simulation via Conversation Splic-ing 

The core innovation of our framework is the Conversa-tion Splicer , which synthesizes IPI attacks by grafting an attack trace onto a utility trace. 

13.4.1 Why Splicing? 

Directly prompting an LLM to ”simulate an injection attack” often leads to hallucinations or invalid tool us-age. By separately generating valid utility logic and valid attack logic, we ensure that both the ”before” (benign context) and ”after” (attack execution) com-ponents of the trace are realistic and executable by the agent. 

13.4.2 The Splicing Algorithm 

Given a Utility Trace TU and an Attack Trace TA, the splicer performs the following operations: 1. Identify Injection Point : The algorithm locates the output of the first retrieval tool call in TU . Let this message be Mretrieval .2. Payload Construction : An adversarial payload is constructed by appending the Attack Instruction (the user prompt from TA) to the legitimate content of Mretrieval .Payload = Content( Mretrieval )⊕Preamble ⊕Prompt( TA)(1) Here, the Preamble is a jailbreak string (e.g., ”Ignore previous instructions”, ”[SYSTEM MES-SAGE]”) selected from a configurable list to vary the stealthiness of the injection. 3. Grafting :

• The simulation preserves the TU history up to the injection point. 21 • The modified Mretrieval (containing the pay-load) is inserted. 

• The subsequent messages from TA (represent-ing the agent’s response to the attack prompt) are appended, replacing the original continu-ation of TU .

13.4.3 Outcome 

The result is a valid conversation history where the agent starts a benign task, retrieves data containing an injection, and immediately ”pivots” to execute the attacker’s command. This method allows us to gen-erate thousands of unique attack variations ( Nutility ×

Nattack × Npreambles ) to thoroughly evaluate the robust-ness of our provenance-based defense.
