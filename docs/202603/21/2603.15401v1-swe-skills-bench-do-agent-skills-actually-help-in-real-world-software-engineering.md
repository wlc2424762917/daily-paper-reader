## 摘要
智能体技能（Agent skills）是在推理时注入的结构化程序性知识包，越来越多地被用于增强大语言模型（LLM）智能体在软件工程任务中的表现。然而，它们在端到端开发环境中的实际效用仍不明确。我们提出了 SWE-Skills-Bench，这是首个需求驱动的基准测试，旨在隔离智能体技能在真实软件工程（SWE）中的边际效用。它将 49 个公开的 SWE 技能与固定提交记录的真实 GitHub 仓库以及具有明确验收标准的需求文档配对，在六个 SWE 子领域中产生了约 565 个任务实例。我们引入了一个确定性验证框架，将每个任务的验收标准映射到基于执行的测试，从而实现了在有无技能情况下的受控配对评估。我们的结果表明，技能注入带来的收益远比其快速普及所暗示的要有限：49 个技能中有 39 个没有带来通过率的提升，平均增益仅为 +1.2%。Token 开销从适度节省到增加 451% 不等，而通过率却保持不变。只有 7 个专业技能产生了显著收益（最高达 +30%），而有 3 个技能由于版本不匹配的指导与项目上下文冲突而导致性能下降（最高达 -10%）。这些发现表明，智能体技能是一种狭窄的干预手段，其效用在很大程度上取决于领域契合度、抽象级别和上下文兼容性。SWE-Skills-Bench 为评估软件工程智能体中技能的设计、选择和部署提供了一个测试平台。SWE-Skills-Bench 可在 https://github.com/GeniusHTX/SWE-Skills-Bench 获取。

## Abstract
Agent skills, structured procedural knowledge packages injected at inference time, are increasingly used to augment LLM agents on software engineering tasks. However, their real utility in end-to-end development settings remains unclear. We present SWE-Skills-Bench, the first requirement-driven benchmark that isolates the marginal utility of agent skills in real-world software engineering (SWE). It pairs 49 public SWE skills with authentic GitHub repositories pinned at fixed commits and requirement documents with explicit acceptance criteria, yielding approximately 565 task instances across six SWE subdomains. We introduce a deterministic verification framework that maps each task's acceptance criteria to execution-based tests, enabling controlled paired evaluation with and without the skill. Our results show that skill injection benefits are far more limited than rapid adoption suggests: 39 of 49 skills yield zero pass-rate improvement, and the average gain is only +1.2%. Token overhead varies from modest savings to a 451% increase while pass rates remain unchanged. Only seven specialized skills produce meaningful gains (up to +30%), while three degrade performance (up to -10%) due to version-mismatched guidance conflicting with project context. These findings suggest that agent skills are a narrow intervention whose utility depends strongly on domain fit, abstraction level, and contextual compatibility. SWE-Skills-Bench provides a testbed for evaluating the design, selection, and deployment of skills in software engineering agents. SWE-Skills-Bench is available at https://github.com/GeniusHTX/SWE-Skills-Bench.

#### 1. 核心问题与整体含义（研究动机和背景）
随着大语言模型（LLM）智能体在软件工程（SWE）领域的广泛应用，一种名为“智能体技能”（Agent Skills）的机制迅速流行。这些技能是结构化的程序性知识包（通常为 Markdown 格式），在推理时注入智能体的上下文窗口，旨在提供特定领域的标准操作程序或代码模板。
**研究动机**：尽管技能包生态发展极快（136天内产生了超过8万个技能），但其在真实、端到端开发任务中的**实际边际效用**尚不明确。现有的基准测试（如 HumanEval 或 SWE-Bench）要么侧重于孤立的代码生成，要么未隔离“技能注入”这一变量。本论文旨在通过构建首个需求驱动的基准测试，回答一个核心问题：**注入这些技能真的能帮助智能体更好地完成软件工程任务吗？**

#### 2. 方法论：核心思想与关键技术
论文提出了 **SWE-Skills-Bench**，其核心思想是**需求驱动的受控实验**。
*   **技能筛选（Stage 1）**：从 8 万多个公开技能中，通过三阶段过滤（类别选择、语义过滤、可行性筛选），选出 49 个涵盖开发者工具、安全测试、API 开发等 6 个领域的 SWE 技能。

*   **需求驱动验证（Stage 3）**：这是该方法论的关键。研究者不依赖主观的“LLM 打分”，而是将需求文档中的验收标准自动转化为**确定性的 pytest 单元测试**。这些测试直接运行智能体生成的代码，验证其功能正确性。
*   **配对评估（Stage 4）**：在 Docker 容器中对比智能体在“有技能注入”和“无技能注入”两种条件下的表现，测量通过率（Pass Rate）的变化。

#### 3. 实验设计
*   **数据集/场景**：包含 49 个真实 SWE 技能，共计约 565 个任务实例。
*   **Benchmark 构成**：每个任务是一个四元组 `(R, E, P, S)`，即：GitHub 仓库、容器环境、需求文档、可选技能文档。
*   **对比方法**：
    *   **Baseline**：仅使用智能体（Claude Code + Claude Haiku 4.5）处理需求。
    *   **Skill-Augmented**：在项目根目录注入 `SKILL.md`，由智能体自主发现并应用。
*   **评估指标**：通过率（Pass Rate）、技能效用增量（$\Delta P$）、Token 开销比例（$\rho$）以及成本效率（CE）。

*   **算力环境**：实验在 Docker 容器（Ubuntu 24.04）中运行，配置为 **CPU-only**，并对每个任务设置了资源限制。
*   **模型使用**：使用了 **Claude Code** 智能体框架，底层模型为 **Claude Haiku 4.5**。
*   **训练时长**：由于该研究属于推理侧的基准测试评估，不涉及模型训练，因此未提及 GPU 训练时长。主要的资源消耗在于调用 LLM API 进行代码生成和运行自动化测试。

#### 5. 实验数量与充分性
*   **实验规模**：共进行了 565 组配对实验（总计 1130 次任务运行）。
*   **覆盖面**：涵盖了 6 个 SWE 子领域，平均每个技能对应约 11 个任务实例。
*   **充分性与公平性**：
    *   **充分性**：实验不仅关注成功率，还深入分析了 Token 消耗和失败案例，提供了定性分析（如 Linkerd 案例）。
    *   **公平性**：通过固定 GitHub 提交记录、使用容器化环境、排除预存技能干扰等手段，确保了实验的可重复性和受控性。

#### 6. 论文的主要结论与发现
1.  **边际收益极低**：39/49 的技能（约 80%）对通过率提升为零。平均通过率仅从 89.8% 提升至 91.0%（+1.2%）。
2.  **Token 开销与性能脱钩**：技能注入会导致 Token 消耗剧增（最高达 451%），但往往并不改变结果。这表明技能改变了推理路径，但不一定提高了正确性。
3.  **特定领域有效**：仅 7 个涉及高度专业化知识（如金融风险公式、云原生流量管理）的技能带来了显著收益（最高 +30%）。
4.  **负面干扰风险**：3 个技能导致了性能下降（最高 -10%）。原因是技能中的陈旧模板或具体参数与当前项目的上下文发生冲突，导致智能体产生幻觉或过度拟合模板。

#### 7. 优点（亮点）
*   **首创性**：填补了 SWE 领域缺乏技能效用评估基准的空白。
*   **确定性验证**：坚持使用基于执行的单元测试（Execution-based），而非不可靠的 LLM 自评，保证了结果的客观性。
*   **真实性**：任务基于真实的 GitHub 仓库和复杂的跨文件操作，而非简单的算法题。
*   **深刻的洞察**：揭示了“技能注入”并非万灵药，并提出了“上下文干扰”这一重要风险点。

*   **模型单一性**：目前仅测试了 Claude Haiku 4.5，不同规模或架构的模型（如 GPT-4o 或 Llama-3）对技能的利用能力可能存在差异。
