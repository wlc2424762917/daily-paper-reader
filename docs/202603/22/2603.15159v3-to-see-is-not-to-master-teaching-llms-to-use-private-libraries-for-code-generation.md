## 摘要
大语言模型（LLMs）在代码生成方面展现出巨大潜力，但在面向私有库的代码生成方面仍存在局限性，其目标是使用私有库中的 API 生成代码。现有方法主要依赖于检索私有库 API 文档，并在推理时将相关知识注入上下文。然而，我们的研究表明这是不够的：即使提供了准确的所需知识，LLMs 仍然难以有效地调用私有库 API。为了解决这一局限性，我们提出了 PriCoder，这是一种通过自动合成数据来教 LLMs 调用私有库 API 的方法。具体而言，PriCoder 将私有库数据合成建模为图构建过程，并在两个图算子之间交替进行：（1）渐进式图演化（Progressive Graph Evolution），通过从基础样本逐步合成更多样化的训练样本来提高数据多样性；（2）多维图剪枝（Multidimensional Graph Pruning），通过严格的过滤流水线提高数据质量。为了支持严格的评估，我们基于测试模型不熟悉的近期发布库构建了两个新的基准测试。在三种主流 LLMs 上的实验表明，PriCoder 显著提升了面向私有库的代码生成能力，在许多设置下 pass@1 的增益超过 20%，同时对通用代码生成能力的影响微乎其微。我们的代码和基准测试已在 https://github.com/eniacode/PriCoder 公开发布。

## Abstract
Large Language Models (LLMs) have shown strong potential for code generation, yet they remain limited in private-library-oriented code generation, where the goal is to generate code using APIs from private libraries. Existing approaches mainly rely on retrieving private-library API documentation and injecting relevant knowledge into the context at inference time. However, our study shows that this is insufficient: even given accurate required knowledge, LLMs still struggle to invoke private-library APIs effectively.   To address this limitation, we propose PriCoder, an approach that teaches LLMs to invoke private-library APIs through automatically synthesized data. Specifically, PriCoder models private-library data synthesis as the construction of a graph, and alternates between two graph operators: (1) Progressive Graph Evolution, which improves data diversity by progressively synthesizing more diverse training samples from basic ones, and (2) Multidimensional Graph Pruning, which improves data quality through a rigorous filtering pipeline. To support rigorous evaluation, we construct two new benchmarks based on recently released libraries that are unfamiliar to the tested models. Experiments on three mainstream LLMs show that PriCoder substantially improves private-library-oriented code generation, yielding gains of over 20% in pass@1 in many settings, while causing negligible impact on general code generation capability. Our code and benchmarks are publicly available at https://github.com/eniacode/PriCoder.

### 1. 核心问题与研究动机
*   **核心问题**：现有的私有库代码生成主要依赖**检索增强生成（RAG）**，即在推理时将 API 文档注入上下文。然而，作者通过实证研究发现“**见之不等于掌握**”：即使提供完美的（Oracle）API 文档，模型在调用 API 时仍频繁出现语法错误、参数误用或逻辑遗漏。
*   **研究动机**：私有库数据在公开语料中极度稀缺，直接合成数据又面临**多样性低**（模型倾向于生成简单调用）和**质量差**（包含大量幻觉 API 或运行错误）的挑战。因此，需要一种能够自动生成高质量、高多样性私有库训练数据的方法，让模型通过微调真正“掌握”API 的使用。

### 2. 方法论（PriCoder 框架）
PriCoder 将私有库数据合成建模为一个**合成图（Synthesis Graph）**的构建过程，通过两个核心算子交替迭代：
*   **渐进式图演化（Progressive Graph Evolution）**：
    *   **初始演化**：从 API 节点出发，随机组合 API 规格说明，引导模型生成基础的编程需求和代码实现。
    *   **迭代演化**：将已生成的多个简单样本节点进行整合，要求模型编写更复杂、涉及多个 API 协同工作的综合性任务，从而提升数据的多样性和复杂度。
*   **多维图剪枝（Multidimensional Graph Pruning）**：
    *   **语法验证**：使用 AST 解析器检查生成的代码是否存在语法错误。
    *   **执行验证**：利用 LLM 自动生成单元测试，并在沙箱环境中运行代码，剔除无法通过测试或产生运行时错误的样本。
    *   **语义功能验证**：采用“LLM-as-a-Judge”模式，评估需求是否合理以及代码是否真正实现了需求意图。
*   **训练与部署**：在清洗后的高质量合成数据集上对模型进行微调（如使用 LoRA）。该方法与 RAG 正交，微调后的模型可结合 RAG 进一步提升性能。

### 3. 实验设计
*   **数据集/基准测试**：
    *   **NdonnxEval**（169 个实例）和 **NumbaEval**（187 个实例）：作者基于 2024-2025 年发布的 `ndonnx` 和 `numba-cuda` 库构建。由于这些库发布时间晚于主流模型的知识截止日期，能有效模拟“私有库”场景。
    *   **HumanEval**：用于评估模型在学习私有库后，通用代码生成能力是否退化。
*   **对比方法**：
    *   **Vanilla**（原始模型直接生成）。
    *   **RAG 变体**：Naive RAG、EpiGen（任务分解）、CAPIR（API 重排序）。
    *   **Oracle**：提供解决该任务所需的全部准确 API 信息作为上限参考。
*   **测试模型**：DeepSeek-Coder-6.7B、Qwen2.5-Coder-7B、LLaMA3.1-8B。

### 4. 资源与算力
*   **硬件环境**：使用了一台配备 **8 张 NVIDIA A100 (80GB)** 显存 GPU 的服务器。
*   **训练时长**：对于每个目标模型，完成数据合成与微调的总耗时**不超过 50 小时**。
*   **数据规模**：针对 `numba-cuda` 合成了 6K 样本，针对 `ndonnx` 合成了 20K 样本。

### 5. 实验充分性、客观性与公平性
*   **实验充分性**：作者在 3 个主流模型和 2 个全新基准上进行了测试，并包含了详尽的**消融实验**（验证图演化和剪枝的必要性）、**数据规模分析**以及**跨模型合成分析**。
*   **客观性与公平性**：
    *   使用了**无偏 pass@k** 和 **exec@k**（执行成功率）作为指标。
    *   所有方法均采用相同的解码参数（Temperature=0.5 等）。
    *   通过构建全新的、模型未见过的库基准，避免了数据泄露导致的评估偏差。

### 6. 主要结论与发现
*   **性能飞跃**：PriCoder 显著优于所有 RAG 基准，在多个设置下将 **pass@1 提升了 20% 以上**。
*   **协同效应**：PriCoder 微调后的模型再结合 RAG，性能通常能达到最优，证明了“内在掌握”与“外在参考”的互补性。
*   **通用能力保持**：在 HumanEval 上的测试显示，PriCoder 几乎**不损害模型的通用编程能力**，甚至在某些情况下因强化了语法规范而略有提升。
*   **鲁棒性**：即使使用较小规模的合成数据（如 1000 条），也能获得显著增益；且使用不同模型合成的数据对微调效果影响较小。

### 7. 优点与亮点
*   **理念新颖**：明确指出了 RAG 在私有库场景下的“天花板”问题，强调了微调在 API 调用技能习得中的重要性。
*   **质量控制严苛**：通过语法、执行、语义三重剪枝，解决了合成数据中常见的“幻觉”和“噪声”问题。
*   **基准贡献**：开源了两个高质量、针对最新库的基准测试，填补了私有库代码生成评估工具的空白。

### 8. 不足与局限
*   **微调成本**：虽然 50 小时在企业级应用中可接受，但相比纯 RAG 方法，PriCoder 仍需额外的计算资源进行模型训练。
*   **依赖初始文档**：数据合成的质量高度依赖于私有库初始 API 文档的清晰度，如果原始文档极度模糊，合成效果可能会受限。
*   **库的覆盖范围**：实验主要集中在 Python 库，对于其他编程语言或极其庞大的企业级单体仓库（Monorepo）的适用性尚待进一步验证。
