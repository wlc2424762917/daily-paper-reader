## 摘要
尽管大语言模型（LLMs）发展迅速，但复杂软件工作流的鲁棒自动化仍是一个悬而未决的问题。在长程（long-horizon）场景中，智能体经常受到级联错误和环境随机性的困扰；动态界面中的一次失误就可能导致任务失败，进而引发幻觉或陷入反复试错。本文介绍了“环境地图”（Environment Maps）：一种持久的、与智能体无关的表示方式，通过将屏幕录像和执行轨迹等异构证据整合到结构化图中，来缓解这些失败。该表示由四个核心组件组成：(1) 上下文（抽象位置），(2) 动作（参数化可供性），(3) 工作流（观测到的轨迹），以及 (4) 隐性知识（领域定义和可重用程序）。我们在 WebArena 基准测试的五个领域中评估了该框架。配备环境地图的智能体实现了 28.2% 的成功率，几乎是受限于会话上下文的基线模型（14.2%）的两倍，且优于能够访问用于生成环境地图的原始轨迹数据的智能体（23.3%）。通过在模型和环境之间提供结构化接口，环境地图为长程规划奠定了持久的基础，且具有可解释性、可编辑性和增量完善性。

## Abstract
Although large language models (LLMs) have advanced rapidly, robust automation of complex software workflows remains an open problem. In long-horizon settings, agents frequently suffer from cascading errors and environmental stochasticity; a single misstep in a dynamic interface can lead to task failure, resulting in hallucinations or trial-and-error. This paper introduces $\textit{Environment Maps}$: a persistent, agent-agnostic representation that mitigates these failures by consolidating heterogeneous evidence, such as screen recordings and execution traces, into a structured graph. The representation consists of four core components: (1) Contexts (abstracted locations), (2) Actions (parameterized affordances), (3) Workflows (observed trajectories), and (4) Tacit Knowledge (domain definitions and reusable procedures). We evaluate this framework on the WebArena benchmark across five domains. Agents equipped with environment maps achieve a 28.2% success rate, nearly doubling the performance of baselines limited to session-bound context (14.2%) and outperforming agents that have access to the raw trajectory data used to generate the environment maps (23.3%). By providing a structured interface between the model and the environment, Environment Maps establish a persistent foundation for long-horizon planning that is human-interpretable, editable, and incrementally refinable.

### 2. 方法论：环境地图（Environment Maps）
论文提出了一种名为 **Environment Maps (M)** 的持久化、与智能体无关的结构化表示，其核心由四个组件构成：$M = (C, A, W, K)$。
*   **核心组件**：
    1.  **Contexts (C)**：抽象位置（如特定的页面或视图），通过 URL 模式识别。
    2.  **Actions (A)**：参数化的可供性（如将“点击 John”抽象为“点击 {用户名}”）。
    3.  **Workflows (W)**：观测到的成功执行轨迹序列。
    4.  **Tacit Knowledge (K)**：隐性知识，包括领域定义（术语表）和过程逻辑（操作指南）。
*   **构建流程（5阶段流水线）**：
    *   **阶段 1：统一步骤序列**。解析原始观测（屏幕录像、轨迹）为结构化步骤。
    *   **阶段 2：动作提取**。从轨迹中提取已执行动作，并从 DOM 中识别潜在动作。
    *   **阶段 3：动作泛化**。利用 LLM 将具体动作抽象为带占位符的参数化模板。
    *   **阶段 4：上下文与知识提取**。通过 URL 归一化对页面进行聚类，并提取领域定义和操作规程。
    *   **阶段 5：整合与合并**。将上述信息合并为统一的 JSON 格式图结构。

### 3. 实验设计
*   **数据集/场景**：使用 **WebArena** 基准测试，涵盖 5 个真实交互环境：电子商务（Magento）、管理后台（CMS）、代码托管（GitLab）、论坛（Reddit）和地图（OpenStreetMap）。
*   **评估基准**：采用 **WebArena Verified** 框架，通过 HTTP 存档（HAR）检查进行严格、可重复的评估。
*   **对比方法**：

    2.  **Trajectory Access**：允许智能体访问原始的人类执行轨迹文件（JSONL 格式）。
    3.  **Environment Map Access**：智能体可访问基于相同轨迹数据生成的结构化环境地图。

*   **实验规模**：在 5 个领域共进行了 **812 个任务** 的评估，样本量充足。

### 6. 主要结论与发现
*   **性能提升**：配备环境地图的智能体成功率达到 **28.2%**，远高于基线（14.2%）和原始轨迹访问组（23.3%）。
*   **结构的重要性**：即使数据源相同，将原始轨迹转化为结构化地图也能显著提升性能，尤其是在 GitLab 和 CMS 等高分支、复杂的 UI 环境中。
*   **泛化能力**：环境地图不仅在有演示的任务上表现好，在从未见过的任务上也表现出更强的泛化能力（从 12.6% 提升至 26.9%）。
*   **交互模式**：使用地图的智能体表现出更主动的查询行为（平均每任务 12 次工具调用），而原始轨迹组几乎不调用查询工具。

### 7. 优点
*   **智能体无关性**：地图是持久的 JSON 构件，可以跨模型（GPT、Claude 等）重用，无需重新训练。
*   **可解释与可编辑**：人类可以直接阅读和修改地图内容，方便纠错和维护。
*   **低成本维护**：相比于昂贵的模型微调，通过更新地图来应对 UI 变更更加高效。
*   **结构化优势**：成功将空间（上下文）、动作（参数化）和语义（知识）整合在一起。

### 8. 不足与局限
*   **环境复杂度的依赖**：在简单、导航路径浅的环境（如 Reddit）中，结构化地图的优势不如在复杂环境中明显。
*   **时效性风险**：随着 Web 环境的频繁更新，地图可能会过时，目前尚缺乏完全自动化的实时更新机制。
*   **跨站点限制**：目前的地图主要针对单一环境，对于跨多个站点的复杂工作流支持尚不完善。
*   **状态验证**：目前的动作前置条件和效果主要是描述性的，尚未实现基于实时状态观测的硬性校验。
