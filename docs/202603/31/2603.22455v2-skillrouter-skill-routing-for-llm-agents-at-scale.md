## 摘要
可重用技能使大语言模型（LLM）智能体能够将特定任务的流程、工具功能（affordances）和执行指南封装成模块化的构建块。随着技能生态系统增长到数万个条目，在推理时展示每个技能变得不可行。这产生了一个技能路由问题：给定一个用户任务，系统必须在下游规划或执行之前识别出相关的技能。现有的智能体技术栈通常依赖于渐进式披露，仅展示技能名称和描述，而隐藏完整的实现主体。我们在基于 SkillsBench 衍生的基准测试上研究了这一设计选择，该基准包含约 8 万个候选技能，针对具有严重重叠的大型技能注册库这一具有实际重要性的场景。在该场景下的代表性稀疏、密集和重排序基准模型中，隐藏技能主体会导致路由准确率下降 31 至 44 个百分点，这表明在这一设定下，完整的技能文本是关键的路由信号，而非次要的元数据改进。受此发现启发，我们提出了 SkillRouter，这是一个紧凑的 1.2B 参数全文本检索与重排序流水线。SkillRouter 在我们的基准测试中实现了 74.0% 的 Hit@1——这是我们评估的基准模型中表现最强的平均 top-1 路由性能——同时其参数量减少了 13 倍，运行速度比最强的基础流水线快 5.8 倍。在针对四个编程智能体的补充性端到端研究中，路由性能的提升转化为了任务成功率的提高，且对于能力更强的智能体，提升效果更为显著。

## Abstract
Reusable skills let LLM agents package task-specific procedures, tool affordances, and execution guidance into modular building blocks. As skill ecosystems grow to tens of thousands of entries, exposing every skill at inference time becomes infeasible. This creates a skill-routing problem: given a user task, the system must identify relevant skills before downstream planning or execution. Existing agent stacks often rely on progressive disclosure, exposing only skill names and descriptions while hiding the full implementation body. We examine this design choice on a SkillsBench-derived benchmark with approximately 80K candidate skills, targeting the practically important setting of large skill registries with heavy overlap. Across representative sparse, dense, and reranking baselines on this setting, hiding the skill body causes a 31--44 percentage point drop in routing accuracy, showing that full skill text is a critical routing signal in this setting rather than a minor metadata refinement. Motivated by this finding, we present SkillRouter, a compact 1.2B full-text retrieve-and-rerank pipeline. SkillRouter achieves 74.0% Hit@1 on our benchmark -- the strongest average top-1 routing performance among the baselines we evaluate -- while using 13$\times$ fewer parameters and running 5.8$\times$ faster than the strongest base pipeline. In a complementary end-to-end study across four coding agents, routing gains transfer to improved task success, with larger gains for more capable agents.

### SkillRouter: 大规模 LLM 智能体技能路由论文总结

#### 1. 核心问题与整体含义（研究动机和背景）
随着 LLM 智能体（Agents）生态的发展，开发者将复杂的工具调用和执行逻辑封装为“技能（Skills）”。当技能库规模达到数万个且功能高度重叠时，如何从海量库中精准找到匹配任务的技能（即“技能路由”问题）成为系统瓶颈。
*   **研究动机**：现有框架通常只向路由系统暴露技能的名称和描述（元数据），而隐藏具体的实现代码（主体）。作者发现，在大规模且高度重叠的技能库中，这种“隐藏主体”的做法会导致路由准确率大幅下降。
*   **核心意义**：论文证明了**完整的技能实现文本是关键的路由信号**，并提出了一个轻量且高效的解决方案。

#### 2. 方法论：核心思想与关键技术
作者提出了 **SkillRouter**，一个包含 1.2B 参数的“检索+重排序（Retrieve-and-Rerank）”全文本流水线。
*   **核心思想**：在检索和重排序阶段均引入完整的技能实现代码（Full-text），而非仅依赖元数据。
*   **关键技术细节**：
    *   **双阶段架构**：由一个 0.6B 的双编码器（Bi-encoder）进行初步检索，再由一个 0.6B 的交叉编码器（Cross-encoder）进行精细重排序。
    *   **负样本过滤（False-negative Filtering）**：针对技能库中存在大量功能相似但名称不同的“近重复”技能，通过名称去重、文本重叠度（Jaccard 相似度）和嵌入相似度三层过滤，防止错误的负样本干扰对比学习。
    *   **列表式重排序损失（Listwise Reranking Loss）**：采用列表式交叉熵损失而非点对点分类，使模型能在前 20 个候选技能中进行细微的功能区分。
    *   **合成数据增强**：使用 GPT-4o-mini 基于技能主体生成 3.8 万个合成查询对，确保训练集与评估集技能完全解耦。

#### 3. 实验设计
*   **数据集与基准（Benchmark）**：
    *   基于 **SkillsBench** 衍生，包含约 **80,000 个候选技能**。
    *   **75 个核心查询**（24 个单技能任务，51 个多技能任务），涵盖 55 个应用领域。
    *   **Easy/Hard 分级**：Hard 级别额外加入了 780 个由 LLM 生成的“功能相近但逻辑错误”的干扰项（Distractors）。
*   **对比方法**：
    *   **稀疏检索**：BM25。
    *   **密集检索**：BGE-Large、Gemini/OpenAI 嵌入模型、Qwen3-Emb (0.6B/8B)。
    *   **重排序/LLM 评判**：Qwen3-Rank、GPT-4o-mini、GPT-5.4-mini。
*   **端到端验证**：在自然技能库下，测试了 Kimi-K2.5、GLM-5、Claude 3.5 Sonnet 和 Claude 3 Opus 四个编程智能体的任务成功率。

#### 4. 资源与算力
*   **硬件**：模型训练在单台 **NVIDIA H20 GPU (96GB)** 上完成。
*   **训练配置**：采用 AdamW 优化器，1 个 Epoch 训练，混合精度（BF16），使用了梯度检查点（Gradient Checkpointing）。
*   **推理效率**：在真实 80k 技能库上，1.2B 流水线的的中值延迟为 **495.8 ms**，比 16B 的基准流水线快 5.8 倍。

#### 5. 实验数量与充分性
*   **实验规模**：进行了多维度的对比，包括不同参数规模（0.6B 到 16B）、不同输入格式（仅元数据 vs 全文本）、不同损失函数（Pointwise vs Listwise）的消融实验。
*   **充分性与公平性**：
    *   通过**长度控制的注意力分析**证明了性能提升并非单纯因为文本变长，而是模型确实捕捉到了代码中的逻辑。
    *   通过**描述质量分层实验**排除了“描述写得烂”这一干扰因素。
    *   实验设计涵盖了从检索指标（Hit@1, Recall@K）到端到端智能体执行成功率的完整链路，评估较为客观。

#### 6. 主要结论与发现
*   **主体文本至关重要**：移除技能实现主体会导致路由准确率下降 **31-44 个百分点**。
*   **小模型胜过大模型**：经过特定任务微调的 1.2B SkillRouter 在 Hit@1 指标上（74.0%）超过了参数量大 13 倍的 16B 基础模型（68.0%）。
*   **路由增益可转化**：路由准确性的提升能直接显著提高下游智能体的任务成功率，且智能体本身能力越强，从优质路由中获益越多。
*   **训练策略的影响**：负样本过滤贡献了 +4.0pp 的提升，而列表式训练对重排序至关重要（比点对点训练高出 30.7pp）。

#### 7. 优点
*   **实用性强**：针对工业级大规模技能库的真实痛点（重叠度高、规模大）。
*   **效率极高**：在保持高精度的同时，极大地降低了推理延迟和显存占用，适合实时部署。
*   **深入的诊断分析**：不仅给出了结果，还通过注意力机制可视化解释了模型为何需要阅读代码主体。

#### 8. 不足与局限
*   **查询样本量较小**：核心测试集仅包含 75 个查询，虽然覆盖领域广，但统计样本量仍有提升空间。
*   **多跳推理限制**：案例分析显示，对于需要复杂逻辑推理（如：任务 A 需要技能 B 提取数据，再由技能 C 处理）的场景，目前的检索机制仍难以完全覆盖。
*   **领域局限**：实验主要集中在编程和技术类技能，对于非技术类、缺乏代码实现的技能路由效果尚未验证。
