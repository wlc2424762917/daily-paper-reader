## 摘要
在企业环境中，大型遗留系统的现代化仍是一项重大挑战，特别是当迁移过程必须在保留特定领域逻辑的同时，符合内部架构框架和共享 API 时。直接应用大语言模型（LLMs）进行代码翻译通常会产生语法正确但无法编译或无法集成到现有生产框架中的输出，这限制了它们在现实世界现代化工作中的实际应用。在本文中，我们提出了 LegacyTranslate，这是一个用于 API 感知代码翻译的多智能体框架，该框架是在一家金融机构将约 250 万行 PL/SQL 迁移到 Java 的持续现代化背景下开发和评估的。其核心思想是使用基于 LLM 的专门智能体，每个智能体负责处理翻译挑战的不同方面。具体而言，LegacyTranslate 由三个智能体组成：初始翻译智能体（Initial Translation Agent）利用检索到的上下文示例生成初始 Java 翻译；API 落地智能体（API Grounding Agent）通过从 API 知识库中检索相关条目，使代码与现有 API 保持一致；细化智能体（Refinement Agent）利用编译器反馈和 API 建议迭代地优化输出，以提高正确性。我们的实验表明，每个智能体都有助于提高翻译质量。仅初始翻译智能体就实现了 45.6% 的可编译输出和 30.9% 的测试通过率。结合 API 落地智能体和细化智能体，编译率额外提高了 8%，测试通过准确率提高了 3%。

## Abstract
Modernizing large legacy systems remains a major challenge in enterprise environments, particularly when migration must preserve domain-specific logic while conforming to internal architectural frameworks and shared APIs. Direct application of Large Language Models (LLMs) for code translation often produces syntactically valid outputs that fail to compile or integrate within existing production frameworks, limiting their practical adoption in real-world modernization efforts. In this paper, we propose LegacyTranslate, a multi-agent framework for API-aware code translation, developed and evaluated in the context of an ongoing modernization effort at a financial institution migrating approximately 2.5 million lines of PL/SQL to Java. The core idea is to use specialized LLM-based agents, each addressing a different aspect of the translation challenge. Specifically, LegacyTranslate consists of three agents: Initial Translation Agent produces an initial Java translation using retrieved in-context examples; API Grounding Agent aligns the code with existing APIs by retrieving relevant entries from an API knowledge base; and Refinement Agent iteratively refines the output using compiler feedback and API suggestions to improve correctness. Our experiments show that each agent contributes to better translation quality. The Initial Translation Agent alone achieves 45.6% compilable outputs and 30.9% test-pass rate. With API Grounding Agent and Refinement Agent, compilation improves by an additional 8% and test-pass accuracy increases by 3%.

#### 1. 核心问题与研究动机
*   **核心问题**：企业级遗留系统（如 PL/SQL、COBOL）的现代化迁移面临巨大挑战。虽然大语言模型（LLM）在通用代码翻译上表现出色，但在处理企业内部特定架构、共享库和私有 API 时，生成的代码往往因无法适配内部框架而导致编译失败或功能错误。
*   **研究背景**：本研究基于一家欧洲金融机构的真实案例，该机构拥有约 250 万行 PL/SQL 代码，正致力于将其迁移到基于 Java 的现代架构。直接使用 LLM 翻译的代码在生产环境中几乎无法编译，因为它们缺乏对企业内部 API 的感知。

#### 2. 方法论
LegacyTranslate 提出了一个包含三个专门智能体的多智能体协作框架：
*   **初始翻译智能体 (Initial Translation Agent)**：
    *   **核心思想**：利用检索增强生成（RAG）提供上下文示例。
    *   **流程**：从已有的（PL/SQL, Java）对参考集中，检索与目标 PL/SQL 语义最相似的 $k$ 个示例，作为 Few-shot 提示词引导 LLM 生成初始 Java 草稿。
*   **API 落地智能体 (API Grounding Agent)**：
    *   **核心思想**：解决代码与内部 API 的对齐问题。
    *   **流程**：基于初始代码和编译器报错信息，从预先构建的“API 知识库”（包含类名、签名、描述等）中筛选出最相关的 API 列表。
*   **细化智能体 (Refinement Agent)**：
    *   **核心思想**：基于反馈的迭代优化。
    *   **流程**：将初始代码、编译器错误信息和筛选出的 API 列表输入 LLM，进行迭代修正。该过程持续到代码编译成功、通过测试或达到终止条件。

#### 3. 实验设计
*   **数据集**：使用从金融机构生产环境中提取的 68 组经过人工验证的 PL/SQL 到 Java 的翻译对。PL/SQL 代码平均长度为 101.7 行。
*   **评估指标**：结构有效性（SV）、编译率（CR）、测试通过率（TPR）。
*   **对比方法（Baselines）**：

    2.  **LLM+Random Examples**：随机选取 3 个示例进行 Few-shot 翻译。
    3.  **LegacyTranslate (w/o API & Refine)**：仅使用初始翻译智能体（消融实验）。

#### 6. 主要结论与发现
*   **API 感知是关键**：在没有 API 引导的情况下，LLM 生成的代码编译率为 0%。引入 API 落地和细化智能体后，编译率提升至 52.9%，测试通过率提升至 33.8%。
*   **多智能体协同效应**：初始翻译智能体贡献了基础能力（45.6% 编译率），而后续智能体通过迭代修正进一步提升了 8% 的编译率和 3% 的测试通过率。

*   **检索质量影响结果**：高维度的密集检索器（如 SFR-Mistral）能提供更相关的示例，直接提升翻译的准确性。
