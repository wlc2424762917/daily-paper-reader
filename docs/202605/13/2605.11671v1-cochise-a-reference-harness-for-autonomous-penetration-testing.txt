## Abstract 

Recent work on LLM-driven autonomous penetration testing re-ports promising results, but existing systems often combine many architectural, prompting, and tool-integration choices, making it dif-ficult to tell what is gained over a simple agent scaffold. We present cochise, a 597 LOC Python reference harness for autonomous penetration-testing experiments. Cochise connects an LLM-driven agent to a Linux execution host over SSH and supports controlled target environments reachable from that jump host. The prototype implements a separated Planner–Executor archi-tecture in which long-term state is maintained outside the LLM context, while a ReAct-style executor issues commands over SSH and self-corrects based on command outputs. The scenario prompt can be adapted to different target environments. To demonstrate the efficacy of our minimal harness, we evaluate it against a live third-party testbed called Game of Active Directory (GOAD). Alongside the harness, we release replay and analysis tools: (i) cochise-replay for offline visualization of captured runs, (ii) cochise-analyze-logs and cochise-analyze-graphs for cost, token, duration, and compromise analysis, and (iii) a corpus of JSON trajectory logs from GOAD runs, allowing researchers to study agent behavior without provisioning the 48—64 GB RAM / 190 GB storage testbed themselves. Cochise is intended not as a state-of-the-art pen-testing agent, but as reusable experimental infrastructure for comparing models, agent architectures, and penetration-testing traces. Tool demo video available at https://youtu.be/2mQimB1ufyI.

## 1 Introduction 

There has been large academic and industrial interest in using LLMs for autonomous network penetration testing. While publica-tions provide performance numbers for their respective prototypes, reusable execution and logging infrastructure is typically missing. This makes it difficult to compare models and architectures under a common protocol or to reuse traces and analysis pipelines across studies. Cochise is intended as a reference harness 1 rather than as a high-performance pentesting agent: it fixes the execution inter-face, model abstraction, state handling, and trajectory format so that researchers can compare models, architectural variants, and

## 2 Motivation, Background And Related Work 

There has been high academic interest in using LLMs for autonomous penetration testing, with current prototypes including incalmo [ 5], cAI [ 3], and pentestGPT 2 [1 ]. These prototypes commonly focus on increasing their respective success rates, provide multiple operation modes, and are subject to constant tinkering. A prior version of our research [ 2 ] was used by PentestGPT as a point of comparison, indicating demand for reusable (baseline) scaffolds in this space. This inspired us to write cochise , a docu-mented reference/baseline harness with explicit execution, logging, replay, and analysis interfaces. 

Penetration Testing as an SE Problem. While our target domain is security-centric, the problem an autonomous penetration-testing agent must solve originates in the software engineering domain.

Realistic security evaluation testbeds are highly dynamic stateful environments where actions can trigger destructive side-effects. Any interaction of the agent with the environment may produce adverse effects, e.g., invoking an exploit can crash the target system, enumeration actions can trigger intrusion detection systems, or running password-based attacks can result in account lock-outs. In addition, defenses such as automated EDR solutions act as active adversaries, creating attacker/defender setups. To achieve success, agents must cope with these side-effects and must combine findings into successful attack paths, making our trajectories inherently multi-step. As the agent can run arbitrary commands on the execution host (Section 3.1), the action space is unbound. In addition, penetration-testing tools are fickle and often complex to use, mandating robust agent error handling and auto-repair functionalities. These properties—partial observability, side-effecting actions, an unbounded action space, and the need for autonomous error recovery—are characteristic of a broader class of agentic SE tasks [ 6 ], and motivate studying autonomous pentesting as an instance of agentic software engineering rather than as security tooling alone. 

Related Harnesses. Architecturally, cochise sits between mini-malist agent scaffolds such as mini-SWE-agent [ 6 ] and full-feature penetration-testing agents such as cAI [ 3 ] and incalmo [ 5]. The clos-est minimalist comparison, mini-SWE-agent, lacks long-horizon trajectory management. In pilot runs against our target, GOAD, it could perform initial exploitation but did not progress to multi-stage compromise. 

## 3 Architecture 

Cochise assumes an assumed breach execution-host setting: the agent begins with SSH access to a Linux host that has network reachability to the target environment and contains the tools re-quired for the experiment. In our evaluation, this host is a Kali Linux VM connected to GOAD, and the agent’s task is to enumerate and compromise Active Directory domains. The harness itself only as-sumes command execution over SSH and structured observation of command outputs. Cochise does not address initial access (phishing, exploit-driven remote code execution, physical access), and we do not model human-in-the-loop adversaries. The execution host is trusted; the testbed network is the only attack surface. 

## 3.1 System Overview 

The overall displayed in Figure 1 consists of the cochise prototype that conceptually connects a cloud- or locally-provided LLM to a third-party penetration-testing testbed, e.g., GOAD. To achieve this, a separate virtual machine is added to the testbed as an execution host. Cochise connects to this execution host over the standard SSH protocol and uses this connection to execute commands, typically to perform attacks within the target environment. We deliberately keep the cochise source code on a separate host from the testbed and connect to the testbed only via SSH. This protects against two specific failure modes: (i) accidental destruction of the agent’s own configuration or logs by the agent itself, and (ii) lateral movement back from a compromised target host to the agent. The agent can execute arbitrary commands on the execution host but cannot modify its own source or logging configuration. In

## 3.2 Agent Architecture 

The architecture comprises of a high-level planner component maintaining long-term state and a low-level short-term executor 

component only using ephemeral periodic state. Both planner and 

executor can submit findings as well as compromised accounts to the knowledge component. The planner module operates as the strategic core of the pro-totype. The planner maintains long-term state outside the LLM context in a structured textual task representation. In our default configuration this is a tree-structured Markdown task list, but the harness does not depend on this exact format. The Planner relies on a continuously updated document known as the Pentest-Task-Tree (PTT) [ 1] as well as the data stored within the knowledge 

component for long-term planning. During each strategy round, the planner receives the current PTT, a summary of the most re-cently executed task, and current world-view from the knowledge 

component. Using this information, it selects the next task to be executed by a per-task instantiated executor . After the executor 

has finished, a new strategy round begins. The planner can submit new information to the knowledge component as well as can revise the PTT autonomously. The executor module functions as a ReAct (Reasoning and Act-ing) agent [ 7] responsible for translating the planner ’s high-level directives into concrete operational commands. To achieve this, it connects to the execution host placed within the test environment via SSH and issues multiple commands designated to solve the task given by the planner . After it believes the task to be achieved, or a maximum number of interaction rounds has been reached, a summary of its invocation is returned to the planner and the 

executor ’s state is discarded, making its memory ephemeral and episodic. A new executor instance is created per task and discarded Cochise: A Reference Harness for Autonomous Penetration Testing

This is a deliberate choice: it bounds the execu-tor’s context window cost (which would otherwise grow across hours of operation), it does not reset the execution host or undo side effects in the target environment, and instead isolates LLM-context-level failures such as malformed local reasoning traces, repeated invalid command attempts, stale short-term observations, and task-specific prompt drift from later executor invocations. It also forces the planner to be the integration point for cross-task knowledge. Both planner and executor implement the Reflexion [ 4] pattern to detect and repair invalid command invocations. 

## 3.3 Logging and Analysis 

Every interaction between the Planner, the Executor, the LLM APIs, and the target network is meticulously captured and serialized into a JSON format per-run log file. The structure of this logging mecha-nism is explicitly designed for downstream data analysis and repro-ducibility. For instance, an LLM invocation is logged with an event key indicating the specific architectural action, such as updating the high-level strategy. This entry contains sub-dictionaries detailing the exact prompt submitted to the model, the raw text completion received, and highly granular cost tracking metrics, including input tokens, output tokens, reasoning tokens, and cached tokens. System commands are logged with separate event identifiers, capturing the exact bash string executed on the remote machine alongside the resulting standard output and standard error streams. The strategic value of this JSON logging format lies in its ability to reduce onramp costs for LLM security research. The live eval-uation environment utilized in our evaluation (GOAD) requires a massive hardware footprint. Specifically, deploying GOAD necessi-tates a system capable of running five concurrent Windows Server virtual machines alongside a Kali Linux attacker node, requiring a minimum of 48GB to 64GB of RAM and roughly 190GB of storage. This resource requirement inherently excludes many academic re-searchers and students from independently verifying or building upon the work. To solve this accessibility barrier, the cochise artifact includes a suite of custom analysis utilities, notably the Python scripts de-signed to parse the JSON logs and a dedicated replay tool. These utilities allow researchers to parse the JSON traces, reconstruct the evolution of the high-level plan, visualize the exact command execution flow, and extract precise cost and token metrics with-out ever provisioning the heavy GOAD virtual environment. The provision of these rich datasets, capturing thousands of complex interactions between state-of-the-art LLMs and a live penetration-testing testbed, provides a high-fidelity corpus of agentic behavior in adversarial environments, ready for immediate analysis and in-tegration into secondary research projects. ## 4 Evaluation 

We evaluate Cochise as a reusable reference harness along three dimensions: compactness, capability, and analyzability. Compact-ness captures whether the scaffold is small enough to inspect and modify. Capability captures whether it can produce meaningful long-horizon trajectories in a live testbed. And analyzability cap-tures whether generated runs can be replayed and converted into reusable empirical data. 

Prototype Complexity. Lines of code are not a measure of agent quality; we use them only as a coarse proxy for inspectability and modification cost. The results highlighted in Ta-ble 1 show that cochise’s 597 lines-of-code compare very favorable to other prototypes, typically being smaller by a factor of 3–13. claude-code , whichs’ code-base we have not included in our eval-uation. In addition, this currently prevents usage of pentestGPT with non-Anthropic models. 

Prototype Performance. In addition to the results presented in [ 2], we preformed a capability evaluation using GOAD. We high-light results of two current frontier LLMs, Google’s Gemini-3-Flash (preview) and Anthropic’s Claude-4.7-Opus , in Table 2. Across all runs, both Opus and Gemini-Flash compromised at least one of the three Active Directory domains within GOAD; in 3/5 runs, both models were able to compromise all three Active Directory domains with average costs of Gemini-Flash being 12x lower. ## 5 Ethics 

Any offensive security research is inherently dual-use research of concern (DURC). In our own line of research, we found the massive improvement in cybersecurity capabilities in off-the-shelf LLMs between our original research [ 2] and this publication especially concerning. We release Cochise to improve the reproducibility and compara-bility of research on autonomous penetration-testing agents, not to provide an operational red-team assistant. As we have created a minimal reference harness, this highlights not security issues cre-ated by our prototype, but by the underlying LLMs, whose providers are increasingly choosing structured access as protective measure. While the readability of our prototype also helps malicious adver-saries, these parties typically use off-the-shelf offensive tooling instead of creating bespoke tools, thus our work should primarily benefit researchers and defenders.
