## 摘要
基于大语言模型（LLM）的智能体越来越多地被部署用于处理流式任务，但它们通常仍是单次问题解决者，无法从过去的交互中学习。从经验中提炼的可重用技能为自进化提供了天然基础，而高质量的技能策展（skill curation）则是其中的关键瓶颈。现有方法要么依赖人工技能策展，要么规定启发式技能操作，或者针对短时程技能操作进行训练。然而，它们仍然难以从间接且延迟的反馈中学习复杂的长期策展策略。为了应对这一挑战，我们提出了 SkillOS，这是一种经验驱动的强化学习（RL）训练方案，用于在自进化智能体中学习技能策展。SkillOS 将一个负责检索和应用技能的冻结智能体执行器与一个可训练的技能策展器配对，后者根据积累的经验更新外部技能库（SkillRepo）。为了提供策展的学习信号，我们设计了复合奖励，并基于技能相关的任务依赖关系在分组任务流上进行训练，其中早期的轨迹用于更新 SkillRepo，而随后的相关任务则用于评估这些更新。在多轮智能体任务和单轮推理任务中，SkillOS 在有效性和效率方面均始终优于无记忆和强记忆基线，且学习到的技能策展器能够泛化到不同的执行器骨干网络和任务领域。进一步的分析表明，学习到的策展器能产生更具针对性的技能使用，而 SkillRepo 中的技能随着时间的推移演变成结构更丰富的 Markdown 文件，编码了更高层级的元技能。

## Abstract
LLM-based agents are increasingly deployed to handle streaming tasks, yet they often remain one-off problem solvers that fail to learn from past interactions. Reusable skills distilled from experience provide a natural substrate for self-evolution, where high-quality skill curation serves as the key bottleneck. Existing approaches either rely on manual skill curation, prescribe heuristic skill operations, or train for short-horizon skill operations. However, they still struggle to learn complex long-term curation policies from indirect and delayed feedback. To tackle this challenge, we propose SkillOS, an experience-driven RL training recipe for learning skill curation in self-evolving agents. SkillOS pairs a frozen agent executor that retrieves and applies skills with a trainable skill curator that updates an external SkillRepo from accumulated experience. To provide learning signals for curation, we design composite rewards and train on grouped task streams based on skill-relevant task dependencies, where earlier trajectories update the SkillRepo, and later related tasks evaluate these updates. Across multi-turn agentic tasks and single-turn reasoning tasks, SkillOS consistently outperforms memory-free and strong memory-based baselines in both effectiveness and efficiency, with the learned skill curator generalizing across different executor backbones and task domains. Further analyses show that the learned curator produces more targeted skill use, while the skills in SkillRepo evolve into more richly structured Markdown files that encode higher-level meta-skills over time.

### 论文总结：SkillOS - 面向自进化智能体的技能策展学习

#### 1. 核心问题与整体含义
当前的 LLM 智能体大多是“一次性”的问题解决者，难以从过去的交互经验中持续学习。虽然“技能（Skills）”被认为是实现智能体自进化的核心，但如何从原始轨迹中提取、更新和删除高质量技能（即**技能策展**）仍是主要瓶颈。现有方法多依赖人工编写或固定的启发式规则，缺乏根据下游任务反馈进行长期优化的能力。本文提出的 **SkillOS** 旨在通过强化学习（RL）训练一个专门的“技能策展器”，使智能体能够根据执行经验自主管理技能库，实现真正的自我进化。

#### 2. 方法论
SkillOS 采用模块化设计，将系统分为两个部分：
*   **冻结的执行器（Agent Executor）：** 负责从技能库（SkillRepo）检索技能并执行任务。
*   **可训练的策展器（Skill Curator）：** 负责观察执行轨迹，通过函数调用（插入、更新、删除）来管理以 Markdown 格式存储的技能。

**核心技术细节：**
*   **分组任务流训练（Grouped Task Streams）：** 为了解决策展反馈延迟的问题，研究者将具有技能依赖关系的相似任务编组。策展器在处理组内早期任务时更新技能库，其效果由组内后续任务的成功率来评估。
*   **强化学习算法：** 使用 **GRPO**（Grouped Reward Policy Optimization）进行策略优化。
*   **复合奖励函数：** 包含四个维度：
    1.  **任务结果奖励（$r_{task}$）：** 后续相关任务的成功率。
    2.  **函数调用奖励（$r_{fc}$）：** 确保生成的技能操作格式正确。
    3.  **内容质量奖励（$r_{cnt}$）：** 由外部 LLM 评估技能的原子性、可重用性和准确性。
    4.  **压缩奖励（$r_{comp}$）：** 鼓励提炼简洁的技能，而非简单复制原始轨迹。

#### 3. 实验设计
*   **实验场景与数据集：**

    *   **单轮推理任务：** AIME24、AIME25（数学竞赛）、GPQA-Diamond（专家级科学问答）。
*   **对比方法（Baselines）：**
    *   **无记忆（No Memory）：** 独立解决每个任务。
    *   **强记忆基线：** ReasoningBank（提炼推理见解）、MemP（基于启发式规则的程序化记忆管理）。
    *   **内部变体：** SkillOS-base（未经过 RL 训练的策展器）、SkillOS-gemini（直接使用 Gemini-2.5-Pro 作为策展器）。

*   **训练时长：** ALFWorld 约 3 天，推理任务约 2.5 天，WebShop 约 5 天。
*   **模型底座：** 策展器使用 Qwen3-8B；执行器测试了 Qwen3-8B/32B、Gemini-2.5-Pro 和 Gemini-3.1-Flash-Lite。

#### 6. 主要结论与发现
*   **性能领先：** SkillOS 在所有基准测试中均优于无记忆和现有的记忆管理方法，成功率提升最高达 9.8%。
*   **效率提升：** 相比最强基线，SkillOS 减少了约 6.0% 的交互步数，表明策展的技能能帮助智能体找到“捷径”。
*   **小模型逆袭：** 经过 RL 训练的 8B 策展器在技能管理任务上优于未经训练的 Gemini-2.5-Pro。
*   **元技能涌现：** 随着训练进行，策展器从最初的盲目插入转向对现有技能的精细更新，并演化出如“失败恢复”、“系统化搜索”等高层级元技能。

#### 7. 优点
*   **模块化解耦：** 策展器与执行器分离，使得技能管理能力可以跨模型迁移，无需重新训练昂贵的执行器。
*   **经验驱动：** 摆脱了人工规则，通过任务反馈闭环学习策展策略。
*   **结构化存储：** 采用 Markdown 格式，既方便 LLM 阅读，也具备良好的可解释性和人工干预空间。

#### 8. 不足与局限
*   **检索机制简单：** 目前依赖 BM25 关键词检索，对于语义复杂的技能匹配可能存在瓶颈。
*   **执行器冻结：** 虽然保证了模块化，但策展器无法根据执行器的特定“偏好”进行协同进化（Co-evolution）。
*   **技能表示简化：** 目前将技能简化为单个 Markdown 文件，尚未支持包含脚本或外部资源的复杂分层技能结构。
*   **算力需求：** 尽管策展器模型较小，但 RL 训练过程仍需要显著的算力支持。
