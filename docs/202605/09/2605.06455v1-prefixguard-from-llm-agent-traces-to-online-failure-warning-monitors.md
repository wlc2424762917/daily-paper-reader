## 摘要
大语言模型 (LLM) 智能体现在执行长周期的工具使用任务，最终结果检查往往对于干预来说为时已晚。在线预警需要针对异构轨迹的轻量级前缀监控器，但手工编写的事件模式（schemas）很脆弱，且部署时的 LLM 评判成本高昂。我们引入了 PrefixGuard，这是一个从轨迹到监控器的框架，包含离线的 StepView 归纳步骤以及随后的监督监控器训练。StepView 从原始轨迹样本中归纳出确定性的类型化步骤适配器，而监控器则从终端结果中学习事件抽象和前缀风险评分器。在 WebArena、$τ^2$-Bench、SkillsBench 和 TerminalBench 上，最强的 PrefixGuard 监控器分别达到了 0.900/0.710/0.533/0.557 的 AUPRC。在每种表示中使用最强的后端，它们比原始文本对照组平均提高了 +0.137 AUPRC。在相同的前缀预警协议下，LLM 评判器的表现仍然明显较弱。我们还推导出了基于评分的精准率-召回率曲线下面积 (AUPRC) 的可观测性上限，该上限将监控器误差与在观测前缀中缺乏证据的失败区分开来。对于有限状态审计，事后确定性有限自动机 (DFA) 提取在 WebArena 和 $τ^2$-Bench 上保持紧凑（29 和 20 个状态），但在 SkillsBench 和 TerminalBench 上扩展到 151 和 187 个状态。最后，首警诊断表明，强大的排名能力并不意味着部署实用性：WebArena 排名表现良好，但无法支持低误报预警，而 $τ^2$-Bench 和 TerminalBench 则保留了更多具有可操作性的早期预警。总之，这些结果将 PrefixGuard 定位为一种实用的监控器合成方案，并为前缀预警何时能转化为可操作的干预提供了明确的诊断。

## Abstract
Large language model (LLM) agents now execute long, tool-using tasks where final outcome checks can arrive too late for intervention. Online warning requires lightweight prefix monitors over heterogeneous traces, but hand-authored event schemas are brittle and deployment-time LLM judging is costly. We introduce PrefixGuard, a trace-to-monitor framework with an offline StepView induction step followed by supervised monitor training. StepView induces deterministic typed-step adapters from raw trace samples, and the monitor learns an event abstraction and prefix-risk scorer from terminal outcomes. Across WebArena, $τ^2$-Bench, SkillsBench, and TerminalBench, the strongest PrefixGuard monitors reach 0.900/0.710/0.533/0.557 AUPRC. Using the strongest backend within each representation, they improve over raw-text controls by an average of +0.137 AUPRC. LLM judges remain substantially weaker under the same prefix-warning protocol. We also derive an observability ceiling on score-based area under the precision-recall curve (AUPRC) that separates monitor error from failures lacking evidence in the observed prefix. For finite-state audit, post-hoc deterministic finite automaton (DFA) extraction remains compact on WebArena and $τ^2$-Bench (29 and 20 states) but expands to 151 and 187 states on SkillsBench and TerminalBench. Finally, first-alert diagnostics show that strong ranking does not imply deployment utility: WebArena ranks well yet fails to support low-false-alarm alerts, whereas $τ^2$-Bench and TerminalBench retain more actionable early alerts. Together, these results position PrefixGuard as a practical monitor-synthesis recipe with explicit diagnostics for when prefix warnings translate into actionable interventions.

随着 LLM 智能体被部署在软件工程、网络安全和金融管理等高风险领域，它们通常需要执行长周期的多步任务。目前的评估手段（如任务校验器）通常只能在任务结束时给出结果，这对于防止不可逆的损害来说为时已晚。
*   **现有挑战**：
    *   **传统运行时验证**：依赖手工编写的事件模式，难以应对异构且不断演变的智能体轨迹。
    *   **LLM-as-judge**：在每个步骤都调用 LLM 进行评判的成本过高，难以实时部署。
    *   **缺乏可解释性**：纯神经预测模型无法提供可审计的符号化状态。
*   **核心目标**：合成一种轻量级、无需部署时 LLM 推理、且具备符号化审计能力的在线失败预警监控器。

### 2. 方法论：核心思想与关键技术
PrefixGuard 采用了一种“从轨迹到监控器”的合成路径，主要包含以下组件：
*   **StepView（离线适配器归纳）**：利用 LLM 在离线阶段通过少量样本归纳出确定性的解析规则，将异构的原始轨迹（如浏览器操作、工具调用、对话）转换为统一的类型化记录（包含元数据、观察、动作、工具、参数、结果、状态等字段）。
*   **微分事件抽象层**：使用 TF-IDF 对 StepView 记录进行编码，并通过 Gumbel-Softmax 算子将连续向量映射到 $K$ 个离散的潜在符号（事件字母表），实现端到端的离散表示学习。
*   **前缀风险评分器（监控器后端）**：支持多种架构，包括 GRU（默认）、Transformer 和软有限状态机（Soft-FSM）。监控器学习预测当前前缀是否处于“临近失败”窗口（定义为终端失败前的最后 $H$ 步）。
*   **DFA 提取与审计**：训练完成后，可以从学习到的硬符号中提取确定性有限自动机（DFA），作为事后审计的符号化产物。
*   **可观测性上限（Observability Ceiling）**：论文推导了一个理论公式，用于区分监控器的误差与由于轨迹本身缺乏失败证据而导致的预测局限。

### 3. 实验设计
*   **数据集/场景**：涵盖了四种不同类型的智能体基准：
    1.  **WebArena**：浏览器导航。
    2.  **$\tau^2$-Bench**：工具驱动的对话。
    3.  **SkillsBench**：编程任务。
    4.  **TerminalBench**：命令行界面（CLI）。
*   **对比方法（Baselines）**：
    *   **LLM 评判器**：GPT-4o-mini 和 DeepSeek-V3 的零样本（Zero-shot）评判。
    *   **PPM 比较**：预测过程监控（Predictive Process Monitoring）中的 LSTM 基准。
    *   **原始文本控制组**：不使用 StepView，直接处理原始文本序列的模型。
*   **评估指标**：主要使用 AUPRC（精准率-召回率曲线下面积），并辅以首警诊断（First-alert diagnostics）来衡量部署实用性。

### 4. 资源与算力
*   **硬件**：使用了本地工作站，配备 2 个 AMD EPYC 7452 CPU，125 GiB RAM，以及 **3 块 NVIDIA A100-PCIE-40GB GPU**。
*   **训练时长**：单个任务的训练时间从 30 分钟（$\tau^2$-Bench）到 3.5 小时（TerminalBench）不等。
*   **推理成本**：部署时无需 LLM，仅涉及轻量级的 TF-IDF 编码和小型神经网络（如单层 GRU）推理，计算开销极低。

### 5. 实验数量与充分性
*   **实验规模**：在 4 个大型基准测试上进行了详尽实验，每个实验运行 3 个随机种子并报告均值和标准差。
*   **消融实验**：进行了字段删除（Field-drop）实验，验证了 StepView 中不同字段对预测性能的贡献。
*   **充分性**：实验设计非常客观，不仅对比了性能，还通过“可观测性上限”分析了不同数据集的内在难度。此外，还探讨了排名质量（AUPRC）与实际部署预警（低误报率下的召回）之间的脱节，体现了严谨的工程视角。

### 6. 主要结论与发现
*   **性能优越**：PrefixGuard 在所有基准上均显著优于 LLM 评判器和原始文本基准。在 WebArena 上 AUPRC 达到 0.900，平均比原始文本对照组提升 0.137。
*   **StepView 的价值**：类型化的证据（尤其是“结果”和“状态”字段）对于识别失败至关重要，且不同领域的关键字段不同。
*   **审计边界**：DFA 提取在 WebArena 和 $\tau^2$-Bench 上非常紧凑（约 20-30 个状态），但在编程和 CLI 任务中会急剧膨胀（超过 150 个状态），说明了复杂任务中有限状态审计的局限性。
*   **排名不等于实用性**：高 AUPRC 并不总是意味着能提供早期的、低误报的预警。例如 WebArena 虽然排名分高，但预警往往发生在失败前夕，而 $\tau^2$-Bench 则能提供更具操作性的早期信号。

### 7. 优点
*   **架构优雅**：将 LLM 的能力用在离线归纳（StepView）而非在线推理，兼顾了灵活性与运行效率。
*   **神经符号结合**：通过 Gumbel-Softmax 桥接了神经预测与符号化 DFA 审计，增强了系统的透明度。
*   **诊断深入**：提出了可观测性上限理论，为评估监控器的“天花板”提供了数学工具。

### 8. 不足与局限
*   **依赖可见证据**：如果失败的原因完全不在执行轨迹中（例如逻辑错误但环境反馈正常），监控器将无法识别。
*   **DFA 扩展性**：在高度灵活的任务（如编程）中，提取的 DFA 过于庞大，失去了人类可读的审计优势。
*   **干预策略缺失**：论文专注于“预警”，但未深入探讨触发预警后应采取的具体干预动作（如回滚或人工介入）。
