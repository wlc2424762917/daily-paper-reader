## 摘要
随着多模态大语言模型（MLLMs）和代码智能体的发展，网页开发已从手动编程转向基于智能体的项目级代码合成。现有的基准测试通常依赖于理想化的假设，特别是针对结构良好、信息丰富的输入以及静态执行设置。相比之下，现实世界的开发受到一个关键瓶颈的制约：非专家用户模糊、低质量的指令与模型理解之间的语义失配，这导致了一种我们称之为“盲目执行”（blind execution）的失败模式。为了填补这一空白，我们推出了 InteractWeb-Bench，这是首个针对非专家低代码用户场景下的网页生成多模态交互式基准测试。InteractWeb-Bench 引入了四种类型的用户智能体和基于人格驱动的指令扰动，以需求工程缺陷分类法为基础，系统地模拟了包括歧义、冗余和矛盾在内的多样化用户行为。我们为智能体开发了一个交互式执行环境，其特点是包含“澄清”（Clarify）、“实现”（Implement）、“验证”（Verify）和“提交”（Submit）的统一动作空间，从而实现迭代式的意图细化、代码合成以及基于视觉反馈的验证。广泛的实验和分析表明，前沿的基于 MLLM 的智能体仍然受困于“盲目执行”，暴露了其在意图识别和自适应交互方面的局限性。

## Abstract
With the advancement of multimodal large language models (MLLMs) and coding agents, the website development has shifted from manual programming to agent-based project-level code synthesis. Existing benchmarks rely on idealized assumptions, especially for well-structured, information-rich inputs and static execution settings. In contrast, real-world development is constrained by a critical bottleneck: the semantic misalignment between ambiguous, low-quality instructions from non-expert users and model understanding, which results in a failure mode that we term blind execution. To address this gap, we introduce InteractWeb-Bench, the first multimodal interactive benchmark for website generation under non-expert low-code user conditions. InteractWeb-Bench introduces four types of user agents and persona-driven instruction perturbations to systematically simulate diverse user behaviors, including ambiguity, redundancy, and contradiction, grounded in requirement engineering defect taxonomies. We develop an interactive execution environment for agents, featuring a unified action space comprising Clarify, Implement, Verify, and Submit, enabling iterative intent refinement, code synthesis, and visual feedback-based validation. Extensive experiments and analysis reveal that frontier MLLM-based agents remain trapped in blind execution, exposing limitations in intent recognition and adaptive interaction.

### 1. 核心问题与整体含义（研究动机和背景）
随着多模态大语言模型（MLLM）的发展，网页开发正从手动编码转向智能体驱动的自动化合成。然而，现有基准测试（如 WebGen-Bench）通常假设用户指令是完美、结构化且信息丰富的。
*   **核心问题**：在现实场景中，非专家用户的指令往往具有**模糊性、冗余性或矛盾性**。
*   **盲目执行（Blind Execution）**：现有智能体在面对低质量指令时，往往不加思考地直接进入代码实现阶段，而不是通过交互来澄清意图。这种“单向执行”策略导致了高失败率、UI 渲染异常和功能幻觉。
*   **研究目标**：构建一个能够模拟真实交互、评估智能体意图识别与自适应交互能力的基准测试。

### 2. 方法论：核心思想与关键技术
论文提出了 **InteractWeb-Bench**，其核心由两个模块组成：
*   **人格驱动的用户代理模块（Persona-Driven User Agent）**：
    *   基于需求工程缺陷分类法，设计了四种人格：**极简型（P-MIN）**（信息缺失）、**啰嗦型（P-RAM）**（低信噪比）、**直觉型（P-INT）**（表达抽象/感性）、**矛盾型（P-CON）**（逻辑冲突）。
    *   通过 metamorphic 算子对原始指令进行扰动，模拟真实用户的表达缺陷。
*   **交互式执行环境与动作空间**：
    *   为智能体提供统一的动作空间：**澄清（Clarify）**（向用户提问）、**实现（Implement）**（编写代码/配置）、**验证（Verify）**（进行 GUI 视觉检查）、**提交（Submit）**（完成任务）。
    *   **视觉反馈循环**：智能体可以自主触发浏览器内核进行渲染，并根据截图和控制台错误进行自我修正。
*   **评估指标（TCR）**：
    *   采用**基于约束槽（Constraint-Slot）**的评分机制。将任务分解为原子约束（Oracle Slots），根据技术复杂度（静态、交互、复杂逻辑）赋予不同权重，计算任务完成率（Task Completion Rate）。

### 3. 实验设计
*   **数据集规模**：从 101 个高质量种子网站出发，通过四种人格扰动扩展为 **404 个动态测试用例**。
*   **难度分级**：根据约束复杂度将任务分为简单（Easy）、中等（Middle）、困难（Hard）三个等级。
*   **对比方法**：评估了 9 种前沿 MLLM 智能体，包括：
    *   **闭源模型**：Qwen3.6-Plus, Kimi-K2.5, GPT-4.1, GPT-4.1-mini, Gemini-3.1-Flash-Lite。
    *   **开源模型**：Qwen3.5-397B/9B, Gemma-4-31B/26B。
*   **评估器**：使用 WebVoyager 框架配合 GPT-5-mini 作为视觉审计员，对生成的网页进行多步自动化测试。

### 4. 资源与算力
*   **算力配置**：对于开源模型（如 Qwen3.5-9B, Gemma-4-31B），研究团队使用了 **8 张 NVIDIA A800 GPU**，并采用 **vLLM 框架**进行推理部署。
*   **闭源模型**：通过官方 API 进行访问。
*   **成本统计**：论文详细列出了各模型生成单个网站的平均成本（如 GPT-4.1 约为 0.475 美元，Gemini-3.1-Flash-Lite 仅为 0.016 美元）。

### 5. 实验数量与充分性
*   **实验组数**：共进行了 404 组完整的智能体执行轨迹实验，涵盖了不同难度和人格维度。
*   **充分性与公平性**：
    *   实验不仅测试了最终的 TCR，还深入分析了**意图对齐得分（IAS）**、**主动澄清命中率（CHR）**、**代码行数（LoC）**及**幻觉率**。
    *   引入了人类专家（3 名计算机专业博士生）进行审美质量的对比评估，验证了模型评估器的客观性。
    *   设置了全局步数限制和错误边界，防止智能体陷入无限循环，确保了评估的效率和公平。

### 6. 主要结论与发现
*   **普遍受困于“盲目执行”**：即使是表现最好的 Qwen3.6-Plus，其 TCR 也仅为 38.78%，说明现有模型在处理模糊需求时极度乏力。
*   **“过度生成”补偿策略**：模型在面对缺失信息时，倾向于通过生成大量冗余代码（LoC 超过 1000 行）来“猜测”用户意图，导致幻觉率激增（部分模型超过 60%），而不是选择主动澄清。
*   **对缺失信息更敏感**：模型在处理“啰嗦型”指令（P-RAM）时表现尚可，但在处理“极简型”指令（P-MIN）时性能大幅下降，反映出其缺乏主动补全需求的能力。
*   **视觉反馈利用率低**：虽然模型会执行验证动作，但往往只进行局部修复，无法根据视觉反馈重新审视底层逻辑错误。

### 7. 优点（亮点）
*   **首创性**：首次提出了针对网页生成任务的交互式、多模态基准测试，填补了静态评估的空白。
*   **真实性**：引入需求工程中的缺陷分类法来模拟用户人格，极大地贴近了现实开发中的“非专家用户”场景。
*   **闭环评估**：构建了从“提问-编码-视觉调试-提交”的完整闭环，能够多维度考察智能体的综合素质。

### 8. 不足与局限
*   **种子任务规模**：101 个种子任务虽然经过扩展，但对于覆盖极其复杂的企业级网页应用来说，规模仍有提升空间。
*   **评估器偏见**：虽然使用了强力的 GPT-5-mini 作为评估器并进行了人类校验，但自动化视觉审计仍可能存在对某些动态交互识别不准的风险。
*   **环境限制**：实验环境禁止了外部 API 调用（仅限 Mock 数据），这限制了对需要真实后端交互的网页任务的评估。
