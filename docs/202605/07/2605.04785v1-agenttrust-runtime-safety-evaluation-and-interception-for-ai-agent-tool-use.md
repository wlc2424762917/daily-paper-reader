## 摘要
现代 AI 智能体通过文件操作、Shell 命令、HTTP 请求和数据库查询等工具调用来执行现实世界的副作用。单个不安全的操作，包括误删、凭据泄露或数据外泄，都可能造成不可逆转的损害。现有的防御措施并不完善：事后基准测试在执行后衡量行为，静态护栏会遗漏混淆和多步上下文，而基础设施沙箱仅限制代码运行位置，而不理解操作的含义。我们提出了 AgentTrust，这是一个运行时安全层，它在执行前拦截智能体工具调用，并返回结构化的判定结果：允许、警告、阻止或审查。AgentTrust 结合了 Shell 反混淆归一化器、用于提供更安全替代方案的 SafeFix 建议、用于多步攻击链检测的 RiskChain，以及用于处理歧义输入的缓存感知型 LLM-as-Judge。我们发布了一个涵盖六个风险类别的 300 场景基准测试，以及另外 630 个独立构建的真实世界对抗场景。在内部基准测试中，仅生产规则集在毫秒级端到端延迟下实现了 95.0% 的判定准确率和 73.7% 的风险等级准确率。在 630 场景基准测试中（在修复后的规则集下评估，且不声称为零样本），AgentTrust 实现了 96.7% 的判定准确率，其中包括约 93% 的 Shell 混淆有效负载准确率。AgentTrust 以 AGPL-3.0 许可证发布，并为兼容 MCP 的智能体提供模型上下文协议（Model Context Protocol）服务器。

## Abstract
Modern AI agents execute real-world side effects through tool calls such as file operations, shell commands, HTTP requests, and database queries. A single unsafe action, including accidental deletion, credential exposure, or data exfiltration, can cause irreversible harm. Existing defenses are incomplete: post-hoc benchmarks measure behavior after execution, static guardrails miss obfuscation and multi-step context, and infrastructure sandboxes constrain where code runs without understanding what an action means.   We present AgentTrust, a runtime safety layer that intercepts agent tool calls before execution and returns a structured verdict: allow, warn, block, or review. AgentTrust combines a shell deobfuscation normalizer, SafeFix suggestions for safer alternatives, RiskChain detection for multi-step attack chains, and a cache-aware LLM-as-Judge for ambiguous inputs.   We release a 300-scenario benchmark across six risk categories and an additional 630 independently constructed real-world adversarial scenarios. On the internal benchmark, the production-only ruleset achieves 95.0% verdict accuracy and 73.7% risk-level accuracy at low-millisecond end-to-end latency. On the 630-scenario benchmark, evaluated under a patched ruleset and not claimed as zero-shot, AgentTrust achieves 96.7% verdict accuracy, including about 93% on shell-obfuscated payloads. AgentTrust is released under the AGPL-3.0 license and provides a Model Context Protocol server for MCP-compatible agents.

### AgentTrust：AI 智能体工具使用的运行时安全评估与拦截

#### 1. 核心问题与整体含义
随着 AI 智能体（如 Claude Code, Cursor, OpenDevin）从单纯的文本生成转向能够执行文件操作、Shell 命令、网络请求等具有“副作用”的自主行动，安全风险也随之剧增。一个错误的命令（如 `rm -rf /`）或敏感信息泄露（如打印 API Key）可能导致不可逆的损失。
论文指出，现有的防御手段存在明显缺陷：
*   **事后基准测试**：仅能在伤害发生后进行评估，无法实时拦截。
*   **静态规则护栏**：容易被简单的代码混淆（如变量替换）绕过，且缺乏对多步操作序列的理解。
*   **基础设施沙箱**：虽然限制了运行环境，但无法理解操作的语义（例如沙箱内的智能体仍可能合法地将数据库外泄到攻击者域名）。
**AgentTrust** 的核心目标是提供一个**实时、语义感知**的安全拦截框架，在智能体执行任何工具调用前给出“允许、警告、拦截、审查”的判定。

#### 2. 方法论
AgentTrust 采用了一种多级流水线架构，核心组件包括：
*   **ShellNormalizer（Shell 归一化器）**：针对 Shell 命令的混淆攻击，实现了 9 种去混淆策略（如变量展开、十六进制/八进制转义解码、别名解析、命令替换模拟等），在不实际执行命令的情况下暴露其真实意图。
*   **ActionAnalyzer & PolicyEngine**：内置 170 条 YAML 配置规则和 42 种风险模式匹配，涵盖文件系统、网络、凭据泄露和系统配置等多个维度。
*   **RiskChain（会话跟踪器）**：通过维护会话历史，识别多步攻击链。例如，单独看“读取 .env 文件”和“执行 curl 请求”可能都是合法的，但 RiskChain 能识别出“读取 -> 编码 -> 外部发送”这一完整的外泄序列。
*   **SafeFix（安全修复引擎）**：当判定为拦截时，不仅阻止操作，还会根据 37 条修复规则建议更安全的替代方案（如将 `chmod 777` 建议修改为 `chmod 755`）。
*   **LLM-as-Judge（增量式评判）**：对于规则难以判定的模糊场景，调用 LLM 进行五维语义评估。为了降低长会话的 Token 成本，引入了**块哈希增量检测**（受 rsync 启发），仅对会话中新增或修改的部分进行评估。

*   **数据集/场景**：

    *   **独立真实世界基准（630 场景）**：包含对抗性探测、现代 DevOps 工作流、云原生操作及复杂的混淆绕过尝试。

#### 4. 资源与算力
*   **本地算力**：核心规则引擎和分析器在 Apple M4 CPU 上运行，单线程处理。

*   **训练时长**：论文未提及大规模模型训练，因为该系统主要基于专家规则、启发式算法和现成的 LLM 推理。

#### 5. 实验数量与充分性
*   **实验规模**：论文进行了多维度的评估，包括在 930 个总场景上的准确率测试、针对 5 个版本的演进消融实验、针对 4 个核心组件的消融实验，以及针对 LLM 缓存机制的效率测试。
*   **充分性与客观性**：实验设计较为严谨。作者通过“独立基准测试”来避免规则过拟合内部数据集的问题，并详细报告了误报率（FPR）和漏报率（FNR）。此外，论文公开了所有代码、规则集和基准测试数据，支持社区复现。

#### 6. 主要结论与发现
*   **高准确率与低延迟**：AgentTrust 生产规则集在内部测试中达到了 **95.0%** 的判定准确率，且端到端延迟保持在 **2 毫秒** 左右，远优于纯 LLM 方案（秒级延迟）。
*   **去混淆的必要性**：ShellNormalizer 使系统在处理混淆攻击时的漏报率降低了约 2.5 倍。
*   **防御深度**：相比于 NeMo Guardrails（存在严重的过度拦截/误报问题，FPR 高达 96%），AgentTrust 在保持高安全性的同时，极大地降低了对正常开发工作的干扰。
*   **混合模式优势**：规则引擎负责处理 95% 的常见风险，LLM 评判层则作为“第二意见”处理剩余的模糊案例，将漏报率进一步压低至 1% 以下。

#### 7. 优点
*   **语义感知**：不同于简单的字符串匹配，它能理解操作的上下文和潜在后果。
*   **工程化程度高**：提供了 MCP 服务器支持，可与主流智能体框架无缝集成；具备完善的“失败安全”（Fail-safe）机制，确保系统异常时不会默认放行。
*   **可解释性**：判定结果包含具体的规则 ID、风险因素和修复建议，而非简单的“Yes/No”。
*   **开源贡献**：发布了高质量的智能体安全基准测试集，填补了该领域的空白。

#### 8. 不足与局限
*   **静态分析限制**：作为运行时拦截层，它无法完全模拟复杂的动态数据流（例如无法预知一个动态生成的脚本在执行时的所有行为）。
*   **多步检测的量化不足**：虽然设计了 RiskChain，但目前的基准测试多为单步动作，缺乏对复杂长程攻击链的系统性量化评估。
*   **环境依赖**：某些判定（如软件包拼写检查）需要外部数据库支持，目前系统尚未完全集成这些外部 Oracle。
*   **语言偏见**：目前的规则和 Prompt 主要针对英文环境。
