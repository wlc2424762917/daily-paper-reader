## 摘要
我们推出了 PhysicianBench，这是一个用于在电子健康档案（EHR）环境下的真实临床场景中评估大语言模型（LLM）智能体执行医生任务的基准测试。现有的医疗智能体基准测试主要侧重于静态知识召回、单步原子操作或动作意图，缺乏针对环境的可验证执行。因此，它们无法捕捉真实临床系统所特有的长程、复合工作流。PhysicianBench 包含 100 个长程任务，这些任务改编自全科医生与专科医生之间的真实会诊案例，且每个任务都经过独立医生小组的评审。任务在包含真实患者记录的 EHR 环境中实例化，并通过商业 EHR 供应商使用的标准 API 进行访问。任务涵盖 21 个科室（如心血管内科、内分泌科、肿瘤科、精神科）和多种工作流类型（如诊断解读、药物处方、治疗计划），平均每个任务需要 27 次工具调用。完成每个任务需要跨就诊记录检索数据、对异构临床信息进行推理、执行具有实际影响的临床操作并生成临床文档。每个任务被分解为结构化的检查点（整个基准测试共 670 个），用于捕捉不同的完成阶段，并由特定任务脚本通过基于执行的验证进行评分。在对 13 个闭源和开源 LLM 智能体的测试中，表现最好的模型成功率仅为 46% (pass@1)，而开源模型最高仅达到 19%，这揭示了当前智能体能力与真实临床工作流需求之间存在巨大差距。PhysicianBench 为衡量自主临床智能体的进展提供了一个真实且基于执行的基准测试。

## Abstract
We introduce PhysicianBench, a benchmark for evaluating LLM agents on physician tasks grounded in real clinical setting within electronic health record (EHR) environments. Existing medical agent benchmarks primarily focus on static knowledge recall, single-step atomic actions, or action intent without verifiable execution against the environment. As a result, they fail to capture the long-horizon, composite workflows that characterize real clinical systems. PhysicianBench comprises 100 long-horizon tasks adapted from real consultation cases between primary care and subspecialty physicians, with each task independently reviewed by a separate panel of physicians. Tasks are instantiated in an EHR environment with real patient records and accessed through the same standard APIs used by commercial EHR vendors. Tasks span 21 specialties (e.g., cardiology, endocrinology, oncology, psychiatry) and diverse workflow types (e.g., diagnosis interpretation, medication prescribing, treatment planning), requiring an average of 27 tool calls per task. Solving each task requires retrieving data across encounters, reasoning over heterogeneous clinical information, executing consequential clinical actions, and producing clinical documentation. Each task is decomposed into structured checkpoints (670 in total across the benchmark) capturing distinct stages of completion graded by task-specific scripts with execution-grounded verification. Across 13 proprietary and open-source LLM agents, the best-performing model achieves only 46% success rate (pass@1), while open-source models reach at most 19%, revealing a substantial gap between current agent capabilities and the demands of real-world clinical workflows. PhysicianBench provides a realistic and execution-grounded benchmark for measuring progress toward autonomous clinical agents.

#### 1. 核心问题与整体含义
*   **研究动机**：尽管大语言模型（LLM）在医学知识问答上表现优异，但将其作为“智能体（Agent）”应用于真实临床工作流仍面临巨大挑战。
*   **核心问题**：现有的医疗评估基准存在三大缺陷：
    1.  **任务简单**：多为单步操作或选择题，缺乏长程（Long-horizon）复合工作流。
    2.  **缺乏执行验证**：仅评估智能体的“意图”，而不验证其在电子健康档案（EHR）系统中实际执行动作（如开处方、下医嘱）后的状态变化。
    3.  **数据脱离现实**：多使用模拟数据或考试题，而非真实的临床病例。
*   **整体含义**：本文提出了 **PhysicianBench**，这是首个基于真实临床病例、使用标准 FHIR API 接口、并在真实 EHR 环境中进行执行验证的长程医疗智能体基准测试。

#### 2. 方法论
*   **核心思想**：将临床任务建模为部分可观测马尔可夫决策过程（POMDP），让智能体通过多步工具调用与 FHIR 兼容的 EHR 服务器交互。
*   **关键技术细节**：
    *   **任务构建**：从斯坦福大学的真实电子会诊（e-consult）案例中提取 100 个任务，涵盖 21 个专科。
    *   **环境模拟**：使用 Docker 容器部署 HAPI FHIR 服务器，预装载脱敏后的真实患者病历。
    *   **工具集**：提供 14 种结构化工具，包括 FHIR 读取（查询病历、化验单、诊断）、FHIR 写入（开医嘱、预约、通信）以及文件操作。
    *   **细粒度评估（Checkpoints）**：将每个任务分解为平均 6.7 个检查点（总计 670 个），分为数据检索、临床推理、动作执行和文档撰写四类。
    *   **评分机制**：结合代码验证（检查数据库状态）、混合验证（提取文本并比对）和 LLM 裁判（基于医生制定的量表评分）。

#### 3. 实验设计
*   **数据集/场景**：100 个经过医生独立评审的任务，涉及心血管、内分泌、肿瘤等 21 个专科。
*   **对比方法**：评估了 12 个主流 LLM 智能体，包括：
    *   **闭源模型**：GPT-5.5、GPT-5.4、Claude 4.6/4.7 (Opus/Sonnet)、Gemini Pro 3.1、Grok-4.20、Qwen3.6-Plus、MiniMax M2.7、MiMo-v2.5-Pro。
    *   **开源模型**：DeepSeek V4-Pro、Kimi-K2.6。
*   **评估指标**：Pass@1（单次成功率）、Pass@3（三次尝试中至少成功一次的概率）、Pass^3（三次尝试全部成功的可靠性指标）以及平均工具调用次数（#Turns）。

#### 4. 资源与算力
*   **算力说明**：论文未明确提及模型训练所需的算力（因为该研究侧重于评估而非训练）。
*   **评估环境**：每个任务在独立的 Docker 容器中运行，使用 HAPI FHIR JPA 服务器和 H2 数据库。智能体通过 API 调用进行推理，实验中设置了最高 100 步的工具调用限制。

#### 5. 实验数量与充分性
*   **实验规模**：对 12 个模型进行了测试，每个模型在 100 个任务上运行 3 次独立实验，总计产生了超过 3600 条完整的智能体交互轨迹。
*   **充分性与公平性**：
    *   **充分性**：实验涵盖了从错误分布分析到专科差异分析的多个维度，并进行了深度的根因分析（Root Cause Analysis）。
    *   **公平性**：所有模型使用统一的轻量级智能体框架和系统提示词，确保评估结果反映的是模型核心能力而非框架差异。
    *   **客观性**：任务经过 11 名专业医生多轮独立评审和修订，确保了临床金标准的权威性。

#### 6. 主要结论与发现
*   **能力差距巨大**：最强模型 GPT-5.5 的 Pass@1 仅为 46.3%，而开源模型最高仅 18.7%。
*   **可靠性不足**：即使是表现最好的模型，其 Pass^3（一致成功率）也仅为 28%，说明成功具有随机性。
*   **瓶颈在于推理**：错误分析显示，约 50% 的失败源于“临床推理”错误，而非简单的工具使用或数据检索问题。
*   **“输出鸿沟”现象**：智能体常在病历记录中写对了处理方案，但却忘记在 EHR 系统中实际下达对应的电子医嘱（Action Execution 失败）。

#### 7. 优点
*   **临床忠实度高**：任务源于真实会诊，而非合成数据，捕捉了真实医疗环境中的复杂共病和数据不规则性。
*   **执行导向**：不仅看智能体“说了什么”，更看它对 EHR 数据库做了什么修改，实现了真正的闭环评估。
*   **标准化接口**：采用 FHIR 标准 API，使得该基准测试具有极强的行业相关性和可扩展性。
*   **诊断性强**：通过 670 个结构化检查点，能够精准定位智能体是在检索、推理还是执行阶段出错。

#### 8. 不足与局限
*   **场景覆盖有限**：目前主要集中在门诊/电子会诊场景，尚未覆盖急诊、住院或手术等更动态的环境。
*   **模态单一**：目前仅处理结构化数据和文本，未引入医学影像（X光、CT）或波形数据（心电图）。
*   **缺乏交互性**：评估是自主完成的，未考虑医生与智能体之间的实时协作或共享决策过程。
*   **静态环境**：EHR 环境在任务执行期间是静态的，不涉及其他医护人员同时修改病历的并发情况。
