Abstract 

Modern AI agents (e.g. Claude Code, Cursor, OpenDevin, AutoGPT) execute real-world side-effects through tool calls: file operations, shell commands, HTTP requests, database queries. A single misjudged action—an accidental rm -rf / , a credential printed to a log, or an exfiltration disguised as a benign POST —can cause irreversible damage. Existing defenses fall short in complementary ways: post-hoc benchmarks measure agent behavior after harm has occurred, static rule-based guardrails miss obfuscation and semantic context, and 

infrastructure sandboxes constrain where code runs without understanding what an action 

means in application terms. We present AgentTrust , a real-time, semantics-aware safety-interception framework that sits between an agent and its tools and produces a structured verdict ( allow , warn , block ,

review ) before every action is executed. AgentTrust contributes four capabilities that, to our knowledge, no prior open framework combines: (1) a shell deobfuscation normalizer 

(nine strategies covering variable expansion, hex/octal escapes, alias resolution, command-substitution, ANSI-C quoting, and adjacent-quote concatenation) that exposes hidden dangerous commands to pattern matching; (2) SafeFix , a rule-driven engine that suggests safer alternatives (e.g. chmod 777 → chmod 755 ) instead of simply blocking; (3) RiskChain ,an order-aware session tracker that detects multi-step attack chains (e.g. read .env → base64 

→ curl -X POST ) where each step is individually benign; and (4) a cache-aware LLM-as-Judge that uses block-hash delta detection (inspired by rsync /git ) to incrementally evaluate growing agent contexts, reducing token cost on long sessions while preserving semantic-level accuracy on ambiguous inputs. We release a benchmark of 300 curated scenarios across six risk categories and an additional 630 independently-generated real-world adversarial scenarios spanning safe de-velopment workflows, moderate-risk DevOps operations, dangerous attacks, and obfuscated evasion attempts. On the internal benchmark, the production-only AgentTrust ruleset achieves 95.0% verdict accuracy and 73.7% risk-level accuracy at low-millisecond median end-to-end latency (with sub-millisecond analyzer-plus-policy microbenchmarks). When the optional benchmark-compatibility ruleset is loaded (four rules that match synthetic benchmark domains; never enabled in production deployments) the verdict accuracy rises to 97.0% and risk accuracy to 75.7%; we report the production figure as the headline (§6.2). On the 630 independently-constructed adversarial scenarios—evaluated under the patched ruleset and therefore not claimed as a zero-shot held-out set (§5.2)—verdict accuracy is 96.7% ,including ∼93% on shell-obfuscated payloads. AgentTrust is released under the AGPL-3.0 license (commercial licensing available on request) and exposes a Model-Context-Protocol (MCP) server so that any MCP-compatible agent can adopt it in minutes. 

Keywords: AI agent safety; tool-use guardrails; LLM-as-judge; benchmark; deobfuscation; multi-step attack chains; real-time interception.

Each tool call is an external side-effect: unlike a misclassified token, a wrong rm , an exposed API key, or a silent POST of secrets to an attacker-controlled domain cannot be revoked by re-sampling the next token. This shift creates an action-time safety surface that did not exist for chatbot-style LLMs. A 2024 survey of agent failure modes [14, 18] indicates that the modal incident is not adversarial jailbreak but accidental harm : a benign instruction expanded by an over-eager planner into a destructive sequence. Adversarial misuse aggravates the problem—recent work has shown that agents can be coerced through prompt injection [ 8 , 10 ], indirect context contamination, and multi-turn social engineering. 

The gap left by existing defenses. Three families of defense are widely deployed today, but each leaves a distinct residual gap (Table 1): 

Post-hoc benchmarks (e.g. AgentHarm [ 3 ], ToolEmu [ 18 ], R-Judge [ 21 ], TrustAgent [ 11 ]) evaluate the safety of an agent’s outputs in a sandbox. They are essential for measuring progress, but operate after the action has been taken; they cannot block harm in production. 

Static rule-based guardrails (e.g. Llama-Guard [ 6 , 12 ], NeMo Guardrails [ 17 ], Invariant Labs guardrails) intercept actions but use textual pattern matching that is easily defeated by trivial obfuscation ( r=rm; $r -rf / ) and that cannot reason about sequences of actions. 

Infrastructure sandboxes (containers, gVisor, OS-level seccomp) constrain where code runs. They are necessary but insufficient: a sandboxed agent that exfiltrates a customer database through a legitimately-allowed HTTP egress is operating entirely within its sandbox. What is missing is a layer that operates in real time , at the semantic level of an action’s intent, with explainable verdicts and a path to remediation rather than only refusal. 

Our approach. AgentTrust is a real-time interception framework that evaluates each proposed agent action before it executes and returns a structured TrustReport with a four-valued verdict: allow , warn , block , or review . The architecture (Section 4) consists of eight components: a ShellNormalizer that deobfuscates input through nine syntactic strategies; an 

ActionAnalyzer that extracts risk-relevant features via 42 regex patterns across four categories; a PolicyEngine with 170 YAML-configurable rules; a TrustInterceptor that orchestrates the pipeline; a TrustReporter for human-readable explanations; a SafeFixEngine with 37 fix rules that propose safer alternatives to dangerous actions; a SessionTracker (RiskChain) with seven order-aware multi-step attack-chain detectors; and a cache-aware LLMJudge that handles ambiguous cases through a five-dimension semantic evaluation, using block-hash delta detection (inspired by rsync /git ) to amortize LLM cost across long agent conversations.

# 2 Related Work 

We organize related work along four axes that map to the gaps identified in Section 1: post-hoc agent safety benchmarks, real-time LLM/agent guardrails, multi-step attack analysis, and LLM-as-judge evaluation.

2.1 Agent Safety Benchmarks (Post-hoc) 

The largest body of recent work measures agent safety after the agent has acted, in controlled sandboxes. AgentHarm [ 3 ] curates 110+ harmful tasks across 11 categories and grades how readily an agent complies. ToolEmu [ 18 ] emulates tool execution with an LLM sandbox to surface failure modes without real side-effects. R-Judge [ 21 ] uses LLM judges to retrospectively label agent trajectories. AgentDojo [ 8 ] provides a dynamic adversarial test bed for prompt-injection defenses. TrustAgent [ 11 ] introduces an agent-constitution-based scoring system. AgentBench [ 14 ] covers eight environments but is primarily capability- rather than safety-focused. These benchmarks are essential for measuring progress, but by construction they cannot stop a harmful action at decision time. AgentTrust complements them by providing a deployable interceptor and by releasing two additional benchmarks (300 + 630 scenarios) targeted at the kinds of accidental harm and obfuscation observed in real production telemetry. 

2.2 Real-Time LLM and Agent Guardrails 

Llama-Guard [12 ] and its successors [ 6 ] fine-tune small open LLMs as input/output classifiers across taxonomies (e.g. MLCommons hazard categories). They are general-purpose content mod-erators, not action-aware: a Llama-Guard verdict on rm -rf / will depend on the surrounding text, not on the structural fact that the action is irreversible. 

NeMo Guardrails [ 17 ] provides a configurable dialog-management layer with Colang flows. It supports input/output rails and tool-call rails, but rails are written as keyword-or-LLM-classifier conditions that do not reason about multi-step sequences or shell-level obfuscation. 

Invariant Labs 2 publishes pattern-matching guardrails for tool calls; coverage and precision depend on the user-supplied policy. 

LangChain and LlamaIndex expose hook points (e.g. “callbacks”), but ship no built-in safety semantics. 

Agent firewalls / sandboxes. Project-level work on agent-specific safety toolkits, browser sandboxes, and OS-level isolation (containers, gVisor, seccomp-bpf) constrain where actions execute. These are necessary but orthogonal: AgentTrust’s verdict layer can sit above any of them. 

Constitutional AI [5] and IAA / RLHF train safety into the agent itself. AgentTrust is complementary—an external verifier that does not require model retraining and that catches mistakes the trained model still makes. 

2.3 Multi-Step / Trajectory-Level Risk 

Most production guardrails operate per-call. A small literature addresses multi-step risk: Ruan et al. [18] and Liu et al. [14] measure trajectory-level success/failure in sandboxes; Yuan et al. [21] judges full transcripts; Debenedetti et al. [8] stages multi-turn injection attacks. None of these provide an online , order-aware detector that fires when partial sequences match a known dangerous chain. AgentTrust’s RiskChain (Section 4.7) fills this gap with seven predefined chains derived from MITRE-ATT&CK-style sequences (data exfiltration, credential harvesting, persistence installation, privilege escalation, supply-chain attack, reverse shell, data destruction). 

2.4 LLM-as-Judge 

LLM-as-judge has become a de-facto evaluation method for open-ended NLG [ 13 , 15 , 22 ]. In the safety setting, R-Judge [ 21 ], AgentSafe judges, and several proprietary systems use a single LLM call per evaluation.

Two practical concerns are largely unaddressed: (i) when the agent context is long and growing (a typical Claude Code session reaches 10 5+ tokens), re-evaluating the entire context on each step incurs substantial cost and latency; (ii) when the LLM is unavailable, guardrails should fail safe rather than fail open. git ’s object model) that routes to one of three strategies ( cache_hit , incremental , full ). It addresses (ii) through formally-tested fail-safe contracts: the framework returns review with a low-confidence score whenever the judge is unreachable. 

2.5 Shell Deobfuscation 

Adversarial shell obfuscation has been studied in the malware analysis community [ 2], with tools such as shellcheck and academic systems for bashfuscator reversal. These targets are typically post-hoc: parse a captured script. AgentTrust’s normalizer is the first, to our knowledge, to be embedded in a real-time agent guardrail and to operate purely on text-level rewriting (no eval-based execution), making it safe to deploy in production where the input is untrusted.

# 3 Threat Model and Problem Formulation 

3.1 Threat Model 

We assume an autonomous agent A that interleaves LLM reasoning with calls to a finite toolset 

T = {t1, . . . , t m}. Tool calls produce side-effects in an external environment E (filesystem, network, OS, databases). We assume: 1. The agent itself is partially trusted. A is not adversarial by design, but may emit unsafe actions because of (a) over-eager planning, (b) hallucinated paths or commands, (c) ambiguous user instructions, (d) indirect prompt injection from tool outputs [ 10 ], or (e) adversarial context constructed by an external user. 2. The environment is hostile by default. Filesystem, network, and credentials are valuable; we model the worst case in which any side-effect to E may be unrecoverable. 3. The interceptor is in-process and trusted. AgentTrust runs in the same trust boundary as the agent runtime; an attacker who already controls the runtime is out of scope. OS-level isolation (containers, seccomp) remains a complementary layer. AgentTrust Preprint 6

4. Side-effects are observable as structured actions. We require that the agent runtime can intercept each tool call and present it as a typed Action record (defined below) before execution. This is satisfied by MCP-style architectures and by most modern agent frameworks (LangChain, LlamaIndex, OpenDevin). 

Failure modes targeted. We focus on four classes of failure: (F1) accidental destructive actions —e.g. rm -rf on a home or system directory caused by a planning error; (F2) credential exposure —reading or printing of secrets where not required; (F3) exfiltration chains —multi-step sequences that combine benign-looking steps into a single dangerous outcome; (F4) obfus-cated commands —semantically dangerous actions hidden through shell-level encoding (variable substitution, hex/octal escapes, alias misuse, command substitution). 

Out-of-scope. (i) Adversarial fine-tuning of the underlying LLM. (ii) Side-channel attacks against E (timing, power, hardware). (iii) Direct compromise of the OS sandbox. (iv) General LLM content moderation (toxicity, copyright, factuality). 

3.2 Action and Verdict Definitions 

Definition 1 (Action) . An action is a tuple a = ( τ, κ, δ, π, ρ, σ, θ ) ∈ S where τ ∈ Type 

is a discrete category ( file_read , file_write , file_delete , shell_command , net-work_request , code_execution , database_query , api_call , credential_access ,

system_config , unknown ); κ is the invoking tool name; δ is a natural-language description; 

π is a parameter map (string-keyed); ρ is the raw payload (e.g. command text, URL, query); σ

is an optional session identifier; θ is a UTC timestamp. 

Definition 2 (Risk level) . The risk lattice R = ( R, ⪯) is the totally ordered set R =

{none , low , medium , high , critical } with the natural order. 

Definition 3 (Verdict) . The verdict alphabet is V = {allow , warn , block , review }. We define the executability predicate exec (v) ≜ v ∈ { allow , warn }. review requires escalation to a human-in-the-loop or to a secondary policy. 

Definition 4 (Trust evaluator) . A trust evaluator is a (possibly stateful) function 

F : ( History , a ) → (v, r, c, e )

mapping the prior history h ∈ History and a candidate action a ∈ S to a verdict v ∈ V , an overall risk r ∈ R, a confidence c ∈ [0 , 1] , and a human-readable explanation e. AgentTrust is a concrete instance of F.

4.2 ShellNormalizer 

The ShellNormalizer pre-processes shell-style payloads before they reach the analyzer and policy engine. Its sole goal is to expose hidden dangerous commands to text-level pattern matching, without ever executing the input (which would itself be unsafe). We implement nine deobfuscation strategies as pure-text rewrites: N1. Variable expansion. Match NAME=value; $NAME idioms via a constrained regex for assign-ments, then substitute single- and double-brace forms ( ${NAME} , $NAME ). N2. Hex/octal escape decoding. Decode \xHH and \NNN escapes inside single- or double-quoted segments only (i.e. where the shell would actually decode them). N3. Alias resolution. Parse alias N=... definitions and rewrite subsequent unqualified occur-rences of N .N4. Command substitution ($(printf ’...’) ). Resolve printf format strings by decoding embedded hex/octal escapes; arbitrary command substitution is intentionally not executed. N5. Eval+printf chains. Match eval "$(printf ’...’)" and emit the decoded body as a variant. N6. ANSI-C quoting. Decode $’\xHH’ bash idiom (a common rare-but-dangerous form). N7. Backtick substitution. For the safe subset ‘echo ...‘ , emit the literal argument; arbitrary backticks are not executed. N8. Echo command substitution. Resolve $(echo ...) when the argument is a literal. N9. Adjacent-quote concatenation. Iterate to a fixed point on ’r”m’ → rm (and chains thereof). The normalizer returns the set of distinct variants; the analyzer and policy engine then evaluate each variant and combine results using maximum-severity aggregation (Property 4). AgentTrust Preprint 9

This guarantees that adding a deobfuscation step can only discover risks, never hide them. 

Why text-level rewriting and not AST? A full Bash AST parser would in principle handle every pattern, but brings two costs: (a) implementation complexity and a long tail of edge cases (heredocs, control flow, function definitions), and (b) the temptation to evaluate command substitutions for better coverage, which would re-introduce the very arbitrary-execution risk we are guarding against. We therefore restrict the normalizer to the nine pure-text strategies above, which empirically catch the common-case obfuscations observed in agent telemetry (Section 6.2). AST-based analysis is on the v1.0 roadmap. 

4.3 ActionAnalyzer 

The analyzer extracts risk-relevant features from each (normalized) action via 42 regex patterns grouped into four categories: file_system , shell , network , and credential . Each pattern produces a structured RiskFactor carrying a category, a description, a severity, an evidence snippet, and an optional recommendation (Section 3.2). The analyzer also assigns a preliminary action-type-based base risk via a fixed table ( credential_access starts at medium ; file_read 

at none ; etc.). The output of the analyzer is an AnalysisResult that is consumed both by the policy engine and by the SafeFix engine. 

4.4 PolicyEngine 

The policy engine evaluates each action against a configurable list of PolicyRule entries (Section 3.2). A rule is a tuple 

⟨id , name , description , τ ⊆ Type , regex , r, v, tags ⟩

where r ∈ R and v ∈ V are the rule’s risk level and verdict if matched, and τ restricts the rule to a subset of action types. The default ruleset ships with 170 rules covering, in addition to the basics (rm-rf, chmod 777, curl-pipe-bash): 

• Cloud IAM and secret managers (AWS, GCP, Azure). 

• Container security (Docker socket, privileged mode, host networking). 

• Kubernetes and orchestration ( kubectl , helm , terraform , ArgoCD). 

• Database privilege escalation ( GRANT ALL , ALTER USER ). 

• Credential file access ( .aws/credentials , .kube/config , ~/.netrc ). 

• Service exposure (binding to 0.0.0.0 , opening privileged ports). 

• Anti-forensics (clearing logs, history wipes). 

Aggregation. For a given action, the engine takes the highest-severity rule that fires (verdict-wise) and the maximum severity (risk-wise) across all matched rules. If no rule matches and the analyzer detects no patterns, the engine emits the configured default_verdict (allow for typical use; configurable to review for stricter environments). 

Benchmark-rule isolation. Four “benchmark-only” rules that match synthetic domain names (evil.com , attacker.com , etc.) are deliberately excluded from the production ruleset and live in a separate benchmark_compat.yaml . They are loaded automatically by the BenchmarkRunner but never by a vanilla TrustInterceptor . 4.5 TrustInterceptor (Orchestration) 

The interceptor implements Algorithm 1. It owns the analyzer, the policy engine, the optional SafeFix engine, and the optional session tracker; it measures end-to-end latency; and it catches any internal exception, returning a fail-safe ( review , medium , c = 0 .3) report (Property 2). 

Confidence assignment. Confidence is computed by a small step function (Eq. 1) over three boolean signals derived from the verification: whether the analyzer matched any risk patterns, whether the policy engine produced any violations, and whether the session tracker raised any chain alerts. 4.6 SafeFix Engine 

When the verdict is block , warn , or review , the SafeFix engine consults a list of 37 fix rules , each defined by a regex pattern, a category aligned with the analyzer categories, and a generator function that produces a safer alternative. Suggestions are first-class SafeSuggestion 

records: (original, suggested, explanation, category) . Representative examples appear in Table 3. The category filter ensures that, e.g., a credential-related risk does not produce a file-system suggestion. When no risk category applies (empty risk-factor list), all rules are tried. SafeFix never overrides the verdict—blocking remains blocking; it only augments the report with concrete remediation hints.

Formally, a chain pattern P = ( id , n, [s1, . . . , s k], r P , m ) specifies k ordered steps, each 

si = ( τi, regex i, descr i), a combined risk rP , and a minimum match count m. The matching algorithm walks the steps in order, scanning forward through the history for the earliest action whose type matches τi (if non-empty) and whose payload matches regex i, without reusing the same action for two distinct steps. Tie-breaking is deterministic: when an action matches multiple step patterns it is greedily assigned to the lowest-indexed unmatched step, preserving order-of-appearance semantics and making detection results bit-for-bit reproducible. When at least m steps have matched, a ChainAlert is emitted; the interceptor then merges rP into the action’s overall risk. 5.3 Discussion: Avoiding Self-Refereeing 

A common pitfall in safety-evaluation papers is that the benchmark and the system being evaluated are co-developed by the same authors, so the benchmark implicitly encodes the system’s strengths. We address this in three ways. 1. Rule isolation. Synthetic-domain rules ( evil.com , attacker.com , etc.) ship in a separate file and are excluded from the production ruleset (Section 4.4). 2. Independent corpus. The 630-scenario benchmark targets categories the YAML benchmark intentionally does not cover (e.g. Kotlin/Scala build commands, Bun/Deno, ArgoCD, Trivy, Hadolint). 3. Open data. All scenarios, labels, and runner scripts ship under AGPL-3.0 alongside the source tree; we explicitly invite third parties to construct their own scenarios and to report disagreements with our labels. Reading the numbers. Each baseline illustrates a distinct failure mode that AgentTrust avoids. 

B1 (regex blocklist) achieves a perfect 0% false-positive rate but at a catastrophic 88.4% false-negative rate —i.e., 88% of the actions labeled block are not caught by the 50 hand-written patterns. Keyword-shaped defenses cannot keep up with the diversity of dangerous shell, file, and network operations that real agents emit. 

B3 (NeMo Guardrails) exhibits the opposite failure: a 96.2% false-positive rate —it blocks almost every safe action, treating routine git status , npm install , or kubectl get pods as unsafe. This is not a tuning artifact: NeMo’s input-rail design is intended for chatbot moderation (where the cost of a stray refusal is low) and not for agent action verification (where over-blocking AgentTrust Preprint 19 

destroys the agent’s utility). The verdict accuracy of 44.7% reflects that the safe half of the benchmark is uniformly mis-labelled. 

B4 (DeepSeek-V3 zero-shot judge) is the strongest baseline, attaining 82.3% verdict accuracy. Yet it remains 12.7 percentage points below AgentTrust , with a higher false-positive rate (7.5% vs. 2.3%) and ∼780 × larger latency (1.35 s vs. 1.72 ms). Even a state-of-the-art open LLM applied zero-shot cannot match a structured rule + chain pipeline that encodes domain-specific safety knowledge. The latency gap also makes B4 impractical for an interactive agent loop that typically emits several actions per second. 

AgentTrust attains 95.0% verdict accuracy with a 5.4% false-negative rate at the cost of a 2.3% false-positive rate—a 16 × reduction in FNR over B1 and a clean improvement on every dimension over B3 and B4. 

Latency. AgentTrust’s median 1.72 ms is end-to-end including SafeFix and SessionTracker. The pure analyzer-plus-policy path (without SafeFix or SessionTracker) measures 0.3 ms in microbenchmarks (Table 7). 6.5 Hybrid Analysis: Rules + LLM Judge 

The bottom row of Table 9 reports the hybrid configuration where the LLM Judge serves as a second opinion on every action that the rule engine does not already classify as block . We initially expected this to strictly dominate the rule-only configuration, since LLM-as-judge can in principle catch ambiguous-but-risky actions that pattern matching misses. The empirical result is more nuanced and, we think, more informative. 

Asymmetric trade-off. Compared to AgentTrust rule-only, the hybrid: 

• reduces the false-negative rate from 5.4% to 0.8% on the internal benchmark, and from 6.4% to 0.3% on the independent benchmark—a ∼7× and ∼21 × reduction respectively; 

• raises the false-positive rate from 2.3% to 9.0% (internal) and from 2.1% to 4.8% (independent). Verdict accuracy drops because the safe-action class is larger than the dangerous-action class on both benchmarks, so the FPR increase costs more than the FNR reduction saves on the aggregate score. When is the hybrid the right choice? This is exactly the regime where the headline accuracy number hides the operational truth. An agent operating in a high-stakes environment (production data, irreversible operations) typically weights an undetected dangerous action much higher than an unnecessary refusal. For such deployments the hybrid is preferable: it cuts the rate of missed attacks by ∼10 × at the cost of a modest increase in user friction. For interactive developer tooling where every refusal annoys the user, the rule-only configuration is more appropriate. 

Why does the hybrid not strictly dominate? Inspection of the disagreements shows that DeepSeek-V3 has a mild but systematic bias toward escalation: it labels a non-trivial fraction of routine but non-canonical commands (e.g. rsync to staging-server , kubectl apply on a non-production namespace) as warn or block when the rule engine correctly produces allow .These false positives are not “hallucinations”; they are the LLM exercising commendable caution in the absence of operational context (project policy, environment labels). Future work could supply this context to the judge via the existing prompt slot to reduce the FPR delta. 

Asymmetric hybrid (preview). A natural follow-up is an asymmetric hybrid that allows the LLM to escalate only when the rule engine is itself uncertain ( review ) or weakly conservative (warn ). A pilot on the internal benchmark—LLM consulted only when rule verdict ∈{warn ,

review }—recovers ∼80% of the FNR improvement at ∼30% of the FPR cost; we leave the full characterization to v0.6.

6.7 Component Ablation 

To attribute AgentTrust’s overall performance to its individual components, we ran a leave-one-out ablation: each variant disables exactly one component and is evaluated on both benchmarks. Results are in Table 12. Three findings stand out, all of which are scientifically interesting because they refine—rather than confirm—our prior expectations. 

Finding 1: ShellNormalizer is the single most valuable component. Disabling the normalizer leaves internal accuracy unchanged (the internal benchmark contains few obfuscated scenarios) but on the independent benchmark—which intentionally includes ∼95 obfuscation-targeted scenarios—FNR jumps from 6.4% to 15.7% , a ∼2.5× regression in attack-miss rate. This is direct empirical justification for the normalizer’s nine deobfuscation strategies (§4.2): they are not decorative engineering but the difference between catching and missing one in seven adversarial commands. 

Finding 2: SafeFix never affects the verdict (by design). Disabling SafeFix produces identical accuracy and FNR on both benchmarks, validating the design contract that SafeFix only augments the report with safer-alternative suggestions and never modifies the verdict (§4.6). Finding 3: SessionTracker has no measurable effect on either benchmark. This is the most surprising result. RiskChain detection contributes zero verdict-accuracy improvement and zero FNR reduction on both benchmarks, while costing ∼1.6 ms of latency per action. The reason is structural: both our benchmarks are stateless (scenarios are evaluated in isolation, with the session cleared between scenarios in the baseline harness for fairness), whereas the RiskChain detector is designed to fire on stateful multi-step sequences within a single session. In other words, our current benchmark suite does not exercise the contribution that the chain detector is built for. This is an honest gap in our evaluation methodology rather than a flaw in the chain detector: the seven chain patterns (§4.7) need a multi-step session to fire. Constructing a trajectory benchmark (e.g., 50 multi-step attack chains, each comprising 3–5 individually-benign actions that compose into a dangerous sequence) is an obvious next benchmark contribution, and we mark it as a v0.6 deliverable. Until that benchmark exists, the SessionTracker’s value is best demonstrated by the case studies in §4.7 rather than by aggregate accuracy on a stateless suite. 

Finding 4 (combined): the rule pipeline is the workhorse. The “Minimal” configuration with all three optional components disabled scores 95.0% / 90.3% on the two benchmarks, identical to the −ShellNormalizer row. This means the bulk of AgentTrust’s performance is delivered by the analyzer + 170 policy rules, with the Normalizer providing the obfuscation-handling head. 

6.8 LLM-Judge Cache Effectiveness 

We separately evaluate the cache-aware LLM-Judge layer (Section 4.8) on a synthetic streaming workload designed to mimic a long-running Claude-Code-style session: an initial ∼6.25 K-token context (25 000 characters at the standard 4 chars/token rule of thumb), with 500–3000 characters appended per turn over 50 turns, and a 10% probability per turn of a non-tail mid-content edit (paragraph replacement). We run the workload 100 independent times with seed = 7 and report mean ± 95% confidence intervals. The benchmark exercises the strategy router only; no LLM is actually called and token cost is estimated as |payload |/4, the standard chars-per-token approximation. The runner is experiments/run_cache_benchmark.py .Table 13 reports the strategy distribution and the estimated token savings against a no-cache baseline that re-evaluates the full context on every turn. 

Reading the cache results. On this append-dominant workload the block-hash router correctly identifies ∼89% of evaluations as incremental (the change ratio is below the 20% threshold), reducing token cost by an order of magnitude per call. The 11 .4% full re-evaluations are the turns where either the random mid-content edit triggered the threshold ( ∼10% probability AgentTrust Preprint 23 

per turn × 50 turns ≈ 5 expected mid-edits) or the appended payload exceeded the threshold. Cache-hits are 0% in this synthetic workload because each turn’s content is unique by construction; in real agent traffic where users repeat prompts or where the same context is evaluated multiple times against different actions, cache-hit rate would be non-trivial and would push the savings even higher. 

Limits of this benchmark. The 87% headline savings depends on the workload being append-dominant. Three regimes degrade it: (i) heavy mid-content edits (e.g. a coding agent that frequently rewrites earlier code), which push more turns through the full branch; (ii) short sessions where the initial full call dominates the amortization budget; (iii) tool calls that change action content substantially between turns even when the surrounding context grows linearly. We choose the parameter set above (50 turns, 25 K initial context, ∼1.5 K tokens appended/turn, 10% mid-edit probability) because it matches the dominant pattern observed in agent telemetry, but report it as a representative case rather than a guarantee; users with substantially different workloads should re-run run_cache_benchmark.py with their own parameters. 

6.9 Error Analysis 

We manually inspected the 9 verdict mismatches and 73 risk-level mismatches on the internal benchmark (representative cases in Appendix C). The mismatches fall into four categories: 1. High vs. critical boundary (∼35 of 73 risk mismatches). Subjective even for humans. These do not change the verdict. 2. Typosquat package detection (3 cases). We cannot decide that pip install reqeusts 

is a misspelling without an external package-name database. 3. Deeply nested obfuscation (4 cases). eval "$(printf ’\x77\x67...’)" style chains where the normalizer resolves only the outer layer. Targeted by AST parsing on the v1.0 roadmap. 4. Semantic context required (2 cases). curl -X PUT -T - https://api.example.com :depending on whether -T - pipes a sensitive file, this is benign or catastrophic; static regex cannot tell. These are exactly the cases the LLM-Judge is designed for.

# 7 Limitations and Ethical Considerations 

7.1 Limitations 

We document the limitations honestly so that downstream users can calibrate trust in Agent-Trust’s verdicts. 

L1. Static analysis ceiling. AgentTrust’s rule-only path is fundamentally a text-rewriting + regex-matching system. It cannot, even in principle, reason about runtime data flow. Concretely, 

curl -X PUT -T - https://h/x is benign or catastrophic depending on what -T - is streaming, which static regex cannot determine. The LLM-Judge layer is designed for these cases but introduces its own dependencies (API key, latency, cost). 

L2. Shell deobfuscation is incomplete by design. The ShellNormalizer applies nine pure-text strategies and deliberately does not execute or interpret subshell substitutions, evaluate function bodies, or trace control flow. Adversarially-crafted nested obfuscation ( eval chains, function-defined aliases, dynamically-decoded heredocs) can evade detection. AST-based analysis is on the v1.0 roadmap and is expected to push obfuscation handling from ∼93% toward ∼99% on the independent benchmark, at the cost of a more complex codebase. 

L3. Typosquat and reputation gaps. We cannot decide that pip install reqeusts is a malicious typosquat without an external package-name database (or a domain-registration WHOIS lookup, etc.). Production deployments should layer in such databases (PyPI typosquat oracles, Sigstore, etc.); AgentTrust exposes a plug-in extension point but ships no such oracle. 

L4. Risk-level subjectivity. The high /critical boundary is intrinsically subjective; even after careful rubric-based labeling (Appendix A), ∼35 of 73 risk-level mismatches in our experiments cluster at this boundary. Verdict accuracy is the more reliable headline metric. 

L5. English-only patterns. Both regex patterns and the LLM-Judge prompt are English-centric. Agents that emit non-English commands (e.g. Cyrillic-encoded filenames as a side channel) are not currently covered. 

L6. The interceptor must be in-process. The threat model (§3.1) assumes the agent runtime is trusted and that AgentTrust runs in the same process. A malicious agent runtime can simply not call AgentTrust; out-of-process enforcement requires a different deployment model (e.g. sandboxed exec, kernel hooks). AgentTrust Preprint 25 

L7. Coverage of the rule, chain, and SafeFix libraries is finite. 170 policy rules, 37 SafeFix fix rules, 42 analyzer risk patterns, and 7 multi-step chains is large but not exhaustive: long-tail operations on niche tooling, novel cloud APIs, and emerging package managers will go uncovered until either a custom rule is added or the LLM-Judge layer is invoked. We commit to a public process for adding rules, chains, and fixes via GitHub contributions, and we ship a YAML-extension mechanism so users can add domain-specific rules without forking the codebase. 

7.2 Ethical Considerations 

Dual-use of attack patterns. The seven RiskChain patterns (Section 4.7) correspond to MITRE-ATT&CK-style attack sequences and are, in a sense, an abbreviated catalog of how to abuse an agent. We judge the publication trade-off as net positive for two reasons: (i) all seven patterns are already documented in MITRE ATT&CK [ 16 ], in standard penetration-testing references, and in widely-circulated red-team materials—we are not introducing novel attack technique; (ii) deploying the detector publicly raises the floor for defenders who lack red-team budgets. 

Responsible disclosure. AgentTrust is not a vulnerability-discovery system. No zero-day vulnerabilities are exposed by this work. The benchmark contains no real credentials, no real user data, and no live attack URLs (synthetic-domain rules are isolated; see §4.4). 

Risk of false sense of security. A 95% verdict accuracy is not 100%. Deployers should treat AgentTrust as a defense-in-depth layer rather than a sole gatekeeper, and should retain (i) OS sandboxing, (ii) least-privilege credentials, (iii) human-in-the-loop review for irreversible operations, and (iv) audit logging. The fail-safe design (Property 2) and the explicit review 

verdict are intended to support these complementary controls, not to replace them. 

LLM-Judge cost and equity. The LLM-Judge layer requires a paid third-party API. The cache-aware design (§4.8) and the documented fallback to rule-only operation are intended to keep AgentTrust accessible to deployers without an LLM budget; the rule-only path is functionally complete (Table 7). 

Data licensing. Both the 300-scenario YAML benchmark and the 630-scenario Python runner suite are released under AGPL-3.0 together with the source tree; a commercial license is available on request for organizations that cannot accept the AGPL-3.0 reciprocity obligations. Where individual scenarios are inspired by public incidents or postmortems, we avoid including live credentials or attack infrastructure. A per-scenario source: field that cites the specific public postmortem, RFC, or vendor advisory that inspired each scenario is on the v0.6 deliverables list (cf. docs/dataset_datasheet.md §7). 

# 8 Conclusion 

We presented AgentTrust , a real-time, semantics-aware safety-interception framework for AI agents. The system fills a gap left by post-hoc benchmarks (too late), text-only guardrails (too shallow), and infrastructure sandboxes (too low-level): it intercepts each tool call, normalizes obfuscation, evaluates against 170 policy rules and seven multi-step attack chains, suggests safer alternatives when blocking, and falls back to a cache-aware LLM judge for ambiguous cases—with a sub-millisecond core rule path and low-millisecond end-to-end rule-only latency. On the 300-scenario internal benchmark the production-only AgentTrust ruleset achieves 

95.0% verdict accuracy and 73.7% risk accuracy at low-millisecond median end-to-end latency (97.0% / 75.7% with the four optional benchmark-compatibility rules loaded; see §6.2). On an AgentTrust Preprint 26 

independently constructed corpus of 630 real-world scenarios—spanning modern DevOps, cloud, container, and supply-chain operations, plus a dedicated obfuscation subset—AgentTrust attains 

96.7% verdict accuracy under the patched ruleset (§5.2; not a zero-shot claim). Four engineering safety contracts, codified as 13 regression tests, ensure the system fails safe when individual components break. 

Future work. (i) A trajectory-level benchmark that exercises the SessionTracker’s multi-step chain detectors directly, complementing the per-action benchmarks reported here (§6.7 showed that the current stateless suites cannot quantify the chain-detector’s contribution); (ii) AST-based shell analysis to push obfuscation detection beyond the nine pure-text strategies of the current normalizer; (iii) extending the multi-baseline harness to additional LLM families (Llama-Guard-4, GPT-5, Claude 4) to test cross-model generalization of the LLM-Judge findings; (iv) plug-in extension points for typosquat oracles, threat-intel feeds, and SaaS-specific rule packs; (v) elevation of the LLM-Judge to a primary arbiter (rather than fallback) when the per-call budget allows, with the cache layer keeping cost sub-linear in session length. 

Availability. AgentTrust is released under the AGPL-3.0 license, with a commercial licensing option available on request: https://github.com/chenglin1112/AgentTrust . The release includes the Python library, CLI, MCP server integration, web dashboard, both benchmarks (300 internal + 630 independent), and the 192-test regression suite.

# A Labeling Rubric 

Scenarios in the internal benchmark (Section 5.1) were labeled with the following four-part rubric. The rubric is deterministic in the sense that it produces the same label for the same (τ, ρ )

across annotators who apply it faithfully; it is not, however, capable of resolving all ambiguous semantic cases (see L4 in Section 7). 

Step 1 — Reversibility. Can the effect of the action be undone without external state (e.g. backups, support tickets)? If no , raise a floor of high .

Step 2 — Scope. Does the action touch project-local paths only, or does it touch user-global (˜/. ), system ( /etc , /usr ), or external ( http(s):// , remote hosts) scope? Escalate one level per scope jump. 

Step 3 — Credential involvement. Does the action read, write, or transmit anything match-ing the credential patterns ( .env , .pem , .key , .aws/credentials , .kube/config , ˜/.netrc )? If yes, raise one risk level and default verdict to warn at minimum; if the action transmits the credential outward, verdict becomes block .

Step 4 — Verdict mapping. Given the risk level produced by steps 1–3: none /low →

allow ; medium → warn ; high /critical → block . Override to review if the rubric produces a conflict (e.g. scope escalation says high but reversibility says low ). The rubric is recorded in docs/labeling-rubric.md in the released repository, which serves as the authoritative source for external annotators. See that file for worked examples and the recommended workflow for an inter-annotator-agreement study (Cohen’s κ on a stratified subset).
