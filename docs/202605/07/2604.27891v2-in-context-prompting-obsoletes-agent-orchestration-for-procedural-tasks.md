## 摘要
智能体编排框架（如 LangGraph、CrewAI、Google ADK、OpenAI Agents SDK 等）在大型语言模型（LLM）之上放置了一个外部编排器，用于跟踪状态并在每一步注入路由指令。我们通过一项对照实验表明，对于程序化任务，这种架构被一种更简单的替代方案所超越：将整个流程放入系统提示词中，并让模型进行自我编排。在旅游预订（14 个节点）、Zoom 技术支持（14 个节点）和保险理赔处理（55 个节点）这三个领域中，我们针对每种情况评估了 200 场对话，并使用 LLM 作为评判（LLM-as-judge）根据五个质量标准进行评分。结果显示，在 5 分制评分中，上下文方法得分为 4.53-5.00，而使用相同模型的 LangGraph 编排器得分为 4.17-4.84。编排系统在旅游、Zoom 和保险对话中的失败率分别为 24%、9% 和 17%，而上下文基准方案的失败率仅为 11.5%、0.5% 和 5%。虽然外部编排对于早期模型可能是必要的，但前沿模型能力的提升使得它在遵循既定流程的多轮对话中已不再必要。

## Abstract
Agent orchestration frameworks -- LangGraph, CrewAI, Google ADK, OpenAI Agents SDK, and others -- place an external orchestrator above the LLM, tracking state and injecting routing instructions at every turn. We present a controlled comparison showing that for procedural tasks, this architecture is dominated by a simpler alternative: putting the entire procedure in the system prompt and letting the model self-orchestrate. Across three domains -- travel booking (14 nodes), Zoom technical support (14 nodes), and insurance claims processing (55 nodes) -- we evaluate 200 conversations per condition using LLM-as-judge scoring on five quality criteria. The in-context approach scores 4.53--5.00 on a 5-point scale while a LangGraph orchestrator using the same model scores 4.17--4.84. The orchestrated system fails on 24% of travel, 9% of Zoom, and 17% of insurance conversations, compared to 11.5%, 0.5%, and 5% for the in-context baseline. While external orchestration may have been necessary for earlier models, advances in frontier model capabilities have made it unnecessary for multi-turn conversations following a defined procedure.

*   **研究动机**：目前业界普遍采用外部编排框架（如 LangGraph, CrewAI, Semantic Kernel 等）来管理 LLM 智能体。这些框架通过外部逻辑跟踪状态并在每一步注入路由指令。然而，这种架构增加了工程复杂度（如“七层代码抽象”），且容易导致错误累积和路由失败。
*   **核心问题**：随着前沿模型（Frontier Models）推理能力和上下文窗口的提升，这种复杂的外部编排是否还有必要？
*   **整体含义**：论文提出，对于遵循特定流程的程序化任务（Procedural Tasks），直接将整个流程图放入系统提示词中让模型“自编排”，其表现全面优于复杂的外部编排框架。

### 2. 方法论
*   **核心思想**：对比“外部编排（Orchestrated）”与“上下文内提示（In-Context Prompting）”两种架构。
*   **关键技术细节**：
    *   **程序表示**：将任务定义为有向图 $F = (N, E, n_0, T, D)$，包含节点（Agent/User 角色）、边（转换条件）、起始点、终止点和决策中心（Decision Hubs）。
    *   **外部编排组（LangGraph）**：每个流程节点对应一个图节点。每当到达决策中心时，编排器会额外调用一次 LLM 来决定下一步走向，并动态注入当前节点的提示词模板。
    *   **上下文内基准组（In-Context）**：将整个流程图序列化为结构化文本（包含所有节点、边和条件），直接放入系统提示词中。模型在每一轮对话中自行判断所处阶段并决定回复内容，无需外部状态跟踪或路由调用。
*   **算法流程**：两种方案均使用相同的底层模型（Claude Sonnet），在相同的初始场景下与模拟用户进行多轮对话。

### 3. 实验设计

    3.  **保险理赔处理**：55个节点，高复杂度，包含 6 个多路决策中心，涉及理赔评估、覆盖范围判定、谈判等。
*   **Benchmark 与对比方法**：
    *   **对比对象**：LangGraph 编排架构 vs. 上下文自编排架构。

### 5. 实验数量与充分性
*   **实验数量**：1,200 场对话是一个相当大的样本量，足以进行统计学显著性分析（所有 15 项对比均达到 $p < 0.005$）。
*   **充分性**：实验覆盖了从简单（14 节点）到复杂（55 节点）的流程，并引入了独立的第三方模型（GPT-4.1）作为裁判，有效排除了“模型自我偏好”的偏差，实验设计客观且公平。

### 6. 主要结论与发现
*   **全面胜出**：在所有领域和所有指标上，上下文内提示方案均显著优于 LangGraph 编排。
*   **失败率对比**：在保险理赔中，编排系统的失败率为 17%，而上下文方案仅为 5%；在 Zoom 支持中，编排系统失败率（9%）是上下文方案（0.5%）的 18 倍。
*   **效率瓶颈**：编排框架由于需要额外的路由调用，导致 LLM 调用次数增加 1.2-1.7 倍，增加了延迟且降低了质量。
*   **核心发现**：编排框架会“碎片化”模型的推理能力。模型只看到当前节点的局部信息，失去了对全局对话弧线的把握；而上下文方案允许模型进行整体推理。

### 7. 优点
*   **架构极简**：证明了“少即是多”，减少了开发者在复杂框架上的学习和维护成本。
*   **鲁棒性强**：消除了由于外部路由逻辑错误导致的循环或过早终止问题。
*   **性能卓越**：在一致性和自然度上表现更好，更像真实的人类专家。

### 8. 不足与局限
*   **上下文窗口限制**：虽然 55 个节点仅占用了约 2% 的上下文窗口，但对于极其庞大（如数千个节点）的流程，可能会面临 Token 溢出或性能下降。
*   **成本略高**：由于每轮对话都携带完整流程，Token 成本比编排方案高出约 30%-40%。
*   **适用范围**：研究仅针对“程序化任务”。对于需要多模型协作、复杂工具调用或非结构化探索的任务，外部编排可能仍有其价值。
*   **合成数据风险**：实验使用的是模拟用户而非真实人类用户，可能无法完全捕捉人类行为的不可预测性。
