## 摘要
大语言模型（LLM）智能体在处理多步任务时越来越依赖技能库，然而，随着技能的添加、重用、修补以及与不断变化的依赖项建立链接，这些库可能会积累持久性缺陷。我们将这种失效模式称为“技能技术债”：即库级别的缺陷，它们可能不会在局部破坏单个技能，但会损害未来的检索、组合和执行。现有的基于技能的智能体主要关注任务时的检索、规划和修复，而库时的维护仍未得到充分探索。我们提出了 SkillOps，这是一个用于维护技能库的与方法无关的插件框架。SkillOps 将每个技能表示为一个有类型的技能合约（P, O, A, V, F），使用分层技能生态系统图组织技能，并从效用、兼容性、风险和验证维度诊断库的健康状况。给定一个原始技能库，SkillOps 会生成一个经过维护的库，现有的检索或规划智能体无需更改其内部代码即可使用该库。在 ALFWorld 上，SkillOps 作为独立智能体实现了 79.5% 的任务成功率，在不增加额外任务时大语言模型调用的情况下，比最强的基准模型高出 8.8 个百分点。作为插件层，它将重检索基准模型的性能提升了 0.68 到 2.90 个百分点。目前的基于规则的维护实现几乎不使用库时的大语言模型调用或 Token，这表明技能库维护可以作为一个低开销的架构层添加。

## Abstract
Large language model agents increasingly rely on skill libraries for multi-step tasks, yet these libraries can accumulate persistent defects as skills are added, reused, patched, and linked to changing dependencies. We call this failure mode skill technical debt: library-level defects that may not break a single skill locally but can harm future retrieval, composition, and execution. Existing skill-based agents mainly focus on task-time retrieval, planning, and repair, while library-time maintenance remains underexplored. We propose SkillOps, a method-agnostic plug-in framework for maintaining skill libraries. SkillOps represents each skill as a typed Skill Contract (P, O, A, V, F), organizes skills with a Hierarchical Skill Ecosystem Graph, and diagnoses library health across utility, compatibility, risk, and validation dimensions. Given a raw skill library, SkillOps produces a maintained library that can be used by existing retrieval or planning agents without changing their internal code. On ALFWorld, SkillOps achieves 79.5 percent task success as a standalone agent, outperforming the strongest baseline by 8.8 percentage points with no additional task-time large language model calls. As a plug-in layer, it improves retrieval-heavy baselines by 0.68 to 2.90 percentage points. The current rule-based maintenance implementation uses nearly zero library-time large language model calls or tokens, showing that skill-library maintenance can be added as a low-overhead architectural layer.

### 1. 核心问题与研究动机
*   **核心问题**：论文识别并定义了**“技能技术债”（Skill Technical Debt）**。随着大模型智能体运行时间的增加，其技能库会因不断添加、修补和重用技能而积累冗余、接口漂移、验证缺失或实现陈旧等缺陷。
*   **研究动机**：现有的研究多集中在“任务执行时”（Task-time）的检索、规划和修复，而忽视了“库维护时”（Library-time）的长期管理。如果库本身存在缺陷（如缺失验证器），任务时的临时修复无法根除问题，导致未来重复失败。

### 2. 方法论：SkillOps 框架
SkillOps 将技能库视为一个自维护的软件生态系统，核心包含以下要素：
*   **技能合约（Skill Contract）**：将每个技能定义为五元组 $(P, O, A, V, F)$，分别代表前置条件、执行操作、产出物、验证器和已知失败模式。
*   **层级技能生态图（HSEG）**：构建包含四种类型边的图结构：依赖（dep）、兼容性（comp）、冗余（red）和替代（alt）。
*   **双环路机制**：
    *   **任务时环路（Task-Time Loop）**：基于 HSEG 进行受约束的规划，在规划阶段插入适配器（Adapter）和验证器（Validator），并进行局部修复。
    *   **库时环路（Library-Time Loop）**：核心维护阶段。通过**五维健康诊断**（效用、冗余、兼容性、失败风险、验证间隙）评估技能，并执行维护动作（合并、修复、退役、添加验证器/适配器）。
*   **CGPD 算法**：一种基于收缩映射原理的风险传播算法，能将上游技能的风险传播至下游，实现预防性维护。

### 3. 实验设计
*   **数据集与场景**：使用 **ALFWorld**（基于文本的家庭操纵任务基准）。
*   **技能库构建**：基于 SkillsBench 的 229 个真实技能，通过注入 6 种技术债模式（如冗余克隆、错误接口等）扩展至 200 到 2000 个技能规模。
*   **对比方法（Baselines）**：
    *   **ReAct**：全量 Prompt 注入。
    *   **LLM_Skill_Planner**：基于 GPT-4o-mini 的语义排名规划。
    *   **Hybrid_Retrieval**：结合 BM25 和向量检索。
    *   **GoS_Style**：基于依赖图的检索。
    *   **SkillWeaver**：具备任务时自我修复能力的基准。

### 4. 资源与算力
*   **硬件设施**：实验运行在配备 3 个 NVIDIA RTX 2080 Ti GPU（11GB 显存）和 3 个 Intel Xeon CPU 的机器上。
*   **模型调用**：所有 LLM 推理均通过 OpenAI API 调用 **gpt-4o-mini**。
*   **成本与时长**：全量评估耗时约 6 小时，总 API 成本仅为 0.27 美元。维护过程主要基于规则，在 CPU 上运行，2000 个技能的维护耗时不足 1 秒。

### 5. 实验数量与充分性
*   **实验规模**：涵盖了 9 种不同的技能库规模（200-2000），每组实验使用 3 个独立随机种子，每种子包含 185 个任务实例。
*   **消融实验**：详细测试了移除任务环路、库环路、外部图、内部合约以及各项维护动作（如修复、验证器添加等）对性能的影响。
*   **充分性评价**：实验设计非常充分且客观。通过“噪声分级”压力测试验证了系统在极端技术债环境下的稳健性，并分析了 Token 消耗和检索 Top-k 的敏感性。

### 6. 主要结论与发现
*   **性能卓越**：作为独立智能体，SkillOps 在 ALFWorld 上达到 79.5% 的成功率，比最强基线高出 8.8%。
*   **插件增益**：作为插件层，它能显著提升重检索型基线的性能（最高提升 2.9pp），且通常能减少任务时的 Token 消耗（通过剪枝冗余技能）。
*   **低开销维护**：基于规则的维护逻辑几乎不产生额外的 LLM 调用成本，证明了库维护可以作为一种低成本的架构层存在。
*   **规模稳定性**：随着技能库规模从 200 增加到 2000，SkillOps 的性能保持稳定，而传统检索方法因噪声增加而显著下降。

### 7. 优点与亮点
*   **架构创新**：首次将“技术债”概念引入智能体技能管理，填补了库级别维护的空白。
*   **即插即用**：SkillOps 输出的是优化后的标准技能库，下游智能体无需修改代码即可受益。
*   **极高效率**：通过类型检查和哈希碰撞等规则实现维护，避免了昂贵的 LLM 持续推理开销。
*   **理论严谨**：引入 CGPD 算法并证明了其收敛性，为风险传播提供了数学支撑。

### 8. 不足与局限
*   **依赖结构化定义**：目前高度依赖 PDDL 风格的技能合约定义，在缺乏结构化文档的真实场景中可能受限。
*   **噪声模拟偏差**：实验中的技术债是人工合成注入的，可能无法完全覆盖真实生产环境中复杂的软件演化问题。
*   **语义理解局限**：当前的 V4 版本主要依赖规则，可能漏掉一些需要深度语义推理才能识别的冗余或冲突。
*   **应用领域单一**：目前仅在 ALFWorld 具身智能场景验证，需进一步在 Web 自动化、软件工程等领域测试。
