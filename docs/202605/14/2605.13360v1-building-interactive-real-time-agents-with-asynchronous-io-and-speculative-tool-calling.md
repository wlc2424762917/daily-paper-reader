## 摘要
针对客户服务和个人助手等一系列下游应用，对智能体 AI（agentic AI）技术的需求日益增长。对于需要智能体与人互动的应用，实时低延迟响应至关重要；例如，在语音控制应用中，通常需要低于 1 秒的延迟才能保证交互的无缝感。然而，如果要求大语言模型（LLM）进行推理并执行包含工具调用的智能体工作流，这可能会增加数秒甚至更长的延迟，这对于实时延迟敏感型应用来说是难以接受的。在本研究中，我们的目标是使具有复杂多轮工具调用的智能体也能实现实时交互。我们提出了异步 I/O（Asynchronous I/O），它将智能体核心的“推理-行动”线程与等待来自用户或环境的额外信息的过程解耦，从而允许在等待外部延迟时重叠进行智能体处理。我们还提出了推测性工具调用（Speculative Tool Calling），作为一种在智能体尚不确定是否已接收到完整信息，或后续是否会提供额外用户信息时管理任务执行的方法。对于强大的云端模型，我们的方法可以开箱即用地应用于现有的实时云端 API，在精度损失极小的情况下提供 1.3-1.7 倍的加速。为了在小型边缘级模型上实现实时交互，我们还提出了一种基于时钟的训练方法，使模型能够处理流式输入和异步响应，并展示了用于有监督微调（SFT）的合成数据生成策略。总之，该方法在多个工具调用基准测试中，使 Qwen2.5-3B-Instruct 和 Llama-3.2-3B-Instruct 模型获得了 1.6-2.2 倍的加速。

## Abstract
There is a growing demand for agentic AI technologies for a range of downstream applications like customer service and personal assistants. For applications where the agent needs to interact with a person, real-time low-latency responsiveness is required; for example, with voice-controlled applications, under 1 second of latency is typically required for the interaction to feel seamless. However, if we want the LLM to reason and execute an agentic workflow with tool calling, this can add can add several seconds or more of latency, which is prohibitive for real-time latency-sensitive applications. In our work, we aim to enable real-time interaction even for agents with complex multi-turn tool calling. We propose Asynchronous I/O, which decouples the core agent reason-and-act thread from waiting for additional information from either the user or environment, thereby allowing for overlapping agentic processing while waiting on external delays. We also propose Speculative Tool Calling as a method to manage task execution when the agent is still unsure if it has received the full information or if additional user information may later be provided. For strong cloud models, our method can be applied out-of-the-box to existing real-time cloud APIs, providing 1.3-1.7$\times$ speedups with minor accuracy loss. To enable real-time interaction with small edge-scale models, we also present a clock-based training methodology that adapts the model to handle streaming inputs and asynchronous responses, and demonstrate a synthetic data generation strategy for SFT. Altogether, this approach provides 1.6-2.2$\times$ speedups with the Qwen2.5-3B-Instruct and Llama-3.2-3B-Instruct models across multiple tool calling benchmarks.

## 论文详细总结（自动生成）

这篇论文由加州大学伯克利分校（UC Berkeley）等机构的研究人员撰写，旨在解决大语言模型（LLM）智能体在实时交互场景（如语音助手）中因工具调用和推理过程导致的显著延迟问题。

### 1. 核心问题与整体含义
*   **研究动机**：实时语音交互通常要求延迟低于 1 秒，但现有的智能体工作流（如 ReAct）是顺序执行的：等待用户说完 -> 模型推理 -> 调用工具 -> 等待工具返回 -> 生成回答。这种“停-等”模式在处理复杂多轮工具调用时会产生数秒甚至更长的延迟，严重破坏交互体验。
*   **核心目标**：实现一种能够“边听边想边做”的异步智能体，在用户说话的同时开始推理和调用只读工具，从而大幅降低端到端响应时间。

### 2. 方法论
论文提出了 **AsyncIO** 框架，包含以下核心技术：
*   **异步 I/O (Asynchronous I/O)**：将智能体的推理-行动线程与用户输入流、环境反馈流解耦。模型不再等待完整输入，而是根据流式更新（`<partial query update>`）持续生成推理（`<think>`）和动作。当新信息到达时，系统会中断当前生成并注入新上下文。
*   **推测性工具调用 (Speculative Tool Calling)**：
    *   **读写分离**：将工具分为“安全（只读）”和“不安全（修改状态）”。只读工具（如搜索）在获取部分信息后立即推测性执行；修改状态的工具（如发短信）则缓冲至“提交点”（用户说完且逻辑确认后）再执行。
    *   **动态修正**：支持模型通过 `REMOVE ID` 或覆盖相同 ID 的方式，撤销或修改之前基于不完整信息发出的错误工具指令。
*   **时钟感知训练 (Clock-based Training)**：针对 3B 等小模型，提出一种模拟流式环境的训练方法。利用 TTS（语音合成）和 WhisperX（对齐）获取单词级时间戳，将延迟转化为“Token 数量”，训练模型在特定时刻决定是“等待（Wait）”还是“行动”。

### 3. 实验设计
*   **数据集/场景**：
    *   **HotpotQA**：测试多跳搜索工具调用能力。
    *   **TinyAgent**：测试个人助理场景下的多步工作流（如发邮件、定会议）。
    *   **Human Instructions**：论文自建的自然语言数据集，包含口语停顿、纠错和歧义，更贴近真实语音交互。
*   **对比基准 (Benchmark)**：标准的同步“推理-行动（Reason-and-Act）”基准。
*   **对比模型**：云端 API（OpenAI Realtime API）和边缘侧模型（Qwen2.5-3B-Instruct, Llama-3.2-3B-Instruct）。

### 4. 资源与算力
*   **算力设备**：实验在单张 **NVIDIA A100 GPU** 上进行基准测试。
*   **训练细节**：使用 LLaMA-Factory 进行微调（SFT）。训练 1 个 epoch，学习率 3e-5，Batch Size 为 16。
*   **数据生成**：使用 Qwen-2.5-32B-Instruct 作为教师模型来合成推理轨迹和修正行为的数据。

### 5. 实验数量与充分性
*   **实验规模**：针对云端 API 抽取了 500 个样本，针对开源模型在两个基准上各抽取 100 个样本进行延迟评估。
*   **充分性**：实验涵盖了从闭源最强 API 到边缘侧小模型的完整光谱，并对比了有无 SFT 的效果。通过自建的“人类指令数据集”验证了模型在极端真实情况下的鲁棒性，实验设计较为客观、全面。

### 6. 主要结论与发现
*   **显著加速**：AsyncIO 在云端 API 上实现了 **1.3-1.7 倍**的加速；在边缘侧 3B 模型上实现了 **1.6-2.2 倍**的加速。
*   **精度保持**：在大幅降低延迟的同时，任务成功率（准确率）仅有轻微下降（通常在 1-3% 以内）。
*   **小模型潜力**：通过专门的异步 SFT 训练，3B 规模的模型能够学会复杂的推测执行和错误修正逻辑，表现接近同步执行的水平。

### 7. 优点
*   **系统与算法协同**：不仅改进了模型生成逻辑，还设计了配套的事件驱动型推理引擎和任务管理器。
*   **安全性考量**：通过推测执行的“提交点”机制，巧妙解决了自动代理可能误操作（如误发邮件）的风险。
*   **真实性高**：利用 TTS 和对齐技术构建的训练数据非常接近真实的流式语音交互物理特性。

### 8. 不足与局限
*   **自然语言处理挑战**：在处理包含大量填充词（如 "um", "uh"）和中途改口的真实人类语音时，模型准确率下降明显（从 60%+ 降至 15-20% 左右），说明流式理解仍有很大提升空间。
*   **写操作延迟瓶颈**：由于“不安全工具”必须等待提交点，对于这类任务，延迟缩减主要来自于推理过程的重叠，而非工具执行本身的重叠。
*   **退化行为**：在复杂流式输入下，小模型偶尔会出现重复生成动作的退化现象。
