## 摘要
我们推出了 SWE Atlas，这是一个针对编程智能体（coding agents）的基准测试套件，涵盖了三种专业的软件工程工作流：代码库问答（124 个任务）、测试编写（90 个任务）和重构（70 个任务）。SWE Atlas 与之前的软件工程（SWE）基准测试在三个关键方面有所不同：它针对的是代表性不足但具有实际重要性的任务类别；采用了全面的、针对特定类别的评估协议；并采用了描述不充分且具有智能体特征的任务表述，从而更好地反映了现实世界的使用情况。其评估框架结合了程序化检查与基于评分量表的评估。这超越了功能正确性，旨在评估软件工程质量，包括测试和重构的完整性、可维护性、可重用抽象以及代码库整洁度。我们在 SWE Atlas 上评估了一系列前沿模型和开源权重模型，发现 GPT-5.4 和 Opus 4.7 取得了最强的综合性能，而即使是最好的开源权重模型得分也较低。我们的分析表明，顶尖模型依赖于广泛的代码库探索和运行时驱动的推理。然而，即使是顶尖模型在处理微妙的边缘情况、复杂的运行时分析以及遵循软件工程最佳实践方面也始终面临困难。总之，SWE Atlas 提供了一个补充性的评估套件，用于衡量编程智能体的正确性和工程质量。

## Abstract
We introduce SWE Atlas, a benchmark suite for coding agents spanning three professional software engineering workflows: Codebase Q&A (124 tasks), Test Writing (90 tasks), and Refactoring (70 tasks). SWE Atlas differs from prior SWE benchmarks in three key ways: it targets underrepresented but practically important task categories, uses comprehensive category-specific evaluation protocols, and adopts under-specified, agentic task formulations that better reflect real-world usage. Its evaluation framework combines programmatic checks with rubric-based assessment. This goes beyond functional correctness, evaluating software engineering quality, including test and refactor completeness, maintainability, reusable abstractions, and codebase hygiene. We evaluate a range of frontier and open-weight models on SWE Atlas and find that GPT-5.4 and Opus 4.7 achieve the strongest overall performance, while even the best open-weight models score poorly. Our analysis suggests that top models rely on extensive codebase exploration and runtime-driven reasoning. However, even top models consistently struggle with subtle edge cases, complex runtime analysis, and adherence to software engineering best practices. Overall, SWE Atlas provides a complementary evaluation suite for measuring both correctness and engineering quality in coding agents.

### 1. 核心问题与整体含义
*   **研究动机**：当前的编程基准测试（如 SWE-bench）主要关注“功能修复”（Bug 修复或功能实现）。然而，现实中的软件工程（SWE）远不止于此，还包括代码库理解、测试编写和重构。
*   **核心问题**：如果只评估功能修复，可能会导致智能体过度拟合为“补丁工”，而缺乏维护代码健康、防止回归和理解复杂架构的“工程师”素质。
*   **整体含义**：SWE Atlas 填补了这一空白，通过引入更具挑战性、描述不充分（Under-specified）且关注工程质量的任务，推动编程智能体向专业工程师水平进化。

### 2. 方法论
SWE Atlas 包含 284 个由专家编写的任务，涵盖三个核心工作流：
*   **代码库问答 (Codebase Q&A, 124个)**：要求智能体进行深度代码理解和运行时分析（如追踪数据流、分析系统压力下的行为），而非简单的静态代码阅读。
*   **测试编写 (Test Writing, 90个)**：要求为特定行为编写生产级的单元测试、集成测试或验收测试。
*   **重构 (Refactoring, 70个)**：在不改变外部行为的前提下改进代码结构（如解耦、提取模块、迁移接口）。
*   **评估框架（核心创新）**：
    *   **混合验证**：结合了**程序化检查**（如测试套件、变异测试、回归测试）和**基于准则（Rubric）的 LLM 评判**。
    *   **专家准则**：为每个任务制定了详细的二进制评分准则（YES/NO），评估代码位置、反模式、可维护性和清理工作等定性指标。

### 3. 实验设计
*   **数据集/场景**：从 18 个活跃维护的开源项目（如 Grafana, k6, MinIO 等）中提取任务，涵盖 Go, Python, JS/TS, C/C++ 等语言。
*   **对比方法**：
    *   **前沿模型**：GPT-5.4 (Codex), Opus 4.7 (Claude Code), Gemini 3.1 Pro。
    *   **开源模型**：GLM 5, Kimi K2.5, Minimax M2.5。
*   **Benchmark 设置**：对比了模型在“原生脚手架”（如 Claude Code CLI）和“统一最小脚手架”（mini-SWE-agent）下的表现。

### 4. 资源与算力
*   **算力环境**：实验在 Harbor 框架下运行，使用 Modal 沙盒环境。
*   **配置细节**：每个任务分配 16 个 CPU，16GB 内存限制，以及 6 小时的运行时间上限。
*   **训练算力**：文中未提及模型训练的算力，因为该论文侧重于基准测试评估而非模型训练。

### 5. 实验数量与充分性
*   **实验规模**：共 284 个任务，每个配置运行 3 次试验（Trials），计算 Pass@1 和 Pass 3（一致性）。
*   **充分性**：实验涵盖了从闭源顶尖模型到开源模型的广泛对比，并进行了消融分析（如脚手架的影响、成本与能力的权衡）。
*   **客观性**：引入了独立专家审查机制（Quality Control），并进行了数据污染检查（Memorization Screen），确保模型不是通过记忆 GitHub 上的原始 PR 来解题。

### 6. 主要结论与发现
*   **性能领先者**：GPT-5.4 和 Opus 4.7 在综合表现上最强，但即使是它们，在 Pass 3（连续三次成功）上的得分也比 Pass@1 降低了 30-50%，说明**一致性仍是巨大挑战**。
*   **工程质量差距**：模型在“功能正确性”上表现尚可，但在“工程严谨性”（由 Rubric 衡量）上普遍落后。例如，在重构任务中，模型常能通过测试，但会留下死代码或未更新文档。
*   **开源模型落后**：最好的开源模型（GLM 5）仍显著落后于闭源前沿模型。
*   **运行时推理**：顶尖模型表现出更强的运行时探索能力（如启动服务器、发送请求），而弱模型更多依赖静态代码阅读。

### 7. 优点
*   **多维度评估**：超越了单一的“测试通过率”，引入 Rubric 评估代码美学和架构合理性。
*   **现实复杂性**：任务描述故意设定为“欠规范”，模拟了真实开发者接到的模糊需求。
*   **跨语言与跨领域**：涵盖了多种主流编程语言和复杂的分布式系统场景。

### 8. 不足与局限
*   **单轮对话限制**：目前仅限于单轮任务，未评估多轮交互、澄清需求或人类协作的能力。
*   **LLM 评判风险**：虽然进行了对齐实验，但基于 LLM 的 Rubric 评分仍可能存在模型偏见或鲁棒性问题。
*   **语言覆盖不均**：虽然涵盖多种语言，但 Go 和 Python 占据了绝大部分任务，C/C++ 任务相对较少且难度极高。
*   **应用限制**：该基准测试主要针对通用编程智能体，可能无法完全代表特定领域（如嵌入式或底层内核开发）的特殊需求。
