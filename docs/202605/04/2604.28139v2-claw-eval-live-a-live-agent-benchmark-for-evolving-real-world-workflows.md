## 摘要
大语言模型（LLM）智能体被期望能够跨软件工具、业务服务和本地工作空间完成端到端的任务单元。然而，许多智能体基准测试在发布时固定了精选的任务集，并主要根据最终响应进行评分，这使得评估智能体应对不断演进的工作流需求的能力或验证任务是否实际执行变得困难。我们推出了 Claw-Eval-Live，这是一个针对工作流智能体的动态基准测试，它将可刷新的信号层（根据公开工作流需求信号在不同版本间更新）与可复现的、带有时间戳的发布快照分离。每个版本都基于公开的工作流需求信号构建，当前版本使用了 ClawHub 前 500 强的技能，并具体化为具有固定装置、服务、工作空间和评分器的受控任务。在评分方面，Claw-Eval-Live 记录执行轨迹、审计日志、服务状态和运行后的工作空间产物，在证据充分时使用确定性检查，仅在语义维度使用结构化 LLM 评判。该版本包含 105 个任务，涵盖受控业务服务和本地工作空间修复，并在统一的公开通过规则下评估了 13 个前沿模型。实验表明，可靠的工作流自动化仍远未解决：领先模型的任务通过率仅为 66.7%，且没有模型达到 70%。失败情况按任务族和执行界面呈现结构化特征，其中人力资源、管理和多系统业务工作流是持久的瓶颈，而本地工作空间修复相对容易但尚未达到饱和。仅靠排行榜排名是不够的，因为通过率相近的模型在整体完成度上可能存在差异，且任务层面的区分度集中在中间难度的任务带中。Claw-Eval-Live 表明，工作流智能体的评估应基于双重基础：新鲜的外部需求和可验证的智能体动作。

## Abstract
LLM agents are expected to complete end-to-end units of work across software tools, business services, and local workspaces. Yet many agent benchmarks freeze a curated task set at release time and grade mainly the final response, making it difficult to evaluate agents against evolving workflow demand or verify whether a task was executed. We introduce Claw-Eval-Live, a live benchmark for workflow agents that separates a refreshable signal layer, updated across releases from public workflow-demand signals, from a reproducible, time-stamped release snapshot. Each release is constructed from public workflow-demand signals, with ClawHub Top-500 skills used in the current release, and materialized as controlled tasks with fixed fixtures, services, workspaces, and graders. For grading, Claw-Eval-Live records execution traces, audit logs, service state, and post-run workspace artifacts, using deterministic checks when evidence is sufficient and structured LLM judging only for semantic dimensions. The release contains 105 tasks spanning controlled business services and local workspace repair, and evaluates 13 frontier models under a shared public pass rule. Experiments reveal that reliable workflow automation remains far from solved: the leading model passes only 66.7% of tasks and no model reaches 70%. Failures are structured by task family and execution surface, with HR, management, and multi-system business workflows as persistent bottlenecks and local workspace repair comparatively easier but unsaturated. Leaderboard rank alone is insufficient because models with similar pass rates can diverge in overall completion, and task-level discrimination concentrates in a middle band of tasks. Claw-Eval-Live suggests that workflow-agent evaluation should be grounded twice, in fresh external demand and in verifiable agent action.

*   **静态任务集与动态需求的不匹配**：大多数基准测试在发布时固定了任务，但现实中的工具栈和业务流程（如 HR、财务、运维）在不断演进，导致基准测试逐渐与实际需求脱节。

### 2. 论文提出的方法论
Claw-Eval-Live 的核心思想是将基准测试分为“可刷新的信号层”和“可复现的发布快照”。

*   **信号到任务的流水线（Signal-to-Task Pipeline）**：
    1.  **信号采集**：从 ClawHub Top-500 热门技能中提取实时需求信号。
    2.  **模式聚类与加权**：将碎片化的技能归类为工作流模式，并根据流行度分配权重。
    3.  **种子扩展与实现**：将模式转化为具体的、可执行的任务脚本、工具定义和初始环境（Fixtures）。
    4.  **区分度感知选择**：使用**混合整数线性规划（MILP）**算法，从候选池中筛选出既能覆盖各家族、又能有效区分模型能力的 105 个任务。
*   **执行与评分机制**：
    *   **受控环境**：提供 18 个受控业务服务（如 CRM、财务、邮件）和沙盒化的本地工作区。
    *   **动作锚定评分（Action-Grounded Grading）**：
        *   **确定性检查**：通过审计日志、API 调用轨迹和运行后的环境状态（如文件是否修复）进行硬性校验（占比 60%-80%）。
        *   **结构化 LLM 评判**：仅针对语义维度（如报告的组织质量）引入 LLM 评判，且必须基于已记录的执行证据。

### 3. 实验设计
*   **数据集/场景**：包含 105 个任务，分为两大类：

### 5. 实验数量与充分性
*   **实验规模**：共 105 个任务，涵盖 22 个细分任务家族。对 13 个模型进行了全量测试，生成了 105 $\times$ 13 的完整得分矩阵。

    *   **多维度**：不仅看总分，还分析了任务家族（HR vs 财务）和执行界面（API vs 终端）的差异。

### 6. 论文的主要结论与发现
*   **自动化瓶颈**：即使是最强的模型（Claude Opus 4.6），通过率也仅为 66.7%，没有模型能突破 70%，说明复杂工作流自动化远未解决。
*   **能力分布不均**：模型在“本地工作区修复”上表现较好（通过率 $>72\%$），但在“跨系统业务流”（如 HR、管理、对账）上表现极差，部分家族甚至全军覆没。
*   **“说”与“做”的脱节**：许多模型能生成流畅的总结报告，但在审计日志中显示其并未调用必要的工具或遗漏了关键的写入操作。
*   **效率差异**：GPT-5.4 在保持高准确率的同时，Token 消耗和成本显著低于 Claude 系列，表现出极高的性价比。

### 7. 优点
*   **动态性（Live）**：引入了信号层刷新机制，使基准测试能够随真实世界需求演进。
*   **高可信度评分**：强调“执行证据”而非“文本相似度”，有效防止了模型的“幻觉成功”。
*   **数学化筛选**：利用 MILP 优化任务子集，确保了基准测试的区分度和代表性。
*   **环境完备**：提供了包含审计日志的受控 API 服务，为智能体研究提供了高质量的沙盒。

### 8. 不足与局限
*   **任务规模限制**：105 个任务虽然精炼，但相比于 WebArena 等大型基准，覆盖的极端边缘案例可能仍有限。
*   **评判模型偏见**：虽然限制了 LLM 评判的范围，但使用 GPT-5.4 作为裁判模型评估其自身或其他模型时，仍可能存在潜在的偏好偏见。
*   **环境简化**：受控服务虽然模拟了真实 API，但真实世界中的网络延迟、权限复杂性和非结构化数据挑战在基准中被部分简化。
*   **时间戳依赖**：作为“动态”基准，其结论具有时效性，需要持续维护更新才能保持权威。
