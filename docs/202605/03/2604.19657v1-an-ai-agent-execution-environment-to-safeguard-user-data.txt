## Abstract 

AI agents promise to serve as general-purpose personal as-sistants for their users, which requires them to have access to private user data (e.g., personal and financial information). This poses a serious risk to security and privacy. Adversaries may attack the AI model (e.g., via prompt injection) to exfil-trate user data. Furthermore, sharing private data with an AI agent requires users to trust a potentially unscrupulous or compromised AI model provider with their private data. This paper presents GAAP ( Guaranteed Accounting for 

Agent Privacy), an execution environment for AI agents that guarantees confidentiality for private user data. Through dynamic and directed user prompts, GAAP collects permis-sion specifications from users describing how their private data may be shared, and GAAP enforces that the agent’s disclosures of private user data, including disclosures to the AI model and its provider, comply with these specifications. Crucially, GAAP provides this guarantee deterministically, without trusting the agent with private user data, and without requiring any AI model or the user prompt to be free of attacks. 

GAAP enforces the user’s permission specification by tracking how the AI agent accesses and uses private user data. It augments Information Flow Control with novel per-sistent data stores and annotations that enable it to track the flow of private information both across execution steps within a single task, and also over multiple tasks separated in time. Our evaluation confirms that GAAP blocks all data disclosure attacks, including those that make other state-of-the-art systems disclose private user data to untrusted parties, without a significant impact on agent utility. 

## 1 Introduction 

Recent advances in agentic AI systems are fundamentally changing how users interact with software systems. Given only a natural-language prompt from the user, an AI agent can invoke external tools (e.g., search engines, calculators, and web APIs [ 46 , 57 , 79 ]) to autonomously complete real-world tasks (e.g., booking travel, purchasing food, and inter-acting with SaaS platforms [ 1, 34 , 71 ]) with minimal human intervention. This capability is further supported by a grow-ing ecosystem of AI-tool interfaces, including plugin frame-works and standards for structured tool invocation [3, 48]. However, executing such tasks often requires agents to have access to private user data , such as credentials, financial data, and personally identifiable information (PII). These agents then embed this data, or derivatives of this data, in the tool calls they make to accomplish the task. For example, in order to send emails, purchase items, or check in to a flight on a user’s behalf, an agent needs access to the user’s e-mail credentials, payment information, and date of birth. As a result, agentic systems necessarily operate over contexts that combine user intent, private data, and external actions. This combination introduces significant risks for security and privacy. Recent work has shown that LLM-based agents are vulnerable to prompt injections and related attacks that manipulate their behavior [ 8 , 20 , 41 , 47 , 53 , 84 ]. Users may also want to avoid sharing their private data with the AI model provider, due to fear of its misuse or that the model provider may be compromised [ 15 , 42 ]. And even in the absence of an attack, AI agents may hallucinate or take mis-taken actions [37, 40] leading to unwanted data disclosure. To address this problem, we develop GAAP ( Guaranteed 

Accounting for Agent Privacy), an execution environment for AI agents that provides confidentiality guarantees for private user data, preventing unintended disclosures to un-trusted parties through tool calls. GAAP enforces strong, fully deterministic confidentiality guarantees by asking the agent to generate a code artifact and performing an informa-tion flow control analysis (IFC) on this artifact. In designing GAAP, our vision is to achieve deterministic privacy guaran-tees on data disclosures without relying on the user’s prompts to the agent or the AI model to be correct or free of attacks . To fulfill this vision, GAAP must address three key challenges. First, GAAP considers the user prompt, model provider, and model context to be fully untrusted, so these components cannot hold private user data or contain the user’s permis-sion policies. To address this challenge, GAAP encapsulates user private data and permission policies in a private data database and a permission database, and allows only the user

to access and modify these items through direct interactions (blocking any agent access). This allows GAAP to provide strong confidentiality guarantees without any trust in the prompt, context, or LLM model. Second, realistic usage of agents includes long sequences of interactions with the model, either through iterative task completion, or across multiple tasks done by the same user, or even across tasks done by different users in the system. All of these interactions might perform tool calls that might lead to indirect private data flows across tasks, e.g., when data shared in a tool call is returned to the user by a subsequent call to the same tool for a different task. To address this prob-lem, GAAP introduces a disclosure log that records all prior disclosures of private data to external services (e.g., sharing a passport number with an airline service), allowing it to track and prevent unintended data flows that occur across tasks and tool calls. This log can also be used for compliance in accordance to government and other regulations. Finally, real-world agents use a vast ecosystem of exter-nal tools (e.g., as Model Context Protocol (MCP) services). Each of tool is associated with different external parties and allows different flows of data. For example, an airline service might return the user’s full name and passport number, but it would never send back the password for the user’s account. To address this, GAAP develops an annotation framework that can be used to describe the associated parties and data flows for each method of each MCP service. The goal is that the framework will be used to develop an open-source crowd-sourced library of annotations for MCP services that can be publicly analyzed and vetted and that users can import to use with GAAP. We provide written annotations for 10 diverse MCP servers to bootstrap this process. We implement GAAP and evaluate it using a collection of tasks and attacks drawn from various sources in the lit-erature and practice. Our evaluation confirms that it pro-vides deterministic privacy guarantees, and that it blocks all data disclosure attacks, including those that succeed in making other state-of-the-art systems disclose private data to untrusted parties. Furthermore, GAAP provides strong privacy guarantees without negatively impact the agent’s utility, allowing it to successfully complete tasks with rates comparable to other state-of-the-art and less private systems. In summary, GAAP makes the following contributions: 1. A system architecture for deterministic confidentiality guarantees in a realistic setting where the user’s prompts, private data, agent context, and AI model may be compro-mised (§3). 2. A private data DB and a permission DB that encapsulate user private data and permission policies, restricting the agent LLM’s data access (§4.2,4.3). 3. A disclosure log that stores all data disclosures to support indirect data flows across tasks and tool calls (§4.5). 4. An annotation framework that enables high precision descriptions of the parties and flows associated with each external tool (§4.4). We start by surveying related work and how existing sys-tems fail to address critical aspects of real-world agents in §2. We then present an overview of GAAP in §3. Because GAAP’s main components are interrelated, we explain them in two passes: a high-level “overview” pass in §3.3, and a de-tailed pass in §4. Finally, we present GAAP’s implementation (§5), an evaluation of GAAP (§6), and a discussion (§7). 

We plan to release GAAP as open-source software. 

## 2 Related Work 

GAAP is one of many privacy-providing systems for agents, and provides an approach to privacy that differs in its fea-tures, threat model, and goal. Table 1 highlights the differ-ences between GAAP and various other categories of sys-tems; we break these down further below. 

Information Flow Control (IFC) Systems. Existing IFC systems [ 11 , 12 , 29 , 31 , 60 , 76 , 82 ] utilize mechanisms similar to GAAP’s IFC core (§3); all use techniques from prior IFC systems [ 16 , 45 , 81 ]. Like GAAP, some of these systems [ 12 ,31 ] generate code artifacts to perform the intended task. These systems differ from GAAP in three key ways. First, these systems aim to preserve control-flow integrity by preventing execution of high-risk actions induced by untrusted data manipulating the control flow. This requires a trusted mechanism to label data as untrusted, and a trusted LLM that executes correctly unless exposed to untrusted data. By contrast, GAAP operates in a threat model (§3.2) where no data is considered trusted and no trusted labels are required, thus avoiding any assumption of trust in the user’s input or context, which can be compromised [ 8, 14 , 41 , 62 ]). In this threat model, GAAP provides confidentiality but not integrity. Second, GAAP builds the user’s privacy policy over time; these systems instead assume a prewritten, static disclosure policy set by the developer over user data. Prudentia [ 29 ]allows users to dynamically override a denied disclosure, but does not persist this decision by updating the policy. Third, GAAP tracks prior disclosures and taints within and across task executions; by contrast, these systems either track taints only through a “single-shot” piece of code [ 12 ] or through a single task execution [ 11 , 29 , 76 ]. Because GAAP’s threat model allows untrusted data for code generation, GAAP can track taints through a “multi-shot” execution (where the agent generates code during task execution based on intermediate results), whereas a system like CaMeL can-not. Unlike IFC systems like Fides and Prudentia, GAAP reasons about transitive and indirect data disclosures across tasks, remembering prior disclosures to external entities.

Trusted Policy-Based Defenses. Approaches including IsolateGPT [ 77 ], Conseca [ 65 ], CeLLMate [ 38 ], and others [ 5 ,12 , 22 , 38 , 49 , 51 , 54 , 63 , 67 , 70 ] aim to guarantee both confi-dentiality and integrity against malicious or confused agents compromised by e.g., prompt injection, model poisoning, or malicious system prompts. They operate by running a variant of “security monitor” that checks either the agent’s inputs (e.g., retrieved user data) or outputs (e.g., tool calls) against a (static or contextually-generated) policy. 1 Approaches us-ing LLMs for policy generation [ 7 , 12 , 31 , 65 , 76 ] follow the dual-LLM pattern [ 74 ] to ensure the security monitor is itself not comprised by any untrusted inputs. By contrast, GAAP aims to only guarantee data confidentiality and operates in a threat model where the agent and all inputs may be un-trusted. These systems can similarly be combined to provide both strong confidentiality and integrity guarantees.

Model-based Threat Detection. Many works use models to detect attacks [ 2 , 24 , 26 , 75 ], or train or prompt mod-els to be robust to malicious inputs via e.g., spotlighting or prompt “delimiters” [ 6 , 23 , 69 , 73 , 80 ]. ## 3 System Overview 

We first describe GAAP’s system model and workflow (§3.1). We then describe GAAP’s threat model and security guaran-tees (§3.2). Because GAAP’s main components are interre-lated, we explain them in two passes: a high-level “overview pass” below in §3.3, and a detailed pass in §4. 

3.1 System Model 

We consider a system model where a user issues prompts to an AI agent, asking it to perform a variety of tasks, such as booking a flight, purchasing an item online, or e-mailing a file to a colleague. The agent takes actions in response to the user’s prompts, issuing calls to external services (remote or local MCP servers), to e.g., send an email, or access a file. The agent remembers context across multiple prompts. To carry out these tasks, the agent needs access to the user’s sensitive data (e.g., payment information, email cre-dentials, personal information, etc.), and this is where GAAP comes into play. GAAP is an execution environment for the agent that aims to ensure that all disclosures of private user data to external services are consistent with the user’s per-mission specifications. It intercepts the agent’s requests to external services, and denies requests that would disclose private user data in ways that the user has not permitted. Even though AI models (e.g., GPT, Claude, etc.) are often hosted remotely, the agent (e.g., Claude Code, Copilot, etc.) often executes locally in the user’s workspace, with access to local resources like the file system. GAAP should be deployed in the environment where the agent executes, which we envision will often be on the user’s local machine. We briefly consider alternative deployment models in §3.2. 

3.1.1 User Workflow. As shown in Figure 1, agent opera-tion with GAAP proceeds as follows: 1. The user issues a prompt to the agent, which runs in GAAP’s execution environment. 2. The agent retrieves some private data by issuing an API call (to an MCP server) such as get_file or by querying user data stored internally by GAAP itself. GAAP tracks these accesses throughout agent execution. 3. When the agent attempts to make an API call to an ex-ternal service, GAAP intercepts and evaluates this call to check for disclosures of user data. 4. If GAAP can conclusively determine whether the API call is permissible or not based on its current understanding of the user’s policy (i.e., which user data items can be shared with which parties), it accordingly either allows or denies the API call, without prompting the user. 5. If GAAP cannot conclusively determine this, then it pauses the agent’s execution, prompts the user with information about the disclosure, and asks the user if it should be al-lowed (e.g., “Should the contents of file2.txt be sent via e-mail to joe.smith@email.com ?”). GAAP saves this decision (i.e., the permission specification ) to automatically evaluate future disclosures (§3.3.3) and skips prompting the user for future disclosures of that data item to the particular external party. GAAP exposes an API for users to update their permission specifications if they wish. 

3.1.2 Private Data Lookups and Tracking. To perform a task, the agent needs access to private user data. The agent can access private data in two ways: (1) lookups in GAAP’s internal store, and (2) calls to external APIs. First, GAAP maintains an internal data store (described more fully in §3.3.2) containing private user data that the agent can access. At runtime, the agent identifies any data items that it needs that are not already in GAAP’s internal store (e.g., user’s date of birth), and prompts the user to add those values to GAAP’s internal store. Users may also add data values proactively, in anticipation of the agent need-ing them for a future prompt; however, we expect users to generally rely on runtime prompts as described above. Second, some user data may exist outside of GAAP’s data store, in external services (e.g., file system, e-mail, or web services) so GAAP conservatively considers the outputs of API calls as containing private data. For example, the user may have shared data with an external service (e.g., a health portal) without using GAAP, or external services may share data among each other. APIs can be annotated using GAAP’s annotation framework (described more fully in §3.3.5) to indicate cases where the outputs of API calls do not contain private data (e.g., APIs for searching public information). To enforce the user’s permission specifications, GAAP tracks how the agent accesses and uses private user data. GAAP achieves this by (1) requiring the agent to generate 

> 4

code that performs the desired task, and (2) applying Infor-mation Flow Control to determine how that code accesses and discloses private user data. §3.3.1 explains this further. 

3.1.3 User Prompts. GAAP considers the user’s prompt “public” and shared directly with the agent. Therefore, GAAP expects the user to avoid including their private data in their prompt. Instead, the user writes their prompts using the keys 

(names) of data items rather than the values, and the agent looks up the corresponding values at runtime via GAAP’s execution environment. For example, instead of prompting the agent with “As a 20-year-old, am I eligible to vote in the United States?”, the user should prompt the agent with “Given my age, am I eligible to vote in the United States?” At runtime, the agent looks up the user’s date of birth (§3.1.2). 

3.2 Threat Model and Security Guarantees Definitions. A data disclosure is a transfer of user data (from either GAAP’s own storage, or the output of an API call) to an external party (through a call to an external service). For example, data disclosures include (1) sending an e-mail containing data from a local file, (2) filling out an online web form using data from an e-mail, and (3) logging in to a web service using credentials that the user gave to GAAP. A private data type is a category of data whose handling should be subject to the user’s policy specification; examples include name, date of birth, SSN, email address, and email password. We then define private user data , or simply user data , to mean any piece of data that contains (or is tainted by or otherwise derived from) at least one private data type; examples include a date of birth “1/1/2000”, or a string con-taining multiple private data types “My name is J. Doe and I was born on 1/1/2000”. GAAP’s focus is controlling dis-closures of private user data to external parties , namely an entity that can be precisely identified, e.g., an organization domain name, a URL, an email address, or an external service. Finally, a permission specification is an allow/deny flag for a pair of a private data name and an external party. 

Security guarantees. We consider a malicious adversary who observes the inputs (prompt and context) to the AI model, controls the outputs (text, code, and tool calls) of the AI model, and controls some or all of the external services with which the agent may interact. GAAP guarantees that such an adversary learns no private user data, except for data that the user has authorized, by their permission specifi-cations, to be disclosed to the AI model provider or external services under the adversary’s control. In this way, GAAP protects the confidentiality of private user data. Basically, GAAP guarantees that the agent can only dis-close data to external services in ways that are consistent with the user’s permission specifications. These guarantees also apply for disclosures to the AI model provider (i.e., the AI model provider is itself modeled as an “external service”). GAAP provides this security guarantee reliably and deter-ministically; it does not rely on an AI model to detect and prevent unintended data disclosures. Thus, it totally elimi-nates unintended data disclosures in the above threat model. 

Discussion. Our threat model (i.e., malicious adversary in the previous paragraph) covers attacks including agent hallu-cinations, malicious model providers, model poisoning, and untrusted tool providers. In particular, it includes prompt injections [ 30 , 36 , 52 ], which OWASP listed as the #1 security threat for LLMs in 2025 [ 50 ]. For example, a prompt injection could happen because the user authorizes disclosure of data containing a prompt injection attack to the AI agent; our system ensures that, even if the agent is attacked, it can only disclose user data according to the user’s explicit permission specifications. GAAP achieves its security guarantee without relying on the correctness or trustworthiness of the agent, user prompts, or private user data. Removing trust from the AI agent and its provider are particularly relevant if the user runs GAAP on their local machine, as they may be more comfortable entrusting their data to GAAP than to a remote AI agent provider. In cases where an AI model provider offers a hosted agent, the provider might deploy GAAP in their own envi-ronment (e.g., in the cloud). In such cases, GAAP still brings value by protecting against attacks on the model (e.g., prompt injections). Existing work on private AI memory [ 4 , 28 , 64 ] safeguards user data from an adversary who controls the AI model serving infrastructure. Private AI memory does not protect against an agent erroneously disclosing user data to external services via API calls. In contrast, GAAP does protect against disclosures to external services, captured by our threat model where the adversary controls not only the AI model, but also some external services. 

Non-guarantees. GAAP does not provide guarantees for user data that is included directly in the agent prompt (hence the usage guidelines in §3.1.3). Furthermore, in addition to learning the user prompts, the adversary also learns metadata about the prompts, e.g., the timing and order in which the user issued them. Finally, GAAP only prevents the agent from violating the confidentiality of user data, and does not aim to preserve data or system integrity. For example, GAAP will not prevent changes in control flow caused by prompt injections that do not lead to unintended data disclosure.

3.3.1 IFC Core. GAAP does not allow the agent to directly call into an MCP server, but instead requires it to generate a 

code artifact that performs the intended task when executed. Because the code artifact consists of regular software, GAAP can apply information flow control (IFC) analysis [ 16 , 45 , 81 ]to analyze information returned by external calls and how how each external call depends on private data. When the user provides a prompt, GAAP prepends a sys-tem prompt that instructs the agent to generate code for the task. Then, GAAP applies IFC to the generated code: it applies taints to private user data, and analyzes how those taints propagate to the inputs of API calls. Any lookup in GAAP’s private data store is tainted with the lookup key, and the output of any API call is tainted with a representation of the API call (e.g., a retrieved file is tainted with the file path); these are the only two ways for the agent to access private data (§3.1.2). Generating a code artifact is an idea used in prior agent environments, in academia [ 12 , 31 ] and industry [ 66 ]. These existing systems constrain the agent by requiring it to solve the problem in a “single shot” at code generation; the code ar-tifact prevents the agent from inspecting intermediate results and adapting its execution accordingly. GAAP bridges this gap by generalizing the execution model to enable “multi-shot” code generation, where some intermediate results are disclosed to the AI agent, which can then generate a new code artifact to continue execution. Crucially, GAAP stores and tracks taints across multiple tasks and agent invocations to support multi-shot code generation while respecting the user’s permission specifications , including for data disclosures to the agent itself. We describe this further in §3.3.4. 

3.3.2 Private Data Database. Recall from §3.1.2 that pri-vate user data items are only accessible to the code artifact via GAAP’s internal store or via API calls. The private data DB is this internal store that holds private user data. It is organized as a key-value store from private data types, e.g., “date of birth” and “email username,” to private data values, e.g., “1/1/2000” and “jdoe@email.com”. The keys are con-sidered public information—they are included in GAAP’s system prompt and user prompts may refer to data items by their keys—but GAAP tracks the disclosures of values. To understand the benefits of having the private data DB, let us consider what would happen if GAAP did not have it. In our running example, the agent would have to search the user’s e-mails or files to try and find the user’s date of birth, airline rewards number, etc. This has several downsides— writing code to search the user’s files/e-mails is brittle, may accumulate more taints than needed (e.g., a file may contain the airline rewards number and also other information), and is not guaranteed to succeed (e.g., no file or e-mail may contain the desired information). The private data DB instead enables the code artifact to ask the user to insert data for the requisite keys; the private data DB “remembers” those values so that next time, the agent can simply retrieve them from the DB instead of asking the user again. 

3.3.3 Permission Database. GAAP must address the chal-lenge of enforcing precise permission specifications, one for every pair of private data type and external party. These per-missions are also dynamic, meaning that they might evolve 

> 6

over time as users perform new tasks and revise their trust assumptions about different parties. To address this and minimize the number of times GAAP needs to ask the user whether they allow a data disclosure, GAAP introduces a permission database that stores allowed data disclosures for specific pairs of private data types and external parties. GAAP incrementally fills its permission DB by querying the user for denial or confirmation whenever a data type is about to be shared with an external party and no relevant permission specification exists in the permission DB yet. Later in time, the user can modify the permissions in the database if their trust model changes and they want to change who can have access to which private data types. In our running example, GAAP asks the user for permis-sion to disclose the date of birth, airline rewards number, etc. the first time that the user checks in. The next time that the user asks to check in for a flight, the permission DB re-members that the user has authorized disclosing these items to the airline. GAAP only prompts the user for items being disclosed for the first time (e.g., the confirmation number). 

3.3.4 Disclosure Log. Another challenge is that private data may also be disclosed indirectly and transitively . This can happen when an agent discloses private data to an ex-ternal service and later (in the same task or in a future task) makes another call to that same external service, which then returns the previously disclosed private data. This data could then be passed to another service as its input. In our run-ning example, suppose that, after the AI agent completes the check-in process, the user asks the agent in a separate prompt to obtain confirmation of the check-in from the air-line’s website or API. The confirmation from the airline may contain information that the user previously shared with the airline (e.g., their full name and passport number). To address this, GAAP introduces a disclosure log that records all prior disclosures of private data (e.g., sharing a passport number with an airline service). This log enables the system to reason about transitive and indirect data dis-closures, by allowing the IFC subsystem to track private data flows through external services and back into the agent program. In our example, GAAP recognizes that any out-puts from the airline service after a check-in attempt may be tainted with any private information that the user previously disclosed to the airline service, and it relies on the disclosure log to identify such previous disclosures so it can apply the corresponding taints. The disclosure log also records disclo-sures to the AI agent and its providers, enabling multi-shot execution (§3.3.1) while respecting the user’s permission specifications. Our disclosure log can only track indirect and transitive disclosures that happen through GAAP. For example, if the user manually uploads private user data to an external ser-vice without using GAAP, then the disclosure log will not record it and the relevant taints will not be included if the agent calls the service to download it. Because of this, as mentioned in §3.1.2 and §3.3.1, GAAP taints the output of each API call with a representation of the API call itself. This may result in a less detailed (but still correct and workable) prompt to the user if the tainted outputs are later disclosed. 

3.3.5 Annotation Framework. The final challenge is that the MCP interfaces provided by external services do not fully capture semantics relevant to security/privacy. In particular, external services may differ in the following aspects: (1) which external parties are associated with a specific call to an external service (e.g., a call to an email service to send an email might leak the email to the email provider and the email recipient ); and (2) which private data previously shared with a service can be returned as part of the service’s output (e.g., a flight status check may return the user’s name, but never their password). Without knowing this information precisely, GAAP must assume the worst case, i.e., all data disclosed to a service may be returned by any API call to it. To address this challenge, GAAP introduces a lightweight annotation framework that supports describing the afore-mentioned properties for each external service (i.e., associa-tions with external parties and what data can be returned). These annotations are meant to be trusted, acting as a partial specification of external services and MCP servers. Impor-tantly, we consider annotations trustworthy because annota-tions for services can be open-sourced and be made available for independent public audit, similar to popular open-source security-sensitive libraries (e.g., OpenSSL).

4.3 Permissions Database 

The permissions DB keeps track of which disclosure permis-sions the user has granted. Each DB row contains an external party identifier, a private data item, and a permission (allow or deny) that indicates whether the item can be shared with this party. An external party can be a specific server, tool, or some other identifier, as specified by annotations in §4.4. When the code artifact attempts to share private data with any external party, GAAP queries the database for the MCP server, the keys of the private data, and the identifier for the external party. If no permission has previously been set, GAAP asks the user for permission to share that private data item with that external party. GAAP allows a user to permit sharing "just once" (and does not store the permission in the 

> 8

DB); however, this single disclosure will persist in the disclo-sure log. Because GAAP intercepts MCP calls, GAAP must check all possible private data items being shared by that call prior to allowing it. If desired, a user can revoke permissions via an API to remove rows from their permissions DB. Upon initialization of a new user’s permissions DB, the database is empty. We offer no default permissions. The reason for this is threefold: (1) the keys in the private data DB or other private data items are set in real time by a user and LLM for each instantiation, and are not necessarily common between users; (2) the external entities with which different users may share data can be substantially different, such as differing MCP servers being installed and different contacts existing in the email server; and (3) users may have differing privacy preferences, with some comfortable sharing data items easily and others carefully protecting them. As a result, when a user first starts using the system, they will face a number of permissions requests. This cost is amortized as the same tools and data items are repeatedly used, leading to decreased permission requests to the user over time. 

4.4 Annotations 

Different external services naturally offer different semantic disclosure granularities (e.g., at the granularity of the entire service, a single API call, or some other identifier such as an email address). GAAP’s annotations allow permissions to be set at varying granularities to make IFC more accurate and taint pass through more precise. Annotations either: (1) define the granularity and method of external entities; or (2) describe how input arguments to tool calls are propagated into outputs of those tool calls. For (1), we annotate each server and tool to specify how to identify the external entity for each tool call—the server, the tool, or some additional string identifier computed from the arguments to a tool call. For the email server’s tool calls, for example, the external entity is the email address to which an e-mail is being sent. This is implemented as a string identifier across all tools of the email server, computed by selecting the appropriate argument to each tool. This reduces permis-sions requests: disclosing to matching entities, even across different tools, do not incur added requests to the user. It also allows for finer-grained permissions: for example, a single email server can represent an unbounded number of differing external entities through email address. This form of annotations can be essential for usability. For example, a task might ask the agent to read a file of reviews and send review scores by email to an external party. With-out annotations on the filesystem and email servers, GAAP asks the user “Do you want to share all filesystem contents with all entities of the email server?” This would be unreason-able. Annotations allow GAAP to ask “Do you want to share the contents of feedback.txt with user@example.com? ,” which is a far more acceptable request. Annotations of type (2) act on tool call parameters, de-scribing whether the external entity for the MCP call may later disclose private data passed via these parameters. For example, if we issue an API call containing “password” and “ordered_food_items” to a food ordering service, we may expect the response to contain the list of food items echoed back to us, but not the password. By default, GAAP would be overly conservative and consider all outputs of the food ordering MCP to be tainted with the values passed into the ar-gument “ordered_food_items” and the values passed into the argument “password,” even though the password is never re-turned. Instead, annotations can mark the value of argument “password” as impossible for the food ordering MCP to later return so GAAP never taints food order outputs with the user’s password. GAAP still logs all private data disclosures to external entities via the disclosure log to compute the set of private data disclosed to an entity (§4.5). Annotations remove from this set any disclosed private data annotated as impossible for the entity to later output. Annotations are specified in JSON. As we assume the entities behind the servers to be untrusted, there is some burden for a user or public source to verify the accuracy of the annotations. We would expect the annotations to be written by a transparent open-source community where they can be vetted, in real world deployment. These annotations are held user-side, so they should always be accessible. In the case of uncertainty with black-box highly untrusted servers, the specification of the default annotations will pass through all data possible, leading to the most stringent protections. 

4.5 Disclosure Log 

The disclosure log is a database with rows representing dis-closures of individual data items to individual external enti-ties. Each row records a disclosure’s corresponding private data item, external entity, time, and causal tool call’s argu-ment names and values. GAAP adds a new entry in this log each time private data is disclosed to an external party. The disclosure log provides privacy guarantees against 

indirect and transitive data flows , e.g., private data uploaded to a web server yesterday may be returned back today. When a code artifact issues an API call to an MCP server, GAAP taints the output data returned by that API call with a rep-resentation of the API call, including the server, tool, and external entity. If and when this output data is about to be dis-closed via a second API call, GAAP queries the disclosure log to discover all of the private data that has been disclosed to the external entity for the first API call and can pass through the first API call (as defined in the annotations, §4.4); GAAP checks if the entity behind the second API call is authorized to see that private data before allowing the disclosure. Because the disclosure log is persistent, this propagation of taints can happen within a single task, or across multiple tasks separated in time. For example, if a task reads the user’s Social Security Number (SSN) and writes it to a file (where 

> 9

a file path is the entity identifier behind the filesystem), we track this disclosure in the disclosure log. When a later task accesses this file, GAAP will taint its contents with the SSN. The disclosure log also acts as a record for accountability of private data sharing—one can easily see all instances when private data was disclosed, and in what form. 

## 5 Implementation 

The GAAP prototype is implemented in Python, with a com-mand line interface for agent interactions. Its core execution environment executes an LLM-generated Python code arti-fact as follows: given the code artifact created by the LLM, it deterministically (lightly) modifies it to add import state-ments, better track the execution, and catch exceptions. The script then runs via IFC implementation, which uses Pyre [ 39 ] with additional functionality on top to support taint tracking of MCP calls and annotations. The GAAP prototype supports the addition of any new MCP servers to the system via a JSON file containing metrics about the server, and without modifications to any code. These servers will automatically be compatible with the IFC analysis and usable by new code artifacts. All databases are implemented with Sqlite [ 61 ]. GAAP is implemented with roughly 3,500 LoC, with an additional 2,600 LoC for the benchmark suite.

LLM-Judge and Conseca can be tuned to achieve different results, but this simply shifts their privacy-utility tradeoffs and successful attacks still exist [ 47 ]. By contrast, GAAP provides deterministic confidentiality guarantees.

Because we are able to batch concurrent permissions requests to the user, we observe 58 real user interactions when we reset the permissions DB after each task and 39 real user interactions without resets (a 32% decrease). Over time, the permissions DB will continue to lead to fewer user queries.

In this task, we ask the agent to read from a remote database and email the results to an external party. It is impossible for the agent, on its own, to know what data has been fed into this remote database and could be returned. In our scenario, we assume that the user previously gave GAAP permission to disclose secret_info.txt to populate the remote database. However, they may have forgotten about this action. Thus, prior to the agent sending the email, GAAP prompts the user if they want to share secret_info.txt with the external party, giving the user the choice of how to protect their private data. 

## 7 Discussion and Conclusion 

A future work direction is the user experience . GAAP can proactively find and add user private data to the private data DB (with user validation of keys and values) via e.g., new user files. Further, as personalized “privacy personas” [ 17 ]and AI-made user decisions improve in accuracy, GAAP could perhaps substitute user decisions with personalized AI ones, lowering user burden (but losing its deterministic guarantee). GAAP also incurs storage overheads for its components. The private data DB only contains data the user themselves input, and which they would likely store regardless. The permissions DB grows proportional to how many times we asked the user, but is limited by the number of data items and external parties. These costs are likely manageable and can be mediated by e.g., “evicting” data or permissions not recently used, at the chance of increasing user queries. How-ever, the disclosure log can grow arbitrarily large. Compactly representing this log and annotations on unimportant or ex-pired disclosures can all help address this overhead. Finally, GAAP can also provide support for private data in prompts by integrating a data scrubbing subsystem that rewrites prompts containing private data (requiring trust 

> 13

in this subsystem), or falling back on tainting the prompt (asking the user if they permit sharing the prompt itself). To conclude, this paper presents GAAP, a system for AI agents that achieves deterministic privacy guarantees with-out relying on the the AI model or user’s prompts to the agent to be correct or free of attacks.
