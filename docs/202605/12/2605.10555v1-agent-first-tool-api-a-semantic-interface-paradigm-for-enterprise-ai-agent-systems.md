## 摘要
随着 AI Agent 从研究原型转向企业生产系统，它们所调用的工具接口仍根植于面向人类的 CRUD（增删改查）范式。本文识别了传统 API 与自主 Agent 需求之间的五个基本架构失配：精确标识符依赖、面向渲染的响应、单次交互假设、等同用户的授权以及不透明的错误语义。我们提出了 Agent 优先工具 API 范式，包含三个集成机制：(1) 六动词语义协议，将工具交互分解为搜索、解析、预览、执行、验证和恢复阶段；(2) 归一化工具契约 (NTC)，提供结构化的决策支持元数据，包括置信度分数、证据链和建议的后续行动；(3) 双层治理流水线，结合了静态能力策略与动态风险升级。该范式在一个生产级多租户 SaaS 平台上得到了实现和验证，该平台在 6 个业务领域提供 85 个注册工具。对 50 个真实操作任务的对比实验表明，Agent 优先 API 实现了 88% 的端到端任务成功率，而优化的 CRUD 基准线为 64%（提升了 37.5%），同时减少了 72.7% 的人工干预，并将自主错误恢复能力提高了 5.8 倍。我们证实该范式与 MCP 等传输层标准是正交且互补的，作为现有工具发现和调用协议之上的语义应用层运行。

## Abstract
As AI agents transition from research prototypes to enterprise production systems, the tool interfaces they consume remain rooted in human-oriented CRUD paradigms. This paper identifies five fundamental architectural mismatches between conventional APIs and autonomous agent requirements: exact-identifier dependence, rendering-oriented responses, single-shot interaction assumptions, user-equivalent authorization, and opaque error semantics. We propose the Agent-First Tool API paradigm, comprising three integrated mechanisms: (1) a Six-Verb Semantic Protocol that decomposes tool interactions into search, resolve, preview, execute, verify, and recover phases; (2) a Normalized Tool Contract (NTC) providing structured decision-support metadata including confidence scores, evidence chains, and suggested next actions; and (3) a dual-layer governance pipeline combining static capability policies with dynamic risk escalation. The paradigm is implemented and validated in a production multi-tenant SaaS platform serving 85 registered tools across 6 business domains. Comparative experiments on 50 real operational tasks demonstrate that Agent-First APIs achieve 88% end-to-end task success rate versus 64% for optimized CRUD baselines (+37.5%), while reducing required human interventions by 72.7% and improving autonomous error recovery by 5.8x. We establish that the paradigm is orthogonal and complementary to transport-layer standards such as MCP, operating as the semantic application layer above existing tool discovery and invocation protocols.

### 1. 核心问题与整体含义
*   **研究动机**：当前的 API（如 REST/CRUD）是为人类操作的 UI 设计的，而非自主 Agent。
*   **核心矛盾**：论文识别了五个关键失配：
    1.  **参数精确性**：CRUD 要求精确 ID，而 Agent 只有模糊的自然语言意图。
    2.  **响应格式**：CRUD 返回渲染数据，而 Agent 需要决策支持元数据（如置信度、证据）。
    3.  **交互模式**：CRUD 假设单次请求，而 Agent 需要多步推理循环。
    4.  **权限模型**：传统权限绑定用户，Agent 需要额外的“工具能力”与“对象范围”双层管控。
    5.  **错误处理**：HTTP 状态码对 Agent 而言信息量不足，无法引导其自主恢复。
*   **整体含义**：该研究主张将工具接口从“表单提交终点”转变为“目标达成协议”，通过语义化的接口设计提升 Agent 在复杂企业环境中的可靠性与安全性。

### 2. 方法论
论文提出了三个核心贡献：
*   **六动词语义协议 (Six-Verb Semantic Protocol)**：将每次工具调用结构化为：
    1.  `semantic_search`：接受模糊输入，返回候选集。
    2.  `resolve_candidates`：处理歧义，引导 Agent 选择正确对象。
    3.  `preview_action`：在执行前进行“干跑（Dry-run）”并评估风险。
    4.  `execute_action`：执行具有幂等性保证的状态变更。
    5.  `verify_result`：执行后自动验证结果是否符合原始意图。
    6.  `recover_from_error`：发生错误时提供结构化的修复建议。
*   **归一化工具契约 (NTC)**：标准化的响应格式，包含：
    *   **置信度 (Confidence)**：基于贝叶斯更新的动态评分。
    *   **证据 (Evidence)**：提供推理依据（如匹配分数、过滤条件）。
    *   **后续行动建议 (Next Actions)**：引导 Agent 进入下一步，减少搜索空间。
*   **双层治理管线**：
    *   **权限分离**：区分“能力（能做什么）”与“对象范围（能看哪个租户/品牌的数据）”。
    *   **动态风险升级**：根据操作影响范围（如跨品牌、批量操作）动态触发人工审批流。

*   **实验规模**：

### 6. 主要结论与发现
*   **性能提升**：任务成功率从 64% 提升至 **88%**。
*   **错误减少**：ID 幻觉错误减少了 **85.7%**，自主错误恢复能力提升了 **5.8 倍**。
*   **治理效果**：人工干预需求降低了 72.7%，动态风险评估能有效拦截高危操作。
*   **成本权衡**：虽然单次 API 调用的延迟增加（约 1.5s）且 Token 消耗增加（约 37%），但由于减少了无效的重试循环，**单次任务的总成本和总时间反而有所优化**。

### 7. 优点
*   **架构前瞻性**：首次系统性地定义了“Agent 原生”的 API 应该长什么样，填补了工具调用协议与后端实现之间的空白。
*   **解决痛点**：通过后端逻辑（语义搜索）分担了 LLM 的推理压力，从根本上解决了 Agent 难以处理精确 ID 的顽疾。
*   **安全性强**：内置的审批流和动态风险评估非常符合企业级生产环境对 Agent 的合规性要求。
*   **兼容性**：与 MCP (Model Context Protocol) 协议互补，可作为其语义应用层。

### 8. 不足与局限
*   **工程成本**：需要维护两套 API 表面（CRUD 供 UI 使用，Agent-First 供 AI 使用），增加了开发负担。
*   **通用性验证**：虽然在 SaaS 系统中验证有效，但在其他异构系统（如遗留的大型机系统或高度非结构化的环境）中的适用性仍待观察。
*   **置信度模型**：目前的置信度校准仍部分依赖人工设定的基准值，未来可能需要更自动化的强化学习机制来优化。
*   **延迟敏感性**：多阶段协议（预览、验证等）增加了交互轮次，对于极度追求实时响应的场景可能存在挑战。
