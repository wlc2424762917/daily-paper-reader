## 摘要
大多数语言模型训练数据展示的是最终产物，而非产生这些产物的过程。我们在工具使用场景下研究了这一问题的一个可处理版本：当模型学习一系列新的 API 领域时，与剥离中间 API 追踪相比，保留工具使用轨迹是否有助于学习？我们使用 QLoRA 在 API-Bank 上对 Llama 3.1 8B Instruct 进行了微调，采用了四个连续的领域块。实验条件 A 从提示词中剥离了之前的 API 请求/响应行，并训练模型预测下一个 API 调用。实验条件 B 保留了轨迹上下文。在单随机种子的初步实验中，全留出生成评估显示，条件 B 的最终精确全调用准确率达到 56.9%，而条件 A 为 39.2%。B 还将最终 API 名称准确率提高了 7.7 个百分点。然而，B 使用了多出 25.1% 的训练 token，运行仅使用了一个种子，且任务是预测下一个调用而非完整的对话成功。

## Abstract
Most language-model training data shows final artifacts, not the process that produced them. We study a tractable version of this question in tool use: when a model learns a stream of new API domains, does keeping tool-use trajectories help compared with stripping the intermediate API trace? We fine-tune Llama 3.1 8B Instruct with QLoRA on API-Bank using four sequential domain blocks. Condition A strips previous API request/response lines from the prompt and trains the model to predict the next API call. Condition B keeps the trajectory context. In a single-seed pilot, full held-out generation evaluation shows that Condition B reaches 56.9\% final exact full-call accuracy compared with 39.2\% for Condition A. B also improves final API-name accuracy by 7.7 points. However, B uses 25.1\% more training tokens, the run uses one seed, and the task is next-call prediction rather than full dialogue success.

### 1. 核心问题与整体含义
*   **研究动机**：大多数语言模型的训练数据仅包含最终结果（如答案、代码），而忽略了产生这些结果的中间过程（推理轨迹）。虽然“过程监督”在数学推理中已被证明有效，但人类真实的思维轨迹难以获取。
*   **核心问题**：在工具使用（Tool-Use）这一具体场景下，当模型连续学习新的 API 领域时，保留中间的 API 调用轨迹（请求与响应记录）是否比仅保留最终动作更有助于模型的适应与记忆？
*   **研究背景**：研究者将工具使用轨迹视为人类内部推理的一种外部代理，通过持续学习（Continual Learning）这一压力测试，观察不同监督格式对模型性能的影响。

### 2. 方法论
*   **核心思想**：对比“剥离上下文”与“保留轨迹”两种监督模式在顺序微调过程中的表现。
*   **关键技术细节**：
    *   **模型基础**：使用 Llama 3.1 8B Instruct 作为基座模型。
    *   **微调技术**：采用 QLoRA（4-bit NF4 量化，Rank=32, Alpha=64）进行高效微调。
    *   **实验条件 A（Stripped-context）**：从提示词中删除之前所有的 API 请求和响应行，仅训练模型根据当前对话预测下一个 API 调用。
    *   **实验条件 B（Trajectory-context）**：在提示词中保留完整的 API 调用轨迹（动作-观察序列）。
    *   **持续学习流程**：将数据集分为四个领域块（$D_1$ 到 $D_4$），模型按顺序在每个块上训练 3 个 Epoch，并在每个阶段后对所有已学和未学的块进行评估。

### 3. 实验设计
*   **数据集**：使用 **API-Bank** 评估基准，它包含结构化的 API 调用、响应和对话上下文。
*   **Benchmark 设置**：将 API-Bank 划分为 4 个不相交的领域块。
*   **对比指标**：
    *   **API 名称准确率**：API 名称是否匹配。
    *   **精确全调用准确率（Exact Full-call）**：API 名称及所有参数键值对必须完全匹配。
    *   **持续学习指标**：平均准确率（AA）、向后转移（BWT，衡量遗忘）、向前转移（FWT）等。
*   **评估方式**：全留出生成评估（Full held-out generation），使用正则表达式解析模型生成的 API 调用。

### 4. 资源与算力
*   **硬件环境**：论文提到实验在 **Google Colab** 上运行，使用了 QLoRA 技术以适应有限的硬件资源。

### 5. 实验数量与充分性
*   **实验组数**：主要进行了 A/B 两个条件的对比，每个条件经历了 4 个阶段的顺序训练和交叉评估（4x4 矩阵）。

    *   **客观性**：研究者诚实地指出了这些局限性，并将其定位为“初步试点研究（Pilot Study）”。

### 6. 主要结论与发现
*   **轨迹监督显著占优**：在最终阶段，保留轨迹的条件 B 达到了 **56.9%** 的精确全调用准确率，远高于剥离轨迹的条件 A（**39.2%**）。
*   **工具选择能力提升**：条件 B 在 API 名称选择上的错误极少（仅 12 次），而条件 A 有 102 次。这表明轨迹上下文能显著帮助模型判断“该用哪个工具”。
*   **格式化挑战**：尽管条件 B 选对了工具，但其生成的 API 调用格式错误（Malformed）较多（101 次 vs A 的 45 次），说明长上下文可能干扰了模型对精确输出格式的掌握。
*   **持续学习表现**：两种条件都存在明显的遗忘现象，但条件 B 在维持跨领域准确率方面表现更稳健。

### 7. 优点
*   **切入点精准**：将复杂的“过程监督”问题简化为可量化的 API 轨迹对比，为 LLM 训练数据构建提供了实用建议。
*   **实战导向**：使用了最新的 Llama 3.1 模型和高效的 QLoRA 技术，实验结果对当前开发者具有参考价值。
*   **深入的错误分析**：不仅关注准确率，还详细分析了“选错工具”与“格式错误”的区别。

### 8. 不足与局限
*   **Token 预算不一致**：条件 B 看到的 Token 更多，性能提升可能部分归功于训练强度的增加。
*   **任务范围有限**：仅评估了“预测下一个 API 调用”，而非端到端的对话成功率。
*   **数据合成性**：API-Bank 的轨迹具有一定的结构化和合成性质，可能无法完全代表真实世界中复杂的人类推理过程。
*   **缺乏多样性**：单种子、单模型、单数据集，结论的泛化性有待进一步验证。
