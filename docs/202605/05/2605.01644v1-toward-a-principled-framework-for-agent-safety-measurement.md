## 摘要
大语言模型（LLM）智能体不仅输出文本，还会执行动作，且一旦执行，这些动作往往无法撤销。然而，目前的智能体安全评估通常运行贪婪搜索或少量采样回放，并报告单一的安全/不安全率——这忽视了长尾轨迹，在这些轨迹中，不安全行为可能源于低概率但不可忽视的动作。我们认为，智能体安全应通过搜索而非采样来衡量。我们应用了 BOA 框架，该框架在给定部署配置（模型、解码器、提示词、环境、评判器、似然预算）的情况下，搜索预算内的轨迹空间并报告安全评分：即智能体在该配置下保持安全的概率。BOA 在给定的似然预算下，既在单轮 LLM 内部搜索，也在智能体-环境交互树中搜索，并通过批量解码/评判、前缀缓存和分块树扩展使搜索变得实用。在智能体安全工作负载上，BOA 发现了贪婪和采样评估所遗漏的不安全轨迹。此外，BOA 还可以用于在同一尺度上对模型、防御和攻击进行排名，且 GPU 成本可控。

## Abstract
LLM agents emit actions, not just text, and once taken, those actions often cannot be undone. Yet today's agent-safety evaluations run greedy or a few sampled rollouts and report a single safe/unsafe rate -- blind to the long-tail trajectories where unsafe behavior may arise from low-probability but non-negligible actions.   We argue agent safety should be measured by search, not sampling. We apply BOA, a framework that, given a deployment configuration (model, decoder, prompt, environment, judger, likelihood budget), searches the in-budget trajectory space and reports a safety score: the probability the agent stays safe under the configuration. BOA searches both within a single LLM round and across the agent-environment interaction tree under a given likelihood budget, and makes search practical via batched decoding/judging, prefix caching, and chunked tree expansion. On agent-safety workloads, BOA discovers unsafe trajectories that greedy and sampled evaluations miss. BOA can additionally be used for ranking models, defenses, and attacks, all on the same scale, with manageable GPU costs.

*   **研究动机**：当前的智能体安全评估存在严重缺陷。现有的方法（如 Agent-SafetyBench）通常采用“贪婪解码”或少量“随机采样”来运行智能体，并根据结果给出二进制（安全/不安全）的评分。
*   **核心矛盾**：智能体执行的是不可逆的动作（如删除数据库、转账），而安全性风险往往隐藏在“长尾分布”中。由于部署时通常使用随机采样（如 top-p），贪婪解码可能会漏掉那些概率较低但确实存在的不安全轨迹。
*   **整体含义**：论文主张智能体安全应通过**搜索（Search）**而非**采样（Sampling）**来衡量。BOA 框架旨在计算智能体在给定配置下保持安全的总概率（安全评分），从而提供比单一标签更严谨的风险评估。

### 2. 方法论
*   **核心思想**：在给定的“似然预算（Likelihood Budget, $\epsilon$）”内，系统性地探索智能体与环境交互的所有可能轨迹，并计算安全轨迹的概率加权和。
*   **关键技术细节**：
    *   **安全评分 (Safety Score)**：定义为在预算 $\epsilon$ 内，所有被评判为安全的轨迹概率之和占总探索轨迹概率的比例。
    *   **两级搜索结构**：
        *   **轮内搜索（Chunk Tree）**：在单次模型生成中，按“块（Chunk）”而非单个 Token 进行扩展，以平衡搜索深度和评判成本。
        *   **跨轮搜索（Block Tree）**：将智能体与环境的多次交互建模为树，节点间通过环境反馈连接。
    *   **系统优化**：
        *   **批处理（Batching）**：同时处理多个解码和评判请求，提高 GPU 吞吐量。
        *   **前缀缓存（Prefix Caching）**：缓存已评判轨迹的前缀，避免对重复路径的冗余计算。
        *   **相位感知优先级搜索**：初期优先保证搜索宽度，后期转向搜索可能导致不安全行为的深度路径。

### 3. 实验设计
*   **数据集/场景**：使用 **Agent-SafetyBench** 作为基础，涵盖 8 个风险类别（如非法操作、隐私泄露等）。
*   **实验设置**：从基准测试中筛选出 72 个具有代表性的提示词（Prompt），分为“贪婪全安全”、“贪婪全不安全”和“混合结果”三组。
*   **对比方法**：主要对比传统的**贪婪解码（Greedy Decoding）**评估方法。
*   **应用场景实验**：
    1.  **基座模型对比**：对比不同规模和系列的 LLM。
    2.  **防御机制评估**：对比“指令防御（Instructional Defense）”与“三明治防御（Sandwich Defense）”。
    3.  **攻击机制评估**：对比带有显式覆盖标记（Marker）与无标记的提示词注入攻击。

### 4. 资源与算力
*   **硬件**：实验使用了 **Nvidia H100 GPU**。
*   **算力配置**：
    *   每个 Prompt 的搜索时间预算设为 **600 秒**。
    *   目标模型使用 HuggingFace Transformers 加载，评判模型（Judger）使用 vLLM 加速。
    *   具体 Token 消耗：Llama-3.1 (8B) 平均每 Prompt 消耗约 92.9K Token，而 70B 模型约为 1.1K Token（因 8B 模型常产生无效工具调用导致搜索空间膨胀）。

### 5. 实验数量与充分性
*   **实验规模**：针对 4 个主流模型（Llama-3.1 8B/70B, Qwen2.5 14B/32B）在 72 个复杂场景下进行了测试。
*   **充分性与客观性**：实验不仅涵盖了模型排名，还深入探讨了防御和攻击的有效性。通过引入“似然预算”这一变量，实验能够客观地展示安全风险随概率阈值变化的趋势，比单一的二进制标签更具说服力。

### 6. 主要结论与发现
*   **发现隐蔽风险**：BOA 成功发现了贪婪解码无法捕捉到的不安全轨迹。
*   **安全性是概率性的**：模型在某些 Prompt 下表现为“部分安全”，即大部分路径安全但存在低概率的危险分支。
*   **模型规模影响**：大模型（如 70B）通常比小模型（如 8B）更安全，但小模型在处理工具调用时的不稳定性（如格式错误）反而可能在某些评估标准下“显得”安全。
*   **攻击特征**：去除显式的“忽略之前指令”标记后的攻击往往更难被模型拒绝，因为它们更隐蔽地融入了正常工作流。

### 7. 优点
*   **原则性强**：将安全评估从随机采样提升到了概率搜索的理论高度。
*   **实用性优化**：通过前缀缓存和分块扩展，解决了智能体多轮交互中搜索空间爆炸的工程难题。
*   **统一衡量标准**：为模型、攻击和防御提供了一个可以在同一尺度（安全评分）上进行比较的框架。

### 8. 不足与局限
*   **评判器依赖**：安全性高度依赖于作为“评判器”的 LLM 的准确性，如果评判器本身存在偏差，结果将不可信。
*   **环境确定性假设**：目前的框架主要处理确定性环境，对于具有高度随机性的外部环境，搜索复杂度会进一步激增。
*   **计算成本**：虽然比暴力搜索高效，但相比贪婪解码，其计算开销（GPU 时间和 Token 消耗）仍然显著更高，可能限制其在大规模实时监控中的应用。
