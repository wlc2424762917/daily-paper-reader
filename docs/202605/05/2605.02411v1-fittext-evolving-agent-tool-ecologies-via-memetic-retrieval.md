## 摘要
用户描述任务的方式与工具文档记录的方式之间存在语义鸿沟。随着 API 生态系统扩展到数万个端点，仅依靠初始查询的静态检索无法弥合这一鸿沟：智能体对所需工具的理解在执行过程中不断演化，但其工具集却并非如此。我们引入了 FitText，这是一个无需训练的框架，通过将检索直接嵌入到智能体的推理循环中，使检索过程动态化。FitText 生成自然语言伪工具描述作为检索探针，利用检索反馈进行迭代优化，并通过随机生成探索多种替代方案。模因检索（Memetic Retrieval）在工具记忆的引导下，对候选描述施加演化选择压力，从而避免冗余搜索。在 ToolRet（包含 4 个领域的 4.3 万个工具）上，FitText 将平均检索排名从 8.81 提升至 2.78；在 StableToolBench（包含 16,464 个 API）上，它实现了 0.73 的平均通过率，相比静态查询检索获得了 24 个百分点的绝对提升。这种增益可以迁移到能够胜任语义算子的基础模型上；在较弱的基础模型下，模因演化搜索会发生反转——放大噪声而非提炼信号——这表明模型能力是进行演化工具探索的前提。

## Abstract
A semantic gap separates how users describe tasks from how tools are documented. As API ecosystems scale to tens of thousands of endpoints, static retrieval from the initial query alone cannot bridge this gap: the agent's understanding of what it needs evolves during execution, but its tool set does not. We introduce FitText, a training-free framework that makes retrieval dynamic by embedding it directly in the agent's reasoning loop. FitText generates natural-language pseudo-tool descriptions as retrieval probes, refines them iteratively using retrieval feedback, and explores diverse alternatives through stochastic generation. Memetic Retrieval adds evolutionary selection pressure over candidate descriptions, guided by a tool memory that avoids redundant search. On ToolRet (43k tools, 4 domains), FitText improves average retrieval rank from 8.81 to 2.78; on StableToolBench (16,464 APIs), it achieves a 0.73 average pass rate--a 24-point absolute gain over static query retrieval. The gains transfer across base models capable of acting as competent semantic operators; under weaker base models, Memetic's evolutionary search inverts--amplifying noise rather than refining signal--surfacing model capacity as a prerequisite for evolutionary tool exploration.

### 论文总结：FitText —— 通过模因检索演化智能体工具生态

#### 1. 核心问题与整体含义（研究动机和背景）
在大型语言模型（LLM）驱动的智能体系统中，如何从海量 API（数以万计）中准确检索工具是一个关键挑战。现有的方法主要存在两个局限：
*   **语义鸿沟（Semantic Gap）：** 用户的高层目标描述（如“帮我退票”）与工具的底层功能文档（如“查询航班状态”）在表达上存在巨大差异。
*   **静态检索的局限性：** 传统方法通常在任务开始前进行一次性检索，但智能体对工具的需求往往随着推理过程的深入而演变，静态检索无法捕捉这种动态变化。
**FitText** 的提出旨在将工具检索直接嵌入智能体的推理循环中，通过动态生成和演化“伪工具描述”来弥合语义鸿沟。

#### 2. 方法论：核心思想与关键技术
FitText 是一个**无需训练（Training-free）**的框架，其核心思想是将工具检索视为一种在异构工具生态中的“演化探索”。

*   **核心思想：** 智能体不直接使用用户查询进行检索，而是根据当前推理状态生成**伪工具描述（Pseudo-tool descriptions）**。这些描述是关于所需工具功能的自然语言假设，作为检索探针。
*   **关键技术策略：**
    1.  **Single-Pass（单次）：** 生成一个伪描述并检索。
    2.  **Multi-Turn Refinement（多轮优化）：** 利用检索到的工具示例作为反馈，迭代修正伪描述。
    3.  **Scattershot（随机散射）：** 在高温度设置下并行生成多个多样化的描述，通过投票机制确定最终工具。
    4.  **Memetic Retrieval（模因检索）：** 引入进化算法。将伪描述视为“模因（Meme）”，通过**交叉（Crossover）**、**变异（Mutation）**和**适应度选择（Fitness Selection）**进行多代演化。
*   **工具记忆（Tool Memory）：** 记录已探索的工具路径，通过 Jaccard 相似度惩罚项避免冗余搜索，引导种群向未探索的语义空间移动。

#### 3. 实验设计
*   **数据集/场景：**
    *   **ToolRet：** 大规模检索基准，包含 43,000 个工具，涵盖代码、Web、自定义工具等 4 个领域。
    *   **StableToolBench：** 包含 16,464 个 API 的端到端智能体任务基准，分为 G1（单工具）、G2（同类多工具）、G3（跨类多工具）三个难度等级。
*   **对比方法：** 静态查询检索（Static Query Retrieval）、Single-Pass、Multi-Turn、Scattershot 以及本文核心的 Memetic 策略。
*   **评价指标：** NDCG@5、Precision@5、Recall@5、Comprehensiveness@5 以及端到端任务通过率（Pass Rate）。

#### 4. 资源与算力
*   **训练算力：** 论文明确指出该框架是 **Training-free** 的，因此**不需要** GPU 进行模型训练。
*   **推理资源：** 实验使用了 **GPT-4.1-mini** 和 **Qwen3-30B** 作为基础模型。
*   **开销说明：** 虽然不需要训练，但 Memetic 策略在推理时会产生多次 LLM 调用（种群规模 N=6~8，进化代数 G=3），相比静态检索增加了 Token 消耗。

#### 5. 实验数量与充分性
*   **实验规模：** 论文在超过 5,000 个查询（ToolRet）和 6 个子场景（StableToolBench）上进行了测试。
*   **消融实验：** 进行了关于查询分析模型（GPT-4.1 vs Qwen3）、优化轮数（T0-T3）以及模型能力对进化效果影响的深入分析。
*   **客观性：** 实验对比了不同规模和类型的模型，并揭示了该方法在弱模型上的失效现象（反转效应），体现了研究的客观性和公平性。

#### 6. 主要结论与发现
*   **性能飞跃：** 在 ToolRet 上，平均检索排名从 8.81 提升至 2.78；在 StableToolBench 上，平均通过率达到 0.73，比静态检索高出 24 个百分点。
*   **复杂任务表现优异：** 在最困难的 G2/G3 任务中增益最明显，证明了动态演化在处理模糊需求时的有效性。
*   **模型能力门槛：** 发现模因搜索存在“能力依赖”。强模型（如 GPT-4）能通过进化提炼信号，而弱模型（如 Qwen3-30B）在进化过程中会放大噪声，导致性能反而下降。
*   **瓶颈转移：** 实验表明，当前智能体性能的瓶颈在于**检索质量**而非规划能力。

#### 7. 优点
*   **即插即用：** 无需对模型进行微调，适用于任何具备基本推理能力的 LLM。
*   **动态适应：** 解决了智能体在长程任务中工具需求变化的问题。
*   **语义对齐：** 通过伪描述生成，巧妙地解决了用户语言与 API 文档之间的表达差异。
*   **进化机制：** 引入模因算法为离散文本空间的优化提供了新思路。

#### 8. 不足与局限
*   **推理延迟与成本：** 多代进化和并行采样显著增加了 API 调用次数和响应延迟，在实时性要求极高的场景下受限。
*   **对基础模型的依赖：** 该方法在较弱的开源模型上表现不佳，限制了其在低算力环境或非前沿模型上的应用。
*   **记忆管理的复杂性：** 随着任务复杂度的增加，如何更高效地管理和检索“工具记忆”仍有待进一步优化。
