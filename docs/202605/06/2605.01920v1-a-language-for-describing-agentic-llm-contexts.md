## 摘要
大语言模型越来越多地被应用于更大的系统（即“LLM 智能体”）中。这些系统会进行一系列 LLM 调用，每次调用都会为 LLM 提供指令、观察结果和交互历史的组合。编码信息的结构设计在最终系统的质量中起着核心作用，从而引发了对上下文工程（context engineering）的投入。因此，传达系统中 LLM 上下文的构成及其随时间的演变过程至关重要。然而，目前尚无此类标准：上下文的构建通常通过非正式的描述、临时图表或直接检查代码来传达，这些方式都无法精确捕捉提示词（prompt）在交互步骤中是如何演变的，也无法体现两种上下文表示策略之间的差异。为解决这一问题，我们引入了代理式上下文描述语言（Agentic Context Description Language, ACDL），这是一种以精确、易读且标准的方式指定 LLM 输入上下文结构和动态的语言，并配有可视化工具。ACDL 提供了用于指定上下文各个方面的构造，例如角色消息序列、动态内容、时间索引引用以及条件或迭代结构，从而独立于任何特定实现来捕捉提示词的完整架构。ACDL 图表可以手绘在白板上，也可以用正式语言编写并进行渲染。我们描述了该语言，通过记录几个现有系统及其变体来演示它，并鼓励社区在日常交流和论文中使用它来描述 LLM 系统的上下文。工具、示例和文档可在 www.acdlang.org 获取。

## Abstract
Large language models are increasingly used within larger systems ("LLM agents"). These make a sequence of LLM calls, each call providing the LLM with a combination of instructions, observations, and interaction history. The design of the encoded information and its structure play a central role in the quality of the resulting system, leading to efforts spent on context engineering. It is therefore critical to communicate the composition of the LLM context in a system, and how it evolves over time. Yet, no standard exists for doing so: context construction is typically conveyed through informal prose, ad hoc diagrams, or direct inspection of code, none of which precisely capture how a prompt evolves across interaction steps or how two context representation strategies differ. To remedy this, we introduce the Agentic Context Description Language (ACDL), a language for specifying the structure and dynamics of LLM input contexts in a precise, readable, and standard manner, along with visualizations. ACDL provides constructs for specifying context aspects such as role message sequences, dynamic content, time-indexed references, and conditional or iterative structure, capturing the full architecture of a prompt independently of any particular implementation. ACDL diagrams can be hand drawn on a whiteboard, or written in formal language which can then be rendered. We describe the language, demonstrate it by documenting several existing systems and their variants, and encourage the community to adopt it for describing LLM systems context, both in day-to-day communication and in papers. Tooling, examples and documentation are available at www.acdlang.org.

### 1. 论文的核心问题与整体含义（研究动机和背景）
*   **核心问题**：在构建 LLM 智能体（Agent）时，如何设计和组织发送给模型的“上下文”（Context）是决定系统性能的关键（即“上下文工程”）。然而，目前缺乏一种标准化的方式来精确描述上下文的结构、内容来源以及随时间的动态演变。
*   **研究背景**：现有的描述方式通常依赖于非正式的文字叙述、随意的流程图或直接阅读源代码。这导致开发者难以准确沟通设计意图，研究者难以复现他人的系统，且难以对比不同上下文策略（如 ReAct 的不同变体）之间的细微差别。
*   **整体含义**：论文提出了 **ACDL（Agentic Context Description Language）**，这是一种专门用于描述代理式 LLM 上下文的领域特定语言（DSL），旨在成为该领域的“UML”或“电路图”。

### 2. 论文提出的方法论：ACDL 语言规范
*   **核心思想**：将上下文抽象为信息的线性序列，通过符号化标签描述每个元素的角色、类型和来源，并利用时间索引和控制流捕捉其动态演化。
*   **关键技术细节**：
    *   **角色消息（Role Messages）**：支持 System (S)、User (U)、Assistant (A)、Tool (T) 和 None (N) 五种角色。
    *   **时间索引（Time Indexing）**：使用 `@T` 表示当前主步骤，`@t` 表示历史步骤，支持子步骤（Sub-steps）嵌套（如 `@T.I`）。
    *   **信息来源（Information Sources）**：区分环境状态（`env`）、系统内部状态（`sys`）和模型响应（`resp`）。
    *   **控制结构**：引入 `ForEach`（循环遍历历史）、`If/Else`（条件注入内容）和 `Switch` 等逻辑。
    *   **片段（Fragments）**：支持可重用的代码块（`StrFrag` 和 `RolesFrag`），提高描述的模块化。
    *   **可视化渲染**：ACDL 代码可以自动渲染为结构化的图表，使复杂的上下文结构一目了然。

### 6. 论文的主要结论与发现
*   **上下文结构至关重要**：实验表明，是否在历史中保留推理轨迹（Reasoning Traces）或如何划分消息角色，会显著影响模型在推理和代码任务上的表现。
*   **ACDL 的有效性**：ACDL 能够以简洁的代码捕捉到 prose（散文式描述）难以表达的逻辑，如“每 25 步注入一次进度报告”或“在特定条件下截断历史”。
*   **标准化价值**：ACDL 为社区提供了一个共享词汇表，有助于提升智能体研究的可复现性和透明度。

### 7. 优点
*   **填补空白**：首次针对智能体上下文工程提出了形式化的描述标准。
*   **工具链完备**：不仅提供了语言规范，还提供了 VS Code 插件、Web 编辑器和 Claude Code 技能，降低了采用门槛。
*   **抽象层次适中**：成功分离了“上下文工程”（宏观结构）与“提示工程”（微观措辞），使设计者能专注于逻辑架构。

### 8. 不足与局限
*   **状态假设限制**：ACDL 目前假设系统状态在上下文构建期间是不可变的。对于那些在构建 Prompt 过程中会产生副作用或改变状态的极端复杂系统，描述起来较为繁琐。
*   **多智能体时钟同步**：在描述具有独立时钟且共享可变状态的多智能体系统时，ACDL 的表达方式尚不够优雅。
*   **学习曲线**：虽然比阅读代码简单，但对于非技术背景的提示词工程师来说，掌握其形式化语法仍需一定时间。
