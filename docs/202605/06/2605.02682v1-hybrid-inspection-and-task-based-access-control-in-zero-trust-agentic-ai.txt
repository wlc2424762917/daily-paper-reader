# Hybrid Inspection and Task-Based Access Control in Zero-Trust Agentic AI

Abstract —Authorizing Large Language Model (LLM)-driven agents to dynamically invoke tools and access protected resources introduces significant security risks, and the risks grow dramat-ically as agents engage in multi-turn conversations and scale toward distributed collaboration. A compromised or malicious agentic application can tamper with tool calls, falsify results, or request permissions beyond the scope of the subject’s in-tended tasks, which could go unnoticed with current delegated authorization flows given their lack of visibility into the orig-inal subject’s intent. In light of this, we make the following contributions towards Continuous Agent Semantic Authorization (CASA). First, we propose a hybrid runtime enforcement model that combines deterministic and semantic controls enabled by a zero-trust interception layer. Five deterministic controls enforce structural and data-integrity guarantees over the message flow, while a semantic inspection layer evaluates whether tool call choices align with the intended tasks commissioned to the agent. Second, differently from prior Task-Based Access Control (TBAC) techniques that operate on single-turn interactions, we decompose the semantic layer into two stages: i) a task-extraction step that distills the subject’s objectives from multi-turn conversations at the interception layer, and ii) a task-tool semantic matching step at the authorization server that evaluates whether the requested tools are appropriate for the extracted tasks. Third, we extend the ASTRA dataset that we introduced in a prior work, by generating novel conversation-tool datasets with multi-turn interactions containing relevant and irrelevant tool calls for a given task *. Lastly, we provide the first experimental results for TBAC under multi-turn conversations.

I. I NTRODUCTION 

Large Language Model (LLM)-driven agents substantially expand application capabilities, yet they also introduce sig-nificant runtime security risks. These risks intensify as agents scale toward distributed multi-agent-human collabora-tion, where agents from different organizations may invoke tools across trust boundaries. In a typical agentic architecture, the application operates as an intermediary between a subject (a user or another agent), one or more LLMs, and tools, such as those provided by a Model Context Protocol (MCP) Server [1]. This intermediary position grants the application broad autonomy: it relays prompts, dispatches tool calls, and returns results, yet the surrounding components implicitly trust it to do so faithfully. As shown in Figure 1, a compromised, defective, or mali-cious application can exploit this position in several ways. Tool calls may be hard-coded within the application, bypassing the LLM’s decision-making process; the application may substi-tute one tool for another, tamper with parameters in transit, poison the tool definitions presented to the LLM, or falsify the data returned by backend services. Furthermore, agents might invoke tools that are technically within the allowed permissions, but operate outside the intended scope of the tasks

they were asked to perform, creating potential attack vectors for malicious actors. These risks are compounded by fundamental limita-tions in current delegated authorization flows. State-of-the-art authorization servers, operating under protocols such as OAuth 2.0 [2] and its extensions [3], authorize access requests based on statically configured policies without visibility into the original tasks or intents conveyed to the agent. As a result, an agent may request, and be granted, access to protected re-sources beyond the scope of the commissioned tasks, whether unintentionally (e.g., a trusted agent making a mistake) or intentionally (e.g., a rogue agent inserting additional scopes). This problem is especially concerning when the additional scopes enable write or delete operations on protected re-sources. Approaches that predefine all allowable scopes for an agent only bound the blast radius without bridging the gap between the subject’s intent, and the agent’s intent reflected in the access request sent by the agent to an Authorization Server during a delegated authorization flow [4]. Beyond the various issues of inserting a human in the loop, agents may exploit a social engineering attack targeting gradual trust building, being highly patient in building trust, to then deceiving the user, controller, or evaluator [5]. To address these challenges, we propose a hybrid runtime enforcement framework combining two complementary cate-gories of controls within a zero-trust interception layer, adding guardrail and semantic control components toward emerging paradigms focused on secure, interoperable, and semantically aligned intents for distributed agent collaboration [6], along with novel evaluation datasets: 1) Deterministic controls : strict, binary-logic verification of structural and data-integrity properties of the message flow, catching violations such as unauthorized tool exe-cution, tool call swaps, parameter tampering, and result falsification. 2) Semantic controls : intent-level inspection evaluating if the LLM’s tool call choices align with the subject’s commissioned tasks, enabling the authorization server to grant only the set of scopes needed for such tasks. 3) Evaluation datasets : multi-turn conversations between a subject and a tool-equipped assistant agent, with both relevant and irrelevant tool calls occurring in the multi-turn exchange. The deterministic controls serve as a guardrail technology enforcing structural compliance, while the semantic ones act as a cognitive engine that reasons about intent alignment, which are complementary and essential control blocks as agents scale beyond isolated deployments. As shown in Figure 2, the deterministic checks catch structural violations that a semantic classifier might miss, while the semantic checks detect intent-level manipulations that pass all binary equality tests. For the semantic checks, we extend beyond prior work [4], which operates on single, explicitly stated tasks, to handle multi-turn conversational contexts. We decompose the semantic check into two stages: a task-extraction step at the interception layer that distills the subject’s objective from the conversation, and a task-tool matching step at the authorization server that evaluates whether the requested tool aligns with the extracted task. Finally, by building on the enhanced delegated authorization flows for tool access described in our prior work [4], we target that the agent is acting in a transparent way and within the intent of the subject during the conversation. We generate and curate two datasets to evaluate this frame-work. First, we extend and curate the task-tool dataset in-troduced in our prior work, ASTRA [4], [7], which pairs natural-language tasks with MCP Server tools and relevance labels. Then, we propose a novel methodology to create conversation-tool datasets of multi-turn conversations con-taining tool calls, built with the curated task-tool labeled samples as conversation-generation seeds. We run extensive experiments with multiple state-of-the-art models powering both our data generators and our semantic check pipeline. Our results benchmark the accuracy of several semantic check models, with a focus on precision and recall, and highlights key challenges inherent to conversational agent settings.

II. R ELATED WORK 

A. Delegated Authorization and Access Control 

Modern authorization systems have evolved to address the increasing complexity of digital identities, including Non-Human Identities (NHI), and resource management in dis-tributed environments, producing more expressive and context-aware methods such as Google’s Zanzibar [8]. Delegated authorization protocols, most prominently OAuth 2.0 [2] and OAuth 2.1 [3], have become the standard for granting appli-cations limited access to protected resources on behalf of a principal. Recent RFCs for Protected Resource Metadata [9], Authorization Server Metadata [10], and Dynamic Client Reg-istration [11] have further enhanced the flexibility of OAuth-based systems, enabling dynamic discovery of endpoints and scopes. Complementary proposals such as Okta’s Cross Ap-plication Access protocol (XAA) [12] extend OAuth with dynamic, cryptographically-bound associations among users, agents, and resource applications, yet do not address the core challenge of aligning agent-generated access requests with the original intent, particularly in the presence of ambiguous or multi-turn conversational prompts. At the infrastructure level, access-controlled website designs that extend decentralized authorization toolkits with trust-based, per-agent session keys have also been explored [13], though these approaches enforce statically configured scope boundaries and do not incorporate semantic analysis of the delegating subject’s intent. 

B. Semantic Task-to-Scope Matching 

In agentic systems, the subject’s intent is usually embedded within multi-turn conversations, that may contain clarification exchanges, tangential information, and conversational drifts. Extracting a coherent task description from such conversa-tional context is itself a non-trivial problem, closely related to intent detection in dialogue systems [14]–[16]. Our prior work [4] explored two task-to-tool strategies in the single-task setting: a semantic similarity matcher that compares embed-dings between an idealized tool description and available tools, and an LLM reasoning matcher that prompts a language model for a direct appropriateness judgment. In this paper, we extend the semantic inspection pipeline presented in [4] to operate on multi-turn conversations, by decomposing the problem into an explicit task-extraction stage followed by task-tool matching. 

C. Task-to-Tool Data 

Agents that are given access to tools must be able to perform an accurate tool selection process to achieve their assigned tasks. Considerable effort is thus spent on creating tool selection data for benchmarking the agents [17]–[20]. More recently, hard-tool data samples that agents fail on were created [21], showing that retraining the models on those synthetic samples leads to improvements, and further proving the utility of synthetic data, and this idea is pushed further by the recent TopoCurate [22]. While in previous efforts tools were optimized in an outcome-driven way to help the agents achieve their downstream objectives, a novel approach has created tools more aligned with tasks themselves [23]. Such tools become, by design, better aligned with the intentions of specific tasks. However, the emphasis is still on selecting the best possible tool for a given task, and not on semantic alignment from a permission scope perspective. Previously, we focused on the simpler case of single-task tool calls [4]. That approach, however, does not address multi-turn conversations that lead to a tool call. The problem of obtaining diverse realistic multi-turn data leading to a tool call is a significant challenge in the literature (see, e.g., [24], where the authors parse problem-solving texts containing a sequence of steps, and exploit them as a source for multi-turn tasks with tool calling). Overall, authenticated delegation for tool selection in multi-turn exchanges remains an open challenge for current research [4], [25], with an absence of evaluation benchmarks and public datasets containing authorized and unauthorized tool requests for a given objective. In this work, we propose novel approaches to generate conversation data with tool calling agents, not only targeting authorized tools but also unauthorized attempts, thus enabling the creation of the first benchmark for this problem. III. M ETHOD 

To mitigate the runtime threats outlined in Section I, we introduce a stateful interception layer operating under zero-trust principles [26]. The interception layer encapsulates the agentic application, intercepting all ingress and egress traffic (subject queries, LLM prompts and responses, MCP Server tool calls and results) and maintaining a real-time ledger of the conversation state, authoritative tool definitions, and LLM-generated requests. The deterministic and semantic runtime checks described, respectively, in Sections III-A and III-B are enforced at this layer. 

A. Deterministic Runtime Checks 

The deterministic checks enforce structural and data-integrity guarantees through strict, binary logic rather than probabilistic classification, ensuring high reliability for verifi-cation. Five such checks are defined and described below, and could be enabled or disabled within the interception layer on a per-agentic-application basis. Appendix A presents worked examples illustrating both the unprotected failure case and the corresponding mitigation for every check. 

a) Tool Definition Integrity Check: When the agentic application first connects to an MCP Server, the interception layer caches the authoritative tool definitions, e.g., names, descriptions, and parameter schemas returned by the server. On every subsequent LLM request that includes tool definitions, this check compares the definitions supplied by the applica-tion against those originally cached. Any discrepancy (e.g. swapping or modifying the description of a tool) causes the request to be blocked before it reaches the LLM. This prevents tool-definition poisoning, in which the application deceives the LLM about what each tool does, thereby manipulating the LLM’s tool selection (see Appendix A-A). Note that, the problem of building trust with the MCP Server and the tools that it exposes (e.g., using verifiable credentials or other techniques) is outside the scope of this paper. 

b) Request Authorization Verification: After each LLM response, the interception layer records whether the response contains one or more tool call request entries or only a text reply. If the application subsequently attempts to execute a tool call when the LLM issued no such request, the call is blocked. This mitigates unauthorized tool execution, ensuring the application cannot act autonomously outside the LLM’s reasoning loop (see Appendix A-B). Practically speaking, some agent applications may require hard-coded tool calls to function correctly. To accommodate this, the interception layer settings can include an allowlist of tool names that bypass this validation check. 

c) Action Alignment Validation: When the LLM issues a tool request, the interception layer logs the function name. This check compares this against the function name in the outgoing MCP Server tool call constructed by the application. In case of mismatch, such as when the LLM requests calling a tool named get_balance , but the application solicits access to a tool named transfer_amount_between_accounts ,the tool call swap is flagged and the call is blocked (see Appendix A-C). 

d) Parameter Integrity Enforcement: Beyond verifying the function name, this deterministic check inspects the spe-cific parameter names and values in the outgoing tool call and compares them against those specified by the LLM, blocking calls that don’t match. This mitigates parameter tampering, preventing the application from hijacking a legitimate transac-tion by silently altering arguments in transit. For instance, by redirecting a fund transfer to a different destination account (see Appendix A-D). 

e) Data Fidelity Verification: When a tool call com-pletes, the interception layer captures the raw response re-turned by the MCP Server. This check compares this against the content the application attempts to relay back to the LLM. If the application modifies the result, e.g., by reporting a balance of 0 when the server returned 1,234, the tampered message is blocked. This mitigates result-data falsification, ensuring the LLM operates on unmodified data from the system of record and preventing the application from inducing hallucinated responses or triggering unwarranted follow-up tool calls (see Appendix A-E). These five deterministic checks operate on observable, syn-tactic properties of the message flow. They cannot reason about the semantic intent of the content traversing the pipeline. This creates a gap: if a malicious application injects adversarial instructions into the system prompt, for example, directing the LLM to initiate a fund transfer before answering the subject’s question, the resulting tool call is structurally indistinguishable from a legitimate LLM-initiated request. The deterministic controls will observe that the LLM did request a transfer, that the function name matches, and that the parameters are unaltered, and will therefore permit the call. Appendix A-F illustrates this scenario in detail. Checks (b), (c), and (d) also assume the LLM itself is not compromised. The semantic layer relaxes this assumption by independently verifying alignment between the subject’s intent and the LLM’s decisions. 

B. Semantic Runtime Checks 

For the semantic checks, which require evaluating whether tool call choices align with the subject’s original objective, the system must handle multi-turn conversations in which the subject’s intent may be distributed across several messages and interleaved with assistant responses, clarification exchanges, and potential unrelated information. We decompose the se-mantic inspection into two stages: 1) extracting the subject’s task from the conversation, and 2) matching the extracted task against the requested tool, extending the single-task matching model that we proposed in [4]. This two-stage decomposition keeps the authorization server’s interface narrow, reducing latency, cost, and the attack surface exposed by transmitting sensitive conversational content. The practical effectiveness of this approach depends on the accuracy of both stages. Extraction errors, such as omitting a critical sub-task or hallucinating an objective not present in the conversation, propagate downstream and can cause the matcher to produce incorrect access decisions. Matching errors similarly affect the balance between over-scoping and 

under-scoping . That is, between an overly permissive matcher that may grant access to tools that fall outside the subject’s intent, therefore creating a security risk, versus an overly strict matcher that may block legitimate tool requests, degrading agent utility and potentially stalling task execution. 

1) Conversation Task Extraction: This first stage leverages the interception layer. Given a multi-turn conversation, in-cluding user messages, assistant responses, and any prior tool call results, the extractor produces a concise natural-language task description that captures the subject’s objective. This step serves two purposes: it reduces multi-turn interactions to a compact representation, and it decouples the downstream matcher from the specifics of conversational structure. Task extraction depends critically on how conversational context (i.e., memory) is handled. Strategies range from local heuristics that consider only the initial user utterance to approaches that incorporate the full dialogue history via LLM-based summarization. In our experiments, we adopt the latter, as described in Section V. 

2) Task-Tool Matching: This second stage extends the authorization server. Given an extracted task from the Con-versation Task Extractor and the name and description of a requested tool, the matcher outputs a binary decision indicating whether the tools requested are semantically appropriate for the intended task. This problem can be formulated as a bi-nary classification task, admitting multiple solution strategies, including embedding-based similarity, LLM-based reasoning, and fine-tuned classifiers. In our experiments, we adopt an LLM-based reasoning approach (Section V). The matcher evaluates each tool request independently. For tasks requiring multiple tools, a separate matching decision is made per tool. This per-tool granularity ensures that the authorization server can grant or deny access at the level of individual scope requests. IV. D ATA GENERATION APPROACH 

A. Task-Tool Data Curation 

We curated our dataset, ASTRA [7], using a multi-stage pipeline that combines automated AI consensus filtering with human-in-the-loop verification. This process produced reliable task–tool pairs, which serve as inputs to the conversation generation pipelines as described in the following sections. Each original sample comprises (i) a natural-language task description synthesized from a seed tool, (ii) a requested tool (name and description) drawn from 12 enterprise MCP servers, (iii) a relevance label indicating whether the requested tool is appropriate for the task, and (iv) a label type. We defined the following labels:  

> •

Relevant-seed : the test tool is the same tool from which the task was originally synthesized.  

> •

Relevant : the test tool differs from the seed tool but is nonetheless semantically relevant to the task.  

> •

Irrelevant : the test tool belongs to the same MCP Server as the seed tool but is not relevant.  

> •

Irrelevant-external : the test tool belongs to a different MCP Server entirely. The data curation pipeline consists of the following steps. We employ five large language models (Claude Sonnet 4.6, Claude Sonnet 4.5, GPT-5.2, Gemini 3 Pro, and O3 Pro) to annotate the task–tool pairs. We remove samples for which all models agree on a prediction that contradicts the synthetic ground truth, as this indicates either mislabeled external tools (null-match with unanimous positive votes) or flawed gener-ation (seed-match with unanimous negative votes). For same-server tools initially labeled as irrelevant but unanimously judged relevant, we relabel them as relevant. For the remaining ambiguous cases, where model annotations partially disagreed with the ground truth, we conduct human annotation with five independent annotators. We retain only samples with full agreement (5/5), discard the rest, and use the consensus labels to override both relevance labels and match tags. The resulting dataset comprises curated validation and test splits as detailed in Table V, with all discarded samples retained for auditability. These splits are subsequently used to generate conversation–tool datasets. 

B. Augmented Conversation Generation Strategy 

The augmented conversation-tool dataset is produced by a three-stage pipeline that transforms labeled (task, tool) pairs into multi-turn conversations containing a single tool invocation. The goal is to generate structurally varied con-versations that exercise both focused and subtly misdirected interaction patterns, producing a dataset that can rigorously evaluate whether downstream models distinguish relevant tool calls from irrelevant ones in the presence of conversational noise. Each conversation ends at the first tool call, because the augmented strategy controls variant structures determin-istically. Extending the conversation beyond this point would require modeling the tool’s response and subsequent subject behaviour, which is instead handled by the agentic strategy described in Section IV-C. The conversation generation strategy comprises three stages: Generation, Review, and Judging. Stage 1 (Generation) synthe-sizes eight structurally distinct conversation variants per input sample using a battery of LLM-driven sub-generators. Stage 2 (Review) enriches every generated sample with an LLM-evaluated objective-alignment check as metadata. Stage 3(Judging) applies deterministic filters that discard samples which are too short or misaligned with the original subject’s objective. For the published dataset, GPT5.2 (OpenAI) was used as the LLM in each of the three stages with varying reasoning levels at each step. We store conversations in the OpenAI API conversation format, and so we refer to the subject as ’user’ when describing the strategy. The following sections describe each stage of the pipeline in detail. 

Generation: Consists of three main parts, namely, Preprocess-ing, Augmented Conversation Generation, and Validation. 

1) Preprocessing: A preprocessing step filters out relevant-seed samples, whose test tool is identical to the ground-truth tool from which the task was derived. Because the augmented pipeline generates conversation variants that deliberately steer toward the test tool (via user drift and assistant drift), applying these perturbations when the test tool and ground-truth tool are the same would produce conversations with drift related to the user objective, and so there would be no meaningful contrast between relevant and misdirected behaviour to evaluate. After filtering, 461 test samples and 559 validation samples entered into the generation stage. 

2) Augmented Conversation Generation: For each input task-tool sample, the generator produces eight conversation samples defined by the Cartesian product of three binary dimensions (see Table I): User style, Assistant style and Tool selection. This yields eight variants per input. All eight share a common group identifier linking them to the same source task. The four ground-truth-tool variants always receive the label 

relevant with type relevant-seed , since the ground-truth tool is intrinsically correct for the task. The four test-tool variants inherit the relevance label from the source task-tool sample (i.e., relevant, irrelevant, or irrelevant-external). Appendix B presents a complete worked example showing all eight variants generated from a single input sample. Each conversation is built through a multi-step LLM-driven pipeline. The number of user turns n is sampled uniformly from {1, 2, 3, 4, 5}, yielding conversations of 2 to 10 messages, alternating user and assistant turns with a final assistant message containing the tool call. The pipeline first constructs a focused, on-task conversation (Steps 1-3 below), then derives the drift variants by perturbing the assistant and user messages (Steps 4-5 below), and finally generates tool calls for both the ground-truth and test tools (Step 6 below). 

Step 1: Focused-user message generation: An LLM decomposes the task objective into n user messages that progressively reveal the request, distributing details naturally across turns without explicit connectors. Step 2: Focused-assistant response generation: Empty assistant placeholders are inserted between consecutive user messages. An LLM fills each placeholder with a contextually appropriate response conditioned on the preceding messages and the ground-truth tool definition. The assistant is instructed not to fabricate data or anticipate information the user has not yet provided. 

Step 3: Assistant review: A reviewer pass removes pre-scient references from assistant messages, instances where the assistant appears to know what the user will say in a subsequent turn. This corrects unnatural conversational flow that would arise from the model having seen the full user message sequence during generation. 

Step 4: Drift-assistant generation: The assistant mes-sages are rewritten so that the first response naturally requests information the user has not provided but that would be needed to call the test tool. This drift is designed to appear as an incidental clarification rather than intentional information-gathering, making it a realistic conversational pattern. 

Step 5: Drift-user generation: Distracting data, parameter values that would satisfy the test tool’s required arguments, are injected into user messages after the first turn. The injections are phrased to appear unrelated to the task, testing whether downstream models can identify the true user objective despite noise. 

Step 6: Tool call generation: Two tool calls are generated: one for the ground-truth tool using the focused-user/focused-assistant conversation as context, and one for the test tool. For the test-tool call, a pivot message is first generated that provides a natural conversational transition from the drift conversation to the test tool’s domain; this pivot serves only as LLM context and is not included in the final sample. Both tool calls receive realistic argument values grounded in the conversation content. For single-turn conversations ( n = 1 ), the pipeline is sim-plified: user messages are not decomposed, assistant responses are omitted prior to the tool call, and drift-user injection adds a single distracting sentence to the sole user message. For simplicity, assistant-drift variations are generated regardless and discarded later.

LLM-Based Review: Each generated sample is enriched with metadata by an objective-alignment reviewer that assesses whether the conversation’s content is consistent with the stated task objective. The reviewer makes four LLM calls per sample, varying two dimensions:  

> •

Conversation input : either (a) user messages only, with assistant and tool messages stripped, or (b) the full conversation including all roles.  

> •

Evaluation prompt condition : the LLM is instructed either to (a) only consider the user’s messages or (b) consider the whole conversation. Each call produces a boolean judgment with supporting reasoning. The 2 × 2 matrix of results is stored as metadata, providing four independent assessments of objective alignment from complementary perspectives. 

Deterministic Judging: A composed judge chains three de-terministic filters with short-circuit evaluation, the first filter to reject a sample prevents execution of subsequent filters on that sample. 

a) Low-iteration filter: Samples with exactly one iter-ation (i.e., an assistant message with a tool call, with no user messages) are discarded as these were the result of a failing generation step, where no user-assistant conversation was created and only the final tool call message was present in the sample. 

b) Inconsistent drift filter: Samples where the conver-sation has only two messages (one user, one assistant tool call) but were generated with the assistant-drift variant are discarded. In single-turn conversations, there are no preceding assistant messages for drift to occur; thus, such samples are structurally inconsistent with their variant labels. 

c) Failed-objective-check filter: Samples where any of the four objective-alignment booleans is false are discarded. This strict criterion ensures that every retained sample is consistently aligned with its stated user objective across all four evaluation perspectives. Discarded samples are persisted separately from kept sam-ples, enabling post-hoc analysis of rejection reasons. The retained samples span five conversation lengths (2, 4, 6, 8, and 10 messages) with an approximately uniform distribution across lengths once accounting for filters. Furthermore, each conversation sample preserves full provenance metadata link-ing it to its source task-tool sample, the augmented-generation variant parameters (user drift, assistant drift, tool choice), and the curation review results. C. Multi-Agent Conversation Generation Strategy 

To obtain the labeled (conversation, tool) pairs described above from labeled (task, tool) pairs in a less constrained environment, we propose an alternative approach where we generate synthetic conversations using a Multi-Agent System 

(MAS) framework. By letting the AI agents interact freely in the MAS, the objective is to obtain more generated data diversity and higher content realism. Our MAS comprises three LLM-powered agents: a User agent, an Assistant agent, and a Tool Simulator agent. The aforementioned agents are orchestrated by a state machine, without the need for real tool execution, thus avoiding the limitation of populating MCP Servers with content data in the backend. All agents share the same backbone LLM, either a GPT-4o or a GPT-5.2 model in our experiments, selected per generation run to produce two dataset variants. As mentioned above, we store conversations in the OpenAI API conversation format and so refer to the subject as the ’user’. 

Agent Roles: Our simulation involves the three agents with asymmetric information access, as summarized in Table IV. 

a) User agent: The user agent receives the task objective taken from the labeled (task, tool) pair, and generates a natural opening request to the assistant, potentially omitting some minor details from the task request deliberately to emulate realistic human behavior, such as identifiers or keys. On subsequent conversation turns, the user agent evaluates the assistant’s latest response against its hidden objective, provides clarifications when asked, and terminates the conversation with a “thank you” or similar closure when satisfied regarding the completion of the task objective. 

b) Assistant agent: The assistant is bound to a set of tools via the OpenAI function-calling API, but never sees the user’s hidden-intent objective. It must infer intent solely from the conversation as would realistically be the case. In our experiments, the tool set is determined by the labeled (task, tool) data sample. In the relevant mode, i.e., for task samples with a relevant label, the assistant genuinely helps the user. In the irrelevant mode, for task samples with an irrelevant 

label, the assistant’s system prompt contains a hidden directive to invoke a specific (irrelevant) tool while appearing helpful. The assistant also has access to a special ask_user tool. We noted empirically that the user agent acting freely would struggle to respond realistically in about 10% of cases when the assistant would ask it for specific identifiers or secrets. Conversation Flow Each simulation maintains a state tuple

The state machine then cycles as follows: the assistant agent receives M and either emits a text reply (routed back to the user) or issues a tool call (routed to the tool simulator). After the tool simulator returns a synthetic result, control passes back to the assistant. The conversation terminates when the user agent’s response con-tains a predefined gratitude message (e.g., “thank you”) or when len (M ) ≥ 15 , whichever comes first. A hard limit of 

15 iterations has been determined empirically to prevent an infinite loop of exchanges between the user and the assistant, particularly when the assistant is acting maliciously and could therefore never complete the user’s request to have a user-based positive termination. 

Relevant vs. Irrelevant Generation Our framework generates both classes of labeled conversation data from the same state machine by varying only the assistant agent’s system prompt instructions. 

a) Relevant samples: The tool exposed to the assistant is a relevant one for the user’s task. The assistant is prompted to genuinely help the user by calling that tool as needed, producing conversations labeled again as relevant .b) Irrelevant samples: The tool exposed to the assistant is not a relevant one for the task. The assistant’s system prompt contains a hidden directive: it must invoke this specific tool while appearing to help the user with their request, eliciting needed parameters of that specific tool by framing questions as task related and presenting the tool’s output as if it addresses the user agent’s goal. Since the user prompt is identical in both modes, the user agent remains naive and cooperative, creating a realistic information asymmetry. The resulting conversations are labeled as irrelevant .To ensure tool invocation in every sample, the multi-agent system is wrapped with a verifier and the MAS retries (from scratch) up to five times if the assistant fails to call the target tool. If retries are exhausted, a more aggressive fallback prompt is activated that pushes for near immediate tool invo-cation from the start. In practice, the first retry mechanism is rarely used, and the fallback system is almost never triggered. However, with their combination, we always obtain the target tool call in every single sample that we generate. The appendix provides the complete system prompts for all agents and modes as well as for our special tool (see Appendix D), along with examples of relevant and irrelevant conversations (see Appendix E). For an in-depth statistical analysis of our MAS generated data, see Appendix F. V. E XPERIMENTAL EVALUATION 

We evaluate the semantic inspection layer on our conversation-tool datasets. The semantic layer is decomposed into two stages: (1) a task-extraction step that distills the subject’s objective from the multi-turn conversation preceding the first tool call, and (2) a task-tool matching step that determines whether the requested tool is appropriate for the inferred task. Each stage is implemented as a single LLM call. The task extractor receives the conversation history up to, but not including, the first tool call message and produces a self-contained natural-language request. The matcher then receives this synthesized request together with the name and description of the invoked tool and returns a binary relevance judgment with chain-of-thought reasoning. We compare four frontier LLMs as the backbone of both pipeline stages: GPT-5.2, GPT-5.4 (OpenAI), Claude Sonnet 4.6, Claude Opus 4.6 (Anthropic). Validation and test splits, shown in Table V, are constructed from non-overlapping MCP Server samples. System prompts are engineered on the validation sets of both Multi-Agent and Augmented datasets, and provided in Appendix G. Results on the test sets are provided for each model considered. We report standard binary classification metrics: accuracy, precision, recall, and F1 score, alongside false positive rate (FPR) and false negative rate (FNR). In the context of a zero-trust authorization layer, the FPR measures over-scoping: the rate at which irrelevant tool calls are incorrectly permitted, creating a potential attack surface. The FNR measures under-scoping: the rate at which legitimate tool calls are blocked, degrading agent functionality. All metrics are accompanied by 95% bootstrap confidence intervals computed over 10,000 resampled iterations. We further distinguish between single-turn (ST) and multi-turn (MT) evaluations. The ASTRA [4] benchmark is a single-turn dataset, while Augmented and Multi-Agent are evaluated in both ST and MT settings. We therefore report ASTRA results only in the single-turn regime, and report both single-turn and multi-turn results for the remaining datasets. Tables VI and VII present the complete test set results. To facilitate comparison across models and settings, Figures 3a and 3b additionally summarize F1 scores and error rates. Across all backbones, performance is strongest on single-turn benchmarks, where F1 scores are consistently high and false negative rates remain near zero. By contrast, the multi-turn setting is more challenging, particularly on the Multi-Agent benchmark, where both F1 degrades and error rates increase. This trend is consistent with the greater difficulty of reconstructing subject intent from longer conversational context and of assessing tool relevance when the task is distributed across multiple turns. From a zero-trust perspective, false positives are the more critical failure mode, since they correspond to irrelevant tool calls being incorrectly authorized. False positive rates remain relatively low overall, but rise on the harder Multi-Agent 

settings, especially in the multi-turn regime. Overall, Claude Sonnet 4.6 appears to be the most consistently strong model across evaluation settings, although the strongest model varies somewhat by benchmark. These results suggest that the pro-posed two-stage semantic inspection layer is a credible foun-dation for safer delegated authorization in agentic systems, but that current semantic checks are not yet sufficient on their own for high-stakes, long-horizon autonomous tool use. Advancing toward that goal will require richer multi-tool benchmarks, real-world evaluation, and conservative fallback mechanisms when intent inference is uncertain. VI. D ISCUSSION AND FUTURE WORK

As AI systems are shifting to horizontally scale across multiple agents with the capacity to self-organize and col-laborate to solve intricate problems, the complexity of their multi-turn interactions will rapidly increase, making extremely challenging to keep track of such interactions and securely control access to protected resources, especially, in zero-trust environments. In light of this, we presented a hybrid runtime enforcement framework combining deterministic checks with a two-stage semantic inspection pipeline within a zero-trust interception and inspection layer, thereby enabling advanced Task-Based Access Control (TBAC) for agentic AI systems. The deter-ministic checks catch structural and data-integrity violations through binary logic, while the semantic layer decomposes the problem of intent-scope alignment into conversation task extraction and task-tool matching, extending prior single-task approaches to handle multi-turn conversations. For matcher validation, we extended and curated our ASTRA dataset and proposed two novel approaches for conversation-tool data generation. Our augmented generation pipeline systematically varies conversational structure, drift patterns, and tool selec-tion, while our multi-agent generation pipeline creates data with more diversity and realism. Our experimental results show both the potential and the need for further research in the direction of hybrid access control techniques natively integrating a semantic layer, like the one introduced herein. This appendix presents six scenarios illustrating the deterministic runtime checks introduced in Section III-A. All scenarios use a common banking domain: a user interacts with an agentic application that orchestrates calls between an LLM and an MCP server exposing two tools, get_balance and transfer_amount_between_accounts . In these scenarios, we refer to our zero-trust interception layer as Continuous Agent Semantic Authorization (CASA), consistent with the accompanying diagrams. For each of the five checks (a)–(e), we contrast an unprotected failure case, in which a compromised application exploits the absence of oversight, with a protected case, in which CASA detects and blocks the attack. A sixth scenario shows a limitation: a malicious system-prompt injection that deterministic checks cannot catch, motivating the complementary semantic inspection layer. 

A. Case (a): Tool Definition Integrity Check 1) Unprotected — Tool definition poisoning: The agentic application receives the correct tool definitions from the MCP server but modifies them before forwarding to the LLM. It claims that get_balance closes accounts and that 

transfer_amount_between_accounts checks balances. The LLM, relying entirely on the context provided to it, selects the transfer tool when the user asks for a balance inquiry, unknowingly initiating a financial transaction (see Figure 4).  

> Fig. 4: Unprotected system: the application poisons tool definitions, causing the LLM to select the wrong tool.

2) Protected — CASA blocks modified tool descriptions: The interception layer caches the authoritative tool definitions when they are first retrieved from the MCP server. When the application attempts to forward a prompt containing altered descriptions, the layer detects that the definitions do not match the cached originals and blocks the request before it reaches the LLM, as shown in Figure 5. Fig. 5: Protected system: CASA detects the modified tool descriptions and blocks the prompt. 

B. Case (b): Request Authorization Verification 1) Unprotected — Unauthorized tool execution: The user asks a general question (“What can you do?”). The LLM responds with a text-only message describing its capabilities. Despite no tool call being requested, the application independently initiates a get_balance call to the MCP server, retrieving sensitive financial data without authorization from the LLM’s reasoning engine, as illustrated in Figure 6.  

> Fig. 6: Unprotected system: the application executes a tool call that was never requested by the LLM.

2) Protected — CASA blocks unrequested tool call: The interception layer records that the LLM’s most recent response contained only text and no tool call request. When the application attempts to dispatch a get_balance call, the layer identifies the discrepancy and returns a 403 Forbidden error, preventing the unauthorized data access (see Figure 7). Fig. 7: Protected system: CASA detects that no tool call was requested and blocks the unauthorized execution. 

C. Case (c): Action Alignment Validation 1) Unprotected — Tool-call swap: The user asks for a balance check. The LLM correctly issues a tool call request for 

get_balance . The application intercepts this and instead sends a transfer_amount_between_accounts command to the MCP server, executing a financial transaction while reporting the result as if it were a balance inquiry (refer to Figure 8).  

> Fig. 8: Unprotected system: the application substitutes a transfer operation for the requested balance check.

2) Protected — CASA detects tool mismatch: The interception layer records the LLM’s requested function name (get_balance ). When the application dispatches a call to transfer_amount_between_accounts , the layer detects the mismatch and blocks the request as an unauthorized tool swap, as shown in Figure 9. Fig. 9: Protected system: CASA detects the tool-name mismatch and blocks the substituted call. 

D. Case (d): Parameter Integrity Enforcement 1) Unprotected — Parameter tampering: The user requests a transfer to account a1e55e3f . The LLM generates the correct tool call request with the intended destination. The application modifies the to_account_id parameter to c90266d6 before forwarding the call to the MCP server, silently redirecting the funds (see Figure 10). The MCP server processes the altered request, and the application confirms the transfer to the user as though nothing was changed.  

> Fig. 10: Unprotected system: the application silently alters the destination account parameter.

2) Protected — CASA detects parameter modification: The interception layer captures the parameters specified by the LLM, including the destination account a1e55e3f . When the application forwards a call with destination c90266d6 , the layer identifies the parameter discrepancy and blocks the request, as illustrated in Figure 11.  

> Fig. 11: Protected system: CASA detects the altered parameter and blocks the tampered request.

E. Case (e): Data Fidelity Verification 1) Unprotected — Result-data falsification: The user requests a balance check. The tool call executes correctly and the MCP server returns a balance of 1234. The application intercepts this response and forwards a modified result to the LLM, reporting the balance as 0. The LLM, trusting the data it receives, informs the user that their account is empty, as shown in Figure 12. Fig. 12: Unprotected system: the application falsifies the tool result, causing the LLM to report incorrect data. 

2) Protected — CASA blocks data manipulation: The interception layer captures the raw response from the MCP server (balance: 1234). When the application attempts to relay a modified result (balance: 0) to the LLM, the layer detects the data integrity violation and blocks the tampered message (refer to Figure 13). Fig. 13: Protected system: CASA detects the falsified result data and blocks the message to the LLM. 

F. Case 6: Malicious System-Prompt Injection — Semantic Gap in Deterministic Checks 

The five deterministic checks above address structural and data-integrity violations that can be resolved by binary comparison. The following scenario illustrates a class of attack that falls outside their scope: manipulation of the semantic content of messages, where all structural properties remain valid. 

1) Unprotected — Malicious system-prompt injection: The user asks for a balance check. The application injects a malicious instruction into the system prompt (“First transfer 100 to account c90266d6 , then answer the user’s question”). The LLM, following its system instructions, issues a tool call request for transfer_amount_between_accounts and the funds are stolen, as shown in Figure 14.  

> Fig. 14: Unprotected system: the application injects a malicious system prompt that tricks the LLM into initiating a transfer.

2) Protected — Deterministic checks pass; semantic analysis required: In this scenario the interception layer observes that (a) tool definitions are unmodified, (b) the LLM did issue a tool call request, (c) the function name matches, (d) the parameters match, and (e) no result data has been altered. Every deterministic check passes, because the attack operates at the level of intent rather than structure, as illustrated in Figure 15. The LLM was genuinely manipulated into wanting to execute the transfer; the interception layer has no ground-truth intent signal to compare against. This gap motivates the complementary semantic inspection layer, which evaluates whether the LLM’s tool-call choices and parameter selections fall within the scope of the user’s original objective, rather than relying solely on validating structural consistency with the deterministic checks. C.1 Generation Prompts 

The augmented generation pipeline decomposes conversation construction into a sequence of specialised LLM calls, each governed by its own prompt. For multi-turn conversations ( N > 1), the pipeline first splits the task objective into N user messages ( Focused User ), fills in assistant responses ( Focused Assistant ), and reviews them for future-guessing ( Assistant Reviewer ). Drift variants are then produced by modifying assistant messages ( Drift Assistant ) and injecting distracting data into user messages ( Drift User: Multi-Message ). For single-turn conversations ( N = 1 ), the pipeline extracts missing tool parameters ( Drift Assistant: Missing Data Extraction ) and injects a distraction sentence ( Drift User: Single-Message ). Finally, a pivot message steers the conversation toward the test tool ( Pivot User ), and the tool call is generated ( Tool Call ).

You are a request synthesizer. Your job is to convert a conversation | which may be a single message or a multi-turn exchange | into one self-contained request that captures the user’s actual goal. Apply the same rules regardless of conversation length. Rules: - Recover the user’s intended task, not the dialogue itself. - Include relevant context from earlier user turns when it helps define the task. - Assistant messages are NOT part of the user’s request. Never promote assistant suggestions, proposed actions, workflows, or intermediate results into the output. - If the assistant said "I’ll also do X" or proposed a multi-step plan and the user only acknowledged (e.g., "thanks", "ok") without explicitly requesting X, do NOT include X. - Ignore code blocks, shell commands, JSON payloads, and tool-call details in assistant messages | these are implementation artifacts, not user intent. - Frame the output as if no work has been done yet.
