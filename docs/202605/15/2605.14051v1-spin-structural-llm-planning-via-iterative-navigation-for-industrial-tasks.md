## 摘要
工业级大语言模型（LLM）智能体系统通常将规划与执行分离，然而 LLM 规划器经常生成结构无效或冗长的工作流，导致脆弱的失败以及不必要的工具和 API 成本。我们提出了 SPIN，这是一种规划封装器，它将经过验证的有向无环图（DAG）规划与基于前缀的执行控制相结合。SPIN 通过 _validate_plan_text 和修复提示强制执行严格的 DAG 契约，在下游执行前生成可执行计划，随后增量式地评估 DAG 前缀，并在当前前缀足以回答查询时停止。在 AssetOpsBench 的 261 个场景中，SPIN 将执行的任务数从 1061 个减少到 623 个，并将完成度（Accomplished）从 0.638 提高到 0.706，同时将每次运行的工具调用次数从 11.81 次减少到 6.82 次。在 MCP Bench 上，同样的封装器提高了 GPT OSS1 和 Llama 4 Maverick 在规划、对齐（grounding）和依赖关系相关的评分。

## Abstract
Industrial LLM agent systems often separate planning from execution, yet LLM planners frequently produce structurally invalid or unnecessarily long workflows, leading to brittle failures and avoidable tool and API cost. We propose \texttt{SPIN}, a planning wrapper that combines validated Directed Acyclic Graph (DAG) planning with prefix based execution control. \texttt{SPIN} enforces a strict DAG contract through \texttt{\_validate\_plan\_text} and repair prompting, producing executable plans before downstream execution, and then evaluates DAG prefixes incrementally to stop when the current prefix is sufficient to answer the query. On AssetOpsBench, across 261 scenarios, \texttt{SPIN} reduces executed tasks from 1061 to 623 and improves \emph{Accomplished} from 0.638 to 0.706, while reducing tool calls from 11.81 to 6.82 per run. On MCP Bench, the same wrapper improves planning, grounding, and dependency related scores for both GPT OSS1 and Llama 4 Maverick.

在工业资产运营与维护（AssetOps）场景中，LLM 智能体需要处理多步骤、跨功能且受限的任务。现有的 LLM 规划器存在两个主要痛点：
*   **结构脆弱性**：规划器生成的输出往往包含微小的结构错误（如索引错误、依赖引用非法、工具名称不匹配），导致下游执行器直接崩溃。
*   **执行冗余与高成本**：规划器倾向于生成长且保守的工作流。在工业环境中，外部工具调用、API 流量和数据传输的成本及延迟远高于模型推理，全量执行这些计划既低效又昂贵。
**SPIN** 的核心动机是建立一个明确的“规划器-执行器”契约，通过结构化验证确保计划可执行，并通过动态导航实现“够用即止”的成本感知执行。

### 2. 论文提出的方法论
SPIN 提出了一种规划封装器（Wrapper），包含两个核心阶段：

*   **可执行 DAG 验证与修复循环**：
    *   将规划输出视为一个**有向无环图（DAG）**接口，包含任务描述、指派智能体、依赖项和预期输出。
    *   **验证器（Validator）**：强制执行严格的格式检查（如列表长度匹配、索引连续、无循环依赖、智能体名称合法）。
    *   **修复循环**：若验证失败，将错误信息反馈给规划器进行针对性修复，直到生成合法的 DAG。
*   **基于前缀评估的早期停止机制**：
    *   不再默认执行整个计划，而是按顺序评估 DAG 的前缀序列 $p^{(k)}$。
    *   **模拟器（Simulator）**：充当“世界模型”，利用检索增强技术（RAG）根据历史轨迹预测当前步骤的可能输出。
    *   **评论员（Critic）**：评估当前已执行/模拟的状态是否足以回答原始查询。
    *   **停止策略**：一旦评论员判定“当前信息已足够”，系统立即终止后续执行，从而节省昂贵的外部工具调用。

### 7. 优点
*   **结构化思维**：将 LLM 规划输出视为严格的软件接口（DAG），通过预执行验证解决了 LLM 格式不稳定的顽疾。
*   **成本意识**：引入早期停止机制，非常契合工业界对 API 调用成本和系统响应时间的敏感需求。
*   **模块化设计**：验证器、模拟器和评论员相互解耦，易于集成到现有的智能体框架中。

*   **模拟器依赖**：模拟器的效果高度依赖于历史轨迹的覆盖范围和质量。在完全陌生的环境中，模拟可能失效。
