## 摘要
基于大语言模型（LLM）的自主智能体正迅速成为一种通用技术，近期如 OpenClaw 等系统通过广泛的工具使用、第三方技能以及与用户环境的深度集成扩展了其能力。与此同时，这些智能体系统通过将不受限的能力与对敏感用户数据的访问相结合，引入了重大的安全风险。在这项工作中，我们从操作系统的视角研究了基于 LLM 的智能体的安全性。我们认为，两者在资源隔离、权限分离和通信调解方面面临着极其相似的挑战。在这一视角的指导下，我们调研了开源智能体的现状，推导出了统一的智能体架构，并系统地分析了潜在的攻击向量。为了验证这一分析，我们进行了一项案例研究，评估了四个广泛使用的类 OpenClaw 智能体。我们发现，即使在攻击者能力有限的情况下，一些保护机制在实践中也会失效，且安全运行需要详尽的系统知识和谨慎的配置。然而，我们也观察到，虽然某些智能体能力在设计上仍不安全，但许多漏洞可以通过使用成熟的操作系统安全技术来缓解。最后，我们针对智能体系统的安全设计提出了一系列建议。

## Abstract
Autonomous agents based on large language models (LLMs) are rapidly emerging as a general-purpose technology, with recent systems such as OpenClaw extending their capabilities through broad tool use, third-party skills, and deeper integration into user environments. At the same time, these agentic systems introduce substantial security risks by combining unconstrained capabilities with access to sensitive user data. In this work, we investigate the security of LLM-based agents through the lens of operating systems. We argue that both face strikingly similar challenges in isolating resources, separating privileges, and mediating communication.   Guided by this perspective, we survey the current landscape of open-source agents, derive a unified agent architecture, and systematically analyze potential attack vectors. To validate this analysis, we conduct a case study evaluating four widely used OpenClaw-like agents. Even under modest attacker capabilities, we find that several protection mechanisms fail in practice and that secure operation requires detailed system knowledge and careful configuration. However, we also observe that while some agentic capabilities remain insecure by design, many vulnerabilities can be mitigated using well-established techniques from operating system security. We conclude with a set of recommendations for the secure design of agentic systems.

*   **核心问题**：随着基于大语言模型（LLM）的自主智能体（如 OpenClaw）深度集成到用户环境并获得工具调用、第三方技能安装等能力，其安全风险呈爆炸式增长。现有的研究多集中在“提示词注入”等模型层面，缺乏系统层面的安全框架。
*   **研究动机**：智能体在处理敏感数据和执行复杂任务时，其行为逻辑与操作系统（OS）高度相似。论文旨在通过**操作系统安全**的视角，重新审视智能体的架构，识别攻击向量，并探讨如何将成熟的 OS 防御机制迁移到智能体设计中。

### 2. 论文提出的方法论
*   **核心思想：OS 类比法**。将智能体组件映射到 OS 概念：
    *   **LLM $\leftrightarrow$ 用户**（不可信的执行主体）。
    *   **智能体运行时 $\leftrightarrow$ 内核**（系统调解者）。
    *   **工具 $\leftrightarrow$ 系统调用**（功能接口）。
    *   **技能 $\leftrightarrow$ 程序**（可执行单元）。
    *   **上下文 $\leftrightarrow$ 内存**（临时存储）。
*   **统一架构模型**：推导出一个包含运行时核心（Runtime Core）、智能体核心（Agent Core）、网关（Gateway）以及持久化/瞬时状态的通用架构，并定义了网络、内核和文件系统三个安全边界。
*   **防御机制迁移**：系统性地梳理了 9 种 OS 安全机制（如进程隔离、沙箱、最小特权原则、TCB 最小化、eBPF 监控等）在智能体中的对应实现方式。

### 3. 实验设计
*   **实验对象**：选择了四种具有代表性的开源智能体：
    1.  **OpenClaw**（通用/功能丰富型）。
    2.  **IronClaw**（安全聚焦型，Rust 编写）。
    3.  **Nanobot**（极简主义型）。
    4.  **NemoClaw**（包装器型，基于容器隔离）。
*   **攻击场景**：设计了 16 个攻击向量，涵盖 5 大类：硬件接口攻击、进程隔离破坏（跨用户数据窃取）、沙箱逃逸、网络过滤绕过、系统日志篡改。
*   **评估基准（Oracle）**：使用 **eBPF（扩展伯克利数据包过滤器）** 在内核层监控智能体行为，确保能够捕捉到文件读写、网络请求等底层违规操作，而不依赖智能体自身的日志。

### 4. 资源与算力
*   **硬件环境**：使用 Debian 13.4 虚拟机，配置为 8 核 AMD EPYC 7713 CPU，16 GB RAM。
*   **模型支持**：主要使用 **Qwen3.5-122b-a10b** 作为后端 LLM。

### 5. 实验数量与充分性
*   **实验规模**：对 4 个智能体在 16 个攻击场景下进行了测试，每个测试执行 3 次以应对 LLM 的随机性。
*   **充分性**：实验覆盖了从代码注入到隐私泄露的多种维度。通过引入不同类别的智能体（从极简到复杂）和不同能力的 LLM（Qwen, Gemini, GPT），验证了漏洞的普遍性。
*   **客观性**：采用虚拟机快照回滚机制确保每次实验的独立性，使用 eBPF 这种非侵入式、底层的监控手段，保证了实验结果的客观公正。

### 6. 论文的主要结论与发现
*   **现有防御脆弱**：即使是宣称安全的智能体（如 IronClaw），在面对精心设计的攻击时仍存在漏洞。OpenClaw（最流行的版本）在所有测试场景中均告破。
*   **隔离缺失**：所有受试智能体都违反了“进程隔离”原则，将受信和不受信数据混入同一个 LLM 上下文，导致跨用户攻击极易实现。
*   **权限分离失效**：文件访问控制往往与输入处理处于同一特权等级，缺乏内核级的强制中介。
*   **OS 机制有效性**：实验证明，通过沙箱化工具执行和严格的网络出口过滤，可以显著降低风险，这验证了借鉴 OS 安全经验的必要性。

### 7. 优点（亮点）
*   **视角独特**：首次系统性地将 AI 智能体安全问题转化为成熟的操作系统安全问题，避免了“头痛医头”的防御思路。
*   **技术硬核**：引入 eBPF 作为监控工具，使安全评估从“观察模型输出”深入到“监控系统调用”层面。
*   **实战导向**：针对当前最热门的开源项目（OpenClaw 等）进行实测，具有极高的现实参考价值。

### 8. 不足与局限
*   **配置偏差**：实验主要基于智能体的默认配置。虽然论文辩称用户通常不具备专家级配置能力，但严格配置下的安全性表现未被充分讨论。
*   **交互渠道单一**：主要测试了 Matrix 和 Telegram 渠道，不同通信协议（如 Slack, Discord）可能引入特定的网关漏洞，未完全覆盖。
*   **LLM 随机性**：尽管做了多次实验，但 LLM 的不可预测性仍可能导致某些边缘攻击案例的漏报或误报。
*   **应用限制**：提出的防御建议（如上下文隔离）可能会增加推理成本（Token 消耗）或降低智能体的任务处理效率，论文未对性能损耗进行量化分析。
