## 摘要
我们定义了 Oracle 投毒，这是一种攻击类别，攻击者通过破坏 AI 智能体在运行时通过工具调用协议查询的结构化知识图谱，导致智能体在推理过程正确的情况下得出错误的结论。与提示注入不同，Oracle 投毒操纵的是智能体进行推理的数据，而非其指令。我们针对一个包含 4200 万个节点的生产级代码知识图谱演示了六种攻击场景，提供了针对生产规模智能体系统的知识图谱投毒的首次实证演示，这与 CTI 嵌入投毒有所不同。主要评估使用了来自三个供应商的九个模型的真实 SDK 工具调用（每个模型 N=30），模型自主调用图查询工具并根据结果进行推理。结果是明确的：在攻击者复杂度适中（L2）的情况下，所有测试模型对投毒数据的信任度均为 100%，在定向查询下，270 次有效试验中有 269 次接受了伪造的安全声明。在开放式提示下，信任度下降到 3-55%，证实了提示框架是一个混杂因素；我们报告了这两种情况。攻击者复杂度梯度揭示了离散的断点，即信任度从 0% 翻转到 100% 的最低技能水平，从而将攻击重新定义为一个“程度”问题，而非“是否可能”的问题。受控交付模式对比显示，内联评估会产生假阴性：GPT-5.1 在内联评估中显示 0% 的信任度，但在模拟和真实的智能体工具使用下均显示 100% 的信任度，这表明交付模式是一个一阶混杂因素。我们评估了五种防御措施；只读访问控制消除了直接变异向量，而其余四种则是部分有效的且依赖于模型。对另外四个平台的分析表明，该攻击可能在整个知识图谱生态系统中具有通用性。

## Abstract
We define Oracle Poisoning, an attack class in which an adversary corrupts a structured knowledge graph that AI agents query at runtime via tool-use protocols, causing incorrect conclusions through correct reasoning. Unlike prompt injection, Oracle Poisoning manipulates the data agents reason over, not their instructions. We demonstrate six attack scenarios against a production 42-million-node code knowledge graph, providing the first empirical demonstration of knowledge graph poisoning against a production-scale agentic system, distinct from CTI embedding poisoning. Primary evaluation uses real SDK tool-use across nine models from three providers (N=30 per model), where models autonomously invoke a graph query tool and reason from results. The result is unambiguous: every tested model trusts poisoned data at 100% at moderate attacker sophistication(L2), with 269 valid trials (of 270) accepting fabricated security claims under directed queries. Under open-ended prompts, trust drops to 3-55%, confirming prompt framing as a confound; we report both conditions. An attacker sophistication gradient reveals discrete break points, a minimum skill at which trust flips from 0% to 100%, reframing the attack as a question not of whether but of how much. A controlled delivery-mode comparison shows that inline evaluation produces false negatives: GPT-5.1 shows 0% trust inline but 100% under both simulated and real agentic tool-use, demonstrating that delivery mode is a first-order confound. We evaluate five defences; read-only access control eliminates the direct mutation vector, while the remaining four are partial and model-dependent. Analysis of four additional platforms suggests the attack may generalise across the knowledge-graph ecosystem.

### 1. 核心问题与研究动机
*   **核心问题**：论文定义了一种名为 **“Oracle Poisoning”（先知投毒）** 的新型攻击类别。其核心在于：攻击者不直接操纵 AI 的指令（如提示注入），而是篡改 AI 在运行时查询的结构化**知识图谱（Knowledge Graph, KG）**。
*   **研究动机**：当前的 AI 智能体（如用于软件工程的智能体）通常通过模型上下文协议（MCP）访问大规模代码知识图谱。智能体倾向于将这些查询结果视为“事实真相”（Oracle），并基于此进行逻辑推理。如果数据源被污染，即使智能体推理过程完美无瑕，也会得出完全错误的结论。

### 2. 方法论
*   **核心思想**：利用 AI 智能体对工具调用（Tool-use）返回结果的盲目信任，通过直接修改图数据库中的节点、边或属性，诱导智能体产生错误的决策。
*   **攻击机制**：
    *   **节点创建**：插入虚假的函数、包或依赖关系。
    *   **属性修改**：修改现有节点的元数据（例如，将一个不安全的函数标记为“已通过安全审计”）。
*   **攻击流程**：
    1.  **侦察**：查询图谱以识别目标节点和命名规范。
    2.  **伪造**：构建符合图谱模式（Schema）的虚假数据。
    3.  **注入**：利用写权限（通常由于缺乏细粒度权限控制）执行 Cypher 语句。
    4.  **触发**：等待智能体查询并基于投毒数据进行推理。
*   **攻击复杂度梯度（L0-L3）**：从简单的命名错误（L0）到具备完美来源证明、安全审计引用和专业术语的专家级伪造（L3）。

### 3. 实验设计
*   **实验场景**：在一个包含 **4200 万个节点** 的生产级代码知识图谱上进行。
*   **攻击场景**：设计了 6 种场景，包括：
    1.  **包分叉投毒**：诱导使用虚假的恶意库版本。
    2.  **调用链安全规避**：伪造不存在的“输入净化函数”，使 AI 误认为 SQL 注入风险已修复。
    3.  **遥测重定向**：使 AI 报告安全事件已监控，实际上数据流向了攻击者。
*   **测试模型（Benchmark）**：涵盖 3 家供应商的 9 种主流模型：
    *   OpenAI: GPT-4o, GPT-4.1, GPT-5 系列。
    *   Anthropic: Claude Sonnet 4.6, Claude Opus 4.5。
    *   Google: Gemini 2.5 Pro, Gemini 2.0 Flash。
*   **对比维度**：
    *   **交付模式对比**：内联文本（Inline）vs 真实 SDK 工具调用。
    *   **提示框架对比**：定向查询（Yes/No）vs 开放式分析。

### 4. 资源与算力
*   **算力说明**：论文未明确提及具体的 GPU 训练算力，因为该研究侧重于**推理阶段**的攻击评估。
*   **环境资源**：使用了 Neo4j 5.x 图数据库、MCP v1.0 协议以及商业级智能体 SDK。实验在真实的生产规模数据环境下运行，而非单纯的实验室模拟。

### 5. 实验数量与充分性
*   **实验规模**：主要评估中每个模型进行 30 次试验（N=30），总计 270 次有效试验。此外还进行了 160 次因子分解实验和多组消融实验。
*   **充分性与客观性**：
    *   **多模型验证**：跨越了不同厂商和架构的模型。
    *   **双盲标注**：由两名独立评分员对模型响应进行分类（信任、怀疑、混合），一致性高达 94%（Cohen’s κ = 0.89）。
    *   **真实性**：使用了真实的 SDK 和生产级图谱，而非仅在 Prompt 中模拟数据。

### 6. 主要结论与发现
*   **普遍易感性**：在 L2（中等复杂度）攻击下，**所有测试模型在真实工具调用场景中的信任率均达到 100%**。
*   **交付模式是关键变量**：发现“内联评估”会产生假阴性。例如，GPT-5.1 在处理文本形式的投毒数据时表现出怀疑，但当数据通过工具调用通道返回时，信任度立即升至 100%。
*   **防御有效性**：
    *   **只读访问控制**：最有效，从源头切断变异路径。
    *   **多工具交叉验证**：能将盲目信任率从 100% 降至 0-25%。
    *   **系统提示词加固**：几乎无效，模型即使在推理中提到要怀疑，最后还是会接受虚假数据。

### 7. 优点
*   **实战化视角**：首次在 4200 万节点规模的生产系统上证明了知识图谱投毒的可行性。
*   **定义了新攻击类**：清晰地将 Oracle Poisoning 与传统的 RAG 投毒或提示注入区分开来。
*   **VUT 防御框架**：提出了“可见性（Visibility）- 理解（Understanding）- 可追溯性（Traceability）”的系统化防御思路。

### 8. 不足与局限
*   **单一生产系统限制**：虽然分析了其他平台，但核心实验主要在一个特定的代码知识图谱上完成。
*   **提示词框架偏差**：定向查询（Is X true?）会显著提高信任率，而开放式查询下模型会更谨慎。
*   **缺乏人类基准**：未对比人类安全分析师在面对同样伪造的图数据时是否也会被欺骗。
*   **应用限制**：攻击前提是攻击者需拥有图数据库的写权限（尽管论文指出这在当前 MCP 集成中很常见）。
