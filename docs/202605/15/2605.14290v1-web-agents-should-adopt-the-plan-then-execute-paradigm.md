## 摘要
ReAct 已成为大语言模型（LLM）智能体的默认架构，许多现有的 Web 智能体也遵循这一范式。我们认为，对于 Web 智能体而言，这并非理想的默认选择。相反，Web 智能体应默认采用“先规划后执行”（plan-then-execute）模式：在观察运行时网页内容之前，先确定一个特定于任务的程序，然后再执行它。原因在于网页内容混合了多方输入。例如，电子商务产品页面可能结合了卖家的列表、客户评论和赞助广告。在 ReAct 模式下，所有这些内容在决定下一步行动时都会流入模型，为提示词注入（prompt injections）操控智能体的控制流提供了直接路径。“先规划后执行”改变了这一边界：不可信的数据可能会影响预定义执行图中的值或分支，但它无法重新定义用户任务，也无法导致模型在运行时合成新的动作。我们分析了流行的 Web 智能体基准测试 WebArena，发现所有任务都与“先规划后执行”兼容，其中 80% 的任务可以通过纯程序化规划完成，无需任何运行时 LLM 子例程。我们确定了在 Web 上采用“先规划后执行”的主要障碍：为了使其良好运行，工具必须清晰地映射到语义动作，且其效果在执行前已知，以便智能体有足够的信息进行规划。Web 并没有自然地暴露这种接口。诸如点击、输入和滚动等浏览器工具具有依赖于页面的含义。在这一层进行规划是近视的：智能体只能看到当前页面上的动作，而后续动作只有在它行动后才会出现。弥补这一差距需要类型化接口，将网站交互从点击和按键转变为任务级操作。这是一个基础设施问题，而非建模问题。Web 任务默认情况下不需要反应性；它们需要类型化、完整且可审计的网站 API。

## Abstract
ReAct has become the default architecture across LLM agents, and many existing web agents follow this paradigm. We argue that it is the wrong default for web agents. Instead, web agents should default to plan-then-execute: commit to a task-specific program before observing runtime web content, then execute it. The reason is that web content mixes inputs from many parties. An e-commerce product page may combine a seller's listing, customer reviews and sponsored advertisements. Under ReAct, all of this content flows into the model when deciding on the next action, creating a direct path for prompt injections to steer the agent's control flow. Plan-then-execute changes this boundary: untrusted data may influence values or branches inside a predefined execution graph, but it cannot redefine the user task or cause the model to synthesize new actions at runtime. We analyze WebArena, a popular web agent benchmark, and find that all tasks are compatible with plan-then-execute, while 80% can be completed with a purely programmatic plan, without any runtime LLM subroutine. We identify the main barrier to adopting plan-then-execute on the web: For it to work well, tools must map cleanly to semantic actions, with effects known before execution, so agents have enough information to plan. The web does not naturally expose that interface. Browser tools such as click, type, and scroll have page-dependent meanings. Planning at this layer is near-sighted: the agent can only see actions on the current page, and later actions appear only after it acts. Closing this gap requires typed interfaces that turn website interactions from clicks and keystrokes to task-level operations. This is an infrastructure problem, not a modeling problem. Web tasks do not need reactivity by default; they need typed, complete, auditable website APIs.

### 1. 核心问题与研究动机
*   **核心问题**：当前 Web 智能体普遍采用 **ReAct** 范式（观察网页 -> 思考 -> 采取行动 -> 重复），这种模式在处理 Web 环境时存在严重的**安全隐患**。
*   **研究动机**：
    *   **提示注入风险**：网页内容是多方输入的混合体（如广告、评论、第三方插件）。在 ReAct 模式下，这些不可信的数据直接进入 LLM 的决策流，攻击者可以通过网页内容实施“提示注入”，劫持智能体的控制流（如诱导转账、泄露隐私）。
    *   **效率低下**：ReAct 智能体通常是“近视”的，只能根据当前页面采取行动，导致重复规划、Token 消耗大且延迟高。
    *   **不可审计性**：动态生成的步骤难以在执行前进行安全审计。

### 2. 方法论：Plan-Then-Execute (PTE) 范式
*   **核心思想**：将任务的“规划”与“执行”彻底分离。智能体在接触任何运行时网页内容之前，先根据用户意图和网站 API 文档生成一个完整的**任务特定程序（Plan）**，然后由执行器运行该程序。
*   **关键技术细节**：
    *   **规划器（Planner）**：仅基于用户目标和受信任的 API 定义生成代码。它不接触不可信的运行时数据。
    *   **受信任 API（Trusted API）**：网站提供的类型化接口（如 REST API 或客户端 SDK）。它将底层的浏览器操作（点击、输入）抽象为语义化的任务操作（搜索产品、加入购物车）。
    *   **执行器（Executor）**：运行预定义的程序。
    *   **LLM 子例程（LLM Subroutines）**：在执行过程中，如果需要处理非结构化数据（如提取评论摘要），可以调用被“隔离”的 LLM。这些子例程只能输出特定类型的数据，**不能改变程序的控制流或合成新动作**。
*   **控制流与数据流分离**：不可信数据只能影响程序中的变量值（数据流），而不能修改程序的逻辑结构（控制流）。

### 6. 主要结论与发现
*   **PTE 的可行性**：WebArena 中 **100% 的任务**都兼容 PTE 范式。
*   **安全性发现**：**81.28%** 的任务属于“纯程序化”任务，完全不需要运行时 LLM 参与决策，从而彻底免疫提示注入。
*   **瓶颈所在**：Web 智能体的发展瓶颈不在于模型建模能力，而在于**基础设施**。目前大多数网站缺乏完整、类型化、可审计的 API 或 SDK。
*   **效率提升**：PTE 减少了冗余的观察-思考循环，显著降低了 Token 成本和延迟。

### 7. 优点
*   **安全性突破**：通过架构设计而非模型微调来解决提示注入问题，提供了更强的安全保证。
*   **确定性与可审计性**：生成的计划可以在执行前进行静态分析和安全策略检查。
*   **工程化思维**：将 Web 交互从“模拟人类点击”提升到“程序化调用”，更符合大规模自动化系统的需求。

### 8. 不足与局限
*   **API 依赖性**：PTE 极度依赖高质量的网站 API 或 SDK。如果网站结构复杂且无 API，构建 SDK 的成本很高。
*   **容错性较弱**：相比 ReAct 的动态调整，PTE 在遇到非预期错误（如搜索无结果）时较为脆弱，需要显式的错误处理逻辑。
*   **不适用于高度动态任务**：对于需要根据执行过程中发现的指令来决定后续步骤的任务（如“阅读邮件并执行其中的所有要求”），PTE 难以胜任。
*   **维护成本**：网站 UI 的频繁变动会导致底层 SDK 失效，需要持续维护。
