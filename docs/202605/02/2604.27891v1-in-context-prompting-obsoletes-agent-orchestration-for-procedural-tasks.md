## 摘要
智能体编排框架（如 LangGraph、CrewAI、Google ADK、OpenAI Agents SDK 等）在 LLM 之上放置了一个外部编排器，用于跟踪状态并在每一步注入路由指令。我们通过一项对照实验表明，对于程序化任务，这种架构被一种更简单的替代方案所超越：将整个流程放入系统提示词中，并让模型进行自我编排。在旅游预订（14个节点）、Zoom 技术支持（14个节点）和保险理赔处理（55个节点）这三个领域中，我们针对每种情况评估了 200 场对话，并使用 LLM 作为评委（LLM-as-judge）根据五个质量标准进行评分。在 5 分制评分中，上下文方法得分为 4.53-5.00，而使用相同模型的 LangGraph 编排器得分为 4.17-4.84。编排系统在旅游、Zoom 和保险对话中的失败率分别为 24%、9% 和 17%，而上下文基准方案的失败率仅为 11.5%、0.5% 和 5%。虽然外部编排对于早期模型可能是必要的，但前沿模型能力的提升使得它在遵循特定流程的多轮对话中已不再必要。

## Abstract
Agent orchestration frameworks -- LangGraph, CrewAI, Google ADK, OpenAI Agents SDK, and others -- place an external orchestrator above the LLM, tracking state and injecting routing instructions at every turn. We present a controlled comparison showing that for procedural tasks, this architecture is dominated by a simpler alternative: putting the entire procedure in the system prompt and letting the model self-orchestrate. Across three domains -- travel booking (14 nodes), Zoom technical support (14 nodes), and insurance claims processing (55 nodes) -- we evaluate 200 conversations per condition using LLM-as-judge scoring on five quality criteria. The in-context approach scores 4.53--5.00 on a 5-point scale while a LangGraph orchestrator using the same model scores 4.17--4.84. The orchestrated system fails on 24% of travel, 9% of Zoom, and 17% of insurance conversations, compared to 11.5%, 0.5%, and 5% for the in-context baseline. While external orchestration may have been necessary for earlier models, advances in frontier model capabilities have made it unnecessary for multi-turn conversations following a defined procedure.

### 1. 核心问题与整体含义
*   **研究动机**：目前开发者投入了大量精力构建复杂的智能体编排框架（如 LangGraph, CrewAI, OpenAI Agents SDK 等）。这些框架通过外部控制器跟踪状态并在每一步注入路由指令。
*   **核心问题**：随着前沿模型（如 Claude 3.5/4.5）能力的增强，这种复杂的外部编排架构是否依然必要？
*   **整体含义**：研究发现，对于“程序化任务”（即遵循特定流程的工作流），外部编排不仅增加了工程复杂度，反而因“推理碎片化”导致性能下降。简单的“上下文内提示词（In-Context Prompting）”方案在质量和可靠性上均全面超越了复杂的编排框架。

### 2. 方法论：核心思想与技术细节
*   **核心思想**：对比“外部编排”与“上下文自编排”两种架构。
    *   **外部编排（LangGraph）**：将流程图的每个节点映射为独立的提示词模板。当到达“决策中心（Decision Hubs）”时，由编排器发起额外的 LLM 调用来决定路由方向。
    *   **上下文内基准（In-Context Baseline）**：将整个流程图（包括所有节点、边缘条件、状态）序列化为结构化文本，直接放入系统提示词中。模型在对话过程中自行理解流程并进行“自编排”。
*   **流程定义**：使用有向图 $F = (N, E, n_0, T, D)$ 表示程序，其中包含节点（角色与模板）、边缘（条件）、起始/终止状态以及决策中心。
*   **技术实现**：两种方案使用完全相同的底层模型（Claude Sonnet 4.5），确保变量唯一。

*   **实验场景**：

*   **算力说明**：论文未涉及模型训练，因此未提及 GPU 训练算力。

### 6. 主要结论与发现
*   **全面胜出**：在所有领域的所有指标上，上下文提示词方案均优于 LangGraph 编排。
*   **可靠性更高**：编排系统的失败率高达 9%-24%，而上下文方案仅为 0.5%-11.5%。
*   **失败模式分析**：外部编排会导致“推理碎片化”，模型只看到当前节点的指令而丢失全局视野，容易产生路由错误、循环或重复提问。
*   **效率悖论**：编排框架为了“节省 Token”而拆分提示词，却因为额外的路由调用增加了延迟，且降低了最终质量。

### 7. 优点
*   **挑战权威**：直击当前 AI 开发中过度工程化（Over-engineering）的痛点。
*   **扩展性验证**：通过 55 个节点的保险案例证明了长上下文模型处理复杂逻辑的能力。
*   **实用性强**：为开发者提供了直接的架构建议——在尝试复杂框架前，先试着把流程写进提示词。

### 8. 不足与局限
*   **合成数据偏差**：实验完全基于模拟用户，真实人类用户的不可预测性可能带来不同结果。
*   **成本问题**：对于极高并发的商业应用，上下文方案带来的 30%-40% 额外 Token 成本可能是一个考量因素。
*   **适用范围限制**：研究明确指出该结论适用于“程序化任务”。对于需要多模型协作、复杂工具调用或非结构化探索任务，外部编排可能仍有价值。
*   **上下文窗口限制**：如果流程图极其庞大（如数千个节点），可能会超出模型的有效注意力范围或上下文窗口。
