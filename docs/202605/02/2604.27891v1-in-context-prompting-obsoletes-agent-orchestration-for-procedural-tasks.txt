## Abstract 

Agent orchestration frameworks—LangGraph, CrewAI, Google ADK, OpenAI Agents SDK, and others—place an external orchestrator above the LLM, tracking state and injecting routing instructions at every turn. We present a controlled comparison showing that for procedural tasks, this architecture is dominated by a simpler alternative: putting the entire procedure in the system prompt and letting the model self-orchestrate. Across three domains—travel booking (14 nodes), Zoom technical support (14 nodes), and insurance claims processing (55 nodes)— we evaluate 200 conversations per condition using LLM-as-judge scoring on five quality criteria. The in-context approach scores 4.53–5.00 on a 5-point scale while a LangGraph orchestrator using the same model scores 4.17–4.84. The orchestrated system fails on 24% of travel, 9% of Zoom, and 17% of insurance conversations, compared to 11.5%, 0.5%, and 5% for the in-context baseline. While external orchestration may have been necessary for earlier models, advances in frontier model capabilities have made it unnecessary for multi-turn conversations following a defined procedure. 

## 1 Introduction 

Substantial engineering effort has gone into building agent orchestration frameworks. LangGraph, CrewAI, Google ADK, OpenAI’s Agents SDK, Microsoft’s Semantic Kernel, Amazon’s Strands, and LlamaIndex all implement the same pattern: an external orchestrator sits above the LLM, tracking conversation state and injecting routing instructions at every turn. These frameworks represent significant investment—both by the companies building them and by the developers adopting them. Wang et al. [2025] found that 80% of developers struggle to select among these frameworks, with LangChain’s abstractions requiring “traversing seven layers of code” for a single change. Meanwhile, the reliability costs of orchestration are well documented: cascading errors, routing failures, and poor cross-trial consistency [Cemri et al., 2025, Zhu et al., 2025, Gupta, 2026, Kapoor et al., 2024]. This paper presents a simple empirical finding: for procedural tasks—conversations that follow a defined workflow from intake to resolution—the orchestration architecture is not just unnecessary but actively harmful. A frontier model given the entire procedure in its system prompt and left to self-orchestrate consistently outperforms the same model when orchestrated via LangGraph, across every domain and every metric we tested. The bottleneck is not better orchestration; orchestration itself is the bottleneck. Our contributions are:

## 2 Method 

The experiment compares simulated customer-service conversations produced under two conditions: (i) an agent orchestrated by LangGraph according to a given procedural workflow represented by a directed graph; (ii) an agent given the same workflow graph in context and otherwise allowed to self-orchestrate. In both conditions, the agent begins with the same initial problem scenario (see Appendices B–D for examples), and the conversation unfolds through dynamic interaction with a simulated user powered by Claude Sonnet. The user simulator receives the same scenario specification in both conditions but generates responses based on the evolving conversation history; it has no knowledge of the underlying flowchart and simply role-plays a customer with given preferences (Appendix H.2). We evaluate n = 200 scenarios per condition per domain (1,200 total conversations), with scenarios designed to cover the full range of flowchart paths, user styles (specific to vague), satisfaction levels (enthusiastic to skeptical), and domain-specific variation. The resulting conversations are then evaluated by an LLM judge across several quality metrics, as described below. 

2.1 Procedure Definition 

We represent procedures as directed graphs F = ( N, E, n 0, T, D ):• N : Nodes with role (agent/user) and prompt template • E ⊆ N × N × C: Edges with optional conditions • n0 ∈ N : Start node • T ⊆ N : Terminal nodes (success, abandonment, escalation) • D ⊆ N : Decision hubs—agent nodes with multiple outgoing edges whose conditions partition the conversation state (e.g., “information complete” vs. “missing documents”) We evaluate on three domains spanning different types and scales of complexity (flowcharts in Appendix E): 

Travel booking (14 nodes, 3 decision hubs, 3 terminal states). Decision hubs route among 4– 6 alternatives based on information completeness, user satisfaction with options, and booking confirmation status. The flowchart includes cycles (e.g., assessment → gathering → re-assessment). 

Zoom technical support (14 nodes, 3 decision hubs, 3 terminal states). A product-specific domain: the agent is a Zoom specialist with knowledge of Zoom’s UI, settings, and error codes. Triage routes based on initial diagnostics, retry logic handles troubleshooting failures, and final routing confirms resolution. Unlike travel booking, this domain requires accurate technical guidance—not just conversational flow. 

Insurance claims processing (55 nodes, 6 decision hubs, 4 terminal states). This domain tests scaling to additional complexity—nearly 4 × the node count of the other domains. The procedure spans five phases: intake and claim type assessment, information gathering with per-type branching, coverage determination with exclusion analysis, settlement negotiation with offer/counter-offer loops, and resolution with follow-up tracking. The 6 multi-way decision hubs yield hundreds of unique paths through nested loops and cross-phase dependencies. 22.2 Two Conditions 

We compare two approaches using the same frontier model (Claude Sonnet 4.5 1) on the same scenarios: 1. LangGraph orchestrated : We implement the orchestration condition using Lang-Graph [LangChain, Inc., 2024], the most widely adopted agent orchestration framework. Each flowchart node maps to a LangGraph graph node. When the graph execution reaches an agent node, that node’s prompt template is inserted into the system prompt before calling the LLM. At decision hubs, the orchestrator makes an additional LLM call to select which edge to follow based on the conversation history and available conditions. User nodes invoke the user simulator. State flows through a dictionary tracking conversation history, current node, and metrics. 2. In-context baseline : The model receives the entire serialized flowchart in its system prompt—all nodes, edges, conditions, and terminal states rendered as structured text—and self-orchestrates. No external state tracking, no routing calls. One API call per agent turn. Agent system prompts are in Appendix H; the per-node prompt templates injected by the orchestrator are in Appendix F. This design isolates the effect of orchestration architecture. Both conditions use the same model, the same procedure, and the same evaluation scenarios. The only difference is whether the model is orchestrated externally via LangGraph or manages the procedure internally. Appendices B–D show side-by-side conversation excerpts from both conditions on identical scenarios. The failure modes we observe—fragmented reasoning, routing errors at decision hubs, and constrained naturalness—are structural consequences of the orchestration architecture: we expect similar properties in any system that injects per-node instructions and routes via additional LLM calls. 

2.3 LLM-as-Judge Evaluation 

We use an approach-agnostic LLM-as-judge methodology [Zheng et al., 2023] where Claude Sonnet evaluates conversation quality without knowing which system produced it. Each conversation is scored on five criteria (1–5 scale) with specific behavioral anchors at every level (full scoring prompt in Appendix I): 

Task Success : Did the agent follow the procedure through to a terminal state? (5 = complete procedure with clear terminal state; 1 = no meaningful progress) 

Information Accuracy : Did the agent correctly use and retain all user-provided information? (5 = every detail correctly reflected; 1 = fabricated details) 

Consistency : Did the agent maintain coherent state across the conversation? (5 = no contradictions or repeated questions; 1 = contradicts itself repeatedly) 

Graceful Handling : How well did the agent handle changes, ambiguity, and edge cases? (5 = smoothly adapts; 1 = any deviation breaks the flow) 

Naturalness : Does the conversation read like talking to a skilled human agent? (5 = indistinguishable from human; 1 = mechanical, scripted) Although both conditions use Claude Sonnet, the judge (also Claude) may systematically prefer unconstrained text (in-context) over template-constrained text (orchestrated) [Panickssery et al., 2024]; we address this with an independent GPT-4.1 replication in §3.2.

3.3 Efficiency Comparison 

Table 3 reveals a structural inefficiency in the orchestrated architecture. The orchestrator makes additional LLM calls at every decision hub to determine which branch to take—calls that are entirely unnecessary in the in-context approach, where the model routes itself. In travel booking, orchestration requires 10.8 LLM calls per conversation versus 8.7 for in-context (1.2 ×). In Zoom support, the ratio is similar: 9.4 versus 6.9 (1.4 ×). In insurance claims, the gap widens to 17.3 versus 10.0 (1.7 ×)because the 55-node procedure contains more decision hubs requiring routing calls. These additional calls introduce latency at every decision hub while achieving lower quality. The in-context approach does consume more tokens per conversation because the serialized procedure is included in every API call. For insurance claims, this is 68K tokens versus 43K for the orchestrated approach. However, this token overhead is the cost of achieving higher quality—the procedure in context enables the holistic reasoning that orchestration prevents.

3.4 Failure Mode Analysis 

Table 5 presents the failure analysis. The LangGraph orchestrator fails more often across all three domains, with the largest gap in Zoom (18 ×) and the highest absolute rate in travel (24%). The Zoom domain shows the most dramatic relative difference: the LangGraph orchestrator fails on 18 conversations (9%) while the in-context baseline fails on just 1 (0.5%)—an 18 × difference. Zoom’s diagnostic branching requires correct routing through technical triage, retry logic, and resolution confirmation; each routing decision is an additional point of failure. In travel booking, the LangGraph orchestrator fails on 48 conversations (24%), the highest absolute rate. Qualitative inspection reveals that the orchestrator’s node-by-node generation causes the model to either truncate conversations prematurely (jumping to a terminal state before the user’s needs are met) or loop between assessment and information-gathering nodes (see Appendix B for an example where the orchestrated agent repeats the same question three times before presenting options). The 34 insurance failures exhibit a different pattern: routing errors at the 6 decision hubs cause the orchestrator to skip phases or loop indefinitely. In a 55-node procedure, each misrouted decision propagates downstream—a claim type misclassification at intake leads to wrong information-gathering questions, wrong coverage determination, and ultimately an unresolved claim. The in-context baseline avoids these failure patterns because the model sees the full procedure and chooses its own pacing while maintaining awareness of the global conversation state. 

## 4 Related Work 

Agent orchestration frameworks. The 2024–2025 period saw rapid proliferation of LLM agent frameworks. LangGraph [LangChain, Inc., 2024], CrewAI [Moura, 2024], Google ADK [Google, 62025], OpenAI Agents SDK [OpenAI, 2025], Semantic Kernel [Microsoft, 2025], Strands Agents [Amazon Web Services, 2025], and LlamaIndex [LlamaIndex, 2025] all implement variants of the state-graph orchestration pattern. These frameworks vary in abstraction level and feature set but share a common architecture: an external controller manages conversation state and injects instruc-tions into the LLM at each step. Wang et al. [2025] surveyed 500+ developers and found that 80% struggle to select among these frameworks, with LangChain’s abstractions requiring “traversing seven layers of code” for a single change. Despite significant community adoption, no controlled study has compared orchestrated performance against the simpler alternative of providing the procedure directly in context. 

Agent reliability and failure modes. A growing body of work documents reliability failures in orchestrated agent systems. Cemri et al. [2025] identified 14 failure modes in multi-agent sys-tems, with specification and planning failures dominating. Cascading errors have been identified as the primary bottleneck in orchestrated systems [Zhu et al., 2025]—a finding consistent with our observation that routing errors at decision hubs propagate downstream. The concept of informa-tion fragmentation —where orchestrated systems impose intrinsic limits on context integration that single-agent systems avoid—has been formalized in recent work [others, 2025], with evidence that multi-agent systems can often be compiled into equivalent single-agent systems with lower token usage and latency [others, 2026]. Agents with 60% pass@1 achieve only 25% consistency across trials [Gupta, 2026], highlighting the gap between single-attempt and repeated performance. Kapoor et al. [2024] argued that most agent evaluations fail to capture these reliability costs and proposed more rigorous evaluation standards. These papers document reliability failures but implicitly assume the solution is better orchestration. Our results suggest the opposite: for procedural tasks, removing 

orchestration improves reliability, and the failure modes they document are structural consequences of the orchestration architecture itself. 

Task-oriented dialogue systems. The task-oriented dialogue (TOD) community has long debated modular versus end-to-end architectures. MultiWOZ [Budzianowski et al., 2018] established multi-domain TOD benchmarks that drove modular pipeline research (NLU → DST → Policy → NLG). SimpleTOD [Hosseini-Asl et al., 2020] demonstrated that these modular pipelines can be collapsed into a single auto-regressive model without loss of quality. SynTOD [Samarinas et al., 2024] used state transition graphs—structurally similar to our flowcharts—for generating training data. AutoTOD [Xu et al., 2024] identified the failure modes of modular systems (error accumulation, poor generalization) and argued for zero-shot autonomous agents. Our work extends this line by showing that even with a frontier model, orchestration degrades performance relative to a simpler in-context approach. 

Workflow automation with LLMs. WorkflowLLM [Fan et al., 2024] enhanced LLMs’ ability to follow multi-step workflows through workflow-specific training data. Hsiao [2025] showed that providing explicit procedural knowledge improves agentic workflow performance, consistent with our finding that procedure-in-context outperforms procedure-as-orchestration. Grohs et al. [2023] demonstrated that LLMs can perform business process management tasks—mining process models and assessing automation suitability—without extensive prompt engineering, suggesting that procedural reasoning is a natural capability of frontier models. AgentBench [Liu et al., 2024b] evaluated LLMs as agents across diverse environments, finding that model capability is the dominant factor in agent performance—supporting our observation that a capable model needs less scaffolding. 

In-context learning and long-context models. ReAct [Yao et al., 2023] demonstrated that inter-leaving reasoning and action traces in prompts improves agent performance. Reflexion [Shinn et al., 2023] extended this by maintaining verbal working memory across episodes. Our in-context condition goes further: rather than prompting with reasoning examples, we provide the entire procedure —all nodes, edges, and conditions—letting the model both reason about and execute the workflow within a single context. This approach is enabled by the rapid expansion of context windows in frontier models, from 4K tokens in 2022 to 200K+ by 2025. Liu et al. [2024a] showed that LLMs struggle to use information positioned in the middle of long contexts, but our procedures are placed at the 

beginning of the context (in the system prompt), avoiding this failure mode. Even our most complex procedure (55 nodes, ∼4,000 tokens serialized) uses only 2% of available context. 

Fine-tuning as an alternative. A complementary approach to in-context prompting is fine-tuning the procedure into a smaller model’s weights. Hsieh et al. [2023] showed that distilling reasoning from large to small models can outperform the large model with less data. Chen et al. [2023] fine-tuned language agents on ReAct-style trajectories and found that fine-tuned small models can 7match prompted large models on specific tasks. Chen et al. [2024] curated diverse agent training data and demonstrated effective agent skill transfer. A companion paper [Dennis et al., 2025] explores this direction, showing that a compiled 8B model achieves quality competitive with a LangGraph orchestrator at 128–462 × lower inference cost. 

LLM-as-judge evaluation. We use LLM-as-judge methodology [Zheng et al., 2023], which has become standard for evaluating open-ended generation quality. Panickssery et al. [2024] showed that LLMs exhibit self-preference bias. We address this directly by replicating all evaluations with an independent judge (GPT-4.1) in §3.2, confirming that the core findings hold across judge families. 

## 5 Discussion 

5.1 Why Orchestration Underperforms 

The results are consistent across domains and metrics: the same model performs worse when orchestrated via LangGraph. Three factors explain this: 

Orchestration fragments reasoning. The orchestrator generates each turn using only the current node’s prompt template, producing locally reasonable but globally less coherent responses. The in-context baseline reasons over the full procedure holistically, maintaining awareness of where the conversation has been and where it needs to go. This explains the consistency gap (4.83–4.99 vs. 4.32–4.55): orchestrated agents lose track of the global conversation arc because they only see one node at a time. 

Orchestration introduces failure modes. The orchestrator adds three failure categories that do not exist in the in-context approach: routing failures (default branch taken when the LLM misclassifies the conversation state), decision ambiguity (multiple edge conditions match simultaneously), and template conflicts (the current node’s instructions may conflict with the conversation context). These failure modes compound across domains, producing failure rates of 9–24%. 

Orchestration constrains the model. Node-by-node template injection constrains the frontier model’s natural conversational style. The in-context baseline lets the model produce unconstrained responses—the procedure is in the prompt but the model chooses how to express it. Under the Claude judge, this produces a naturalness gap (5.00 vs. 4.58–4.84), though the GPT-4.1 replication finds this gap largely disappears (§3.2), suggesting the effect is partly judge-dependent. 

5.2 Limitations Synthetic domains. All three domains use synthetic procedures and simulated users. While the procedures are modeled on real-world workflows (enterprise travel booking, Zoom product support, insurance claims processing), we have not validated on production conversation data. Real users may behave differently from our Claude-powered simulator. 

LLM-as-judge. Our GPT-4.1 replication (§3.2) confirms that the core findings are robust to judge choice, but also reveals that the naturalness advantage is judge-dependent: the Claude judge finds a large gap while GPT-4.1 finds near-parity. Naturalness results should be interpreted cautiously. 

Procedure must fit in context. The in-context approach requires the serialized procedure to fit in the model’s context window alongside the conversation history. For our largest procedure (55 nodes, 

∼4,000 tokens serialized), this is manageable with current context windows (200K+ tokens). For substantially larger procedures, context competition may become a constraint. 

Cost. The in-context approach costs 1.3–1.4 × more per conversation than orchestration (Table 4), with the 55-node insurance procedure costing $0.22 per conversation. For high-volume production workloads, this cost may motivate alternative approaches such as fine-tuning the procedure into a smaller model’s weights. 

Single framework. We evaluate only LangGraph as the orchestration baseline. While LangGraph is the most widely adopted framework and our implementation follows its standard patterns, other frameworks (CrewAI, Google ADK) may yield different results. However, the core architecture— per-node template injection with LLM-based routing—is shared across frameworks, and our failure modes are structural consequences of this architecture rather than LangGraph-specific bugs. 85.3 When Orchestration Might Still Matter 

Our results apply to a specific class of tasks: multi-turn conversations following a defined procedure. Orchestration may still provide value for: 

Multi-model pipelines. When different steps require different models (e.g., a vision model for document processing, a code model for technical execution), an orchestrator coordinates handoffs between models. Our comparison holds the model constant; heterogeneous pipelines are out of scope. 

Tool use with external state. When the agent must interact with external systems—databases, APIs, file systems—an orchestrator manages tool calls and state persistence. Our procedures are conversational; tool-augmented workflows may benefit from orchestration. 

Non-procedural tasks. Open-ended tasks without a defined workflow—creative writing, research, exploration—may benefit from frameworks that provide structure. Our finding is specific to procedural tasks where the workflow is known in advance. 

Smaller models. Our comparison uses a frontier model (Claude Sonnet). Less capable models may benefit from the guardrails that orchestration provides, compensating for weaker instruction-following with external structure. We do not claim that orchestration is universally harmful—only that for frontier models on procedural tasks, it provides no benefit and measurable harm. 

## 6 Conclusion 

Across three domains and five quality criteria ( n = 200 per condition per domain), putting the procedure in the system prompt dominates LangGraph orchestration. Under the Claude Sonnet judge, all 15 comparisons significantly favor in-context prompting ( p < 0.005 , d = 0 .37 –1.01 ). An independent GPT-4.1 replication confirms the core finding: 11 of 15 comparisons significantly favor in-context prompting, with zero favoring orchestration. Task success, information accuracy, and consistency are robust across both judges; the naturalness advantage is judge-dependent and should be interpreted cautiously (§3.2). The practical implication is direct: for procedural tasks where the procedure fits in context, try putting it in the prompt before reaching for a framework. The cost of this approach—higher per-conversation token usage and frontier model pricing—is real and may motivate alternatives such as compiling the procedure into a smaller model’s weights. But the quality comparison is unambiguous: for the class of tasks we tested, orchestration adds engineering complexity, failure modes, and latency while reducing quality. More broadly, these results suggest that as frontier models grow more capable, the value of external scaffolding diminishes for well-defined tasks. The frameworks were built when models needed more help; the models may have outgrown the scaffolding.

## B Conversation Examples: Travel Booking 

We present excerpts from actual evaluation conversations, comparing orchestrated and in-context approaches on the same scenario specification. Conversations are lightly edited for length; user responses differ between conditions due to dynamic simulation. 

Scenario: 4 friends traveling to Spain for 9 days, e2,600 budget, interested in extreme sports, art galleries, scenic retreats, and street food. Budget travel style, prefer boutique hotels. 

In-context agent (task success: 5, naturalness: 5, 20 turns): 11 Agent: Hello! I’d be happy to help you with your travel plans. What kind of trip are you thinking about? 

User: Hi! I’m planning a trip to Spain with three friends—so four of us total. We’re looking at about 9 days and have around e2600 to work with. We’re really into extreme sports, checking out art galleries, scenic retreats, and street food. 

Agent: That sounds like an amazing trip! Spain is perfect for that mix of adventure, culture, and food. Just to make sure I can put together the best options for you—when are you thinking of going, and do you have specific regions in mind? 

User: We’re flexible on dates. As for location, we’re open to suggestions! Maybe a mix would be good so we can get different vibes. Oh, and we’d really prefer boutique hotels over chains. 

Agent: Perfect! A mix sounds ideal for what you’re after, and boutique hotels will definitely add to the experience. Let me put together a few different itinerary options. . . 

[Agent presents three options. User provides feedback: “Option 1 feels too rushed, Option 3 seems too city-heavy—we really want those scenic retreats.” Agent asks clarifying questions about preferred scenery and extreme sports activities.] 

User: A mix of coastal and mountain would be ideal. For extreme sports we’re thinking rock climbing, canyoning, maybe paragliding. 

Agent: Perfect, that gives me a much clearer picture! Let me create some revised options that give you that coastal-mountain balance with those specific activities. . . 

[Agent presents revised options incorporating feedback. User selects Option 2 (Ronda + Tarifa) and asks about art galleries in the area. Agent provides detailed, integrated suggestions for art alongside the adventure itinerary.] 

LangGraph orchestrated agent (task success: 2, naturalness: 4, 14 turns): 

Agent: Hello! I’d be happy to help you plan your next trip. What kind of travel are you thinking about? 

User: Hi! I’m looking to book a trip to Spain for 9 days with three of my friends. We’re interested in extreme sports, art galleries, scenic spots, and street food. Our budget is e2600 and we prefer boutique hotels. 

[Agent asks about dates; user confirms late May. Agent presents three options.] 

User: These are interesting but not quite right. We really want boutique hotels, not hostels— that’s important to us. Also, with e2600 total for 4 people, the budget per person seems tight. . . 

Agent: Thanks for that feedback—you’re absolutely right. With e2,600 total for 4 people over 9 days including flights, accommodation, activities, and food, boutique hotels may be challenging. . . 

User: Wait, our budget is e2600 per person , not total! 

Agent: Oh wow, that changes everything—yes, e2,600 per person absolutely works for boutique hotels! 

[The orchestrated agent misinterprets the budget as per-group rather than per-person. While it recovers after correction, this information loss—caused by the node-by-node architecture not retaining the full conversation context across routing decisions—results in wasted turns and an incomplete booking flow.] 

The in-context agent maintains the full context throughout, iteratively refining options based on detailed feedback. The orchestrated agent loses critical information (per-person vs. total budget) across routing boundaries, requiring the user to correct a misunderstanding that the in-context agent never makes. 

## C Conversation Examples: Zoom Support 

Scenario: Other participants cannot hear the user. Microphone icon shows no audio levels. User on Mac (Ventura), browser-based Zoom. 

In-context agent (task success: 5, naturalness: 5, 11 turns): 

Agent: Hi there! I’m here to help you with any Zoom issues you’re experiencing. What’s going on with Zoom today? 

12 User: Hi! I’m hosting a meeting right now and the other participants are saying they can’t hear me at all. My microphone icon isn’t showing any audio levels when I talk. 

Agent: That sounds frustrating, especially during an active meeting! Let me help you troubleshoot this quickly. Are you using the Zoom desktop app or joining through your web browser? And what operating system are you on? 

User: I’m on a Mac running Ventura, and I’m using Zoom through my web browser, not the desktop app. 

Agent: Got it—Mac with the web browser version. That actually helps narrow this down, as the browser version has some specific audio quirks. Check your Mac’s System Settings 

> Privacy & Security > Microphone, and make sure your browser has permission to access the microphone. 

User: Oh, that was it! I checked System Settings and my browser didn’t have permission to access the microphone—I must have denied it at some point. I turned it on, refreshed the meeting, and now I can see the audio levels moving! 

Agent: That’s fantastic! I’m so glad it’s working now. That macOS microphone permission is sneaky—it can get disabled during system updates or if you accidentally clicked “Don’t Allow” at some point. 

LangGraph orchestrated agent (task success: 3, naturalness: 3): 

Agent: Hello! I’m here to help you with any Zoom issues. What problem are you running into today? 

User: I’m having trouble with my audio during meetings. Other people are saying they can’t hear me, and my microphone icon isn’t showing any sound levels when I talk. 

[Agent asks about platform details; user confirms Mac Ventura, browser-based Zoom. Agent suggests checking System Settings > Privacy & Security > Microphone. User confirms Chrome already has permission.] 

Agent: Let me check: in Zoom’s audio settings, what microphone is selected? 

User: I see “MacBook Pro Microphone” with a checkmark, and there’s also “Same as System”—should I try switching? 

Agent: Let’s try the audio test: click “Test Mic” and speak—do you see the green bar move? 

User: I spoke pretty loudly but the green bar stayed completely flat—no movement at all. 

[Conversation continues for several more turns without resolving the issue. The orchestra-tor cycles through additional troubleshooting steps but never identifies the browser-level permission as the root cause.] 

The in-context agent identifies the most likely cause (browser microphone permission) on its first diagnostic suggestion and resolves the issue in 6 agent turns. The orchestrated agent, constrained to the current node’s troubleshooting template, works through each step sequentially without the holistic view needed to prioritize the most probable diagnosis.

If user wants to ABANDON → close gracefully [user_abandoning] 

14 HANDLE_RESPONSE (Hub 2): 

Respond to the user’s reaction to the options. 1. If they PICKED AN OPTION: Confirm their choice, summarize key details, ask if ready to finalize [option_selected] 2. If they WANT MODIFICATIONS: Acknowledge feedback, suggest adjustments [needs_revision] 3. If they have QUESTIONS: Answer helpfully [answered_question] 4. If CHANGING REQUIREMENTS: Go back to assessment [changing_requirements] 5. If ABANDONING: Close gracefully [user_abandoning] 6. If NEEDS HUMAN: Escalate [needs_human]

Handle the user’s final response. 1. If they CONFIRMED: Express enthusiasm, provide tips, close warmly [confirmed] 2.
