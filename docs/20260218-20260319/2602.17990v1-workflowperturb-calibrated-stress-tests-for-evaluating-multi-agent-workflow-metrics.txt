Abstract 

LLM-based systems increasingly generate structured workflows for complex tasks. In practice, automatic evaluation of these work-flows is difficult, because metric scores are of-ten not calibrated, and score changes do not directly communicate the severity of workflow degradation. We introduce W ORKFLOW PER -

> TURB

, a controlled benchmark for studying workflow evaluation metrics. It works by apply-ing realistic, controlled perturbations to golden workflows. WORKFLOW PERTURB contains 4,973 golden workflows and 44,757 perturbed variants across three perturbation types (Miss-ing Steps, Compressed Steps, and Description Changes), each applied at severity levels of 10%, 30%, and 50%. We benchmark multi-ple metric families and analyze their sensitivity and calibration using expected score trajecto-ries and residuals. Our results characterize sys-tematic differences across metric families and support severity-aware interpretation of work-flow evaluation scores. Our dataset will be released upon acceptance. 

1 Introduction 

Large Language Models (LLMs) are increasingly used to generate multi-step, dependency-rich work-flows for high-stakes settings, where failures can have real operational or scientific consequences. Such workflows now appear in scientific automa-tion, biomedical information access, enterprise copilots, and cloud and DevOps assistants (Bran et al., 2023; Jin et al., 2023; Microsoft, 2026; Chan-drashekar, 2025). In these settings, correctness depends not only on wording but on ordered depen-dencies and the presence of critical steps. Evaluating these workflows remains unexpect-edly difficult. Existing workflow metrics each cap-ture only a narrow slice of correctness: lexical metrics fail under paraphrasing, structural metrics may overlook semantic drift, and semantic metrics can assign favorable scores to workflows missing critical steps. A central practical challenge is cal-ibration: it is often unclear what a score means in terms of functional risk. For example, does a drop from 0.90 to 0.84 reflect harmless rephrasing or the loss of an essential dependency? Without calibrated interpretation, metric values are difficult to use for regression testing, model comparison, or automated filtering. With the widespread deployment of LLM-generated workflows in industry, incorrect or de-graded workflows can have severe consequences. In cloud and DevOps settings, flawed orchestration may lead to outages or misconfigured infrastruc-ture; in scientific and biomedical domains, sub-tle workflow deviations can alter analytical con-clusions. Production systems are also dynamic: models are deprecated and upgraded, for example under Azure OpenAI model retirement policies 1,orchestration libraries evolve, and engineers revise prompts and agent definitions. Each such change may alter workflow structure or semantics. This raises a core practical question: how can we deter-mine whether a modified workflow remains func-tionally equivalent to a validated reference? How can such checks be integrated into CI/CD pipelines without manual review? At the heart of this prob-lem lies reliable workflow evaluation. To make this concrete, consider a validated production workflow for deploying a service, in-cluding resource provisioning, access controls, de-ployment, and observability configuration. An LLM-generated workflow may preserve terminol-ogy while omitting access hardening, compressing observability into a vague step, or reordering de-pendencies. Many metrics still assign high scores due to surface overlap or coarse structural similar-ity, leaving engineers uncertain whether deviations are benign or dangerous.

To address these challenges, we introduce WORKFLOW PERTURB , a benchmark of over 44,000 workflows constructed through controlled perturbations. We generate three perturbation types, Missing Steps, Compressed Steps, and Description Changes, each applied at graded intensity levels of 10%, 30%, and 50%. These structured degradations enable systematic analysis of metric calibration un-der known severity, revealing which metrics are robust to benign edits, which fail under functional degradation, and how workflow scores can be inter-preted more reliably in production settings. 

2 Related Work 

Agentic evaluation and tool-use benchmarks. 

Recent work has focused on evaluating large lan-guage models in agentic and tool-using settings, where models must select, invoke, and sequence external functions. The Berkeley Function Calling Leaderboard (BFCL) (Patil et al., 2025) provides a standardized benchmark for evaluating LLMs on re-alistic function-calling tasks, emphasizing correct-ness of tool selection, and execution outcomes, and moving beyond single-turn generation to end-to-end task success in tool-augmented environments. While BFCL evaluates agent performance through execution success, it does not study how automatic evaluation metrics behave under different types or severities of workflow degradation. 

Diagnosing agent failures from execution traces. 

Several recent approaches analyze execution tra-jectories to diagnose and localize agent failures. AGENTRX (Barke et al., 2026) examines agent execution traces consisting of intermediate actions, tool calls, and environment feedback to identify failure causes and error propagation. TRAIL (Desh-pande et al., 2025) similarly uses trace-based rea-soning over structured action sequences and inter-mediate states to localize issues in agent execution. These methods operate on runtime execution traces and assume access to environment interactions and task outcomes. Our setting differs in that we do not rely on execution traces or success signals; instead, we study static workflow representations and eval-uate how automatic metrics respond when work-flows deviate from golden references in controlled and graded ways, enabling systematic analysis of metric calibration, sensitivity, and blind spots. 

3 WORKFLOW PERTURB 

We model a workflow as a directed acyclic graph (DAG) G = ( V, E ), where each node v ∈ V rep-resents a step described in natural language and each edge (u, v ) ∈ E denotes a precedence con-straint. Given a golden workflow G and a candi-date workflow G′, an evaluation metric produces a score s(G, G ′) ∈ [0 , 1] (or a normalized variant) intended to reflect workflow quality or similarity. Rather than proposing a new workflow generator, our focus is to analyze metric behavior : under con-trolled degradations of known severity, do metric scores change in an interpretable and calibrated manner? We study this question by introducing structured perturbation types, graded severity lev-els, and an expected score trajectory that character-izes how an ideal metric should respond as work-flow quality degrades 

3.1 Benchmark Construction Source Data. We begin with the set of workflows in WORFBENCH (Qiao et al., 2024) and retain those with |V | ≥ 5 nodes. This threshold ensures each workflow has enough steps to meaningfully perturb and evaluate metric sensitivity. 

Perturbation Patterns. Through exploratory analysis of LLM-generated workflows, we iden-tified several issues that degrade workflow quality, among which three recurring patterns emerged. We model these patterns as controlled perturbations applied to golden-standard workflows: • Missing Steps: LLM-generated workflows of-ten omit essential actions that are required for complete task execution. Such omissions result in incomplete or truncated execution paths when compared to the golden workflow. • Compressed Steps: Generated workflows fre-quently contain fewer nodes than their golden counterparts, condensing multiple fine-grained actions into a single broader step. While this may preserve most of the content, it alters the structural properties of the workflow and leads to mismatches under graph-based evaluation. • Description Changes: Even when workflows preserve the correct structure, step descriptions often differ syntactically from the golden ref-erence. This creates lexical variation that can confuse surface-level metrics while retaining se-mantic meaning. Functional Implications of Perturbations. Al-though WORKFLOW PERTURB evaluates static workflow representations, the perturbations are de-signed to reflect realistic failure modes with poten-tial execution-level consequences. Missing steps may eliminate required actions, leading to incom-plete or failed task execution. Compressed steps can merge distinct tool invocations or dependency boundaries, potentially altering execution order or granularity. In contrast, description changes pre-serve structural dependencies but may introduce se-mantic ambiguity. Our objective is to study how au-tomatic metrics respond to these controlled degra-dations; evaluating their downstream execution im-pact is complementary future work. To systematically study the impact of these is-sues on evaluation metrics, we apply each perturba-tion type at three severity levels {10% , 30% , 50% }

of the nodes in the golden workflow are affected as illustrated in Figure 2. These graded levels not only mimic realistic variations observed in practice but also allow us to probe the calibration of metrics, i.e., whether metric scores degrade proportionally with increasing perturbation severity. 

Perturbation Generation and Validation. For each workflow and perturbation configuration, we employ large language models to generate variants. To ensure correctness, each variant undergoes auto-mated static validation checks: • Node Count Consistency: Given a golden work-flow with 10 nodes, applying a 10% removal per-turbation must produce a workflow with precisely 9 nodes. Any deviation from this expected count is flagged as inconsistent. The nodes selected for removal are chosen randomly. • Change Count Verification: For a 30% pertur-bation, exactly 3 nodes must be altered, either removed, compressed, expanded, or paraphrased. If fewer or more nodes are modified, the output is rejected. For removal and compression oper-ations, the results are verified by comparing the number of nodes before and after the perturbation. For paraphrasing, we explicitly pass the specific nodes to be paraphrased to ensure that exactly the required number of nodes are modified. These checks target the two most common generation failure modes: (i) deleting or merging too many nodes, and (ii) over- or under-applying paraphrasing. All prompt templates used in this stage along with our iterative refinement strategy are documented in Appendix A.

Changes. For the description-change perturbation, we do not rely on unconstrained paraphrasing, which often introduces hallucinations. Instead, we explicitly pre-select α × | Vgolden | nodes (where α ∈ { 0.1, 0.3, 0.5}) for paraphrasing. The model is instructed to only rephrase the text of those nodes without adding or removing factual content. For example, a step such as “Check server logs for errors” may be paraphrased to 

“Inspect the server log files for error entries” , while the graph structure and node set remain unchanged.

Score Assignment and Perturbation Alignment. 

Each workflow is assigned a score reflecting per-turbation severity, with lower scores indicating de-viation from the golden workflow. For Missing Steps and Compressed Steps , the score equals the remaining workflow fraction, computed as 1 − p

for perturbation percentage p (e.g., p=30% → 0.7,

p=50% → 0.5). In the case of Compressed Steps, this decrease is justified by the loss of struc-tural granularity: multiple fine-grained actions (of-ten corresponding to distinct tool invocations) are merged into broader nodes, violating the one-tool-per-step abstraction and reducing fidelity to the golden execution graph. For Description Changes ,the score remains constant across severity levels, since these edits affect only textual descriptions and not the underlying workflow structure. 

4 Metrics 

We evaluate seven metrics spanning structural, lex-ical, semantic, order-based, and judgment-based families. Structural metrics (Graph F1, Chain F1) measure topological consistency via subgraph matching and sequence alignment. Lexical met-rics (BLEU, GLEU) compute n-gram overlap. BERTScore measures contextual embedding sim-ilarity. Kendall’s τ evaluates rank-order consis-tency. LLM-as-Judge provides rubric-based holis-tic assessment. Full definitions are provided in Appendix B. 

4.1 Structural Metrics Chain F1. Following (Qiao et al., 2024), predicted node chains are aligned with valid topological or-derings of the golden workflow, and the longest increasing subsequence (LIS) is computed. Preci-sion and recall are defined as pchain = l/ |Vpred | and 

rchain = l/ |Vgolden |, where l is the LIS length. The final score is their harmonic mean (F1). 

Graph F1. The predicted and golden workflow graphs are aligned using a Maximum Common Induced Subgraph (MCIS) algorithm. If the largest shared subgraph contains k nodes, precision and recall are defined as pgraph = k/ |Vpred | and rgraph =

k/ |Vgolden |. The score is their harmonic mean (F1). 4.2 Lexical Metrics BLEU (Papineni et al., 2002) is a precision-oriented n-gram matching metric with a brevity penalty, while GLEU (Napoles et al., 2015) bal-ances n-gram precision and recall and is more ro-bust for shorter or paraphrased step descriptions. 

4.3 Semantic Metric BERTScore (Zhang et al., 2020) compares con-textual embeddings of step descriptions from pre-trained transformer models and computes precision, recall, and F1 via token-level cosine similarity. 

4.4 Ordering Metric Kendall’s τ (Kendall, 1938) measures rank corre-lation between generated and gold step orderings, defined as τ = ( C − D)/  12 n(n − 1) , where C

and D are the numbers of concordant and discor-dant pairs among n aligned steps. Scores range from −1 to 1.

4.5 LLM-as-Judge 

To complement automatic metrics, we use an LLM-as-Judge evaluation with GPT-4o (Azure OpenAI). The model is prompted with the task description, golden workflow, generated workflow, and a rubric defining 0–5 scores covering correctness, complete-ness, ordering, and clarity, with anchor examples. 

5 Results and Analysis 

We evaluate metric behavior across perturbation types and severities, assigning each workflow a pre-defined degradation score for interpretability. All reported LLM-as-Judge scores are linearly normal-ized from the original 0–5 scale to the [0,1] range for comparability across metrics. 

Overall trends. Across all perturbation types, metric scores generally decrease as perturbation severity increases, confirming that the benchmark introduces progressively harder deviations from the golden workflows. However, degradation patterns differ substantially across metric families. Struc-tural metrics (Graph F1, Chain F1) track struc-tural distortions closely, lexical metrics (BLEU, GLEU) are highly sensitive to token-level varia-tion, embedding-based similarity (BERTScore) is less sensitive to structural perturbations because it does not directly encode graph completeness, while ordering-based metrics (Kendall’s τ ) mainly change when perturbations alter relative step prece-dence (e.g., via compression or deletion). 6 Key Insights and Practical Guidance 

Workflow validation under change. In indus-trial settings, workflows frequently evolve due to model upgrades, prompt revisions, or orchestration library changes. Our results show that different metric families respond differently to structural versus textual changes. Structural metrics are re-liable indicators of missing or compressed steps, making them suitable for detecting functional re-gressions when workflows are modified. Lexical metrics detect paraphrasing but may over-penalize deletions. LLM-as-Judge provides a holistic sig-nal aligned with perceived workflow quality but introduces variance. These distinctions are critical when validating workflow updates. 

CI/CD integration. WorkflowPerturb enables severity-aware calibration of evaluation metrics, allowing practitioners to interpret scores relative to expected degradation patterns. In a CI/CD pipeline, candidate workflows generated after model or or-chestration updates can be compared against ap-proved golden workflows using a calibrated metric bundle. Deviations beyond empirically validated thresholds can automatically trigger review or roll-back. By understanding which metrics are sensi-tive to which failure modes, automated validation pipelines can be configured to prioritize structural regressions while tolerating benign paraphrasing. 

Metric selection in practice. No single metric suffices across all perturbation types. We recom-mend reporting a compact bundle of complemen-tary metrics and interpreting them jointly with re-spect to perturbation severity. WorkflowPerturb provides a controlled testbed for selecting and cali-brating such metric bundles before deployment. 

7 Conclusion 

We introduced W ORKFLOW PERTURB , a bench-mark for evaluating workflow evaluation metrics under controlled structural and textual perturba-tions. Our analysis reveals that commonly used metrics exhibit distinct sensitivity patterns and cal-ibration gaps across failure modes. These findings highlight the importance of severity-aware interpre-tation when validating workflow updates in produc-tion systems. By enabling calibration and metric selection, WorkflowPerturb provides practical sup-port for automated workflow validation in CI/CD pipelines and evolving LLM-driven platforms. References

A Prompt Templates 

The following prompts were used to generate per-turbed workflow variants. Placeholders in curly braces {} denote the specified perturbation type or severity level. 

A.1 Missing Step Prompt 

Given the following workflow, please remove exactly {10%, 30%, 50%} of its steps. Return only the modified workflow as a numbered list of steps, preserving the order of the remaining steps. Do not add or paraphrase any content. 

A.2 Compressed Step Prompt 

Given the following workflow, please merge consecutive steps until exactly {10%, 30%, 50%} of the original steps are merged. Each merged step should concisely describe the actions of the combined steps. Return only the modified workflow as a numbered list of steps. Do not remove or paraphrase steps outside the merged groups. 

A.3 Description Change Prompt 

Given the following workflow, please paraphrase the descriptions of exactly {10%, 30%, 50%} of its steps. Ensure that the meaning of each paraphrased step remains unchanged. Do not remove, merge, or add any steps. Return only the modified workflow as a numbered list of steps. 

A.4 Iterative Refinement Prompt (Error Injection) 

When a generated output fails validation, we re-inject the error message into the prompt. For exam-ple: 

Your output had 8 nodes instead of 9 for a 10% removal. Please regenerate the workflow and ensure that exactly one node is removed. Return only the modified workflow as a numbered list of steps. 

This process is repeated up to three times per model before escalation to a stronger model. 

A.5 Iterative Refinement and Escalation 

Variants failing validation enter a feedback-driven refinement loop. The validation error is explicitly injected back into the next prompt. For example: 

“Your output had 8 nodes instead of 9 for a 10% removal. Please regenerate the workflow and ensure exactly one node is removed.” 

If the corrected output again fails validation (e.g., paraphrased 4 nodes instead of 3), the process is repeated up to three times. After three unsuccessful iterations, the task is escalated to a more reliable generation method.

B.3 BLEU and GLEU 

BLEU (Papineni et al., 2002) measures n-gram precision between candidate and gold step descrip-tions, with a brevity penalty to discourage very short outputs. GLEU (Wu et al., 2016) balances precision and recall over n-grams, making it more robust to short sequences. Both are computed at the step-text level and then averaged across workflows. 

B.4 BERTScore 

BERTScore (Zhang et al., 2020) computes similar-ity between candidate and gold step texts using con-textual embeddings (we use all-mpnet-base-v2 ). For each candidate token, the maximum cosine similarity with gold tokens is taken, yielding pre-cision, recall, and F1. Scores are averaged across workflow steps.

B.6 LLM-as-Judge Prompts 

For the LLM-as-Judge evaluation, we designed a rubric-based prompt that instructs the model to compare a generated workflow against a gold refer-ence. Workflows are formatted with explicit node and edge lists. 

B.6.1 Rubric 

• 0 = Completely Incorrect or Irrelevant • 1 = Major Flaws (breaks core logic) • 2 = Significant Issues (omits several key steps or adds wrong steps) • 3 = Noticeable Errors (minor reorder or one extra/missing non-critical step) • 4 = Very Good (only one minor step missing) • 5 = Perfect Match (fully adheres, detailed, accurate)

The model is instructed to: 1. List nodes and edges from both workflows. 2. Match nodes by content. 3. Identify missing, extra, or reordered steps. 4. Assign an initial score using the rubric. 5. Perform a self-check: confirm if the initial score strictly follows the rubric. 6. Adjust to a final score if necessary. 

B.6.4 Expected Output 

The model must return exactly one JSON object with the fields: 

{"ThoughtChain": "<step-by-step reasoning>", "initial_score": <int 0–5>, "self_check": { "consistent": "Yes"|"No", "final_score": <int 0–5> }, "justification": "<one-sentence summary>" }

B.6.5 Implementation Notes 

We use GPT-4o via Azure OpenAI with tempera-ture 0.0 to minimize randomness. Each workflow pair is evaluated across multiple prompt templates, and scores are averaged to reduce variance.
