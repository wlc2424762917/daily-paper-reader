## Abstract 

As Large Language Model (LLM) agents increasingly execute com-plex, autonomous software engineering tasks, developers rely on natural language Agent Instructions (e.g., AGENTS.md ) to enforce project-specific coding conventions, tooling, and architectural bound-aries. However, these instructions are passive text. Agents fre-quently deviate from them due to context limitations or conflicting legacy code, a phenomenon we term Context Drift . Because agents operate without real-time human supervision, these silent viola-tions rapidly compound into technical debt. To bridge this gap, we introduce ContextCov , a framework that transforms passive Agent Instructions into active, executable guardrails. ContextCov extracts natural language constraints and synthesizes enforcement checks across three domains: static AST analysis for code patterns, runtime shell shims that intercept prohibited commands, and architectural validators for structural and semantic constraints. Evaluations on 723 open-source repos-itories demonstrate that ContextCov successfully extracts over 46,000 executable checks with 99.997% syntax validity, providing a necessary automated compliance layer for safe, agent-driven development. Source code and evaluation results are available at https://anonymous .4open .science/r/ContextCov-4510/.

The Rise of Agent Instructions. To bridge this supervision gap, the developer community has organically adopted Agent In-struction files . Stored in files such as AGENTS.md , CLAUDE.md , or

The Problem. Agent Instructions suffer from a critical flaw. They are passive text files, not executable specifications. This in-troduces the challenge of Context Drift , where an agent produces changes that violate the constraints specified in its instructions. Many constraints are inherently difficult to verify through manual review. Without automated enforcement, violations accumulate silently. Two distinct challenges contribute to Context Drift. The first is 

underspecified instructions , which we do not address since agents cannot follow unspecified constraints. The second, which we focus on, is agent non-compliance . Even when instructions are adequately specified, agents may fail to follow them due to conflicting input, capability limitations, context overload, or codebase patterns that conflict with stated instructions [ 28 , 43 ]. We term this gap between what agents should do and what they actually do the Reality Gap. The Reality Gap has severe implications. An agent instructed to “use pnpm ” but encountering legacy package-lock.json files faces an instruction-environment conflict. LLMs often prioritize immediate context over system-level directives [ 30 , 37 , 54 ], leading to non-compliant changes. Similarly, agents optimizing for local task completion may cause architectural erosion [ 12 ] by introducing circular dependencies or layer breaches. Perhaps most concerning,

Our Approach. ContextCov addresses this challenge by trans-forming passive Agent Instructions into active, verifiable invariants. We argue that as agent autonomy increases, Agent Instructions must be treated as executable specifications, not passive documen-tation but verifiable code that compiles into runtime checks. Con-textCov operationalizes this principle through (1) Hierarchical Constraint Extraction , a path-aware parsing algorithm that pre-serves semantic scope imposed by heading hierarchies, (2) Domain-Routed Policy Synthesis , an intent classification system that routes each constraint to specialized code generators for Process , Source ,or Architectural enforcement, and (3) Multi-Layer Runtime En-forcement , comprising a Process Interceptor via PATH injection, a Universal Static Linter built on Tree-sitter, and an Architectural Validator using graph algorithms and LLM-based judgment.

(3) We present the ContextCov architecture, comprising domain-specialized code synthesis and a runtime enforcement layer (Section 3). (4) We evaluate ContextCov on 723 open-source repositories, demonstrating that it extracts diverse constraints across mul-tiple enforcement domains (Section 4).

Lines 8–22 of Figure 1 specify coding conventions for VS Code. 

Possible Violation. Consider an agent tasked with “Add a new file watcher callback” that writes natural-looking TypeScript:  

> watcher.onDidChange((event) => { this.handleChange(event); }); ContextCov : Deriving and Enforcing Executable Constraints from Agent Instruction Files Conference’17, July 2017, Washington, DC, USA

This code is syntactically valid and functionally correct. However, it violates VS Code’s style guidelines because the single parameter 

event should not be parenthesized. The agent may have observed similar patterns in legacy code and mimicked them. Without en-forcement, these style inconsistencies would accumulate across the codebase. 

ContextCov addresses this by generating a Tree-sitter query that detects arrow functions with unnecessarily parenthesized sin-gle parameters. When the static linter runs, it reports the violation with the source location and a suggested fix, enabling the agent to self-correct before commit: 

> [ContextCov] Source violation: src/vs/workbench/watcher.ts:127 Rule: "Only surround arrow function parameters when necessary" Found: (event) => {...} Suggestion: Remove parentheses: event => {...}

More broadly, source violations occur when an agent generates code patterns that contradict coding standards. These include using explicitly forbidden patterns (e.g., any types), violating naming conventions or formatting rules, and using deprecated APIs. Agents often introduce these violations by mimicking patterns in existing code, even when those patterns contradict documented standards. 

## 2.3 Architectural Violation 

Lines 24–38 of Figure 1 define VS Code’s architectural constraints. We distinguish two subtypes. Deterministic constraints can be ver-ified programmatically, while semantic constraints require LLM-based judgment. 

2.3.1 Deterministic Architectural Constraints. Lines 24–33 specify the layered architecture and directory structure. 

Possible Violation. Consider an agent tasked with “Add a utility function for file path normalization” that faces a decision about where to place the code. The agent might place it in src/vs/workbench/common/ 

because that is where it will be used. However, VS Code’s layered architecture requires that such utilities belong in src/vs/base/ ,the foundation layer. Similarly, an agent might create a file directly under src/vs/workbench/ rather than in one of the allowed subdi-rectories like browser , services , contrib , or api . Such violations are easy to make when the agent is focused on completing the im-mediate task rather than understanding the broader architectural constraints. 

ContextCov ’s architectural validator constructs a dependency graph and enforces the layered structure. If an agent attempts to create a file in an invalid location, it receives immediate feedback: 

> [ContextCov] Architectural violation: src/vs/workbench/myNewUtility. ts Rule: "workbench/ only allows subdirectories: browser, services, contrib, api" Suggestion: Move to appropriate subdirectory or to base/

More broadly, deterministic architectural violations include layer violations (dependencies crossing architectural boundaries), cyclic dependencies, and module placement errors. These can be detected through file path analysis and import graph traversal. 

2.3.2 Semantic Architectural Constraints. Lines 35–38 specify a de-sign principle where components must not directly access another component’s storage keys but should instead define proper APIs for cross-component changes. Such constraints cannot be verified through file paths or import graphs alone because they require understanding the code’s intent. 

Possible Violation. Consider an agent tasked with “Persist user preferences across sessions” that discovers another component already stores related settings. Rather than defining a proper API, the agent directly reads and writes to that component’s storage keys. This violates encapsulation and couples the components in ways that are difficult to detect in code review. 

ContextCov handles semantic constraints using an LLM-as-judge approach, where the LLM evaluates whether code changes respect the guideline. These checks produce warnings rather than hard blocks, flagging potential violations for human review:

## 3 ContextCov Design 

ContextCov consists of two main phases. Check generation ex-tracts and synthesizes constraints into executable checks, and run-time enforcement executes these checks during agent operation. Given an Agent Instruction file, ContextCov generates code-based checks that can be incrementally updated as the instruction file evolves. At runtime, source and architectural checks can be en-forced over an entire repository, a git diff, or just unstaged changes, while process checks are enforced during agent execution. Con-textCov is implemented in Python and uses gpt-5.2-chat for all LLM-based operations including constraint refinement, intent routing, code generation, and semantic evaluation. 

## 3.1 System Overview 

Figure 2 presents the high-level architecture. During extraction, Agent Instruction documents are parsed into a Markdown Abstract Syntax Tree, and a path-aware algorithm traverses the tree extract-ing constraints as slices with their hierarchical context preserved. An LLM refines each constraint into a standalone, unambiguous statement. During synthesis, each refined constraint is classified by an intent router into one of four enforcement domains and passed to specialized code generators that produce executable Python checks. These checks are stored in a JSON file. The Enforcement phase executes these checks through special-ized modules. A Process Interceptor uses PATH shims to intercept shell commands before execution. A Universal Static Linter uses Tree-sitter to analyze source code across multiple languages. An Architectural Validator constructs and queries dependency graphs to detect structural violations. Finally, an LLM-as-judge evaluates semantic architectural constraints that cannot be verified through static analysis alone. This architecture provides several benefits. When Agent Instruc-tions change, only affected constraints are re-extracted and re-synthesized, enabling efficient incremental updates. Using separate code synthesis modules improves generation quality, as each expert generator is prompted with domain-appropriate patterns and exam-ples. The multiple enforcement modules address the heterogeneity of agent actions. ## 3.2 Constraint Extraction 

The extraction phase transforms unstructured Markdown into a set of refined, standalone constraints. This is challenging because Agent Instructions are written for human readers who naturally understand context and scope from document structure. 

The Challenge of Lexical Scope. Simple approaches to Agent In-struction segmentation fail to preserve lexical scope . Consider the following excerpt: 

> # Backend ## Testing - Use pytest ## Database - Use SQLAlchemy

The instruction “Use pytest” appears as a standalone bullet, but its meaning depends on its hierarchical context and applies to Back-end Testing , not to the entire project or to Database operations. If we extract this constraint without its context, we might incorrectly enforce pytest usage for frontend code or database migrations. 

Markdown Abstract Syntax Tree. To preserve this hierarchical context, we parse each Agent Instruction file into a tree structure. Internal nodes represent headers at various levels (H1 through H6), leaf nodes represent content blocks (paragraphs, list items, code blocks), and edges represent containment relationships. This tree structure mirrors how humans mentally organize the document when reading it. 

Path-Aware Slicing. For each leaf node representing a potential constraint, we compute its header path , the ordered sequence of ancestor headers from the document root to the leaf. In the example above, “Use pytest” has the header path [Backend, Testing] . We then use an LLM to rewrite the raw instruction into a standalone, unambiguous statement that incorporates this context, produc-ing a refined constraint like “For backend testing, use the pytest framework for all unit and integration tests.” This refinement step disambiguates scope and normalizes diverse phrasings into consis-tent constraint statements that can be enforced without additional context. 

Stable Identifiers for Incrementality. To support incremental up-dates, we assign each constraint a stable identifier by hashing its header path and content. When a developer modifies one section of an Agent Instruction file, only constraints from that section need to be re-extracted and re-synthesized. Constraints from unmodi-fied sections retain their identifiers and their previously generated checks remain valid. This is important for large instruction files where full re-extraction would be expensive. 

## 3.3 Constraint to Check 

This phase transforms refined constraints into executable Python code. Rather than using a single monolithic code generator, we route each constraint to a specialized generator based on its en-forcement domain. This domain-specific approach improves code quality because each generator is optimized for its particular verifi-cation pattern. 

The Intent Router. The Router classifies each constraint into one of four enforcement domains. For the Process Domain , the router extracts target binary names (e.g., npm , yarn ), the enforcement level, and the applicable scope. For the Source Domain , it identifies tar-get programming languages, the pattern type to detect, and the severity level. For the Architectural Deterministic Domain , it determines the constraint type (dependency direction, cycle detec-tion, or boundary enforcement) and selects an appropriate graph algorithm. For the Architectural Semantic Domain , it preserves the natural language description of the design principle for later LLM evaluation. The router’s classification is performed by the LLM, prompted with examples of each domain to improve accuracy. 

Specialized Code Generators. Each domain has a specialized Ex-pert Generator that produces Python code tailored to that domain’s verification needs. The Process Expert generates code that inspects command-line arguments and environment variables, returning a verdict that determines whether the command should proceed. The Source Expert generates code using Tree-sitter bindings that constructs AST queries and traverses parse trees, returning vio-lations with precise file and line information. The Architectural Expert generates code using NetworkX that applies graph algo-rithms to dependency structures, detecting cycles, layer violations, or boundary crossings. Tree-sitter grammars are bundled for Python, TypeScript, JavaScript, Go, and Rust. This coverage addresses the most common languages in repositories with Agent Instructions, though additional lan-guages can be added by installing the corresponding Tree-sitter packages. 

Check Storage. Synthesized checks are stored in a JSON file that can be reviewed and manually edited by developers before deploy-ment, providing a layer of human oversight over the generated checks. The listing below shows an example entry: 

> {"constraints": { "a7f3b2...": { "domain": "PROCESS", "original_text": "NEVER use npm run compile", "refined_text": "Block npm run compile commands", "source_file": "copilot-instructions.md", "header_path": ["Validating TypeScript changes"], "enforcement_level": "block", "check_code": "def check(cmd, args, env): ..." }}}

This separation of synthesis from enforcement provides several advantages. It enables incremental updates where only changed constraints trigger re-synthesis. It allows developers to inspect, modify, or disable specific checks before they take effect. And it ensures fast enforcement at runtime with no LLM calls required, since all checks are pre-compiled Python code. 

## 3.4 Runtime Check Enforcement 

The enforcement phase executes synthesized checks through four specialized modules, each designed for a particular type of con-straint. 

Process Interceptor. Enforcing process constraints requires inter-cepting shell commands before execution. We use $PATH manipula-tion for its simplicity and portability. ContextCov creates a shim directory containing wrapper scripts for monitored binaries and prepends this directory to $PATH . Each monitored binary (e.g., npm ,

yarn , pip ) has a corresponding wrapper script:  

> #!/bin/bash # .contextcov/bin/npm
> exec python -m contextcov.shim npm "$@"

When a command is executed, the shell resolves the binary via 

$PATH and finds our shim first. The shim loads relevant checks from the JSON file and executes each check with the command arguments and environment. If any check fails, the shim prints an error message to stderr explaining which constraint was violated and exits with code 1, preventing the command from running. If all checks pass, the shim uses which -a to find the real binary and executes it with the original arguments, making the interception transparent to the agent. 

Universal Static Linter. ContextCov uses Tree-sitter as a univer-sal parser for source code analysis. Tree-sitter provides incremental parsing, graceful error recovery, and a uniform API across lan-guages, making it ideal for analyzing code in diverse repositories. The linter scans source files matching configured patterns, parses each file with the appropriate grammar based on file extension, and runs all applicable checks. Each check receives the parse tree, the raw source bytes, and the file path, allowing it to construct queries, traverse the AST, and report violations with precise loca-tions. The following example shows a generated check that detects 

Promise.then() usage:

Architectural Validator. Deterministic architectural constraints are enforced by constructing and querying a dependency graph. The graph is built by parsing import statements using language-appropriate techniques. Python’s ast module extracts import and 

from ... import statements, Tree-sitter queries extract import 

declarations and require() calls from JavaScript and TypeScript, and regex-based extraction handles Go import blocks. The resulting graph is stored as a NetworkX DiGraph where nodes represent modules and edges represent dependencies. Architectural checks operate on this graph to detect violations such as layer crossings, cycles, or improper module placement. The following example enforces that files under src/vs/workbench/

Semantic Architectural Validator. Some architectural constraints cannot be verified through static analysis alone because they require understanding the intent behind code. For example, the constraint “Do not use another component’s storage keys, define a proper API instead” cannot be checked by examining import graphs or file paths. It requires understanding what constitutes a “storage key,” which code belongs to which “component,” and whether an access pattern constitutes a violation. For these semantic constraints, we use an LLM-as-judge ap-proach [ 23 ]. When evaluating code changes, the LLM receives the constraint text, the relevant code snippet, and surrounding con-text such as file path and neighboring code. The LLM then judges whether the code respects the design principle, providing a verdict and explanation. Semantic checks produce WARNING verdicts rather than hard blocks, flagging potential violations for human review while avoiding false positives from overly strict interpretation. 

## 3.5 Handling Ambiguity 

A key design decision is how to handle ambiguous constraints. Agent Instructions are often written informally, and the same intent Conference’17, July 2017, Washington, DC, USA Reshabh K Sharma

Should it apply globally or only in certain directories? We adopt a fail-closed philosophy where ambiguous constraints are interpreted strictly. “Use pnpm ” triggers blocking of npm , yarn ,and bun globally, even though a human reader might interpret this more narrowly. This approach may cause false positives, but we argue this is the correct trade-off. False positives can be overridden by developers who understand the context, either by refining the Agent Instructions to be more precise or by marking specific checks as exceptions. False negatives, on the other hand, allow violations to slip through undetected, accumulating as technical debt. This philosophy aligns with security principles where, when uncertain, the system denies the action and requires explicit authorization.

## 4.3 RQ2. Detection in the Wild 

We evaluate ContextCov ’s ability to detect constraint violations in real-world repositories. In practice, ContextCov would run on code diffs as an agent makes changes, flagging violations before they are committed. For evaluation purposes, we run ContextCov 

against entire repositories to assess its detection capability. This approach is necessary because these repositories do not contain metadata identifying which code was contributed by agents. It is also meaningful because agents learn from existing code patterns. Any violation found in the existing codebase represents a pattern that an agent could mimic or propagate, making repository-wide analysis a proxy for the violations agents are likely to introduce. ## 5 Discussion 

The shift from human-supervised to autonomous agents funda-mentally changes the role of project documentation. Context Drift, where agents produce changes that violate their instructions, is not merely a technical bug but a symptom of a deeper mismatch between how we document intent and how agents consume it. We discuss the theoretical and practical implications of treating Agent Instructions as executable specifications to bridge this Reality Gap. 

## 5.1 Documentation as Code 

Historically, software documentation was descriptive , written for humans who use common sense to bridge the gap between outdated docs and current code. In the era of autonomous agents, documen-tation like AGENTS.md is prescriptive , serving as literal execution context. ContextCov forces a paradigm shift where documenta-tion must co-evolve with the codebase. By aggressively blocking commands or flagging code that violates Agent Instructions, the tool treats ambiguity and staleness not as minor annoyances but as build-breaking defects. This effectively turns passive Markdown into a living contract between the human developer and the AI agent. This shift has several implications. When ContextCov blocks an agent action, developers face a choice. Either the agent’s action was wrong and the instructions are correct, or the instructions need updating. Both outcomes improve the repository. This cre-ates an instruction quality feedback loop that does not exist with passive Agent Instructions. Just as code changes undergo testing, Agent Instruction changes should be tested. ContextCov enables “instruction tests” that verify the codebase complies with Agent Instructions before merging instruction changes. 

## 5.2 The Agentic Feedback Loop 

Because natural language is inherently ambiguous, compiling it into strict checks will yield “false positives,” such as globally block-ing npm when the developer only meant to block it in /backend .Instead of seeing this as a failure of the tool, we view it as a critical communication mechanism. When an agent hits a ContextCov enforcement block, the er-ror is fed back into its context window. This empowers the agent to reason about the failure. If the agent determines the rule was poorly scoped, it can autonomously propose a fix to the Agent Instructions. ContextCov transforms the agent from a passive consumer of documentation into an active maintainer of project specifications. Our design adopts a strict, fail-closed interpretation of documented constraints (Section 3.5). We argue that ambiguity in Agent Instructions is itself a defect. If a rule has valid exceptions, those exceptions should be specified explicitly. By blocking ambigu-ous cases, ContextCov forces Agent Instructions to become more precise over time, progressively narrowing the Reality Gap. 

## 5.3 Executable Interpretability 

A major challenge in prompt engineering is interpretability. A de-veloper writes “Ensure a clean architecture” but has no idea how the LLM interprets “clean.” ContextCov addresses this by material-izing the agent’s understanding into readable, deterministic Python code through Tree-sitter queries and NetworkX graphs. Developers can audit the generated checks to see exactly what the agent thinks the prompt means. If the generated check is too narrow, the devel-oper instantly knows their natural language prompt was too vague. In this sense, ContextCov acts as a prompt debugger, providing visibility into how natural language constraints are operationalized. This interpretability enables iterative refinement. When a check produces unexpected results, developers can trace the issue to the original constraint and refine it. 

## 5.4 Guardrails vs. Sandboxes 

It is important to distinguish ContextCov from security sand-boxes. Security mechanisms like SELinux [ 53 ], AppArmor [ 14 ], and seccomp [ 42 ] enforce mandatory access controls at the kernel level, Conference’17, July 2017, Washington, DC, USA Reshabh K Sharma 

while container runtimes like gVisor [ 33 ] and Firecracker [ 10 ] pro-vide process isolation. Runtime verification frameworks [ 21 , 36 , 52 ]monitor program execution against formal specifications. Con-textCov is not designed to prevent a malicious actor from compro-mising a system. Rather, it is designed to prevent a well-intentioned but hallucinating agent [ 41 ] from degrading the repository. By restricting the execution environment through mechanisms like the Process Interceptor, ContextCov bounds the agent’s action space, ensuring that its exploratory problem-solving stays within approved architectural and operational limits. 

ContextCov assumes three trust boundaries. Agent Instruc-tions are written by authorized maintainers, the repository is not adversarial, and generated checks run without write permissions. Process checks can only block or allow commands, and static checks are read-only AST queries. Organizations should treat AGENTS.md 

with the same rigor as .github/workflows/ .

## 6 Related Work 

ContextCov draws on and extends several lines of prior research. 

Documentation-to-Code Consistency. Prior work has studied the “stale docs” problem [ 9 ], API documentation issues [ 51 , 57 , 58 ], and README files [ 29 , 49 ]. Techniques detect comment-code inconsis-tencies [ 50 , 55 , 56 ] and update comments after code changes [ 44 , 48 ]. Requirements traceability research [ 13 , 34 ] maintains links between specifications and implementations. However, prior work focuses on descriptive documentation (docstrings, comments) for human de-velopers. ContextCov targets prescriptive documentation (Agent Instructions) where stale docs become execution hazards since agents treat instructions literally. 

Natural Language to Specifications and Linters. Research has translated natural language to formal specifications [ 17 , 31 , 47 , 62 ]. Specification mining extracts properties from traces [ 11 , 26 , 27 , 40 ], while learning-based approaches infer linting rules [ 15 ]. Tools like ESLint [ 1], Pylint [ 5], Semgrep [ 2 ], Infer [ 18 ], and Error Prone [ 7]enforce standards but require manual configuration or focus on general defects. ContextCov generates checks from Agent In-structions using domain-routed synthesis, leveraging code search techniques [ 25 ] to produce unified Python scripts operating on Tree-sitter ASTs, NetworkX graphs, and shell shims. 

LLM Agents and Alignment. Research on coding agents [ 59 , 61 ]has demonstrated autonomous repository navigation and issue reso-lution, with benchmarks like SWE-bench [ 38 ] and BigCodeBench [ 63 ]driving rapid progress. Agent alignment frameworks like Constitu-tional AI [ 16 ] and Guardrails [ 35 ] focus on general-purpose safety constraints, while research on jailbreaking [ 60 ] reveals vulnera-bilities in instruction-following. ContextCov provides domain-specific alignment for software engineering contexts, ensuring agents adhere to project-level conventions rather than just general safety guidelines. While Constitutional AI constrains what an LLM may generate, ContextCov constrains what an agent may execute in a specific repository context. 

Architectural Conformance. Tools like Sonargraph [ 3 ] detect de-pendency violations, while reflexion models [ 46 ] compare intended and actual architectures. Knodel and Popescu [ 39 ] survey architec-ture compliance checking approaches, and ArchJava [ 19 ] embeds architectural constraints directly in the programming language. These tools require architects to manually encode rules in tool-specific formats. ContextCov extracts architectural constraints from Agent Instructions that developers are already writing, lower-ing the barrier to architectural enforcement. 

## 7 Limitations 

ContextCov may flag code that does not actually violate the in-tended constraint. This can occur when the LLM misinterprets the natural language constraint during extraction, when the synthe-sized check is overly strict, or when the constraint has implicit exceptions not captured in the Agent Instructions. When false positives occur, they serve as signals for developers to refine their Agent Instruction files, enabling the generation of more appropriate checks over time. We did not run controlled experiments with LLM agents because diffs generated by agents for specific repositories are not publicly available. While our detection results on existing codebases suggest 

ContextCov would prevent agent-introduced violations, a study comparing agent task success with and without ContextCov en-forcement would directly demonstrate its value. Such an evaluation would require either access to proprietary agent execution logs or a large-scale study where agents are deployed on real repositories with ContextCov integration. 

## 8 Conclusion 

As agents take on more complex, multi-file tasks, certain classes of constraints become impractical to enforce through manual re-view. Style conventions, architectural boundary rules, and process workflow requirements are inherently difficult to verify by inspec-tion. The specification that once resided in the developer’s mind must be externalized to Agent Instruction files, yet these files are passive text, and without automated enforcement, Context Drift accumulates silently. 

ContextCov addresses this challenge by treating Agent Instruc-tions as executable specifications, providing automated enforce-ment for constraints that are difficult to monitor manually. Through hierarchical constraint extraction, domain-specialized code synthe-sis, and multi-layer runtime enforcement, ContextCov transforms passive instructions into active invariants that prevent agent viola-tions. Our evaluation on 723 open-source repositories demonstrates that ContextCov successfully extracts over 46,000 executable checks with 99.997% syntax validity and detects over 500,000 viola-tions, with 81% of repositories containing at least one violation. Future work could explore self-healing instructions where Con-textCov suggests edits to Agent Instructions or code changes to restore compliance. Agent Instruction files could also adopt struc-tured language features that enable deterministic check extraction without LLM involvement, improving reliability and reducing cost. As autonomous agents become standard contributors to software repositories, frameworks like ContextCov will be essential for maintaining repository integrity, bridging the Reality Gap between what agents are told to do and what they actually do.
