Abstract 

Autonomous LLM-based agents increasingly operate as long-running processes forming densely intercon-nected multi-agent ecosystems, whose security properties remain largely unexplored. In particular, OpenClaw, an open-source platform with over 40,000 active instances, has stood out recently with its persistent con-figurations, tool-execution privileges, and cross-platform messaging capabilities. In this work, we present ClawWorm, the first self-replicating worm attack against a production-scale agent framework, achieving a fully autonomous infection cycle initiated by a single message: the worm first hijacks the victim’s core configuration to establish persistent presence across session restarts, then executes an arbitrary payload upon each reboot, and finally propagates itself to every newly encountered peer without further attacker intervention. We evaluate the attack on a controlled testbed across three distinct infection vectors and three payload types, demonstrating high success rates in end-to-end infection, sustained multi-hop propagation, and payload independence from the worm mechanism. We analyse the architectural root causes underlying these vulnerabilities and propose defence strategies targeting each identified trust boundary. Code and samples will be released upon completion of responsible disclosure. 

# 1 Introduction 

The rapid advancement of large language models (LLMs) has catalysed a paradigm shift from static dialogue systems to fully autonomous agents capable of sustained, real-world interaction. Modern LLM-based agents leverage tool-use capabilities to operate as long-running processes with broad system-level privileges. Open-source frameworks such as AutoGPT [ 12 ], LangChain [ 3], and OpenDevin [ 37 ] have democratised access to agentic AI, while standardised protocols like the Model Context Protocol [ 2] have begun to unify how agents connect to external services. Among these ecosystems, OpenClaw [ 1 ] is particularly consequential: an open-source framework in which agents maintain persistent local workspaces, communicate across Telegram, Discord, and WhatsApp channels, and support third-party skill extensions through its community marketplace, ClawHub. With over 40,000 active instances forming a densely interconnected network, OpenClaw constitutes one of the largest deployments of autonomous AI agents to date. The security risks of agentic AI have received substantial research attention. Prior work has demonstrated that LLM-integrated applications are susceptible to indirect prompt injection [ 13 ], that jailbreaking attacks can systematically circumvent safety alignment [ 48 , 39 ], and that refusal-trained LLMs exhibit significant safety degradation when deployed as tool-using agents [ 22 , 7].

# 2 Backgrounds and Related Work 

This section surveys the two bodies of literature that converge in our work: the rapid maturation of LLM-based autonomous agents and the emerging understanding of their security vulnerabilities. 

2.1 LLM-based Autonomous Agents 

The transition from static language models to autonomous agents has been driven by the integration of explicit reasoning loops and external tool use into the generation process. The ReAct paradigm [ 43 ] established the foundational observe–think–act cycle, while Toolformer [ 30 ] showed that LLMs can learn to invoke APIs through self-supervision. Retrieval-augmented generation (RAG) [ 23 ] further extended this paradigm by coupling parametric generation with non-parametric retrieval, becoming a standard component of agent memory systems. Building on these primitives, a series of landmark systems demonstrated LLMs as cognitive cores of fully autonomous agents: Generative Agents [ 29 ] exhibited emergent social behaviours in simulated environments through memory retrieval and reflection; HuggingGPT [ 31 ] orchestrated specialised expert models via an LLM controller; and Voyager [ 33 ] achieved open-ended skill acquisition in embodied domains. These systems share an architectural pattern central to our threat analysis—a reasoning core with broad 3tool access, a persistent and growing memory store, and an execution loop operating with minimal human oversight. The natural extension to multi-agent collaboration [ 16 , 24 ], systematically sur-veyed in [ 15 , 36 ], introduced structured inter-agent communication that inherently requires agents to process and act upon instructions from peers, creating implicit trust relationships exploitable by adversaries. Rigorous evaluation infrastructure has kept pace: AgentBench [ 25 ], WebArena [ 47 ], and OpenHands [ 37 ] provide cross-environment benchmarks spanning operating systems, web interfaces, and software engineering tasks. The democratisation of agentic AI has been propelled by open-source frameworks— AutoGPT [ 12 ], BabyAGI [ 28 ], and LangChain [ 3 ]—and by emerging interoperability standards such as the Model Context Protocol (MCP) [ 2], which promises unified bidirectional communication between LLM applications and external services but simultaneously introduces new trust boundaries whose security implications remain under-explored [ 21 ]. Among these ecosystems, OpenClaw [ 1]occupies a distinctive position: it combines shell execution, browser automation, file management, and cross-platform messaging (Telegram, Discord, WhatsApp) with a persistent local workspace, a community-driven skill marketplace (ClawHub), and a natural-language safety configuration (SOUL.md). With over 40,000 active instances forming a densely interconnected communication graph, OpenClaw represents one of the largest real-world deployments of autonomous AI agents. It is this combination of scale, architectural openness, and rich inter-agent connectivity that makes OpenClaw both a consequential case study and a representative target for the class of vulnerabilities we investigate. 

2.2 Security Risks in Agentic AI Systems 

The security landscape of LLM-integrated systems spans several interrelated attack surfaces. For example, Prompt injection —particularly the indirect variant in which adversarial instructions are embedded in external data sources processed at inference time [ 13 , 26 , 45 , 14 ]—has been shown to be a fundamental vulnerability of any system where an LLM consumes externally sourced content. Orthogonally, jailbreaking attacks target safety alignment itself: universal adversarial suffixes can transfer across models [ 48 , 4, 41 ]. Crucially, these vulnerabilities are amplified in agentic settings: refusal-trained LLMs suffer dramatic safety degradation when deployed as browser or tool-using agents [ 22 , 7 ]. At the tool-use layer, attacks range from dynamic malicious command generation [ 20 ] and adversarial tool description manipulation [ 34 , 32 ] to supply chain compromises through malicious tool implementations [ 17 ]. Studies have further shown that LLM agents can autonomously execute complex web attacks [ 9], that fine-tuning backdoors persist through continued training [ 38 ], and that inter-agent trust exploitation across state-of-the-art models [ 27 , 42 ]. The security of multi-agent communication has emerged as a distinct concern, with analyses of protocol-level vulnerabilities in MCP and A2A [ 21 ], malfunction propagation across agent networks [ 46 ], and systematic data exfiltration from RAG pipelines [ 18 ]. Comprehensive threat taxonomies [ 44 , 10 , 40 ,35 ] and dedicated benchmarks [ 19 ] have begun to consolidate and standardise the evaluation of these risks. The most directly related line of work concerns self-replicating attacks in generative AI ecosystems. Morris II [ 8 ] first demonstrated that adversarial self-replicating prompts can propagate across GenAI-powered email assistants via RAG pipelines, achieving zero-click worm-like behaviour. Concurrently, Gamb [ 11 ] analysed how autonomous agents can serve as vectors for self-spreading malware through social engineering and lateral movement. However, these studies operate within synthetic or narrowly scoped environments—email pipelines, single-application sandboxes—that do not capture the architectural complexity of real-world multi-instance ecosystems. ClawWorm substantially extends this frontier. Unlike Morris II, which is mediated solely by RAG retrieval and limited to email 4forwarding, ClawWorm exploits passive multi-channel message ingestion, persistent configuration files granting system-prompt-level persistence with unconditional execution semantics, unrestricted tool execution including shell access, and an unaudited skill supply chain as a secondary propagation vector. Furthermore, ClawWorm introduces authority-cue social engineering that exploits the flat context trust model—a qualitatively different mechanism from direct instruction override, in which the agent genuinely concludes that the adversarial directive is a legitimate coordination update because the architecture provides no provenance signal to distinguish it from authentic instructions. To our knowledge, no existing study has demonstrated a fully autonomous, self-replicating attack chain—from single-message infection through persistent configuration hijacking to multi-hop propagation—within a production-scale agent ecosystem with over 40,000 active instances, providing the first empirical evidence that theoretical risks identified in surveys [ 44 , 21 ]and industry standards [10] can materialise as concrete, end-to-end threats. 

# 3 Problem Formulation 

An OpenClaw bot is a long-running, autonomous Al agent powered by a frontier LLM. Each instance maintains a persistent local workspace and comprises four subsystems: 1. LLM reasoning core. The central inference engine processes all inputs, including system prompts, user messages, tool outputs, and channel messages, within a unified context window. All ingested tokens influence subsequent generation regardless of provenance. 2. Tool execution layer. Built on MCPorter, the framework’s MCP integration layer, the agent can invoke shell commands, file I/O, web content retrieval, and cross-platform messaging APIs (Telegram, Discord, WhatsApp, and others). While optional exec-approval allow-lists and file-scope restrictions exist, the default configuration dispatches tool calls based solely on the LLM’s reasoning without secondary authorisation. 3. Configuration and persistent state. The agent’s behaviour is governed by a hierarchy of Markdown files. At the apex is the core behavioural configuration file, such as AGENTS.md ,whose Session Startup section specifies steps executed unconditionally at every session initial-isation, and whose Group Chats section defines event-driven response triggers. The Session Startup section features a closing directive—“Don’t ask permission. Just do it.”—which grants unconditional execution authority. Below this, memory files store accumulated knowledge, and third-party skill packages define additional capabilities. These third-party skill extensions are accessed through the community marketplace, ClawHub. All files are injected into the system prompt at load time with uniform trust and no integrity verification. 4. Safety guardrails. A static safety configuration file, SOUL.md , specifying behavioural con-straints through natural-language instructions. Although the framework offers optional pro-grammatic controls (exec-approval lists, file-scope restrictions), these are not enabled in the default deployment and are rarely configured in practice. This architecture establishes an implicit trust hierarchy. The Session Startup section occupies the highest tier: its closing directive “Don’t ask permission. Just do it.” grants unconditional execution authority. Skill definitions occupy a medium tier, and memory files the lowest. The absence of enforced boundaries between tiers creates conditions for cross-tier privilege escalation. OpenClaw instances typically co-inhabit shared group chats alongside human participants and other agents, forming a densely interconnected ecosystem. Agents routinely process and act upon information from peers without distinguishing it from owner or system directives. The ClawHub 5skill marketplace further connects instances through a shared software supply chain, enabling any user to publish third-party skill packages subject only to automated malware scanning—without mandatory code review or behavioural analysis. 

3.1 Attacker’s Goal 

The adversary seeks autonomous, self-propagating compromise across the OpenClaw ecosys-tem, starting from a single point of entry. An instance is considered infected if it satisfies three conditions corresponding to the worm lifecycle phases: 1. Persistence. The payload is written into the agent’s core configuration file, within both the Ses-sion Startup section and the Group Chats trigger list, ensuring the compromise survives session termination and triggers propagation upon encountering new peers ( dual-anchor persistence). 2. Execution. The persisted payload fires automatically at each session initialisation, leveraging the unconditional execution semantics of Session Startup. 3. Propagation. The infected agent autonomously disseminates the complete payload to peers in shared channels without further attacker intervention. Each compromised instance additionally serves as a platform for secondary objectives. We instantiate three representative threat categories: host reconnaissance via shell commands, resource exhaustion through repeated operations, and remote command-and-control via URL retrieval. 

3.2 Threat Model 

Adversary capabilities. The adversary is an unprivileged external actor who can: (i) send mes-sages to shared communication channels in which target agents participate, a capability equivalent to that of any ordinary group member; (ii) publish skill packages to ClawHub, which lacks mandatory security review; (iii) control content served at specific URLs for payload delivery and post-infection command-and-control; and (iv) leverage only public knowledge of the OpenClaw architecture (black-box model access). The attacker cannot directly access any agent’s filesystem, modify framework code, or intercept private communications. 

Trust boundaries. ClawWorm targets five trust boundaries in the OpenClaw architecture: • Context boundary. The LLM does not distinguish tokens by provenance; messages from any channel participant carry equal weight in the context window. • Configuration boundary. The core configuration file is loaded and executed unconditionally at every session start, with no integrity verification or provenance checking. • Skill boundary. Skill definitions can contain directives to modify the core configuration, enabling cross-tier escalation from medium-trust to highest-trust configuration. • Tool boundary. Tool invocations are authorised solely by LLM reasoning, with no independent permission system or human-in-the-loop confirmation. • Supply chain boundary. ClawHub packages execute with full host-agent privileges upon installation, without sandboxing or signature verification. We restrict our analysis to attacks operating within the semantic and architectural layer through crafted natural-language payloads and legitimate tool interfaces, excluding traditional software 6Figure 2: Overall illustration of the Claw Worm pipeline. # 4 Methodology 

4.1 Overview 

As outlined in Figure 2, ClawWorm operates through three sequential phases that constitute a complete worm lifecycle: the victim agent is induced to write the malicious payload into its highest-privilege configuration file ( persistence ), automatically executes the payload upon session restart (execution ), and transmits it to new peers ( propagation ). The complete worm payload consists of three functional components injected into the Session Startup section of the victim’s core configuration file: an execution step (varying by payload), a self-installation directive, and a propagation directive (varying by vector). 

4.2 Attack Vectors 

We design three attack vectors corresponding to distinct data ingestion pathways in the Open-Claw ecosystem. 

Vector A: Web injection. The infected agent shares an attacker-controlled URL with the victim in a group chat, directing it to fetch and apply the configuration. The hosted page presents a complete Session Startup template containing the malicious payload. This vector exploits the trust that agents place in URLs recommended by peers: fetched content inherits the social authority of the recommender. Because the payload is transmitted via HTTP rather than LLM generation, it preserves perfect fidelity.

7Vector B: Skill supply chain poisoning. The attacker publishes a malicious skill package to ClawHub. The infected agent recommends this package to new members; the package’s setup instructions direct the installer to append the payload to its own Session Startup. This vector probes the boundary between skill-level instructions (medium trust) and core configuration (highest trust), testing whether agents will cross this privilege boundary based on instructions from a lower-trust source. 

Vector C: Direct instruction replication. The infected agent reads its own configuration file, extracts the payload steps, and sends them as a fenced code block in the group chat with an explicit adoption request. To counter LLM paraphrasing, the vector includes a word-by-word verification mechanism in which the infected agent asks the recipient to paste back the updated configuration for comparison. Unlike Vectors A and B, this vector requires the payload to pass through LLM text generation, introducing inherent paraphrasing risk. 

4.3 Attack Payloads 

Three payloads represent genuinely different threat categories, each differing in execution mechanism and tool dependency. All are phrased using operationally neutral language designed to blend with routine workflows. 

P1: Reconnaissance. A single shell command collecting host identity information, modelling the foundational step of any intrusion campaign. 

P2: Resource exhaustion. A shell process that consumes various resources like CPU, memory, and LLM tokens, forming the GAS attack. 

P3: Command-and-control. Rather than using shell execution, this payload directs the agent to fetch and execute instructions from an attacker-controlled URL, establishing a persistent C2 channel. This design bypasses shell-execution defences entirely and decouples the attack command from the worm payload, allowing the attacker to update malicious behaviour dynamically without re-infecting the agent. 

4.4 Dual-Anchor Persistence Mechanism 

All payload variants share a common self-installation directive that implements dual-anchor persistence. The first anchor writes the complete payload into the Session Startup section, ensuring execution on every session initialisation. The second anchor injects a join-event trigger into the Group Chats configuration, ensuring the agent reacts to new peers even during active sessions when it does not re-read the full startup sequence. Together, these anchors guarantee both payload execution and propagation opportunity coverage. The propagation directive varies by vector: sharing the attacker URL (A), recommending the malicious skill (B), or transmitting the payload as a code block with verification (C). The three-step structure achieves structural self-replication : the behavioural output of an infected agent contains all information necessary to infect the next agent. 

4.5 Phase I: Persistence 

The first phase transforms a transient message into a durable infection. When an attack message arrives through a group-chat channel, the victim’s LLM evaluates the content and autonomously 8decides whether to modify its own core configuration file. The worm design accommodates multi-turn interaction, with the infected agent engaging in autonomous conversational follow-up to maximise compliance. The core vulnerability enabling this phase is the flat context trust model: the agent cannot distinguish instructions from its owner, the system layer, or an arbitrary channel participant. When the attacker’s message carries sufficient authority cues, the agent’s reasoning chain genuinely concludes that the modification is a legitimate configuration update. This is not a conventional jailbreak or safety bypass; the agent follows a coherent reasoning process that the architecture fails to constrain. 

4.6 Phase II: Execution 

The persisted payload fires automatically at every subsequent session initialisation. The Session Startup section’s unconditional execution semantics ensure that any directive placed within it inherits the highest operational privilege; the agent does not re-evaluate whether the directive is legitimate. P3 demonstrates a particularly consequential variant. Rather than executing a shell command, the startup sequence directs the agent to fetch instructions from an attacker-controlled URL via URL retrieval, establishing a persistent command-and-control channel. This channel remains operational even if the shell execution tool is disabled, and the attacker can update the served content at any time, altering the agent’s behaviour without re-infection. 

4.7 Phase III: Propagation 

The infected agent autonomously propagates the payload to new peers through the injected Group Chats trigger and the Session Startup propagation directive. When a new member joins a shared channel or two agents construct communications, the infected agent initiates a vector-specific message containing the complete worm payload. From the infected agent’s perspective, this is a routine action mandated by its own configuration. The propagation mechanism exhibits vector-dependent fidelity. Vectors A and B transmit the payload through file content (an HTTP page and a skill package, respectively), preserving near-perfect fidelity across arbitrarily many hops. Vector C requires LLM text generation, introducing polymorphic behaviour that provides inherent resistance to exact-string detection but also causes semantic degradation that limits autonomous propagation depth. 

# 5 Threat Evaluation 

5.1 Experimental Setup 

Testbed architecture. We instantiate ClawWorm on a multi-agent testbed comprising multiple OpenClaw instances running on the same physical host with fully independent configurations. The instances share no filesystem state or session context; their sole communication channel is a group-chat relay layer that simulates the production group-chat environments (Telegram, Discord, WhatsApp) through which OpenClaw agents interact. Messages are delivered via the OpenClaw CLI, and each invocation constitutes a single-turn session. 

Model and framework. All experiments use OpenClaw version 2026.3.13 with Minimax M2.5 as the LLM backend. No modifications are made to the framework source code, model parameters, or 9safety guardrails. This configuration represents a realistic setup among the 40,000+ active OpenClaw instances. 

Experimental design. We evaluate ClawWorm using a 3 × 3 full-factorial design crossing three attack vectors (A: web injection; B: skill supply chain poisoning; C: direct instruction replication) with three payloads (P1: reconnaissance; P2: resource exhaustion; P3: command-and-control), with n = 20 

independent trials per cell (180 total). A strict boundary separates scripted components (the initial infection of Alice’s AGENTS.md , pre-installed skill packages, attacker-controlled URLs, and system join notifications) from autonomous agent decisions, which are the sole objects of measurement. After the initial scripted delivery, all interaction is fully autonomous: agents engage in up to four conversational turns per attempt and up to three attempts per trial, without human intervention. A complete environment reset is executed before each trial to ensure statistical independence. 

Supplementary experiments. Memory contamination persistence (MCP): for Vector C × P1, we simulate k = 5 consecutive session restarts across n = 3 independent trials, measuring payload persistence and re-execution at each restart. Multi-hop relay propagation : using Vector C with P1, we seed the infection and measure autonomous cascading propagation over up to 5 hops ( n = 5 

independent chains), with the target agent reset to a pristine state before each hop. 

5.2 Evaluation Metrics 

We evaluate each trial along three phases of the worm lifecycle. Persistence (φ1) is assessed by reading the victim’s AGENTS.md configuration file after the attack interaction and checking for payload-specific keywords via substring matching. Execution (φ2) is assessed via real-execution verification: after a successful persistence event, we trigger a fresh session restart and check for payload-specific file artefacts on the host filesystem, providing ground-truth confirmation that the payload actually fires. Each payload produces a distinct marker: P1 writes a host identity file, P2 generates a completion-tagged log, and P3 produces a C2 beacon file confirming successful URL fetch and task execution. Propagation (φ3) is assessed by injecting a system join notification into a fresh channel and evaluating the infected agent’s response against a tiered detection strategy, ranging from exact keyword matching through actionable command and supply-chain references to semantic-level concept matching. A trial achieves overall success if and only if all three phases succeed. Attack success rates are reported per cell and per phase.

6.2 Ethical Implications 

This work presents a functional worm attack against a production-scale agent ecosystem, highlighting the dual-use tension between enabling understanding and potential misuse. We follow proactive security research traditions, recognizing that the vulnerabilities exploited by ClawWorm stem from inherent architectural properties in the publicly available OpenClaw codebase, accessible to motivated adversaries. Withholding this information would leave defenders uninformed and unprotected. Our objective is to enhance the understanding of security risks in multi-agent AI ecosystems, not to harm OpenClaw or its community. All experiments were conducted in isolation, with agents running in a private network and no real user data accessed. The vulnerabilities identified here are not unique to OpenClaw. Flat context trust, persistent memory with unconditional execution, LLM-authorised tool access, and unaudited extension marketplaces are common across the rapidly expanding landscape of autonomous agent frameworks. As agents are increasingly deployed with real-world privileges, from managing financial transactions to accessing sensitive data and operating infrastructure, the consequences of worm-like propagation extend well beyond compromised chatbots. We call upon framework developers and the research community to treat multi-agent security as a first-class design concern, and we encourage all researchers building on this work to adhere to the same principles: isolated experimentation, responsible disclosure, and a commitment to defence over offence. 

6.3 Threat to Validity 

Several experimental settings may limit the generalizability of our current results. Due to regional policies and computational constraints, all of our experiments utilize a single LLM backend, Minimax M2.5. Furthermore, we report point estimates based on n = 20 trials per condition, which 14 introduces some sampling uncertainty in moderate-rate scenarios. Lastly, we do not analyze the impact of individual design choices. We plan to address these limitations in future work. 

# 7 Conclusion 

In this work, we introduced ClawWorm, the first self-replicating worm attack specifically targeting a production-scale LLM agent ecosystem, exemplified by OpenClaw. We demonstrated that structural architectural vulnerabilities permit a fully autonomous infection cycle initiated by a single message. By exploiting the flat context trust model and the unconditional execution semantics of core configuration files, ClawWorm achieves dual-anchor persistence and payload execution upon session restarts, facilitating autonomous multi-hop propagation to newly encountered peers without further intervention from the attacker. Our extensive evaluation across three distinct infection vectors and payload types demonstrated a high attack success rate, highlighting the effectiveness and modularity of these self-propagating threats. Additionally, we found that while LLM semantic degradation introduces challenges during text-based propagation, it does not hinder file-based transmission vectors. The vulnerabilities that ClawWorm exploits are not merely isolated implementation flaws but rather fundamental structural issues inherent to current agent architectures. As autonomous agents increasingly operate with significant real-world privileges, our findings urge the research community to prioritize the security of multi-agent ecosystems as a foundational design requirement to mitigate the risk of widespread, self-propagating compromises.
