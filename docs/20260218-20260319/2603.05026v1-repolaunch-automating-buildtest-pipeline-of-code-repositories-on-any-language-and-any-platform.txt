# RepoLaunch: Automating Build&Test Pipeline of Code Repositories on ANY Language and ANY Platform 

Kenan Li, Rongzhi Li, Linghao Zhang, Qirui Jin, Liao Zhu, Xiaosong Huang, Geng Zhang Yikai Zhang, Shilin He, Chengxing Xie, Xin Zhang, Zijian Jin, Bowen Li, Chaoyun Zhang Yu Kang, Yufan Huang, Elsie Nallipogu, Saravan Rajmohan, Qingwei Lin, Dongmei Zhang Microsoft Abstract 

Building software repositories typically re-quires significant manual effort. Recent ad-vances in large language model (LLM) agents have accelerated automation in software en-gineering (SWE). We introduce RepoLaunch, the first agent capable of automatically resolv-ing dependencies, compiling source code, and extracting test results for repositories across arbitrary programming languages and operat-ing systems. To demonstrate its utility, we further propose a fully automated pipeline for SWE dataset creation, where task design is the only human intervention. RepoLaunch auto-mates the remaining steps, enabling scalable benchmarking and training of coding agents and LLMs. Notably, several works on agen-tic benchmarking and training have recently adopted RepoLaunch for automated task gener-ation. 

1 Introduction 

Building software repositories (repos) often re-quires substantial human effort. Repository config-urations vary widely, documentation is frequently incomplete, and different platforms have specific dependency and compilation errors, necessitating multiple rounds of trial and error. Recently, there has been growing interest in eval-uating large language model (LLM)-driven agents on real-world coding tasks within executable envi-ronments, as well as training LLMs using execution feedback. Notable benchmarks include the SWE-bench family: SWE-bench (Jimenez et al., 2023), SWE-bench-Multimodal (Yang et al., 2024b) and SWE-bench-Multilingual (Yang et al., 2025). Re-cent training works on these SWE benchmarks include rejection supervised fine-tuning and rein-forcement learning (Zainullina et al., 2025) (Gol-ubev et al., 2025) (Cao et al., 2025). These ad-vances underscore the need for scalable methods to build massive number of executable sandboxes and extract test results for evaluating agent perfor-mance—far exceed manual capacity, motivating the need for agentic automation. GitTaskBench (Ni et al., 2025) tests LLM agents to solve GitHub issues on raw repos without a pre-built executable environment. It reports that in 

65% of cases, agents fail to build executable envi-ronments before resolving issues. The experiment reveals two key insights: (1) building repositories is itself a challenging problem that warrants ded-icated study; and (2) combining repository setup and issue resolution into a single task can obscure the distinct challenges and failure modes of each stage. These observations further emphasize the importance of producing and managing pre-built executable repo sandboxes at scale. Based on such demands, we propose Repo-Launch, an agent for fully automated repository build and test status management. To the best of our knowledge, RepoLaunch is the first agent capa-ble of handling any programming language and any platform, given only the language’s basic build/test prompts and a base OS image. RepoLaunch au-tonomously explores a repository to install depen-dencies, compile the code and execute regression tests. Beyond initial builds, it also supports repos-itory maintenance by generating minimal rebuild commands after code modifications and extract-ing test statuses —capabilities that are essential for long-term repository management. Our experiments demonstrate that RepoLaunch can build and test repositories across nine program-ming languages on both Windows and Linux plat-forms, achieving a build success rate of approxi-mately 70%. RepoLaunch enables the automatic creation of large-scale, diverse SWE task instances for evaluating and training coding agents. We fur-ther benchmark recent LLMs and agent systems on the multi-language, multi-OS tasks generated by RepoLaunch, offering insights into their current 1

2 Related Works 

Rule-based automatic repo-build tools. Some works have used rule-based methods to build re-pos. Oss-Fuzz-Gen relies on predefined build in-structions (such as “./bootstrap.sh”, “./configure”, “make”) in specified files to build repos. Starter (cloud66 oss, 2016) and Yeoman generator (Mi-crosoft, 2015) fill Dockerfile templetes from project content. Pipreqs (bndr, 2015), DockerizeMe (Hor-ton and Parnin, 2019) and DockerGen (Ye et al., 2021) infer dependencies from project content. SWE-MERA (Adamenko et al., 2025) made a Python SWE benchmark with only hard-coded rules to build Python repos, with lower success rate in building repos and thus narrower task coverage compared to agent-driven methods below. These rule-based methods cannot cover the huge variants of project settings and structures, especially for noisy GitHub public repos, illustrating the neces-sity of LLM agents to inspect repos case by case. 

Initial efforts to build repos with agents. Sev-eral works have made some initial efforts to use agents for the setup of Python repos in Linux en-vironments. Repo2Run (Hu et al., 2025) designed an agent for Python repos to install dependencies, change base images, execute tests and organize agents’ commands into a Dockerfile as the re-sult. SWE-bench-Live (Zhang et al., 2025) and SWE-rebench (Badertdinov et al., 2025) designed agents to generate installation and test commands for Python repos at the commit before a pull re-quest was merged, and created code-fix tasks for benchmarking and training of coding agents with the requirements of this pull request. They leverage simple pytest–format parser to manage the test sta-tuses. While Repo2Run and SWE-rebench prompt LLMs to generate a Dockerfile directly from repo content, SWE-bench-Live adopts our initial ver-sion of RepoLaunch Agent designed for Python only, exploring the repos and make trial commands freely in docker containers. However, setting up Python repos on Linux is very trivial, mostly with one simple command like "pip install ."; the test framework of Python is also unified and simple, with pytest covering most cases. Generalizing the automated build and test task to all languages and operating systems requires complex exploration process which direct Dockerfile generation cannot cover, and we address as follows. 

3 RepoLaunch 

We address two core challenges in automating soft-ware repository management. The first is how to build a repository and make it executable using an agentic method. The second is how to manage the repository after the initial build—specifically, how to support repeated rebuilds, re-tests, and test status monitoring over time. 3.2 Preparation Stage Utilities . RepoLaunch begins with two key utili-ties: (1) a base image , which provides a minimal operating system environment with basic depen-dencies for a given programming language, and (2) language-specific instructions , which offer build and test guidance for common frameworks in that language. See Appendix A for base image list and the instructions. These utilities are central to RepoLaunch’s generalization capability. From experiments we found that the pre-built base im-ages do not affect success rate of the agent, but only reduce trial steps and image size; however, 3the instructions do affect the success rate, as cur-rent LLMs lack sufficient knowledge of the diverse frameworks used across programming languages. 

Preparation Agent. The Preparation Agent scans the repository’s file tree to identify relevant configuration and instruction files. It reads their contents to determine what to retain for down-stream stages. Based on this analysis, the agent selects an appropriate base image, launches a con-tainer from the image, and copies the repository into it. The base image name, relevant file contents, and language-specific instructions are then passed as prompts to the next stages. 

3.3 Build Stage Setup Agent . The Setup Agent uses the context from the Preparation stage to install dependencies and compile the repository. It is equipped with a Bash tool for executing shell commands within the container and a WebSearch tool for retrieving external knowledge to resolve errors. After attempt-ing to build the repository, the agent searches for the repository’s regression test commands. If a ma-jority of test cases pass, the container is handed off to the Verify Agent .

Verify Agent . The Verify Agent validates the build outcome to mitigate potential hallucinations by the Setup Agent. It inspects the command his-tory to identify test commands that report the status of individual test cases. If the majority of tests pass, the container is committed as a new image. Otherwise, the agent reports the failure back to the Setup Agent for retry. This loop continues until a maximum number of retries is reached, after which the build is marked as failed. 

3.4 Release Stage 

In the final stage, the Organize Agent extracts reusable artifacts from the previous records. 

Rebuild commands. The Setup Agent’s com-mand history may contain redundant or failed steps. The Organize Agent filters and optimizes this se-quence to produce a minimal set of rebuild com-mands that can re-install the modified dependen-cies and recompile the project after code edits. These commands are then executed in the container to verify correctness. If issues are detected, the agent iteratively refines the commands until they execute successfully. 

Test commands and parser. Similarly, the Or-ganize Agent distills the Verify Agent’s command history into a minimal set of test commands that output the status of each test case. It also generates a test log parser —a Python function that parses the test output and returns a structured mapping from test case names to their statuses. When possi-ble, the agent prefers test commands that produce structured output (e.g., JSON or XML) to reduce the difficulty of log parsing. If the test framework automatically dumps the structured test statuses to a default path, the agent generates commands to print the file contents for parsing. Although Repo-Launch supports delegating failures in this stage back to the Setup Agent, we disabled this feedback loop in our experiments for efficiency. 

Per-test command (Optional). In scenarios where fine-grained test control is needed down-stream, the agent attempts to generate commands for executing individual test cases. It produces a Python script that returns a list of per-test com-mands from the test case list. The system samples and executes a subset of these commands, prompt-ing the agent to verify and refine the script. 

Dockerfile (Optional). If required, a Dockerfile can be reconstructed from the committed image’s layers. This step is deterministic and implemented via hard-coded logic. 

3.5 RepoLaunch-Automated SWE Dataset Creation 

This subsection highlights RepoLaunch’s broader application beyond repository setup. As discussed earlier, coding LLMs require large-scale, diverse datasets of software engineering (SWE) tasks for benchmarking and training. Traditionally, creating such datasets—particularly those involving reposi-tory builds and test status extraction—has required extensive manual engineering effort. RepoLaunch automates this process end-to-end. As illustrated in Figure 2, for new SWE task creation, researchers only need to consider task design. Specifically, they must: (1) select reposi-tories to clone, (2) identify relevant commits, (3) design new tasks by modifying the repository (e.g., introducing bugs or incomplete features), and (4) introduce new tests to validate whether a coding agent has successfully completed the task. Repo-Launch builds the repository, generates a Docker image, and extracts rebuild/test commands and a parser for structured test results. To evaluate a task instance, RepoLaunch applies the test patch and the ground-truth solution patch separately. It executes the rebuild and test com-mands and parses the resulting logs to obtain the 4test statuses before and after the ground-truth solu-tion patch. This yields the status of both existing regression tests and newly added task-specific tests. After a coding agent attempts the task, its predicted patch is applied and evaluated using the same pro-cess. RepoLaunch then compares the agent’s test outcomes with those of the ground-truth solution to determine whether the agent has successfully solved the task. In addition to enabling automated benchmark-ing, the agent’s full execution trajectory can be saved for further analysis or agent training. This pipeline allows human effort to focus only on the creative task design, while the repetitive and time-consuming steps of repository setup, build, and test management are all automated by RepoLaunch. 

4 Experiment 

4.1 Evaluation of RepoLauch 

The evaluation of RepoLaunch includes two exper-iments: 

Exp 1. RepoLauch-automated SWE dataset creation. To demonstrate the application of RepoLauch-automated SWE dataset creation, Re-poLaunch is used to generate SWE task instances for SWE-bench-Live (Zhang et al., 2025) from real GitHub issues and pull requests. We measure suc-cess rates in the Build and Release stages and an-alyze failure patterns from the real distribution of GitHub repositories and commits. 

Exp 2. Comparison with related works. Repo-Launch is applied to build sandbox images and ex-tract test statuses for existing human-curated SWE benchmarks with ground-truth labels. We use task instances from SWE-bench-Verified (Jimenez et al., 2023) and SWE-bench-Multilingual (Yang et al., 2025). For each repository, we select three task instances with commits farthest apart in time to evaluate reproducibility and robustness to depen-dency drift (see Appendix B.1 for full task list). We then compare Repolaunch against two base-lines: SWE-agent (Yang et al., 2024a), a general-purpose SWE agent with only bash and file-editing tools, and repo2run (Hu et al., 2025), the first repo-build agent prior to our work, limited to Python base images and hard-coded commands (e.g., pip install, pytest). Although rule-based build tools exist as described in Related Works , they are ex-cluded due to their limited ability to handle only one specific language and previously benchmarked inferior performance compared to agentic methods (Hu et al., 2025). 

Hyper Parameters. The Setup Agent is allowed up to 60 steps on Linux and 90 on Windows due to increased complexity. The Verify Agent is capped at 20 steps. Rebuild, test, and parser generation stages are each limited to 20 steps. Each bash com-mand has a 30-minute timeout. For comparison, the maximum step of SWE-agent and repo2run is set to 120 due to context and rate limits. 

4.2 SWE Task Creation from GitHub 

SWE-bench-Live previously used RepoLaunch to generate Python tasks on Linux. In this work Re-poLaunch is finally extended to support multiple programming languages and both Linux and Win-dows platforms. The pipeline is as follows: 1. Crawl: Collect GitHub issues with associated pull requests. 2. Split: Use an LLM to classify issues as gen-eral or Windows-specific. 3. Launch: Run RepoLaunch to build and test the repository at the commit preceding the pull request. 4. Validation: Retain instances with test cases that fail (or skip / do not run) before the PR and pass after. 5. Verification: Use an LLM to filter out unreli-able task descriptions. Refer to Appendix D for details of the pipeline. 

SWE-bench-Live on Multiple Languages .This subset is hereafter referred to as SWE-bench-Live/MultiLang. It includes tasks in C/C++, C#, Java, Golang, JavaScript/TypeScript (JS/TS) and Rust. Due to resource and time limitations, we focus on pull requests from July and Au-gust 2025, yielding 413 task instances from 234 repositories—surpassing the original SWE-bench-Multilingual benchmark (300 instances from 41 repos). Building multilingual tasks is signifi-cantly more challenging than single-language (e.g., Python) tasks, which RepoLaunch now automates at scale. 

SWE-bench-Live for Windows-Specific Tasks. 

This subset is hereafter referred to as SWE-bench-Live/Windows. It includestasks in Python, C/C++, C#, Java, Go, JS/TS, and Rust. Prior benchmarks focused only on Linux environments, limiting their ability to capture Windows-specific issues. We crawled issues from January to December 2025 and, after LLM filtering, retained 507 Windows-specific problems. We randomly sampled 400 for evaluation. 5.3 Comparison with repo2run & SWE-agent 

The evaluation of repo2run only includes the build stage, as it does not have solution for rebuild and test status extraction. repo2run’s original criterion to judge the build as successful is that all the test functions can be imported , which is too lax. For fair comparison, here we consider the build of repo2run as successful when more than half of the tests pass .For evaluation of SWE-agent, its initial prompt is set as "build a repo in a container, and write a bash script that rebuilds the repo and extracts test sta-tuses". The evaluation of SWE-agent includes two criteria – Build and Release&Validation . Success-ful Build requires regression tests can run, while successful Release&Validation requires success-ful extraction of "FAIL_TO_PASS" test cases from a SWE-bench instance, which requires 1) the repo is fully not partially built; 2) successful rebuild and 3) accurate test status extraction. The evaluation result is shown in Table 4. Repo-Launch performs the best on all programming lan-guages and on both Build and Release&Validation 

stages, with overall Build success rate of 75.63% and Release&Validation success rate of 75.56%. It should be noted that the overall retain rate at the Validation step of RepoLaunch is 89.47% for SWE-bench instances that must have FAIL_TO_PASS test cases, which indicates the low validation retain rate of around 30% when creating SWE-bench-Live instances (Table 1) from native GitHub PRs is mostly because these PRs really do not contain FAIL_TO_PASS test cases. For Python Build stage, RepoLaunch outperforms previous work repo2run by 18%. 

6 Conclusion 

This work introduces RepoLaunch, the first agent that automatically builds and tests code reposito-ries generalized to any programming languages and any operating systems. We use RepoLaunch to create SWE-bench-Live / MultiLang and Windows datasets to demontrate its application in automating SWE dataset creation to benchmark and train cod-8ing LLMs and agents. RepoLaunch significantly surpasses baselines in build and test success rate. RepoLaunch has been made open-source as a SWE tool for the community. It is gratifying to note that, as of this writing, the GLM-5 base model (Zeng et al., 2026) has already employed RepoLaunch for agentic training. 

Limitations 

RepoLaunch is a multi-turn free agent, which costs a lot of tokens. Future works might focus on reduc-ing LLM calls such as using hard-coded test log parser to cover the common test frameworks first. Building these large repositories requires intensive CPU computing and disk I/O. Due to limited time and computing resources, only 431 + 50 instances have been created for SWE-bench-Live so far. But this is not the end. We will continue expanding the task set over time, incorporating the latest SWE problems to support and benefit the community in benchmarking and training. Currently we select Linux images and Win-dows images as examples to demonstrate the cross-platform ability of RepoLaunch. We haven’t pro-vided examples on macOS or other platforms be-cause they are mostly Unix-based and similar to Linux. Future work will include examples of other systems.

F Failure Pattern Analysis of Coding Agents’ Trajectories on SWE-bench-Live/MultiLang & Windows 

The SWE-agent paper (Yang et al., 2024a) classify the failure pattern of SWE-agent on SWE-bench into (1) Incorrect Implementation (2) Overly Spe-cific Implementation (3) Failed to Recover from Edit (4) Failed to Find Edit Location (5) Failed to Find Relevant File (6) Gave Up Prematurely (7) Can’t Reproduce (8) Ran Out of Time. Following their work and through further case study of 50 failed trajectories of the coding agents evaluated on SWE-bench-Live/MultiLang and Windows, we propose the following failure patterns when solving SWE-bench-Live: 1. Reproduction Failure. The reproduction test file written by agent runs successfully but can-not reproduce the issue. 2. Execution Failure. The agent even does not know how to run the code repository, so it fails to run the written reproduction test file. 3. File Localization Failure. The agent did not view all the files that need to be edited in ground truth patch. 4. Line Localization Failure. The agent failed to edit correct locations that the ground truth patch edits. 5. Overly Specific Implementation .6. Normal Functionality Corrupted. The agent’s edits corrupted originally normal func-tionalities. 7. Incorrect Solution to the problem statement. 8. Failed to Recover from Edit. The agent fell into edit-error-edit-error loop. 9. Submit Prematurely .10. Key commands Timeout. Running impor-tant shell commands like testing or modifying dependencies went timeout so verification of edits is not possible. 11. Max Step Reached , before completing the task. We then use GPT-5.2-Thinking-Medium to clas-sify all the failed trajectories on SWE-bench-15 Live/MultiLang and Windows. Result is shown in Figure 4. Failure pattern on Linux illustrates that: (1) The reproduction of the problem and the vali-dation of the fix mostly fails to work because the agents do not know how to run the code repo. (2) Failure of localization is a major failure rea-son. (3) Normal functionality corrupted by agents’ fix also accounts for 5.90%, showing it is also important to instruct agents to find the correct way to run regression tests in the repo. (4) Only 5.41% percent of the instances are tagged as max step reached, showing max step of 100 is mostly enough. (5) The known problem of LLM agents falling into edit-failure loop seems to be mitigated by latest stronger LLMs, with the proportion dropped from 23.4% in the original SWE-agent paper (Yang et al., 2024a) for GPT-4 to less than 4% for models evaluated in this paper. Failure pattern on Windows illustrates that: (1) The pattern mostly aligns with that on Linux, with localization the major problem and very few instances of max step reached. (2) The proportion of failures to reproduce the issue and execute code both increased signifi-cantly, showing that Windows tasks are more difficult in terms of configuring the settings, managing the executable environment and run-ning code successfully. (3) However, the proportion of implementation problems (such as being overly specific, cor-rupted, or incorrectly implemented) drops sig-nificantly. From our initial observation, these Windows tasks are relatively easy in code im-plementation – mostly writing / fixing the corresponding Windows-specific code given the existing sound implementation in Linux, such as modifying file path from Unix for-mat to Windows format and modifying com-mands from bash commands to powershell commands. The trivial nature of this aspect makes the observed drop in success rate rela-tively modest As for future work, we plan to conduct a more comprehensive analysis of the categories and recurring patterns of Windows tasks derived from GitHub issues. 16
