## 摘要
现代分布式系统集成了异构服务、具有不同模式（schema）版本的 REST API、GraphQL 端点以及具有专有负载的物联网（IoT）设备，这些系统长期受到模式不匹配问题的困扰。传统的静态适配器需要为每一对模式进行手动编码，且无法在运行时处理新的组合。我们提出了 SAGAI-MID，这是一种基于 FastAPI 的中间件，利用大语言模型（LLM）在运行时动态检测并解决模式不匹配问题。该系统采用五层流水线：混合检测（结构差异对比加 LLM 语义分析）、双重解决策略（单次请求 LLM 转换和 LLM 生成的可重用适配器代码）以及三层保障栈（验证、集成投票、基于规则的备选方案）。我们通过 Bass 等人的互操作性策略构建架构，将其从设计时的人工产物转变为运行时能力。我们在涵盖 REST 版本迁移、IoT 到分析平台的桥接以及 GraphQL 协议转换的 10 个互操作性场景中，对来自两个供应商的 6 个 LLM 进行了评估。性能最佳的配置达到了 0.90 的 pass@1 准确率。CODEGEN 策略的表现始终优于 DIRECT 策略（平均 pass@1 为 0.83 对比 0.77），而不同模型之间的成本差异超过 30 倍，且准确率并未随之成比例增长；最准确的模型反而是最便宜的。我们讨论了软件架构师将 LLM 作为运行时架构组件所带来的启示。

## Abstract
Modern distributed systems integrate heterogeneous services, REST APIs with different schema versions, GraphQL endpoints, and IoT devices with proprietary payloads that suffer from persistent schema mismatches. Traditional static adapters require manual coding for every schema pair and cannot handle novel combinations at runtime. We present SAGAI-MID, a FastAPI-based middleware that uses large language models (LLMs) to dynamically detect and resolve schema mismatches at runtime. The system employs a five-layer pipeline: hybrid detection (structural diff plus LLM semantic analysis), dual resolution strategies (per-request LLM transformation and LLM-generated reusable adapter code), and a three-tier safeguard stack (validation, ensemble voting, rule-based fallback). We frame the architecture through Bass et al.'s interoperability tactics, transforming them from design-time artifacts into runtime capabilities. We evaluate SAGAI-MID on 10 interoperability scenarios spanning REST version migration, IoT-to-analytics bridging, and GraphQL protocol conversion across six LLMs from two providers. The best-performing configuration achieves 0.90 pass@1 accuracy. The CODEGEN strategy consistently outperforms DIRECT (0.83 vs 0.77 mean pass@1), while cost varies by over 30x across models with no proportional accuracy gain; the most accurate model is also the cheapest. We discuss implications for software architects adopting LLMs as runtime architectural components.

*   **核心痛点**：传统的互操作性解决方案依赖于静态适配器（Static Adapters），需要开发人员为每一对模式手动编写转换代码。这种方式不仅耗时耗力，且无法处理运行时出现的未知模式组合，导致系统集成非常脆弱。
*   **论文目标**：提出一种名为 **SAGAI-MID** 的中间件，利用大语言模型（LLM）在运行时动态检测并解决模式不匹配问题，将传统的“设计时”适配转变为“运行时”能力。

#### 2. 方法论
SAGAI-MID 是一个基于 FastAPI 的 ASGI 中间件，其核心是一个**五层流水线架构**：
*   **输入层（Input Layer）**：拦截 HTTP 请求，通过路由匹配确定源模式（Source Schema）和目标模式（Target Schema）。
*   **检测层（Detection Module）**：采用**混合检测**机制。首先通过确定性算法进行结构差异分析（Structural Diff），随后利用 LLM 进行语义分析（识别命名语义和单位不匹配）。
*   **解析层（Resolution Engine）**：提供两种策略：
    *   **DIRECT 策略**：对每个请求实时调用 LLM 进行数据转换。
    *   **CODEGEN 策略**：利用 LLM 生成 Python 适配器代码，编译并缓存。后续请求直接运行本地代码，无需再调用 LLM。
*   **保障层（Safeguard Layer）**：应对 LLM 的非确定性，设有三级防御：
    1.  **验证**：使用 Pydantic 和 JSON Schema 校验输出。
    2.  **集成投票（Ensemble Voting）**：验证失败时，运行多个 LLM 实例并取多数票。
    3.  **规则兜底（Fallback）**：若 AI 彻底失效，使用基于模糊匹配和预设规则的确定性算法进行“尽力而为”的转换。
*   **监控层（Monitoring）**：记录延迟、成本、Token 消耗和准确率指标。

*   **实验规模**：每个模型在 10 个场景下分别运行 DIRECT 和 CODEGEN 策略，并开启/关闭保障机制，每组组合重复 3 次。总计进行了数百次实验调用。

#### 6. 主要结论与发现
*   **CODEGEN 胜过 DIRECT**：CODEGEN 策略的平均 pass@1 准确率（0.83）高于 DIRECT（0.77），且在复杂场景下优势更明显。

#### 7. 优点
*   **理论结合实践**：将经典的软件架构互操作性策略（Bass et al.）与现代生成式 AI 结合，提供了清晰的理论映射。
*   **创新的 CODEGEN 策略**：通过生成代码并缓存，有效解决了 LLM 运行时的延迟和高昂成本问题，使其具备工业化潜力。
*   **防御性设计**：三层保障栈的设计充分考虑了 AI 的不可靠性，体现了严谨的软件工程思维。
