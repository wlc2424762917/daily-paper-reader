## 摘要
基于 ReAct 的工具调用型大语言模型（LLM）自主智能体存在三个局限性：串行延迟、上下文二次方增长，以及易受提示词注入和幻觉的影响。近期研究趋向于将规划与执行分离，但在各种情况下，模型仍与执行机制耦合。我们为 LLM 智能体引入了一种系统级抽象，将智能体工作流的执行与 LLM 推理层解耦。我们定义了两个核心抽象：(1) 意图门控执行 (IGX)，一种在执行时强制执行意图的安全范式；(2) 执行内核，负责管理调度、工具分发、依赖解析、故障处理和安全。在 KAIJU 中，LLM 预先进行规划，通过依赖感知的参数注入来乐观地并行调度工具。工具通过 IGX 基于四个独立变量进行授权：范围、意图、影响和许可（外部审批）。KAIJU 支持三种自适应执行模式（Reflect、nReflect 和 Orchestrator），提供适用于复杂调查、深度分析或研究的渐进式细粒度执行控制。针对 ReAct 基准的实验评估表明，由于规划开销，KAIJU 在简单查询上存在延迟劣势，在复杂度适中时性能趋于一致，而在需要并行数据采集的计算查询上具有结构性优势。除了延迟之外，这种分离还强制执行了仅靠提示词无法实现的行为保证。代码开源于 https://github.com/compdeep/kaiju

## Abstract
Tool-calling autonomous agents based on large language models using ReAct exhibit three limitations: serial latency, quadratic context growth, and vulnerability to prompt injection and hallucination. Recent work moves towards separating planning from execution but in each case the model remains coupled to the execution mechanics. We introduce a system-level abstraction for LLM agents which decouples the execution of agent workflows from the LLM reasoning layer. We define two first-class abstractions: (1) Intent-Gated Execution (IGX), a security paradigm that enforces intent at execution, and (2) an Executive Kernel that manages scheduling, tool dispatch, dependency resolution, failures and security. In KAIJU, the LLM plans upfront, optimistically scheduling tools in parallel with dependency-aware parameter injection. Tools are authorised via IGX based on four independent variables: scope, intent, impact, and clearance (external approval). KAIJU supports three adaptive execution modes (Reflect, nReflect, and Orchestrator), providing progressively finer-grained execution control apt for complex investigation and deep analysis or research. Empirical evaluation against a ReAct baseline shows that KAIJU has a latency penalty on simple queries due to planning overhead, convergence at moderate complexity, and a structural advantage on computational queries requiring parallel data gathering. Beyond latency, the separation enforces behavioural guarantees that ReAct cannot match through prompting alone. Code available at https://github.com/compdeep/kaiju

#### 1. 核心问题与研究动机
传统的基于 ReAct（推理-行动）架构的 LLM 智能体在处理复杂任务时存在三个核心局限性：
*   **串行延迟：** 必须等待上一个工具执行完并观察结果后才能进行下一步推理，导致响应缓慢。
*   **上下文二次方增长：** 每一轮对话都要携带之前所有工具的执行结果，导致 Token 消耗呈 $O(n^2k)$ 增长，容易超出上下文窗口并降低模型注意力。
*   **安全性脆弱：** 安全约束主要依赖提示词（Prompt），容易受到提示词注入攻击或模型幻觉的影响，且攻击者可以通过观察报错信息进行自适应攻击。

#### 2. 方法论与核心技术
KAIJU 提出了**推理层与执行层解耦**的系统级抽象，核心组件包括：
*   **执行内核（Executive Kernel）：** 负责管理调度、工具分发、依赖解析、故障恢复和安全。它将任务转化为**有向无环图（DAG）**，支持并行执行。
*   **意图门控执行（IGX）：** 一种确定性的安全范式，通过四个独立变量进行授权：
    1.  **范围（Scope）：** 允许调用的工具白名单。
    2.  **意图（Intent）：** 调用者的操作权限等级（由系统而非 LLM 设定）。
    3.  **影响（Impact）：** 工具本身的风险等级（只读、写入、删除等）。
    4.  **许可（Clearance）：** 委托给外部 HTTP 端点的资源级授权。
*   **参数注入机制：** 允许在规划阶段定义数据流（param_refs），在执行阶段动态注入上游节点的输出。
*   **三种自适应执行模式：**
    *   **Reflect：** 在依赖波次（Waves）之间进行反思和重新规划。
    *   **nReflect：** 每完成 N 个节点进行一次反思。
    *   **Orchestrator：** 为每个节点配备轻量级观察者，实现最高频的实时反馈。

*   **Token 效率：** KAIJU 将 Token 复杂度从 $O(n^2k)$ 降低到 $O(nk)$，有效避免了长任务中的上下文崩溃。
*   **可靠性：** 在天文学基准测试中，KAIJU 完成率为 100%，而 ReAct 为 80%（因上下文溢出或中途放弃）。

#### 7. 优点与亮点
*   **结构化解耦：** 首次将 LLM 智能体的执行逻辑从推理循环中彻底剥离，类似于操作系统的内核抽象。
*   **并行调度：** 通过 DAG 和参数引用机制，实现了比 ReAct 更高效的工具并行化。
*   **防御深度：** IGX 的四变量模型提供了比单纯提示词过滤更强的确定性安全保障。
*   **自适应性：** 反思循环（Reflection Loop）允许系统在规划不足时动态修正，兼顾了预规划的效率和运行时的灵活性。

#### 8. 不足与局限
*   **规划开销：** 对于只需 1-2 步的简单查询，预规划增加了不必要的延迟。
*   **规划器依赖：** 系统的表现高度依赖于初始规划器的分解能力，若规划器严重失误，会增加反思轮数和成本。
*   **动态影响分类：** 对于通用工具（如 Shell）的影响等级判定仍依赖模式匹配，存在一定的误判风险。
*   **应用限制：** 目前主要针对工具调用型任务，对于纯对话或创意写作类任务提升有限。
