## 摘要
由大语言模型（LLM）驱动的编程智能体日益依赖于模型上下文协议（MCP）等工具调用协议，在开发者的工作站上执行文件读写。当写入因内容过滤、截断或会话中断而失败时，智能体通常无法接收到结构化信号，导致草稿丢失，并因盲目重试而浪费 Token。我们提出了 Resilient Write，这是一个 MCP 服务端，它在智能体与文件系统之间介入了一个六层持久化写入界面。这六个层级——预检风险评分、事务性原子写入、断点续传安全分块、结构化类型错误、带外暂存盘存储以及任务连续性移交信封——彼此正交且可独立采用。每一层都对应于 2026 年 4 月一次真实智能体会话中观察到的具体故障模式，在该会话中，内容安全过滤器静默拒绝了包含脱敏 API 密钥前缀的草稿。在使用该系统撰写本文的过程中，还衍生出了分块预览、格式感知验证和日志分析这三个额外工具。包含 186 个测试的测试套件验证了每一层的正确性；与朴素基准和防御性基准的定量对比显示，该系统使恢复时间缩短了 5 倍，智能体自纠错率提高了 13 倍。Resilient Write 已在 MIT 许可证下开源。

## Abstract
LLM-powered coding agents increasingly rely on tool-use protocols such as the Model Context Protocol~(MCP) to read and write files on a developer's workstation. When a write fails -- due to content filters, truncation, or an interrupted session -- the agent typically receives no structured signal, loses the draft, and wastes tokens retrying blindly. We present \textbf{Resilient Write}, an MCP server that interposes a six-layer durable write surface between the agent and the filesystem. The layers -- pre-flight risk scoring, transactional atomic writes, resume-safe chunking, structured typed errors, out-of-band scratchpad storage, and task-continuity handoff envelopes -- are orthogonal and independently adoptable. Each layer maps to a concrete failure mode observed during a real agent session in April~2026, in which content-safety filters silently rejected a draft containing redacted API-key prefixes. Three additional tools -- chunk preview, format-aware validation, and journal analytics -- emerged from using the system to compose this paper. A 186-test suite validates correctness at each layer, and quantitative comparison against naive and defensive baselines shows a 5x reduction in recovery time and a 13x improvement in agent self-correction rate. Resilient Write is open-source under the MIT license.

*   **研究背景**：当写入失败时，智能体通常收不到明确信号，导致其丢失已生成的草稿，并陷入盲目重试的死循环，浪费大量 Token 和时间。
*   **整体含义**：本文提出在智能体与文件系统之间建立一个“持久化写入界面”，通过六层防御机制确保写入过程可预测、可恢复且具备审计能力。

### 2. 方法论：核心思想与关键技术
Resilient Write 作为一个 MCP 服务器，采用了**六层正交架构**，每一层解决特定的故障模式：
*   **L0: 预检风险评分 (Risk Scoring)**：在写入前使用确定性分类器（正则匹配）扫描内容。通过公式计算风险分 $s$，识别 API 密钥、JWT、私钥等敏感模式，若分数过高则提前预警，避免被下游过滤器静默拦截。
*   **L1: 事务化原子写入 (Safe Write)**：采用“临时文件 -> fsync 同步 -> 哈希校验 -> 原子重命名”的流程，确保文件要么完整写入，要么完全不改动，防止产生损坏的半截文件。
*   **L2: 断点续传分块 (Chunking)**：支持将大文件拆分为编号分块。如果中间某个分块失败，只需重试该分块，最后通过 `chunk_compose` 合并并校验完整性。
*   **L3: 结构化类型错误 (Error Envelopes)**：将模糊的字符串错误替换为 JSON 格式的结构化错误包，包含错误类型（如 `blocked`, `stale_precondition`）、重试预算和改进建议。
*   **L4: 内容寻址暂存盘 (Scratchpad)**：提供带外存储，用于存放不宜进入代码库的敏感数据或二进制大对象，通过 SHA-256 哈希进行去重。
*   **L5: 任务接力信封 (Handoff)**：在 `HANDOFF.md` 中记录任务状态、已完成文件的哈希和后续步骤，允许新会话在中断处无缝恢复。

### 8. 不足与局限
*   **单工作区限制**：目前仅支持绑定到单个工作区根目录，缺乏多工作区协同能力。
*   **缺乏跨文件事务**：虽然单个文件写入是原子的，但不支持多个文件同时写入的“全成功或全失败”（Rollback）。
*   **本地化局限**：日志和暂存盘均存储在本地，不支持分布式环境下的状态同步。
*   **分类器覆盖面**：L0 层的正则匹配可能无法覆盖所有新型的敏感信息格式，需要持续维护规则库。
