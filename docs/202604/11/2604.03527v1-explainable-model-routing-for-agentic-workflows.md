## 摘要
现代智能体工作流将复杂任务分解为专门的子任务，并将其路由到不同的模型，以在不牺牲质量的前提下最小化成本。然而，当前的路由架构仅专注于性能优化，导致模型能力与成本之间的潜在权衡未被记录。由于缺乏清晰的依据，开发人员无法区分“智能效率”（即针对合适任务使用专门模型）与由预算驱动的模型选择所导致的潜在失败。我们提出了 Topaz，这是一个为智能体路由引入正式可审计性的框架。Topaz 用一个具有内在可解释性的路由器取代了静默的模型分配，该路由器包含三个组件：(i) 基于技能的画像，将不同基准测试中的性能综合为细粒度的能力画像；(ii) 完全可追溯的路由算法，利用基于预算的多目标优化，生成关于技能匹配分数如何与成本进行权衡的清晰追踪记录；(iii) 面向开发者的解释，将这些追踪记录转化为自然语言，允许用户审计系统逻辑并迭代调整成本与质量的权衡。通过使路由决策具有可解释性，Topaz 使用户能够理解、信任并有效地引导路由化的智能体系统。

## Abstract
Modern agentic workflows decompose complex tasks into specialized subtasks and route them to diverse models to minimize cost without sacrificing quality. However, current routing architectures focus exclusively on performance optimization, leaving underlying trade-offs between model capability and cost unrecorded. Without clear rationale, developers cannot distinguish between intelligent efficiency -- using specialized models for appropriate tasks -- and latent failures caused by budget-driven model selection. We present Topaz, a framework that introduces formal auditability to agentic routing. Topaz replaces silent model assignments with an inherently interpretable router that incorporates three components: (i) skill-based profiling that synthesizes performance across diverse benchmarks into granular capability profiles (ii) fully traceable routing algorithms that utilize budget-based and multi-objective optimization to produce clear traces of how skill-match scores were weighed against costs, and (iii) developer-facing explanations that translate these traces into natural language, allowing users to audit system logic and iteratively tune the cost-quality tradeoff. By making routing decisions interpretable, Topaz enables users to understand, trust, and meaningfully steer routed agentic systems.

### 2. 论文提出的方法论
Topaz 框架包含三个核心阶段：
*   **基于技能的画像（Skill-Based Profiling）**：
    *   定义了一个包含 8 种技能（数学、逻辑、代码、工具使用、事实知识、写作质量、指令遵循、总结）的分类法。
    *   **模型画像**：通过 LLM 分析公开基准测试（Benchmarks）的描述，将得分分解为细粒度的技能分数。
    *   **任务画像**：对工作流中的子任务进行复杂度、技能需求、Token 长度和质量敏感度（$q_t$）的评估。
*   **可追溯的路由算法**：
    *   **目标导向路由（Objective-based）**：通过加权权衡质量（技能匹配度）和成本。公式考虑了局部质量敏感度和全局成本敏感度（$c_{global}$），为每个任务独立选择最优模型。
    *   **预算导向路由（Budget-based）**：在给定总预算 $B$ 的情况下，使用**动态规划（DP）**算法在子任务序列中寻找整体质量最大化的模型组合。
*   **解释生成（Explanation Generation）**：
    *   系统记录路由过程中的中间数值（如技能匹配分、成本惩罚）。
    *   利用 LLM（如 Gemini 3.0 Flash）将这些数值轨迹转化为自然语言，解释为何选择该模型（例如：“因为该任务逻辑复杂度高，虽然 Claude 贵，但其逻辑技能匹配度远超其他模型”）。

### 6. 主要结论与发现
*   **路由灵活性**：Topaz 能根据开发者的偏好平滑切换策略。在高性能模式下倾向于 Claude/Gemini Pro，在极端省钱模式下则能精准识别出哪些简单任务（如工单分类）可以安全地交给 Mistral。
*   **解释的忠实性**：生成的自然语言解释直接源于数值轨迹（Match Score 和 Cost Penalty），避免了传统 XAI 常见的“事后编造”问题。
*   **关键发现**：即使在极高成本压力下，Topaz 也会优先保留核心复杂任务（如技术诊断）的高质量模型分配，而牺牲非核心任务的质量，这证明了其“智能效率”。

### 7. 优点
*   **内在可解释性**：不同于事后解释（Post-hoc），Topaz 的解释是基于决策逻辑的直接翻译，具有更高的忠实度。
*   **细粒度控制**：引入了“技能画像”而非单一的性能分，使路由能针对特定能力（如代码或数学）进行优化。
*   **闭环反馈**：允许开发者通过调整质量敏感度（$q_t$）来迭代优化路由逻辑，具有很强的实用性。

### 8. 不足与局限
*   **画像依赖性**：路由效果高度依赖于 LLM 对基准测试和任务的初始画像评估，如果画像评估有误，路由决策也会随之偏差。
*   **动态性限制**：目前的成本估算基于预估的 Token 长度，而 LLM 的实际输出长度具有随机性，可能导致实际支出超出预算。
*   **下游行为解释缺失**：Topaz 解释了“为什么选这个模型”，但无法解释“选中的模型为什么给出了这个错误答案”，即它只管路由层，不管推理层。
*   **校准因子的人工干预**：文中提到的校准因子 $\kappa$ 需要手动调优，这在自动化部署中可能成为瓶颈。
