## 摘要
Android 应用严重依赖数据操作功能 (DMFs) 通过 CRUDS 操作处理应用特定数据，因此其正确性对可靠性至关重要。然而，检测数据操作错误 (DMEs) 具有挑战性，因为它们依赖于特定的 UI 交互序列，且表现为逻辑漏洞。现有的自动化 UI 测试工具面临两个主要挑战：UI 路径覆盖不足导致无法充分触发 DMF，以及依赖手动编写的测试脚本。为了解决这些问题，我们提出了一种利用大语言模型 (LLMs) 进行 DME 检测的自动化方法。我们开发了 LDMDroid，一个针对 Android 应用的自动化 UI 测试框架。LDMDroid 通过引导 LLMs 执行状态感知过程来生成 UI 事件序列，从而提高了 DMF 的触发成功率。它还利用视觉特征来识别数据状态的变化，提高了 DME 验证的准确性。我们在 24 个真实世界的 Android 应用上评估了 LDMDroid，结果表明其 DMF 触发成功率优于基准方法。LDMDroid 发现了 17 个独特的漏洞，其中 14 个已得到开发者确认，11 个已修复。该工具已在 https://github.com/runnnnnner200/LDMDroid 公开发布。

## Abstract
Android apps rely heavily on Data Manipulation Functionalities (DMFs) for handling app-specific data through CRUDS operations, making their correctness vital for reliability. However, detecting Data Manipulation Errors (DMEs) is challenging due to their dependence on specific UI interaction sequences and manifestation as logic bugs. Existing automated UI testing tools face two primary challenges: insufficient UI path coverage for adequate DMF triggering and reliance on manually written test scripts. To address these issues, we propose an automated approach using Large Language Models (LLMs) for DME detection. We developed LDMDroid, an automated UI testing framework for Android apps. LDMDroid enhances DMF triggering success by guiding LLMs through a state-aware process for generating UI event sequences. It also uses visual features to identify changes in data states, improving DME verification accuracy. We evaluated LDMDroid on 24 real-world Android apps, demonstrating improved DMF triggering success rates compared to baselines. LDMDroid discovered 17 unique bugs, with 14 confirmed by developers and 11 fixed. The tool is publicly available at https://github.com/runnnnnner200/LDMDroid.

*   **核心问题**：
    1.  **触发难**：DME 往往隐藏在特定的 UI 交互序列中，现有的自动化测试工具（如随机测试或基于规则的测试）难以理解应用语义，无法有效触发复杂的 DMF 路径。

### 2. 方法论：核心思想与技术细节
LDMDroid 框架主要包含以下四个关键阶段：
*   **DUM（受控数据）识别**：
    *   通过算法自动识别 UI 页面中的数据容器。核心逻辑是寻找具有**结构相似性**（Tree Edit Distance）和**位置对齐性**（中心坐标与边界框大小）的 UI 组件簇，并过滤掉静态字符串常量。
*   **状态感知的 DMF 触发**：
    *   **通用步骤引导**：为五种 CRUDS 操作定义了通用的交互步骤（如“打开创建页 -> 输入信息 -> 提交 -> 返回”）。
    *   **UI 变更提取**：在每步操作后，利用 LLM 对比操作前后的 UI 截图，总结 UI 状态的变化（如“点击添加按钮后出现了创建对话框”）。
    *   **任务进度追踪**：将 UI 变更反馈给 LLM，使其感知当前处于通用步骤的哪一阶段，从而规划下一步动作，减少 LLM 的幻觉。
*   **逻辑错误验证（Test Oracle）**：
    *   将 DME 定义为自然语言提示词（Prompt），让 LLM 对比操作前后 DUM（数据容器）的状态变化。例如，在“创建”操作后，验证新条目是否出现在列表中且内容匹配。
*   **交替测试策略**：
    *   将收集到的 DMF 实例与随机生成的 UI 事件交替执行，以模拟真实用户场景并探测深层逻辑漏洞。

*   **自动化 Oracle**：无需人工编写复杂的测试脚本，利用 LLM 对 DUM 状态的语义理解实现了通用的逻辑错误检测。
*   **实用性强**：在真实世界应用中发现了大量深层逻辑漏洞，且已获得开源社区认可。

### 8. 不足与局限
*   **特定 UI 限制**：对于非列表形式的数据展示（如地图节点、绘图画布、单对象页面），DUM 识别算法效果较差。

*   **操作深度**：对于需要极长交互序列或特定前置条件（如复杂的账号注册流程）的 DMF，触发成功率仍有提升空间。
