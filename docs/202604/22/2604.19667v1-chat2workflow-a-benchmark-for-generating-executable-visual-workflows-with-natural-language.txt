Abstract 

At present, executable visual workflows have emerged as a mainstream paradigm in real-world industrial deployments, offering strong reliability and controllability. However, in current practice, such workflows are almost en-tirely constructed through manual engineering: developers must carefully design workflows, write prompts for each step, and repeatedly re-vise the logic as requirements evolve—making development costly, time-consuming, and error-prone. To study whether large language models can automate this multi-round interaction pro-cess, we introduce Chat2Workflow , a bench-mark for generating executable visual work-flows directly from natural language, and pro-pose a robust agentic framework to mitigate recurrent execution errors. Chat2Workflow is built from a large collection of real-world busi-ness workflows, with each instance designed so that the generated workflow can be transformed and directly deployed to practical workflow platforms such as Dify and Coze. Experi-mental results show that while state-of-the-art language models can often capture high-level intent, they struggle to generate correct, stable, and executable workflows, especially under complex or changing requirements. Although our agentic framework yields up to 5.34% resolve rate gains, the remaining real-world gap positions Chat2Workflow as a foundation for advancing industrial-grade automation.

2.3 Dataset Construction 

Figure 3 outlines our construction and evaluation framework. To address the absence of multi-turn workflow datasets, we collected production-level Dify and Coze configurations from official sources and GitHub. Based on these workflows derived from actual production, we reverse-engineered the workflow creation instruction set. To ensure that each round’s instruction is based on the previous ones and may add, modify, or refine workflow behavior, we selectively cluster the workflows in the same task context, then rewrite each of them as a round and the whole as a task. To facilitate subsequent evaluation, the written instructions clearly define the input and output variables and give explicit hints about the use of some key nodes. Ultimately, we obtained 27 tasks as the dataset, with each containing 2 to 4 rounds of instructions. 27 tasks can be further classified into the following six types according to the scenarios: Research, Document, Enterprise, Developer, Education and AIGC. More details are in Appendix A. Although the same task objective can be achieved through different workflows, some specific nodes will inevitably be used, which will be recorded in advance for the corresponding rounds. After the collection and arrangement of the task instructions, we equipped each instruction with three test cases to actually carry out end-to-end testing. For task rounds with relatively clear outputs, the test cases will provide reference answers. For other general cases, only task input is provided. Test cases are primarily derived from existing public datasets, followed by scattered materials from the Internet, and finally synthesized data of language models. The design of test cases takes into account the task requirements of historical rounds, striving to comprehensively cover various situations of multi-branch workflows. 

2.4 Data Selection and Processing 

We selected the 20 most frequently used nodes to fully cover our dataset. To reduce generation complexity, we simplified the nodes’ I/O variable interfaces: the language agent only needs to pro-vide key-value pairs for essential primary variables, while secondary variables default. In order to endow the language agent with the usage of each node and corresponding I/O interfaces, we craft node knowledge base for 20 node types, serving as a crucial part of the system prompt used for understanding the node ecosystem. On the basis of the above principles, we collect the official node documents, select the specified contents as needed for necessary rewriting, and then organize them into a predefined structured form. 

2.5 Quality Control and Dataset Statistics 

For the constructed data, we conduct a manual ver-ification process for every individual input–output pair, ensuring the correctness, completeness, pri-vacy and overall quality of the dataset. During this process, workflows that failed to meet our quality standards or violated predefined rules and compliance criteria are systematically identified and removed. This included cases with ambigu-ous instructions, invalid intermediate steps, time-varying sources, or incoherent workflow behav-iors. After this rigorous filtering and validation process, we curate Chat2Workflow, a high-quality dataset consisting of 27 tasks with 79 multi-turn instructions in total, each accompanied by three test instances (totalling 237 instances) to assess whether the generated workflows adhere to the specified execution requirements. 

2.6 Evaluation Protocol 

Unlike general generation tasks such as text and images, the quality of a workflow cannot be judged entirely based on its representation form itself. It is essential to check whether it can be executed normally and what the execution outcome is. Therefore, we adopted a two-stage progressive evaluation method. The first stage aims at the workflow representation form itself, checking whether it is consistent with CoT sequence and whether the generated workflow JSON can be converted into a valid YAML file. The second stage involves verifying whether the workflow can be successfully imported into the platforms, correctly executed, and produce results that satisfy the specified requirements. We introduce two metrics, pass rate and resolve rate , corresponding to the first and second stages respectively.

3 Experiments 

3.1 Experimental Setup 

To thoroughly assess workflow generation capabil-ities, we validate 15 representative models on the Chat2Workflow benchmark, reporting the average results across three independent runs: 1) Four closed-source LMs: GPT-{5.1, 5.2}, Claude-Sonnet-4.5, and Gemini-3-Pro-Preview. 2) Eleven open-source LMs: Qwen-3-{8B, 14B, 32B}, Qwen3-235B-A22B, Qwen3-Coder-480B-A35B-Instruct, GLM-{4.6,4.7}, DeepSeek-{V3.1,V3.2}, Kimi-K2-{Instruct, Thinking}. Furthermore, to explore the performance upper bound, we select two representative models (GPT-5.1 and GPT-5.2) for an advanced agentic evalua-tion, which is detailed in Section 3.5. The experiments are conducted on the 1.9.2 version of Dify. More details are in Appendix B.

Table 1 presents our experimental results, which we use to analyze the following research questions. 

Q1: What is the gap between the workflow pass rate and resolve rate? For each kind of task, we show both the pass rate %Pas. and resolve rate %Res. , representing the format legality and practical usability of workflows respectively. It can be observed that the resolve rate of all models is lower than the pass rate, with the largest disparity observed in GLM-4.6, reaching an average difference of 20.96%. Although Kimi-K2-Instruct shows the lowest gap of 5.21%, both of its absolute values are at a low level compared to the highest visible resolve rate of 71.59%. In addition, in Developer scenario, it has also shown a significant decline. In terms of specific task scenarios, the gap of GLM-4.6 in Education task scenarios has reached an astonishing 43.44%. We conclude that a legal workflow representation that meets the format requirements is still far from successful execution. The former merely serves as a fundamental and necessary condition, but not a sufficient one for achieving the latter, which often has the phenomenon of inflated scores. 

Q2: To what extent do existing LLM agents approach the capabilities of real workflow de-sign experts? Since the seed dataset is collected from typical workflows commmonly used by peo-ple, together with the mechanism for creating executable workflows through interactive dialogues and the requirements for using files as input and output, our benchmark can to a large extent repre-sent the real-world scenarios. Considering whether the workflow can solve the ultimate problem only, Gemini-3-Pro-Preview performs the best on average, with its absolute performance of 71.59%, which is still far from the level of a practical and deployable workflow planner. Given the limited number of node types considered, real-world workflow creation is inherently more complex; consequently, Gemini-3-Pro-Preview is supposed to show inadequate performance on complex workflows involving a larger set of nodes.

3.3 Analysis of User Requirement Changes from Chat Interactions 

Each task goes through 2 to 4 rounds of dialogue, and each round is regarded as a subtask. The work-flow generation requirements for each round will be based on historical conversations, with additions, deletions or modifications made to the existing structure, which poses a further challenge to the model’s ability of complex task understanding and long-horizon instruction following. To prevent skewed representations—as only 4 tasks contain four dialogue rounds—we illustrate the trend of the two evaluation metrics over the first three rounds for each model, where each data point is the average of three experimental runs (Figure 4). It can be intuitively observed from the table that for most models, the quality of the generated workflow shows a stable downward trend as the number of interaction rounds increases, which corroborates the challenges mentioned above. The slight improvement observed for GPT-5.2 model from the second to the third round is due to some tasks with only two interaction rounds, which affect the second round but not the third. Furthermore, it can be observed that in most models, a common phenomenon occurs that as the rounds progress, the absolute value of the slope corresponding to the broken line gradually decreases. It might be explained as diminishing marginal returns. The workflows that can still be legally parsed and successfully executed in the second round have already undergone one screening, and they possess a certain ability to understand changes in workflow requirements. With more rounds of modifications, the rate of performance degradation would be reduced, reflect-ing the distinction between qualitative changes and cumulative quantitative effects.

4 Related work 

4.1 LLM-based Agents 

The rise of LLM has greatly promoted the develop-ment of AGI, providing a capability foundation for LLM-centered AI agents. Since LLM-based agents can fully leverage the powerful task understanding capabilities of LLM, they are widely used to solve complex real-world problems with the aid of general tools (Schick et al., 2023; Patil et al., 2024), collaborative frameworks (Hong et al., 2024; Qian et al., 2023), and extensive API repositories (Qin et al., 2024; Tang et al., 2023; Zhang et al., 2024a; Yang et al., 2024). Many prompt-based methods (Wei et al., 2022; Yao et al., 2022; Zhu et al., 2026) are proven effective for improving performance. However, these agent methods or frameworks mainly focus on end-to-end effects, paying little attention to the requirements and norms of the intermediate processes in solving tasks, such as planning and reasoning paths, which is not conducive to stable and reliable execution, as well as the reproduction of results across various domains (Ge et al., 2024; Xie et al., 2024; Liu et al., 2024; Yang et al., 2026; Liang et al., 2026). 

4.2 Workflow and Language Agent Planning 

Serving as an intermediate state, workflow provides a way to bridge the gap between task goals and the specific executable steps. On the one hand, by breaking down the task into several executable atomic steps and organizing them into execution plans based on formal logical relationships (van der Aalst, 1997; Dijkman et al., 2008), the workflow achieves complete transparency of the reasoning path, effectively enhancing the interpretability and reliability of the results. On the other hand, the introduction of prior organizational structures makes it possible to construct the logic of complex tasks using established patterns (van der Aalst et al., 2003; Commoner et al., 1971) and robotic process automation (Ivanˇ ci´ c et al., 2019; Hofmann et al., 2020). Early workflow relied on meticulous manual design to avoid the hallucinations. How-ever, hand-crafted workflows are time-consuming and labor-intensive. Therefore, many studies have shifted their focus to using LLM for automated workflow generation (Shen et al., 2023a; Zeng et al., 2023), including iterative synthesis (Zhang et al., 2024b; Li et al., 2024) and knowledge-augmented planning frameworks (Ye et al., 2023; Zhu et al., 2024; Wornow et al., 2024; Huang et al., 2024). More related work can be found in Appendix C. 

5 Conclusion 

We introduce Chat2Workflow, a benchmark for evaluating LLMs’ ability to generate deployable visual workflows from natural language, and show that even when augmented with an agentic frame-work, current models remain fragile under struc-tural constraints and requirement changes. We hope Chat2Workflow can provide both a realistic testbed and a concrete target for future research on automated workflow engineering, and will help bridge the gap between language model-driven workflow design and industrial deployment. 

Limitations 

Despite its contributions, Chat2Workflow has sev-eral limitations. First, while the high-quality dataset is manually verified , the current scale may not encompass the near-infinite variety of logic found in complex industrial business processes. Second, we simplified node interfaces to prior-itize executability, which may not fully capture the intricate parameter configurations required in some real-world deployments. Finally, the current system incorporates only 20 high-frequency node types. These specific nodes were selected as they effectively cover most standard workflow scenarios; however, a wide range of valuable tools remain to be incorporated.

The description of the task scenarios is as follows: 

Research includes 5 tasks, which focus on systematically obtaining, understanding, and in-tegrating knowledge from unstructured data. They typically take papers, books, or research topics as input, emphasizing in-depth reading of the content, logical organization, and critical thinking skills. 

Document includes 5 tasks, whose core value lies in transforming raw inputs such as files, images, and tables into structured results that are computable, reusable, and processable downstream. They serve as an important representative of the “AI preprocessing layer” in data processing. 

Enterprise includes 4 tasks, which are targeted at real Enterprise scenarios, with the goal of automating, standardizing, and providing decision support for complex business processes through AI. Their inputs are mostly internal or business-related documents from enterprises. 

Developer includes 3 tasks, which serve scenar-ios like software development, system design, and technical understanding. The core objective is to lower the threshold for technical understanding and development and enhance engineering efficiency. 

Education includes 4 tasks, which revolve around the complete life cycle of “teaching and learning”, with the goal of building a closed-loop system from learning planning, teaching content generation, assessment to feedback. 

AIGC includes 6 tasks, which are positioned at the core of “generating directly usable content or materials from 0 to 1”, emphasizing creative expression and multimodal output capabilities. The input is often highly abstract or brief instructions, while the output is complete, publishable, and consumable content results. 

B Experimental Settings 

Dify platform requires the necessary extensions to be prepared in advance. A set of common models is installed to make the workflows runnable and the re-sults reproducible. Workflows in Dify are natively represented in YAML format. We have developed a code framework for rule-based transformation to convert the JSON format workflow generated by the language agent into valid YAML format. This ensures that workflows can be effectively generated by agents through simplified JSON representation while remaining fully compatible with the Dify execution environment. In addition, the CoT reasoning sequence is uniformly used as the output method. The node types included in the knowledge base are as follows: Start, End, LLM, Question Classifier, Code, Document Extractor, HTTP Request, If-Else, List Operator, Parameter Extractor, Template, Variable Aggregator, Iter-ation, Iteration-Start, Text to Speech, Text to Image, Mermaid Converter, Markdown Exporter, Google Search and Echarts. Specifically, the Question Classifier and Parameter Extractor node types are strictly designated to the Qwen-3-Max (qwen3-max ), whereas the LLM node type is fixed to the Qwen-3-VL-Plus ( qwen3-vl-plus ). Text to Image node type are fixed to use the Z-image-Turbo ( z-image-turbo ) and Text to Speech node type are fixed to use the GPT-4o-Mini-TTS ( gpt-4o-mini-tts ). For Qwen-3-VL-Plus, temperature is 0.7, max_tokens is 32768, other hyperparameters used during decoding are all set to default values. The system prompt and SKILL.md for workflow generation will be integrated into the GitHub repository soon. As for the Agentic Baseline method, the opencode framework we use is version 1.3.17. 

C More Related Work on Automated Workflow Evaluation 

It is crucial to evaluate the quality of the workflow created by LLM. Previous studies have tried to automate workflow evaluation in tool learning scenarios (Li et al., 2023; Shen et al., 2023b) and fine-grained planning tasks (Chen et al., 2024; Ye et al., 2024; Zheng et al., 2024), using metrics to assess format validity and tool consistency. But these works still have some limitations: Firstly, Most of them remain at the stage of abstract expression of the workflow (Qiao et al., 2024), only checking key elements without considering actual execution effects (Guo et al., 2024), struc-tural dependencies (Lal et al., 2024), or platform-specific constraints (Shen et al., 2024). They may identify obvious format errors but cannot ensure usability. Secondly, these works focus on single-round evaluation, neglecting scenarios where requirements change over time.
