## 摘要
AI 智能体作为自主数字参与者，需要智能体原生协议。现有方法包括 GUI 自动化和基于 MCP 的技能，但由于缺乏统一的顶层框架和关键组件，且各独立模块存在缺陷，导致 Token 消耗高、交互碎片化、安全性不足。为解决这些问题，我们提出了 ANX，这是一种开放、可扩展、可验证的智能体原生协议和顶层框架，集成了 CLI、Skill 和 MCP，通过协议创新、架构优化和工具补充解决了上述痛点。其四大核心创新包括：1) 智能体原生设计（ANX Config、Markup、CLI），具有高信息密度、灵活性和强适应性，可减少 Token 消耗并消除不一致性；2) 人机交互结合了 Skill 的灵活性，可实现智能体可执行指令和人类可读 UI 的双重渲染；3) 支持 MCP 的按需轻量级应用，无需预注册；4) 基于 ANX Markup 的机器可执行 SOP，消除了歧义，确保了长程任务和多智能体协作的可靠性。作为系列研究的第一篇，我们重点介绍了 ANX 的设计，展示了其包含 ANXHub 的 3EX 解耦架构，并进行了初步的可行性分析和实验验证。ANX 确保了原生安全性：绕过大语言模型（LLM）的 UI 到核心（UI-to-Core）通信使敏感数据脱离智能体上下文；仅限人工确认的机制防止了自动化滥用。使用 Qwen3.5-plus/GPT-4o 进行的表单填写实验表明，与基于 MCP 的技能相比，ANX 分别减少了 47.3% (Qwen3.5-plus) 和 55.6% (GPT-4o) 的 Token 消耗；与 GUI 自动化相比，分别减少了 57.1% (Qwen3.5-plus) 和 66.3% (GPT-4o) 的 Token 消耗；且执行时间比基于 MCP 的技能缩短了 58.1% 和 57.7%。

## Abstract
AI agents, autonomous digital actors, need agent-native protocols; existing methods include GUI automation and MCP-based skills, with defects of high token consumption, fragmented interaction, inadequate security, due to lacking a unified top-level framework and key components, each independent module flawed. To address these issues, we present ANX, an open, extensible, verifiable agent-native protocol and top-level framework integrating CLI, Skill, MCP, resolving pain points via protocol innovation, architectural optimization and tool supplementation. Its four core innovations: 1) Agent-native design (ANX Config, Markup, CLI) with high information density, flexibility and strong adaptability to reduce tokens and eliminate inconsistencies; 2) Human-agent interaction combining Skill's flexibility for dual rendering as agent-executable instructions and human-readable UI; 3) MCP-supported on-demand lightweight apps without pre-registration; 4) ANX Markup-enabled machine-executable SOPs eliminating ambiguity for reliable long-horizon tasks and multi-agent collaboration. As the first in a series, we focus on ANX's design, present its 3EX decoupled architecture with ANXHub and preliminary feasibility analysis and experimental validation. ANX ensures native security: LLM-bypassed UI-to-Core communication keeps sensitive data out of agent context; human-only confirmation prevents automated misuse. Form-filling experiments with Qwen3.5-plus/GPT-4o show ANX reduces tokens by 47.3% (Qwen3.5-plus) and 55.6% (GPT-4o) vs MCP-based skills, 57.1% (Qwen3.5-plus) and 66.3% (GPT-4o) vs GUI automation, and shortens execution time by 58.1% and 57.7% vs MCP-based skills.

### 论文总结：ANX 智能体原生交互协议与 3EX 解耦架构

#### 1. 核心问题与研究背景
随着 AI 智能体（Agent）逐渐成为互联网的核心使用者，传统的以人为中心的交互范式（如 GUI 自动化）在智能体应用中暴露出严重缺陷：
*   **效率低下**：GUI 自动化导致极高的 Token 消耗和响应延迟。
*   **交互碎片化**：缺乏统一的顶层框架来协调 GUI、MCP（模型上下文协议）和 CLI。
*   **安全性不足**：敏感数据（如密码）必须经过 LLM 上下文，存在泄露风险；且缺乏不可绕过的人工确认机制。
*   **语义模糊**：自然语言描述的标准作业程序（SOP）在复杂长程任务中容易产生歧义，导致执行不稳定。

#### 2. 方法论：ANX 协议与 3EX 架构
论文提出了 **ANX (AI Native eX)**，这是一个协议优先、架构解耦的顶层框架，核心思想是将交互逻辑从 LLM 逻辑中抽离。

*   **ANX 协议层**：
    *   **ANX Markup & Config**：一种高信息密度的结构化标记语言，比纯文本或 Markdown 更简洁，支持双向渲染（智能体可读指令 vs 人类可读 UI）。
    *   **ANX CLI**：轻量级、跨平台的命令行载体，消除不同平台间的描述不一致。
    *   **结构化 SOP**：通过 `sources` 和 `targets` 显式定义任务依赖和分支逻辑，实现机器可执行的确定性工作流。
*   **3EX 解耦架构**：
    *   **Expression（表达层）**：使用 ANX Markup 定义任务元数据和安全注释。
    *   **Exchange（交换层 - ANXHub）**：一个全球化的应用市场，支持动态语义发现，实现“即用即走”，无需预安装。
    *   **Execution（执行层 - ANX Core/Node）**：解析指令并驱动执行节点，采用“渐进式披露”减少 Token 占用。
*   **核心安全机制**：
    *   **UI-to-Core 隔离**：敏感字段直接由 UI 传给执行核心，完全绕过 LLM 上下文。
    *   **仅限人工确认**：关键操作触发的确认弹窗没有程序化接口，必须由人类手动完成。

#### 6. 主要结论与发现
*   **Token 效率显著提升**：相比 MCP，ANX 减少了 47.3% (Qwen) 和 55.6% (GPT-4o) 的增量 Token；相比 GUI，减少幅度高达 57.1% 和 66.3%。
*   **执行时间大幅缩短**：ANX 的执行速度比 MCP 快约 58%，比 GUI 快约 60% 以上。
*   **架构优势**：效率提升的主因在于 ANX 将动态数据解析（如远程下拉列表）从 LLM 上下文转移到了执行层（ANX Core），避免了冗余的数据传输。
*   **安全性验证**：通过 UI-to-Core 机制成功实现了敏感数据的物理隔离。

#### 7. 优点
*   **协议原生性**：首次提出了针对 Agent 特性的结构化标记语言，解决了 Token 膨胀问题。
*   **安全设计深入**：将安全性视为“一等公民”，通过架构设计而非提示词工程（Prompt Engineering）来保护隐私。
*   **解耦彻底**：3EX 架构实现了任务定义、发现与执行的完全分离，支持大规模工具集的动态扩展。
*   **人机协同**：支持同一套协议在 Agent 指令和人类 UI 之间无缝切换。

#### 8. 不足与局限
*   **任务覆盖面**：目前的量化实验集中在表单填写，对于更复杂的计算密集型任务或极大规模（50+步骤）的 SOP 验证尚不充分。
*   **生态依赖**：ANX 的安全性依赖于受信任的 ANX Core 和官方 Hub，若用户使用被篡改的核心组件，安全机制将失效。
*   **延迟挑战**：虽然执行时间缩短，但 ANXHub 的全球语义搜索在超大规模库中可能带来额外的冷启动延迟。
*   **社会工程学风险**：协议无法阻止 LLM 通过诱导手段欺骗用户在非 ANX UI 渠道泄露信息。
