## 摘要
使用外部工具的大语言模型智能体通常通过反应式执行来实现，即在每次观察后重复计算推理，这增加了延迟以及对错误传播的敏感性。本研究提出了“先画像后推理”（PTR），这是一种用于结构化工具增强推理的有界执行框架。在该框架中，语言模型首先合成一个显式工作流，由确定性或受保护的算子执行该工作流，验证器评估生成的轨迹，且仅在原始工作流不再可靠时才调用修复机制。本文开发了一种数学公式，将整个流水线表示为画像、路由、执行、验证、修复和推理算子的组合；在有界修复下，语言模型调用次数在名义情况下被限制为两次，在最坏情况下为三次。在六个基准测试和四个语言模型上与 ReAct 基准进行的对比实验表明，PTR 在 24 种配置中的 16 种里实现了成对精确匹配优势。结果表明，PTR 在以检索为中心和重分解的任务中特别有效，而当成功取决于大量的在线自适应时，反应式执行仍然更具优势。

## Abstract
Large language model agents that use external tools are often implemented through reactive execution, in which reasoning is repeatedly recomputed after each observation, increasing latency and sensitivity to error propagation. This work introduces Profile--Then--Reason (PTR), a bounded execution framework for structured tool-augmented reasoning, in which a language model first synthesizes an explicit workflow, deterministic or guarded operators execute that workflow, a verifier evaluates the resulting trace, and repair is invoked only when the original workflow is no longer reliable. A mathematical formulation is developed in which the full pipeline is expressed as a composition of profile, routing, execution, verification, repair, and reasoning operators; under bounded repair, the number of language-model calls is restricted to two in the nominal case and three in the worst case. Experiments against a ReAct baseline on six benchmarks and four language models show that PTR achieves the pairwise exact-match advantage in 16 of 24 configurations. The results indicate that PTR is particularly effective on retrieval-centered and decomposition-heavy tasks, whereas reactive execution remains preferable when success depends on substantial online adaptation.

#### 1. 核心问题与整体含义
传统的工具增强型大语言模型（LLM）智能体（如 ReAct 框架）通常采用**反应式执行（Reactive Execution）**模式。在这种模式下，模型在每执行一个工具并获得观察结果后，都需要重新进行推理以决定下一步行动。
*   **研究动机**：反应式执行存在三大缺陷：1) **高延迟**（频繁调用 LLM）；2) **高成本**（Token 消耗大）；3) **错误传播**（中间步骤的幻觉或检索错误会干扰后续决策）。
*   **核心目标**：提出一种**有界执行框架**，将语义推理集中在任务的开始（规划）和结束（总结）阶段，中间过程尽可能通过确定性逻辑完成，从而降低复杂度并提升鲁棒性。

#### 2. 方法论：PTR 框架
PTR 框架将智能体的工作流分解为六个阶段，核心思想是“先合成显式工作流，再进行确定性执行”：
1.  **画像 (PROFILE)**：LLM 仅调用一次，生成一个包含工具调用序列、参数占位符、假设条件和分支规则的完整工作流对象。
2.  **路由 (ROUTE)**：基于风险函数（考虑架构、历史、规模等因素）确定执行模式（纯净模式、受保护模式或可修复模式）。
3.  **执行 (EXECUTE)**：这是一个**确定性循环**。利用“自动解析规则”处理参数，利用“分支规则”处理局部逻辑跳转，无需 LLM 参与。
4.  **验证 (VERIFY)**：执行结束后，由确定性算法评估执行轨迹的“信任分”（基于失败次数、空结果等指标）。
5.  **修复 (REPAIR, 可选)**：若信任分过低，触发**至多一次** LLM 调用来修正工作流并重新执行。
6.  **推理 (REASON)**：最后一次 LLM 调用，基于执行轨迹和验证结果生成最终报告。
*   **有界性**：PTR 保证 LLM 调用次数在名义下为 2 次，最坏情况下（触发修复）为 3 次。

#### 6 主要结论与发现
*   **性能优势**：PTR 在 24 种配置中的 16 种里优于 ReAct。
*   **任务适配性**：
    *   PTR 在**检索密集型**（TriviaQA/NQ-Open）和**任务分解型**（StrategyQA）任务中表现极佳，因为这些任务的工作流相对固定。
    *   ReAct 在**多跳检索**（HotPotQA）和**高度灵活的符号推理**（AQuA-RAT）中仍具优势，因为这些任务需要根据中间观察结果实时大幅调整策略。
*   **效率**：PTR 显著降低了 LLM 调用次数，且在大多数情况下实现了更低的成本和更高的准确率。

#### 7. 优点
*   **架构创新**：通过数学化定义算子，将智能体行为从“黑盒推理”转变为“可观测、可审计的确定性状态机”。
*   **高效率**：严格限制 LLM 调用次数，解决了反应式智能体在长轨迹下的延迟和成本爆炸问题。
*   **鲁棒性**：引入验证和有界修复机制，能够识别并纠正结构性执行错误。

#### 8. 不足与局限
*   **动态适应能力有限**：对于那些“下一步行动完全取决于上一步具体内容”的开放式探索任务（如复杂多跳搜索），预先规划的 PTR 显得过于僵化。
*   **样本量限制**：每项测试仅 50 个样本，对于 EM 差异较小的配置，统计显著性可能不足。
*   **依赖初始规划**：如果第一步的 PROFILE 阶段出现严重偏差，后续的确定性执行和有限修复可能无法挽回。
