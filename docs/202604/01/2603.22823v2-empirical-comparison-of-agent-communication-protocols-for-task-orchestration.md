## 摘要
背景：如今，人工智能智能体系统正从单一工具交互向复杂的多智能体编排转型。因此，出现了两种竞争性的通信协议：一种是规范智能体如何调用外部工具的工具集成协议，另一种是使自主智能体能够相互发现并委派任务的智能体间委派协议。尽管这些协议已被数十家企业合作伙伴广泛采用，但文献中尚不存在对这些协议的实证比较。目标：本研究旨在开发首个系统性基准，在三个复杂度级别的标准化查询下，比较仅工具集成、多智能体委派以及混合架构，并量化在响应时间、上下文窗口消耗、货币成本、错误恢复和实现复杂度方面的权衡。

## Abstract
Context. Nowadays, artificial intelligence agent systems are transforming from single-tool interactions to complex multi-agent orchestrations. As a result, two competing communication protocols have emerged: a tool integration protocol that standardizes how agents invoke external tools, and an inter-agent delegation protocol that enables autonomous agents to discover and delegate tasks to one another. Despite widespread industry adoption by dozens of enterprise partners, no empirical comparison of these protocols exists in the literature. Objective. The goal of this work is to develop the first systematic benchmark comparing tool-integration-only, multi-agent delegation, and hybrid architectures across standardized queries at three complexity levels, and to quantify the trade-offs in response time, context window consumption, monetary cost, error recovery, and implementation complexity.

### 1. 核心问题与整体含义（研究动机和背景）
随着大语言模型（LLM）智能体从单一工具交互向复杂的多智能体协作转型，业界出现了两种竞争性的通信范式：
*   **Model Context Protocol (MCP)**：由 Anthropic 提出，标准化了智能体如何调用外部工具（Agent-to-Tool）。
*   **Agent-to-Agent Protocol (A2A)**：由 Google 提出，标准化了自主智能体之间的发现与任务委派（Agent-to-Agent）。
**核心问题**：尽管这两种协议在工业界已被广泛采用，但缺乏实证数据来指导开发者在不同复杂度的任务下应如何选择架构。本研究旨在填补这一空白，量化不同架构在性能、成本和复杂度上的权衡。

### 2. 论文提出的方法论
研究者开发了一个名为“开源健康分析器（Open Source Health Analyzer）”的应用，并实现了三种对比架构：
*   **架构 A (纯 MCP 架构)**：采用单智能体模式，通过 stdio 传输协议连接四个 MCP 工具服务器。所有工具返回的数据都累积在同一个上下文窗口中。
*   **架构 B (A2A 多智能体架构)**：采用“协调者-专家”模式。协调者通过 HTTP/JSON-RPC 将子任务委派给四个独立的专家智能体（每个专家运行一个 FastAPI 服务）。每个专家拥有独立的上下文窗口。
*   **架构 C (混合架构)**：引入一个轻量级的启发式路由层。根据查询关键词预判复杂度，简单查询走 MCP 路径，复杂查询走 A2A 路径。
*   **核心逻辑**：所有架构共享相同的应用逻辑，即从 GitHub、npm、OSV.dev 和 Stack Overflow 四个数据源提取信息并进行综合分析。

### 3. 实验设计
*   **数据集与场景**：设计了 30 个标准化自然语言查询，分为三个复杂度等级：
    *   **简单**：单数据源、单项目查询。
    *   **中等**：多数据源、单项目查询。
    *   **复杂**：多数据源、多项目对比分析。
*   **Benchmark 流程**：每种架构对每个查询执行 5 次，总计 450 次实验。
*   **对比指标**：端到端延迟（秒）、Token 消耗总量、货币成本（美元）、API 调用次数、代码行数（LOC）及圈复杂度。
*   **基准模型**：统一使用 `claude-sonnet-4-20250514` 以消除模型差异。

### 4. 资源与算力
*   **算力说明**：论文未提及具体的 GPU 型号或训练时长，因为实验主要基于现有的 LLM API 调用。
*   **部署环境**：所有 A2A 智能体部署在单机的 localhost 环境上，以消除网络波动对延迟的影响。

### 5. 实验数量与充分性
*   **实验规模**：共进行了 450 次受控实验（30 查询 × 3 架构 × 5 重复），样本量足以进行统计学分析。
*   **统计严谨性**：使用了 95% Bootstrap 置信区间、Mann-Whitney U 检验（配合 Bonferroni 校正）以及 Cliff’s delta 效应量分析。
*   **可靠性**：实验中包含故障注入测试（随机关闭 API）和幻觉检测模块，确保了结果的客观性和公平性。

### 6. 主要结论与发现
*   **性能交叉效应（Crossover Effect）**：
    *   在**简单任务**中，MCP 显著更快（8.9s vs 19.8s），因为它没有 HTTP 通信和委派开销。
    *   在**复杂任务**中，A2A 反超 MCP（45.1s vs 51.8s）。
*   **上下文窗口膨胀（Context Bloat）**：在复杂任务下，MCP 的单窗口模式导致 Token 消耗是 A2A 的 **3.1 倍**。
*   **成本优势**：对于复杂查询，A2A 相比 MCP 可降低约 **39%** 的货币成本，这归功于分布式上下文保持了每个智能体输入的精简。
*   **实现代价**：A2A 的代码复杂度显著高于 MCP（代码行数多出 2.2 倍），主要源于 HTTP 服务和协议样板代码。

### 7. 优点
*   **首次实证**：填补了 MCP 与 A2A 协议对比的学术空白。
*   **量化贡献**：首次量化了单智能体架构在处理复杂任务时的“上下文窗口膨胀”现象及其经济后果。
*   **实用性强**：提出了一个清晰的决策框架（简单任务用 MCP，复杂任务用 A2A，未知复杂度用混合路由）。

### 8. 不足与局限
*   **模型单一性**：仅测试了 Claude Sonnet，结果在其他模型（如 GPT-4o 或开源模型）上的泛化性有待验证。
*   **网络环境理想化**：实验在 localhost 运行，未考虑真实分布式系统中的网络延迟，这可能会削弱 A2A 的延迟优势。
*   **领域局限**：仅测试了开源项目分析这一领域，其他领域（如多模态任务或长程规划）的特征可能不同。
*   **样本重复次数**：每个查询仅重复 5 次，虽然总数不少，但对于捕捉 LLM 的极端随机性可能仍显不足。
