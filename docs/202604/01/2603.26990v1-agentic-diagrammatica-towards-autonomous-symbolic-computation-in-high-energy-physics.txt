# Abstract 

We present Diagrammatica , a symbolic computation extension to the HEP-TAPOD agentic framework, which enables LLM agents to plan and execute multi-step theoretical calculations. Symbolic computation poses a distinctive reliability challenge for LLM agents, as correctness is governed by implicit mathematical conventions that are not encoded in a form that can be easily checked in the computational backend. We identify two complementary reme-dies, tool-constrained computation and targeted knowledge grounding, and pursue the first as the primary architecture. Concretely, we concentrate the agent’s action distribution onto tool calls with convention-fixing semantics, in which the agent specifies a compact, human-auditable diagram specifica-tion and a trusted backend performs the symbolic or numerical manipulations exactly. The toolkit provides two complementary calculation paths consum-ing a shared diagram specification: Naive Dimensional Analysis (NDA) for order-of-magnitude rate estimates and Exact Diagrammatic Analysis (EDA) for tree-level symbolic calculations via automatic FeynCalc code generation, both supplemented by automatic Feynman diagram enumeration and a navi-gable theory knowledge base. The architecture is validated on two benchmarks: (1) an exhaustive catalog of all tree-level, single-vertex 1 → 2 partial decay widths across scalar, fermion, and vector parents, with complete massless and threshold limits and Standard Model validation; and (2) an NDA sensitivity study of the muon decay multiplicity μ → νμ ¯νe + n(e+e−) + e−, determining the maximum observable n at current and planned muon experiments.

21 Introduction 

Symbolic computation has a long and distinguished tradition in high-energy physics (HEP), sup-ported by a rich ecosystem of computer algebra systems (CAS) and specialized packages. Dedicated systems such as REDUCE [1], FORM [2], Cadabra [3], and Redberry [4] provide HEP-oriented sym-bolic engines, while general-purpose platforms like Mathematica [5] and SymPy [6] host widely used packages including FeynCalc [7, 8], FeynArts/FormCalc [9, 10], Package-X [11, 12], and FIRE [13]. These tools are powerful, but even routine calculations are manual, multi-step processes. For ex-ample, computing tree-level decay widths or leading-order cross sections requires: constructing an amplitude from Feynman rules, squaring it, performing spin sums, evaluating Dirac traces, substi-tuting kinematics, and extracting a partial width. Many steps require familiarity with both the physics and the idiosyncrasies of the specific software package being used. This is why hands-on tutorials are now routinely scheduled at graduate-level summer schools and workshops [14]. Large language models (LLMs) have recently emerged as general-purpose computational interfaces capable of planning and executing multi-step workflows through structured tool invocation [15, 16]. In our previous work, we introduced the HEP Toolkit for Agentic Planning/Programming, Orchestration, and Deployment (HEPTAPOD) framework [17], demonstrating how an LLM agent can orchestrate Monte Carlo simulation pipelines through schema-validated tool invocation. Since then, a number of papers employing agentic LLMs have appeared that target different stages of the HEP workflow, including event generation [18], collider analysis pipelines [19,20], experimental data analysis [21, 22], experimental design and detector simulation [23], theory hypothesis generation and reinterpretation [24], lattice QCD automation [25], and broader multi-domain science [26]. In all of these systems the LLM orchestrates domain-specific computational engines rather than replacing them, providing the procedural knowledge that connects individual steps into coherent workflows. This paper introduces Diagrammatica 1, a symbolic computation extension to the HEPTAPOD framework that enables LLM agents to plan and execute multi-step theoretical calculations. Sym-bolic computation poses a distinctive challenge compared to the Monte Carlo orchestration demon-strated in [17]. Frontier LLMs are already capable of impressive symbolic manipulation (solving integrals, simplifying expressions, writing working computer algebra code), but reliability , not capa-bility, is the bottleneck. The multi-step calculations typical of HEP phenomenology require dozens of symbolic operations chained together under a shared set of implicit conventions that must re-main internally consistent throughout. Autoregressive generation provides no guarantee that this consistency will be maintained across an extended workflow, even when each individual step is well within the model’s capabilities. An LLM navigating this space without external structure can produce plausible but silently wrong results, including dropped terms, misapplied identities, and inconsistent conventions. Even purpose-built transformer models trained to perform specific sym-bolic subtasks, such as predicting squared amplitudes directly from Feynman diagrams [28], can achieve high but imperfect accuracy, illustrating the difficulty of replacing traditional CAS tools for exact symbolic computation. Additionally, auditing free-form symbolic code for such errors is 

at least as hard as writing it from scratch. In this paper we identify two complementary remedies

The first is tool-constrained computation : instead of asking the agent to generate the symbolic calculation directly, the action space is restricted to schema-validated tool calls that fix conventions by construction, where the agent specifies a compact, human-auditable diagram specification and a trusted backend performs the algebra exactly. The second is targeted knowledge grounding : delivering domain-specific conventions and worked examples to the agent at the moment of each critical decision, rather than loading reference material in bulk. Tool constraints provide structural reliability that is independent of the model’s knowledge, but bounds the agent’s capabilities to the tool designer’s foresight; knowledge grounding preserves the full action space but offers only probabilistic suppression of errors. This paper primarily pursues the tool-constrained path, realized through a shared diagram specification and deterministic symbolic tools, while using a navigable theory knowledge base as a complementary layer (section 2.3). This approach contrasts with the predominant paradigm in the growing literature on LLM agents in HEP [15, 16, 18–26], in which the agent writes free-form scripts or shell commands executed against existing software, with the reliability burden falling entirely on the model’s ability to produce correct code from context. The key distinction of the approach pursued here (as well as in Ref. [17]) is to design tool schemas that reduce the agent’s action space to a physically meaningful set of choices, fixing domain conventions by construction and providing a more natural computational interface for the LLM. This yields two concrete advantages: reliability , as correctness of the mechanical steps becomes a property of the interface rather than of the model, eliminating entire classes of silent errors independently of model capability, and interpretability , as the agent’s decisions are compressed into a small number of human-readable fields that a domain expert can audit at a glance rather than reviewing pages of generated code where errors may be invisible without re-deriving the calculation from scratch. The toolkit provides two complementary calculation paths at different fidelity levels, both consuming a shared diagram specification: 1. NDA (Naive Dimensional Analysis): Order-of-magnitude estimates of decay widths, cross sections, and branching ratios using dimensional analysis, phase space volumes, and coupling power counting. 2. EDA (Exact Diagrammatic Analysis): Complete tree-level calculations via automated FeynCalc [7, 8] code generation. Produces symbolic expressions, L ATEX output, and Python-callable numerical evaluations from a single diagram specification. This two-fidelity structure mirrors how phenomenologists naturally work, allowing for order-of-magnitude estimates to guide the decision to commit to a full calculation. It also extends the toolkit’s reach beyond what either path could achieve alone, since exact analytic results are restricted to low-multiplicity final states where phase space integration is tractable, while NDA provides useful rate estimates for arbitrarily complex processes. Both paths consume a shared, LLM-compatible, convention-fixing diagram specification, introduced in this work, and are further supplemented by automatic Feynman diagram enumeration via FeynGraph [29] as well as a navigable theory knowledge base. We define two benchmark tasks in this paper to serve as both architecture validation and as repro-ducible test cases for other agentic platforms. Task 1 (section 4.1) is an exhaustive catalog of all 2tree-level, single-vertex 1 → 2 decay widths across scalar, fermion, and vector parents, validated against Standard Model partial widths. Task 2 (section 4.2) is an NDA-based sensitivity study of 

μ → νμ ¯νe + n(e+e−) + e− as a function of multiplicity n, stressing diagram enumeration and rate estimation across a large variety of diagrams. Both tasks are presented through curated, annotated agent transcripts from single Claude Code [30] sessions, demonstrating the full workflow from task prompt to compiled deliverables without manual intervention. The remainder of this paper is organized as follows. Section 2.1 establishes the case for tool-constrained agentic calculation through an entropy decomposition of the agent’s action distribution, with a token-level illustration for Z → e+e−. Section 3 presents the toolkit architecture: the shared diagram specification, the NDA and EDA calculation paths, the FeynGraph enumeration engine, and the theory knowledge base. The architecture is validated on two benchmarks: section 4.1 presents the 1 → 2 decay rate catalog (Task 1), exercising the EDA path, and section 4.2 presents the muon decay sensitivity study (Task 2), exercising the NDA path. Section 5 discusses implications and future directions.

2.1 Reliability, uncertainty, and the case for tool-constrained calculation

A tool is an external computational backend—a function, script, or service—exposed to the agent in the initialization context C0 through a convention-dependent schema specifying its name, inputs, and return types. A tool call is then a token sequence that forms a structured JSON object matching this schema, which the inference serving infrastructure can intercept and route to that backend (typically the user’s machine). Importantly, adding or removing tools doesn’t change the model’s token vocabulary, but for particular tasks it can significantly reshape the conditional distribution of eq. (2.2) by concentrating probability mass onto the structured subset of the action space corresponding to valid tool invocations. Some inference providers enforce this concentration even more strongly 3through constrained decoding [31], in which tokens incompatible with the tool schema are masked to zero probability at each generation step. The practical value of concentrating p(At | Ct) depends on the task. The obvious candidates are exact arithmetic, external state access, and formal grammar adherence. More subtle, and more relevant here, are tasks where the correctness criteria are implicit — nowhere encoded in a form the computational backend can check against reliably. Symbolic computation occupies this category. Unlike compiled code, where type errors and syntax violations halt execution and provide immediate feedback, a computer algebra system has no equivalent system to check its input against. Correctness is defined implicitly by the mathematical conventions of the domain, which are external to the system itself. An LLM navigating this space without external structure can only rely on statistical correlations in its training data, correlations that are themselves unreliable, since conventions in the literature are not always unanimous. A high-energy physics symbolic calculation is a particularly acute instance, where conventions are numerous, interrelated, and drawn from a literature in which different sources make incompatible choices. A correct calculation may have to simultaneously respect consistent choices for the metric signature ( +−−− vs. −+++ ), spinor normalization ( ¯uu = 2 m vs. ¯uu = 1 ), the sign of the covariant derivative, the normalization of SU( N ) generators ( Tr[ T aT b] = δab /2 vs. δab ), the choice of gauge, and the phase conventions in CKM and PMNS matrices, among others. An LLM trained on the full literature will have seen all of these conventions represented, often without explicit declaration, and the statistical correlations it relies on are precisely the wrong tool for enforcing internal consistency across a multi-step calculation where every convention choice must be made once and propagated uniformly. Trusted symbolic computation with LLM agents therefore requires that the action space be restricted to tools where a fixed set of conventions is encoded in the computational backend, or that additional context be provided to the agent fixing the conventions explicitly. Developing tools that provide these semantic constraints reliably for common HEP computations is the main focus of this work. The simplest way to realize this is to alter what inputs the agent operates on: instead of asking the model to realize the calculation directly in free-form symbolic text, one asks it to specify a typed action whose arguments correspond to physically meaningful choices—vertex class, external spins, topology, mass assignments, and other convention-fixing data—and the backend performs the algebra exactly. In this design, the LLM is responsible for selecting the correct calculation, while the symbolic engine is responsible for executing it. A similar principle underlies the reliability of LLM-based theorem proving in Lean [32], where probability mass is concentrated on valid proof tactics verified against a structured proof state rather than on free-form natural-language reasoning. The resulting verification asymmetry shifts the locus of human oversight from a high-dimensional verification problem to a low-dimensional one. It is easier to audit a compact, human-readable diagram specification with a few physically meaningful fields than a full generated script. • Context uncertainty ∆C (L) is the degradation of attention and retrieval fidelity as context length grows [35]. Unlike ∆T , this term increases with L, introducing a fundamental tension in which adding context to resolve task ambiguity simultaneously inflates context-induced errors. There is a natural scale LC marking the onset of this degradation. When LT ≪ LC , the task is fully resolved before degradation sets in and reliable operation is possible. When LT ≳ LC ,the documentation needed to resolve the task pushes the context past the degradation onset, 5and the workflow becomes unreliable regardless of how much reference material is provided. The remedy is to increase the information density of each context token by structuring domain knowledge as compact, retrievable nodes rather than raw long-form text [36], exposing exactly the information the agent needs in a small number of tokens and shifting LC to larger values. Crucially, when such a node is retrieved on demand at the moment the agent faces a convention-dependent choice, it not only avoids inflating ∆C but also partially suppresses ∆E by resolving the specific ambiguity at the point where execution variability originates. It is the timing of delivery , not the content, that distinguishes this from bulk documentation (see section 2.3). Finally, it is reasonable to expect an irreducible floor ε > 0 in the action entropy of any finite autoregressive model. Even with perfect context and a fully constrained action space, the stochastic sampling procedure and finite model capacity prevent the conditional distribution from collapsing to a delta function on the correct action. Whether such a floor can be made rigorous, and how it scales with model size, is an open question; for the present purposes it suffices to note that the three architectural remedies above—structured specifications, tools, and skills—are designed to reduce the controllable components of ∆ as far as possible, leaving whatever irreducible residual remains as the dominant source of uncertainty.

2.3 Tool constraints versus knowledge grounding 

The execution uncertainty ∆E identified in the previous section admits two complementary remedies that, despite both involving the delivery of information to the agent, operate through qualitatively different channels and offer fundamentally different tradeoffs. The first mechanism, emphasized in this paper, is tool-constrained computation . Instead of asking the agent to generate the symbolic calculation, the action space is restricted to schema-validated tool calls that fix conventions by construction. The agent specifies what to compute through a

8diagram specification, and a trusted backend determines how to compute it by performing the algebra exactly. This drives ∆E to near zero structurally , since correctness is a consequence of the interface design rather than the model’s knowledge, and holds regardless of what context the agent has seen. The cost is that the agent’s capabilities are bounded by the tool designer’s foresight. It can compute any decay width the tools support, but it cannot notice that two seemingly unrelated results satisfy an unexpected identity, propose a novel parameterization, or derive a relation that the tool vocabulary does not anticipate. The tools define the ceiling. The second mechanism is knowledge-grounded computation . As discussed in section 2.1, bulk refer-ence material loaded into context does not suppress ∆E because the agent must still retrieve and apply the right convention at the right moment from a long context, which is essentially a ∆C prob-lem. What distinguishes targeted knowledge injection is timing. A skills-graph node retrieved on demand at the exact moment the agent faces a convention-dependent choice directly concentrates the token distribution at that position, resolving the specific ambiguity that drives execution vari-ability. This is not “more context” in the sense of the ∆C analysis but rather context that acts at the 

locus of ∆E . The distinction from task-specifying context, which reduces ∆T by clarifying what to compute, is equally important. Targeted knowledge addresses how to execute a step that is already specified, not what the step should be. The result is a partial, probabilistic suppression of ∆E , not the structural guarantee of tool constraints, but a genuine reduction that scales with the quality and specificity of the injected knowledge. The advantage is that the full action space remains available. A knowledge-grounded agent could in principle recognize patterns across calculations, derive new identities from first principles, or explore parameterizations that the tool designer never considered. The ceiling is set by the model’s reasoning capability, not by the tool vocabulary. In practice, the choice between these mechanisms is not binary but a spectrum, and the optimal design point depends on the nature of the task. Mechanical steps where creativity adds no value and errors are catastrophic (Dirac traces, spin sums, polarization averaging, phase space integration) should be tool-constrained. Strategic steps where judgment and exploration are valuable (which process to study, what limits to take, how to organize a survey of many processes) benefit from free-form reasoning grounded in domain knowledge. The Diagrammatica toolkit already makes this split implicitly: the agent reasons freely about strategy while using tools for tactical execution. Naming it explicitly as a design principle clarifies both the strengths and the limitations of the current architecture. The two approaches are also complementary in practice, since an agent that explores freely with knowledge grounding can confirm its results via EDA tool calls, combining the discovery potential of the first approach with the reliability guarantees of the second. This suggests a natural trajectory for future agentic systems, beginning with the tool-constrained architecture presented here and progressively expanding the boundary of free-form reasoning as the knowledge base deepens and verification tools mature. 

# 3 Architecture and design 

The uncertainty decomposition of section 2.1 identifies three architectural requirements for a reliable agentic symbolic toolkit: a compact structured representation that the agent can construct with 9low task uncertainty ∆T , a skills layer that supplies targeted domain knowledge to suppress context uncertainty ∆C , and schema-validated deterministic tools that drive execution uncertainty ∆E to-ward zero for the mechanical steps of the calculation. The six-step calculation procedure introduced in section 2.2 for S → f ¯f is universal, since every 1 → 2 decay follows the same sequence. Step 1, writing the invariant amplitude, is the only process-specific ingredient. Steps 2–6 are identical for every supported topology and are automated entirely by the ComputeSymbolicAmplitude tool (see fig. 2), which maps a structured diagram specification to a complete, self-contained FeynCalc script. This division, where the agent selects the physics and the backend executes the algebra, is the concrete realization of the ∆E suppression argued for in section 2.1. The key architectural enabler is a shared Diagram specification (section 3.1), a JSON dataclass consumed by all toolkit components that provides a common structured representation of the physics across stages. Beyond the three entropy-motivated requirements, the architecture is organized around two addi-tional design principles: 1. Persistent artifacts and provenance. Generated scripts, parsed result sidecars, and ex-ported Python functions are saved as concrete artifacts in the agent’s session sandbox rather than discarded as transient outputs. Downstream tools consume results by reference, enabling auditable multi-step pipelines without manual transcription of symbolic expressions. 2. Multi-fidelity pathways. The toolkit provides two calculational paths at different fidelity levels, both consuming the same diagram specification. NDA (section 3.2) produces order-of-magnitude decay width and cross section estimates from dimensional analysis, phase space volumes, and coupling power counting, requiring no external software and applicable to arbi-trary n-body final states. The EDA path (section 3.4) produces exact tree-level symbolic results via automatic FeynCalc code generation. The current scope of the EDA path is restricted to 

1 → 2 decays and 2 → 2 scattering processes, where the phase space integration is analytic and the matrix element dependence on final-state kinematics drops out (for 1 → 2) or reduces to Mandelstam variables (for 2 → 2). Extension to n ≥ 3 final-state particles, where phase space integration over non-trivial matrix elements is required, is left to future work. For such pro-cesses, where exact analytic phase space integration is generally not available, numerical Monte Carlo tools such as MadGraph [37], already integrated into the HEPTAPOD framework [17], become the primary calculational path. MadGraph also serves as an independent numerical cross-check on the EDA and NDA results for simpler processes. Both paths can be fed by automatic Feynman diagram enumerations engines, here we use FeynGraph [29] (section 3.3). Agreement in scaling and order of magnitude between paths provides an internal consistency check; disagreement identifies where assumptions, topology assignments, or kinematics require inspection. The remainder of this section describes each component in turn. 

3.1 The LLM-compatible diagram specification 

The central architectural element of Diagrammatica is an LLM-compatible Feynman diagram for-mat. A single structured JSON object specifies a decay or scattering process and serves simultane-ously as the input to all toolkit components and as a natural object for an LLM agent to construct, 10 inspect, and manipulate. Existing HEP software represents processes through formats designed for traditional programmatic workflows. FeynArts [9] uses deeply nested Mathematica objects ( TopologyList , FeynmanGraph )tied to the Mathematica ecosystem, and parton-level-generators like CompHEP [38], CalcHEP [39], and MadGraph [37] specify processes through a domain-specific command syntax (for example, 

generate p p > t t˜ QCD<=2 ). These formats are expressive within their respective ecosystems, but none provides a single structured object that simultaneously serves as input to tools at differ-ent fidelity levels and is natural for an LLM to construct under the token-entropy constraints of section 2.2. The Diagrammatica diagram specification is designed for agentic manipulation, with a small, fixed vocabulary of field names, schema-validated enumerated types for vertex structures, human-readable particle labels, and a flat JSON structure requiring no ecosystem-specific knowl-edge. The Diagrammatica Diagram data structure has two modes. Both share the same top-level schema with fields for initial- and final-state particles (label, spin, and optionally mass), vertices (Lorentz-structure type and coupling), propagators, color factors, and optional topology hints. 3.3 Enumerating diagrams with FeynGraph 

FeynGraph [29] is an efficient Feynman diagram enumeration engine written in Rust with Python bindings. It serves as a common entry point to both calculation paths, automatically enumerating all Feynman diagrams contributing to a given process (though the agent can also construct diagrams manually when the process is known in advance). The Standard Model vertex set is available by default; BSM models can be loaded via UFO [43] model files. Given initial- and final-state particle lists, FeynGraph enumerates all topologically distinct diagrams consistent with the model vertices. The Diagrammatica wrapper around FeynGraph then converts each enumerated diagram to the shared Diagram dataclass of section 3.1 via automatic vertex-type inference, mapping SM particle content to the vertex vocabulary of table 1 without requiring the agent to know the Lorentz structure in advance (e.g., a W boson coupled to a fermion pair is assigned "left-handed" , a Z boson "vector-axial" ). 3.4 Computing diagrams with EDA 

The EDA path generates complete, self-contained Mathematica/FeynCalc scripts from symbolic di-agram specifications, executes them via wolframscript , and returns parsed structured results. The 13 code generator maps the abstract diagram description to concrete FeynCalc function calls covering the procedure outlined in section 2.2, handling three topology classes: direct 1 → 2 decays, 1 → 2

decays with one internal propagator, and 2 → 2 scattering with s-, t-, or u-channel exchange. Each call computes a single diagram for a specific set of particles, a single vertex type, and a single coupling symbol per chiral projection. When different final-state flavors do not interfere (e.g., 

Z → e+e− vs. Z → μ+μ−), inclusive widths are assembled by the agent through multiple tool calls, keeping each individual computation simple and auditable. A notable limitation of this single-coupling-per-vertex design is that it does not capture the flavor structure arising from non-diagonal coupling matrices. A typical EDA workflow chains 3–5 tool calls. After the agent specifies a Diagram (either by hand or via FeynGraph), ComputeSymbolicAmplitude generates the script, RunWolframScript executes it, optional SimplifyResult calls apply limits or substitutions, and ConvertToPython exports the final formula as a callable Python function.

# 4 Benchmarks 

The two benchmark tasks defined in this work serve complementary roles in validating the ar-chitecture of section 3. Task 1 (section 4.1) exercises the EDA path on a systematic sweep of all supported vertex families, testing the shared diagram specification, code generator, and multi-fidelity cross-checks. Task 2 (section 4.2) exercises the NDA path and FeynGraph enumeration engine on a problem with strong combinatorial growth, stressing diagram ranking and approximate rate estimation across large diagram classes. Both tasks are presented through curated, annotated agent transcripts from single sessions. Extended agent transcripts can be found in appendices C and D, while the full system prompts and complete, unedited conversation logs are available in the accompanying data release (see section 6). Throughout this work, we use Claude Opus 4.6 [30] as the LLM agent via Claude Code, Anthropic’s CLI for agentic development. Like HEPTAPOD [17], the Diagrammatica toolkit is implemented in the Orchestral AI framework [44], which supports all major LLM providers and allows for all tools to be exposed via the Model Context Protocol (MCP) [45]. Since MCP has been adopted 15 as the standard tool-integration protocol across the major agentic frameworks (LangChain/Lang-Graph, OpenAI Agents SDK, CrewAI, Microsoft Agent Framework, and PydanticAI), the same tools are immediately consumable by any MCP-compatible agent, independent of the underlying model provider. Each task is run in a fresh sandbox with a task-specific subset of HEPTAPOD tools pre-authorized and a structured system prompt documenting the tool interfaces and available workflows. The system prompts do not contain task-specific instructions; the agent must plan its own approach from the combination of the task prompt and the tool documentation.

For each vertex family admitting dual parameterizations—scalar–pseudoscalar versus chiral for SFF, and vector–axial versus chiral for VFF—the agent computed both bases as separate ComputeSym-bolicAmplitude calls, yielding 20 code-generation calls producing 19 distinct FeynCalc scripts. The code generator handles all downstream algebra—Dirac traces, spin sums, polarization sums, phase space, and kinematic simplification—producing a self-contained WolframScript with structured output markers. 2. Batch execution: All 19 scripts were dispatched in a single RunWolframScriptBatch call with 5-way concurrency. Validation: With all 19 formulas in hand, the agent entered a systematic validation phase. It ran six chiral ↔ VA cross-checks via RunWolframScript , confirming all algebraic differences to be exactly zero in Mathematica. Synthesis: The agent wrote four standalone Python plotting scripts (executed to generate publication figures) and compiled a 12-page L ATEXreference document containing all 19 formulas in both coupling bases, the cross-check and validation tables, and a phenomenological analysis of the catalog. Coupling. Each pair introduces two QED vertices (photon emission and γ → e+e− conversion), contributing a factor e4 = (4 πα )2 ≈ 8.4 × 10 −3 to |M| 2.

# 5 Conclusions 

In this work, we introduced Diagrammatica , a symbolic computation extension to the HEPTAPOD agentic toolkit that enables LLM agents to plan and execute multi-step theoretical calculations in high-energy physics. The conceptual backbone of the architecture is an entropy decomposition of the agent’s action distribution into task, context, and execution components ( ∆T , ∆C , ∆E ), which identifies execution uncertainty as the primary obstacle to reliable agentic symbolic computation. Unlike traditional programming, where compilers and runtime checks can at least enforce syntactic and type correctness, symbolic computation is governed by implicit mathematical conventions that no computational backend can efficiently verify independently. We argued that this challenge admits two complementary remedies, tool-constrained computation that drives ∆E toward zero structurally and targeted knowledge grounding that suppresses it probabilistically, and pursued the first as the primary architecture. The toolkit is organized around a shared LLM-friendly diagram specification that serves as a single source of truth across two complementary calculation paths. Naive Dimensional Analysis provides order-of-magnitude rate estimates applicable to arbitrary final-state multiplicities, while Exact Dia-grammatic Analysis produces complete tree-level symbolic results through automatic FeynCalc code generation. Both paths consume the same structured JSON specification and are fed by FeynGraph for automatic Feynman diagram enumeration, with a navigable theory knowledge base exposed 29 through the same tool interface. By treating diagram specifications, symbolic results, and generated scripts as persistent, referenceable artifacts within the agent’s orchestration loop, the architecture preserves transparency, provenance, and human oversight throughout multi-step workflows. Two benchmarks, each completed autonomously from a single user prompt, validated the architec-ture. Task 1 exercised the EDA path on an exhaustive catalog of all tree-level, single-vertex 1 → 2

decay widths, producing 19 independent formulas across 6 vertex families with complete massless and threshold limits, chiral cross-checks, and Standard Model validation at the few-percent level. Task 2 exercised the NDA path and FeynGraph on the muon decay μ → νμ ¯νe + n(e+e−) + e−, enu-merating over 150,000 tree-level diagrams across four multiplicities to map the experimental frontier from established measurements through the reach of Mu3e and HiMB, with MadGraph cross-checks confirming both the NDA scaling and published data. Both tasks are specified with sufficient detail to serve as reproducible test cases for other agentic platforms, and we encourage other groups to attempt them and report their results. Several design principles emerged from this work: 1. The reliability of agentic symbolic computation is determined less by the model’s familiarity with domain conventions than by whether the tool interface encodes them. Concentrating convention-dependent choices into schema-validated fields eliminates entire classes of silent errors by construction, independently of model capability. 2. A multi-fidelity organization, in which approximate and exact calculations consume the same structured specification, lets the agent match computational cost to the question being asked while providing internal consistency checks. 3. Treating generated scripts, symbolic results, and intermediate outputs as persistent, refer-enceable artifacts rather than transient context allows downstream tools to consume results by reference, avoiding the context-window inflation that would otherwise degrade attention and retrieval fidelity over multi-step pipelines. 4. Co-locating procedural tools and theoretical reference material in the same tool interface allows the agent to look up conventions at the moment of each critical decision, suppressing execution uncertainty through targeted retrieval rather than bulk documentation. These principles point to a broader observation about the role of tools in agentic scientific compu-tation. The growing literature on LLM agents in HEP [15, 16, 18–26] has predominantly adopted a 

code generation paradigm: the agent writes free-form scripts or shell commands that are executed against existing software, with the reliability burden falling entirely on the model’s ability to pro-duce correct code from context. This approach inherits the full execution uncertainty ∆E analyzed in section 2.1, since the agent must independently resolve every domain-dependent choice at the to-ken level. The HEPTAPOD framework [17] was designed from the outset around schema-validated tool invocation rather than code generation, and Diagrammatica continues this approach into the domain of symbolic computation, where the stakes are particularly high because correctness is gov-erned by implicit mathematical conventions rather than compiler-enforceable rules. The key is to design tool schemas whose valid completions fix not only the format but the domain conventions of the calculation, so that every schema-compliant invocation maps to a correct computation by con-30 struction. This yields two distinct advantages. The first is reliability : correctness of the mechanical steps becomes a property of the interface rather than of the model, eliminating entire classes of silent errors independently of model capability. The second is interpretability : because the agent’s decisions are compressed into a small number of human-readable fields, a domain expert can audit the full specification at a glance, in sharp contrast to reviewing pages of generated code where errors may be invisible without re-deriving the calculation from scratch. We believe convention-fixing tool design, in which the structure of the action space encodes as much domain knowledge as the model that navigates it, deserves wider adoption in agentic HEP workflows. The present architecture suggests several natural directions for future development. Perhaps the most natural next step is to seamlessly integrate Diagrammatica’s symbolic outputs to HEPTA-POD’s existing Monte Carlo tools (MadGraph [37], Pythia [47], Sherpa [48]), which would close the loop between symbolic theory and simulated events within a single agentic session and realize the full potential of the HEPTAPOD framework as an end-to-end research tool. More specifically, the NDA path can be extended to more refined power-counting prescriptions, such as generalized NDA rules [49] for arbitrary EFTs or warped/composite NDA for strongly coupled scenarios [50], with the tool interface designed to accommodate alternative scaling prescriptions without changing the diagram specification. For the EDA path, three extensions would significantly broaden the toolkit’s scope: one-loop calculations via Package-X [11, 12] for scalar integral evaluation, flavor-indexed coupling matrices enabling decay widths with full interference structure for non-diagonal flavor couplings, and automatic parsing of UFO model files from FeynRules [43] into FeynGraph vertex rules for BSM diagram enumeration without manual model specification. More broadly, the tool-constrained and knowledge-grounded approaches discussed in section 2.3 suggest a trajectory for increasingly autonomous systems that begins with the tool-constrained architecture presented here and progressively expands the boundary of free-form reasoning as the knowledge base deepens and verification tools mature. Multi-agent configurations, where specialized agents for symbolic calculation, Monte Carlo validation, and data comparison are coordinated by a planning agent, represent a natural evolution toward this goal. Diagrammatica provides the symbolic computation layer required for increasingly capable AI-assisted research in high-energy physics. The design principles identified here, including convention-fixing semantics, multi-fidelity tool hierarchies, shared structured specifications, and theory-grounded interfaces, offer a blueprint for reliable agentic computation that can extend a researcher’s effec-tive capabilities beyond what either the researcher or the agent could achieve working indepen-dently.
