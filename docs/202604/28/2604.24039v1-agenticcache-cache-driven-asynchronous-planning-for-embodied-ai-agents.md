## 摘要
具身智能体（Embodied AI agents）越来越多地依赖大语言模型（LLMs）进行规划，但逐步骤的 LLM 调用带来了严重的延迟和成本问题。在本文中，我们展示了具身任务表现出强烈的规划局部性（plan locality），即下一个规划在很大程度上可以根据当前规划进行预测。基于此，我们提出了 AgenticCache，这是一个通过重用缓存规划来避免逐步骤 LLM 调用的规划框架。在 AgenticCache 中，每个智能体查询频繁规划转换的运行时缓存，而后台缓存更新器（Cache Updater）则异步调用 LLM 以验证和细化缓存条目。在四个多智能体具身基准测试中，AgenticCache 在 12 种配置（4 个基准测试 x 3 个模型）下平均将任务成功率提高了 22%，模拟延迟降低了 65%，Token 使用量减少了 50%。因此，基于缓存的规划重用为实现低延迟、低成本的具身智能体提供了一条切实可行的路径。代码可在 https://github.com/hojoonleokim/MLSys26_AgenticCache 获取。

## Abstract
Embodied AI agents increasingly rely on large language models (LLMs) for planning, yet per-step LLM calls impose severe latency and cost. In this paper, we show that embodied tasks exhibit strong plan locality, where the next plan is largely predictable from the current one. Building on this, we introduce AgenticCache, a planning framework that reuses cached plans to avoid per-step LLM calls. In AgenticCache, each agent queries a runtime cache of frequent plan transitions, while a background Cache Updater asynchronously calls the LLM to validate and refine cached entries. Across four multi-agent embodied benchmarks, AgenticCache improves task success rate by 22% on average across 12 configurations (4 benchmarks x 3 models), reduces simulation latency by 65%, and lowers token usage by 50%. Cache-based plan reuse thus offers a practical path to low-latency, low-cost embodied agents. Code is available at https://github.com/hojoonleokim/MLSys26_AgenticCache.

### 1. 核心问题与整体含义
*   **研究动机**：具身智能体通常遵循“感知-规划-行动”循环。目前主流做法是每一步都同步调用 LLM 生成规划，这导致了严重的**推理延迟**（智能体必须停下来等回复）和**高昂成本**（Token 消耗大）。
*   **核心发现**：作者发现具身任务具有强烈的“**规划局部性（Plan Locality）**”——即下一个规划往往可以从当前状态和前一个规划中高度预测（例如：抓取物体后，下一步大概率是运输）。
*   **整体含义**：通过引入类似计算机体系结构中的“缓存”机制，让智能体优先复用历史成功的规划转换，仅在必要时异步咨询 LLM，从而实现低延迟、低成本的实时交互。

### 2. 方法论
AgenticCache 的核心思想是**异步解耦规划与执行**，主要包含以下关键技术：
*   **运行时缓存结构**：每个智能体维护一个本地缓存，存储“2-gram”规划转换模式（即 $\langle 规划 P_i \to 规划 P_j \rangle$）。每个条目包含元数据（如步骤数、持有物品数等状态范围）。
*   **评分与选择机制**：通过公式 $S = C \cdot I$ 选择最佳候选规划。其中 $C$ 是局部转换频率（Count），$I$ 是全局重要性（Importance，即该规划被 LLM 确认的比例）。
*   **异步缓存更新器（Cache Updater）**：
    *   **确认（Confirmation）**：如果后台 LLM 的回复与缓存预测一致，则增强该缓存条目的权重。
    *   **修正（Correction）**：如果 LLM 认为当前缓存规划错误，更新器会立即**替换**正在执行的动作，确保鲁棒性。
*   **查询抑制（Suppression）**：为了节省成本，当缓存与 LLM 达成一致或正在执行修正动作时，系统会暂时停止发送冗余的 LLM 请求。
*   **预填补（Warm-start）**：可选方案，利用少量离线专家轨迹初始化缓存，解决“冷启动”问题。
