## 摘要
有状态的工具调用 LLM 智能体将上下文窗口视为工作内存，然而目前的智能体框架对驻留性和持久性的管理仅是尽力而为，导致了反复出现的故障：压缩后的状态丢失、重置时跳过刷新以及破坏性回写。我们提出了 ClawVM，这是一个虚拟内存层，它将状态管理为具有最小保真度不变性的有类型页，在 Token 预算下提供多分辨率表示，并在每个生命周期边界进行经过验证的回写。由于框架已经负责组装提示词、协调工具并观察生命周期事件，它是天然的执行点；将契约置于此处使得驻留性和持久性变得确定且可审计。在合成工作负载、12 个真实会话追踪和对抗性压力测试中，经离线真值确认，只要最小保真度集合符合 Token 预算，ClawVM 就能消除所有策略可控故障，且每轮增加的策略引擎中位开销小于 50 微秒。

## Abstract
Stateful tool-using LLM agents treat the context window as working memory, yet today's agent harnesses manage residency and durability as best-effort, causing recurring failures: lost state after compaction, bypassed flushes on reset, and destructive writeback. We present \textsc{ClawVM}, a virtual memory layer that manages state as typed pages with minimum-fidelity invariants, multi-resolution representations under a token budget, and validated writeback at every lifecycle boundary. Because the harness already assembles prompts, mediates tools, and observes lifecycle events, it is the natural enforcement point; placing the contract there makes residency and durability deterministic and auditable. Across synthetic workloads, 12 real-session traces, and adversarial stress tests, \textsc{ClawVM} eliminates all policy-controllable faults whenever the minimum-fidelity set fits within the token budget, confirmed by an offline oracle, and adds median <50 microseconds of policy-engine overhead per turn.

### ClawVM: 面向有状态工具调用 LLM 智能体的框架管理虚拟内存 —— 论文总结

#### 1. 论文的核心问题与整体含义（研究动机和背景）
随着 LLM 智能体（Agent）在长时任务（如编码、自动化办公）中的应用，智能体需要管理跨越数百个工具调用和多个会话的复杂状态。现有的智能体框架（如 OpenClaw）将上下文窗口视为“工作内存”，但在管理状态的驻留（Residency）和持久性（Durability）时，通常采用“尽力而为”的启发式策略。这导致了三大类故障：
*   **驻留故障**：上下文压缩（Compaction）时丢失了关键指令或计划。
*   **持久性故障**：在会话重置时未及时刷新状态，或在回写时发生破坏性覆盖。
*   **可观测性故障**：检索失败时缺乏明确的错误代码，难以诊断。
论文的核心动机是借鉴操作系统（OS）的虚拟内存管理机制，在智能体框架（Harness）层建立一套确定性的、可审计的状态管理契约。

#### 2. 论文提出的方法论：核心思想与关键技术
ClawVM 的核心思想是将智能体状态抽象为**类型化页面（Typed Pages）**，并由框架强制执行内存契约。
*   **类型化页面与不变性**：将状态分为 Bootstrap（引导/策略）、Constraint（约束）、Plan（计划）、Preference（偏好）、Evidence（证据）和 Conversation（对话片段）。每种页面都有“最小保真度不变性”，规定了其在预算压力下允许降级的底线。
*   **多分辨率表示（Multi-resolution Residency）**：页面可以以四种精度存在：Full（全文）、Compressed（压缩文本）、Structured（结构化字段）、Pointer（指针/句柄）。当 Token 预算紧张时，页面按需降级而非直接丢弃。
*   **故障模型（Fault Model）**：定义了可观测的策略故障，如 `refetch`（重新获取已驱逐的状态）、`bootstrap fault`（压缩后丢失引导信息）、`flush-miss`（重置时丢失脏数据）。
*   **验证回写协议（Validated Writeback）**：在生命周期边界（如压缩、重置）执行三阶段提交：结构化暂存、确定性验证（检查架构、权限、非破坏性）、作用域提交。
*   **选择策略**：采用确定性的两阶段算法。第一阶段安装所有硬固定（Hard-pinned）页面；第二阶段根据边际效用（考虑重算成本、 recency 等）贪婪地升级页面分辨率。

    *   **真实追踪**：来自 12 个真实的 Claude Code 会话（涵盖编码、DevOps、交易、网页抓取等）。

*   **评估指标**：策略可控故障数、抖动指数（Thrash Index）、Token 效率、系统开销。

#### 4. 资源与算力
*   **算力说明**：论文未提及大规模 GPU 训练，因为 ClawVM 是一个运行在智能体框架层的系统插件。
*   **实现细节**：使用 Python 实现（约 1300 行代码），作为轻量级中间层运行。
*   **运行开销**：实验记录显示，ClawVM 策略引擎的中位开销小于 **50 微秒（μs）**，峰值内存占用小于 **83 KB**。这表明其对硬件资源的需求极低，几乎不影响推理延迟。

#### 5. 实验数量与充分性
*   **实验规模**：涵盖了 24 种配置（4 种负载 × 6 种 Token 预算水平）、12 个长达 100-200 轮的真实会话追踪、30 个任务级重放实验以及针对预算饥饿和极端抖动的对抗性压力测试。
*   **消融实验**：进行了详细的加法和减法消融实验，验证了指针解析、自动固定和生命周期回写等功能的必要性。
*   **客观性**：通过离线 Oracle（真值）对比证明了在线策略已达到故障消除的最优解。实验设计考虑了不同 Token 预算的压力，覆盖了从单会话到多会话的复杂边界，具有较高的客观性和说服力。

#### 6. 主要结论与发现
*   **故障消除**：在所有 Token 预算水平下，ClawVM 彻底消除了所有策略可控故障（从基准方法的平均 67.8 个降至 0 个）。
*   **鲁棒性**：即使在极端的生命周期压力（如每 3 轮压缩一次）下，ClawVM 依然能保持状态的完整性。
*   **策略无关性**：故障消除主要归功于其结构化约束（如硬固定和强制回写），而非特定的启发式评分权重，这证明了系统设计的稳健性。
*   **性能**：ClawVM 在显著提升可靠性的同时，引入的计算开销在智能体运行时间中可以忽略不计。

#### 7. 优点
*   **系统化建模**：首次将成熟的虚拟内存理论完整引入智能体状态管理，解决了“尽力而为”策略的随机性问题。
*   **多分辨率机制**：通过降级而非丢弃，优雅地处理了 Token 预算压力，避免了信息的全有或全无。
*   **可观测性**：通过故障模型使内存管理决策变得可审计、可重放，极大方便了开发者调试。
*   **非侵入性**：作为框架层插件，不需要重新训练模型或更换检索后端。

#### 8. 不足与局限
*   **语义正确性**：ClawVM 只能保证数据“存得进、取得出、不丢失”，但无法验证模型生成的更新内容在语义上是否正确（即模型写了错误的信息，系统也会视为有效更新）。
*   **工具模拟依赖**：重放实验依赖于工具调用的规范化签名，对于具有高度非确定性输出的复杂工具，其重放保真度可能受限。
*   **在线部署验证**：虽然进行了实时钩子验证，但大规模、多用户的在线生产环境部署数据在文中相对较少，主要侧重于确定性重放分析。
