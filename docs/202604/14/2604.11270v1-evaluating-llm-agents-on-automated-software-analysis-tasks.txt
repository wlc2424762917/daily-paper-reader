## Abstract 

Numerous software analysis tools exist today, yet applying them to diverse open-source projects remains challenging due to en-vironment setup, dependency resolution, and tool configuration. LLM-based agents offer a potential solution, yet no prior work has systematically studied their effectiveness on the specific task of 

automated software analysis , which, unlike issue solving or general environment setup, requires installing and configuring a separate analysis tool alongside the target project, generating tool-specific prerequisites, and validating that the tool produces meaningful anal-ysis outputs. We introduce AnalysisBench, a benchmark of 35 tool-project pairs spanning seven analysis tools and ten diverse C/C++ and Java projects, each with a manually constructed reference setup. Using AnalysisBench, we evaluate four agent architectures across four LLM backends. Our custom agent, AnalysisAgent, achieves 

manually verified success rates of 94% (Gemini-3-Flash, 33/35 tasks), compared to 77% for the best baseline (ExecutionAgent). Beyond quantitative results, we identify key limitations in existing agents, including stage mixing, poor error localization, and premature ter-mination, and show that agentic architecture matters more than LLM capability alone. We further find that whole-program analy-ses and symbolic execution are the most difficult tasks, that Java toolchains pose greater challenges than C/C++, and that LLM-self-validated success consistently overstates manually verified success.

## 1 Introduction 

To support developers in ensuring software quality, security, and performance, decades of research and engineering have produced a wide variety of software analysis tools . For example, developers can choose from a diverse range of static analyzers [ 2, 20 , 57 ], symbolic execution engines [ 7, 48 , 52 ], fuzzers [ 17 , 66 ], and profilers [ 12 , 21 ,23 , 61 ]. When adopted widely, these tools have a proven impact: For instance, Google’s static analysis platform Tricorder surfaces hundreds of thousands of warnings per day across the company’s

Unfortunately, applying a given analysis tool to a new project is notoriously difficult [ 3]. One part of this challenge is creating a compatible execution environment for the analysis tool itself, which may require a specific compiler version, complex dependen-cies, and non-trivial configuration. Another part of the challenge is preparing the target project for analysis, which may involve building the project with special flags, generating tool-specific pre-requisites (e.g., LLVM bitcode), or launching a runtime environment (e.g., JVM) with specific parameters. Typically, developers want to perform these tasks in an isolated, containerized environment to avoid polluting their local setup and ensure reproducibility. This environment needs to be compatible with both the analysis tool and the target project, which adds further complexity. Finally, de-velopers must validate that the analysis actually ran and produced meaningful outputs (e.g., warnings, test cases, call graphs), as a mis-configured setup may skip most or all of the project’s code and give a false sense of success. Due to these challenges, adoption of analy-sis tools in practice remains limited. Prior work has documented that the usability and setup burden of static analyzers is a major barrier to their adoption [ 3 , 32 ], and similar observations have been made for fuzzers [17] and symbolic execution engines [7]. Given the multi-step, environment-intensive nature of auto-mated software analysis, LLM-based agents are a natural fit: they can plan and execute shell commands, interpret error messages, and iteratively adapt to unforeseen failures. Such agents have al-ready demonstrated success in related tasks, including issue solv-ing [ 28 , 60 , 62 , 67 ], automated program repair [ 5 , 10 , 45 ], web au-tomation [ 60 ], and environment construction [ 4 , 6, 16 , 41 ]. However, end-to-end execution of software analysis tools remains underex-plored. Unlike project-only environment setup [ 6 ], automated soft-ware analysis requires the agent to install and configure a separate analysis tool alongside the target project, produce tool-specific pre-requisites (e.g., LLVM bitcode, Java bytecode), and validate that the tool produces meaningful analysis outputs rather than merely run-ning without errors. As we show in our evaluation, simply changing the task description in an existing agent yields substantially lower success rates than a purpose-built agentic architecture (e.g., the best baseline averages 57% verified success across LLM backends vs. 79% for AnalysisAgent, a gap of 20 percentage points). To systematically study the capabilities and limitations of LLM-based agents on this task, we construct AnalysisBench , the first benchmark for evaluating LLM-based agents on end-to-end auto-mated software analysis . AnalysisBench evaluates an agent’s ability to set up and execute diverse analysis tools on open-source projects from scratch, spanning the full pipeline from environment con-struction to evidence validation. ## 2 AnalysisBench 2.1 Task Definition 

The automated software analysis task is about automatically apply-ing a software analysis tool to a real-world software project. The task is to be performed end-to-end, i.e., starting from a fresh envi-ronment and finishing with verifiable evidence that a meaningful analysis has been successfully completed. 

Inputs. Each task is defined by a tool-project pair (𝑇 , 𝑃 ) and an execution interface. 

• Tool specification 𝑇 . Tool name and acquisition method (e.g., up-stream repository URL or a released package). 

• Target project 𝑃 . Repository URL and pinned revision (commit hash or tag) of the target project to be analyzed. 

• Execution interface. A writable workspace, environment interac-tion tools (e.g., terminal), and a budget ( 5 h , $2 per task) ensuring comparable evaluation.

Terminology. Throughout this paper, we use artifacts to refer to concrete build and analysis outputs (e.g., bitcode files, analysis reports), and evidence to refer to artifacts that demonstrate mean-ingful analysis completion on the target project. 

Task objective. Given (𝑇 , 𝑃 ), the agent must (i) provision an isolated Docker container, (ii) install the tool and its dependencies, (iii) fetch and build (when required) the target project, (iv) execute the analysis tool on the target project, and (v) emit evidence that the analysis tool processed project-specific inputs (e.g., source files) and produced tool-specific output artifacts (e.g., warnings) beyond trivial outputs such as tool version strings or help messages. 

Outputs. A successful run produces an evidence package: 

• Reproducible environment. E.g., a Dockerfile 

• Complete execution trace. The full transcript of executed com-mands, including stdout/stderr, return codes, and key environ-ment metadata (tool version, OS image, commit hashes). 

• Tool-specific analysis artifacts. Logs and output files from execut-ing 𝑇 on 𝑃 (e.g., warnings, test cases, call graphs). 

Evidence-based success criteria. We consider a task solved successfully if the environment is reproducible, the tool is installed and runnable, the target project is built and prepared, the tool is invoked on project-relevant inputs, and the outputs contain verifi-able, project-specific analysis evidence (e.g., warnings referencing project paths, generated test cases, or call graphs). We operational-ize these criteria through tool-specific validation checks detailed in §2.3 and discuss limitations in §5.

## 2.3 Validation of Success 

A central challenge in evaluating agentic automation of software analysis is distinguishing genuine end-to-end analysis from su-perficial “success” signals. For example, an agent might install an analysis tool, run its –help command, and then declare success without actually analyzing the target project. To thoroughly check for meaningful analysis, we manually construct reference artifacts and validation mechanisms for each tool-project pair. 

Reference environment construction . For each (𝑇 , 𝑃 ) pair, we manually construct a containerized reference setup that installs 𝑇 ,clones 𝑃 , performs the required build steps, and runs the analysis under the benchmark budget. 

Reference artifacts . We execute each reference setup to pro-duce a reference evidence package: full logs, key intermediate build outputs, and tool-specific analysis artifacts (e.g., static analysis re-ports, call graphs, generated tests, fuzzing queues/crashes, or JVM diagnostic snapshots). These artifacts define what a meaningful execution looks like for the (𝑇 , 𝑃 ) pair. 

Validation process . When an agent completes a task, we re-execute the agent’s produced setup (e.g., Dockerfile and scripts) to confirm reproducibility, and compare the agent’s artifacts against the reference evidence package. To further validate the analysis run, we may extend the analysis beyond the limit the agent has chosen. Outlier or suspicious values (e.g., unusually low coverage, empty reports, or results inconsistent with the reference) are flagged and discussed among authors. To support reproducibility, our replica-tion package includes the detailed validation protocol. 1 Specifically, we check: 

• Structural evidence: Expected files and directories exist (KLEE’s 

klee-out directory, AFL++’s queue directory, etc.). 

• Project references: Logs or reports reference project-specific paths, symbols, or build artifacts. 

• Semantic evidence: Tool-specific indicators of analysis progress, such as partial or completed path exploration, AFL coverage growth, generated call graphs, or static analysis warnings. This reference-backed validation supports human auditing and helps spot premature termination by requiring artifacts that are dif-ficult to produce without running analysis on the target repository. 

## 3 Agents for Automated Analysis 

To study the extent to which LLM-based agents can perform the au-tomated software analysis task defined in §2, we evaluate four agent architectures on AnalysisBench. We first describe three baseline agents adapted from prior work (§3.1), then analyze their common failure modes (§3.2), and finally introduce AnalysisAgent, a custom agent designed to address these limitations (§3.3). Across all agents, we provide the same task specification (§2) and the same budget (5 h and $2 per task). The key difference is how each agent plans and executes its actions and when it decides to stop. All baseline agents rely on the LLM’s own judgment to decide when to stop, without explicit output validation checks.

## 3.1 Baseline Agents 

RAG-Agent. Retrieval-augmented generation (RAG) is a common paradigm to enhance LLM capabilities by retrieving external knowl-edge [ 34 ]. We re-implement this paradigm for the automated soft-ware analysis task in RAG-Agent , following the design of a RAG agent described by LangChain [ 15 ] but using LiteLLM to support multiple LLM backends. We include RAG-Agent to represent the “plan-then-execute” paradigm that is common in LLM-based au-tomation pipelines. Unlike the other agents, RAG-Agent does not interactively debug inside the container; instead, it synthesizes com-plete scripts upfront and can only revise them based on execution feedback, without fine-grained, command-level interaction. The agent first issues web queries to collect installation and invocation guidance, e.g., in the form of tool READMEs, GitHub issues, and tutorials. Then, the agent synthesizes executable artifacts: a Docker-file, a setup script that installs dependencies and builds the project, and a launch script that runs the analysis. 

Adaptation. We seed retrieval with the tool name, its upstream repository, the project name, and its repository URL. 

Mini-SWE-Agent. An agent architecture designed for software engineering tasks [ 38 , 62 ]. Instead of producing a single large script, it alternates between reasoning and tool actions (shell commands, file edits), updating its plan based on observed errors. This style is well suited to incremental debugging and refinement. While conceptually simple, Mini-SWE-Agent has demonstrated strong performance on software engineering tasks and is one of the leading openly available agents on SWE-bench Verified [31]. 

Adaptation. We modify the prompt to specify the automated soft-ware analysis task. 

ExecutionAgent. ExecutionAgent [ 6] is an environment-setup agent designed to build a given project and execute its test suite in a container environment. Compared to Mini-SWE-Agent, it priori-tizes creating a runnable container specification and then executing the workload within that environment. 

Adaptation. We disable ExecutionAgent’s test-specific retrieval mechanisms (e.g., test-suite discovery and CI workflow parsing), which are not applicable to the analysis task. 

## 3.2 Limitations and Failure Trends 

While the baselines differ in planning and execution style, early experiments reveal three common limitations when applied to au-tomated software analysis. 

1) Missing procedural structure leads to stage mixing. All baselines occasionally attempt operations in the wrong order, such as cloning a repository into a non-existent container context, in-voking analyzers before the tool’s dependencies are installed or before a build artifact of the project exists. Because error messages are often locally scoped (e.g., missing binary, missing path, miss-ing library), agents frequently respond with additional installation attempts instead of correcting the underlying ordering mistake. 

2) Verbose logs obscure root causes, impeding recovery. Long logs produced when executing commands may obscure the root cause of a failure. RAG-Agent in particular suffers from this prob-lem due to its monolithic scripts, but Mini-SWE-Agent and Exe-cutionAgent also occasionally face this challenge. In all cases, poor error localization reduces the agent’s ability to form targeted hy-potheses (e.g., “the project uses CMake but container is missing 

ninja-build ”), wasting cycles on uninformed trial and error. 

3) Weak success validation causes premature termination. 

All baselines sometimes stop after observing superficial signals of success. Common examples include: 

• Tool presence only: Commands like klee --version or infer --help succeed, and hence, the agent declares success. 

• Toy execution: The tool runs on a trivial input (e.g., a small file or example program), but not on the target project. 

• Partial pipeline: The project builds successfully, but the analysis is skipped or misconfigured, producing no tool-specific results. 

Example 1: Stage mixing and environment thrashing (Mini-SWE-Agent, CSA + curl, Gemini-3-Flash). Mini-SWE-Agent correctly re-ports 3 warnings, yet the run is recorded as a failure. Stage mixing causes the agent to interleave package installation with analysis attempts, wasting 17 of 25 iterations on dpkg lock contention from concurrent package-manager invocations. Although the analysis does complete, a stale Docker-cleanup status message misleads the agent into believing the environment has failed, so it never recognizes its own success. This example illustrates how stage mix-ing (limitation 1) wastes cycles and how weak success validation 

(limitation 3) causes the agent to discard valid results. 

Example 2: Repetitive cycling without progress (RAG-Agent, WALA + Checkstyle, DeepSeek-V3.2). RAG-Agent uses all 120 iterations trying to build WALA from source (Maven, Gradle, different JDKs, hallucinated version tags) without ever diagnosing the root cause (missing C++ toolchain for WALA’s cast module). This example illustrates poor error localization (limitation 2). 

## 3.3 AnalysisAgent 

Motivated by these limitations, we introduce AnalysisAgent, a custom agent built on three principles: staged execution, single-action cycles with log condensation, and evidence-based validation. 

3.3.1 Principle 1: Explicitly Staged Execution. The agent follows a four-stage linear progression with per-stage instructions: (1) Docker setup . Writes a Dockerfile and auxiliary files. No con-tainer exists yet, so terminal commands are disabled. The frame-work enforces a tool whitelist, allowing only write_to_file 

and read_file ; any other action is rejected. The agent designs a compatible base image and installs build dependencies. (2) Tool setup . The framework automatically advances to this stage once the container is running and responds to a shell command. In this stage, we instruct the agent to install the analysis tool and conduct a successful smoke test of the analysis tool, e.g., by invoking it on a small toy example or showing --help .(3) Project setup . We give the agent instructions to build the project and generate tool-specific artifacts (e.g., compilation database Evaluating LLM Agents on Automated Software Analysis Tasks Conference’17, July 2017, Washington, DC, USA 

for static analyzers or bitcode for KLEE). We instruct the agent to check the availability of such artifacts before switching stage. (4) Analysis run . In this stage, the agent is instructed to apply the installed tool to the project (i.e., not only toy examples) and pro-vide the outputs in a results directory. When the agent declares the task complete, an LLM-as-judge validates the claim; if the judge rejects, execution continues if cycles remain (Principle 3). The transition from Stage 1 to Stage 2 is automatic: the framework advances once the Docker container is running and responsive. Between later stages (2 →3, 3 →4), the agent determines when stage goals are met based on its assessment of whether progress has been made as specified in the prompt. In each stage, the prompt carries over a summary of what hap-pened in previous stages (e.g., which versions of the tool and its dependencies were installed, compilation results, preview of latest actions). The carry-over information contains recent commands from different stages and a synthesized summary obtained by a separate LLM query. While staged pipelines are common in CI/CD, our decomposition is tailored to automated software analysis. 

3.3.2 Principle 2: Single-Action Cycles with Deterministic Log Con-densation. Weaker LLMs struggle to express correctly formatted commands. AnalysisAgent therefore triggers actions in two steps. First, the agent receives stage context, recent observations, and structured feedback from prior failures, and outputs free-form rea-soning with proposed next steps. Second, a separate LLM call ex-tracts the first concrete action from this response, formats it as an executable command (following a per-tool JSON schema that we define for each agentic tool), and the system executes it, returning the observation (stdout/stderr, exit code, environment metadata) for the next cycle. Invalid actions (e.g., stopping the container, host-to-container copies) are rejected by the framework with an error message returned to the agent, and the cycle is consumed without effect. As build systems and analysis tools may produce thousands of lines of output, we need to condense outputs. Unlike prior work that relies on LLM-based summarization [ 6 ], AnalysisAgent uses deterministic pattern matching to extract diagnostic signals: the first 1,000 characters (command + initial progress), lines matching any of 70 manually curated failure patterns (e.g., compiler/linker errors, missing files, package errors), and the last 1,000 characters (exit status + summary). The action-observation loop follows the ReAct paradigm [ 64 ]; the key differences are the separation of plan-ning from extraction and the deterministic log condensation, both tailored to the long, noisy outputs of build systems and tools. 

3.3.3 Principle 3: Evidence-Based Validation. When the agent de-clares the task complete, the system constructs an evidence package 

comprising three components: (1) stage summaries (successful/-failed commands, environment choices), (2) recent analysis-stage observations, and (3) output file locations. This package is then sub-mitted to an independent LLM-as-judge . The judge uses the same LLM backend as the agent. To mitigate circular bias, the validator operates on the structured package (not raw conversation history), uses a separate prompt focused solely on output validation, and has no access to the agent’s reasoning or conversation context. The judge verifies that (i) the invocation targets the specified project and (ii) output artifacts consistent with tool completion exist (e.g., KLEE

## 4.1 Experimental Setup 

Configuration. We evaluate all agents on the full Analysis-Bench benchmark of 35 tasks (§2.2) using four LLM backends of varying capability and cost (Table 3). All agents share the same, per-task stopping limits: a maximum of 120 agentic cycles, a $2 API cost cap, and a 5 h wall-clock timeout; reaching any limit interrupts the task and marks it as failed. For agents supporting retries (Analysis-Agent, ExecutionAgent), each retry counts as additional cycles within the same per-task budget. We repeat each configuration 

𝑛 = 3 times to account for non-determinism. 

Success metrics. We assess task outcomes with two complemen-tary metrics. First, a task is self-validated if the agent terminates and claims task completion. For AnalysisAgent, this includes passing its internal LLM-as-judge validation (§3.3); for baseline agents, it means the agent exited without error and reported success. Self-validated success may include false positives. To account for false positives, we inspect self-validated cases and call a task manually verified if it passes the manual validation process described in §2.3. 

Statistical testing. We perform pairwise significance tests bet-ween AnalysisAgent and each baseline. For self-validated success, where both agents are evaluated on the same 35 tasks, we use Mc-Nemar’s one-sided exact test [ 39 ] to check whether AnalysisAgent can solve a task more often than the baselines in cases when one agent solves it and the other does not. For verified success, where only one run is available, we use Fisher’s one-sided exact test to check whether AnalysisAgent solves more tasks. All 𝑝 -values are corrected for multiple comparisons (3 baselines × 4 LLMs) using the Holm-Bonferroni method [ 25 ] at 𝛼 =0.05 . We also test each agent’s absolute success rate against chance ( 𝐻 0 : 𝑝 ≤ 0.5) via exact bino-mial tests with 95% Wilson confidence intervals. Beyond 𝑝 -values, we report Cohen’s ℎ for effect size, odds ratios with 95% confidence intervals, and the Cochran-Mantel-Haenszel test to confirm that results hold across LLM backends. 

## 4.2 RQ1: Effectiveness Across Agents and LLMs

Verified success. Manually verified success rates span a wide range across agent architectures: RAG-Agent averages only 10% across LLM backends (range 3–23%), showing that naive retrieval-based prompting is insufficient for multi-step analysis workflows. Mini-SWE-Agent, a general-purpose coding agent, reaches an aver-age of 37% but exhibits high LLM sensitivity (9–63%), suggesting that without task-specific scaffolding, effectiveness depends heavily on the model’s capabilities. ExecutionAgent’s tool-aware adapta-tions raise the average to 57%, with less variation across LLMs (40–77%). AnalysisAgent achieves the highest manually verified success, averaging 79% across LLM backends (range 54–94%). It also achieves the best overall verified success rate of 94% with both Gemini-3-Flash and DeepSeek-V3.2 (33 of 35 tasks), demonstrating that purpose-built scaffolding can reliably handle the multi-step workflows required to install tools, build target projects, and pro-duce meaningful analysis evidence. We highlight Gemini-3-Flash in the abstract because it also achieves the lowest mean time (36 minutes) among the two tied backends.

4.3.2 Analysis Outputs. Beyond binary success, we examine what

Suc-cessful runs produce meaningful analysis evidence across all seven tools: Infer reports issues spanning thread safety, resource leaks, and null dereferences, while WALA constructs call graphs with a median of 39K nodes and 4.3M edges. 4.3.3 Failure modes. We classify 182 failing trajectories (of 560 ver-ified instances) into root-cause categories, excluding infrastructure crashes and missing logs. Categories are non-exclusive (28% have multiple), so percentages in Figure 4 may exceed 100%: 

• Docker/build failure (67): The agent is stuck in a Docker-build loop, fails to install the analysis tool, or cannot compile the target project, never reaching the analysis stage. 

• Analysis tool misuse (73): The analysis tool is installed, but the agent invokes it incorrectly (wrong API classes, stale imports, wrong flags or input formats) or cannot produce required prereq-uisites, such as LLVM bitcode or a fuzzing harness. 

• Malformed LLM output (42): The LLM produces output that the agent framework cannot parse as a valid tool call, causing cycles to be consumed without any action. 

• Budget/time exhausted (34): The iteration budget or wall-clock limit is exhausted, typically co-occurring with other categories. 

• Incorrect analysis result (21): The agent completes execution and submits a result, but the output is rejected by the validator or found to be incorrect during manual validation. Each agent exhibits a distinct failure profile (Figure 4). RAG-Agent and ExecutionAgent primarily fail during environment con-struction: RAG-Agent is dominated by Docker/build failures (60%), reflecting its tendency to exhaust all 120 Dockerfile-generation iterations without learning from repeated build errors; Execution-Agent is dominated by malformed LLM output (50%), followed by Docker/build failures (20%). The malformed-output failures are concentrated on DeepSeek-V3.2, which frequently returns empty re-sponses that the framework cannot parse as valid tool calls, silently consuming cycles without progressing past environment setup. Mini-SWE-Agent fails at output quality and tool usage: Incorrect analysis results (48%) and analysis tool misuse (44%) dominate, as the agent often completes the workflow but produces inadequate analysis results or misuses the tool. Finally, AnalysisAgent has largely solved environment setup (only 7% of its failures) and in-stead fails during tool invocation: Analysis tool misuse (50%) is the leading cause, driven by incorrect WALA API usage, failed Infer build integration, and KLEE bitcode extraction failures. Two recur-ring signatures stand out: 33% of failing runs involve compile-fix loops (3+ consecutive compilation errors without resolution) and 23% include repeated validation rejections (5+ rejections); these thresholds describe observed behavior, not tuned parameters.

## 4.4 RQ3: Efficiency and Costs 

Figure 5 reports per-task resource consumption broken down by agent and LLM backend, covering monetary cost (computed from per-token pricing) and wall-clock time. 

High variance across tasks . Figure 5 shows large standard deviations relative to the means across nearly all metrics. For exam-ple, ExecutionAgent with DeepSeek-V3.2 averages $1.34 per task but with a standard deviation of $0.46, and its token consumption varies substantially ( 2351 K ± 817 K). This high variance implies that per-task cost is difficult to predict from the agent-model pair alone: Individual task characteristics (such as build-system com-plexity, dependency resolution difficulty, and whether the agent encounters early failures that trigger retries) appear to strongly in-fluence the cost profile. From a practical standpoint, this means that budget planning for LLM-based analysis pipelines must account for particularly expensive tasks and not average only. 

Capability-efficiency interaction . Model capability and ef-ficiency are tightly coupled. The weakest backend (GPT-5-nano) consumes the most calls and wall-clock time (mean 78 calls and 144 minutes per task), despite having the lowest per-token pricing. Stronger backends substantially reduce interaction length and ex-ecution time: DeepSeek-V3.2 achieves a mean of 56 calls and 47 minutes per task, and Gemini-3-Flash reaches 36 minutes mean time. Hence, token price alone is not a reliable proxy for per-task cost, as weaker models require more cycles and more tool executions. 

Failure cost and model sensitivity . Failed runs are consis-tently more expensive than successful runs, with 2.77 × more cycles, 4.07 × longer duration, 1.27 × higher cost, and 1.76 × more retry attempts. The magnitude of this inflation varies by model: Gemini-3-Flash shows the highest sensitivity (3.3 × more cycles on failure vs. success) despite the lowest absolute count on success ( ∼30 cy-cles), while DeepSeek-V3.2 degrades more gracefully (2.0 ×). In other words, a failing run does not just miss the goal: It consumes far more time and money than a successful one. Choosing a weaker or cheaper model does not save cost if it also fails more often. Similarly, reducing the timeout could curb the overhead of failed runs, but at the risk of terminating runs that would have eventually succeeded, potentially lowering overall success rates. 

Language ecosystem differences . Java tasks are systematically more resource-intensive than C/C++ (e.g., mean 189 vs. 110 minutes and 269 vs. 176 calls for AnalysisAgent with GPT-5-nano; the same trend holds across all agent–model combinations), consistent with the higher failure rates reported in §4.2. We attribute this gap to the Java analysis toolchain rather than to the language itself. First, the Java tools in our benchmark carry heavier setup dependencies (e.g., Infer requires Buck, opam, OCaml, JDK, and Clang/LLVM), whereas several C/C++ tools have lighter dependency chains (e.g., cflow requires only Autotools). Second, the Java tools universally require a successful full build of the target project before analysis can begin: Infer and WALA operate on bytecode, and SJK must attach to a running JVM, while C/C++ tools, such as cflow and CSA, can operate more directly on source or integrate into the compi-lation step. Third, the Java tools demand more complex runtime orchestration. For instance, SJK requires launching the target as a background process and attaching the profiler, a coordination pattern absent from the C/C++ tools. 

## 5 Discussion 5.1 Threats to Validity 

Our benchmark covers seven tools and ten projects, which may not represent the full diversity of real-world analysis scenarios. In particular, (i) all target projects are command-line programs with well-defined entry points, whereas analyzing libraries may require additional harness construction; (ii) we focus on open-source tools, excluding commercial analyzers (e.g., Coverity) that may have dif-ferent setup characteristics; and (iii) our project selection is limited to C/C++ and Java, leaving other ecosystems (e.g., Python, Rust) unexplored. Moreover, the 35 tasks are not statistically indepen-dent because multiple tasks share projects and tools. With 𝑛 =3

repetitions, variance estimates are rough, particularly for weaker models (e.g., ExecutionAgent with GPT-5-nano: 30 ± 37 %), where non-determinism is highest; we report standard deviations for trans-parency but caution against over-interpreting them. Manual validation was performed by a single primary validator who re-executed each agent setup and compared results against the reference artifacts; outlier cases were discussed among authors. While we did not measure formal inter-rater agreement, the vali-dation relies on reproducible, artifact-based checks (file existence, Conference’17, July 2017, Washington, DC, USA Islem Bouzenia, Cristian Cadar, and Michael Pradel 

project-specific references, tool output comparison) rather than subjective judgment, and all verdicts, logs, and rubrics are included in the replication package for independent verification. We measure success through tool-specific checks rather than numeric thresholds (e.g., we do not require a minimum number of KLEE paths or AFL++ corpus entries). This binary criterion eval-uates whether the agent-produced setup is correct and functional ,not whether the analysis itself is exhaustive. The low medians in Table 5 (e.g., AFL++ 0.4% coverage, KLEE 2 test cases) reflect short runtimes (30–60 s for dynamic tools), not setup quality; we re-run borderline setups with extended budgets to confirm functionality. For instance, we let two AFL++ setups run for several hours and observed crashes and hangs, confirming that the agent-produced harnesses and configurations are correct and that longer runtimes yield deeper analysis. A tiered metric (e.g., partial vs. full comple-tion) is a natural extension, but evidence of a correct, functional setup is already a notable achievement given the task complexity. 

## 5.2 Implications and Future Work 

For practitioners. LLM agents can automate the deployment of analysis tools with high reliability when using strong models and structured agent architectures. For the best-performing configu-rations (AnalysisAgent with Gemini-3-Flash or DeepSeek-V3.2), the mean cost remains under $1 per task (Figure 5), making this practical for CI/CD integration. Users should, however, incorporate evidence validation rather than relying on self-validated success. 

For researchers. AnalysisBench can serve as a testbed for evaluat-ing new agent architectures and LLM capabilities on environment-intensive tasks. Extending the benchmark to additional tools, lan-guages, and project types (including libraries) is a natural next step. The most impactful next step is a controlled user study comparing agent-assisted setup time to fully manual setup, which would quan-tify the practical time savings for developers. Another promising direction is to explore adaptive agent strategies that detect early failure patterns (e.g., compile-fix loops) and dynamically switch strategies or reallocate budget to mitigate wasted resources. Finally, we envision AnalysisAgent enabling large-scale studies of analysis tools across many codebases. 

## 6 Related Work 

Environment construction and build reproducibility. Building soft-ware reliably and reproducibly is a well-known problem [ 37 ]. Sev-eral techniques to address this problem have been proposed, in-cluding a technique for fixing dependency errors to make Python builds reproducible [ 42 ] and a learning-based approach to localize and fix build errors in Java projects [ 56 ]. Reproducibility challenges affect not only project builds but also, e.g., bug datasets [ 69 ] and computational notebooks [ 59 ]. A related problem is to identify and install dependencies required to run code, e.g., addressed by DockerizeMe [ 26 ], which infers dependencies required to execute a Python code snippet without import errors. Our work targets multiple languages and build ecosystems, and it focuses on the broader task of applying analysis tools, not just building projects. 

Agents for environment setup. ExecutionAgent introduced the idea of delegating the task of setting up an execution environment of a given project to an LLM-based agent [ 6]. Other recent work further explores this idea [ 16 , 29 , 41 , 63 ] and uses it as a first step toward creating SWE-bench-like [ 31 ] benchmarks of repository-level tasks [ 58 , 63 ]. Similar to our work, these approaches use LLM-based agents to create a containerized environment and set up a given project inside it. In contrast, our work requires setting up not just the project but also a separate analysis tool with its own dependencies, and applying it to produce verifiable results. This dual requirement (tool installation, artifact preparation, analysis invocation) explains the higher failure rates vs. prior work [ 6]. Our evaluation directly compares to ExecutionAgent, showing Analysis-Agent outperforms it. 

Other software engineering agents. LLM-based software engineer-ing agents have attracted considerable attention in recent years. Issue solving and program repair are among the most prominent tasks, with notable examples including SWE-Agent [ 62 ], RepairA-gent [ 5 ], AutoCodeRover [ 67 ], OpenHands [ 60 ], FixAgent [ 33 ], AgentCoder [ 30 ], Magis [ 55 ], MarsCode Agent [ 36 ], and Trae Agent [ 18 ]. Other tasks addressed by software engineering agents include generating issue-reproducing tests [ 1 , 10 , 43 , 44 ], root cause analysis [ 46 ], and debugging computational notebooks [ 22 ]. Our work targets automated software analysis, introduces a new benchmark, and proposes an agent design tailored to this task. 

Agent design and prompt engineering. ReAct [ 64 ] improves agent reasoning via interleaved reasoning and acting, and Reflexion [ 49 ]via verbal self-reflection. Our single-action cycle design aligns with the goal of immediate feedback. The staged structure of Analysis-Agent is complementary to these techniques and future work may integrate self-reflection into AnalysisAgent. 

Benchmarks. AnalysisBench contributes a novel task and bench-mark to the growing space of benchmarks for evaluating LLMs and LLM agents. Other benchmarks include SWE-bench [ 31 ] for issue solving, AgentBench [ 35 ] for interactive tasks, SUPER [ 4 ] for setting up and executing tasks from research repositories, NoCode-bench [ 13 ] for documentation-based code editing, benchmarks for assessing code reasoning abilities [ 14 ] and code generation abili-ties [ 65 ], and several benchmarks for reproducing and replicating scientific results [ 4, 27 , 50 , 51 ]. In addition to benchmarking LLMs, prior work has proposed benchmarks for evaluating program anal-ysis tools, e.g., fuzzers [ 24 , 40 ]. While some existing work requires agents to set up environments and execute tasks, AnalysisBench fo-cuses specifically on automated software analysis : the agent must in-stall a separate analysis tool, prepare tool-specific prerequisites (e.g., LLVM bitcode, compilation databases), and validate that the tool produces meaningful analysis outputs on the target project. This dual setup requirement (tool + project) and the need for domain-specific output validation distinguish our benchmark from prior agent evaluation suites. 

## 7 Conclusion 

We introduce the task of automated software analysis and present AnalysisBench, a benchmark of 35 tool-project pairs, together with AnalysisAgent, a purpose-built agent that achieves 94% verified success. Our evaluation across four agent architectures and four LLM backends shows that task-specific architecture provides gains Evaluating LLM Agents on Automated Software Analysis Tasks Conference’17, July 2017, Washington, DC, USA 

that model scaling alone does not, that self-validated success is unreliable without evidence-based validation, and that tool-side us-ability (especially Java toolchains) remains a bottleneck that neither stronger models nor better scaffolding fully resolve.
