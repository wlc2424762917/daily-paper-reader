## 摘要
代码智能体正在飞速发展，但对其进行调试也变得日益困难。随着框架在复杂任务中编排并行工具调用和多阶段工作流，智能体的状态转换和错误传播变得难以观察。在这些运行过程中，早期的失误可能会使智能体陷入无意义的循环，甚至级联成根本性错误，形成隐蔽的错误链，从而难以判断智能体何时偏离轨道以及原因。现有的智能体追踪分析要么侧重于简单的交互，要么依赖小规模的人工检查，这限制了它们在实际编码工作流中的可扩展性和实用性。我们提出了 CodeTracer，这是一种追踪架构，它通过不断演进的提取器解析异构运行产物，将完整的状态转换历史重构为具有持久化内存的层级追踪树，并执行故障起始定位，以精准识别故障源头及其下游链条。为了实现系统性评估，我们从四个广泛使用的代码智能体框架在多种代码任务（如错误修复、重构和终端交互）中生成的大量执行轨迹中构建了 CodeTraceBench，并在阶段和步骤级别上为故障定位提供监督。实验表明，CodeTracer 的表现显著优于直接提示和轻量级基准模型；此外，在相同的预算下，回放其诊断信号能够稳定地恢复原本失败的运行。我们的代码和数据已公开。

## Abstract
Code agents are advancing rapidly, but debugging them is becoming increasingly difficult. As frameworks orchestrate parallel tool calls and multi-stage workflows over complex tasks, making the agent's state transitions and error propagation hard to observe. In these runs, an early misstep can trap the agent in unproductive loops or even cascade into fundamental errors, forming hidden error chains that make it hard to tell when the agent goes off track and why. Existing agent tracing analyses either focus on simple interaction or rely on small-scale manual inspection, which limits their scalability and usefulness for real coding workflows. We present CodeTracer, a tracing architecture that parses heterogeneous run artifacts through evolving extractors, reconstructs the full state transition history as a hierarchical trace tree with persistent memory, and performs failure onset localization to pinpoint the failure origin and its downstream chain. To enable systematic evaluation, we construct CodeTraceBench from a large collection of executed trajectories generated by four widely used code agent frameworks on diverse code tasks (e.g., bug fixing, refactoring, and terminal interaction), with supervision at both the stage and step levels for failure localization. Experiments show that CodeTracer substantially outperforms direct prompting and lightweight baselines, and that replaying its diagnostic signals consistently recovers originally failed runs under matched budgets. Our code and data are publicly available.

随着大模型代码智能体（Code Agents）在处理复杂软件工程任务（如修复 Bug、重构代码）时变得日益复杂，其内部的**状态转换和错误传播过程**变得极难观察。
*   **痛点**：智能体在长序列执行中，早期的微小失误（如环境配置错误、错误的假设）会通过级联效应导致最终失败，形成“隐蔽错误链”。
*   **现状**：现有的评估仅关注最终结果（成功/失败），缺乏对过程的诊断；现有的追踪分析要么太简单，要么依赖难以规模化的人工检查。
*   **目标**：实现代码智能体执行轨迹的自动化、结构化追踪，并精准定位故障发生的起始点（Failure Onset Localization）。

### 2. 方法论：CodeTracer 架构
CodeTracer 将异构的智能体运行日志转化为可理解、可诊断的结构化数据，包含三个关键阶段：
*   **演进式提取（Evolving Extraction）**：针对不同智能体框架生成的杂乱文件目录，系统会自动识别布局并适配/合成解析器，将原始日志标准化为包含动作、观察、代码差异（Diff）和验证结果的“规范化步骤”。
*   **树状索引（Tree Indexing）**：将扁平的步骤序列重构为**分层追踪树（Hierarchical Trace Tree）**。
    *   **探索节点**：仅查看环境而不改变状态的操作。
    *   **状态改变节点**：修改代码或环境的操作，会触发树的分支转换。这种结构能清晰展示智能体是在何种上下文背景下做出的决策。
*   **故障定位与诊断（Diagnosis）**：遍历追踪树，通过结构化查询定位“故障关键阶段”和“错误相关步骤”，并生成诊断证据。
*   **反思重放（Reflective Replay）**：将诊断出的错误信号作为提示词反馈给智能体，使其在相同预算下尝试修复之前的错误。

### 6. 主要结论与发现
*   **证据-行动鸿沟（Evidence-to-action gap）**：智能体经常能通过探索找到正确的诊断证据，但却无法将其转化为正确的修复行动。
*   **迭代收益递减**：智能体的成功率通常在 40 次迭代左右达到饱和，增加更多步数往往只会导致无效的循环，而非能力的提升。
*   **错误模式迁移**：环境错误多发生于早期，而错误的假设和验证误读则集中在后期。
*   **诊断有效性**：CodeTracer 的 F1 分数显著优于基线（最高达 48%），且通过重放诊断信号，能显著提升原本失败任务的成功率（Pass@1）。

### 7. 优点
*   **结构化创新**：提出的分层追踪树（Trace Tree）有效地解决了长程轨迹难以导航的问题。
*   **实用性强**：不仅能诊断，还能通过“反思重放”直接提升智能体的性能。
*   **基准贡献**：CodeTraceBench 为社区提供了首个大规模、步骤级标注的代码智能体诊断数据集。
*   **跨框架兼容**：演进式提取机制使其能够处理各种不同格式的智能体日志。

### 8. 不足与局限
*   **标注主观性**：尽管有标注指南，但对于“错误关键步骤”的判定仍带有一定的人为解释偏差。
*   **离线评估限制**：目前的重放实验是在预先收集的轨迹上进行的，可能无法完全模拟在线交互中环境的动态变化。
*   **应用范围**：主要集中在软件工程和终端交互领域，对于其他类型的智能体（如网页浏览、多模态智能体）的适用性尚未验证。
*   **成本问题**：虽然比人工便宜，但对于极长轨迹的深度诊断仍需消耗大量的 LLM Token。
