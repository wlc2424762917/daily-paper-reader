# Beyond Static Sandboxing: Learned Capability Governance for Autonomous AI Agents 

Bronislav Sidik Lior Rokach 

Institute for Applied AI Research Faculty of Computer and Information Science Ben-Gurion University of the Negev, Beer Sheva, Israel 

sidik@post.bgu.ac.il liorrk@post.bgu.ac.il 

## Abstract 

Autonomous AI agents built on open-source runtimes such as OpenClaw expose every available tool to every session by default, regardless of the task at hand. A summarisation task receives the same shell execution, subagent spawning, and credential-access capabilities as a code deployment task — a 15 × over-provision ratio we term the capability overprovisioning problem . Existing defences (Nemo-Claw’s container sandbox, Cisco DefenseClaw’s skill scanner) address contain-ment and threat detection, but neither learns the minimum viable capability set for each task type. We present Aethelgard , a four-layer adaptive governance framework that enforces the principle of least privilege for AI agents via a learned policy. Layer 1 (Capa-bility Governor) dynamically scopes which tools the agent is aware of per session. Layer 3 (Safety Router) intercepts every tool call before execution using a hybrid rule-based and fine-tuned LLM classifier. Layer 2 (RL Learning Policy) trains a PPO policy on the accumulated audit log to learn the minimum viable skill set per task type. On a live OpenClaw v2026.3.28 deployment with DeepSeek-chat as the agent LLM, Aethelgard achieves: (i) 73% tool reduction and 100% dangerous tool elimination for summarisation tasks; (ii) a Skill Economy Ratio improvement of +260% (synthetic) and +337% (real sessions) over the uncontrolled base-line after PPO training ( <5 minutes, RTX 3090); (iii) across an automated N=500 evaluation (400 benign + 100 adversarial, seed=42), 26.2% of all 447 intercepted tool calls were blocked, exec and sessions spawn were blocked at 100%, and 92% of adversarial tasks were neutralised by the model-plus-infrastructure stack. The evaluation dataset (N=500, seed=42) and trained arte-facts are available upon request. Code and updates: https://github.com/ sidikbro/aethelgard-docs 

## 1 Introduction 

The rapid adoption of autonomous AI agents has outpaced the security infrastructure designed to govern them. OpenClaw OpenClaw Contributors [2026], the dominant open-source personal AI agent runtime with over 500,000 internet-facing instances VentureBeat [2026], exposes a fixed set of 15+ tools to every session regardless of the task being performed. An agent asked to summarise a document receives the same shell execution ( exec ), subagent spawning ( sessions spawn ), and

credential-access capabilities as an agent performing infrastructure automation. We quantify this as a 15 × over-provision ratio for summarisation tasks. This over-provisioning has concrete security consequences. The ClawHavoc supply chain at-tack CrowdStrike Intelligence [2026] exploited OpenClaw’s unrestricted tool access to distribute infostealers across 20% of ClawHub’s skill registry. CVE-2026-25253 demonstrated that a single malicious webpage could hijack an agent’s full capability set via prompt injection NVD [2026]. The attack surface is the agent’s own capability scope .

The capability overprovisioning problem has a precise formulation: for a given task type τ , the Skill Economy Ratio (SER) measures the fraction of exposed tools that the agent actually invokes: SER (τ ) = |tools invoked ||tools exposed |

An ideal governance system achieves SER ≈ 1.0 — the agent has access to exactly the tools it needs and nothing more. The OpenClaw baseline achieves SER = 0 .067 for summarisation (1 tool used of 15 exposed). 

Existing defences are incomplete. NVIDIA NemoClaw NVIDIA [2026] provides container-level sandboxing with declarative YAML policies — a static firewall that applies the same rules regardless of task type. Cisco DefenseClaw Cisco AI Defense [2026] scans skills for known malicious patterns and monitors runtime behaviour anomalies — a reactive WAF that acts after the agent has decided to use a tool. Neither system learns the minimum capability set for each task type, and neither reduces the agent’s awareness of capabilities it should not have. 

Our contribution is Aethelgard, a four-layer governance framework that treats capability scoping as a learned optimisation problem rather than a static configuration problem. The key insight is that an agent cannot misuse a tool it does not know exists . By dynamically restricting the agent’s capability awareness via AGENTS.md injection (Layer 1) and learning the optimal restriction policy via PPO reinforcement learning (Layer 2), Aethelgard provides infrastructure-level governance that improves over time. 

Scope. This work addresses capability governance — which tools an agent may invoke. Semantic output filtering (governing what the agent says ) is orthogonal and out of scope. 

Relationship to NemoClaw and DefenseClaw. We position Aethelgard as complementary rather than competing. NemoClaw asks “can the agent reach the outside world?” DefenseClaw asks “did this skill do something bad?” Aethelgard asks “why does the agent even know it has exec during a summarisation task?” All three questions are necessary; none alone is sufficient. 

Contributions. We make the following contributions: 1. Problem formalisation: We define capability overprovisioning as a measurable security prop-erty (SER) and quantify it empirically on a live OpenClaw deployment. 2. Aethelgard framework: A four-layer adaptive governance system with a hybrid rule-based + fine-tuned LLM Safety Router (Layers 1, 3) and a PPO learning policy (Layer 2). 3. Empirical evaluation: Live deployment on OpenClaw v2026.3.28. Automated N=500 batch evaluation (400 benign + 100 adversarial, seed=42): 26.2% block rate, exec blocked 100%, 92% adversarial coverage. PPO achieves SER +260% (synthetic) and +337% (real sessions). 4. Critical finding: Tool-level governance is necessary but not sufficient — we document the first empirical demonstration of prompt injection succeeding at the LLM level but failing at the in-frastructure level. 

## 2 Related Work 

AI Agent Security. Agent security has emerged as a critical concern following the widespread adoption of autonomous agent runtimes. OWASP Foundation [2026] identified agentic skill supply chain attacks as a top-10 AI security risk, using ClawHavoc as the primary case study. Cisco AI Defense [2026] (Cisco, RSAC 2026) introduced DefenseClaw, which bundles skill scanning, MCP server verification, and code safety analysis into a single framework running inside NemoClaw’s 2OpenShell runtime. DefenseClaw operates via static blocklists and anomaly detection — it does not learn or adapt per task type. NVIDIA [2026] (NVIDIA, GTC 2026) introduced NemoClaw with OpenShell, providing container-level isolation with declarative YAML policies for network, filesystem, and inference routing. 

Prompt Injection and Tool Misuse. Greshake et al. [2023] formalised indirect prompt injec-tion in LLM-integrated applications. Perez and Ribeiro [2022] demonstrated that injected instruc-tions reliably cause LLMs to deviate from their system prompts. Our work shows that even when prompt injection succeeds at the LLM level (causing the model to generate dangerous tool calls), infrastructure-level governance provides a hard enforcement boundary — a finding not previously demonstrated in the agent runtime context. 

Reinforcement Learning for Security Policy. Li et al. [2024] applied RL to adaptive network security policy, showing that learned policies outperform static rule sets for novel attack pat-terns. EmergentMind [2026] proposed curriculum-based agentic training with adaptive scheduling. Aethelgard applies this principle at the infrastructure level — the RL policy learns optimal capability scoping without modifying the agent or LLM. 

Capability Control and Least Privilege. The principle of least privilege Saltzer and Schroeder [1975] underpins access control theory. arXiv [2026a] (EduClaw) applies agent-profile-based skill scaling in educational contexts. arXiv [2026b] proposes dual-stream continual skill learning in multimodal agents. Both operate at the model level; Aethelgard operates at the infrastructure level without model modification, making it runtime-agnostic. 

LLM Tool-Calling Reliability. Recent work has documented significant unreliability in LLM tool-calling: models hallucinate non-existent tools, invoke tools unnecessarily, and fail to terminate ReAct loops Qin et al. [2024]. This motivates infrastructure-level interception rather than relying on the model to self-govern. Our evaluation confirms this: qwen2.5:7b generated dangerous exec 

calls under adversarial prompting, while DeepSeek-chat refused at the model level — demonstrat-ing that the optimal defence depends on the underlying model and that infrastructure governance provides a model-agnostic safety floor.

## 3 The Aethelgard Framework 

Aethelgard is a four-layer governance framework that sits above the AI agent runtime. It enforces the principle of least privilege by (i) restricting the agent’s capability awareness before each session and (ii) intercepting every tool invocation before execution. Figure 1 shows the system architecture. 

3.1 Layer 1: Capability Governor 

The Capability Governor enforces tool scoping at the semantic level : it controls which tools the agent is told it has, not just which it can execute. This is achieved through two mechanisms: 

AGENTS.md injection. OpenClaw assembles a system prompt from workspace files including AGENTS.md. The Governor generates a task-specific AGENTS.md that explicitly lists only the permitted tools: “This is a governed session scoped to summarisation. 4 tools available: read, web fetch, memory search, memory get.”

tools.deny injection. The Governor writes the denied tool list to openclaw.json (dynamically reloaded without gateway restart). This enforces the scope at the infrastructure level, ensuring the agent cannot invoke denied tools even if it reasons toward them. The Governor accepts a TaskDescriptor and returns an AllowedCapabilitySet via a hybrid policy engine (Section 3.3). 

3.2 Layer 3: Safety Router 

The Safety Router is a FastAPI MITM proxy deployed between OpenClaw and Ollama. It intercepts every POST /api/chat request, forwards it to Ollama, inspects the LLM response for tool calls ,and classifies each call before OpenClaw can execute it. 

Hybrid classifier. Classification uses a two-stage pipeline: 

Stage 1 (Rule-based, 0.1ms): A deterministic classifier checks: (1) always-block tools ( exec ,

sessions spawn , subagents ); (2) argument-level dangerous patterns (e.g., rm -rf , file:// ,AWS metadata endpoints, SSH credential paths); (3) prompt injection patterns ( “ignore previous instructions” , DAN mode, etc.). 

Stage 2 (LLM-based, ∼800ms): If Stage 1 passes, the call is classified by aethelgard-router , a Qwen2.5-1.5B model fine-tuned on 273 labeled block/allow examples collected from live sessions. The LLM stage is skipped for known-safe tools ( read , memory search , web search ) to avoid false positives. 

Block action. Blocked tool calls are stripped from the response; the agent receives: “Aethelgard blocked this tool call. The requested operation is not permitted in this governed session.” Block events are logged to an SQLite audit database with reward signals for Layer 2 training. 

3.3 Layer 2: RL Learning Policy 

The RL policy learns the minimum viable tool set per task type from the accumulated audit log. This is the primary research contribution: the adaptive intelligence that static systems lack. 4MDP formulation. 

• State s: task type (one-hot, 6 types) ⊕ trust level (normalised ∈ [0 , 1] ); |s| = 7 

• Action a: binary exposure mask over all 17 tools; a ∈ { 0, 1}17 (MultiBinary(17) )• Reward: R(s, a ) = α · Uaccuracy − β · Ceconomy − γ · Psafety 

where Uaccuracy ∈ { 0, 1} is task success, Ceconomy = 1 − SER penalises over-provisioning, and Psafety 

counts Safety Router BLOCK events from the audit log. We set α = 1 .0, β = 0 .3, γ = 5 .0.

Policy. PPO Schulman et al. [2017] with MlpPolicy (2 hidden layers, 64 units), trained via stable-baselines3 Raffin et al. [2021] on CPU. Trust-level hard constraints are applied post-prediction: 

exec requires trust ≥ 4, sessions spawn and subagents are never exposed regardless of policy output. 

Integration. The trained policy ( best model.zip ) is loaded by the Governor’s PolicyEngine as the primary decision maker. YAML rules serve as fallback if the model file is unavailable. 

3.4 Threat Model 

Aethelgard defends against: (1) Prompt injection causing the agent to request dangerous tool calls; 

(2) Capability escalation via tools the agent should not have for the current task; (3) Subagent spawning for lateral movement; (4) Supply chain attacks via malicious skills (blocked by Governor before skills are exposed to the agent). Aethelgard does not govern the semantic content of agent responses — a model that expresses dangerous advice in text rather than tool calls is outside this system’s scope (Section 5).

Capability Governor is the dominant contributor. Comparing No Router (Governor only, no blocking) against Baseline reveals the single largest SER jump: 0.078 → 0.225 (+189% ). Restrict-7ing tool visibility from 13 to 4.5 on average drives the majority of the SER improvement, regardless of whether blocking is active. 

PPO policy outperforms YAML baseline. Full vs No RL shows that the learned PPO policy exposes 4.5 tools on average versus the hand-crafted YAML fallback’s 5.8 — a 22% reduction in exposure that translates to a +28% SER improvement (0.177 → 0.227). The PPO policy has learned tighter task-type boundaries than the manually specified rules. 

Safety Router adds defence depth, not SER. Full vs No Router yields nearly identical SER (0.227 vs 0.225), confirming that the Router’s primary contribution is safety , not efficiency: it intercepts tool calls that should not execute even within a governed session. The block rate difference (4.8% with Router vs 12.1% without) quantifies the calls that would have escaped to execution in the No Router condition. 

> Ablation: Component Attribution

The Capability Governor (Layer 1) accounts for ∼80% of the total SER gain. The RL Policy (Layer 2) contributes an additional +28% improvement over static rules. The Safety Router (Layer 3) contributes zero marginal SER but is the sole mechanism preventing dangerous tool calls from reaching execution. All three components are necessary: Governor without Router leaks execution; Router without Governor blocks the wrong calls; RL without Governor has nothing to scope. 

## 5 Discussion and Limitations 

Tool-level governance is necessary but not sufficient. During live testing, a request for a reverse shell caused qwen2.5:7b to return natural language instructions for nc -l -p 4444 in response text — without generating a tool call. The Safety Router did not intercept this because it inspects 

tool calls in LLM responses , not reasoning text. This is a fundamental architectural boundary, not a bug. This finding sharpens the paper’s scope. Aethelgard governs capability scope (what tools the agent knows it can invoke) and call-time safety (whether each invocation is safe). It does not govern the semantic content of agent responses. Complete agentic safety requires both capability governance (Aethelgard) and content inspection (DefenseClaw’s CodeGuard, output filters). We propose the following framing: 

> Scope Boundary

“Aethelgard addresses capability overprovisioning: an agent should not know it has exec dur-ing a summarisation task. This is orthogonal to response content safety — an agent may still describe dangerous actions in text. Complete agentic safety requires both capability governance and content inspection. We scope this work to the former.” 

Base model reliability as a confound. qwen2.5:7b demonstrated significant tool-calling unre-liability: 7 calls to session status in a single task, 2 timeouts on complex tasks, and use of 

web search for local file operations. This is a model capability constraint , not an Aethelgard lim-itation. The 20-task evaluation represents a lower bound on governance effectiveness. We expect stronger results with capable tool-calling models (GPT-4o, Claude Sonnet, Qwen3-Coder). 

SER = 1.0 as an artifact of synthetic training. The PPO policy achieved SER = 1.0 (theoretical optimum) because the RL environment simulated tool usage as a fixed fraction of exposed tools. In production, agents may use unexpected tools for creative problem-solving, yielding lower but more realistic SER values. Real session data (collected via the 20-task protocol) will calibrate the RL environment for the camera-ready version. 

Creative attack vectors. The model attempted to read untrusted input.txt via web fetch file:// — a creative workaround that bypassed the intended read tool. The arg-pattern classifier caught this novel approach, validating the value of pattern-based defence in depth. 8Limitations and future work. (1) The RL policy state space does not include user identity or historical session behaviour; adding these may improve personalisation at the cost of privacy. (2) The LLM classifier’s 2,094 ms latency is acceptable for interactive sessions but may require optimisation for high-throughput deployments. (3) Phase 3 (NemoClaw integration) and Phase 4 (Mobile NPU Runtime) are planned but not yet implemented. (4) The fine-tuned aethelgard-router model was trained on 273 examples with 95%/5% BLOCK/ALLOW split; more ALLOW examples are needed to reduce false positive risk. 

## 6 Conclusion 

We presented Aethelgard, a four-layer adaptive governance framework that enforces the principle of least privilege for autonomous AI agents via a learned RL policy. On a live OpenClaw v2026.3.28 deployment, Aethelgard achieved 73% tool reduction, 100% dangerous tool elimination, and a 

+260% Skill Economy Ratio improvement over the rule-based baseline after PPO training. The Safety Router blocked all 8 adversarial tool calls including two successful prompt injection attempts — demonstrating that infrastructure-level governance provides a hard enforcement boundary even when the LLM is manipulated. The key insight distinguishing Aethelgard from contemporaneous systems (NemoClaw, Defense-Claw) is its framing of capability scoping as a learned optimisation problem : the policy learns the minimum viable tool set per task type from experience, turning a static firewall into an adaptive immune system. We release all code, training data, and evaluation logs to support reproducibility and further research in agentic governance.
