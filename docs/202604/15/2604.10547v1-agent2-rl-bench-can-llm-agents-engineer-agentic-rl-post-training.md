## 摘要
我们引入了 Agent^2 RL-Bench，这是一个用于评估智能体强化学习（RL）后训练的基准测试，旨在考察大语言模型（LLM）智能体能否自主设计、实现并运行完整的 RL 流水线以提升基础模型。这一能力至关重要，因为 RL 后训练正日益成为模型对齐与专业化的核心驱动力，然而现有的基准测试大多是静态的：仅靠监督微调（SFT）即可获得强劲结果，导致交互式 RL 工程化能力未得到充分测试。Agent^2 RL-Bench 通过涵盖三个层级的六个任务解决了这一问题——从静态的基于规则的训练到包含轨迹采集的闭环在线 RL——每个层级都引入了前一层级所不具备的结构性要求。该基准测试提供了带有评分 API 的隔离工作区、记录每次提交和代码修订的运行时检测工具，以及生成结构化运行报告的自动化事后分析，从而实现了对智能体驱动的后训练行为的首次自动化诊断。通过对涵盖 5 个智能体系统和 6 个驱动 LLM 的多个智能体堆栈的研究，我们发现智能体在交互式任务中取得了显著增益——在 ALFWorld 上，纯 RL 智能体通过 SFT 预热和带有在线采样的 GRPO 将性能从 5.97 提升至 93.28——但在其他任务上进展甚微（如 DeepSearchQA 仅提升 2.75，处于评估噪声范围内）；此外，驱动模型的选择对交互式任务影响巨大，在相同框架下，更换驱动模型可使交互式改进从接近零提升至 78 个百分点。更广泛地，该基准测试揭示了在固定预算下，监督流水线在智能体驱动的后训练中占据主导地位，而在线 RL 仅在 ALFWorld 上成为了最终的最佳路径。代码开源地址为 https://github.com/microsoft/RD-Agent/tree/main/rdagent/scenarios/rl/autorl_bench。

## Abstract
We introduce Agent^2 RL-Bench, a benchmark for evaluating agentic RL post-training -- whether LLM agents can autonomously design, implement, and run complete RL pipelines that improve foundation models. This capability is important because RL post-training increasingly drives model alignment and specialization, yet existing benchmarks remain largely static: supervised fine-tuning alone yields strong results, leaving interactive RL engineering untested. Agent^2 RL-Bench addresses this with six tasks across three levels -- from static rule-based training to closed-loop online RL with trajectory collection -- each adding a structural requirement that prior levels do not impose. The benchmark provides isolated workspaces with a grading API, runtime instrumentation that records every submission and code revision, and automated post-hoc analysis that generates structured run reports, enabling the first automated diagnostic of agent-driven post-training behavior. Across multiple agent stacks spanning five agent systems and six driver LLMs, we find that agents achieve striking interactive gains -- on ALFWorld, an RL-only agent improves from 5.97 to 93.28 via SFT warm-up and GRPO with online rollouts -- yet make only marginal progress on others (DeepSearchQA: +2.75 within evaluation noise), and that driver choice has a large effect on interactive tasks -- within the same scaffold, switching drivers changes interactive improvement from near-zero to +78pp. More broadly, the benchmark reveals that supervised pipelines dominate agent-driven post-training under fixed budgets, with online RL succeeding as the final best route only on ALFWorld. Code is available at https://github.com/microsoft/RD-Agent/tree/main/rdagent/scenarios/rl/autorl_bench.

#### 1. 核心问题与整体含义（研究动机和背景）
论文探讨的核心问题是：**LLM 智能体是否具备自主设计、实现、调试并运行完整的强化学习（RL）后训练流水线的能力**，以提升另一个基础模型的性能。
*   **研究动机**：虽然 RL 后训练（如 PPO、GRPO）是当前大模型对齐和专业化的核心，但现有的智能体基准测试（如 SWE-bench 或 MLE-bench）大多偏向代码生成或静态机器学习任务。在这些任务中，仅靠监督微调（SFT）往往就能取得好结果，而无需智能体去处理复杂的交互式 RL 工程（如在线轨迹采集、奖励函数设计、训练稳定性控制等）。
*   **整体含义**：该研究定义了“智能体化 RL 后训练”（Agentic RL Post-training）这一新领域，旨在测试智能体在长程、系统级工程挑战中的表现。

#### 2. 方法论：核心思想与技术细节
论文提出了 **Agent^2 RL-Bench**，其核心思想是通过**训练结构覆盖（Training Structure Coverage）**来诊断智能体的能力。
*   **三层级任务结构**：
    *   **L1（静态规则级）**：如 GSM8K、HumanEval，使用确定性校验器，测试基础数据构建和 SFT 能力。
    *   **L2（静态评判级）**：如 AlpacaEval，引入非确定性的 LLM 评判信号，测试针对偏好信号的优化。
    *   **L3（交互采样级）**：如 ALFWorld、WebShop、DeepSearchQA，要求智能体实现环境交互、多步轨迹采集和闭环在线 RL。
*   **关键技术设施**：
    *   **隔离工作区**：为每个智能体提供独立环境，包含基础模型、训练数据和评分 API。
    *   **运行时监测（Instrumentation）**：记录智能体的每一次代码修订、模型提交和运行日志。
    *   **自动化事后分析**：自动生成结构化的运行报告和案例研究，用于诊断智能体为何成功或失败。
*   **算法流程**：智能体在给定预算（如 12 小时）内，需自主经历“阅读任务 -> 编写训练代码 -> 运行训练 -> 提交模型评分 -> 根据反馈迭代”的循环。

#### 3. 实验设计
*   **数据集/场景**：涵盖数学推理（GSM8K）、代码生成（HumanEval）、通用指令遵循（AlpacaEval）、具身智能交互（ALFWorld）、电商搜索交互（WebShop）和深度搜索问答（DeepSearchQA）。
*   **对比方法（智能体堆栈）**：
    *   **5 种智能体框架**：OpenHands、OpenCode、Claude Code、Codex CLI、Gemini CLI。
    *   **6 种驱动模型**：GPT-4o、GPT-5.2/5.4（预研版本）、Claude 3.5/4.x 系列、Gemini 2.5-flash。
    *   **消融实验**：针对 Claude Code 设置了四种模式：仅 SFT、仅 RL、自由模式（Free）、多任务模式（Multi）。
*   **基准模型**：被优化的目标模型包括 Qwen2.5-7B-Instruct 和未对齐的 Qwen3-8B-Base。

#### 4. 资源与算力
*   **工程预算**：每个任务通常分配 **12 小时** 或 24 小时的自主工程时间。
*   **硬件资源**：文中提到部分实验使用了 **H200 GPU**（例如在 DeepSearchQA 追踪实验中使用了 2 × H200）。
*   **算力消耗**：论文详细记录了 Token 消耗，例如 Claude Code 在 24 小时内处理了约 2.9 亿个 Token，估计成本约 796 美元。

#### 5. 实验数量与充分性
*   **实验规模**：进行了跨 6 个任务、多个智能体框架与驱动模型组合的系统性测试。
*   **充分性评价**：
    *   **优点**：实验设计了从静态到动态的梯度，能够清晰定位智能体能力的断层；包含了详细的消融实验（模式切换、驱动模型切换）。
    *   **局限**：由于成本极高，部分 8B 模型的受控实验为**单次运行（Single-run）**，缺乏多次重复的统计置信区间（尽管在 DeepSearchQA 上做了 20 次重复实验来验证评估噪声）。

#### 6. 主要结论与发现
*   **交互式增益显著但极不均衡**：在 ALFWorld 上，智能体能将模型得分从 5.97 提升至 93.28；但在 DeepSearchQA 上几乎没有实质性进展。
*   **驱动模型是决定性因素**：在相同框架下，更换驱动模型（如从 GPT-4o 换到 GPT-5.2）可使 ALFWorld 的改进幅度从 0.74pp 飙升至 78.35pp。
*   **SFT 仍占主导地位**：在固定预算下，智能体更倾向于（且更擅长）构建复杂的 SFT 流水线，而非稳定的在线 RL 系统。
*   **对齐差距的闭合依赖全堆栈**：对于未对齐的基础模型，只有最强的智能体+驱动模型组合（如 Claude Code + Opus 4.6）能将其提升至指令遵循水平，弱组合甚至会导致模型退化。

#### 7. 优点
*   **结构化诊断**：通过 L1-L3 的分层，首次量化了“写 RL 代码”与“运行 RL 系统”之间的巨大鸿沟。
*   **真实工程模拟**：不限制智能体的方法，允许其自主选择 SFT、DPO 或 GRPO，真实反映了 AI 工程师的工作流程。
*   **深度行为分析**：提供了丰富的 Trace（追踪）分析，揭示了智能体如何通过修复 Prompt 格式不一致（而非改进算法）获得巨大提升。

#### 8. 不足与局限
*   **评估噪声**：部分交互式任务（如 DeepSearchQA）存在 ±3pp 的评估噪声，导致智能体难以通过迭代获得正向反馈。
*   **非完全因子设计**：受限于成本，并未测试所有智能体框架与所有驱动模型的排列组合。
*   **内容过滤干扰**：实验中提到 Azure 的内容过滤器干扰了约 4% 的 AlpacaEval 评判，可能对结果产生轻微偏差。
*   **任务多样性**：虽然结构覆盖好，但任务总数（6个）相对较少，未来可扩展更多复杂的 RL 场景。
