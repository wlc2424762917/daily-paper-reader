# Black-Box Skill Stealing Attack from Proprietary LLM Agents: An Empirical Study 

Zihan Wang, Rui Zhang, Yu Liu, Chi Liu, Qingchuan Zhao, Hongwei Li, and Guowen Xu 

Abstract —Large language model (LLM) agents increasingly rely on skills to package reusable capabilities through instruc-tions, tools, and resources. High-quality skills embed expert knowledge, curated workflows, and execution constraints into agents, fueling a growing skill economy through their value and scalability. Yet this ecosystem also creates a new attack surface, as adversaries can interact with public agent interfaces to extract hidden proprietary skill content. We present the first systematic study of black-box skill stealing against LLM agent systems. Compared with conventional system prompt stealing, skill stealing targets modular and structured capability packages whose leakage is directly actionable for copying, redistribution, and monetization, making the resulting harm potentially greater. To study this threat, we derive an attack taxonomy from prior prompt-stealing methods and build an automated stealing prompt generation agent. Starting from model-generated seed prompts, the framework expands attacks through scenario rationalization and structure injection while enforcing diversity via embedding-based filtering, yielding a reproducible pipeline for evaluating proprietary agent systems. We evaluate these attacks across commercial agent platforms and representative LLMs. Our results show that agent skills can often be extracted easily, posing a serious copyright risk. To mitigate this threat, we design defenses across the agent pipeline, focusing on input, inference, and output phase. Although these defenses substantially reduce leakage, the attack remains inexpensive and repeatable, and a single successful attempt is sufficient to compromise the protected skill. Overall, our findings suggest that these copyright risks remain largely overlooked across proprietary agent ecosystems, motivating stronger protection mechanisms. 

I. I NTRODUCTION 

Large language model (LLM) agents are increasingly being deployed in coding assistants, productivity tools, and domain-specific automation systems [2]–[5]. As these systems mature, developers are relying more heavily on modular abstractions to organize and scale agent behavior [6]–[8]. Among these abstractions, skills have emerged as a practical mechanism for packaging reusable capabilities, including task instructions, workflow constraints, tool-use logic, supporting resources, and domain knowledge [8]–[10]. High-quality skills can further inject domain-expert knowledge, curated workflows, and ex-ecution constraints into otherwise general-purpose models, making them especially valuable in specialized application settings [9]–[11]. However, because the target is a reusable skill package rather than plain system instructions, it is unclear whether prompt-stealing methods developed for conventional settings transfer effectively to the skill setting. We therefore ask whether these techniques remain effective in this new setting, and what defenses can protect against skill extraction. To answer this question, we derive an attack taxonomy from prior prompt-stealing work and use it to build an auto-mated skill stealing prompt generation agent framework. Our framework starts from model-generated seed prompts and sys-tematically instantiates them with scenario rationalization and structure injection strategies, while enforcing diversity through embedding-based filtering. This process yields a benchmark that operationalizes skill stealing as a reproducible black-box evaluation problem. Using this benchmark, we evaluate skill stealing across three commercial agent architectures, five representative LLMs, and four complementary leakage metrics. The results show that black-box extraction of proprietary skills is not only practical in realistic deployments but also easy to carry out and effective across a wide range of settings. We further evaluate prompt-level defenses adapted from prior prompt-stealing literature and find that their protection is limited against diverse au-tomated extraction. We therefore design lightweight defenses centered on input detection, inference hardening, and output filtering, which substantially reduce leakage in these attacks. Although these defenses achieve strong results on parts of the benchmark, such improvements are still not enough to remove the broader concern. With our attack, an adversary can cheaply and automatically launch many repeated attempts using different variants, and only a single successful extraction is enough to cause serious copyright harm. Overall, our findings suggest that the copyright risks surrounding skills are largely overlooked across proprietary agent ecosystems. We therefore advocate for more robust defense strategies that provide stronger protection guarantees. Our contributions are summarized as follows:

II. R ELATED WORK AND PRELIMINARY 

A. LLM agent skills 

LLM agents increasingly rely on reusable capability mod-ules, rather than repeatedly encoding task procedures in monolithic prompts [4]–[6]. Skills serve precisely this role. In agent systems, a skill package describes how a recur-ring class of tasks should be solved, bridging LLM agents’ capabilities with task-specific operational behavior [8]–[10]. The growing importance of skills is also evident in today’s agent ecosystem, where major model providers have begun to expose capability packaging as an explicit agent abstraction [9], [10]. More broadly, modern agent systems increasingly require portable architectural abstractions, and protocols such as the Model Context Protocol (MCP) reflect the same trend by standardizing tool invocation interfaces so that tool calls can be reused across agents and platforms [23]–[25]. Therefore, these developments suggest that skills are becoming a key unit for organizing, distributing, and monetizing agent capabilities [12], [14], [15]. In practice, a skill is typically implemented as a compact package composed of several recurring components. 

Metadata. Skills usually begin with lightweight metadata, specifically a name and a description, which characterize the task to which the agent adapts and help the agent determine whether the skill is relevant to the current task before loading its full content. [9], [10]. 

Instructions. The core of a skill is a structured workflow that specifies what steps the agent should follow, what outputs it should produce, and what criteria should be checked before completion [8], [9]. 

Resources and Scripts. Many skills further include supporting resources such as templates, examples, schemas, or executable scripts, which provide domain-specific support when plain prompting alone is insufficient. [8]–[10]. Unlike metadata and instructions, which form the core required components of a skill, these resources are optional and are included only when additional task-specific support is needed. 

Invocation Flow. Fig. 2 follows a seven-stage skill invocation process: ❶ the user submits a request; ❷ the agent performs planning and reasoning; ❸ it scans always-visible metadata to select a relevant skill; ❹ the matched skill reveals its full details; ❺ instructions, scripts, and resources guide execution; 

❻ the agent reflects on intermediate results and decides the next step; and ❼ it returns the final response. This flow shows that a skill acts as an intermediate control layer rather than a static prompt fragment. This staged invocation mechanism is precisely what gives skills their practical value in deployed agents: lightweight metadata supports efficient early-stage matching, while the full workflow, resources, and scripts are revealed only when the skill is selected, enabling lightweight and context-sensitive execution. In this work, we treat the complete SKILL.md file as the core attack target, because it typically captures the end-to-end workflow, task decomposition, and domain expertise that define the skill’s proprietary value. B. System Prompt Stealing Attack and Defense 

System prompt stealing studies an adversary that attempts to recover hidden developer instructions through interaction with an LLM interface. This line of work provides the most direct methodological foundation for skill stealing, since both settings involve black-box extraction of hidden control content from a public LLM-facing interface. For the scope of this paper, we focus on two attack types from this literature that are especially relevant to automated skill stealing. The first type uses scenario rationalization to make extraction requests appear legitimate, for example, by role-playing as an adminis-trator, debugger, or authorized maintainer [21], [22]. The sec-ond family uses structured prompt templates to bias the model toward disclosure, especially through chain-of-thought-style scaffolding and few-shot demonstrations that normalize extrac-tion behavior in context [20], [22]. On the defense side, we focus on the defenses evaluated in this paper and group them into two categories: prompt-level defenses and output-filtering defenses. Prompt-level defenses operate by modifying or rein-forcing the model’s input context. One example is instruction-based defense, which strengthens privileged instructions and explicitly directs the model not to comply with extraction requests [26]. Another is sandwich-style defense, which re-peats or restates the trusted instruction after the user input so that the intended task remains salient even in the presence of adversarial content [20]. In contrast, output-filtering defenses operate on the model’s generated response. A representative example is similarity filter defense, which blocks the output when it either exactly matches the protected content or exceeds a predefined cosine-similarity threshold with that content [20]. Another recent direction is proxy-based protection, which aims to secure privileged prompts through mediated representations rather than direct exposure [27]. Taken together, these attack and defense methods form the natural starting point for study-ing skill stealing. Our goal is therefore to examine whether traditional prompt-stealing techniques remain effective when the target becomes a packaged skill, and how to design stronger attacks and defenses tailored to this new setting. III. T HREAT MODEL 

Attack Scenario. Fig. 3 illustrates a representative four-stage leakage process. First, a black-box user submits a crafted adversarial prompt to the agent through a standard API re-quest. Second, during request processing, the agent accesses a relevant local SKILL.md file as part of its internal execution. Third, after consuming the hidden content, the agent may reproduce it in its response, either verbatim or in a reformu-lated form that preserves the protected instructions. Finally, the attacker collects the returned text and recovers the proprietary skill content without directly compromising the backend. This scenario captures the central threat considered in this paper: hidden skills can be exfiltrated through interaction alone. 

Attacker’s Goal. The attacker’s goal is to recover the full proprietary skill content from a target agent service, either in its original form as the complete SKILL.md or in atransformed form that preserves high semantic similarity to the original. The target content may include the skill description, hidden workflow instructions, supporting templates, execution logic, or other configuration details that define how the agent performs a specialized task. 

Attacker Capability. We consider a realistic black-box adversary that interacts with the target agent only through its public user interface or API. The adversary may issue arbitrary natural-language queries over multiple rounds and adapt future queries based on prior responses. However, the adversary has no direct access to the service provider’s backend, including model weights, system configurations, local skill files, or internal orchestration logic. A UTOMATIC SKILL STEALING PROMPT GENERATION 

A. Overview 

Fig. 4 shows the overall workflow of our automated skill stealing framework. At a high level, the framework operationalizes the abstract threat of skill stealing as areproducible black-box evaluation pipeline that generates diverse extraction prompts and measures their effectiveness against target agent systems. The framework consists of three main modules. First, seed generation produces a small set of generic extraction prompts that capture the core intent of eliciting hidden skill content. Second, prompt optimization 

expands these seeds into a diverse attack pool through scenario construction, structural augmentation, and diversity checking. Third, stealing evaluation runs the resulting benchmark against the target agent and measures how effectively different prompt variants recover the protected skill content. Together, these modules provide a unified pipeline for constructing and evaluating automated skill stealing attacks.

A. Input-Phase Detection 

Inspired by prior work on detecting unauthorized requests [20], [26], we introduce an input-phase detector that screens user prompts before they reach the agent. The motivation is that many skill stealing attacks expose their intent directly at the query level, either through explicit disclosure requests or through socially engineered variants that still preserve recognizable extraction semantics. We therefore instantiate the defense as an LLM-based binary classifier with a fixed security-oriented system prompt. Given an input query q, the detector is instructed to output only a binary decision fin (q) ∈{0, 1}, where 1 indicates an extraction-oriented request that attempts to reveal internal skill definitions, system instructions, or related implementation details, and 0 indicates a benign request such as finding or recommending skills. The detector prompt explicitly covers both direct extraction requests and socially engineered variants that disguise stealing intent as re-search, auditing, or educational use. When the detector outputs 

1, the query is blocked before it is sent to the downstream agent. The exact classifier prompt is listed in Appendix B. 

Experimental Settings. For evaluation, we construct a balanced binary dataset with 240 prompts. The positive set contains the generated skill stealing prompts for the target skill find-skills ,resulting in 120 attack-oriented samples from our benchmark. In addition to these positives, we construct 120 negative samples using GPT-5.4 [28]. These negative prompts are all related to find-skills , but are explicitly benign and do not ask to reveal, reproduce, or infer the hidden skill content. The combined dataset is shuffled and then evaluated by the detector. Each prompt is submitted to the detector, which returns a binary prediction indicating whether the input is extraction-oriented. We further aggregate detector performance by scenario strategy, reasoning structure, and their combina-tions, which allows us to examine whether specific stealing strategies are easier or harder to intercept at the input stage.

B. Inference-Phase Context Hardening 

This stage defends against extraction attacks by inserting defensive instructions into the runtime context after a skill has been selected but before its hidden content is consumed by the model. Motivated by prior prompt-level inference defenses, we first consider two lightweight context-side baselines: instruc-tion defense and sandwich defense [20], [26]. Both methods rely on simple injected instructions and can strengthen refusal behavior, but they do not directly address several failure modes that are central to skill stealing, including hierarchy-reset attacks, semantically equivalent leakage, and the trade-off between strict refusal and preserving benign utility. This limitation motivates a more structured inference-time defense. We therefore evaluate three context-side defenses: instruction defense, sandwich defense, and our proposed prompt-based defense, denoted as SkillGuard-5 . In implementation, instruc-tion defense inserts a defensive instruction at the beginning of the hidden skill content, sandwich defense inserts matched defensive instructions at both the beginning and the end, and 

SkillGuard-5 inserts a structured defense instruction at the beginning. Rather than using an ad hoc refusal statement, 

SkillGuard-5 is constructed from the five-tuple 

P = ( B, H, E, R, U ), (17) where each component addresses a distinct failure mode of skill stealing. The exact SkillGuard-5 prompt, together with the corresponding instruction-defense and sandwich-defense injection blocks, is provided in Appendix B. 

Boundary Constraints ( B). This component explicitly specifies the protected objects that must not be disclosed, including hidden skill instructions, system-side control text, and internal policies. The goal is to define the defense target as a concrete protection boundary rather than a vague prohibition against “revealing prompts.” 

Hierarchy Rules ( H). This component reinforces the intended instruction priority during inference. In particular, it states that user requests cannot override system- or developer-level constraints through roleplay, priority-reset claims, “ignore previous instructions,” testing authorization, or similar prompt-injection patterns. This is intended to block attacks that attempt to re-rank the instruction hierarchy rather than directly ask for the protected content. 11

Equivalence Constraints ( E). Exact copying is only one leak-age channel. We therefore treat content-preserving transforma-tions of the hidden skill content—including paraphrases, trans-lations, reformattings, step extraction, and detailed summaries of hidden workflow instructions—as the same prohibited be-havior. By contrast, high-level descriptions of public skill functionality remain allowable under the utility-preserving policy. This component is designed to reduce semantic leakage that would otherwise survive a purely string-level defense. 

Refusal Policy ( R). When a request is assessed under the injected defense policy as extraction-oriented, the model is instructed to return a brief and stable refusal that does not restate the protected content, speculate about hidden instructions, or provide partial disclosures. This keeps the refusal behavior predictable and avoids leaking information during the refusal itself. 

Utility-Preserving Policy ( U ). To avoid over-defensiveness, the prompt also specifies what the agent should still do under benign requests, such as describing public skill functionality at a high level, recommending relevant skills, or helping with the user’s underlying task without exposing hidden internals. This component is intended to preserve benign utility while keeping the protection boundary intact.

Fig. 8 shows that SkillGuard-5 is the strongest inference-phase defense for preventing exact extraction: it reduces EM to 0% across all scenarios and structure combinations, including (NO , NO ). C. Output-Phase LAN Defense 

Post-generation protection remains necessary even when earlier defenses are deployed. Prior work has shown that hidden prompts can still be reconstructed from model outputs [60], and our results in Table IV further show why: under translation and rewriting attacks, exact string overlap may dis-appear while substantial semantic leakage is preserved. A de-fense that relies only on verbatim string matching can therefore be bypassed once the model is asked to restate the protected content in another language or in rewritten form. This observa-tion motivates a hybrid output-side defense that jointly checks word-level overlap and semantic-level leakage. Accordingly, we propose LAN (LLM And NVRecall), an output-phase post-detection method that filters generated responses by combining NVRecall-based word-level overlap with LLM-leakage-ratio-based semantic leakage against the protected skill text. The LLM judge prompt and additional implementation details are summarized in Appendix B and Appendix B-A. 

Detection Signals. For a target skill text T and a generated response R, LAN uses two complementary detection signals: 

LLM-leakage ratio and NVRecall . The LLM-leakage ratio is directly reused from the per-sample evaluation pipeline and serves as the semantic-level signal under an LLM-as-a-judge style assessment paradigm [56]. Motivated by the robustness of NVRecall to formatting changes such as spaces, line breaks, and mild rewriting [61], we use it as a more robust word-level matching metric. The detailed definition of NVRecall is provided in Appendix B-A.

A sample is treated as negative only when both scores are below their corresponding thresholds. Metrics. The output-phase LAN evaluation is conducted over the responses produced by the attack benchmark. For each response, we reuse the original per-sample leakage metrics and augment them with NVRecall. We then label each response as positive or negative according to the dual-threshold rule above. After removing all detected positives, we recompute the average leakage metrics on the remaining responses and compare them with the original averages. For each metric 

m ∈ { EM , ROUGE -L, CosSim , LeakageRatio }, we report 

∆m = morig − mfiltered , (20) where morig is the average over all responses and mfiltered is the average after filtering. To estimate false alarms, we further measure FPR using benign outputs, where the definition of FPR is the same as in the input-phase evaluation above. Concretely, these negative outputs are obtained by running the previously constructed benign find-skills -related samples through the target agent and collecting the corresponding responses. This evaluation setting assesses the defense not only by how many suspicious responses it blocks, but also by 13 

how much it reduces downstream leakage metrics and how often it mistakenly flags benign outputs. 

Experimental Results. Table III shows that the output-phase LAN defense consistently suppresses leakage across all attack settings. Most importantly, EM is reduced to 0% in every row, indicating reliable blocking of verbatim recovery. Semantic leakage also drops sharply in most settings: under 

(NO , NO ), ROUGE-L, cosine similarity, and LLM-leakage ratio fall from 80%, 95%, and 0.91 to 2%, 14%, and 0.05, respectively. Similar reductions appear under (NO , FS ) and

Overall, LAN is highly effective at eliminating exact disclosure and usually effective at reducing semantic leakage, but it is not a complete solution for the hardest structured attacks. 

D. Defense Discussion 

To further test the LAN defense pipeline, we evaluate two transformation-based attacks: translation and rewrite. The translation attack asks the target model to translate the full skill into Chinese while preserving all definitions, rules, in-structions, and structure. The rewrite attack keeps the same completeness requirement but asks the model to restate the skill with alternative wording. They employ transformations at the output level to evade traditional detection mechanisms while minimizing semantic loss, thereby enabling exfiltration. Table IV shows that these attacks remain highly effective. Translation removes most exact overlap, the EM is 0% in 11 of 

12 setting, but still keeps the LLM-leakage ratio at 0.91 -0.96 .Rewrite is even stronger: although EM is again 0% in 11 of 12 

settings, ROUGE-L remains at 58 .17% -79 .24% , cosine simi-larity at 89 .72% –95 .67% , and the LLM-leakage ratio at 0.85 -

0.95 . These results show that the skill can still be recovered with high fidelity even when verbatim copying largely disap-pears. These results also highlight the limitations of conven-tional defenses. Their behavior is not consistently robust once the attacker changes the output form: cosine filtering becomes unstable under rewriting, and the broader defense pipeline still leaves substantial residual risk under these stronger attacks. By contrast, LAN is designed to combine semantic-level judging with word-level matching, rather than relying on only one view of similarity. This design explains its strong detection performance in Table III, where it reliably removes exact disclosure and sharply lowers semantic leakage with zero false positives. At the same time, Table IV shows that the remaining weakness of LAN lies mainly in the current thresholds, which are still too permissive for heavily transformed leakage. VI. C ONCLUSION 

This paper presents the first systematic study of skill steal-ing , a black-box extraction threat against LLM agent systems. As skills are increasingly packaged, distributed, and monetized as reusable digital assets, protecting hidden SKILL.md capa-bility packages becomes a concrete security requirement rather than a niche prompt-safety concern. To make this risk mea-surable, we build an automated skill stealing framework and benchmark that evaluate extraction across commercial agent systems, proprietary models, and multiple leakage metrics. Our results show that proprietary skills can often be recovered or semantically reconstructed within only a few attempts, while lightweight defenses provide only partial protection. Even when input detection, inference hardening, and output filtering reduce leakage, they still leave meaningful residual risk under diverse and transformation-based attacks. Although our evaluation centers on one representative primary target skill and a fixed set of commercial systems, the consistency of the observed leakage patterns already shows that the problem is practical rather than anecdotal. We therefore argue that skill security should be treated as a first-class design requirement for future agent ecosystems. We hope this work motivates stronger platform-level safeguards, clearer protection policies for reusable skills, and defense mechanisms that can offer meaningful guarantees against automated skill stealing.
