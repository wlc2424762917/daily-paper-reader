## 摘要
代码智能体正飞速发展，但对其进行调试也变得日益困难。随着框架在复杂任务中编排并行工具调用和多阶段工作流，智能体的状态转换和错误传播变得难以观察。在这些运行过程中，早期的失误可能会使智能体陷入无意义的循环，甚至级联成根本性错误，形成隐蔽的错误链，从而难以判断智能体何时以及为何偏离轨道。现有的智能体追踪分析要么侧重于简单的交互，要么依赖小规模的人工检查，这限制了它们在实际编程工作流中的可扩展性和实用性。我们提出了 CodeTracer，这是一种追踪架构，它通过不断演进的提取器解析异构运行产物，将完整的状态转换历史重构为具有持久化内存的层级追踪树，并执行故障起始定位以精准识别故障源头及其下游链条。为了实现系统化评估，我们构建了 CodeTraceBench。该基准测试源自四个广泛使用的代码智能体框架在多种代码任务（如错误修复、重构和终端交互）中生成的大量执行轨迹，并在阶段和步骤级别提供故障定位的监督信号。实验表明，CodeTracer 的表现显著优于直接提示和轻量级基准模型；此外，在相同预算下，回放其诊断信号能够稳定地恢复原本失败的运行。我们的代码和数据已公开。

## Abstract
Code agents are advancing rapidly, but debugging them is becoming increasingly difficult. As frameworks orchestrate parallel tool calls and multi-stage workflows over complex tasks, making the agent's state transitions and error propagation hard to observe. In these runs, an early misstep can trap the agent in unproductive loops or even cascade into fundamental errors, forming hidden error chains that make it hard to tell when the agent goes off track and why. Existing agent tracing analyses either focus on simple interaction or rely on small-scale manual inspection, which limits their scalability and usefulness for real coding workflows. We present CodeTracer, a tracing architecture that parses heterogeneous run artifacts through evolving extractors, reconstructs the full state transition history as a hierarchical trace tree with persistent memory, and performs failure onset localization to pinpoint the failure origin and its downstream chain. To enable systematic evaluation, we construct CodeTraceBench from a large collection of executed trajectories generated by four widely used code agent frameworks on diverse code tasks (e.g., bug fixing, refactoring, and terminal interaction), with supervision at both the stage and step levels for failure localization. Experiments show that CodeTracer substantially outperforms direct prompting and lightweight baselines, and that replaying its diagnostic signals consistently recovers originally failed runs under matched budgets. Our code and data are publicly available.

### 1. 核心问题与整体含义（研究动机和背景）
随着大语言模型（LLM）驱动的代码智能体（Code Agents）在处理复杂软件工程任务（如修复 Bug、重构代码）时变得日益复杂，其执行过程呈现出**长序列、多阶段、并行化**的特点。
*   **痛点**：当智能体任务失败时，开发者很难从成千上万行的异构日志中定位“第一处错误”发生在哪里。早期的微小失误往往会引发连锁反应，导致智能体陷入无效循环或产生根本性错误。
*   **研究目标**：实现**故障起点定位（Failure Onset Localization）**，即自动识别轨迹中最早触发失败的关键决策点，并提供可解释的证据，从而支持智能体的调试与自我修复。

### 2. 方法论：核心思想与关键技术
CodeTracer 提出了一套将异构运行记录转化为结构化、可追溯状态的架构，包含三个核心阶段：
*   **演进式提取（Evolving Extraction）**：针对不同智能体框架产生的杂乱文件布局，系统会自动扫描目录并生成布局规范。它维护一个解析器注册表，若遇到新格式则自动合成并注册新解析器，将原始日志标准化为包含动作、观察、Diff 和验证结果的步骤记录。
*   **层级树索引（Tree Indexing）**：将扁平的步骤序列重构为**层级追踪树**。
    *   **探索节点（Exploration Nodes）**：仅查看环境而不改变状态的操作（如 `ls`, `grep`）。
    *   **状态改变节点（State-changing Nodes）**：修改代码或环境的操作（如 `patch`, `compile`），这些操作会诱发状态转移，生成子节点。
*   **结构化诊断（Diagnosis）**：诊断代理遍历追踪树，通过“验证回归”、“Diff 强度”、“回溯频率”等特征进行评分，最终输出故障发生的阶段、具体错误步骤及支撑证据。

### 6. 主要结论与发现
*   **证据-行动鸿沟（Evidence-to-Action Gap）**：智能体经常能通过探索找到正确的诊断证据，但无法将其转化为正确的代码修改动作。
*   **成功率饱和点**：增加迭代步数（max_iterations）在达到约 40 步后收益迅速递减，额外的步数多用于无效循环而非纠错。
*   **阶段性错误规律**：环境错误多发生于早期，而假设错误和补丁错误集中在后期。
*   **诊断性能**：CodeTracer 在故障定位上显著优于基线，F1 分数达到 46%-48%，且通过注入诊断信号，原本失败的任务重放成功率提升了 5%-10%。

### 7. 优点与亮点
*   **架构创新**：将复杂的非结构化日志转化为层级树，符合软件工程中“状态转移”的本质逻辑。
*   **实用性强**：提出的“演进式提取器”解决了不同智能体框架日志格式不统一的痛点，具有很强的通用性。
*   **高质量资源**：贡献了 CodeTraceBench，填补了代码智能体过程级（Process-level）评估基准的空白。
*   **闭环验证**：不仅做诊断，还通过“重放”实验证明了诊断结果能切实帮助智能体修复错误。

### 8. 不足与局限
*   **标注主观性**：尽管有指南，但对于长轨迹中“关键错误点”的判定仍存在一定的人工解释偏差。
*   **场景覆盖**：主要集中在终端和代码修复任务，对于涉及图形界面（GUI）或复杂多智能体协作的场景覆盖不足。
*   **离线限制**：目前的重放实验是在匹配预算的离线设置下进行的，无法完全模拟在线交互中人类干预或动态环境变化的影响。
*   **模型依赖**：诊断效果高度依赖于底层模型（如 GPT-5/Claude）的推理能力，在弱模型上的表现尚未充分验证。
