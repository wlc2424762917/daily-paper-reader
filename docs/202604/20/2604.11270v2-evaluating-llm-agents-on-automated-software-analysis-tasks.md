## 摘要
尽管目前存在众多的软件分析工具，但由于环境搭建、依赖解析和工具配置等方面的困难，将这些工具应用于各种开源项目仍然具有挑战性。基于大语言模型（LLM）的智能体提供了一种潜在的解决方案，但此前尚无研究系统地探讨它们在自动化软件分析这一特定任务上的有效性。与问题解决或通用环境搭建不同，该任务需要针对目标项目安装并配置独立的分析工具，生成工具特定的先决条件，并验证工具是否产生了有意义的分析输出。我们推出了 AnalysisBench，这是一个包含 35 个“工具-项目”对的基准测试，涵盖了 7 种分析工具和 10 个不同的 C/C++ 及 Java 项目，每个项目都配有人工构建的参考配置。利用 AnalysisBench，我们在四种 LLM 后端上评估了四种智能体架构。我们自定义的智能体 AnalysisAgent 实现了 94% 的人工验证成功率（使用 Gemini-3-Flash，完成 33/35 个任务），而表现最好的基准智能体（ExecutionAgent）成功率为 77%。除了定量结果外，我们还识别了现有智能体的主要局限性，包括阶段混淆、错误定位能力差以及过早终止，并证明了智能体架构比单纯的 LLM 能力更为重要。我们进一步发现，全程序分析和符号执行是最困难的任务，Java 工具链比 C/C++ 带来的挑战更大，且 LLM 自我验证的成功率始终高于人工验证的成功率。

## Abstract
Numerous software analysis tools exist today, yet applying them to diverse open-source projects remains challenging due to environment setup, dependency resolution, and tool configuration. LLM-based agents offer a potential solution, yet no prior work has systematically studied their effectiveness on the specific task of automated software analysis, which, unlike issue solving or general environment setup, requires installing and configuring a separate analysis tool alongside the target project, generating tool-specific prerequisites, and validating that the tool produces meaningful analysis outputs. We introduce AnalysisBench, a benchmark of 35 tool-project pairs spanning seven analysis tools and ten diverse C/C++ and Java projects, each with a manually constructed reference setup. Using AnalysisBench, we evaluate four agent architectures across four LLM backends. Our custom agent, AnalysisAgent, achieves manually verified success rates of 94% (Gemini-3-Flash, 33/35 tasks), compared to 77% for the best baseline (ExecutionAgent). Beyond quantitative results, we identify key limitations in existing agents, including stage mixing, poor error localization, and premature termination, and show that agentic architecture matters more than LLM capability alone. We further find that whole-program analyses and symbolic execution are the most difficult tasks, that Java toolchains pose greater challenges than C/C++, and that LLM-self-validated success consistently overstates manually verified success.

*   **核心问题**：尽管软件分析工具（如静态分析、模糊测试、符号执行）功能强大，但由于环境配置复杂、依赖解析困难以及工具特定的前置要求，将这些工具应用于多样化的开源项目极其耗时且易出错。
*   **研究动机**：现有的 LLM 智能体主要关注代码修复或通用环境搭建，缺乏对“自动化软件分析”这一特定任务的系统研究。该任务具有独特性：它不仅需要配置项目环境，还需要安装独立的分析工具、生成特定中间产物（如 LLVM bitcode），并验证分析输出的有效性。

### 2. 核心方法论：AnalysisAgent
作者提出了 **AnalysisAgent**，其核心设计基于三个原则：
*   **显式阶段化执行（Staged Execution）**：将任务划分为四个线性阶段：(1) Docker 环境搭建；(2) 分析工具安装与冒烟测试；(3) 目标项目构建与产物准备；(4) 最终分析运行。每个阶段有特定的指令和准入准则，防止“阶段混淆”。
*   **单动作循环与确定性日志压缩（Log Condensation）**：采用 ReAct 模式，但通过确定性模式匹配（而非 LLM 总结）来压缩冗长的构建日志。保留前 1000 字符、匹配 70 多种错误模式的行以及后 1000 字符，帮助模型精准定位错误。
*   **基于证据的验证（Evidence-Based Validation）**：引入“LLM 作为裁判（LLM-as-judge）”机制。智能体完成任务后需提交包含阶段总结、运行日志和输出文件位置的“证据包”，由独立的 LLM 验证是否产生了项目相关的分析结果，而非仅仅是工具的帮助信息。

### 6 主要结论与发现
*   **架构胜过模型**：专门设计的 AnalysisAgent 在 Gemini-3-Flash 下达到了 **94%** 的人工验证成功率，远超最佳基准（77%）。这证明了针对任务的架构设计比单纯提升模型规模更有效。
*   **生态系统差异**：Java 任务的失败率远高于 C/C++，主因是 Java 工具链（如字节码生成、JVM 附加）更为沉重且依赖复杂。
*   **验证偏差**：LLM 自我验证的成功率普遍高于人工验证，说明智能体容易被表面的“运行成功”信号误导，必须引入严格的产物检查。
*   **任务难度**：全程序分析和符号执行是最难自动化的任务，因为它们对构建产物（如编译数据库、bitcode）有极高要求。

### 7. 优点
*   **首创性**：填补了 LLM 智能体在自动化软件分析领域评估的空白。
*   **实用性**：提出的日志压缩和阶段化执行策略直接解决了 LLM 处理长文本构建日志时的“迷失”问题。
*   **高质量基准**：AnalysisBench 提供了人工构建的参考配置，为后续研究提供了可靠的衡量标准。

### 8. 不足与局限
*   **项目类型受限**：目前仅覆盖了命令行程序（CLI），未涉及库（Library）或框架，后者通常需要编写复杂的测试桩（Harness）。
*   **语言覆盖**：仅限于 C/C++ 和 Java，未探讨 Python、Rust 等其他流行语言。
*   **动态分析深度**：由于任务时间限制，动态分析工具（如 AFL++）虽然配置成功，但运行时间较短，产生的分析深度（如代码覆盖率）有限。
*   **人工验证成本**：虽然结果可靠，但依赖人工验证难以在大规模（如数千个项目）下扩展。
