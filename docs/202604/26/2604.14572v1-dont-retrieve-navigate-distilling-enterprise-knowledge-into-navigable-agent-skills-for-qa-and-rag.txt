## Abstract 

Retrieval-Augmented Generation (RAG) grounds LLM responses in external evidence but treats the model as a passive consumer of search re-sults: it never sees how the corpus is organized or what it has not yet retrieved, limiting its abil-ity to backtrack or combine scattered evidence. We present C ORPUS 2S KILL , which distills a doc-ument corpus into a hierarchical skill directory offline and lets an LLM agent navigate it at serve time. The compilation pipeline iteratively clusters documents, generates LLM-written summaries at each level, and materializes the result as a tree of navigable skill files. At serve time, the agent receives a bird’s-eye view of the corpus, drills into topic branches via progressively finer sum-maries, and retrieves full documents by ID. Be-cause the hierarchy is explicitly visible, the agent can reason about where to look, backtrack from unproductive paths, and combine evidence across branches. On WixQA, an enterprise customer-support benchmark for RAG, C ORPUS 2S KILL 

outperforms dense retrieval, RAPTOR, and agen-tic RAG baselines across all quality metrics. 

## 1. Introduction 

Enterprise knowledge bases contain thousands of hetero-geneous documents: support articles, product guides, pol-icy documents, and FAQs (Gao et al., 2024). Large lan-guage models alone cannot answer questions about these corpora reliably: they lack access to private, frequently updated content and are prone to hallucinating plausible but incorrect details (Lewis et al., 2020; Gao et al., 2024). Retrieval-Augmented Generation (RAG; Lewis et al., 2020) was introduced to solve these problems by grounding LLM responses in retrieved evidence. The LLM receives these fragments but has no understanding of how the corpus is organized, what topic areas exist beyond the retrieved set, or whether the best-matching documents were even in the retrieval window. It cannot reason about where else to look, whether its evidence is complete, or how the retrieved passages relate to the broader knowledge structure. In short, conventional RAG treats the LLM as a passive consumer of search results, leaving the model’s planning and reasoning capabilities largely unused. Agentic RAG systems (Yao et al., 2023; Schick et al., 2023) partially address this by allowing the LLM to issue mul-tiple search queries iteratively. But even with tool access, the agent operates without a map: it must guess productive search terms, has no way to survey the corpus structure, and cannot systematically narrow its focus from broad topics to specific documents. Each search query is a shot in the dark, guided only by the agent’s prior knowledge and whatever fragments previous queries returned. For the business-type query above, an agentic retrieval system would need to in-dependently discover that the answer involves both the pay-ments ecosystem and account verification policies — topics whose connection is not apparent from keyword similarity alone. Hierarchical approaches such as RAPTOR (Sarthi et al., 2024) and GraphRAG (Edge et al., 2024) organize docu-ments into trees or graphs with LLM-generated summaries at internal nodes. These summaries capture thematic struc-ture that flat embeddings miss. However, these systems still rely on embedding-based similarity search at query time: RAPTOR indexes all tree nodes in a flat vector store, while GraphRAG requires a graph database and entity extraction pipeline. The hierarchy enriches the retrieval candidates but remains invisible to the LLM at query time; the model still receives search results without understanding the orga-nizational structure that produced them. More recent work pushes further: StructRAG (Li et al., 2025) restructures re-trieved information into tables or graphs at inference time, and BookRAG (Wang et al., 2025) builds hierarchical in-dices with agent-based planners. Yet these systems still invoke embedding-based search at query time and do not give the agent direct access to the organizational structure itself. Meanwhile, a parallel line of work on LLM agent skills (Wang et al., 2023; Jiang et al., 2026) has shown that agents perform complex tasks more effectively when given struc-tured, reusable knowledge modules that they can selectively load and follow. Agent platforms now support filesystem-based skill packages 1 that the agent discovers via metadata

and reads on demand, a mechanism known as progressive disclosure (Section 2.3). This suggests an alternative to re-trieval: rather than searching for passages, distill the corpus into a navigable skill hierarchy and let the agent explore it. We propose exactly this: make the hierarchy itself the pri-mary interface for the LLM. Rather than searching over ahierarchy, the agent navigates through it. This is the core idea behind C ORPUS 2S KILL (Figure 1): a one-time offline compilation phase transforms a document corpus into a nav-igable skill tree, and at serve time, the LLM agent browses this tree to find information, much like a human navigating a well-organized file system or table of contents. We evaluate C ORPUS 2S KILL on WixQA (Cohen et al., 2025), an enterprise customer-support benchmark designed for evaluating RAG systems over real-world knowledge bases. C ORPUS 2S KILL achieves the highest quality across all metrics, outperforming flat retrieval, RAPTOR, and a multi-turn agentic RAG system, while requiring no em-bedding index or vector database at serve time. These re-sults suggest a complementary design point for knowledge-grounded systems: invest computation at compile time to make the corpus navigable , rather than solely at query time to make it searchable. We summarize our contributions as follows. 1. We identify corpus navigability as an underexplored dimension in current RAG systems: LLMs lack a struc-tured overview of the knowledge base, which limits their ability to systematically locate the best evidence. 2. We propose C ORPUS 2S KILL , a compile-then-navigate framework: an offline pipeline transforms a raw corpus into a hierarchical skill directory via iterative clustering and LLM-based summarization, and at serve time an LLM agent navigates this hierarchy through file brows-ing and document lookup tools, requiring no embedding index or vector database at serve time. 3. We evaluate on WixQA (Cohen et al., 2025) against five baselines and conduct ablation studies on hierarchy structure, exploration budget, and serving model choice. 

## 2. Background: Skills in LLM Agents 

2.1. From Tools to Skills 

Modern LLM agents interact with external systems through 

tools : atomic, stateless functions such as web search or database queries (Schick et al., 2023; Yao et al., 2023). Tools give an agent the ability to act but do not tell it how to approach a complex task. Agent skills fill this gap: reusable modules that package procedural knowledge or multi-step workflows into a form the agent can load and follow (Jiang et al., 2026). Unlike a tool, a skill shapes reasoning over mul-tiple steps, persisting across tool calls to guide the agent’s 2Don’t Retrieve, Navigate: Distilling Enterprise Knowledge into Navigable Agent Skills 

decisions throughout a task. This distinction is increasingly important as agents tackle longer-horizon tasks where choos-ing the right sequence of tool calls requires domain-specific strategy, not just tool availability. Formally, Jiang et al. (2026) define a skill as a tuple S =(C, π, T, R ): an applicability condition C, an execution policy π, a termination condition T , and a reusable interface 

R. This distinguishes skills from one-off plans (no reusable interface), episodic memories (no execution policy), and atomic tools (no multi-step structure). 

2.2. Skill Representations 

Skills take several forms in practice: • Code-as-skill. Voyager (Wang et al., 2023) stores suc-cessful action sequences as executable functions in a growing library. • Natural-language playbooks. SkillX (Wang et al., 2026) and EvoSkill (Alzubi et al., 2026) distill agent experience into structured textual instructions. • Filesystem-based packages. A directory containing a structured instruction file ( SKILL.md ) alongside op-tional scripts and data assets. This is the pattern adopted by recent agent platforms and the one C ORPUS 2S KILL 

builds on. Code-based skills offer precision but require execution en-vironments; natural-language skills are portable and inter-pretable at the cost of ambiguity. Filesystem-based packages balance the two by combining structured text with auxiliary resources that the agent can access on demand. 

2.3. Progressive Disclosure 

Loading all skill content at startup would overwhelm the context window; loading nothing would leave the agent unable to choose. The filesystem pattern resolves this via 

progressive disclosure (Shneiderman, 1996): 1. Metadata (at startup): skill name and one-line descrip-tion ( <100 tokens each). 2. Instructions (on demand): the full SKILL.md is read when the agent selects a skill. 3. Resources (as needed): supporting files remain on disk, consuming zero tokens until accessed. The agent thus maintains awareness of many skills while paying the token cost only for those actually used. 

2.4. From Procedural Skills to Informational Skills 

In prior work, skills encode how to perform a task : work-flows, decision trees, or code patterns. CORPUS 2S KILL 

repurposes the same mechanism to encode what a corpus contains . Each skill directory represents a topical parti-tion, with SKILL.md and INDEX.md files that summarize its contents at increasing levels of detail. The agent fol-lows the same progressive disclosure workflow (description 

→ SKILL.md → INDEX.md → document) but navigates an information hierarchy rather than executing procedu-ral instructions. This reinterpretation—from procedural to informational skills—is the central design insight of C OR -

2S KILL .

## 3. C ORPUS 2S KILL Framework 

CORPUS 2S KILL operates in two phases: a one-time offline 

compile phase that transforms a raw document corpus into a browsable skill tree, and a per-query online serve phase 

where an LLM agent navigates this tree with full visibil-ity into the corpus structure. Figure 2 illustrates the full architecture. 

3.1. Problem Setup 

Let D = {d1, . . . , d N } be a corpus of N documents. Given a natural-language query q, the task is to produce an answer 

a grounded in a subset Dq ⊆ D of relevant documents. Standard RAG systems select Dq via a retrieval function 

f (q, D) → D q that operates as a black box from the LLM’s perspective: the model receives Dq but has no visibility into what D contains or why Dq was chosen. CORPUS 2S KILL replaces this opaque retrieval step with 

agent-driven navigation . The compile phase constructs a hierarchical skill forest S = {s1, . . . , s K } where each skill 

sk is a navigable directory tree of summaries covering a top-ical partition of D. Following the agentic skills formalism of Jiang et al. (2026), each skill can be viewed as a tu-ple (C, π, T, R ): the applicability condition C is the skill’s summary (the agent reads it to decide relevance); the execu-tion policy π is the navigation workflow (descend through summaries, then retrieve documents); the termination con-dition T is the agent’s judgment that sufficient evidence has been gathered; and the reusable interface R is the stan-dardized file structure ( SKILL.md , INDEX.md ) and the 

get document tool. At serve time, the agent selects and navigates skills autonomously, making the document selec-tion process transparent and steerable.

Documents are truncated to a maximum character length and embedded using a sen-tence embedding model (Reimers & Gurevych, 2019), pro-ducing a dense vector for each document. These embed-dings capture semantic similarity between documents and serve as the foundation for the subsequent clustering step. 

Iterative hierarchical clustering. We build a multi-level hierarchy through an iterative bottom-up process controlled by two parameters: the branching ratio p (target number of children per cluster) and the maximum number of top-level clusters K. At each level, the current set of n embeddings is partitioned into ⌈n/p ⌉ groups via K-Means (Lloyd, 1982) on L2-normalized vectors. Each resulting cluster is then summarized by an LLM that reads representative member texts and produces a concise description of the cluster’s top-ical coverage, the types of questions the documents answer, and key terms or features mentioned. These summaries are re-embedded using the same sentence embedding model, and the resulting embeddings become the input to the next round of K-Means. This process repeats until the number of remaining clusters falls below K, yielding a set of root clusters that form the top level of the hierarchy. This iterative embed-cluster-summarize loop is inspired by the recursive tree construction in RAPTOR (Sarthi et al., 2024), but adapted for our navigability goal. We use K-Means with hard assignments so that each document maps to exactly one path in the directory tree, and we materialize the hierarchy as browsable files rather than indexing it in a flat vector store. These two choices, deterministic partitioning and filesystem materialization, are what enable agent-driven navigation at serve time. Small clusters below a minimum size are merged into the nearest centroid to avoid orphan groups. 

Labeling. After the clustering hierarchy stabilizes, the LLM generates a short (2 to 5 word) filesystem-safe label for each non-leaf node. These labels serve a dual purpose: they provide human-readable directory names for inspection (e.g., wix-commerce-monetization ) and, more im-portantly, they give the agent semantically meaningful entry points for topic routing. When the agent reads a SKILL.md 

listing subgroups such as “wix-payments-ecosystem” or “domain-dns-management,” the labels alone often suffice to identify the relevant branch without reading full summaries. 

Skill tree construction. The hierarchy is materialized into a filesystem structure designed for agent navigation: • Root clusters become top-level skill directories, each containing a SKILL.md with the cluster summary and a listing of child groups. • Sub-clusters become subdirectories with INDEX.md

3.3. Serve Phase 

At serve time, the pre-compiled skill directories are up-loaded to Anthropic’s Skills API. The API implements pro-gressive disclosure: skill names and one-line descriptions are pre-loaded into the agent’s context, but full file content is loaded only when the agent explicitly reads a file. This design is critical for token efficiency. Pre-loading all naviga-tion files would flood the context with thousands of tokens of irrelevant summaries; loading nothing would leave the agent blind to the corpus. Progressive disclosure strikes a balance: the agent sees just enough (skill names and descrip-tions, approximately 200 tokens total) to make an informed first decision, then incrementally loads detail as it descends. 

Tools. The agent receives two tools: (1) 

code execution , which allows browsing SKILL.md 

5Don’t Retrieve, Navigate: Distilling Enterprise Knowledge into Navigable Agent Skills 

and INDEX.md files through view commands, providing the agent with full visibility into the hierarchy; and (2) 

get document(doc id) , a custom tool that retrieves the full text of a document from the document store given its identifier. The two-tool design reflects the two types of information the agent needs: navigational (where to look) and evidential (what to cite). Keeping them separate means the agent can survey many document summaries cheaply before committing to the token cost of reading full documents. 

Navigation workflow. Because skill descriptions are pre-loaded, the agent begins each query with a bird’s-eye view of the corpus, knowing what topics exist and roughly how many documents each topic covers. A typical query pro-ceeds in 2 to 3 turns: 1. The agent identifies the most relevant skill from pre-loaded descriptions and reads its SKILL.md to under-stand the subgroup structure. 2. The agent navigates into the relevant subgroup’s 

INDEX.md , which lists document identifiers with brief titles and summaries. 3. The agent calls get document to retrieve the full text of one or more promising documents and synthesizes a grounded answer. At each step, the agent makes an informed decision guided by summaries describing each branch. If a SKILL.md 

lists ten subgroups and the agent has only explored one, it knows nine remain—an awareness that flat retrieval cannot provide. This structural visibility enables two behaviors: tar-geted backtracking (abandoning an unpromising branch for a more relevant one) and cross-branch synthesis (combining evidence from multiple subgroups within or across skills). The system prompt enforces a grounding rule (every claim must trace to a retrieved document, not a summary file) and a concise-answer format (word limits, no preamble); see Section D. 

## 4. Experiments 

4.1. Setup Dataset. We evaluate on WixQA (Cohen et al., 2025), a customer support QA benchmark built from the Wix knowl-edge base. The corpus contains 6,221 support articles cov-ering website building, e-commerce, SEO, marketing, and platform features. The evaluation set consists of 200 expert-written questions with gold-standard answers and gold arti-cle IDs. 

Compilation. We compile the WixQA corpus with branching ratio p=10 and maximum top-level clusters K=7 .This produces a 3-level hierarchy with 6 top-level skills, 665 navigation files, and a 13 MB document store. Compilation takes 6.5 minutes on a 32-CPU server. 

Baselines. All methods use the same LLM for answer generation and the same evaluation protocol. We compare five baselines spanning three retrieval paradigms: BM25 

(sparse keyword retrieval (Robertson & Zaragoza, 2009)), 

Dense (Qwen3-Embedding-0.6B (Zhang et al., 2025) with a FAISS index (Johnson et al., 2019)), Hybrid (Reciprocal Rank Fusion (Cormack et al., 2009) of BM25 and Dense), 

RAPTOR (hierarchical tree retrieval (Sarthi et al., 2024) with UMAP+GMM+BIC clustering), and Agentic (an LLM agent with iterative access to BM25, Dense, and Hybrid tools). The first three are single-shot (top-5 passages, one LLM call); RAPTOR uses collapsed tree retrieval; Agentic is allowed up to 10 interaction rounds. Full implementation details are in Section H. 

Metrics. We report four lexical overlap metrics: Token F1 

(harmonic mean of token-level precision and recall against the gold answer), BLEU (Papineni et al., 2002) (n-gram precision with brevity penalty), ROUGE-1 and ROUGE-2 

(Lin, 2004) (unigram and bigram recall). We additionally re-port two LLM-judged metrics following the LLM-as-judge paradigm (Zheng et al., 2023), inspired by RAGAS (Es et al., 2024): Factuality measures how factually correct the generated answer is relative to the gold answer (1–5 scale, normalized to 0–1), and Context Recall measures how well the retrieved context covers the essential claims in the gold answer (same scale). We also report input token usage and per-query cost. Details on metric computation and fairness standardization across methods are in Section I.

## 5. Limitations 

Cost. The per-query cost of C ORPUS 2S KILL is dominated by input tokens: the Skills API includes navigation file con-tent in every call, so even relatively small SKILL.md and 

INDEX.md files accumulate significant token counts across turns. C ORPUS 2S KILL is best suited for high-value queries where answer quality justifies the cost premium, such as complex multi-hop support questions or compliance-critical lookups. For high-volume, low-stakes queries, single-shot retrieval or RAPTOR remain cost-effective. Prompt caching could substantially reduce this overhead by reusing repeated skill file content across queries within the same session. 

API constraints. The framework relies on Anthropic’s Skills API, which imposes limits of 8 skills per request, 200 files per skill, and 30 MB per skill. These constraints shape the hierarchy design: the 8-skill limit caps top-level clusters, and the 200-file budget constrains tree depth before requir-ing compact merging (Section 4.4). The WixQA corpus fits comfortably (6 skills, ≤133 files each), but larger corpora would require compact merging or multi-level partitioning. As Skills APIs evolve and constraints relax, even narrower hierarchies could improve navigation precision. 

Hard single-path clustering. Each document is assigned to exactly one cluster path, a deliberate choice that en-ables filesystem materialization and deterministic naviga-tion. However, documents that span multiple topics (e.g., billing and subscriptions) can only appear under one branch, creating a routing bottleneck when queries target the other topic. Our failure analysis (Section G) confirms that top-level routing errors account for the majority of failures. Soft or multi-parent assignment could mitigate this at the cost of duplicated content and more complex navigation; we leave this exploration to future work. 

Compilation. The one-time compilation costs $5–$10 in LLM calls plus 6.5 minutes of wall-clock time, negligi-ble for enterprise deployments handling thousands of daily queries. The pipeline is fully automated and deterministic given fixed random seeds, but does not yet support incremen-tal updates—adding documents requires recompilation. 

## 6. Related Work 

Cluster-based document organization. Scatter/Gather (Cutting et al., 1992) introduced the paradigm of navigating a pre-computed cluster organization rather than matching by similarity, grounded in the cluster hypothesis (van Rijs-bergen, 1979; Liu & Croft, 2004). Where Scatter/Gather required users to manually select and refine clusters in an 8Don’t Retrieve, Navigate: Distilling Enterprise Knowledge into Navigable Agent Skills 

interactive loop, C ORPUS 2S KILL delegates this navigation entirely to an LLM agent that reasons about cluster sum-maries at each level. 

Hierarchical summarization for retrieval. RAPTOR (Sarthi et al., 2024) recursively clusters and summarizes text chunks into a tree, retrieving via layer-by-layer traver-sal or collapsed-tree vector search. C ORPUS 2S KILL shares the compile-time embed-cluster-summarize loop but mate-rializes the hierarchy as navigable files and replaces fixed traversal with agentic navigation. GraphRAG (Edge et al., 2024) builds a knowledge graph with community-level sum-maries; unlike C ORPUS 2S KILL , it requires a graph database at serve time. HiRAG (Huang et al., 2025) augments RAG with a hierarchical knowledge graph that interleaves entity and community levels; it still relies on graph retrieval at query time rather than agent-driven file navigation. 

Retrieval-augmented generation. RAG (Lewis et al., 2020) grounds LLM responses in retrieved passages, with dense passage retrieval (Karpukhin et al., 2020) and late-interaction models (Santhanam et al., 2022) pushing re-trieval quality. Self-RAG (Asai et al., 2024) teaches the model to critique its own retrieval; IRCoT (Trivedi et al., 2023) interleaves retrieval with chain-of-thought reasoning for multi-hop questions, using intermediate reasoning steps to formulate better queries. Recent surveys (Gao et al., 2024; Singh et al., 2025) organize RAG along axes of granularity, timing, and agentic control. C ORPUS 2S KILL occupies a dis-tinct point on this spectrum: rather than improving retrieval, it replaces embedding-based search with agent-driven file navigation over a pre-compiled hierarchy. 

Structure-aware and agentic RAG. StructRAG (Li et al., 2025) dynamically restructures retrieved information into tables, graphs, or catalogues at inference time, letting the LLM reason over structured views. BookRAG (Wang et al., 2025) builds a hierarchical index mirroring a document’s table of contents and uses an agent-based planner to query it. A-RAG (Du et al., 2026) exposes hierarchical retrieval in-terfaces (keyword, semantic, chunk-read) to an LLM agent, and SPD-RAG (Akay et al., 2026) assigns a dedicated sub-agent per document in a multi-agent pipeline. NaviRAG (Dai et al., 2026) builds a hierarchical view of documents and uses an LLM agent to actively navigate it, detecting in-formation gaps and retrieving at the appropriate granularity. HCAG (Wu & Deng, 2026) constructs a multi-resolution knowledge base offline and performs hierarchical retrieval with a multi-agent discussion at query time. All of these sys-tems retain some form of retrieval or search infrastructure at serve time. C ORPUS 2S KILL differs by materializing the hierarchy as a static filesystem of skill files and navigating it with a single agent using file browsing and document lookup rather than embedding-based search. 

Tool-use agents. Toolformer (Schick et al., 2023) and ReAct (Yao et al., 2023) established that LLMs can au-tonomously select and use external tools. C ORPUS 2S KILL 

extends this paradigm with domain-specific, read-only tools (file browsing and document lookup) over a static hierarchy, making navigation deterministic and reproducible given the same compiled skill tree. 

Agent skill construction. Voyager (Wang et al., 2023) builds a skill library from successful action sequences as code; SkillX (Wang et al., 2026) and EvoSkill (Alzubi et al., 2026) distill agent trajectories into reusable knowl-edge. These create skills from trajectories , whereas C OR -

> PUS

2S KILL creates skills from document corpora via clus-tering and summarization, a fundamentally different input modality. 

Positioning. CORPUS 2S KILL combines offline hierarchi-cal clustering, materialization as navigable skill files, and LLM-driven top-down navigation into a system that replaces embedding-based retrieval with agent-driven file browsing at serve time. While NaviRAG and HCAG share the mo-tivation of active hierarchical navigation, C ORPUS 2S KILL 

is distinguished by its filesystem-based skill representation, progressive disclosure via SKILL.md /INDEX.md files, and reliance on a standard code-execution tool rather than custom retrieval APIs. Table 6 in the Appendix provides a detailed feature comparison. 

## 7. Conclusion 

CORPUS 2S KILL demonstrates that giving an LLM agent a structured, browsable overview of a document corpus leads to higher-quality answers than providing it with retrieved passages alone. The key insight is that modern LLM agents are capable navigators when given the right map: 

a pre-compiled hierarchy of summaries and topic clusters enables informed, top-down exploration that exploits the agent’s reasoning abilities rather than treating it as a pas-sive consumer of search results. Because the agent reads summaries describing each branch, it can make informed routing decisions and combine evidence across branches when queries span multiple topics—an advantage that flat retrieval, which offers no corpus overview, cannot match. While the approach incurs higher per-query cost due to nav-igation overhead, it achieves the highest quality on WixQA and suggests a complementary design point for knowledge-grounded systems: invest computation at compile time to make the corpus navigable, rather than at query time to make it searchable. Hierarchy depth grows logarithmically (O(log p N )), so the framework scales to larger corpora. Future work includes evaluation on more domains, incre-mental compilation, and prompt caching to cut query cost. 9Don’t Retrieve, Navigate: Distilling Enterprise Knowledge into Navigable Agent Skills 

## Impact Statement 

CORPUS 2S KILL is designed for enterprise knowledge-grounded QA, a setting where incorrect answers can lead to real-world harm (e.g., wrong compliance guidance or billing instructions). Like all LLM-based systems, it can produce confidently wrong answers when the agent nav-igates to irrelevant documents or misinterprets retrieved content. Hierarchical summaries produced during compi-lation may also expose corpus structure that organizations consider sensitive. Hard clustering means that documents spanning multiple topics are routed to a single branch, cre-ating blind spots that may disproportionately affect certain query types. Finally, because compilation is a batch process, the system cannot reflect real-time corpus updates, risking stale or outdated answers until recompilation occurs. We recommend human-in-the-loop verification for high-stakes deployments.

## G. Failure Analysis 

We examine the 62 queries (31%) on which C ORPUS 2S KILL 

scores Factuality ≤0.4 or Context Recall ≤0.2. By compar-ing each query’s retrieved documents against the gold article IDs (using prefix matching, since C ORPUS 2S KILL uses trun-cated content hashes), we identify three failure modes: 

Navigation miss (38 queries). The agent retrieves doc-uments but none match the gold article, indicating it de-scended into the wrong branch of the hierarchy. For ex-ample, a query about reordering CMS collection items is routed to a general site-editing skill instead of the CMS-specific one; a query about blocking time slots in a staff schedule navigates to a generic booking group rather than the staff-management leaf. These misroutes occur at the top level of the hierarchy, where broad skill summaries fail to distinguish closely related topics. 

Partial navigation (19 queries). The agent’s retrieved set includes the gold document, yet scores remain low. In these cases the gold article is retrieved alongside several non-relevant documents, diluting the context provided to the answer synthesis step. The agent correctly identified the general topic area but over-retrieved from neighboring leaf groups. 

Synthesis error (3 queries). The agent retrieves contextu-ally relevant documents (Context Recall ≥0.6) but produces a factually incorrect answer, typically by over-generalizing across multiple articles or misinterpreting conditional in-structions as universal. 

No documents retrieved (2 queries). Two queries (“How do I clear my browser cache?” and a Gmail password reset question) resulted in zero retrieved documents. Both have gold articles in the corpus (Wix publishes help articles for these topics), but the agent could not locate a matching skill branch and correctly declined to answer rather than hallucinating. 

Takeaway. The dominant bottleneck is initial topic rout-ing: 38 of 62 failures stem from the agent choosing the wrong top-level branch. The compact-mode ablation (Sec-tion 4.4), which produces narrower top-level clusters, par-tially mitigates this by providing finer-grained topic separa-tion at the first navigation step. 

## H. Baseline Implementation Details 

All baselines use the same LLM (Claude Sonnet) for an-swer generation and the same evaluation prompts for fair comparison. 

BM25. We implement BM25 sparse keyword retrieval us-ing the rank bm25 library (Robertson & Zaragoza, 2009). Documents are tokenized and indexed at the full-article level. At query time, we retrieve the top-5 articles by BM25 score and pass them as context to a single LLM call. 

Dense. We encode all documents using Qwen3-Embedding-0.6B (Zhang et al., 2025), producing 1024-dimensional embeddings stored in a FAISS flat index (Johnson et al., 2019). We use the document prompt template for encoding articles and the query prompt template for encoding questions. At query time, we retrieve the top-5 nearest neighbors by cosine similarity and pass them as context to a single LLM call. 18 Don’t Retrieve, Navigate: Distilling Enterprise Knowledge into Navigable Agent Skills 

Hybrid. We combine BM25 and Dense scores using Re-ciprocal Rank Fusion (Cormack et al., 2009) with k=60 

and equal weighting ( α=0 .5). The fused ranking selects the top-5 passages, which are passed to a single LLM call. 

RAPTOR. We faithfully implement the RAPTOR algo-rithm (Sarthi et al., 2024): UMAP (McInnes et al., 2018) for dimensionality reduction, Gaussian Mixture Model (GMM) soft clustering (McLachlan & Peel, 2000) with BIC-based cluster count selection (Schwarz, 1978), and recursive sum-marization of clusters. We use the same embedding model (Qwen3-Embedding-0.6B) as the Dense baseline for con-sistency. At query time, collapsed tree retrieval searches across all hierarchy levels simultaneously, returning the top-5 passages from any level. 

Agentic. An LLM agent is given access to three search tools (search bm25 , search dense ,

search hybrid ), each returning top-k results. The system prompt instructs the agent to search first, then answer based on accumulated evidence, choosing BM25 for exact terms and product names, dense for conceptual queries, and hybrid as a default strategy. The agent can issue up to 10 rounds of iterative retrieval, reformulating sub-queries based on intermediate results. Evidence is accumulated across all turns; when the agent stops, it produces a structured JSON response containing the answer and curated top-5 references selected from the full evidence pool. For evaluation fairness, context recall is computed over the last 5 retrieved documents (matching the context window of single-shot baselines). This baseline tests whether agentic multi-turn retrieval over traditional search infrastructure can match navigation-based exploration. 

## I. Evaluation Metrics 

Lexical metrics. Token F1 tokenizes both the predicted and gold answers into word tokens and computes the har-monic mean of token-level precision and recall. BLEU 

(Papineni et al., 2002) measures modified n-gram precision (up to 4-grams) with a brevity penalty to penalize overly short outputs. ROUGE-1 and ROUGE-2 (Lin, 2004) mea-sure unigram and bigram recall respectively, capturing how much of the gold answer’s content appears in the prediction. 

LLM-judged metrics. Both Factuality and Context Re-call use an LLM judge (Zheng et al., 2023) following the RAGAS framework (Es et al., 2024), scoring on a 1–5 scale normalized to 0–1. Factuality evaluates whether the claims in the generated answer are factually consistent with the gold answer, independent of the retrieved context. Context Recall evaluates whether the retrieved context contains suf-ficient information to answer the question, by checking how many essential claims in the gold answer can be attributed to the provided context. 

Context Recall standardization. A critical design deci-sion is the standardization of Context Recall measurement across different retrieval paradigms (single-shot, multi-turn, and navigation-based). For fair comparison, all methods follow the same protocol: (1) collect the last 5 documents or passages retrieved by the method; (2) join them with separator tokens; (3) truncate to a uniform 8,000-character window. The LLM judge receives this standardized context. This ensures that single-shot baselines contribute their top-5 retrieved passages, Agentic contributes the last 5 documents from its multi-turn retrieval, and C ORPUS 2S KILL con-tributes the last 5 documents retrieved via get document 

tool calls. 

Cost metrics. Input token count is the total number of tokens sent to the LLM across all turns. Per-query cost is computed using the model’s published per-token pricing (in-put and output tokens combined). For single-shot methods, cost reflects one LLM call; for multi-turn methods (Agen-tic, C ORPUS 2S KILL ), it reflects all turns including tool-call reasoning. 19
