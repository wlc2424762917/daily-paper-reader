## 摘要
模型上下文协议 (MCP) 已成为连接大语言模型 (LLM) 智能体与外部工具的通用接口，但其对无状态、急切模式注入的依赖带来了一种隐形的每轮开销——即“MCP 税”或“工具税”。根据从业者报告，在典型的多服务器部署中，这种开销通常在 1 万到 6 万个 token 之间。这种负载膨胀了键值 (KV) 缓存，且随着上下文利用率接近约 70% 的已知性能断裂点，会导致推理能力下降，并将 token 预算转变为一项持续的运营成本。我们提出了工具注意力 (Tool Attention)，这是一种中间件层机制，它将“Attention Is All You Need”范式从针对 token 的自注意力推广到针对工具的门控注意力。工具注意力结合了：(i) 基于句子嵌入的意图模式重叠 (ISO) 分数；(ii) 强制执行前置条件和访问范围的状态感知门控函数；以及 (iii) 一个两阶段延迟模式加载器，该加载器在上下文中保留紧凑的摘要池，并仅为前 k 个门控工具加载完整的 JSON 模式。我们在一个模拟的包含 120 个工具、6 台服务器的基准测试中进行了评估，其每台服务器的 token 计数根据真实 MCP 部署的公开审计进行了校准。在此模拟中，工具注意力直接将测得的每轮工具 token 数减少了 95.0%（从 47.3k 降至 2.4k），并将有效上下文利用率（一种 token 比例指标）从 24% 提高到 91%。关于任务成功率、延迟、成本和推理质量的端到端数据是基于测得的 token 计数结合已发布的部署遥测数据得出的预测值；这些数据并非在实时 LLM 智能体上测得，我们在文中明确标注了这些预测值。总而言之，这些结果支持一个简单的论点：协议级效率，而非原始上下文长度，才是可扩展智能体系统的关键约束。

## Abstract
The Model Context Protocol (MCP) has become a common interface for connecting large language model (LLM) agents to external tools, but its reliance on stateless, eager schema injection imposes a hidden per-turn overhead the MCP Tax or Tools Tax that practitioner reports place between roughly 10k and 60k tokens in typical multi-server deployments. This payload inflates the key-value cache, is associated with reasoning degradation as context utilization approaches published fracture points around 70%, and turns token budgets into a recurring operational cost. We introduce Tool Attention, a middleware-layer mechanism that generalizes the "Attention Is All You Need" paradigm from self-attention over tokens to gated attention over tools. Tool Attention combines (i) an Intent Schema Overlap (ISO) score from sentence embeddings, (ii) a state-aware gating function enforcing preconditions and access scopes, and (iii) a two-phase lazy schema loader that keeps a compact summary pool in context and promotes full JSON schemas only for top-k gated tools. We evaluate on a simulated 120-tool, six-server benchmark whose per-server token counts are calibrated to public audits of real MCP deployments. In this simulation, Tool Attention directly reduces measured per-turn tool tokens by 95.0% (47.3k -> 2.4k) and raises effective context utilization (a token-ratio quantity) from 24% to 91%. End-to-end figures for task success, latency, cost, and reasoning quality are reported as projections derived from the measured token counts combined with published deployment telemetry; they are not measured on live LLM agents, and we mark projected values explicitly throughout. Taken together, the results support a simple thesis: protocol-level efficiency, not raw context length, is a binding constraint on scalable gentic systems. The code for this work is accessible at https://github.com/asadani/tool-attention

### 1. 核心问题与整体含义（研究动机和背景）
*   **核心问题：** 现有的模型上下文协议（MCP）在连接智能体与工具时，由于 API 的无状态性，每一轮对话都必须重新注入所有工具的完整 JSON Schema。这导致了所谓的**“工具税”（Tools Tax）**：在典型部署中，每轮对话会产生 10k 到 60k 个 token 的额外开销。
*   **负面影响：**
    *   **经济成本：** 极大地增加了 API 调用费用。
    *   **推理退化：** 当上下文利用率超过 70% 时，模型推理质量会大幅下降（出现幻觉、参数混淆）。
    *   **安全风险：** 攻击者可以通过在工具描述中注入恶意指令（工具中毒攻击）来劫持智能体。
*   **研究意义：** 论文认为，协议层面的效率优化（而非单纯增加上下文长度）是构建可扩展智能体系统的关键。

### 2. 方法论：核心思想与关键技术
Tool Attention 借鉴了 Transformer 的注意力机制，将“对 token 的注意力”推广为“对工具的门控注意力”。其核心流程包括：
*   **意图-模式重叠评分 (ISO Score)：** 使用轻量级句子嵌入模型（如 `all-MiniLM-L6-v2`）计算用户查询与工具摘要之间的余弦相似度，识别潜在相关的工具。
*   **状态感知门控 (State-aware Gating)：** 在语义匹配的基础上，增加确定性的前置条件检查（如：是否已授权、是否已完成前置步骤），过滤掉当前状态下不可用的工具。
*   **两阶段延迟模式加载 (Two-phase Lazy Loading)：**
    *   **阶段 1（摘要池）：** 在上下文中仅保留所有工具的极简摘要（每个约 60 token），确保模型知道工具的存在，且这部分内容易于被 Prompt 缓存。
    *   **阶段 2（模式提升）：** 仅针对 ISO 评分最高的 Top-k 个工具，从注册表中动态加载并注入完整的 JSON Schema。
*   **幻觉拒绝门 (Hallucination Gate)：** 中间件会拦截模型尝试调用但未被注入完整模式的工具，并返回结构化错误，引导模型重新选择或澄清意图。

### 3. 实验设计
*   **数据集/场景：** 构建了一个包含 **120 个工具、6 个服务器**（涵盖 GitHub、数据库、Slack、Jira 等领域）的合成 MCP 测试基准。
*   **对比方法 (Baselines)：**
    1.  **Full-Schema (B1)：** 原始 MCP 模式，每轮注入全部 120 个工具定义。
    2.  **Static Pruning (B2)：** 静态剪枝，由人工预选 30 个工具。
    3.  **Simple Retrieval (B3)：** 简单的 Top-k 向量检索，不含状态门控和延迟加载。
    4.  **CLI Lazy Discovery (B4)：** 类似命令行界面的按需发现模式。
*   **评估指标：** 直接测量 Token 消耗和上下文利用率；通过模拟遥测数据**预测**任务成功率、延迟、成本和推理质量。

### 8. 不足与局限
*   **模拟偏差：** 核心性能指标（成功率、质量）是基于公式预测而非实测，在复杂真实场景下的表现可能存在偏差。
