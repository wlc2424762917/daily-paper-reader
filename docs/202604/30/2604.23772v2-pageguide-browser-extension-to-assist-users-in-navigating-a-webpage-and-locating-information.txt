## 1 Introduction 

AI chatbots have become a primary tool for information retrieval, with 62.8% of the internet users citing information retrieval as a top reason for going online [ 10 ]. In WebArena, VisualWebArena, and WebVoyager [ 16 , 20 , 47 ]) platforms, chatbots read the page content, autonomously execute actions ( e.g., clicking buttons and links or filling out forms), and return answers in a chat sidebar. Yet despite this capability, a fundamental problem remains: These autonomous agents return answers without referencing where they are extracted from, and act without pausing for user verification .Users must blindly trust every automated step and manually cross-check answers against a webpage—making it challenging to inspect, correct, or intervene (see Figure 1a). Commercial LLM assistants ( e.g., RealWebAssist [ 43 ], Browser-Use [ 7], ChatGPT Atlas [ 30 ], Dia Browser [ 36 ]) and browser-integrated agents ( e.g., Claude in Chrome [ 4], Gemini in Chrome [ 12 ], OpenAI Operator [ 31 ]) optimize for en-to-end web navigation but make no attempt to ground their outputs in the page or expose intermediate decisions to the user. Grounded reasoning approaches such as HoT [ 29 ] and ScreenAI [ 5] link model outputs to evidence spans, but do so within the model’s response text—the webpage itself is left unchanged, so users still have to locate the evidence themselves. Standard keyword search ( e.g., Ctrl+F) is limited to exact string matching and cannot answer questions or highlight supporting evidence in context. Content filtering tools such as ad blockers [15 ] rely on fixed CSS rules and cannot adapt to a user’s semantic intent. None of these approaches give users a way to verify answers, confirm actions, or semantically hide unwanted content—all on the page. Three specific user needs remain unaddressed. (1) Finding and verifying information on dense, cluttered pages is tedious: users must scan long pages manually to find where AI answers may be generated based on [ 46 ]. (2) Completing multi-step tasks with step-level oversight is risky without step-level oversight [8 , 19 , 21 , 24 , 27 , 37 , 41 , 44 ]: users often need to follow many steps (e.g., clicking, scrolling, navigating menus) to complete tasks such as changing a password. However, chatbot instructions are long and do not point to where each step should be performed on the page, making them hard to follow. (3) Staying focused amid distracting content is increasingly difficult. While 1 in 3 of the internet users already use ad blockers [ 15 ] and prior work shows distraction reduction improves task performance [ 13 , 23 ], users still lack control over what they want to hide. For example, users may want to temporarily hide political posts during election periods so they can focus on essential tasks and reduce exposure to distracting or emotionally charged content [ 25 , 33 ]. However, existing tools rely on fixed rules that cannot adapt to users’ specific goals or to novel webpage layouts [18]. We present PageGuide , a browser extension that addresses all three needs by grounding LLM answers directly in the HTML DOM ( e.g., a specific paragraph, button, or link on the page) via visual overlays. PageGuide offers three main features: (1) Find 

links model answers to specific HTML DOM elements via inline citations, highlighting supporting evidence in-place so users can instantly verify any claim without leaving the page (Figure 1b); (2) Guide shows step-by-step instructions to perform a task, one step at a time, with each target element highlighted on the page and the user confirming each step before proceeding—following the mixed-initiative copilot model [ 2, 17 , 34 , 41 ] (Figure 2); and (3) Hide interprets a natural-language hide request, scores HTML DOM elements against the user’s intent, and presents a review-able checklist before hiding the elements, each with a per-element justification (Figure 3). To evaluate PageGuide , we conduct a within-subject controlled user study ( 𝑁 =94 ) on websites (Section 3.7), where each participant completed 6 tasks (2 per mode) under both a control condition (standard browser tools) and an extension condition, with order-ing counterbalanced to mitigate learning effects. We measure task completion time, accuracy, behavioral signals (Ctrl+F usage, page visits), and post-task Likert ratings. Our study aims to answer three central research questions: 

RQ1: Does PageGuide improve task completion accuracy and suc-cess rate across all three features? 

RQ2: Does PageGuide reduce task completion time and interaction effort ( e.g., Ctrl+F usage, mouse clicks, page visits) across Find ,

Guide , and Hide tasks? 

RQ3: Do users perceive completing tasks with PageGuide as easier and less effortful compared to completing the same tasks without it? By systematically comparing between two conditions (control and extension), we aim to provide rigorous empirical evidence on whether grounding LLM outputs in the HTML DOM leads to measurable improvements in task efficiency, accuracy, and user experience across all three interaction features. In summary, this paper makes three contributions :(1) A unified Find–Guide–Hide framework that grounds LLM an-swers in the HTML DOM, prioritizing in-situ inspectability over opaque end-to-end automation. ## 2 Related Work 

Web agents are built to serve a purpose: helping users accomplish goals on the web more efficiently and accurately than they could alone . Yet the dominant design paradigm ( i.e., fully autonomous agents that plan, click, and return a final result) conflates com-pleting a task with serving the user . When a user needs to ver-ify a claim on a cluttered page, follow a multi-step task, or stay focused amid irrelevant content, end-to-end automation offers little help. Prior work can be understood through two orthog-onal design choices that determine how well a system actually serves users in these situations: (i) autonomy level , whether the system acts on behalf of the user or alongside them; and (ii) out-put grounding , whether the answer is returned without showing where it comes from on the page or linked to specific, visible ele-ments. Systems that score high on autonomy and low on grounding (most existing agents) maximize speed at the cost of verifiability. 

PageGuide inverts this trade-off: it acts as a copilot that grounds every output ( e.g., answers, guidance steps, and hiding decisions) directly in the HTML DOM, keeping the user in control at every stage. 

Web Agents and Browser Automation. LLM-based web agents have made rapid progress on benchmarks that measure end-to-end task success [ 11 , 16 , 20 , 47 ]. Early systems such as Auto-GPT [ 32 ] and BabyAGI [ 28 ] decompose high-level goals into sub-tasks, execute them via code or tool calls, and iterate until done. More re-cently, browser-integrated systems—OpenAI Operator [ 31 ], Claude in Chrome [ 3], Gemini in Chrome [ 12 ], Browser Use [ 7 ], Dia Browser [36 ], and ChatGPT Atlas [ 30 ]—embed LLM planners directly in the browser, enabling click, type, scroll, and navigate actions on pages. All of these systems sit at the high-autonomy, low-grounding end of the design space: they handle the task for the user and return a final result, but their intermediate decisions are opaque and they do not show where the output comes from on the page. , , Nguyen et al., Tin Nguyen, Thang T. Truong, Runtao Zhou, Trung Bui, Chirag Agarwal, and Anh Totti Nguyen 

This opacity carries real costs. Recent benchmarks show that au-tonomous web agents often make incorrect decisions in high-stakes situations—such as deleting data, making unintended purchases, or following deceptive “Download” buttons that expose personal information [ 8, 21 , 24 , 37 , 44 ]. A partial remedy is the copilot model named Cowpilot [ 41 ], where the agent proposes actions and the user is able to pause, reject, or take alternative action when needed. 

PageGuide extends this idea further by not only requesting confir-mation but grounding each proposed action in the specific HTML DOM element it will affect, hinting what will happen next, so users can verify intent before execution rather than trusting an opaque plan. 

Chain-of-Thought and Grounded Reasoning. Chain-of-Thought (CoT) prompting improves LLM accuracy by externalizing interme-diate reasoning steps [ 1, 26 , 35 , 39 ]. Highlighted Chain-of-Thought (HoT) [ 29 ] takes this further by embedding XML-style evidence tags in each reasoning step, linking conclusions to specific input spans so users can trace why an answer is produced. Vision-language work on web interfaces reinforces this direction: ScreenAI [ 5 ] and WebQuest [ 38 ] show that reliable web interaction requires aligning model outputs with screen elements and multimodal page evidence, not just fluent text. These approaches share a common limitation: evidence is sur-faced in the model’s output—a sidebar answer, a tagged span, a cited region—while the webpage in the browser stays exactly as it is. The user sees the model point to evidence but must scroll back to the page and locate it themselves. PageGuide closes this gap by moving evidence directly onto the page: instead of returning “the answer is supported by element 42,” it places a highlight on 

element 42 in the HTML DOM, making the evidence visible exactly where the user is already looking. This shifts grounding from a feature of the model’s text output into a live, interactive layer on the webpage itself—one that users can inspect, override, and act on without switching context. 

Content Moderation and Selective Filtering. Existing filtering tools operate at the rule level: Ad blockers [ 15 ] match URLs and CSS selectors, Purpose-Mode [ 23 ] simplifiers hide predefined element classes, and task-focused readers strip non-content regions [ 13 , 15 ,23 ]. These rules are static, require manual updates as page layouts change, and cannot respond to a user’s natural-language intent ( e.g., “hide posts that feel politically divisive” ). Research on personalized moderation confirms that users want more agency over what gets filtered and clearer explanations for why [ 18 ], yet current tools offer neither: they hide content silently, with no per-element justification and no way for the user to review or override individual decisions. 

PageGuide ’s Hide mode approaches this problem through the 

output grounding len: rather than applying a fixed rule, it interprets the user’s intent at query time, scores each HTML DOM element against that intent using an LLM, and surfaces a one-sentence hiding reason for every hidden element directly on the page. The user can inspect each decision, restore individual elements, and re-run filtering as new content loads in infinite-scroll feeds. This turns content hiding from an opaque background process into a transparent, user-steered interaction. “ How many

## 3 Method 3.1 System Overview 

PageGuide is a Manifest v3 Chrome extension that injects content scripts into webpages, giving it both read and write access to the HTML DOM in real time. As shown in Figure 4, the system follows a three-step pipeline: the Router classifies the user’s query and dispatches it to the appropriate handler; each handler reads the HTML DOM to ground its answers in the current page state; and the handler produces a final state by mutating the HTML DOM directly, highlighting evidence, surfacing step-by-step instructions, or hiding distracting elements. The three handlers differ in how they interact with the user after the initial HTML DOM read: 

• Find is single-pass: the agent identifies supporting evidence spans and renders inline highlights on the page in one shot. 

• Guide is iterative: the agent proposes one action at a time, the user confirms or skips it (the Feedback loop in Figure 4, and the agent re-reads the HTML DOM on the new page before generating the next step. 

• Hide is single-pass: the agent scores all HTML DOM ele-ments against the user’s hiding intent and applies PageGuide : Browser extension to assist users in navigating a webpage and locating information , , 

display:none to matched elements, attaching a one-sentence reason to each. Across all three features, every text output is coupled to a HTML DOM mutation: Find returns a text answer with clickable inline citations that scroll to and highlight the source element on the page; Guide outputs predicted actions alongside a visual highlight on the target element; and Hide hides matched elements in place with an attached reason. The page itself is always the primary output surface. To ensure that these cross-component interactions function correctly in a browser (e.g., MS Edge, Chrome), we main-tain a comprehensive end-to-end testing framework built on Play-wright, and integrate it with automated continuous integration (CI) via GitHub Actions. (see Appendix I).

Before the Router or any handler can reason about the page, PageGuide 

must convert the HTML DOM into a form the LLM can work with. Using a Set-of-Marks (SoM) [ 20 ] style approach, every visible, text-bearing, or interactive node is assigned a unique integer index, pro-ducing a structured element index D = {( 𝑖𝑑 𝑗 , text 𝑗 , tag 𝑗 , bbox 𝑗 )} 𝑚 𝑗 =1

that records each element’s content, tag type, and on-screen bound-ing box. This index is passed to the Router for intent classification and to whichever handler is invoked, enabling the LLM to refer to specific page elements by index rather than parsing raw HTML. For Guide , the HTML DOM is re-read after each user-confirmed action so that the next step reflects the current page state.

## 3.4 Find Mode: Grounded Information Retrieval 

Find addresses a core limitation of existing LLM-based web agents: answers that are fluent but unverifiable because there is no pointer on the screen for users to follow or locate information. Given query 

𝑞 and element index D, Find prompts the LLM (see the full prompt in Figure 14 in Appendix) to produce a natural language answer in which every factual claim is annotated with an inline citation 

[N:"exact phrase"] , where N is the SoM index of the supporting HTML DOM element and exact phrase is the precise text span to highlight. For example, a Find answer for “Who directed this movie?” might read: “The movie was directed by [45:"Nolan"] and produced by [12:"Emma"] .” 

PageGuide then resolves each citation: it locates the HTML DOM node at index N, wraps the matching text span in a color-coded animated overlay (colors are drawn from a luminance-aware palette so highlights remain distinguishable against any page background), and scrolls the viewport to the first cited element. The final state visible to the user has two coupled components (see Figure 1b): 

• In-page highlights: each cited span is highlighted directly on the page; clicking a highlight re-scrolls and re-pulses the element. 

• Side panel answer: the full answer with all citations ren-dered as clickable anchors is shown in a collapsible panel alongside the page. This keeps evidence in situ : the user verifies a claim by glancing at the highlighted element on the same page rather than switching context. Find Prompt (condensed)

## 3.6 Hide Mode: Transparent Content Hiding 

Hide enables users to hide irrelevant or distracting content using a natural language description ( e.g., “hide all sponsored posts” ; see Figure 3). Unlike static ad-blockers that match fixed CSS selectors or URL patterns, Hide uses LLM to identify elements that the users want to hide, adapting to novel page layouts and semantically-specified criteria. Given query 𝑞 and index D, the LLM (see the full prompt in Figure 15 in the Appendix) returns a ranked list 𝐻 ⊆ D of elements to hide, each paired with a one-sentence justification and a short content snippet. Before any element is hidden, a floating summary pop-up is shown to the user. Each row in the dialog displays a numbered badge, the hiding reason in purple, the content snippet, and a a jump button ( 

> ↑

) that scrolls the page to that element so the user can inspect it in context. Every row is checked by default; the user can uncheck any item to exclude it from hiding. Only after the user clicks Confirm does the agent apply display:none to the selected elements. This confirm-before-hide design is a deliberate transparency choice: the user reviews and approves every hiding decision be-fore it takes effect, consistent with mixed-initiative principles that preserve meaningful user control [2]. ## 3.7 Data Collection 

To evaluate how well PageGuide assists users in finding correct information, guiding them through realistic web tasks, and hiding specific webpage content, we construct the PageGuide dataset, which consists of three subsets: Find , Guide , and Hide . For details on how each dataset is constructed, please refer to Appendix F. 

Find Subset. For Find subset, we randomly sample 5 questions from Google Natural Questions [ 22 ], which provides questions and answers grounded in webpage content. However, this source alone is not sufficient for the user study, as many examples are centered on Wikipedia-style pages with relatively short content, making the answers easy to find. To better evaluate the efficiency of the 

Find feature in diverse, long, and content-heavy webpage settings, we augment the task set with 5 additional FIND tasks from other websites. 

Guide Subset. The goal of the Guide dataset is to examine how well PageGuide assists users in completing web-based tasks through step-by-step guidance. This requires tasks that reflect real user needs and require a certain degree of human supervision (refer to Figure 2). Existing datasets, such as Mind2Web [ 11 ], Online-Mind2Web [ 42 ], and WebArena [ 47 ], are primarily designed to evaluate autonomous web agents rather than to examine how an agent assists a human user in completing a task . As a result, some tasks are less appropriate for a user study since they are not well aligned with assistive guidance. We therefore construct a Guide subset consisting of 138 tasks spanning 37 websites. From this larger task pool, we select 7 representative tasks for the user study. 

Hide Subset. The objective of the Hide dataset is to study whether 

PageGuide can hide specific webpage elements in response to a user’s request. This requires tasks that pair a hide request with the exact webpage elements that should be concealed. To the best of our knowledge, existing benchmarks do not capture this specific problem. We therefore introduce a dedicated Hide subset consisting of 108 tasks, including 67 easy, 22 medium, and 19 hard instances. We define task difficulty based on the number of target item types specified in the user request: easy tasks involve hiding a single type of target ( e.g., “Hide all advertisements on the page” ), medium tasks involve two types ( e.g., “Hide all posts that relate to Microsoft or Amazon” ), and hard tasks involve three or more ( e.g., “Hide all food items that contain pineapple, pepper and ginger” ). We then select 10 Hide tasks for the user study. We also evaluate the agent’s per-formance on the full Hide dataset to provide broader quantitative context for our user-study results (see Appendix D).

RQ1) Step-by-Step Guidance Increases Task Completion Ac-curacy For Find tasks, participant accuracy increases from 81% in the control condition to 86% (+5 pp) with PageGuide as shown in Figure 5. ## 5 Limitations and Future Work 

Single-mode dispatch limits composite tasks. The current in-tent router maps each query to exactly one mode, which works well for clearly scoped requests but fails on composite queries that span multiple features, e.g., “find the settings page and guide me through changing my password” requires both a Find and a Guide 

step. Future work will replace the single-mode router with a multi-step planner that decomposes composite queries into an ordered sequence of mode invocations ( e.g., Find → Guide ), handling tasks that currently require users to issue multiple separate queries. 

Evidence highlights lack cross-page persistence. Find ’s in-page highlights occasionally overwhelmed participants on dense pages with many matching spans. Additionally, the current imple-mentation highlights only within a single page; users who navigate across multiple pages to gather information must re-issue the query each time. Future work will introduce a re-ranking step that surfaces the most relevant evidence first, and explore multi-page context :persisting and aggregating highlights across page visits so users can build up a cross-page evidence trail within a single session. 

Per-step confirmation in GUIDE may introduce overhead When task time is measured only over correctly completed tasks, PageGuide 

is consistently faster than the control condition in the GUIDE tasks (Figure 6). The remaining limitation is that the per-step confirma-tion model adds overhead for users who already know the pro-cedure. Future work will explore adaptive step granularity that compresses self-evident actions into a single confirmation, and multi-turn correction ( e.g., “skip this step” , “undo the last action” ) so experienced users can move at their own pace. 

Content hiding has no memory across sessions. Hide ’s confir-mation dialog added review overhead that some participants found cumbersome for long lists of matched elements. More broadly, users currently have no memory across sessions: content they routinely hide must be re-specified each time. Future work will introduce a user preference history that persists hiding decisions across sessions—learning what a user dislikes over time and applying those preferences automatically on revisited pages, with the ability to review and edit the saved profile. 

## 6 Conclusion 

We presented PageGuide , a browser extension that grounds LLM answers directly in the HTML DOM across three interaction fea-tures: Find locates and highlights supporting evidence in-place so users can instantly verify answers; Guide delivers step-by-step instructions one action at a time with user confirmation at each step; and Hide hides distracting content with a per-element justifi-cation shown in a pre-action review dialog. In all three features, the agent’s output is embedded in the page itself rather than returned as plain text in a sidebar. A controlled user study ( 𝑁 =94 ) against unaided browsing showed consistent gains across all features. Find reduced task time by 15.2% and Ctrl+F usage from 24.1% to 4.4% of tasks, with page visits drop-ping by 44.3%. Guide improved navigation completion by 7.7% while keeping users in full control of every action. Hide reduced task time by 50.8% and improved content hiding accuracy by 25.7%, enabling users to identify targeted elements they would otherwise overlook. These results support the core thesis of this work: grounded, in-situ interaction—where every answer, step, and hiding decision is anchored to a specific HTML DOM element—is more effective than opaque autonomy for tasks where users need to understand why an answer is correct, how a procedure should unfold, or what has been PageGuide : Browser extension to assist users in navigating a webpage and locating information , , 

changed on a page. We hope PageGuide serves as a step toward web agents that augment rather than replace human judgment—tools that earn trust by showing their work.

## D Model Performance D.1 Intent Router Reliably Classifies User Intent 

The intent router is the first and most critical component in PageGuide (in Section 3): a misrouted query sends the user’s request to the wrong execution mode, causing failures regardless of how well Find , Guide , or Hide perform individually. Before evaluating each mode in isolation, we therefore need to establish that the router operates with high reliability across all three task categories. 

Experiment We constructed a labeled evaluation set of 1,208 natural language queries covering three routing classes: find (finding information, 969 queries), guide (step-by-step guidance, 132 queries), and hide (hiding content, 107 queries). Queries for the find class were drawn from the Natural Questions and QASPER datasets [9, 22], guide queries from the Online-Mind2Web benchmark [11], and hide 

queries from a custom dataset collected across 100 diverse websites (see Section 3.7). We report per-class accuracy and error type to identify systematic failure modes. 

Results The intent router achieves a highly reliable 97.68% overall accuracy, confirming that it provides a robust gateway for downstream execution (Table 1). It correctly classifies all guide queries (100%) and maintains high accuracy across both the find (97.4%) and hide (97.2%) classes. The 25 errors in the find class are entirely misrouted to guide , a misclassification that can arise from ambiguous phrasing ( e.g., “find me a...” ) where information retrieval overlaps with navigation intent. Lastly, the hide class errors consist of three queries misclassified as 

find , which typically occurs when an ambiguous query that do not convey the hide intent clearly ( e.g., “I only want courses that have higher than 4.6 star.” ).

## D.2 Gemini-3-Flash is Prefered for the Find task 

To quantify how accurately Find identifies and highlights supporting evidence, we evaluate it on two question-answering datasets that explicitly require grounding: Natural Questions (NQ) [ 22 ], where answers are anchored to specific spans within Wikipedia articles, and QASPER [ 9 ], where evidence is grounded within lengthy academic PDFs. Both datasets provide human-annotated evidence spans, making them a natural fit for assessing retrieval precision alongside answer quality. 

Experiment We evaluated Find on two datasets. First, on the NQ development set, we rendered each Wikipedia article as a webpage within the extension environment, asking the system to highlight the supporting span(s) and generate an answer to a factoid question. We report four metrics for NQ: Precision (fraction of highlighted spans that are correct), Accuracy (fraction of correct spans that are highlighted), F1 

(harmonic mean of precision and accuracy over span overlap), and Answer Correctness (exact-match accuracy of the final answer string). Second, we evaluated Find on the full QASPER dataset (1,189 samples), a scientific question-answering benchmark built on NLP papers. Each question is answered using evidence extracted from the corresponding paper. For QASPER, we report Answer F1 (token-level overlap between predicted and gold answers) and Evidence F1 (token-level overlap between predicted and gold evidence spans). Across both datasets, we compare two backbone models: and . For QASPER, we additionally compare against LED-base [ 6] as a retrieval-specialized baseline.

We construct the dataset through three separate pipelines for Find , Guide and Hide subsets. (refer to Figure 17). 

Find Subset To construct the Find subset, we collect webpages on the internet and examples from existing resources, such as Google Natural Questions Dataset[ 22 ]. We then filter for long, content-dense pages in which the answer is present but not immediately obvious, requiring users to locate specific information within substantial page content. From these pages, we extract requests and manually validate the resulting examples before exporting the final dataset. In total, the Find subset contains 10 questions (see Figure 17A). 

Guide Subset The construction of Guide subset follows a four stage pipeline: collect webpages, select official documentations, extract guide requests and export dataset. (refer to Figure 17b). Firstly, we select popular websites based on global web traffic rankings, using the list of most-visited websites compiled from Similarweb and Semrush [ 40 ]. We then collect official documentation, help-center articles, and tutorial pages associated with these websites on different common user-facing tasks ( e.g., changing a setting, applying a filter, or navigating to a feature.) Next, we extract requests from the help pages and, in some cases, rewrite these procedures into an actual user requests. Finally, we perform manual quality control to remove outdated, ambiguous, or duplicate tasks. The resulting Guide examples each contain a natural-language request, the reference length and the corresponding website link. 

Hide Subset For Hide subset, we first collect a diverse set of commonly used webpages across different domains, such as social media, forums, shopping sites, and content platforms. For each page, we save a rendered HTML snapshot so that the page structure and visible elements can be inspected during annotation. (refer to Figure 17c) We then build an annotation tool that allows annotators to examine each webpage and hide requests based on the specific content shown on that page. For each request, annotators identify the set of page elements that should be hidden and ground the task to the corresponding page content. After annotation, we manually review the tasks to remove trivial, unclear, or duplicate examples and to ensure that each request is feasible and supported by the visible page. The final Hide examples therefore each include a webpage, its crawled HTML, a hide request, and the grounded target elements to be removed. , , Nguyen et al., Tin Nguyen, Thang T. ## G Beyond Basic Functionality: Enhancing Web Extensions with PDF Reading, Visual Question Answering, and Page-Off Support 

PageGuide extends beyond the core Find , Hide , and Guide functionalities with three additional features: PDF reading, visual question answering, and Page-Off support. These features are motivated by an important limitation of conventional web extensions: user needs are often not confined to the visible content of the current webpage. In practice, users frequently interact with documents such as PDFs, compare webpage content with uploaded images, or ask questions that go beyond the active tab. Supporting these settings allows PageGuide 

to provide a more complete and flexible browsing experience, rather than restricting assistance to page-bound interactions alone. 

PDF Reading Many webpages contain or link to information in PDF form, such as papers, reports, manuals, and forms. However, this content is often difficult to browse efficiently using standard web tools. The PDF reading feature allows users to upload a PDF and ask questions about its content, enabling the agent to support document-centered information seeking within the same interaction framework as webpage assistance (Figure 18).

## H Limitations of Existing Extensions 

Existing browser-integrated AI systems—including Gemini Chatbot (Figure 21), Browser Use (Figure 22), MolmoWeb (Figure 23), and Gemini Agent (Figure 24)—share a common limitation: their outputs are not grounded in visible page evidence. ## I End-to-End Testing Framework 

PageGuide employs Playwright as its end-to-end (E2E) testing framework, integrated into an automated continuous integration (CI) pipeline using GitHub Actions. This pipeline is triggered on each push and pull request to the main branch, invoking npm run ci to execute manifest validation, unit tests, and the full Playwright browser suite in sequence. The rationale for E2E testing stems from the inherent limitations of unit testing in the context of browser extension development: numerous failure modes appear only when code executes within a real Chromium environment, where it must interact with native browser APIs, extension-specific permissions, and live page contexts. Accordingly, E2E tests are designed to detect integration-level defects that unit tests cannot surface, including extension initialization failures, content-script injection errors, UI regressions, and cross-component interactions among the side panel, options page, and background service worker. The Playwright configuration instantiates Chromium with a persistent browser context, enabling the unpacked extension to be loaded via the –load-extension and –disable-extensions-except launch flags. Test coverage encompasses extension loading and initialization, service worker and background script startup, side panel rendering and user interaction, options page functionality, and PDF viewer integration. Additional edge cases are also tested, including pages with minimal content, extended scrollable layouts, and image-dense documents. Collectively, this framework provides empirical validation that PageGuide operates correctly as a fully integrated browser extension, rather than relying solely on the correctness of individual, isolated modules.
