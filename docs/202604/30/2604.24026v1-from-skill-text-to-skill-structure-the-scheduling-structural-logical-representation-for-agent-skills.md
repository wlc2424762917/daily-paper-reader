## 摘要
大语言模型（LLM）智能体日益依赖可重用的技能，即结合了指令、控制流、约束和工具调用的能力包。然而，在大多数当前的智能体系统中，技能仍由文本密集的工件表示，包括 SKILL.md 风格的文档和结构化记录，其机器可用的证据在很大程度上仍嵌入在自然语言描述中。这给以技能为中心的智能体系统带来了挑战：管理技能集合以及利用技能支持智能体，都需要对调用接口、执行结构和具体的副作用进行推理，而这些内容往往纠缠在单一的文本表层中。因此，技能知识的显式表示可能有助于使这些工件更易于机器获取和利用。借鉴 Schank 和 Abelson 在语言知识表示方面的经典著作中的记忆组织包（Memory Organization Packets）、脚本理论（Script Theory）和概念依赖（Conceptual Dependency），我们引入了据我们所知的首个针对智能体技能工件的结构化表示，它解耦了技能级调度信号、场景级执行结构以及逻辑级动作和资源使用证据：即调度-结构-逻辑（Scheduling-Structural-Logical, SSL）表示。我们使用基于 LLM 的归一化器实例化了 SSL，并在两个任务（技能发现和风险评估）的技能语料库上对其进行了评估，其表现显著优于纯文本基准：在技能发现中，SSL 将 MRR 从 0.573 提高到 0.707；在风险评估中，它将宏观 F1 从 0.744 提高到 0.787。这些发现表明，显式的、基于源的结构使智能体技能更易于搜索和审查。研究还表明，SSL 应被视为向更具可检查性、可重用性和操作可行性的智能体系统技能表示迈出的务实一步，而非一个成熟的标准或管理与使用技能的端到端机制。

## Abstract
LLM agents increasingly rely on reusable skills, capability packages that combine instructions, control flow, constraints, and tool calls. In most current agent systems, however, skills are still represented by text-heavy artifacts, including SKILL.md-style documents and structured records whose machine-usable evidence remains embedded largely in natural-language descriptions. This poses a challenge for skill-centered agent systems: managing skill collections and using skills to support agent both require reasoning over invocation interfaces, execution structure, and concrete side effects that are often entangled in a single textual surface. An explicit representation of skill knowledge may therefore help make these artifacts easier for machines to acquire and leverage. Drawing on Memory Organization Packets, Script Theory, and Conceptual Dependency from Schank and Abelson's classical work on linguistic knowledge representation, we introduce what is, to our knowledge, the first structured representation for agent skill artifacts that disentangles skill-level scheduling signals, scene-level execution structure, and logic-level action and resource-use evidence: the Scheduling-Structural-Logical (SSL) representation. We instantiate SSL with an LLM-based normalizer and evaluate it on a corpus of skills in two tasks, Skill Discovery and Risk Assessment, and superiorly outperform the text-only baselines: in Skill Discovery, SSL improves MRR from 0.573 to 0.707; in Risk Assessment, it improves macro F1 from 0.744 to 0.787. These findings reveal that explicit, source-grounded structure makes agent skills easier to search and review. They also suggest that SSL is best understood as a practical step toward more inspectable, reusable, and operationally actionable skill representations for agent systems, rather than as a finished standard or an end-to-end mechanism for managing and using skills.

### 论文总结：从技能文本到技能结构（SSL 表示法）

#### 1. 论文的核心问题与整体含义
随着大语言模型（LLM）智能体的发展，其能力逐渐被封装为“技能”（Skills）。然而，目前的技能表示主要依赖于非结构化的文本（如 `SKILL.md` 文档），这导致了**表示瓶颈**：
*   **机器难处理**：机器难以直接从冗长、嘈杂的文本中推理出技能的调用接口、执行逻辑和潜在副作用。
*   **信息纠缠**：调用接口（何时用）、执行阶段（怎么做）和底层资源操作（动了什么）混合在一起，不利于技能的自动化管理、检索和安全审查。

本研究旨在提出一种结构化的中间表示，将技能文本转化为机器可理解、可推理的图结构。

#### 2. 方法论：SSL 表示法
论文借鉴了 Schank 和 Abelson 的经典语言知识表示理论，提出了 **SSL（Scheduling-Structural-Logical）** 三层表示架构：

*   **核心思想**：
    1.  **调度层 (Scheduling Layer)**：借鉴“记忆组织包 (MOPs)”，定义技能的外部接口。包括技能目标、输入/输出契约、标签、意图签名及粗粒度的控制流特征。
    2.  **结构层 (Structural Layer)**：借鉴“脚本理论 (Script Theory)”，将执行过程划分为有序的“场景”（Scenes），如准备、获取、推理、行动、验证等，并定义场景间的转换规则。
    3.  **逻辑层 (Logical Layer)**：借鉴“概念依赖 (Conceptual Dependency)”，记录原子级的动作（如 READ, WRITE, CALL_TOOL）和资源边界（如内存、文件系统、网络、凭据），用于追踪数据流和操作证据。

*   **技术实现（归一化器）**：
    使用基于 LLM 的归一化器（Normalizer），通过四阶段提示词流程（提取记录 -> 分解场景 -> 扩展逻辑步 -> 验证对齐），将原始文本转换为符合 JSON Schema 的三层关联图。

#### 6. 主要结论与发现
*   **检索性能提升**：在技能发现中，SSL-Rich 表示法将 MRR 从 0.573 提升至 **0.707**。这表明结构化的接口和工作流信号比原始文本更易于匹配。

*   **互补性**：SSL 并非要取代原始文档，而是作为其“证据接口”。实验显示“文档 + SSL”的组合通常能获得最佳效果。

#### 7. 优点
*   **理论支撑深厚**：将 20 世纪 70 年代的经典语言表示理论应用于现代 LLM 智能体领域，逻辑自洽。
*   **解耦清晰**：成功将复杂的技能文档拆解为调度、结构、逻辑三个维度，为后续的自动化工具链（如静态分析、策略检查）奠定了基础。
*   **实用性强**：不仅提出了表示法，还提供了配套的归一化工具和高质量的标注数据集。

#### 8. 不足与局限
*   **静态分析限制**：SSL 是从静态文档中提取的，无法捕捉动态生成的代码或运行时下载的负载所产生的行为。
*   **归一化偏差**：依赖 LLM 进行提取，可能存在信息遗漏、过度正则化或对模糊描述的误读。
*   **代码语义缺失**：如案例 3 所示，如果风险源于技能生成的代码（而非技能本身的动作），SSL 可能会低估其风险。
*   **应用范围**：目前仅在管理和审查层面验证，尚未直接测试 SSL 如何在智能体实时规划和执行中发挥作用。
