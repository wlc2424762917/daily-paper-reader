## Abstract 

Deep learning compilers and vendor libraries deliver strong baseline performance but are bounded by finite, engineer-curated catalogs. When these omit needed optimizations, practitioners substitute hand-written CUDA or CUTLASS, demanding expertise in GPU microarchitecture and C++ template metaprogramming. Recent LLM-based agents target kernel generation in raw CUDA, forcing rediscovery of optimizations already encoded in mature libraries. We present FACT (Framework for Agentic CUTLASS Transpila-tion), a framework that employs a three-stage, agent-driven work-flow optimizing PyTorch modules through multi-pattern compo-sition while grounding synthesis in CUTLASS C++. (1) Pattern discovery: an LLM agent inspects the traced graph, matches sub-graphs to optimization rules, retrieves vetted examples from an architecture-specific index, and outputs prioritized patterns. (2) Pattern realization: each pattern is implemented as a CUTLASS kernel wrapped in a PyTorch extension, verified, and auto-tuned by sweeping parameters inferred from the CUTLASS hierarchy. (3) Pattern composition: extensions are loaded together into a single composed module for end-to-end benchmarking. We evaluate the workflow using KernelBench’s evaluation frame-work and provided modules on an NVIDIA A100. On Level 1, we ap-ply the workflow to three GEMM workloads (square matrix multiply, batched matrix multiply, and large-𝐾 matrix multiply). Auto-tuned CUTLASS kernels improve over PyTorch cuBLAS baseline by 1.06 ×–

1.18 ×. On Level 3 MiniGPT block, composing fused multi-head attention with fused MLP GEMM+GELU yields 2.79 × end-to-end speedup. Our work couples agentic graph-level pattern discovery with auto-tuning and a dynamic pattern table, offering a practical path from traced PyTorch to deployable kernels by automating CUTLASS kernel synthesis and auto-tuning.

## 1 Introduction 

Deep learning compilers such as TVM [ 2], Ansor [ 14 ], and Py-Torch Inductor [ 1] have made significant strides in automating the compilation of tensor programs. However, their ability to generate high-performance GPU kernels is fundamentally constrained by the optimization patterns that their developers have codified. When a new model architecture or a new GPU generation is released, de-veloping optimized kernels that exploit its novel hardware features requires significant engineering effort, and integrating these opti-mizations into compiler backends adds further delay before they become available to practitioners. Even when optimized kernels are eventually integrated into a compiler—as CUrator [ 6 ] does by in-corporating CUTLASS [ 10 ] into TVM’s Bring-Your-Own-Codegen (BYOC) pipeline—the compiler relies on a fixed, developer-specified pattern registry for graph node matching and fusion, which can-not discover or add new optimization patterns without manual intervention. The alternative to waiting for compiler support is to hand-optimize kernels using low-level languages such as CUDA or vendor-specific template libraries such as CUTLASS. Expert GPU programmers can write custom kernels that exploit architecture-specific features and integrate them into a PyTorch computation graph via custom exten-sions. This approach routinely achieves performance competitive with or exceeding compiler output, but it demands deep expertise in GPU microarchitecture and months of engineering effort per kernel. Over the past two years, large language model (LLM)-based agents have demonstrated remarkable capabilities in code genera-tion, program synthesis, and multi-step reasoning [ 3 , 8, 12 ]. These capabilities have been applied directly to GPU kernel optimiza-tion: CUDA Agent [ 3] scales agentic reinforcement learning to generate high-performance CUDA kernels; KernelBlaster [ 4] uses memory-augmented in-context RL to transfer optimization knowl-edge across tasks; Astra [ 13 ] employs a multi-agent workflow to iteratively transform existing kernels using profiling-driven feed-back; and StitchCUDA [ 8 ] targets end-to-end GPU programs with a planner/coder/verifier decomposition.

However, these approaches share a common limitation: they syn-thesize kernels in raw CUDA, which forces them to rediscover—in generated source code—optimizations that mature libraries have already encapsulated as vetted, architecture-parameterized tem-plates. Despite the promise of agentic kernel optimization, we are not aware of any prior framework that combines library-grounded synthesis with a dynamic pattern registry—addressing both the static registry limitation of compiler-based approaches and the raw-code generation focus of prior agentic systems. We propose FACT, a framework that combines multi-pattern composition with library-grounded synthesis through a dynamic pattern registry that grows as the agent discovers new patterns— where each pattern corresponds to a subgraph identifiable by an optimization rule (e.g., GEMM, FMHA), data type, and target GPU architecture—unlike static registries, this enables accumulation across workloads. FACT performs architecture-specific synthesis by matching patterns to the target GPU’s capabilities. It also enables architecture-specific auto-tuning where the agent automatically infers which parameters to sweep at each level of CUTLASS’s hier-archical API. FACT realizes this through a three-stage agentic workflow. (1) Pat-tern Discovery: an LLM agent analyzes the PyTorch computation graph and proposes optimization patterns by retrieving vetted ex-amples from an architecture-specific kernel repository. (2) Pattern Realization: each pattern is implemented as a CUTLASS kernel, ver-ified for correctness, auto-tuned, and added to the dynamic pattern registry. (3) Pattern Composition: realized kernels are composed with the original computation graph for end-to-end benchmarking. Both synthesis and auto-tuning navigate CUTLASS’s three-level hierarchy. At the tile level, synthesis configures Tensor Core prim-itives. At the kernel level, synthesis selects pipelining strategies appropriate for the target architecture. At the grid level, synthesis chooses scheduling policies that balance load across thread blocks. These levels must be addressed coherently—incompatible choices across levels cause failures. Auto-tuning sweeps parameters at each level (tile shapes, pipeline stages, split-K factors) to find optimal configurations. The architecture-specific kernel repository grounds both processes in vetted instantiations. The contributions of this paper are as follows: 

• Library-grounded agentic kernel synthesis with a dy-namic pattern registry. We ground the agent’s synthesis in CUTLASS C++ templates rather than raw CUDA, enabling reuse of architecture-parameterized patterns that accumu-late across workloads. 

• Architecture-specific auto-tuning where the agent in-fers relevant parameters based on the target architec-ture and kernel type. For Ampere multistage pipelines, the agent sweeps tile shapes and pipeline stages; for Hopper warp-specialized kernels, it sweeps kernel schedules and cluster shapes. The search space is derived from CUTLASS’s architecture-specific instantiations. 

• KernelBench-style evaluation on representative prob-lems spanning CUTLASS’s capabilities. We demonstrate the workflow on GEMM variants exercising different sched-uling policies (Data-Parallel, Batched, Stream-K) and on a multi-pattern transformer block, achieving speedups of 1.06 ×–2.79 ×.

## 2 Background & Motivation 2.1 KernelBench Evaluation Framework 

KernelBench [ 12 ] introduces a standardized benchmark for evaluat-ing large language model (LLM) agents on the task of generating op-timized GPU kernels. The framework adopts a two-stage evaluation methodology centered on correctness verification and performance measurement . In the correctness stage, kernels synthesized by the agent are compiled and executed against a reference PyTorch imple-mentation; the output tensors are compared element-wise to ensure functional equivalence. This stage verifies that the agent-generated kernel preserves the semantics of the original computation graph. In the performance stage, correctly executing kernels are bench-marked using CUDA events to measure end-to-end execution time on target GPU hardware. KernelBench reports speedup relative to a baseline—typically the standard PyTorch eager-mode execu-tion or a vendor library implementation—thereby quantifying the optimization capability of the agent. A key contribution of KernelBench is its hierarchical task for-mulation across three difficulty tiers. Level 1 tasks involve single linear algebra operators (e.g., matrix multiplication, transpose) that directly map to established library patterns. Level 2 tasks introduce fused operators that combine two or more primitive operations, requiring the agent to discover and implement kernel fusion op-timizations. Level 3 tasks present complex computation blocks from real deep learning models, challenging the agent to navigate multi-operator patterns and interdependent scheduling decisions. This progressive difficulty structure enables granular assessment of agent capabilities across different levels of optimization complexity. Crucially, KernelBench defines the optimization problem as tran-spiling a baseline PyTorch module into a functionally equivalent kernel implementation. The input to the agent is a PyTorch com-putation graph extracted from the target model. The agent’s ob-jective is to synthesize optimized code that, when compiled and loaded as a PyTorch extension, replaces the baseline subgraph in the model’s execution trace. This formulation closely mirrors real-world workflows in which practitioners identify performance-critical subgraphs through profiling and replace them with cus-tom kernels. By standardizing both the input format (computation graphs) and the evaluation protocol (correctness checking followed by performance benchmarking), KernelBench provides a repro-ducible benchmark for comparing kernel generation agents across diverse hardware architectures and operator patterns. 

## 3 Task Definition 

We adopt a task formulation aligned with KernelBench’s transpi-lation paradigm, with two key extensions. First, we target multi-pattern composition rather than isolated operator synthesis: the agent operates over the complete computation graph of a baseline PyTorch model and produces a transformed version that reflects kernel-level optimizations applied at multiple subgraphs. Second, we introduce a dynamic pattern table that serves as a persistent optimization catalog, enabling discovered patterns to be reused across models and accumulated over time. Formally, let G = (𝑉 , 𝐸 ) denote the computation graph of a baseline PyTorch model, where each node 𝑣 ∈ 𝑉 represents a ten-sor operation and each edge 𝑒 ∈ 𝐸 represents a data dependency. The agent’s objective is to discover a set of optimization opportu-nities P = {𝑝 1, 𝑝 2, . . . , 𝑝 𝑛 }, where each pattern 𝑝 𝑖 corresponds to a subgraph G𝑖 ⊆ G that can be replaced by an optimized kernel implementation. Each pattern 𝑝 𝑖 is associated with a rule —such as GEMM, FMHA (Fused Multi-Head Attention), or epilogue fusion— that determines the structural template of the kernel synthesis problem. The pattern table T serves as the central abstraction for orga-nizing and retrieving kernel implementations. Each entry in T

maps a tuple (𝑟, 𝜏, 𝛼 ) to a concrete CUTLASS kernel implementa-tion, where 𝑟 denotes the optimization rule, 𝜏 denotes the data type configuration (e.g., FP32, BF16, FP8), and 𝛼 denotes the target GPU architecture (e.g., Ampere, Hopper). This multi-dimensional index-ing enables the agent to select kernel instantiations that match the computational pattern, data type constraints, and hardware con-text. Importantly, the table is dynamic : when the agent discovers a new pattern and synthesizes a correct kernel implementation, the pattern and its associated kernel are added to T , expanding the optimization catalog for future queries. Figure 1 illustrates the pattern mapping hierarchy. At the top level, the agent identifies patterns from the computation graph, each corresponding to a well-defined optimization rule. For each rule, the pattern table maintains multiple CUTLASS kernel implementations specialized across data types, input shapes, and GPU architectures. This design enables efficient retrieval: given a target subgraph, data type constraints, and hardware platform, the agent can query 

T to retrieve candidate kernels, then benchmark them to select the highest-performing variant. Unlike static pattern libraries in which all possible specializations must be known ahead of time, our dynamic table allows the optimization catalog to grow as the agent encounters new patterns and architectures. 

## 4 Proposed Approach 

Figure 2 illustrates our three-stage agentic workflow for whole-model kernel optimization. The workflow maps directly to CUT-LASS’s API hierarchy [ 10 ] and leverages LLM-based actions to orchestrate complex optimization tasks that would otherwise re-quire manual intervention. Each stage decomposes into a sequence of well-defined actions—graph extraction and analysis, kernel syn-thesis and verification, composition and extension loading—that the agent executes systematically. By structuring optimization as a pipeline of discrete actions, we enable the agent to reason about each decision point independently while maintaining coherence across the entire workflow.

## 4.2 Pattern Realization 

The second stage synthesizes concrete CUTLASS kernels for each prioritized pattern. For each pattern 𝑝 𝑖 ∈ P , the agent executes an iterative workflow with six actions, including a feedback loop for correctness verification and an optional auto-tuning step. 

Action 1: Select supporting examples. The agent queries the examples index to retrieve CUTLASS kernel implementations that serve as references for synthesizing the target pattern. These supporting examples are selected based on similarity to the target pattern in terms of optimization rule, data type, input shape, and GPU architecture. The agent may retrieve multiple examples that, when combined, provide the necessary components to realize the pattern. 

Action 2: Synthesize CUTLASS + PyTorch extension. Guided by the supporting examples, the agent synthesizes a CUTLASS kernel implementation. This synthesis operates at CUTLASS’s three API levels [ 10 ], organized from innermost to outermost as shown in Figure 3. From innermost to outermost: (1) Tile-level primitives configure Tensor Core operations; (2) Kernel-level pipelining determines how memory-copy and compute overlap—warp-specialized designs on Hopper versus multistage pipelines on Ampere; (3) Grid-level scheduling controls work distribution across thread blocks. Each level presents distinct configura-tion decisions that the agent must resolve correctly for the target architecture and workload. 

At the tile level (innermost circle), the agent configures matrix multiply primitives for Tensor Core efficiency. This involves select-ing the appropriate instruction shape, operand datatypes, swizzling mode, and memory layout (MN-major vs K-major). At the kernel level (middle circle), the agent selects a software pipelining strategy appropriate for the target architecture. For Hop-per GPUs, warp-specialized designs dedicate producer warps to TMA loads and consumer warps to WGMMA compute, leverag-ing warpgroup register reallocation. For Ampere GPUs, multistage pipelines use cp.async to prefetch data while computing on the current buffer, with all warps taking dual roles. At the grid level (outermost circle), the agent configures work partitioning across thread blocks to ensure load balance and high device utilization. CUTLASS supports Data Parallel (standard 2D tiling), Split-K (partitioning along K dimension with turnstile re-duction), and Stream-K (persistent CTAs processing fractional tiles to eliminate wave quantization). The synthesized kernel is wrapped as a PyTorch extension, com-piled into a shared library (.so file), and registered as a custom operation. 

Action 3: Create per-pattern ModelNew(nn.Module). The agent constructs a new PyTorch module, ModelNew , that integrates the synthesized CUTLASS extension. This module inherits from 

nn.Module and implements the forward method to invoke the cus-tom CUTLASS operation. The module preserves the input/output interface of the original subgraph, enabling drop-in replacement within the computation graph. 

Action 4: Verification and benchmarking. The agent exe-cutes ModelNew against a reference implementation using identical test inputs. For correctness verification, output tensors are com-pared element-wise to ensure functional equivalence. For perfor-mance benchmarking, the kernel is executed multiple times on the target GPU hardware, and CUDA events measure end-to-end exe-cution time. The resulting speedup relative to the baseline PyTorch implementation is recorded. At this point, the agent evaluates whether the pattern is accepted .If the pattern passes both correctness verification and achieves satisfactory performance, the workflow proceeds to Action 5. If the pattern fails verification or performs poorly, the workflow loops back to Action 1: the agent re-selects supporting examples, inspects the synthesized code to identify correctness issues, and modifies the implementation to fix the problems. This iterative refinement continues until the pattern is accepted. 

Action 5: Auto-tune kernel. After a pattern passes correctness verification, the agent may perform auto-tuning to improve per-formance beyond the initial configuration. The auto-tuning search space is not hardcoded —it is inferred from the three CUTLASS API levels chosen during Action 2. At the tile level, the agent reads the CUTLASS test files for the target architecture and data type to extract all valid threadblock and warp tile shapes used in existing instantiations, forming the set of tile candidates to sweep. At the kernel level, the swept parameter depends on the pipelining strat-egy: for Ampere multistage pipelines the agent sweeps the number of pipeline stages (typically 2–3, constrained by shared-memory ca-pacity), whereas for Hopper warp-specialized kernels the pipeline , , Heidari and Nikolopoulos 

depth is managed automatically by TMA and the agent instead sweeps the kernel schedule (e.g., KernelTmaWarpSpecialized vs. 

KernelTmaWarpSpecializedPingpong ) and cluster shapes. At the grid level, Split-K problems sweep the split-K factor, while Data Parallel and Stream-K schedules require no additional parameter. To avoid recompiling a separate binary for each configuration, the agent uses a parameterized code generator that emits a sin-gle .cu file containing multiple template instantiations—one per variant—compiled into one shared library with a runtime dispatch index. An auto-tuning script sweeps the full search space, bench-marks each configuration (warmup + timed iterations with CUDA synchronization), records GFLOPS and hardware efficiency, and saves the complete results as JSON. The best-performing configu-ration is selected and used in subsequent stages. 

Action 6: Add to pattern table. Once a pattern is accepted, the agent adds it to the dynamic pattern table T . The entry includes the pattern 𝑝 𝑖 , the synthesized CUTLASS kernel implementation, the auto-tuning results (if performed), the benchmark results (speedup metrics), and metadata about the supporting examples used. The entry is indexed by the tuple (𝑟, 𝜏, 𝛼 ), enabling future queries to retrieve the kernel without re-synthesis. This action expands the optimization catalog, allowing the agent to accumulate knowledge across patterns and reuse kernels for similar workloads. 

## 4.3 Pattern Composition 

The third stage assembles the realized kernels into an optimized model ready for deployment. The composition action constructs a new computation graph G′ by replacing each subgraph G𝑖 with a placeholder node that invokes the corresponding CUTLASS ker-nel. This transformation preserves the semantic equivalence of the model while replacing suboptimal PyTorch operators with special-ized kernels. The agent then executes an extension loading action. Each kernel is compiled into a shared library (.so file) and registered as a PyTorch custom op. The compiled extensions are loaded into the Python runtime, making the optimized kernels callable from PyTorch’s eager execution mode and torchscript compilation path. With the extensions loaded, the agent performs end-to-end bench-marking of the optimized model. The baseline model and the opti-mized model are executed on identical inputs, and wall-clock time measures the overall speedup. This action validates that the compo-sition of individual kernel optimizations translates to model-level performance improvement. The final output is an optimized PyTorch model with CUTLASS kernels integrated as extensions, ready for deployment in infer-ence or training workloads. The dynamic pattern table T persists across optimization sessions, enabling the agent to accumulate opti-mization knowledge and reuse kernels across different models and architectures. 

## 5 Performance Evaluation 5.1 Experimental Setup 

Hardware. All experiments were conducted on an NVIDIA A100-SXM4-80GB GPU (Ampere architecture, SM80), which provides a theoretical peak throughput of 156 TFLOPS in TF32 tensor core mode [ 9]. Additional patterns targeting the H200 GPU (Hopper architecture, SM90) are under development and will be reported in future work. 

Benchmark problems. We evaluate on three KernelBench Level 1 problems drawn from the GEMM family (Problems 1, 3, and 6), each exercising a different grid-level scheduling policy, and on KernelBench Level 3 problem 44_MiniGPTBlock , a MiniGPT-style transformer block with batch size 128, sequence length 512, and embedding width 768. with per-batch dimensions 512 × 1024 and 1024 × 2048 , evaluating Batched GEMM via CUTLASS’s kBatched mode. Software environment. Kernels were implemented using CUT-LASS 3.8.0 (git commit ea46e277), compiled with NVCC 12.4.131 through PyTorch’s torch.utils.cpp_extension interface, and loaded as custom CUDA extensions. All experiments were con-ducted on Ubuntu 24.04 (Linux 6.17.0-14-generic, x86_64) with CUDA 12.4 and PyTorch 2.4.1+cu124. Auto-tuning methodology. Auto-tuning is performed by the agent as Action 5 in the Pattern Realization stage (Section 4.2). The agent infers the search space from the CUTLASS test files for the target architecture and data type, extracting valid thread-block and warp tile shapes, then sweeps pipeline stages or kernel schedules depending on the architecture. For the square matrix multiplication (Problem 1), the agent swept 14 threadblock tile shapes—7 with 𝐾 tile = 16 and 7 with 𝐾 tile = 32 , spanning instruc-tion shapes 16 × 16 × 16 and 16 × 16 × 32 —crossed with 7 pipeline stages (2–8), yielding 98 configurations in total. Each configura-tion was compiled as a standalone shared library, executed with 5 warmup iterations followed by 20 timed iterations, and measured using time.perf_counter with CUDA synchronization. All re-ported timings are averaged over 20 trials; variance was negligible (<1% across configurations), so we report means only. Performance is reported in TFLOPS and as a percentage of the relevant peak throughput (156 TFLOPS for TF32, 312 TFLOPS for FP16 on A100). Configurations that exceeded shared-memory capacity or regis-ter limits at high pipeline stages were recorded as launch failures (status 7). The extreme K/M aspect ratio (2048) makes Stream-K the ap-propriate grid-level scheduling policy [ 11 ]: with only 256 output elements per row, standard Data Parallel tiling produces too few full tiles to saturate all 108 SMs, whereas Stream-K assigns fractional K-tiles to thread blocks for even load distribution. During Stage 2 verification, the agent discovered that using FP16 for both inputs and accumulator caused overflow: summing 524,288 FP16 products (each ∼0.25 for uniform [0, 1) inputs) produces values up to ∼131K, well beyond FP16’s maximum representable value of 65,504. The agent resolved this by switching to mixed precision—FP16 for the A and B operands (to leverage the higher throughput of FP16 tensor cores) with FP32 accumulation and FP32 output.

## 6 Related Work 

Agentic GPU kernel generation and optimization. Recent work demonstrates that LLM-driven agents can generate and op-timize CUDA kernels by closing the loop with compilation, cor-rectness checking, and profiling. KernelBlaster proposes memory-augmented in-context RL to transfer optimization knowledge across tasks and improve KernelBench speedups [ 4]. CUDA Agent scales agentic RL with automated verification/profiling to learn high-performance CUDA kernel generation behavior [ 3]. StitchCUDA extends this direction to end-to-end GPU programs with a planner/-coder/verifier decomposition and rubric-based RL, emphasizing robust rewards from real executions [ 8]. Astra studies a multi-agent workflow that starts from existing CUDA kernels (rather than purely from high-level specs) and iteratively applies profiling-driven transformations to improve performance [13].

Subgraph-aware auto-tuning. Closest to our dynamic pattern table is prior work on exploiting subgraph similarities for efficient auto-tuning [ 7 ], which introduces FamilySeer, a framework that caches and reuses tuning results across similar tensor program subgraphs. Our work differs in three key respects: (1) we use LLM agents to discover patterns automatically rather than relying on manual subgraph extraction; (2) our pattern table is indexed by architecture, data type, and shape in addition to subgraph structure; and (3) we focus on CUTLASS template instantiation rather than search-space exploration within a single compiler backend. 

Library-centric execution engines and tuning. CUrator fo-cuses on efficient LLM execution by integrating CUDA libraries (cuBLAS/CUTLASS) through compiler infrastructure and profiling-driven selection of tiling and split-K strategies for GEMM vari-ants [ 6 ]. Collectively, these systems motivate our focus on expand-ing the search space to newer CUTLASS 3.x kernel schedules (e.g., warp-specialized cooperative and pingpong) and enabling agent-driven exploration for Hopper-specific performance regimes. 

## 7 Conclusion and Future Work 

We presented FACT, a framework that combines library-grounded agentic kernel synthesis with a dynamic pattern registry and archit ecture-specific auto-tuning. Our three-stage workflow—Pattern Dis-covery, Pattern Realization, and Pattern Composition—enables LLM agents to optimize PyTorch modules by instantiating CUTLASS templates for multi-operator patterns, with mandatory auto-tuning that infers search spaces from CUTLASS test files. On KernelBench FACT: Compositional Kernel Synthesis with a Three-Stage Agentic Workflow , , 

Level-1 GEMM problems, auto-tuned CUTLASS kernels achieved 1.06 ×–1.18 × speedups over PyTorch’s cuBLAS baseline. On a Level-3 MiniGPT block, composing fused multi-head attention with fused MLP GEMM+GELU yielded 2.79 × end-to-end speedup, outperform-ing compiler baselines (Inductor: 1.86 ×, Torch-TensorRT: 1.81 ×). 

Future work. Several directions remain for extending FACT. First, we are extending the framework to support Hopper architec-ture (H200 GPU), targeting warp-specialized kernels with TMA and WGMMA instructions. Second, we plan to compare FACT against agentic baselines (KernelBlaster, CUDA Agent, StitchCUDA, As-tra) on the same KernelBench problems to quantify the benefits of library-grounded synthesis. Finally, we are working on open-sourcing the generated CUTLASS kernels and workflow prototype to enable reproducibility and community engagement.
