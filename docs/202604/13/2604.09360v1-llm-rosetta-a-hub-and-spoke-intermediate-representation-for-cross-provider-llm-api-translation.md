## 摘要
大语言模型（LLM）供应商的快速激增——每个供应商都提供专有的 API 格式——导致了一个碎片化的生态系统，使得应用程序与特定供应商紧密耦合。切换或桥接供应商需要 $O(N^2)$ 个双边适配器，阻碍了可移植性和多供应商架构。我们观察到，尽管存在显著的语法差异，但主要的 LLM API 共享一个通用的语义核心：实际挑战在于语法变化的组合表面，而非深层的语义不兼容。基于这一发现，我们提出了 LLM-Rosetta，这是一个基于轴辐式中间表示（IR）的开源转换框架。该 IR 通过一个包含 9 种类型的内容模型和 10 种类型的流式事件模式，捕捉了共享的语义核心——包括消息、内容部分、工具调用、推理轨迹和生成控制。模块化的操作组合转换器架构使得每种 API 标准都能被独立添加。LLM-Rosetta 支持请求和响应负载的双向转换（供应商-到-IR-到-供应商），包括具有状态上下文管理的块级流式传输。我们为四种 API 标准（OpenAI Chat Completions、OpenAI Responses、Anthropic Messages 和 Google GenAI）实现了转换器，涵盖了绝大多数商业供应商。实证评估表明，该框架具有无损的往返保真度、正确的流式行为以及低于 100 微秒的转换开销——在提供双向性和供应商中立性的同时，其性能可与 LiteLLM 的单次转换方法相媲美。LLM-Rosetta 通过了 Open Responses 合规性测试套件，并已在阿贡国家实验室投入生产部署。代码可在 https://github.com/Oaklight/llm-rosetta 获取。

## Abstract
The rapid proliferation of Large Language Model (LLM) providers--each exposing proprietary API formats--has created a fragmented ecosystem where applications become tightly coupled to individual vendors. Switching or bridging providers requires $O(N^2)$ bilateral adapters, impeding portability and multi-provider architectures. We observe that despite substantial syntactic divergence, the major LLM APIs share a common semantic core: the practical challenge is the combinatorial surface of syntactic variations, not deep semantic incompatibility. Based on this finding, we present LLM-Rosetta, an open-source translation framework built on a hub-and-spoke Intermediate Representation (IR) that captures the shared semantic core--messages, content parts, tool calls, reasoning traces, and generation controls--in a 9-type content model and 10-type stream event schema. A modular Ops-composition converter architecture enables each API standard to be added independently. LLM-Rosetta supports bidirectional conversion (provider-to-IR-to-provider) for both request and response payloads, including chunk-level streaming with stateful context management. We implement converters for four API standards (OpenAI Chat Completions, OpenAI Responses, Anthropic Messages, and Google GenAI), covering the vast majority of commercial providers. Empirical evaluation demonstrates lossless round-trip fidelity, correct streaming behavior, and sub-100 microsecond conversion overhead--competitive with LiteLLM's single-pass approach while providing bidirectionality and provider neutrality. LLM-Rosetta passes the Open Responses compliance suite and is deployed in production at Argonne National Laboratory. Code is available at https://github.com/Oaklight/llm-rosetta.

### 论文总结：LLM-Rosetta 跨供应商 LLM API 转换框架

#### 1. 核心问题与整体含义（研究动机和背景）
随着大语言模型（LLM）供应商（如 OpenAI, Anthropic, Google 等）的激增，API 格式呈现碎片化状态。每个供应商都有专有的消息格式、工具调用规范和流式传输协议。这导致了两个主要问题：
*   **供应商锁定（Vendor Lock-in）：** 开发者必须编写特定于供应商的胶水代码，难以灵活切换。
*   **$O(N^2)$ 适配难题：** 若要在 $N$ 个供应商之间实现互操作，传统做法需要建立 $N^2$ 个双边适配器，维护成本极高。
论文观察到，尽管语法（Syntax）差异巨大，但主流 API 的语义核心（Semantic Core）高度一致。因此，研究动机是建立一种**轴辐式（Hub-and-Spoke）**架构，通过中间表示（IR）将复杂度降低至 $O(N)$。

#### 2. 方法论：核心思想与关键技术
LLM-Rosetta 的核心是定义了一个中立的**中间表示（IR）**和一套**模块化转换架构**：
*   **语义 IR 设计：**
    *   **9 种内容模型：** 涵盖文本、图像、音频、文件、工具调用、工具结果、推理轨迹（Reasoning/Thinking）、拒绝（Refusal）和引用（Citation）。
    *   **10 种流式事件模式：** 规范化了不同厂商的 SSE（Server-Sent Events）事件，包括块开始/结束、增量文本、工具调用增量等。
*   **Ops-组合转换器架构：** 将转换逻辑解耦为四个正交模块：
    1.  **ContentOps：** 处理原子内容（如 Base64 图像编码）。
    2.  **MessageOps：** 处理角色映射和对话结构。
    3.  **ToolOps：** 处理工具定义和调用逻辑。
    4.  **ConfigOps：** 处理生成参数（Temperature, Top-p 等）。
*   **双向转换与状态管理：** 支持“供应商 $\leftrightarrow$ IR $\leftrightarrow$ 供应商”的双向路径。针对流式传输，引入了 `StreamContext` 进行状态化管理，以处理不同供应商事件顺序不一致的问题。
*   **元数据模式：** 提供“剥离（Strip）”模式（生成纯净 IR）和“保留（Preserve）”模式（确保无损往返转换）。

#### 7. 优点
*   **架构优雅：** 轴辐式设计将 $O(N^2)$ 复杂度有效降低，新增供应商只需实现一个适配器。
*   **双向性：** 不同于 LiteLLM 等单向代理，它支持请求和响应的全双向转换。
*   **流式处理能力：** 解决了流式传输中复杂的增量合并与状态同步难题。
*   **工业级质量：** 经过生产环境验证，且完全开源（MIT 协议）。

#### 8. 不足与局限
*   **模态覆盖：** 目前主要针对聊天（Chat）模式，尚未覆盖嵌入（Embeddings）、微调（Fine-tuning）或批处理 API。

*   **维护压力：** 随着供应商 API 的快速迭代，转换器模块需要持续的人工维护。
