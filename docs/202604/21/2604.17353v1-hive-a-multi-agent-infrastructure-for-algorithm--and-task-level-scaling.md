## 摘要
大语言模型正越来越多地被部署为复杂的智能体系统，并随任务复杂度的增加而扩展。尽管先前的工作已广泛探索了模型级和系统级扩展，但算法级和任务级扩展在很大程度上仍未得到解决，这限制了智能体系统的全部潜力。在算法层面，分配额外的推理时计算可以增强工作流能力，但会引入跨路径冗余，即多个推理分支之间存在重叠计算。在任务层面，复杂任务可以分解为子问题并委派给多个智能体，以提高可扩展性和并行性。然而，现有基础设施的调度机制无法感知多个智能体的存在，从而错失了优化资源分配的机会。我们提出了 Hive，这是一种支持算法级和任务级扩展的多智能体基础设施。Hive 具有一个描述前端，用于捕获每个智能体的行为并支持测试时扩展算法。利用这一规范，我们的后端引入了两个关键机制：Logits Cache，通过在冗余采样路径中重用中间 Logits 来缓解算法层面的跨路径冗余；以及智能体感知调度（Agent-Aware Scheduling），根据任务层面的智能体贡献高效分配计算和 KV 缓存资源。实验表明，Logits Cache 在重采样方面实现了平均 1.11 倍至 1.76 倍的加速，而智能体感知调度将热点缺失率降低了 33% 至 51%。

## Abstract
Large language models are increasingly deployed as complex agentic systems that scale with task complexity. While prior work has extensively explored model- and system-level scaling, algorithm- and task-level scaling remain largely unaddressed, constraining the full potential of agentic systems. At the algorithm level, allocating additional inference-time computation can enhance workflow capacity but introduces cross-path redundancy: overlapping computations across multiple reasoning branches. At the task level, complex tasks can be decomposed into subproblems and delegated across multiple agents for improved scalability and parallelism. However, existing infrastructures' scheduling is unaware of the existence of multiple agents, missing opportunities to optimize resource allocation.   We propose Hive, a multi-agent infrastructure that enables algorithm- and task-level scaling. Hive features a description frontend that captures per-agent behavior and supports test-time scaling algorithms. Leveraging this specification, our backend introduces two key mechanisms: Logits Cache that reuses intermediate logits across redundant sampling paths to mitigate cross-path redundancy at the algorithm level, and Agent-Aware Scheduling that efficiently allocates compute and KV-cache resources according to agent contributions at the task level. Experiments show that Logits Cache achieves an average speedup of $1.11\times$-$1.76\times$ for re-sampling, and Agent-Aware Scheduling reduces the hotspot miss rate by $33\%$-$51\%$.

### 论文总结：Hive —— 用于算法级与任务级扩展的多智能体基础设施

#### 1. 核心问题与整体含义（研究动机和背景）
随着大语言模型（LLM）从简单的问答引擎转向复杂的智能体（Agentic）系统，推理过程变得更加状态化且复杂。论文指出，现有的推理优化主要集中在**模型级**（如量化）和**系统级**（如并行化），而忽视了**算法级**和**任务级**的扩展需求：
*   **算法级扩展（Algorithm-level Scaling）：** 如“思维树”（Tree-of-Thoughts）等测试时扩展（Test-time Scaling）算法会产生大量分支，导致**跨路径冗余**（不同分支间存在重复的计算和采样）。
*   **任务级扩展（Task-level Scaling）：** 复杂任务被分解为多个异构智能体协作，但现有推理引擎无法感知智能体的角色差异，导致资源分配（如 KV 缓存）低效。
**Hive** 的提出旨在通过统一的基础设施，同时解决这两类扩展带来的效率瓶颈。

#### 2. 论文提出的方法论
Hive 由描述前端和推理后端组成，核心技术包括：
*   **前端描述与异步运行时：** 采用类似 Python 协程（Coroutine）的抽象来定义智能体行为，并引入“监督者”（Supervisor）机制将测试时扩展算法与智能体逻辑解耦。
*   **Logits Cache（算法级优化）：** 
    *   **核心思想：** 缓存已生成分支的 Logits（概率分布），当新分支在相同前缀下重采样时，直接复用 Logits 进行采样，跳过昂贵的模型前向计算。
    *   **采样策略：** 提供“步进采样”（Step-wise）和“热点采样”（Hotspot）。后者通过计算熵和重要性得分，仅在不确定性高的“热点”位置进行重采样，进一步提高复用率。
*   **智能体感知调度（任务级优化）：**
    *   **贡献建模：** 基于博弈论中的 **Shapley Value** 近似计算每个智能体的贡献度。贡献度由“内在效用”（调用频率、Token 量、并发度）和“协作效用”（与其他智能体共享缓存的程度）共同决定。
    *   **优先级驱逐：** 在内存压力下，调度器优先保留高贡献度智能体的 KV 缓存，而非简单的 LRU（最近最少使用）策略。

#### 3. 实验设计
*   **场景与数据集：**
    *   **算法级：** 使用 **24点游戏（24-Game）** 数据集，模拟思维树（ToT）的搜索过程。
    *   **任务级：** 基于 **R3A**（一种用于硬件 RTL 修复的多智能体框架），包含 Decision、Patcher、Viewer、Summary、Searcher 五种角色。
*   **Benchmark 与对比方法：**
    *   以高性能推理框架 **SGLang** 为基准（Baseline）。
    *   对比了开启与关闭 Logits Cache 的性能差异，以及 LRU 调度与智能体感知调度的缓存命中率。

#### 5. 实验数量与充分性
*   **实验规模：** 算法级实验抽取了 100 个实例进行测试；任务级实验模拟了 21 轮完整的多智能体协作流程。
*   **充分性：** 实验涵盖了不同温度（Temperature）参数对采样的影响，并详细分析了不同智能体角色的优先级演变。
*   **客观性：** 实验在受限内存（mem-fraction-static = 0.23）下进行，以真实模拟高压环境下的调度表现，对比设计较为公平。

#### 6. 主要结论与发现
*   **加速效果：** Logits Cache 在重采样任务中实现了 **1.11x 至 1.76x** 的平均加速，且在低温度设置下效果更显著。
*   **缓存优化：** 智能体感知调度将 KV 缓存的“热点缺失率”降低了 **33% – 51%**，并减少了约 20% - 30% 的 Token 驱逐量。
*   **异构性验证：** 实验证实了多智能体系统中不同角色对资源的贡献度存在显著差异，证明了感知调度算法的必要性。

#### 7. 优点
*   **维度创新：** 首次系统性地提出了针对“算法级”和“任务级”扩展的推理基础设施。
*   **Logits 复用机制：** 相比传统的 Prefix Caching（仅复用 KV 缓存），Logits Cache 进一步挖掘了采样阶段的冗余，对搜索类算法意义重大。
*   **数学支撑：** 引入 Shapley Value 进行智能体贡献建模，使资源调度具有较强的理论依据。

#### 8. 不足与局限
*   **模型规模有限：** 实验主要基于 8B 参数模型，尚未在大规模模型（如 70B+ 或混合专家模型 MoE）上验证其 Logits 存储开销与收益的平衡。
*   **领域特定：** 案例研究集中在硬件验证（RTL Repair），虽然具有代表性，但在更通用的多智能体场景（如多智能体辩论、复杂软件工程）下的泛化性有待进一步验证。
*   **存储开销：** Logits Cache 需要额外的内存/显存来存储概率分布，论文虽提到存储在主存中，但未详细讨论大规模并发下的内存带宽瓶颈。
