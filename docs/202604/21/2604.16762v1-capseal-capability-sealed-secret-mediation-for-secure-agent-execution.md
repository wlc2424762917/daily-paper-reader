## Abstract
Modern AI agents routinely depend on secrets such as API keys and SSH credentials, yet the dominant deployment model still exposes those secrets directly to the agent process through environment variables, local files, or forwarding sockets. This design fails against prompt injection, tool misuse, and model-controlled exfiltration because the agent can both use and reveal the same bearer credential. We present CapSeal, a capability-sealed secret mediation architecture that replaces direct secret access with constrained invocations through a local trusted broker. CapSeal combines capability issuance, schema-constrained HTTP execution, broker-executed SSH actions, anti-replay session binding, policy evaluation, and tamper-evident audit trails. We describe a Rust prototype integrated with an MCP-facing adapter, formulate conditional security goals for non-disclosure, constrained use, replay resistance, and auditability, and define an evaluation plan spanning prompt injection, tool misuse, and SSH abuse. The resulting system reframes secret handling for agentic systems from handing the model a key to granting the model a narrowly scoped, non-exportable action capability.

### 1. 论文的核心问题与整体含义
*   **研究动机**：现代 AI 智能体在自动化任务中高度依赖 API 密钥和 SSH 凭据。然而，当前的部署模型通常直接将这些“不记名凭据”（Bearer Credentials）暴露给智能体进程（通过环境变量或本地文件）。
*   **核心挑战**：由于大语言模型（LLM）存在被**提示词注入（Prompt Injection）**、工具滥用或恶意指令操控的风险，智能体可能在执行任务的同时，将这些密钥通过日志、工具参数或网络请求泄露出去。
*   **整体含义**：论文提出将密钥处理模式从“交给模型一把钥匙”转变为“授予模型一个受限的、不可导出的操作能力”，通过中介机制实现密钥的“使用而不接触”。

### 2. 论文提出的方法论
*   **核心思想**：引入一个本地可信的**代理（Broker）**作为中介。智能体不再直接持有密钥，而是向 Broker 请求一个“能力句柄（Capability Handle）”，由 Broker 在受控环境中代为执行涉及密钥的操作。
*   **关键技术细节**：
    *   **能力签发（Capability Issuance）**：Broker 根据策略签发具有特定范围（Scope）的句柄，例如限制 HTTP 请求只能访问特定的 API 路径。
    *   **类型化执行器（Typed Executors）**：
        *   **HTTP 执行器**：支持基于 JSON Schema 的请求体约束，Broker 负责在发送前注入密钥并过滤响应中的敏感信息。
        *   **SSH 执行器**：采用“Broker-exec”模式，Broker 代为执行预定义的命令模板，禁止端口转发，并验证主机指纹。
    *   **会话绑定与防重放**：利用 Unix 域套接字（UDS）提取进程身份，并结合单调序列号、Nonce 和时间戳防止指令重放。
    *   **不可篡改审计（Audit Trail）**：基于哈希链（Hash-Chained）记录所有安全相关事件，提供可验证的证据。

### 6. 论文的主要结论与发现
*   **安全性**：CapSeal 将 HTTP 密钥泄露率从 **1.000（完全泄露）降低到了 0.000**。在 SSH 场景下，由于采用了 Broker 代执行模式，也成功阻止了凭据外泄。
*   **可用性**：在所有良性任务测试中，CapSeal 保持了 **100% 的成功率**，证明安全加固未破坏原有功能。
*   **性能开销**：CapSeal 是所有中介方案中延迟最低的。HTTP 额外开销仅为 **0.149ms**，SSH 额外开销为 **0.208ms**，远优于 S1 和 S2 方案。

### 7. 优点
*   **架构创新**：通过“能力中介”彻底隔离了智能体与原始密钥，从根本上解决了提示词注入导致的密钥泄露。
*   **细粒度约束**：不仅限制访问目标，还能通过 Schema 约束请求内容的“形状”，实现了语义级的安全控制。
*   **高性能实现**：采用 Rust 编写，利用 UDS 和高效的验证逻辑，使安全增强带来的延迟几乎可以忽略不计。
*   **标准化集成**：支持模型上下文协议（MCP），易于集成到现有的智能体生态中。

### 8. 不足与局限
*   **信任边界限制**：假设本地操作系统和 Broker 进程是可信的，无法防御具有 Root 权限的本地攻击者（虽然文中提到未来可通过 TEE 缓解）。
*   **SSH 兼容性**：Broker-exec 模式虽然安全，但对于需要复杂交互或透明 SSH 转发的既有工作流可能存在兼容性挑战。
*   **协议覆盖面**：目前主要针对 HTTP 和 SSH，对于其他专有协议或复杂的数据库连接协议仍需进一步扩展。
*   **审计存储**：目前的审计链存储在本地，尚未实现远程透明度日志（Transparency Log）的自动锚定。
