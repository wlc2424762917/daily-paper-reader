## 摘要
智能体工作流由一系列相互依赖的大语言模型（LLM）调用组成，已成为现代 AI 系统中的主流工作负载。由于推测性执行和并行探索，这些工作流在重叠的提示词（prompt）和中间结果中表现出大量的冗余。现有的 LLM 推理服务系统（如 vLLM）专注于优化单个推理调用，而忽略了跨调用的依赖关系，导致了显著的效率低下。本文从数据系统的视角重新思考了 LLM 和智能体服务，并引入了 Helium。这是一个感知工作流的服务框架，它将智能体工作负载建模为查询计划，并将 LLM 调用视为一等算子（first-class operators）。Helium 集成了主动缓存和感知缓存的调度，以最大限度地提高提示词、KV 状态和工作流之间的复用。通过这些技术，Helium 将经典的查询优化原则与 LLM 推理服务相结合，在各种工作负载上相比最先进的智能体服务系统实现了高达 1.56 倍的加速。我们的结果表明，跨工作流的端到端优化对于构建可扩展且高效的基于 LLM 的智能体至关重要。

## Abstract
Agentic workflows are composed of sequences of interdependent Large Language Model (LLM) calls, and they have become a dominant workload in modern AI systems. These workflows exhibit extensive redundancy from overlapping prompts and intermediate results due to speculative and parallel exploration. Existing LLM serving systems, such as vLLM, focus on optimizing individual inference calls and overlook cross-call dependencies, leading to significant inefficiencies. This paper rethinks LLM and agent serving from a data systems perspective and introduces Helium, a workflow-aware serving framework that models agentic workloads as query plans and treats LLM invocations as first-class operators. Helium integrates proactive caching and cache-aware scheduling to maximize reuse across prompts, KV states, and workflows. Through these techniques, Helium bridges classic query optimization principles with LLM serving, achieving up to 1.56x speedup over state-of-the-art agent serving systems on various workloads. Our results demonstrate that end-to-end optimization across workflows is essential for scalable and efficient LLM-based agents.

*   **研究背景**：
    *   现有的推理引擎（如 vLLM）侧重于优化单个推理请求，将 LLM 调用视为黑盒或独立任务，缺乏对全局工作流结构的感知。
    *   现有的智能体框架（如 LangGraph）虽然能编排复杂逻辑，但无法深入推理引擎层进行 KV 缓存（KV Cache）的精细化复用。
*   **研究动机**：借鉴数据库系统的查询优化原则，将智能体工作流视为“查询计划”，将 LLM 调用视为“一级算子”，通过全局优化消除冗余并提升吞吐量。

### 2. 方法论与关键技术
Helium 框架将工作流建模为有向无环图（DAG），核心技术包括：
*   **查询优化器（Query Optimizer）**：
    *   **算子剪枝与公共子图消除（CSE）**：识别并合并结构相同、输入一致的子图，确保重复计算只执行一次。
    *   **提示词缓存替换**：对于确定性算子（如温度为 0 的 LLM 调用），若输入已处理过，则直接用 `CacheFetch` 算子替换，绕过推理过程。
*   **查询处理器（Query Processor）**：
    *   **模板化基数树（Templated Radix Tree, TRT）**：一种创新的数据结构，不仅建模 Prompt 的前缀结构，还捕获了算子间的依赖关系。
    *   **主动缓存管理（Proactive Caching）**：不同于现有的被动缓存，Helium 在执行前识别静态前缀并预热 GPU 中的 KV 缓存。
    *   **缓存感知调度（Cache-Aware Scheduling）**：基于成本模型（Cost-based）的贪心算法。它通过最小化“Token Step”（总 Token 处理步数）来分配任务，优先保证具有相同前缀的任务在同一 Worker 上连续执行，以最大化缓存命中率。

### 6. 主要结论与发现

*   **扩展性**：随着工作流复杂度和 Batch Size 的增加，Helium 的性能优势更加明显，证明了端到端全局优化的必要性。

### 7. 优点与亮点
*   **视角独特**：首次从数据库系统（Data Systems）视角系统性地重构了 LLM 智能体服务的抽象层。
*   **主动性**：将“被动等待请求”转变为“主动分析工作流结构”，通过预热缓存和智能调度解决了长 Prompt 带来的预填充（Prefill）瓶颈。
*   **低开销**：TRT 结构的内存占用远低于 SGLang 等系统（仅为其 1/20 左右），适合大规模并发场景。

### 8. 不足与局限
*   **动态控制流限制**：目前的 DAG 抽象对于包含复杂循环（Loops）或基于运行时输出动态改变拓扑结构的工作流支持有限。
*   **外部 API 干扰**：实验主要集中在本地 LLM 调用，未深入探讨外部工具调用（Tool Use）带来的不可预测延迟对调度算法的影响。
*   **模型单一性假设**：目前假设工作流使用相同的基座模型，对于混合模型（MoE 或不同尺寸模型混合）的编排优化尚待研究。
