## 摘要
大语言模型（LLM）智能体正日益被期望作为能够解决开放式用户请求的通用系统。虽然现有的基准测试侧重于开发专用智能体的领域感知环境，但评估通用智能体需要更真实的设置，挑战其在统一环境中跨多种技能和工具进行操作的能力。我们推出了 General AgentBench，这是一个为评估搜索、编码、推理和工具使用领域的通用 LLM 智能体提供统一框架的基准测试。利用 General AgentBench，我们系统地研究了顺序缩放（迭代交互）和并行缩放（采样多条轨迹）下的推理时缩放行为。对十个领先 LLM 智能体的评估显示，从特定领域评估转向这种通用智能体设置时，性能出现了大幅下降。此外，我们发现这两种缩放方法在实践中均未能产生有效的性能提升，其原因在于两个根本限制：顺序缩放中的上下文上限（context ceiling）和并行缩放中的验证差距（verification gap）。代码已在 https://github.com/cxcscmu/General-AgentBench 公开。

## Abstract
LLM agents are increasingly expected to function as general-purpose systems capable of resolving open-ended user requests. While existing benchmarks focus on domain-aware environments for developing specialized agents, evaluating general-purpose agents requires more realistic settings that challenge them to operate across multiple skills and tools within a unified environment. We introduce General AgentBench, a benchmark that provides such a unified framework for evaluating general LLM agents across search, coding, reasoning, and tool-use domains. Using General AgentBench, we systematically study test-time scaling behaviors under sequential scaling (iterative interaction) and parallel scaling (sampling multiple trajectories). Evaluation of ten leading LLM agents reveals a substantial performance degradation when moving from domain-specific evaluations to this general-agent setting. Moreover, we find that neither scaling methodology yields effective performance improvements in practice, due to two fundamental limitations: context ceiling in sequential scaling and verification gap in parallel scaling. Code is publicly available at https://github.com/cxcscmu/General-AgentBench.

### 2. 方法论：核心思想与关键技术
*   **核心思想**：构建一个名为 **General AgentBench** 的统一评估框架，将不同领域的工具集成到共享接口中，模拟真实的人机交互。
*   **关键技术细节**：
    *   **统一工具接口**：采用 **Model Context Protocol (MCP)** 协议，将所有基准环境实例化为 MCP 服务器，由统一的 Host 管理全局工具注册表。智能体必须从庞大的工具池中自行识别并调用工具。
    *   **顺序缩放（Sequential Scaling）**：通过延长交互历史（增加轮数）来分配更多计算资源，支持持续推理、反思和探索。
    *   **并行缩放（Parallel Scaling）**：独立采样 $K$ 条轨迹。研究重点在于“自选（Self-choice）”能力，即智能体能否从 $K$ 个候选答案中识别出正确的一个（包括点选和两两对比排序）。
    *   **上下文管理**：由于工具描述和交互历史极长，框架自然地将任务推向了“长上下文”处理范畴。

### 6. 主要结论与发现
*   **性能大幅退化**：从特定领域转向通用设置时，大多数模型性能下降 **10% 到 30%**。Claude Sonnet 4.5 表现最稳健，而 Gemini 2.5 Pro 在某些领域下降超过 60%。
*   **上下文天花板（Context Ceiling）**：顺序缩放并非越长越好。当交互历史超过模型固有的有效处理长度时，性能会进入平台期甚至因推理不稳定性而下降。
*   **验证差距（Verification Gap）**：并行缩放虽然提升了理论上限（pass@K），但模型在“自选”正确答案时表现糟糕。即使是 GPT-5 作为外部验证器，也难以准确评估复杂轨迹的正确性。
*   **跨领域工具红利**：强大的智能体（如 Claude）能自发地跨领域调用工具（例如在搜索任务中调用 Hugging Face API 获取结构化信息），这证明了通用环境的潜在价值。

### 7. 优点：亮点与创新
*   **真实性**：打破了“一个任务对应一套工具”的理想化假设，更贴近真实的 AI Assistant 使用场景。
*   **技术前瞻性**：采用了 MCP 协议，为未来智能体生态的标准化评估提供了范例。
*   **深度诊断**：不仅指出“不行”，还通过注意力分析和验证实验解释了“为什么不行”（上下文上限和验证差距）。

### 8. 不足与局限
*   **采样深度限制**：受限于 API 成本，并行缩放的 $K$ 值仅取到 4，未能探索更大规模采样下的饱和点。
*   **验证器单一**：自选策略主要依赖模型自身或 GPT-5，缺乏更专业、基于规则或强化学习训练的专用轨迹验证器。
*   **动态性不足**：虽然环境是交互式的，但任务目标仍是静态给定的，未涉及用户意图在交互过程中发生漂移的更复杂情况。
